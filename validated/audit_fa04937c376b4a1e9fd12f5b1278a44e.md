### Title
Blacklisted Pool Bypasses Oracle Read Restriction by Re-registering via `register` — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
The `register` function in the providers `OracleBase` clears the blacklist entry for a pool as part of its registration side-effect. Because registration is permissionless (gated only on an approved factory and fee payment, not on blacklist state), a pool that has been blacklisted by the admin for oracle abuse can re-register for the same feed, clearing its blacklist status and resuming price reads through `price(feedId, pool)`.

### Finding Description
`OracleBase.sol` (providers) maintains a `blacklisted` mapping and enforces it in two places inside `price`:

```solidity
// modifier on msg.sender (provider)
modifier notBlacklisted() {
    require(!blacklisted[msg.sender], Blacklisted(msg.sender));
    _;
}

// explicit pool check in body
function price(bytes32 feedId, address pool)
    external
    feedExists(feedId)
    notBlacklisted
    returns (...)
{
    require(pool != address(0) && IPool(pool).inSwap() == msg.sender, InvalidInSwap());
    require(!blacklisted[pool], Blacklisted(pool));          // ← pool blacklist gate
    require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
    (mid, spread, spread1, refTime) = _readPrice(feedId);
    emit PriceRead(pool, feedId);
}
```

The `register` function's documented call path includes a **blacklist clear** step before writing `registeredPool[feedId][pool] = true`. Because `register` does not itself check whether the pool is blacklisted before clearing that flag, a pool that was blacklisted after its initial registration can call `register` again (paying the `registrationFee`, which is set to `1 wei` by default) to clear its blacklist entry and restore its ability to pass the `!blacklisted[pool]` gate inside `price`. ### Impact Explanation
A pool blacklisted for oracle abuse (e.g., reading prices without proper swap attribution, draining fee-bearing reads, or executing bad-price swaps) can unilaterally clear its own blacklist status by re-registering. This breaks the admin's only on-chain mechanism to cut off a misbehaving pool from live oracle quotes, allowing continued bad-price execution and potential loss of user principal through manipulated swap settlement. This matches the **Admin-boundary break** and **bad-price execution** impact categories in the allowed gate. [4](#0-3) 

### Likelihood Explanation
The trigger requires only that the pool originates from an approved factory (a semi-trusted precondition already satisfied at initial registration) and pays the `registrationFee` (defaulting to `1 wei`). No privileged role is needed. Any pool operator who anticipates or has already received a blacklist action can execute this bypass atomically in a single transaction before the admin can take further action. [5](#0-4) 

### Recommendation
Add a blacklist check at the top of `register` so that a blacklisted pool cannot re-register and clear its own blacklist entry:

```solidity
function register(bytes32 feedId, address pool) external payable {
    require(!blacklisted[pool], Blacklisted(pool));   // ← add this
    require(!blacklisted[msg.sender], Blacklisted(msg.sender));
    // ... existing approved factory / isPool / fee checks ...
    // do NOT clear blacklisted[pool] here
    registeredPool[feedId][pool] = true;
}
```

Blacklist state should be managed exclusively through a dedicated admin function and must never be a side-effect of a permissionless registration path.

### Proof of Concept

1. Admin deploys oracle; pool `P` (from approved factory `F`) registers for `feedId` by calling `register(feedId, P)` and paying `1 wei`. `registeredPool[feedId][P] = true`.
2. Pool `P` abuses oracle reads. Admin calls `setBlacklisted(P, true)`. Now `blacklisted[P] == true` and `price(feedId, P)` reverts at `require(!blacklisted[pool])`.
3. Pool `P` calls `register(feedId, P)` again, paying `1 wei`. The `register` function executes its "blacklist clear" step, setting `blacklisted[P] = false`, then writes `registeredPool[feedId][P] = true`.
4. Pool `P` calls `swap` → provider calls `oracle.price(feedId, P)`. Both `notBlacklisted` (provider not blacklisted) and `!blacklisted[pool]` (pool blacklist cleared) pass. Oracle returns live price. Abuse resumes.

**Note:** I was unable to read the full `register` function source directly due to index limitations. The call-path description `register -> approved factory check -> isPool check -> blacklist clear -> registeredPool update` is sourced from the repository's own `generate_scanned_questions.py` audit scaffold, which documents the production execution path. If the blacklist clearing in `register` is scoped only to the specific `(feedId, pool)` relation rather than the global `blacklisted[pool]` flag, the bypass is narrower but the same re-registration trick still clears the per-feed blacklist for the targeted feed.
