### Title
Creator Can Replay Pusher Consent Signature After `revokePusher()` to Silently Re-Establish Revoked Delegation, Feeding Wrong Prices Into Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` uses a deadline-bound EIP-191 signature as the pusher's consent. Because the signature contains no nonce and the deadline is chosen by the creator (not the pusher), a creator can replay the same signature after the pusher has called `revokePusher()`, silently re-establishing the delegation. The pusher then unknowingly continues writing price data into the creator's namespace instead of their own, feeding wrong prices into any pool that reads from the creator's feeds.

---

### Finding Description

**Step 1 – Initial delegation.**
Creator A calls `allowPushers(deadline, [P], [sig])`. The signed message is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

`msg.sender` is creator A; `deadline` is chosen by A (e.g. `now + 365 days`). Pusher P signs this hash once. The result is `namespaceRemapping[P] = A`. [1](#0-0) 

**Step 2 – Pusher revokes.**
P calls `revokePusher()`, setting `namespaceRemapping[P] = address(0)`. P now believes its pushes go to its own namespace. [2](#0-1) 

**Step 3 – Creator replays the old signature.**
Creator A calls `allowPushers(deadline, [P], [sig])` again with the identical arguments. The deadline check passes (deadline has not expired). The ECDSA recovery succeeds (same hash, same signature). `namespaceRemapping[P] = A` is re-written. [3](#0-2) 

**Step 4 – Pusher's updates land in the wrong namespace.**
When P subsequently calls `fallback()`, the oracle reads `namespaceRemapping[msg.sender]` and, finding `A`, routes every slot write into A's namespace instead of P's own namespace. [4](#0-3) 

P's calldata encodes slot IDs and prices intended for P's own feeds. Those values are now stored under A's namespace key `(A_address << 96 | slotId)`. Any pool registered to A's feeds reads P's prices — prices for a different asset, different decimal scale, or different market — and executes swaps against them.

The code's own comment acknowledges the risk but misidentifies the mitigation:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"*

The deadline does not prevent re-establishment **within** the deadline window; it only prevents re-establishment after the deadline expires. Because the deadline is set by the creator, not the pusher, the pusher has no unilateral way to invalidate the signed consent before expiry. [5](#0-4) 

---

### Impact Explanation

Pools registered to creator A's feeds call `getOracleData(feedId)`, which decodes the slot stored under A's namespace. After the replay, that slot contains P's price data (U64x32-encoded price, Codebook256 spread indices) intended for a different feed. The decoded `mid`, `spread0`, and `spread1` values are wrong for A's market. Any swap executed against those values violates swap conservation: the trader receives more or less than the oracle/bin curve permits, or the pool fails to receive the correct input amount. [6](#0-5) 

Simultaneously, P's own feeds stop receiving updates (P's pushes are redirected), causing stale prices in any pool reading P's namespace — a second bad-price execution vector.

---

### Likelihood Explanation

- Creator A already holds a valid, unexpired signature from P (obtained during the original `allowPushers` call).
- No new on-chain interaction from P is required; A acts alone.
- P has no way to detect the re-establishment without monitoring `PusherAuthorized` events.
- The window of exposure equals the remaining time until the deadline — potentially months if A chose a long deadline.
- The trigger is a single transaction by A, requiring no special privilege beyond namespace ownership.

---

### Recommendation

Include a per-pusher nonce in the signed message so that `revokePusher()` (or `removePushers()`) increments the nonce and permanently invalidates all prior signatures:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(
    block.chainid, address(this), deadline,
    pusher, msg.sender,
    pusherNonce[pusher]   // <-- add nonce
))

// In revokePusher / removePushers:
pusherNonce[pusher]++;    // invalidates all prior signatures
namespaceRemapping[pusher] = address(0);
```

Alternatively, let the pusher choose and sign the deadline (not the creator), so the pusher controls the maximum window of their own consent.

### Proof of Concept

```
1. Creator A deploys / uses CompressedOracleV1.
2. Pusher P signs: keccak256(abi.encode(chainid, oracle, deadline=now+365days, P, A))
3. A calls allowPushers(now+365days, [P], [sig])
   → namespaceRemapping[P] = A
4. P calls revokePusher()
   → namespaceRemapping[P] = address(0)
   → P believes it now pushes to its own namespace
5. A calls allowPushers(now+365days, [P], [sig])  // same call, same sig
   → namespaceRemapping[P] = A  (re-established, no new consent from P)
6. P calls fallback() with slot data for P's own ETH/USD feed (e.g. price=X, slotId=0)
   → oracle reads namespaceRemapping[P] = A
   → writes X into A's slot 0 (A's BTC/USD feed)
7. Pool registered to A's BTC/USD feedId reads price X (ETH/USD price)
   → swap executes at wrong price → bad-price execution, loss of user funds
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
