### Title
`allowContractPushers` Allows Silent Namespace Hijack via Live-Call Re-Delegation Without Existing-Mapping Guard - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowContractPushers` proves consent via a live `isPusher(creator)` staticcall instead of a cryptographic signature. Because the function never checks whether `namespaceRemapping[pusher]` is already set to a different creator, any second creator for whom the contract returns `true` can silently overwrite the existing delegation. This is the direct analog of the external EIP-1271 inconsistency bug: the live contract call (like EIP-1271) can behave differently for different callers, and there is no protection against a second caller exploiting that difference to redirect a contract pusher's writes away from the original creator's namespace.

---

### Finding Description

`allowPushers` (EOA path) requires the pusher to sign an EIP-191 message that binds `(block.chainid, address(this), deadline, pusher, msg.sender)`. The signature is deterministic and per-creator: the pusher must explicitly produce a new signature for each creator they consent to, and the deadline prevents post-revocation replay.

`allowContractPushers` (contract path) replaces the signature with a live `pusher.isPusher(msg.sender)` staticcall. The function contains only one guard:

```solidity
if (pusher == msg.sender) { revert NoSelfRemapping(); }
```

There is no check that `namespaceRemapping[pusher]` is already set, and no check that the existing mapping owner has consented to the change. After the live call returns `true`, the mapping is unconditionally overwritten:

```solidity
namespaceRemapping[pusher] = msg.sender;   // line 231 — no prior-mapping guard
```

If the contract pusher's `isPusher` function returns `true` for multiple callers — a realistic scenario for shared price relay contracts, upgradeable contracts, or any contract that does not restrict its `isPusher` to a single creator — any of those callers can invoke `allowContractPushers` and silently redirect the pusher's future fallback writes to their own namespace.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

After the overwrite, every subsequent push from the contract pusher lands in the attacker's namespace, not the original creator's namespace. The original creator's namespace receives no further updates.

---

### Impact Explanation

Price providers (`PriceProvider`, `PriceProviderL2`, `ProtectedPriceProvider`, `AnchoredPriceProvider`) all perform a staleness check before returning a quote:

```solidity
if (refTime == 0) return true;   // _isStale: zero timestamp → stale
```

Once the original creator's namespace stops receiving updates, `refTime` stays at its last value and eventually crosses `MAX_TIME_DELTA`. The provider returns `(0, type(uint128).max)`, which causes `getBidAndAskPrice` to revert with `FeedStalled`. Every pool swap that depends on that feed reverts, making the pool's swap flow completely unusable until the creator re-establishes a delegation — which requires the contract pusher to cooperate.

The corrupted value is the `namespaceRemapping[pusher]` entry, which controls where every future fallback push lands. The downstream corrupted value is the stale `timestampMs` in the original creator's namespace slots, which propagates through `getOracleData` → `price` → provider → pool swap.

---

### Likelihood Explanation

The trigger is unprivileged: `allowContractPushers` is a public function with no access control. The only requirement is that the contract pusher's `isPusher` function returns `true` for the attacker's address. This is realistic for:

- Shared price relay contracts designed to serve multiple creators.
- Upgradeable contracts whose `isPusher` logic changes after the original delegation.
- Any contract that does not restrict `isPusher` to a single hardcoded creator.

The attacker does not need to control the contract pusher itself — they only need to find a contract pusher that returns `true` for their address and is already delegated to a legitimate creator.

---

### Recommendation

Add a guard in `allowContractPushers` that prevents overwriting an existing delegation without the original creator's explicit consent:

```solidity
function allowContractPushers(address[] calldata pushers) external {
    uint256 l = pushers.length;
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];

        if (pusher == msg.sender) {
            revert NoSelfRemapping();
        }

        // NEW: prevent silent overwrite of an existing delegation
        address existing = namespaceRemapping[pusher];
        if (existing != address(0) && existing != msg.sender) {
            revert AlreadyDelegated(pusher, existing);
        }

        (bool ok, bytes memory res) = pusher.staticcall(
            abi.encodeWithSignature("isPusher(address)", msg.sender)
        );
        require(ok);
        bool allowed = abi.decode(res, (bool));
        require(allowed);

        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

Alternatively, require the existing creator to call `removePushers` before a new delegation can be established, mirroring the explicit revocation model already present in `revokePusher` and `removePushers`.
