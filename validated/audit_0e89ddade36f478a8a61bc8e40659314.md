### Title
Creator can replay pusher consent signature to re-establish delegation after `revokePusher()` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1::allowPushers` accepts an EIP-191 pusher-consent signature that is bound to `(chainid, oracle, deadline, pusher, creator)` but carries **no nonce and is never marked as used**. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original signature — as long as the deadline has not expired — to silently re-establish the delegation. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the signed consent.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no per-pusher nonce, no consumed-signature set, and no state that changes between the first and any subsequent call with the same `(deadline, pusher, creator)` tuple. The only gate is `_ensureDeadline`, which checks `block.timestamp <= deadline`. [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` back to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because the mapping is reset to its default value and the signature carries no consumed marker, the creator can call `allowPushers` again with the identical `deadline` and `signature` arguments, passing every check and writing `namespaceRemapping[pusher] = msg.sender` again: [4](#0-3) 

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after the replay still lands in the creator's namespace, not the pusher's own namespace, defeating the revocation.

The code comment itself acknowledges the deadline is the only replay barrier: *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* The deadline is supposed to bound the replay window, but it does not prevent the creator from replaying within that window an unlimited number of times. [6](#0-5) 

---

### Impact Explanation

The pusher's `revokePusher()` call is the only self-service mechanism for a pusher to stop their price data from flowing into a creator's namespace. If the creator replays the consent, every subsequent push by that pusher continues to overwrite the creator's oracle slots. Any pool whose `PriceProvider` reads from those slots will receive prices the pusher intended to stop publishing into that namespace. If the pusher is simultaneously pushing different (e.g., corrected or differently-scaled) data into their own namespace, the pool still consumes the stale creator-namespace data — a bad-price execution path.

---

### Likelihood Explanation

The creator has a direct economic incentive to keep a pusher delegated (continuous price feeds). Replaying the signature is a single on-chain call with no cost beyond gas. The window is as long as the original deadline — typically hours to days. No privileged role beyond being the creator (namespace owner) is required.

---

### Recommendation

Track consumed signatures with a per-pusher nonce that increments on every `revokePusher` / `removePushers` call, and include that nonce in the signed digest:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on every revocation

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusherNonce[pusher], pusher, msg.sender))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
namespaceRemapping[pusher] = address(0);
```

This makes every previously issued consent immediately invalid after revocation, regardless of whether the deadline has expired.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline T+1 day.
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = pusher.sign(
    keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
);

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator);

// 3. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
assert(oracle.namespaceRemapping(pusher) == address(0));

// 4. Creator replays the SAME signature — no revert, delegation restored.
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator); // revocation bypassed

// 5. Pusher's next push still lands in creator's namespace.
vm.prank(pusher);
(bool ok,) = address(oracle).call(buildSlotWord(slotId, tsMs, rawPrice));
assertTrue(ok);
// oracle.getOracleData(feedIdOf(creator, slotId, pos)).price == rawPrice  ← bad price in creator namespace
// oracle.getOracleData(feedIdOf(pusher,   slotId, pos)).price == 0        ← pusher's own namespace empty
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
```
