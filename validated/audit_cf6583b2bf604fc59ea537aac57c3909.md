### Title
Stale contract-pusher delegation persists after `isPusher` authorization is revoked, allowing unauthorized price injection into creator namespace - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowContractPushers` verifies consent via a one-time live `isPusher(creator)` staticcall and permanently writes `namespaceRemapping[pusher] = creator`. The oracle never re-checks `isPusher` on subsequent fallback pushes. If the pusher contract's authorization is later revoked (e.g., via an upgrade, governance change, or internal state mutation that causes `isPusher` to return `false`), the stale `namespaceRemapping` entry persists, and the now-unauthorized pusher can continue injecting arbitrary prices into the creator's namespace indefinitely — exactly as a liquidated position continues to accrue rewards because the NFTPositionManager is never notified.

### Finding Description

`allowContractPushers` establishes delegation by calling `isPusher(creator)` once at registration time: [1](#0-0) 

The code comment explicitly states the design rationale: *"consent is proven by a LIVE `isPusher(creator)` staticcall instead of a signature, so there is nothing to replay and no deadline is needed."* This implies the live check is the ongoing trust anchor. However, the live check is performed **only once** at delegation time. After `namespaceRemapping[pusher] = msg.sender` is written, the fallback push path resolves the namespace purely from storage: [2](#0-1) 

There is no re-verification of `isPusher` on each push. If the pusher contract's internal authorization changes — for example, a proxy upgrade, a governance vote, or a mutable allowlist that removes the creator — `isPusher(creator)` would now return `false`, but `namespaceRemapping[pusher]` still points to the creator. The pusher can continue writing into the creator's namespace via the fallback with any price and a fresh timestamp, bypassing the monotonicity guard by simply incrementing the timestamp.

The contrast with the EOA path is instructive: `allowPushers` requires a deadline precisely because *"an undated signature could re-establish a delegation AFTER the pusher revoked it."* The contract-pusher path has no deadline and no re-check, so a revoked contract pusher is in a strictly worse position than a revoked EOA pusher — the EOA pusher's old signature at least expires.

The stale `namespaceRemapping` entry is the direct analog of the stale `_balances[tokenId][_assetHash]` in the external bug: a state that should reflect current authorization does not, and the downstream consumer (the oracle's price read path) uses the stale state to produce values that reach live swaps.

### Impact Explanation

Prices stored in the creator's namespace are consumed by `PriceProvider`, `ProtectedPriceProvider`, and `AnchoredPriceProvider` via `getOracleData` / `_price`: [3](#0-2) 

Those providers feed `getBidAndAskPrice()` directly into `MetricOmmPool.swap`: [4](#0-3) 

A compromised or governance-revoked contract pusher can push a crafted price (with a fresh timestamp that passes the monotonicity guard and within any configured `priceGuard` bounds) into the creator's namespace. That price propagates through the provider's staleness check (timestamp is fresh), spread check, and price guard, and reaches the pool as the live bid/ask. Traders then execute swaps at the attacker-controlled price, causing direct loss of LP principal or swap conservation failure.

### Likelihood Explanation

The trigger is semi-trusted and reachable without privileged access:

1. Any creator that delegates to a contract with mutable `isPusher` logic (proxy, governance-controlled allowlist, upgradeable contract) is exposed.
2. The pusher contract need not be malicious at delegation time — a legitimate upgrade or governance action that removes the creator from the pusher's allowlist is sufficient.
3. The creator may not notice the authorization change immediately; the window between the `isPusher` change and a manual `removePushers` call is the attack window.
4. No special role or admin key is required to push via the fallback — any EOA or contract can call the fallback once it holds a valid `namespaceRemapping` entry.

### Recommendation

Re-verify `isPusher(creator)` on every fallback push for contract pushers, or introduce a deadline for contract-pusher delegations analogous to the EOA path. The simplest fix is to check `isPusher` in the fallback before resolving the namespace:

```solidity
// In fallback(), after resolving creator from namespaceRemapping:
if (creator != msg.sender) {
    // Re-verify live consent for contract pushers
    (bool ok, bytes memory res) = msg.sender.staticcall(
        abi.encodeWithSignature("isPusher(address)", creator)
    );
    if (!ok || !abi.decode(res, (bool))) {
        // Silently fall back to own namespace instead of creator's
        creator = msg.sender;
    }
}
```

Alternatively, add a deadline to `allowContractPushers` (matching the EOA path) so that stale delegations expire automatically.

### Proof of Concept

```solidity
// MutablePusher: isPusher is controlled by an owner flag
contract MutablePusher {
    address public owner;
    bool public authorized = true;
    constructor(address _owner) { owner = _owner; }
    function isPusher(address) external view returns (bool) { return authorized; }
    function revoke() external { require(msg.sender == owner); authorized = false; }
    // push helper
    function push(address oracle, bytes memory payload) external {
        (bool ok,) = oracle.call(payload);
        require(ok);
    }
}

function test_staleContractPusherInjectsAfterRevoke() public {
    CompressedOracleV1 oracle = new CompressedOracleV1(address(this), 5);
    address creator = address(0xC0FFEE);
    MutablePusher pusher = new MutablePusher(creator);

    // 1. Creator delegates to pusher while isPusher returns true
    address[] memory pushers = new address[](1);
    pushers[0] = address(pusher);
    vm.prank(creator);
    oracle.allowContractPushers(pushers);
    assertEq(oracle.namespaceRemapping(address(pusher)), creator);

    // 2. Creator revokes pusher's internal authorization
    vm.prank(creator);
    pusher.revoke();
    assertFalse(pusher.isPusher(creator)); // isPusher now returns false

    // 3. Pusher still pushes into creator's namespace — namespaceRemapping unchanged
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 badPrice = (uint48(9_999_999) << 16) | (uint48(4) << 8) | uint48(2);
    uint256 word = (uint256(tsMs) << 8) | uint256(0); // slotId=0
    word |= uint256(badPrice) << 208; // position 0
    bytes memory payload = abi.encodePacked(word);

    vm.prank(address(pusher));
    (bool ok,) = address(oracle).call(payload);
    assertTrue(ok, "revoked pusher still writes");

    // 4. Bad price is now in creator's namespace, readable by price providers
    IOffchainOracle.OracleData memory data =
        oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
    assertEq(data.price, U64x32.decode(uint32(badPrice >> 16)),
        "bad price injected into creator namespace after revocation");
}
```

    function allowContractPushers(address[] calldata pushers) external {
        uint256 l = pushers.length;
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            require(ok);
            bool allowed = abi.decode(res, (bool));
            require(allowed);

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
