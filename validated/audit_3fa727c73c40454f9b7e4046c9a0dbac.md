### Title
`allowPushers` Consent Signature Replay Within Deadline Window Allows Creator to Silently Re-establish a Revoked Pusher Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

After a pusher calls `revokePusher()` to clear their delegation, the creator can replay the pusher's original EIP-191 consent signature — unchanged, within the same deadline window — to re-write `namespaceRemapping[pusher] = creator`. The mapping is not permanently cleared on revocation because `allowPushers` carries no nonce or revocation counter. Every subsequent fallback push by the pusher therefore lands in the creator's namespace again, against the pusher's explicit intent.

### Finding Description

`allowPushers` verifies a signature bound to `(block.chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

The signed message contains no nonce, no revocation counter, and no "used" flag. The only freshness gate is the deadline. `revokePusher()` clears the mapping: [2](#0-1) 

But `allowPushers` does not check whether the pusher has already revoked. The creator can immediately call `allowPushers` again with the identical signature and re-set `namespaceRemapping[pusher] = creator`. The fallback push path resolves the namespace at call time: [3](#0-2) 

So every push the pusher makes after their revocation still lands in the creator's namespace, not their own.

The code comment acknowledges the deadline is the only replay guard: [4](#0-3) 

But the deadline only prevents replay *after* it expires. Within the window (up to the maximum the caller chose), the creator can replay the signature an unlimited number of times, undoing every `revokePusher()` call the pusher makes.

### Impact Explanation

A pusher who wants to redirect their data feed to their own namespace (or to a different creator's namespace) cannot do so within the deadline window. Every push they make continues to overwrite the original creator's storage slots. If the pusher is a shared infrastructure node that serves multiple creators, the creator holding the replayed signature can monopolise the pusher's writes, starving any other creator's namespace of fresh data. Pools whose `AnchoredPriceProvider` reads from the starved namespace receive a stale timestamp, causing `_isStale` to return `true` and `getBidAndAskPrice` to revert `FeedStalled`: [5](#0-4) [6](#0-5) 

This renders swaps and liquidity operations on the affected pool unusable for the duration of the deadline window.

### Likelihood Explanation

The creator has a direct economic incentive to keep the pusher's data flowing into their namespace. The replay requires only a single on-chain call with a signature the creator already holds. No front-running or special timing is needed — the creator can call `allowPushers` at any point before the deadline expires, including immediately after observing the pusher's `revokePusher` transaction on-chain.

### Recommendation

Add a per-pusher revocation nonce to the signed message and increment it on every `revokePusher` / `removePushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
namespaceRemapping[pusher] = address(0);
```

Any previously issued signature becomes invalid the moment the nonce advances, making revocation permanent regardless of the deadline.

### Proof of Concept

```solidity
function testRevokedDelegationCanBeReplayedByCreator() public {
    uint256 deadline = block.timestamp + 30 days;

    // 1. Pusher signs consent for creator
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature — revocation is silently undone
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);

    // BUG: mapping is creator again; pusher's revocation had no lasting effect
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 5. Pusher's next push still lands in creator's namespace, not their own
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 raw = _packRaw(999_999, 1, 1);
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
    assertTrue(ok);
    // Data is in creator's namespace — pusher cannot redirect it
    assertEq(
        oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
        U64x32.decode(uint32(raw >> 16))
    );
    // Pusher's own namespace stays empty
    assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0)).price, 0);
}
```
