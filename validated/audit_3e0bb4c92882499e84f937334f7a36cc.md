### Title
No Upper Bound on Per-Bin Additional Fees Bypasses the Admin Fee Cap System - (File: metric-core/contracts/MetricOmmPool.sol, metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setBinAdditionalFees` on `MetricOmmPool` and `setPoolBinAdditionalFees` on `MetricOmmPoolFactory` accept `addFeeBuyE6` and `addFeeSellE6` as raw `uint16` values with **no upper-bound validation**. The global admin spread fee is explicitly capped by `maxAdminSpreadFeeE6` (enforced in `setPoolAdminFees`), but the per-bin additional fee path has no analogous cap, allowing a pool admin to set per-bin fees up to `type(uint16).max` = 65,535 E6 (~6.55%) without any protocol-level check.

### Finding Description

The factory enforces a hard cap on global admin spread fees:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

The per-bin additional fee path has no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

And on the pool itself:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;   // ← no cap check
    s.addFeeSellE6 = addFeeSellE6; // ← no cap check
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

These per-bin fees are then added directly to the base fee during every swap:

```solidity
// MetricOmmPool.sol:999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

```solidity
// MetricOmmPool.sol:1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),
``` [5](#0-4) 

The hard cap constants only govern the global fee path:

```solidity
// MetricOmmPoolFactory.sol:44-45
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20%
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000;
``` [6](#0-5) 

The per-bin additional fee is entirely outside this cap hierarchy. `uint16` max = 65,535 E6 ≈ 6.55% additional fee per bin, additive on top of whatever global spread fee is already set.

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` to set the maximum per-bin additional fee (~6.55%) on any bin. During swaps through that bin, traders pay the global spread fee **plus** the uncapped per-bin fee. This causes:

- **Swap conservation failure**: traders receive less token output than the oracle/bin curve permits at the stated global fee rate.
- **Admin-boundary break**: the pool admin exceeds the intended fee cap system — the global admin spread fee is capped at `maxAdminSpreadFeeE6`, but the per-bin additional fee path has no corresponding cap, allowing the admin to impose additional fees that bypass the protocol's cap enforcement entirely.

### Likelihood Explanation

The pool admin is a semi-trusted role that can be set by the factory owner per pool. Any pool admin can trigger this unilaterally with a single transaction. No timelock, no co-signer, no additional privilege is required beyond holding the `poolAdmin[pool]` role. The factory's `setPoolBinAdditionalFees` passes the values through without any validation. [2](#0-1) 

### Recommendation

Add an upper-bound check in `setPoolBinAdditionalFees` (and/or in `setBinAdditionalFees`) analogous to the global admin fee cap:

```solidity
// In MetricOmmPoolFactory.setPoolBinAdditionalFees:
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Or introduce a dedicated `HARD_MAX_BIN_ADDITIONAL_FEE_E6` constant (e.g., `200_000`) and enforce it in both the factory pass-through and the pool setter, mirroring the pattern used for `HARD_MAX_SPREAD_FEE_E6`. [7](#0-6) 

### Proof of Concept

```solidity
// Pool admin sets per-bin additional fee to uint16 max on bin 0
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, type(uint16).max);
// addFeeBuyE6 = 65535 → ~6.55% additional fee, no revert

// Trader swaps through bin 0 and pays global spread fee + 6.55% extra
// The factory's maxAdminSpreadFeeE6 cap is completely bypassed for this fee component
(,,, uint16 buyFee, uint16 sellFee) = PoolStateLibrary._binState(pool, 0);
assertEq(buyFee, 65535); // succeeds — no cap enforced
assertEq(sellFee, 65535);
```

The global admin fee setter would revert at the same value if `maxAdminSpreadFeeE6 < 65535`, but `setPoolBinAdditionalFees` accepts it silently, confirming the missing cap. [8](#0-7)
