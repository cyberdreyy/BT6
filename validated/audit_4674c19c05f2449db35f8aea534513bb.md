### Title
Missing Sequencer Uptime Check in L2 Price Providers Allows Stale Prices to Reach Pool Swaps — (File: `smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are the L2-specific price providers deployed on Arbitrum, Optimism, and similar chains. Neither contract checks the Chainlink sequencer uptime feed before consuming oracle prices. If the L2 sequencer goes offline, oracle reports (Pyth Lazer / Chainlink Data Streams) cannot be submitted to the chain. When the sequencer restarts, the last stored stale price is consumed by pool swaps if the downtime was shorter than `MAX_TIME_DELTA`, enabling bad-price execution against LPs.

---

### Finding Description

Both `PriceProviderL2.sol` and `ProtectedPriceProviderL2.sol` implement a staleness guard via `_isStale()` parameterised by `MAX_TIME_DELTA` (constructor-enforced range: 1 second – 7 days). Neither contract queries a Chainlink sequencer uptime feed. [1](#0-0) 

```solidity
function _isStale(
    uint256 refTime, uint256 nowTs,
    uint256 maxDelta, uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

This is the **only** temporal guard before prices reach the pool. There is no check for whether the sequencer is currently live or whether it recently restarted. [2](#0-1) 

The `FUTURE_TOLERANCE` immutable (up to 1 hour) is designed for L2 clock skew between the oracle publisher and the sequencer's `block.timestamp`. It does not protect against sequencer downtime — during downtime, no new reports arrive, so `refTime` stays in the past, not the future. [3](#0-2) 

The constructor enforces `FUTURE_TOLERANCE ≤ 1 hour` and `MAX_TIME_DELTA ≤ 7 days`: [4](#0-3) 

A `MAX_TIME_DELTA` of even one hour means a sequencer outage shorter than one hour leaves the stale price fully consumable. The same logic applies identically in `ProtectedPriceProviderL2`: [5](#0-4) 

The registry ABI confirms a `ChainlinkVerifierL2` component **does** exist with a `sequencerUptimeFeed` and `GRACE_PERIOD`, but neither `PriceProviderL2` nor `ProtectedPriceProviderL2` inherit or call it — the check is absent from the price-delivery path that pools actually invoke. [6](#0-5) 

---

### Impact Explanation

**Bad-price execution**: stale bid/ask quotes reach pool swaps. When the sequencer restarts after a downtime shorter than `MAX_TIME_DELTA`:

1. The oracle's stored price is stale (real market has moved).
2. `_isStale()` returns `false` — the price passes.
3. `getBidAndAskPrice()` returns the stale bid/ask to the pool.
4. The pool executes the swap at the stale price.
5. The attacker receives more tokens than the current market permits; LPs absorb the loss.

This satisfies the allowed impact gate: **bad-price execution — stale bid/ask quote reaches a pool swap**, with direct loss of LP principal. [7](#0-6) 

---

### Likelihood Explanation

- L2 sequencer downtime is a documented, recurring risk. Arbitrum experienced a ~7-hour outage in 2022; Optimism has also had incidents.
- No special privileges are required. Any user can call `pool.swap()` the moment the sequencer restarts.
- The attack window is the entire period `[sequencer_restart, oracle_first_update]`, which can be minutes to hours depending on keeper latency.
- `MAX_TIME_DELTA` is a deployment-time parameter with no on-chain floor below 1 second; operators commonly set it to minutes or hours for practical liveness.

---

### Recommendation

Add a sequencer uptime check to both L2 providers, called at the top of `getBidAndAskPrice()`:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour post-restart

error SequencerDown();
error GracePeriodNotOver();

function _checkSequencer() internal view {
    (, int256 answer, uint256 startedAt,,) =
        sequencerUptimeFeed.latestRoundData();
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD)
        revert GracePeriodNotOver();
}
```

Pass `_sequencerUptimeFeed` as a constructor argument (mirroring the existing `ChainlinkVerifierL2` pattern already present in the registry) and call `_checkSequencer()` before the oracle read in `_getBidAndAskPrice()`.

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 3600` (1 hour), backed by a Chainlink Data Streams oracle.
2. Oracle stores price: ETH/USDC = $3 000 at `refTime = T`.
3. Arbitrum sequencer goes offline at `T + 10 s`. No new reports can be submitted.
4. Real ETH price drops to $2 700 (−10 %) during the outage.
5. Sequencer restarts at `T + 1800 s` (30 minutes of downtime).
6. Attacker immediately calls `pool.swap(zeroForOne=false, ...)` (buy ETH with USDC).
7. Pool calls `PriceProviderL2.getBidAndAskPrice()`.
8. `_isStale(T, T+1800, 3600, futureTol)` → `(T+1800 − T) = 1800 ≤ 3600` → **not stale**.
9. Pool quotes bid/ask around $3 000; attacker buys ETH at $3 000 when market is $2 700.
10. Attacker profits ~$300/ETH; LPs are left short by the same amount. [8](#0-7) [9](#0-8)
