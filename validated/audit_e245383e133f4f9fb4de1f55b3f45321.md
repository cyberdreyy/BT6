### Title
`addLiquidity` Bypasses Emergency Pause, Allowing LP Funds to Enter a Paused Pool â€” (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.swap()` is protected by `whenNotPaused`, but `addLiquidity()` is not. When an admin pauses a pool (e.g., due to oracle compromise or price anomaly), new liquidity can still be deposited. Those funds are then exposed to the same risk that triggered the pause the moment the pool is unpaused, undermining the emergency pause mechanism.

### Finding Description

`MetricOmmPool` defines a `whenNotPaused` modifier and a three-level `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused). Only `swap()` enforces this guard:

```solidity
// MetricOmmPool.sol line 224
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
```

`addLiquidity()` carries no such guard:

```solidity
// MetricOmmPool.sol line 188
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

A pool is paused precisely because its current state is unsafe for user funds â€” most commonly because the attached `IPriceProvider` / oracle is returning bad prices. While `swap()` is blocked, `addLiquidity()` remains open, so any LP (including automated vaults or uninformed users) can deposit tokens into the pool during the pause window. Those tokens are then fully exposed to the unsafe oracle the instant the pool is unpaused.

### Impact Explanation

When the pool is unpaused with the underlying oracle issue still present (or before the fix is confirmed), swaps execute against the bad bid/ask price returned by `_getBidAndAskPriceX64()`. LP funds added during the pause are drained by arbitrageurs at the incorrect price. This is a direct loss of LP principal â€” the exact class of loss the pause mechanism is designed to prevent.

### Likelihood Explanation

- Automated LP vaults (e.g., rebalancers) may call `addLiquidity` without checking `pauseLevel`.
- The pause window between detection and fix can be minutes to hours; a single block of exposure is sufficient.
- No special privilege is required: any address can call `addLiquidity` on a paused pool.

### Recommendation

Add `whenNotPaused` to `addLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

`removeLiquidity` intentionally omits the guard (users must be able to exit during emergencies), so no change is needed there.

### Proof of Concept

1. Deploy `MetricOmmPool` with a mutable `PriceProvider` backed by a `PythOracle` / `ChainlinkOracle`.
2. Simulate oracle compromise: push a manipulated price into the oracle.
3. Admin calls `MetricOmmPoolFactory.setPause(poolAddress, 1)` â†’ `pauseLevel = 1`.
4. Confirm `swap()` reverts with `PoolPaused`.
5. Call `addLiquidity(owner, salt, deltas, ...)` â€” succeeds, tokens are deposited.
6. Admin calls `setPause(poolAddress, 0)` (unpauses before oracle is fixed).
7. Attacker calls `swap()` at the manipulated price, draining the LP's newly deposited tokens.
