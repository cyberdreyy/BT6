### Title
`allowPushers` consent signature replay re-establishes revoked pusher delegation within deadline window — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never invalidates it after use. Within the deadline window, the creator can replay the identical signature to re-establish a delegation the pusher already revoked via `revokePusher()`, silently redirecting the pusher's future slot writes back into the creator's namespace.

---

### Finding Description

The consent signature in `allowPushers` is bound to `(block.chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

There is no nonce, no per-use invalidation, and no revocation counter in the signed payload. The deadline is the only replay guard, but it only limits the *window* — it does not prevent the creator from calling `allowPushers` a second time with the exact same `(deadline, sig)` pair while the deadline is still live.

The code comment itself acknowledges the gap: [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But because the signature is never consumed or invalidated, the creator can immediately call `allowPushers` again with the same arguments and restore `namespaceRemapping[pusher] = creator`. The pusher's self-revocation is not final within the deadline window.

After re-delegation, the `fallback` push path resolves the namespace from `namespaceRemapping[msg.sender]`: [4](#0-3) 

So every subsequent push by the pusher — who believes they are writing to their own namespace — silently lands in the creator's namespace instead. This is the exact "silent feed hijack" the delegation path invariant is supposed to prevent.

---

### Impact Explanation

The corrupted value is `namespaceRemapping[pusher]`, which is restored to `creator` after the pusher revoked it. All subsequent `fallback` pushes from the pusher are attributed to the creator's namespace. If the pusher's signing key is compromised (the scenario that typically motivates revocation), an attacker can push arbitrary prices into the creator's oracle slots. Those slots are consumed by `PriceProvider` / `AnchoredPriceProvider` via `getOracleData` → `price()`, and the corrupted mid/spread values reach live pool swaps as bid/ask quotes — satisfying the bad-price execution impact gate.

---

### Likelihood Explanation

The creator is a valid semi-trusted trigger: they hold the consent signature off-chain from the original `allowPushers` call and can replay it in a single transaction at any point before the deadline expires. Deadlines are typically set days in the future (the test suite uses `block.timestamp + 1 days`). The pusher's revocation — the security action that should terminate write authority — is therefore reversible by the creator for the entire remaining deadline window with zero additional cost.

---

### Recommendation

Add a per-pusher nonce or a used-signature bitmap so each consent signature can only establish delegation once:

```solidity
mapping(bytes32 => bool) private _usedConsentSigs;

// inside allowPushers, after signature recovery:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedConsentSigs[sigHash], "signature already used");
_usedConsentSigs[sigHash] = true;
```

Alternatively, include a monotonic per-pusher nonce in the signed payload so that `revokePusher` increments the nonce and invalidates all prior signatures.

---

### Proof of Concept

```
T=0   pusher signs: consent = sign(chainid, oracle, deadline=T+1day, pusher, creator)
T=1   creator calls allowPushers(T+1day, [pusher], [consent])
         → namespaceRemapping[pusher] = creator  ✓

T=2   pusher detects key compromise, calls revokePusher()
         → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are safe)

T=3   creator calls allowPushers(T+1day, [pusher], [consent])  ← SAME signature
         → namespaceRemapping[pusher] = creator  ← delegation silently restored

T=4   attacker (holding compromised pusher key) calls oracle.fallback(badPriceSlotWord)
         → namespace resolved to creator's namespace
         → creator's feed slot overwritten with attacker-controlled price/spread
         → PriceProvider.getBidAndAskPrice() returns attacker-controlled bid/ask
         → pool swap executes against bad price
```
