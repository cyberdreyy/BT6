### Title
`allowPushers` EIP-191 Consent Signature Has No Nonce, Allowing Replay After `revokePusher` — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher self-revokes via `revokePusher()`, anyone who observed the original `allowPushers` transaction can replay it with the identical signature and deadline to silently re-establish the delegation — redirecting the pusher's future slot writes back into the creator's namespace against the pusher's will, and potentially feeding stale or attacker-controlled prices into pools.

### Finding Description

`allowPushers` constructs a signed digest over `(chainid, address(this), deadline, pusher, msg.sender)` and checks only that the deadline has not yet expired:

```solidity
// CompressedOracle.sol L192-211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;
    ...
}
```

`_ensureDeadline` only checks `block.timestamp <= deadline` — there is no nonce, no used-signature bitmap, and no per-pusher revocation counter:

```solidity
// OracleBase.sol L124-126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
```

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`, but this does not invalidate the original signature:

```solidity
// CompressedOracle.sol L238-243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

Because the signed message contains no one-time value, the exact same `(deadline, pushers[], signatures[])` calldata that was submitted in the original `allowPushers` transaction is still valid and can be re-submitted by anyone — including the creator or a third-party attacker — as long as `block.timestamp <= deadline`.

The code's own NatSpec acknowledges the replay concern but only addresses the case of an *undated* signature; it does not address replay *after revocation* within a still-valid deadline window:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The deadline is the intended guard, but it only bounds the window — it does not prevent repeated re-establishment within that window.

### Impact Explanation

After a pusher self-revokes, the creator (or any observer of the original on-chain transaction) can replay `allowPushers` to re-delegate the pusher. The pusher's subsequent `fallback()` pushes are then routed into the creator's namespace:

```solidity
// CompressedOracle.sol L315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

This means:
- A pusher who revoked because they were compromised, or because they no longer wish to serve a creator, can be forcibly re-delegated.
- Their slot writes — potentially stale, incorrect, or attacker-controlled prices — land in the creator's oracle namespace.
- Any pool registered against a `feedId` in that namespace will consume those prices during swaps, resulting in bad-price execution (stale, inverted, or unbounded bid/ask reaching a live pool swap).

### Likelihood Explanation

Delegation signatures are typically issued with long deadlines (days to months) to avoid operational friction. All `allowPushers` calldata is permanently visible on-chain. Any party — the creator, a griefing third party, or an attacker who compromised the pusher — can replay the transaction at any time before the deadline. No special privilege is required; `allowPushers` is a public function.

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` mapping. The simplest fix is to add a per-pusher nonce that is incremented on every successful delegation or revocation, and include it in the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))
);
...
pusherNonce[pusher]++;

// In revokePusher / removePushers:
pusherNonce[msg.sender]++;  // or pusherNonce[pusher]++
```

This ensures that any previously issued consent signature is invalidated the moment the delegation state changes.

### Proof of Concept

```
1. Pusher signs consent for Creator with deadline = block.timestamp + 365 days.
2. Creator calls allowPushers(deadline, [pusher], [sig]).
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher calls revokePusher().
   → namespaceRemapping[pusher] = address(0)  ✓
4. Creator (or any observer of step 2's on-chain calldata) calls
   allowPushers(deadline, [pusher], [sig])  ← identical calldata, still within deadline.
   → _ensureDeadline passes (deadline not yet expired)
   → ECDSA.recover returns pusher  (same signature, same digest)
   → namespaceRemapping[pusher] = creator  ← delegation silently re-established
5. Pusher's subsequent fallback() pushes land in creator's namespace.
   → Stale / attacker-controlled prices are written to creator's oracle slots.
   → Pools registered against creator's feedIds consume bad prices during swaps.
