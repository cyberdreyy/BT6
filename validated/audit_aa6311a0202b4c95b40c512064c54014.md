### Title
Pusher Revocation Is Ineffective Within the Deadline Window — Creator Can Replay Old Consent Signature to Re-Establish Delegation After `revokePusher()` - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies only that the deadline has not expired and that the pusher's EIP-191 signature is valid. It does not check whether the pusher has already self-revoked via `revokePusher()`. Because no nonce or used-signature bitmap is tracked, a creator who still holds a non-expired consent signature can call `allowPushers` again with the exact same signature immediately after the pusher revoked, silently re-establishing the delegation. The code comment explicitly states the deadline is the mechanism that prevents re-establishment after revocation — but the deadline only blocks replay *after it expires*, not during the valid window.

---

### Finding Description

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [1](#0-0) 

`allowPushers` only checks `_ensureDeadline(deadline)` and signature validity before writing `namespaceRemapping[pusher] = msg.sender`:

```solidity
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;
    ...
}
``` [2](#0-1) 

`_ensureDeadline` only checks `block.timestamp <= deadline`: [3](#0-2) 

There is no check that `namespaceRemapping[pusher] == address(0)` (i.e., that the pusher has not already revoked). There is no nonce, no used-signature set, and no revocation flag. The signed consent message is `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))` — it does not encode any revocation state. [4](#0-3) 

The code comment itself acknowledges the risk but mischaracterizes the fix: *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it"* — yet the deadline only closes the window after it expires, not during the valid window. [5](#0-4) 

The `fallback()` push path resolves the namespace at call time from `namespaceRemapping[msg.sender]`, so any re-established delegation immediately routes the pusher's writes back into the creator's namespace: [6](#0-5) 

---

### Impact Explanation

**Bad-price execution reaching a pool swap.**

If a pusher's private key is compromised, the pusher calls `revokePusher()` to stop the attacker from writing bad prices into the creator's namespace. The creator — acting in good faith or otherwise — replays the original (still-valid, non-expired) consent signature via `allowPushers`. The delegation is immediately restored. The attacker continues pushing arbitrary prices into the creator's namespace. Those prices are read by `CompressedOracleV1.price()` → `AnchoredPriceProvider._getBidAndAskPrice()` → `MetricOmmPool.swap()`, producing bad bid/ask quotes that drive pool swaps at incorrect prices, causing direct loss to LPs and traders.

---

### Likelihood Explanation

- Deadlines are set by the creator and can be arbitrarily long (e.g., 30 days, 1 year). During the entire window the old signature remains replayable.
- The creator needs only to call `allowPushers` with the stored signature — a single transaction.
- A creator running an automated keeper that re-establishes delegation on any detected revocation event would trigger this silently and immediately.
- The pusher has no on-chain mechanism to permanently invalidate their consent short of waiting for the deadline to expire.

---

### Recommendation

Track revoked/used consent signatures with a per-pusher nonce or a `usedSignatures` bitmap. The simplest fix is to add a `pusherNonce[pusher]` counter that is incremented on every successful `allowPushers` call and included in the signed digest. A `revokePusher()` call should also increment the nonce, permanently invalidating all previously issued consent signatures for that pusher:

```solidity
// In the signed digest:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In allowPushers, after writing the mapping:
pusherNonce[pusher]++;

// In revokePusher, after clearing the mapping:
pusherNonce[msg.sender]++;
```

Alternatively, add a check at the top of `allowPushers` that reverts if `namespaceRemapping[pusher] != address(0)` (i.e., the pusher is already delegated), forcing the creator to first call `removePushers` before re-delegating — though this does not prevent replay of the exact same signature after a `removePushers` call.

### Proof of Concept

```solidity
// 1. Creator delegates pusher with deadline = now + 30 days
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
// namespaceRemapping[pusher] == creator ✓

// 2. Pusher's key is compromised; pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 3. Creator replays the SAME signature (deadline still valid)
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
// namespaceRemapping[pusher] == creator again — revocation undone

// 4. Attacker (with compromised pusher key) pushes bad price into creator namespace

uint48 badRaw = _packRaw(9_999_999, 5, 0); // manipulated price
vm.prank(pusher); // attacker controls this key
(bool ok,) = address(oracle).call(_wordAt(0, 0, badRaw, tsMs));
assertTrue(ok);

// 5. Bad price is now live in creator's namespace, consumed by price provider → pool swap
IOffchainOracle.OracleData memory data = oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
assertEq(data.price, U64x32.decode(uint32(badRaw >> 16))); // manipulated price confirmed
