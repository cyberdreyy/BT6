The `pay()` function in `PeripheryPayments.sol` is the critical code to examine. I have all the information needed.

The vulnerability is real and confirmed. Here is the full analysis:

---

### Title
Residual ETH on Router Subsidizes Subsequent User's WETH Swap, Causing Prior User to Lose Funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` reads `address(this).balance` without any per-user attribution. When a prior user leaves excess ETH on the router (by sending more `msg.value` than their swap consumes and not calling `refundETH()`), a subsequent user whose swap token is WETH will have their payment silently subsidized by that residual ETH. The prior user loses their ETH; the subsequent user underpays.

---

### Finding Description

The `pay()` function handles WETH payments by first checking the router's native ETH balance: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

When `nativeBalance >= value`, the function deposits exactly `value` ETH — **not all of it** — and transfers WETH to the pool. Any excess ETH above `value` remains on the router after the transaction. Critically, there is no per-user accounting: the balance is a shared contract-level variable.

The `receive()` function blocks plain ETH transfers from non-WETH addresses: [2](#0-1) 

However, ETH sent via `msg.value` in a `payable` function call (e.g., `multicall{value: X}(...)` or `exactInputSingle{value: X}(...)`) is credited to the contract balance regardless of `receive()`. Both entry points are `payable`: [3](#0-2) [4](#0-3) 

The swap callback `_justPayCallback` calls `pay()` with the payer set to `msg.sender` of the original swap call: [5](#0-4) 

---

### Impact Explanation

**User A loses ETH; User B underpays for their swap input.**

Step-by-step:

1. User A calls `exactInputSingle{value: 2 ETH}(params)` where `params.amountIn` corresponds to 1 ETH of WETH.
2. In the swap callback, `pay(WETH, userA, pool, 1e18)` is invoked. `nativeBalance = 2 ETH >= value = 1 ETH`, so exactly 1 ETH is deposited and sent to the pool as WETH. **1 ETH remains on the router.**
3. User A does not call `refundETH()`. The 1 ETH is stranded on the router across transaction boundaries.
4. User B calls `exactInputSingle(params)` with WETH as `tokenIn`, needing 1 ETH worth of WETH. User B sends no ETH (`msg.value = 0`).
5. In the callback, `pay(WETH, userB, pool, 1e18)` is invoked. `nativeBalance = 1 ETH >= value = 1 ETH`, so **User A's stranded ETH** is deposited and sent to the pool as WETH.
6. `safeTransferFrom(userB, ...)` is **never called**. User B pays nothing from their own wallet.
7. User A has permanently lost 1 ETH; User B received a free swap input.

This is a direct loss of user principal (User A) and a swap conservation failure (User B's input is not sourced from User B).

---

### Likelihood Explanation

- Users routinely send slightly more ETH than needed to ensure a swap succeeds, especially when the exact WETH amount is uncertain at signing time.
- The protocol's own documentation acknowledges this pattern: "unused ETH can be reclaimed via `refundETH` in the same multicall."
- Any user who forgets to include `refundETH()` in their multicall, or who calls `exactInputSingle` directly with excess ETH, creates an exploitable state.
- An attacker can monitor the mempool for transactions that leave ETH on the router and immediately follow with a WETH swap to claim it.

---

### Recommendation

In `pay()`, when `token == WETH` and `payer != address(this)`, only use native ETH that was sent in the **current transaction** (i.e., `msg.value`), not the entire `address(this).balance`. One approach is to pass `msg.value` through the call stack and use it as the cap for native ETH consumption, or to track per-transaction ETH attribution in transient storage. Alternatively, enforce that `refundETH()` is always called at the end of any payable multicall by checking `address(this).balance == 0` after each multicall completes.

### Proof of Concept

```solidity
// Foundry integration test sketch
function test_residualEthSubsidizesNextUser() public {
    // User A sends 2 ETH but swap only needs 1 ETH; no refundETH called
    vm.deal(userA, 2 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
        pool: pool,
        tokenIn: address(weth),
        recipient: userA,
        amountIn: 1 ether,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // 1 ETH stranded on router
    assertEq(address(router).balance, 1 ether);

    // User B swaps WETH → token1, needs 1 ETH of WETH, sends no ETH
    uint256 bWethBefore = weth.balanceOf(userB);
    vm.prank(userB);
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool,
        tokenIn: address(weth),
        recipient: userB,
        amountIn: 1 ether,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));

    // User B's WETH balance unchanged — paid with User A's ETH
    assertEq(weth.balanceOf(userB), bWethBefore, "B paid nothing");
    // Router ETH drained
    assertEq(address(router).balance, 0);
    // User A lost 1 ETH permanently
}
```
