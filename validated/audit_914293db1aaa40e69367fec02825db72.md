### Title
Creator can replay pusher's EIP-191 consent signature within the deadline window to silently re-establish delegation after `revokePusher` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` has no revocation nonce and does not check whether `namespaceRemapping[pusher]` was previously cleared by `revokePusher`. A creator who holds a pusher's still-valid consent signature can call `allowPushers` again after the pusher self-revokes, restoring `namespaceRemapping[pusher] = creator` without any fresh consent from the pusher.

### Finding Description
The signature domain is `(block.chainid, address(this), deadline, pusher, creator)`. After `revokePusher` zeroes `namespaceRemapping[pusher]`, nothing in that domain changes. `_ensureDeadline` passes as long as `block.timestamp <= deadline`. `ECDSA.recover` returns the same pusher address. Line 209 unconditionally overwrites `namespaceRemapping[pusher] = msg.sender`. The revocation is silently undone.

Exact replay sequence:
1. Pusher signs consent with `deadline = block.timestamp + 1 days`.
2. Creator calls `allowPushers` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher` → `namespaceRemapping[pusher] = address(0)`.
4. Creator calls `allowPushers` again with the **identical** `(deadline, sig)` before the deadline → `namespaceRemapping[pusher] = creator` again.

### Impact Explanation
After step 4 the pusher's fallback pushes land in the creator's namespace again. If the pusher has already started pushing into their own namespace (the expected post-revocation state), those writes are silently redirected to the creator's feed slots. A pool consuming the creator's feed receives prices that were intended for a different namespace, constituting bad-price execution. Simultaneously, the pusher's own namespace becomes stale; any pool reading from it receives a timestamp-0 or outdated price that every consumer rejects as stale, breaking that feed's liveness.

### Likelihood Explanation
The window is bounded by the original deadline (up to the creator's chosen value, e.g. 1 day). Within that window the attack requires only a single `allowPushers` call from the creator — no special privilege, no off-chain coordination beyond holding the already-used signature. Any creator who wants to retain a pusher against the pusher's will can execute this.

### Recommendation
Add a per-pusher revocation nonce to the signature domain:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on every revoke

// in allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusher, msg.sender, pusherNonce[pusher]))

// in revokePusher:
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

After `revokePusher` increments the nonce, the old signature covers a stale nonce and `ECDSA.recover` will no longer match, making re-delegation impossible without a fresh signature from the pusher.

### Proof of Concept

    // 1. Pusher signs consent

    bytes memory sig = abi.encodePacked(r, s, v);

    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    vm.prank(creator);

    vm.prank(pusher);
    oracle.revokePusher();

    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);

}
