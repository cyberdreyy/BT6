### Title
`AnchoredPriceProvider._isStale` Rejects Valid Oracle Prices on L2 Due to Missing Future-Timestamp Tolerance — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` — described as "the one standard provider for public pools" — uses an L1-only staleness check that unconditionally treats any oracle `refTime` ahead of `block.timestamp` as stale. The protocol is explicitly deployed on Base (L2), where `block.timestamp` is set by the sequencer and can lag behind real wall-clock time. Pyth Lazer and Chainlink Data Streams timestamps are anchored to real time. When the oracle timestamp is even one second ahead of `block.timestamp`, every swap through a pool using `AnchoredPriceProvider` reverts with `FeedStalled`, making the pool completely unusable during those periods.

The protocol already acknowledges this L2 behavior: it ships `PriceProviderL2` and `ProtectedPriceProviderL2` with a `FUTURE_TOLERANCE` parameter for exactly this reason. `AnchoredPriceProvider` has no L2 variant and no such tolerance.

---

### Finding Description

`AnchoredPriceProvider._isStale` is the L1 variant:

```solidity
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // ← any future refTime → stale, no tolerance
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

This is called inside `_readLeg`, which is the only path through which `AnchoredPriceProvider` reads oracle data:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [2](#0-1) 

When `_readLeg` returns `ok = false`, `_getBidAndAskPrice` returns `(0, type(uint128).max)`, and `getBidAndAskPrice` reverts with `FeedStalled`. [3](#0-2) 

The protocol's own L2-aware staleness check, present in `PriceProviderL2` and `ProtectedPriceProviderL2`, tolerates oracle timestamps slightly ahead of `block.timestamp`:

```solidity
if (refTime > nowTs) {
    // refTime in the future: tolerate only within futureTol
    return (refTime - nowTs) > futureTol;
}
``` [4](#0-3) 

`AnchoredPriceProvider` has no equivalent parameter and no L2 variant. [5](#0-4) 

The `refTime` value originates from `data.timestampMs.toSeconds()` stored in the provider oracle (`PythOracle` / `ChainlinkOracle`). For Pyth Lazer, this is the per-feed `FeedUpdateTimestamp` (property ID 12), which is the real-world time the price was observed: [6](#0-5) 

For Chainlink Data Streams V3/V4, it is `observationsTimestamp * 1000` (real wall-clock seconds converted to ms): [7](#0-6) 

Both are real-time anchored. On Base, `block.timestamp` is the sequencer's reported timestamp and can lag behind real time by several seconds. When `refTime` (real time) > `block.timestamp` (sequencer time), `_isStale` returns `true` and the swap reverts.

---

### Impact Explanation

Every swap through a pool whose price provider is `AnchoredPriceProvider` reverts with `FeedStalled` whenever the oracle's real-time timestamp exceeds the sequencer's `block.timestamp`. This makes the pool completely unusable for the duration of the lag. Because Base sequencer clock skew is a structural property of the chain (not a transient anomaly), this condition recurs regularly. LPs cannot earn fees; traders cannot swap; the pool's core function is broken.

---

### Likelihood Explanation

- The protocol explicitly targets Base (L2) for deployment.
- The protocol already ships L2 variants (`PriceProviderL2`, `ProtectedPriceProviderL2`) that add `FUTURE_TOLERANCE` for exactly this reason, confirming the developers are aware of the L2 clock-skew problem.
- `AnchoredPriceProvider` is described as "the one standard provider for public pools," meaning it is the primary provider for production pools on Base.
- Pyth Lazer and Chainlink Data Streams timestamps are real-time; Base `block.timestamp` is sequencer-controlled and routinely lags by 1–5 seconds.
- No privileged action is required to trigger this: it occurs passively on every swap during any lag period.

---

### Recommendation

Introduce an `AnchoredPriceProviderL2` variant (or add an optional `futureTolerance` immutable to `AnchoredPriceProvider`) that mirrors the L2-aware staleness logic already present in `PriceProviderL2`:

```solidity
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol   // e.g. 60 seconds, bounded at construction
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

Bound `futureTol` at construction (e.g. `<= 1 hours`) consistent with `PriceProviderL2`'s `FutureToleranceOutOfBounds` guard. [8](#0-7) 

---

### Proof of Concept

1. Deploy `AnchoredPriceProvider` on Base pointing to a `PythOracle` with `MAX_REF_STALENESS = 60`.
2. Pyth Lazer pushes a price update; the stored `timestampMs` corresponds to real time `T = 1_000_000_005` (seconds).
3. A swap is initiated in a block where `block.timestamp = 1_000_000_000` (sequencer 5 s behind real time).
4. `_readLeg` calls `oracle.price(feedId, pool)` → `refTime = 1_000_000_005`.
5. `_isStale(1_000_000_005, 1_000_000_000, 60)`:
   - `refTime > nowTs` → `true` → returns `true` (stale).
6. `_readLeg` returns `ok = false`.
7. `_getBidAndAskPrice` returns `(0, type(uint128).max)`.
8. `getBidAndAskPrice` reverts `FeedStalled`.
9. The swap fails. Every subsequent swap in every block where the sequencer lags behind real time fails identically.
