### Title
Pusher Consent Signature Has No Nonce, Allowing Creator to Replay `allowPushers` and Silently Re-establish Delegation After `revokePusher` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts an EIP-191 consent signature whose domain is `(chainid, address(this), deadline, pusher, creator)`. There is no nonce and no used-signature bitmap. A creator who holds a pusher's original consent can call `allowPushers` repeatedly with the same signature any number of times before the deadline expires. A pusher who calls `revokePusher()` to stop writing into the creator's namespace will have that revocation silently undone the next time the creator replays the original signature, causing the pusher's subsequent fallback pushes to land in the creator's namespace instead of the pusher's own.

---

### Finding Description

`allowPushers` signs over `(chainid, address(this), deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = creator`: [1](#0-0) 

`revokePusher` clears the mapping to `address(0)`: [2](#0-1) 

Because the signature domain contains no nonce and no "not-before" field, the same `(deadline, pusher, creator)` tuple is valid for the entire window `[T_sign, deadline]`. The creator can call `allowPushers` again with the identical signature bytes after the pusher has revoked, restoring `namespaceRemapping[pusher] = creator` without any new consent from the pusher.

The code comment itself acknowledges the risk but misidentifies the deadline as a sufficient fix: [3](#0-2) 

The deadline prevents replay *after* it expires, but does nothing to prevent replay *within* the window. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the original consent.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So any push the pusher makes after believing they have revoked will silently land in the creator's namespace, not their own.

---

### Impact Explanation

The creator's namespace is the source of truth for `feedIdOf(creator, slotIndex, positionIndex)`. That feed id is bound to an `AnchoredPriceProvider` which calls `getBidAndAskPrice()` during live pool swaps: [5](#0-4) 

If the pusher, believing they have revoked, begins pushing data for a different asset pair or a different price scale into their own namespace, those slot words are silently redirected to the creator's namespace. The pool then reads the wrong price (wrong asset, wrong scale, or stale data from a pusher who intended to stop) and executes swaps at a bad bid/ask. This is a direct bad-price execution path reaching live pool swaps.

---

### Likelihood Explanation

The trigger requires the creator to retain the original consent bytes and replay them after the pusher revokes. This is a deliberate creator action, not an accidental one. However:

- The creator is not the oracle admin (the `ADMIN_ROLE` holder); they are a namespace owner whose actions are not unconditionally trusted.
- The pusher's revocation is a security action (e.g., key rotation, stopping service, or switching to their own namespace). The creator overriding it is a trust-boundary violation.
- The pusher has no on-chain way to detect the re-delegation without monitoring `PusherAuthorized` events, so the hijack can be silent.

Likelihood is **Medium**: requires a creator who is willing to act against the pusher's explicit revocation, but the mechanism is trivially executable with no special access.

---

### Recommendation

Add a per-pusher nonce to the consent domain and increment it on every successful `allowPushers` call (or on every `revokePusher`/`removePushers` call). The signed hash should become:

```
keccak256(abi.encode(chainid, address(this), deadline, pusher, creator, nonce[pusher]))
```

Alternatively, record used `(deadline, pusher, creator)` tuples in a consumed-signature set and revert on replay. Either approach makes a replayed consent invalid after the pusher has revoked.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = block.timestamp + 1 days.
//    Creator calls allowPushers → namespaceRemapping[pusher] = creator.

// 2. Pusher calls revokePusher() → namespaceRemapping[pusher] = address(0).
//    Pusher believes they are now pushing to their own namespace.

// 3. Creator calls allowPushers again with the SAME signature bytes
//    (deadline still in the future) → namespaceRemapping[pusher] = creator again.

// 4. Pusher pushes a slot word intended for their own namespace.
//    fallback() resolves: creator = namespaceRemapping[msg.sender] = creator (not address(0)).
//    The word lands in the creator's namespace, not the pusher's.

// 5. AnchoredPriceProvider reads feedIdOf(creator, slot, pos) → wrong price → bad swap.
```

The existing test `testRevokePusherRestoresOwnNamespace` verifies the happy path (revoke → own namespace) but does not test the creator replaying the original consent after revocation, which is the missing invariant check. [6](#0-5)

        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "mapping should clear");
