### Title
`allowPushers` Consent Signature Is Replayable Within the Deadline Window, Permanently Bypassing a Pusher's Self-Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` binds a pusher's consent to a `deadline` but stores **no nonce and no used-signature record**. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original consent signature (while the deadline is still live) to re-establish `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone, and any automated pusher that continues to emit slot words will have those writes land in the creator's namespace against the pusher's intent, feeding potentially bad prices into every pool that reads from that namespace.

---

### Finding Description

`allowPushers` constructs the consent hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only freshness gate is `_ensureDeadline(deadline)`, which rejects calls after the deadline but places **no restriction on how many times the same `(deadline, pusher, creator)` triple can be accepted before it expires**. [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because `allowPushers` does not mark the signature as consumed, the creator can call it again with the **identical** `(deadline, pusher, sig)` tuple immediately after the pusher revokes, writing `namespaceRemapping[pusher] = creator` again. The code's own comment acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and claims the deadline prevents it — but the deadline only prevents use *after expiry*, not repeated use *before* expiry. [4](#0-3) 

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every slot word the pusher emits after the creator's replay lands in the creator's namespace, not the pusher's own, and is immediately readable by any pool registered against that namespace's feed IDs.

---

### Impact Explanation

An automated pusher (e.g., a keeper bot) that calls `revokePusher()` to detach from a creator — perhaps because the creator is routing its prices to a pool the pusher no longer trusts — will have its revocation silently undone on the next block. The pusher's subsequent slot writes continue to overwrite the creator's namespace feeds. Any pool that reads those feeds via `price(feedId, pool)` receives prices the pusher no longer intends to provide to that creator, which may be stale, mis-scaled, or otherwise bad relative to the pool's expected source. This satisfies the **bad-price execution** impact: an unconsented bid/ask quote reaches a live pool swap.

---

### Likelihood Explanation

The attack is fully permissionless for the creator: it requires only replaying a signature they already hold, with no additional privilege. The only precondition is that the deadline has not yet expired. Pushers commonly sign consents with multi-day or multi-week deadlines (the test suite uses `block.timestamp + 1 days`), giving the creator a wide replay window. An automated pusher that does not monitor `PusherAuthorized` events on-chain will not detect the re-delegation. [6](#0-5) 

---

### Recommendation

Track consumed consent signatures with a `mapping(bytes32 => bool) usedConsentHash` and revert if the hash has already been accepted. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, so a revoked pusher can invalidate all prior signatures by incrementing their nonce via a self-call.

---

### Proof of Concept

```
1. Pusher signs: hash = keccak256(abi.encode(chainid, oracle, deadline=T+1day, pusher, creator))
2. Creator calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are detached)
4. Creator immediately calls allowPushers(T+1day, [pusher], [sig])  ← SAME sig, deadline still valid
   → namespaceRemapping[pusher] = creator  ← revocation silently undone
5. Pusher's automated bot emits a slot word via fallback()
   → namespace resolves to creator (step 4), not pusher
   → creator's feed slot is overwritten with pusher's price data
6. Pool registered against creator's feedId reads the price via price(feedId, pool)
   → receives a price the pusher no longer consents to provide to this creator
``` [2](#0-1) [7](#0-6)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

    fallback() override external {
        uint256 end;
        uint256 namespace;

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

        assembly ("memory-safe") {
            end := calldatasize()
            namespace := shl(96, creator) // [creator:20][zeros:12]
        }

        // 4 * 6 + 7 + 1 = 32 bytes per slot
        if (end == 0 || end % 32 != 0) revert BadCalldataLength();

        for (uint256 ptr = 0; ptr < end; ptr += 32) {
            uint256 word;
            assembly ("memory-safe") {
                word := calldataload(ptr)
            }
            // casting to 'uint8' is safe we want LSB
            // forge-lint: disable-next-line(unsafe-typecast)
            uint8 slotId = uint8(word);
            TimeMs timestampMs = toTimeMs(word >> 8 & X56);
            timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
            bytes32 key = bytes32(namespace | uint256(slotId));
            uint256 old = uint256(_loadStorage(key));
            TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

            bool newer = timestampMs.isAfter(oldTimestampMs);
            if (!newer) continue;

            _writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));

    function testAllowPushersDelegatesNamespace() public {
        uint256 deadline = block.timestamp + 1 days;
        _allowPusher(deadline);
        assertEq(oracle.namespaceRemapping(pusher), creator, "pusher should map to creator");

        // delegated push lands in the CREATOR namespace, not the pusher's own
        uint56 tsMs = uint56(block.timestamp * 1000);

        IOffchainOracle.OracleData memory own = oracle.getOracleData(oracle.feedIdOf(pusher, 2, 3));
