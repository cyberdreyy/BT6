### Title
`PriceProviderL2` and `ProtectedPriceProviderL2` lack Arbitrum sequencer uptime check, allowing stale oracle prices to reach pool swaps — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

Both L2 price provider contracts rely solely on a `MAX_TIME_DELTA` staleness window to reject stale oracle data. Neither checks whether the Arbitrum (or any L2) sequencer is currently live. When the sequencer goes down and later recovers, the last-pushed oracle price — which was valid at push time — can still pass the staleness check if the outage lasted less than `MAX_TIME_DELTA`. That stale price then reaches the pool's `getBidAndAskPrice()` call and drives a swap at a price that no longer reflects the market.

---

### Finding Description

`PriceProviderL2._isStale` and the identical copy in `ProtectedPriceProviderL2._isStale` implement the only temporal guard before a price is forwarded to the pool:

```solidity
// PriceProviderL2.sol lines 135-150
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;   // future-tolerance branch
    }
    return (nowTs - refTime) > maxDelta;         // past-staleness branch
}
``` [1](#0-0) 

The identical logic appears in `ProtectedPriceProviderL2`: [2](#0-1) 

There is no call to a Chainlink sequencer-uptime feed or any equivalent liveness signal anywhere in either contract.

**Attack path:**

1. At time `T`, an off-chain pusher stores a fresh price in the oracle (e.g. via `ChainlinkOracle.updateReport`). The stored `timestampMs` converts to `refTime ≈ T`.
2. The Arbitrum sequencer goes offline at time `T`. No new L2 blocks are produced; `block.timestamp` is frozen at `T`.
3. The real-world market price moves significantly (e.g. a 5 % crash) during the outage.
4. The sequencer recovers at time `T + D` (where `D < MAX_TIME_DELTA`). `block.timestamp` jumps to `T + D`.
5. A trader calls `swap()` on the pool. The pool calls `getBidAndAskPrice()` on `PriceProviderL2`, which calls `_isStale(T, T+D, MAX_TIME_DELTA, FUTURE_TOLERANCE)`. Because `D < MAX_TIME_DELTA`, the check returns `false` — the price is accepted as fresh.
6. The pool executes the swap at the pre-outage (stale) price, giving the trader a better-than-market rate at the LP's expense.

The `FUTURE_TOLERANCE` immutable (up to 1 hour) compounds the issue: a price pushed with `refTime = T + FUTURE_TOLERANCE` just before the outage extends the effective acceptance window to `MAX_TIME_DELTA + FUTURE_TOLERANCE`. [3](#0-2) 

The `MAX_TIME_DELTA` constructor bound is 7 days, so a pool operator could inadvertently configure a very wide window: [4](#0-3) 

---

### Impact Explanation

A stale bid/ask quote reaches `MetricOmmPool.swap()`. The pool's `SwapMath` computes output amounts against the stale price, so the trader receives more tokens than the current market rate permits. LPs bear the loss because their reserves are depleted at an unfavorable rate. This is a direct loss of LP principal matching the "bad-price execution" and "swap conservation failure" impact categories.

---

### Likelihood Explanation

Arbitrum sequencer outages are documented historical events (multiple incidents in 2022–2023). The vulnerability requires only that the outage duration is shorter than `MAX_TIME_DELTA` — a condition that is almost always true for any reasonable staleness window (minutes to hours). No privileged access or malicious setup is required; any trader can submit a swap immediately after sequencer recovery.

---

### Recommendation

Add a Chainlink L2 sequencer uptime feed check at the top of `_getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`, following the pattern recommended by Chainlink:

```solidity
// Example addition to PriceProviderL2 constructor / immutables:
AggregatorV2V3Interface public immutable sequencerUptimeFeed;
uint256 public immutable GRACE_PERIOD; // e.g. 3600 seconds

// In _getBidAndAskPrice(), before the oracle read:
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
// answer == 0 means sequencer is up; 1 means down
if (answer != 0) return (0, type(uint128).max); // sequencer down → stale
if (block.timestamp - startedAt < GRACE_PERIOD) return (0, type(uint128).max); // grace period
```

This ensures that prices pushed before a sequencer outage are never consumed during or immediately after recovery.

       _isStale(0, 300, 600, 0) → (300 - 0) = 300 < 600 → false (not stale).
