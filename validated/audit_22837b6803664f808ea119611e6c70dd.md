### Title
Creator Can Replay Valid Pusher Signature Within Deadline Window to Re-Establish an Explicitly Revoked Delegation, Causing Stale Prices in the Pusher's Pool — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` does not check whether the pusher has already self-revoked their delegation. A creator who holds a still-valid (deadline not yet expired) pusher consent signature can call `allowPushers` again after the pusher calls `revokePusher()`, silently re-writing `namespaceRemapping[pusher] = creator`. The pusher's subsequent fallback pushes then land in the creator's namespace instead of the pusher's own namespace, starving the pusher's own pool of fresh price data and enabling stale-price execution.

---

### Finding Description

The code comment on `allowPushers` explicitly acknowledges the replay-after-revocation risk and claims the deadline solves it:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [1](#0-0) 

However, `_ensureDeadline` only checks `block.timestamp <= deadline`. [2](#0-1) 

Within the deadline window the check passes unconditionally. There is no flag, nonce, or revocation counter that would distinguish a fresh consent from a replayed one. After `revokePusher()` sets `namespaceRemapping[pusher] = address(0)`, `allowPushers` treats that zero value identically to "never delegated" and overwrites it with `msg.sender` again — the exact same pattern as the external bug where `maxSellPercent = 0` was misread as "uninitialized" and reset to a default. [3](#0-2) 

The fallback push path resolves the namespace at call time:

```
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after revocation — intended for their own namespace — is silently redirected back into the creator's namespace the moment the creator replays the old signature.

---

### Impact Explanation

- **Pusher's pool receives stale prices.** The pusher's pool is keyed to `feedIdOf(pusher, slotIndex, positionIndex)`. Once the delegation is re-established, every push the pusher makes updates the creator's slot, not the pusher's. The pusher's slot timestamp freezes at the last pre-revocation value. Any pool consuming that feed will read a stale quote and execute swaps at the wrong price — a direct bad-price execution impact.
- **Creator's pool receives unintended data.** If the pusher has switched to pushing data for a different asset or price range (their own pool), the creator's pool now ingests that mismatched data, potentially inverting or unbounding the bid/ask quote it presents to traders.

Both outcomes satisfy the "bad-price execution" allowed impact gate.

---

### Likelihood Explanation

- Deadlines are routinely set days or weeks in the future (the test suite uses `block.timestamp + 1 days`).
- The pusher's revocation intent is clear and on-chain, but the creator retains the off-chain signature and can replay it at any time before expiry.
- The creator need not be malicious: they may replay the signature believing the delegation lapsed due to a bug, or to restore service after a perceived pusher outage — exactly the "accidental admin override" pattern of the external report.
- No privileged role is required; any creator who previously called `allowPushers` successfully holds the replay capability.

---

### Recommendation

Introduce a per-pusher revocation nonce. Increment it inside `revokePusher` and `removePushers`, and include it in the signed digest inside `allowPushers`:

```solidity
mapping(address => uint256) public revocationNonce;

// in revokePusher / removePushers:
revocationNonce[pusher]++;

// in allowPushers digest:
keccak256(abi.encode(
    block.chainid, address(this), deadline,
    pusher, msg.sender,
    revocationNonce[pusher]   // ← binds consent to current revocation epoch
))
```

A signature produced before a revocation commits to the old nonce and is permanently invalid after `revokePusher` increments it, regardless of whether the deadline has passed.

### Proof of Concept

```
// 1. Pusher P signs consent for creator A; deadline = now + 1 day
bytes memory sig = sign(PUSHER_KEY, abi.encode(chainid, oracle, deadline, P, A));

// 2. Creator A establishes delegation
vm.prank(A);
oracle.allowPushers(deadline, [P], [sig]);
// namespaceRemapping[P] == A  ✓

// 3. Pusher P revokes — intends to push for own pool from now on
vm.prank(P);
oracle.revokePusher();
// namespaceRemapping[P] == address(0)  ✓

// 4. P pushes fresh price data — expects it to land in P's namespace
vm.prank(P);
oracle.call(wordAt(slotId, pos, freshPrice, tsMs));
// lands in A's namespace, NOT P's — P's pool still reads stale price

// 5. Creator A replays the SAME signature (deadline still valid)
vm.prank(A);
oracle.allowPushers(deadline, [P], [sig]);   // ← no revert
// namespaceRemapping[P] == A again

// 6. All subsequent P pushes continue to feed A's namespace
//    P's pool: getOracleData(feedIdOf(P, slotId, pos)).timestampMs == stale
//    A's pool: getOracleData(feedIdOf(A, slotId, pos)).price == P's unintended data
```

    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

    /// @notice Allows a pusher to self-revoke their delegation. After revocation the
    ///         wallet pushes into its OWN namespace again (the registrationless default).
    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
