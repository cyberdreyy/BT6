### Title
`allowPushers` Consent Signature Replay Within Deadline Window Allows Creator to Re-establish Revoked Pusher Delegation — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` uses a deadline to prevent replay of a pusher's EIP-191 consent signature after expiry. However, there is no nonce or used-signature tracking, so the same signature is replayable an unlimited number of times **within** the deadline window. A creator can call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple after the pusher has called `revokePusher()`, silently re-establishing the delegation the pusher explicitly cancelled. The code's own comment acknowledges the replay risk but the deadline only closes the post-expiry window, leaving the within-window window open.

### Finding Description

`allowPushers` signs consent over `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`. [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)`, which reverts when `block.timestamp > deadline`. [2](#0-1) 

There is no nonce, no per-signature consumed-flag, and no check that `namespaceRemapping[pusher]` is currently `address(0)` before writing. The code comment explicitly acknowledges the risk:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [4](#0-3) 

Because `allowPushers` unconditionally overwrites `namespaceRemapping[pusher] = msg.sender` without checking whether the mapping was already cleared by a revocation, the creator can replay the original signature to restore the delegation. [5](#0-4) 

After re-delegation, the pusher's `fallback()` pushes resolve the namespace to the creator again: [6](#0-5) 

### Impact Explanation

After the pusher revokes and begins pushing into their **own** namespace (to serve their own pools), the creator replays the consent to redirect those pushes back to the creator's namespace. The pusher's own namespace stops receiving updates. Any pool whose `AnchoredPriceProvider` reads from `feedIdOf(pusher, slotIndex, positionIndex)` will see a timestamp that no longer advances; `_isStale` returns `true` and `_readLeg` returns `ok = false`: [7](#0-6) 

`getBidAndAskPrice` then returns the `(0, type(uint128).max)` sentinel and the pool reverts `FeedStalled` on every swap — broken core pool functionality, unusable swap/liquidity flows for the duration of the attack. [8](#0-7) 

### Likelihood Explanation

The attack requires only that the creator still holds a valid (non-expired) consent signature. Consent signatures are typically issued with multi-day or multi-week deadlines (the test suite uses `block.timestamp + 1 days`). Any creator who received a pusher's consent and later had that pusher revoke can replay the signature at any time before expiry. No privileged role is needed; the creator is a normal EOA. [9](#0-8) 

### Recommendation

Track consumed signatures. The simplest fix is a `mapping(bytes32 => bool) private _usedConsents` keyed on the signature hash (or on `keccak256(abi.encode(deadline, pusher, creator))`), set to `true` on first use and checked before writing `namespaceRemapping`. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, so a revoked pusher must issue a fresh signature with the new nonce before delegation can be re-established.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = T + 1 day.
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes — intends to push into own namespace.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature before deadline expires.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds — no revert
assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation restored

// 5. Pusher's subsequent pushes land in creator's namespace, not pusher's own.
//    Pools reading feedIdOf(pusher, ...) now see a frozen timestamp → FeedStalled.
``` [10](#0-9)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
