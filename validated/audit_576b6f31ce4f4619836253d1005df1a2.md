### Title
`AnchoredPriceProvider` applies a single `MAX_REF_STALENESS` to both legs of a synthetic two-feed pair, allowing stale prices from the faster-updating feed to reach pool swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

### Summary

`AnchoredPriceProvider` supports a synthetic ratio mode (`quoteFeedId != 0`) that computes `price(baseFeedId) / price(quoteFeedId)` (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). Both legs are validated against the same immutable `MAX_REF_STALENESS`. When the two feeds have different update frequencies, the deployer is structurally forced to set `MAX_REF_STALENESS` to the larger heartbeat to keep the pool functional, rendering the staleness check for the faster-updating feed ineffective and allowing stale prices from that leg to reach pool swaps.

### Finding Description

`AnchoredPriceProvider._readLeg()` is the sole staleness gate for both feeds: [1](#0-0) 

```solidity
function _readLeg(bytes32 feedId)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

    if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
    ...
}
```

Both `baseFeedId` and `quoteFeedId` are dispatched through this same function: [2](#0-1) 

`MAX_REF_STALENESS` is a single immutable set once at construction: [3](#0-2) 

There is no mechanism to assign a per-feed staleness threshold. The deployer faces an inescapable trade-off:

| Choice | Effect |
|---|---|
| `MAX_REF_STALENESS` = min(heartbeat_base, heartbeat_quote) | Correct for both feeds, but the slower feed triggers false-stale reverts, making the pool unusable |
| `MAX_REF_STALENESS` = max(heartbeat_base, heartbeat_quote) | Pool stays live, but the faster feed's staleness check is rendered ineffective |

In practice, deployers choose the second option to keep the pool functional. This is the exact structural analog to the external report's `ORACLE_FRESHNESS_THRESHOLD` bug.

### Impact Explanation

When `MAX_REF_STALENESS` is set to accommodate the slower feed, the faster feed's price can be arbitrarily stale (up to `MAX_REF_STALENESS` seconds old) without triggering the staleness guard. The stale leg price flows through `_computeBidAsk` and is returned as the pool's bid/ask: [4](#0-3) 

A stale numerator or denominator in the synthetic ratio produces an incorrect mid price. The band clamp (`refBid`/`refAsk`) is derived from this same stale mid, so it does not correct the error. Traders can exploit the divergence between the stale on-chain price and the true market price to extract value from LPs — a direct bad-price execution impact.

### Likelihood Explanation

The synthetic mode is an explicit, documented feature of `AnchoredPriceProvider` (NatSpec: "Optional second feed for synthetic ratio quoting"). It is activated whenever `quoteFeedId != bytes32(0)` at construction. Feeds on the same oracle (e.g., `CompressedOracleV1`, Chainlink Data Streams) can have materially different update frequencies — a liquid crypto pair may update sub-second while an RWA feed updates hourly. The deployer has no on-chain mechanism to enforce matching heartbeats, and the contract provides no per-leg staleness parameter.

### Recommendation

Add a second immutable `MAX_QUOTE_REF_STALENESS` and apply it exclusively to the `quoteFeedId` leg:

```solidity
uint256 public immutable MAX_REF_STALENESS;       // for baseFeedId
uint256 public immutable MAX_QUOTE_REF_STALENESS; // for quoteFeedId

function _readLeg(bytes32 feedId, uint256 maxStaleness)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
    if (_isStale(refTime, block.timestamp, maxStaleness)) return (mid, spreadBps, refTime, false);
    ...
}
```

Call sites pass `MAX_REF_STALENESS` for `baseFeedId` and `MAX_QUOTE_REF_STALENESS` for `quoteFeedId`, matching each feed's actual heartbeat.

### Proof of Concept

1. Deploy `AnchoredPriceProvider` with:
   - `baseFeedId` = BTC/USD feed (heartbeat: 1 second on Chainlink Data Streams)
   - `quoteFeedId` = RWA/USD feed (heartbeat: 3600 seconds)
   - `MAX_REF_STALENESS` = 3600 seconds (set to accommodate the RWA feed)
2. The BTC/USD keeper stops pushing updates. After 1 second the BTC price is stale, but `MAX_REF_STALENESS` = 3600 s, so `_isStale` returns `false` for up to 3600 seconds.
3. During this window, `_getBidAndAskPrice()` returns a synthetic BTC/RWA ratio anchored to the stale BTC price.
4. A trader observes the on-chain stale BTC price is lower than the true market price and calls `swap()` to buy BTC at the artificially low synthetic ratio, extracting LP funds. [5](#0-4)
