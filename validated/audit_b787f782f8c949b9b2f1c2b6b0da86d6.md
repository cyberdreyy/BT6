### Title
Permissionless `createPool` accepts any arbitrary price provider without approved-factory validation — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.createPool` is fully permissionless and validates the price provider only by checking that `priceProvider.token0()` and `priceProvider.token1()` match the pool's token pair. There is no check that the price provider was deployed by an approved factory (e.g., `AnchoredProviderFactory`). An attacker can deploy a contract that satisfies the token-match check but returns arbitrary bid/ask prices, attach it to a newly created pool, and cause every swap in that pool to execute at a manipulated price.

### Finding Description

`_validatePriceProvider` is the sole gate called from both `createPool` and `proposePoolPriceProvider`:

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [1](#0-0) 

The two checks are: `priceProvider != address(0)` and token-pair match. Nothing else. The `AnchoredProviderFactory` exposes `isProvider(address)` precisely to answer "was this provider deployed by the curated factory with envelope-validated parameters?" — but `_validatePriceProvider` never calls it. [2](#0-1) 

A malicious contract that returns the correct token addresses from `token0()`/`token1()` and returns attacker-controlled values from `getBidAndAskPrice()` passes all factory-level checks and is stored as the pool's immutable or mutable price provider. The pool is then registered in the factory's canonical registry (`poolToIdx[pool] != 0`, so `isPool` returns `true`), making it indistinguishable from a legitimate pool to any on-chain or off-chain consumer that relies on the registry. [3](#0-2) 

The `abuse-protection-integration.md` design document explicitly states that "public-pool eligibility is then the machine-checkable predicate `recognizedFactory.isProvider(p)`", confirming the intended guard — but it is absent from the deployed code. [4](#0-3) 

### Impact Explanation

Every swap calls `getBidAndAskPrice()` on the price provider. If the provider returns an inverted, unbounded, or attacker-controlled bid/ask pair, `SwapMath` computes the wrong output amount. Traders receive incorrect amounts (too little or too much), and the pool's token balances diverge from LP claims — direct loss of trader principal or pool insolvency. This matches the "bad-price execution" and "swap conservation failure" impact categories. [5](#0-4) 

### Likelihood Explanation

`createPool` is fully permissionless — no role is required. The attacker needs only to deploy a contract implementing `IPriceProvider` with the correct `token0()`/`token1()` return values. The pool appears legitimate in the factory registry, making it indistinguishable from a valid pool to users who rely on `isPool` or the `PoolCreated` event without separately verifying `AnchoredProviderFactory.isProvider`. [6](#0-5) 

### Recommendation

Add an approved-factory check inside `_validatePriceProvider`. The `AnchoredProviderFactory` already exposes `isProvider(address)` for this purpose. The pool factory should maintain an admin-curated set of approved provider factories and require that the price provider is recognized by at least one of them before a pool is deployed or a provider update is executed.

### Proof of Concept

```solidity
contract MaliciousPriceProvider {
    address public immutable token0;
    address public immutable token1;
    uint128 public bid = 1;
    uint128 public ask = type(uint128).max;

    constructor(address _t0, address _t1) { token0 = _t0; token1 = _t1; }

    // Passes _validatePriceProvider — correct tokens returned
    function getBidAndAskPrice() external view returns (uint128, uint128) {
        return (bid, ask); // attacker-controlled prices
    }
    function setPrice(uint128 b, uint128 a) external { bid = b; ask = a; }
}

// Attacker deploys and attaches to a real pool
MaliciousPriceProvider mp = new MaliciousPriceProvider(token0, token1);
address pool = factory.createPool(PoolParameters({
    token0: token0, token1: token1,
    priceProvider: address(mp),
    priceProviderTimelock: type(uint256).max, // immutable — no recovery
    // ... other valid params
}));

assert(factory.isPool(pool)); // true — pool appears legitimate

// Attacker manipulates prices at will; every swap executes at the bad price
mp.setPrice(1, type(uint128).max);
```

The pool passes `isPool` and emits a `PoolCreated` event identical in structure to any legitimate pool. The only on-chain signal of the malicious provider is the provider address itself, which users have no protocol-enforced way to validate against `AnchoredProviderFactory.isProvider`.
