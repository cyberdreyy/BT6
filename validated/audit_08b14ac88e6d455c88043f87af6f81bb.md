### Title
Missing L2 Sequencer Uptime Check in `PriceProviderL2` and `ProtectedPriceProviderL2` Allows Stale-Price Swaps After Sequencer Restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are deployed on L2 chains (Arbitrum, Base, Optimism, etc.) via `PriceProviderFactoryL2`, but neither contract checks whether the L2 sequencer is live or has recently restarted. The only L2-specific guard is `FUTURE_TOLERANCE` (clock-skew tolerance), not a sequencer uptime feed check. During the grace window immediately after a sequencer restart — before authorized pushers can push fresh prices into `CompressedOracleV1` — the last pre-downtime price remains within `MAX_TIME_DELTA` and passes all staleness checks. A public trader can race the pushers and execute a swap against a price that is materially stale, extracting value from the pool.

---

### Finding Description

`PriceProviderL2._getBidAndAskPrice` reads the stored price from `CompressedOracleV1` and passes it to `_computeBidAsk`:

```solidity
// PriceProviderL2.sol L208-216
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spread, , uint256 refTime) =
        IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

    // 2. Staleness check
    if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
        return (0, type(uint128).max);
    }
    ...
}
```

The only L2-specific guard is `_isStale` with `FUTURE_TOLERANCE`:

```solidity
// PriceProviderL2.sol L135-150
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

`MAX_TIME_DELTA` is set by the provider creator at construction and can be up to 7 days. There is no call to a Chainlink sequencer uptime feed (`AggregatorV3Interface`) and no grace-period enforcement after sequencer restart.

`ProtectedPriceProviderL2` has the identical structure:

```solidity
// ProtectedPriceProviderL2.sol L203-209
function _computeBidAsk(uint256 price, uint256 spread, uint256 refTime)
    internal view returns (uint128, uint128)
{
    // 1. Staleness check
    if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
        return (0, type(uint128).max);
    }
    ...
}
```

When the L2 sequencer goes down, authorized pushers cannot push new prices to `CompressedOracleV1`. When the sequencer restarts, there is a window — potentially minutes — before pushers detect the restart, prepare a signed update, and have it included in a block. During this window, the pre-downtime price is still stored in the oracle with its original `refTime`. If the downtime was shorter than `MAX_TIME_DELTA`, `_isStale` returns `false` and the stale price is served to the pool as a valid bid/ask.

The registry confirms `PriceProviderFactoryL2` is deployed on Arbitrum, Base, Optimism, and other L2s, and the `ChainlinkVerifierL2` artifact in the registry (which does expose `sequencerUptimeFeed` and `GRACE_PERIOD`) has no corresponding `.sol` source file connected to these price providers.

---

### Impact Explanation

A public trader can submit a swap immediately after sequencer restart, before pushers update the oracle. If the market moved significantly during downtime (e.g., ETH dropped 10%), the pool's bid/ask is computed from the pre-downtime price. The trader buys the underpriced asset from the pool, and the pool receives less than the current market value of the input. This is a direct loss of LP principal — the pool's token balances no longer cover LP claims at fair value.

**Impact class:** Bad-price execution — stale bid/ask quote reaches a pool swap, causing swap conservation failure and LP asset loss.

---

### Likelihood Explanation

L2 sequencer outages are a documented, recurring event on Arbitrum and Optimism. The attacker's action (submitting a swap) is fully unprivileged. The attacker only needs to monitor for sequencer restarts and submit a swap before the pushers do. The race window is bounded by pusher latency (detection + signing + submission), which is realistically seconds to minutes. With `MAX_TIME_DELTA` values up to 7 days, any downtime shorter than that leaves the stale price valid.

---

### Recommendation

Add a sequencer uptime check to both `PriceProviderL2` and `ProtectedPriceProviderL2`, following the Chainlink pattern:

```solidity
// In constructor: store immutable sequencerUptimeFeed address and GRACE_PERIOD
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _checkSequencer() internal view {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0: sequencer is up; answer == 1: sequencer is down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` / `_computeBidAsk()`. If `sequencerUptimeFeed` is `address(0)` (L1 deployment), skip the check.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 3600` (1 hour).
2. Pusher pushes ETH price = $3000 at `T = 0`. `refTime = T`.
3. Arbitrum sequencer goes down at `T + 10 min`. No new pushes possible.
4. ETH market price drops to $2500 during downtime.
5. Sequencer restarts at `T + 30 min`. `block.timestamp = T + 30 min`.
6. Attacker immediately calls `pool.swap(...)` before pushers can push the new price.
7. `_isStale(T, T+1800, 3600, futureTol)` → `(T+1800 - T) = 1800 < 3600` → **not stale**.
8. Pool quotes bid/ask around $3000. Attacker buys ETH at $3000 when market is $2500.
