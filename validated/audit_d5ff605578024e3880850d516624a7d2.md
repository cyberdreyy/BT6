### Title
Pusher Delegation Signature Replay in `allowPushers` Defeats `revokePusher` Within the Deadline Window — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature but tracks **no per-signature nonce and no used-signature bitmap**. The only replay guard is the deadline. Within the deadline window a creator who holds a valid (unexpired) signature can call `allowPushers` repeatedly with the same bytes, re-establishing a delegation that the pusher already revoked via `revokePusher`. This makes `revokePusher` ineffective for the entire lifetime of the original deadline and enables a creator to hijack a pusher's data stream away from a second creator, starving that creator's feeds of updates and delivering stale prices to any pool that reads those feeds.

---

### Finding Description

`allowPushers` builds the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

After a successful call the mapping is updated:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because no nonce is consumed and no "used" flag is set, the creator can immediately call `allowPushers` again with the **identical** `(deadline, pusher, signature)` triple, writing `namespaceRemapping[pusher] = creator` back. The pusher's revocation is silently undone.

The code comment acknowledges the replay risk but treats the deadline as the complete mitigation:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The deadline limits the **window** of replay but does not prevent replay **within** that window. A deadline of 1 day (common in practice) leaves a 24-hour window during which the creator can re-establish the delegation an unlimited number of times.

**Extended namespace-hijack scenario:**

1. Pusher P signs consent for CreatorA with deadline D1 (e.g., `now + 1 day`).
2. CreatorA calls `allowPushers` → `namespaceRemapping[P] = CreatorA`.
3. P calls `revokePusher()` → `namespaceRemapping[P] = address(0)`.
4. P signs fresh consent for CreatorB with deadline D2.
5. CreatorB calls `allowPushers` → `namespaceRemapping[P] = CreatorB`.
6. CreatorA replays the original signature (D1 still valid) → `namespaceRemapping[P] = CreatorA`.
7. All subsequent fallback pushes from P land in CreatorA's namespace; CreatorB's feeds receive no updates.

The fallback push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

After step 6, every push from P writes into CreatorA's slots; CreatorB's slots are frozen at their last timestamp.

---

### Impact Explanation

Any pool whose `AnchoredPriceProvider` (or direct price provider) reads from CreatorB's `CompressedOracleV1` feeds will observe a frozen `refTime`. The `AnchoredPriceProvider` staleness guard:

```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;
    return (nowTs - refTime) > maxDelta;
}
``` [6](#0-5) 

will eventually trip, causing `getBidAndAskPrice` to return `(0, type(uint128).max)` and the pool to revert with `FeedStalled`:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
``` [7](#0-6) 

Every swap, add-liquidity, and remove-liquidity call that touches the price provider will revert, making the pool completely unusable. This is a **broken core pool functionality** impact — the pool cannot process any user action until the pusher re-delegates to CreatorB or a new pusher is onboarded.

---

### Likelihood Explanation

- The attacker is CreatorA — any address that previously obtained a valid pusher signature. This is a **semi-trusted but unprivileged** role; no admin key is required.
- The attack is a single public transaction (`allowPushers`) with calldata the attacker already possesses.
- The window is the full deadline duration (up to days), and the attack can be repeated every time the pusher re-revokes.
- The pusher has no on-chain mechanism to invalidate the old signature before the deadline expires.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` bitmap. The simplest fix is a per-pusher nonce that is incremented on every successful `allowPushers` call and included in the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

Alternatively, use a `mapping(bytes32 => bool) usedDigests` to mark each digest as consumed on first use, preventing any re-use regardless of deadline.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import "forge-std/Test.sol";
import {CompressedOracleV1} from
    "smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from
    "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract ReplayDelegationTest is Test {
    CompressedOracleV1 oracle;

    uint256 constant PUSHER_KEY  = 0xBEEF;
    uint256 constant CREATOR_A_KEY = 0xAAAA;

    address pusher;
    address creatorA;
    address creatorB = address(0xB0B);

    function setUp() public {
        oracle   = new CompressedOracleV1(address(this), 0);
        pusher   = vm.addr(PUSHER_KEY);
        creatorA = vm.addr(CREATOR_A_KEY);
        vm.warp(1_700_000_000);
    }

    function _sign(uint256 pk, uint256 deadline, address _pusher, address _creator)
        internal view returns (bytes memory)
    {
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, _pusher, _creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(pk, digest);
        return abi.encodePacked(r, s, v);
    }

    function testReplayAfterRevoke() public {
        uint256 deadline = block.timestamp + 1 days;
        bytes memory sig = _sign(PUSHER_KEY, deadline, pusher, creatorA);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // Step 1: CreatorA establishes delegation
        vm.prank(creatorA);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creatorA);

        // Step 2: Pusher revokes
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0));

        // Step 3: Pusher re-delegates to CreatorB
        // (CreatorB calls allowPushers with a new sig — omitted for brevity)
        // Simulate: set mapping directly to creatorB
        // In reality CreatorB would call allowPushers with a fresh sig from pusher.

        // Step 4: CreatorA REPLAYS the original signature — still within deadline
        vm.prank(creatorA);
        oracle.allowPushers(deadline, pushers, sigs); // same sig, no revert!

        // Pusher's revocation is undone; delegation is back to CreatorA
        assertEq(oracle.namespaceRemapping(pusher), creatorA,
            "REPLAY: revocation defeated within deadline window");
    }
}
```
