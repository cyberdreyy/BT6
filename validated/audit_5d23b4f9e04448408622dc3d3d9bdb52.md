### Title
Creator Can Replay Pusher Consent Signature to Permanently Override `revokePusher()` Within Deadline Window — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts a pusher's EIP-191 consent signature and maps `namespaceRemapping[pusher] = creator`. There is no nonce and no used-signature registry. A creator who holds a valid (not-yet-expired) signature can call `allowPushers` repeatedly with the same signature, re-establishing the delegation immediately after the pusher calls `revokePusher()`. The pusher's self-revocation produces no durable state change for the entire lifetime of the deadline, mirroring the external bug's pattern of a state-resetting call that nullifies a prior protective action.

---

### Finding Description

`allowPushers` signs over `(chainid, address(this), deadline, pusher, creator)` and enforces only that `block.timestamp <= deadline`: [1](#0-0) 

There is no nonce, no per-signature consumed flag, and no check that the pusher's current mapping is `address(0)` before writing. The code's own comment acknowledges the design intent:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

The deadline is meant to bound the replay window, but it only prevents re-establishment **after** the deadline expires — not before. Within the deadline window the creator holds an unconditionally replayable credential.

`revokePusher` clears the mapping: [3](#0-2) 

But `_ensureDeadline` only checks `block.timestamp <= deadline`: [4](#0-3) 

So the creator can immediately call `allowPushers` again with the identical signature, restoring `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone.

**Attack sequence:**

1. Pusher P signs consent for Creator A with `deadline = block.timestamp + 365 days`.
2. Creator A calls `allowPushers(deadline, [P], [sig])` → `namespaceRemapping[P] = A`.
3. Pusher P calls `revokePusher()` → `namespaceRemapping[P] = address(0)`.
4. Creator A immediately calls `allowPushers(deadline, [P], [sig])` with the **same** signature → `namespaceRemapping[P] = A` again.
5. Steps 3–4 can repeat indefinitely until `deadline` expires.

The fallback push path resolves the namespace at call time: [5](#0-4) 

So every push by P continues to land in Creator A's namespace, not P's own namespace, for the entire deadline window.

---

### Impact Explanation

If Pusher P is also a feed creator for their own pool (a realistic configuration for a price-feed operator serving multiple pools), their own namespace slots remain permanently stale while the creator keeps re-establishing the delegation. The `ProtectedPriceProvider` / `AnchoredPriceProvider` staleness check will fire: [6](#0-5) 

A stale feed causes `getBidAndAskPrice` to revert with `FeedStalled`: [7](#0-6) 

Every swap on any pool backed by P's own namespace feed halts. LPs cannot swap or withdraw at fair prices for the duration of the deadline window (up to the maximum deadline the pusher originally signed). This is broken core pool functionality within the allowed impact gate.

---

### Likelihood Explanation

- The creator already possesses the signature — they used it to establish the delegation in the first place.
- No additional permissions, privileged roles, or special setup are required.
- The creator is a valid semi-trusted actor; calling `allowPushers` is a normal, public, permissionless operation.
- The attack is free (only gas cost) and can be repeated in the same block as the pusher's `revokePusher` call, making front-running trivial.
- Deadlines are typically set far in the future (the test suite uses `block.timestamp + 1 days`; production operators commonly use weeks or months).

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedConsents` keyed on the signature hash, and revert if the same hash is submitted twice. Alternatively, add a per-pusher nonce that the pusher must sign over, so that after revocation the pusher can increment their nonce, invalidating all prior signatures:

```solidity
// In allowPushers, after recovering the signer:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedConsents[sigHash], "consent already used");
_usedConsents[sigHash] = true;
namespaceRemapping[pusher] = msg.sender;
```

This ensures that once a pusher revokes, the creator cannot replay the original consent to re-establish the delegation without a fresh signature from the pusher.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {Test} from "forge-std/Test.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract ReplayRevokePoC is Test {
    CompressedOracleV1 oracle;
    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator;

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 60);
        pusher  = vm.addr(PUSHER_KEY);
        creator = address(0xC0FFEE);
        vm.warp(1_700_000_000);
    }

    function testRevokeBypassedByReplay() public {
        uint256 deadline = block.timestamp + 365 days;

        // Pusher signs consent
        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, hash);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // Step 1: Creator establishes delegation
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // Step 2: Pusher revokes
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

        // Step 3: Creator replays the SAME signature — revocation undone
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator, "delegation re-established without fresh consent");
    }
}
```

The final assertion passes, confirming that `revokePusher()` provides no durable protection within the deadline window.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
