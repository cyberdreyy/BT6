### Title
`allowPushers` EIP-191 Signature Is Replayable Before Deadline, Enabling Creator to Silently Re-Establish Pusher Delegation After Revocation — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` uses a deadline to guard against signature replay, but the deadline only prevents replay *after* it expires. Before expiry, the identical pusher signature can be submitted an unlimited number of times. A creator can therefore call `allowPushers` with a previously-used signature to re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher called `revokePusher()`, silently redirecting all subsequent pushes back into the creator's namespace.

---

### Finding Description

The signed payload in `allowPushers` is:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no consumed-signature bitmap, and no per-pusher sequence counter. The only freshness gate is `_ensureDeadline(deadline)`, which passes for any call where `block.timestamp <= deadline`. [2](#0-1) 

The code's own NatSpec acknowledges the risk but treats the deadline as a complete fix:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline does not fix the problem — it only bounds the replay window. Within that window, the signature is unconditionally reusable.

`revokePusher` zeroes the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [4](#0-3) 

But `allowPushers` unconditionally overwrites it back:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [5](#0-4) 

There is no state that distinguishes "this signature was already consumed" from "this signature has never been used."

The `fallback` push path resolves the effective namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [6](#0-5) 

So every push after a silent re-delegation lands in the creator's namespace, not the pusher's own namespace.

---

### Impact Explanation

After the creator replays the signature:

1. The pusher's pushes are silently written to the creator's storage slots (`namespace = creator << 96 | slotId`), not the pusher's own slots.
2. The pusher's own feed IDs (`feedIdOf(pusher, slotIndex, positionIndex)`) receive no further updates — their stored `timestampMs` freezes.
3. Any `AnchoredPriceProvider` or pool configured against the pusher's own feed IDs will subsequently read a stale timestamp, which every staleness guard is supposed to reject as a bad price.
4. If the staleness guard threshold is wide enough (or if the pool's `maxTimeDrift` is generous), the frozen price passes through and reaches swap execution — a bad-price execution impact.

This is a **silent feed hijack**: the pusher believes they have revoked, the creator has re-established control, and the pusher's own feeds go dark without any on-chain event visible to the pusher after the re-delegation.

---

### Likelihood Explanation

- The pusher must have signed a delegation with a non-trivial deadline (common for operational convenience — e.g., 7–30 days).
- The creator must be adversarial and retain the original calldata.
- The pusher must call `revokePusher()` before the deadline expires.
- No privileged role is required; the creator is a normal caller of `allowPushers`.

This is a realistic scenario whenever a pusher terminates a relationship with a creator mid-deadline.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on each successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]   // ← add nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;   // ← consume
namespaceRemapping[pusher] = msg.sender;
```

This is the exact fix recommended for the VotingMachine analog: replace a permissive inequality (`< deadline`) with a strict one-time-use check (`nonce == expected`).

### Proof of Concept

```
Setup:
  creator  = address A  (CREATOR_KEY)
  pusher   = address B  (PUSHER_KEY)
  deadline = block.timestamp + 30 days

Step 1 — Pusher signs consent:
  hash = keccak256(abi.encode(chainid, oracle, deadline, pusher, creator))
  sig  = sign(PUSHER_KEY, toEthSignedMessageHash(hash))

Step 2 — Creator establishes delegation:
  oracle.allowPushers(deadline, [pusher], [sig])
  → namespaceRemapping[pusher] == creator  ✓

Step 3 — Pusher revokes:
  vm.prank(pusher); oracle.revokePusher()
  → namespaceRemapping[pusher] == address(0)  ✓

Step 4 — Creator replays the IDENTICAL sig (deadline not yet passed):
  oracle.allowPushers(deadline, [pusher], [sig])   // same calldata, no revert
  → namespaceRemapping[pusher] == creator  ✓  (revocation nullified)

Step 5 — Pusher pushes a price update (believing they are in their own namespace):
  vm.prank(pusher); address(oracle).call(slotWord)
  → price lands in creator's slot, pusher's own feedId receives no update
  → pusher's feedId timestampMs is frozen → stale price on next pool read
```
