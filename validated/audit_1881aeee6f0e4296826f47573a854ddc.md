### Title
L2 Price Providers Do Not Check Sequencer Uptime Before Consuming Oracle Prices — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly designed for L2 deployment (they carry a `FUTURE_TOLERANCE` immutable to handle L2 clock skew) but neither contract checks whether the L2 sequencer is active before consuming oracle prices. When the sequencer goes down and then restarts, oracle data that was last pushed before the outage can still be within `MAX_TIME_DELTA` yet be significantly stale relative to real market prices. Swaps executed in that window receive bad bid/ask quotes, causing direct LP loss.

---

### Finding Description

Both L2 price providers read oracle data through `IPricedOracle.price(feedId, pool)` and then apply a staleness check:

**`PriceProviderL2._getBidAndAskPrice`** (lines 208–248):
```solidity
(uint256 mid, uint256 spread, , uint256 refTime) =
    IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [1](#0-0) 

**`ProtectedPriceProviderL2._computeBidAsk`** (lines 203–238):
```solidity
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [2](#0-1) 

Neither contract has a `sequencerUptimeFeed` field, a `GRACE_PERIOD` constant, or any call to a Chainlink sequencer uptime aggregator. The only L2 accommodation is `FUTURE_TOLERANCE`, which tolerates oracle `refTime` slightly ahead of `block.timestamp` — it does not detect sequencer downtime. [3](#0-2) 

By contrast, the compiled ABI registry already contains a `ChainlinkVerifierL2` entry with both a `sequencerUptimeFeed` storage slot and a `GRACE_PERIOD` constant, confirming the protocol is aware of the requirement but has not wired it into the price providers: [4](#0-3) 

The oracle data in this system is push-based: pushers call `updateReport` / `pushUpdate` to store prices on-chain. When the L2 sequencer is down, no pusher transactions are executed. When the sequencer restarts, the last stored price (from before the outage) is still within `MAX_TIME_DELTA` for a window equal to `MAX_TIME_DELTA − outage_duration`. During that window, `_isStale` returns `false` and the stale pre-outage price is delivered to the pool as a valid bid/ask.

---

### Impact Explanation

A pool swap during the post-restart window receives a bid/ask derived from a price that may be minutes or hours old. If the real market moved significantly during the outage (e.g., a sharp ETH drop during an Arbitrum outage), a trader can:

- **Buy the base token at the stale ask** (below real market ask) — the pool receives less quote token than the current market value of the base token it delivers, directly reducing LP claims.
- **Sell the base token at the stale bid** (above real market bid) — the pool pays more quote token than the current market value of the base token it receives.

This is a direct loss of LP principal, matching the "bad-price execution" and "pool insolvency" impact categories.

---

### Likelihood Explanation

L2 sequencer outages are not hypothetical: Arbitrum has experienced multiple documented outages. The protocol explicitly targets Arbitrum, Base, and HyperEVM deployment. The trigger requires no privileged action and no malicious setup — any user can submit a swap immediately after the sequencer restarts. The window of exploitability is bounded by `MAX_TIME_DELTA` minus the outage duration, which can be tens of minutes for a typical `MAX_TIME_DELTA` of 1–2 hours.

---

### Recommendation

Add a sequencer uptime check to both `PriceProviderL2` and `ProtectedPriceProviderL2`, following the Chainlink pattern:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _checkSequencer() internal view {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0 means sequencer is up; 1 means down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` / `_computeBidAsk()` in both L2 providers. If the sequencer is down or within the grace period, return the stall sentinel `(0, type(uint128).max)` so `getBidAndAskPrice` reverts with `FeedStalled`.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 2 hours`.
2. Pusher stores a price at `T=0` (e.g., ETH = $3000).
3. Arbitrum sequencer goes down at `T=30min`; real ETH price drops to $2500.
4. Sequencer restarts at `T=90min`.
5. At `T=91min`, attacker calls `swap()` on the pool. `_isStale` checks: `block.timestamp − refTime = 91min < 120min (MAX_TIME_DELTA)` → not stale. The pool quotes ETH at $3000.
6. Attacker buys ETH at the stale $3000 ask while real market is $2500 — a ~17% discount extracted from LP reserves.
