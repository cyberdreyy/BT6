### Title
`PriceProviderL2._isStale()` silently extends the effective staleness window by `FUTURE_TOLERANCE`, allowing stale prices to reach pool swaps — (File: `smart-contracts-poc/contracts/PriceProviderL2.sol`)

### Summary
The L2 staleness check in `PriceProviderL2._isStale()` tolerates oracle timestamps up to `FUTURE_TOLERANCE` seconds ahead of `block.timestamp` to accommodate sequencer clock skew. However, the check fails to account for this tolerance when computing the effective age of the price later. A price pushed with `refTime = block.timestamp + FUTURE_TOLERANCE` is accepted immediately and then remains valid for `MAX_TIME_DELTA + FUTURE_TOLERANCE` total seconds — not `MAX_TIME_DELTA` as the parameter name and intent imply. This is the direct analog of the `syncRewards()` bug: just as that function added an extra cycle to `rewardsCycleEnd` when called late, `_isStale()` silently adds `FUTURE_TOLERANCE` to the effective expiry of every future-timestamped price.

### Finding Description

`PriceProviderL2._isStale()` is:

```solidity
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;

    if (refTime > nowTs) {
        // refTime in the future: tolerate only within futureTol
        return (refTime - nowTs) > futureTol;   // ← only checks the gap, not the age
    }

    // refTime in the past or equal: check age
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

When `refTime > nowTs` and `(refTime - nowTs) <= futureTol`, the function returns `false` (not stale) and the age branch is **never reached**. The price is accepted. Later, when `nowTs` catches up and surpasses `refTime`, the age branch fires — but `refTime` is already `futureTol` seconds ahead of the original push time, so the price does not expire until `nowTs - refTime > maxDelta`, i.e., at wall-clock time `pushTime + futureTol + maxDelta`.

Effective expiry timeline for a price pushed at time `T` with `refTime = T + futureTol`:

| Wall-clock time | `refTime > nowTs`? | Stale? |
|---|---|---|
| `T` | yes, gap = `futureTol` | no (gap ≤ futureTol) |
| `T + maxDelta` | no, age = `maxDelta - futureTol` | no (age ≤ maxDelta) |
| `T + maxDelta + futureTol` | no, age = `maxDelta` | no (age = maxDelta, boundary) |
| `T + maxDelta + futureTol + 1` | no, age = `maxDelta + 1` | **yes** |

The price lives for `maxDelta + futureTol` instead of `maxDelta`. `FUTURE_TOLERANCE` is bounded to 1 hour by the constructor:

```solidity
if (_futureTolerance > 1 hours) revert FutureToleranceOutOfBounds();
``` [2](#0-1) 

So the effective staleness window is silently extended by up to 1 hour beyond `MAX_TIME_DELTA`.

The L1 counterpart (`AnchoredPriceProvider._isStale()`) correctly rejects all future timestamps unconditionally (`if (refTime > nowTs) return true`), confirming the L2 version's behavior is a divergence, not a shared design choice. [3](#0-2) 

### Impact Explanation
`PriceProviderL2._getBidAndAskPrice()` feeds the bid/ask directly into pool swaps via `getBidAndAskPrice()`. A price that should have expired `FUTURE_TOLERANCE` seconds ago is still returned as valid, causing pool swaps to execute at a stale bid/ask. Depending on how much the market has moved during the extra window, LPs receive less than the fair price for their liquidity (swap conservation failure / LP principal loss). With `MAX_TIME_DELTA` as short as 1 second (constructor lower bound) and `FUTURE_TOLERANCE` up to 1 hour, the relative extension can be extreme. [4](#0-3) 

### Likelihood Explanation
The trigger is any authorized oracle pusher setting `timestampMs` to a value slightly ahead of the current block timestamp (within `FUTURE_TOLERANCE`). The oracle's compressed push path explicitly carries `timestampMs` as a pusher-supplied field. No unprivileged access is required — a valid, authorized pusher is sufficient (semi-trusted trigger). The condition is reachable on any L2 deployment where `FUTURE_TOLERANCE > 0`.

### Recommendation
Cap `refTime` at `nowTs` before the age check so that a future-timestamped price is treated as if it arrived at the current block, not at its claimed future time:

```solidity
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;

    if (refTime > nowTs) {
        if ((refTime - nowTs) > futureTol) return true;
        refTime = nowTs;   // ← cap: treat as "just arrived"
    }

    return (nowTs - refTime) > maxDelta;
}
```

This preserves the clock-skew tolerance (future prices within `futureTol` are accepted) while ensuring the effective staleness window is exactly `maxDelta`, not `maxDelta + futureTol`.

### Proof of Concept

Scenario: `MAX_TIME_DELTA = 1 hour`, `FUTURE_TOLERANCE = 1 hour`.

1. At `T = 1000`, an authorized pusher submits a price update with `timestampMs = (T + 3600) * 1000` (i.e., `refTime = T + 3600 = 4600`).
2. Oracle stores and returns `refTime = 4600`.
3. Pool calls `getBidAndAskPrice()` at `T = 1000`:
   - `_isStale(4600, 1000, 3600, 3600)` → `refTime > nowTs`, gap = 3600 ≤ 3600 → **not stale**. Price accepted.
4. At `T = 4600` (3600 seconds later, `MAX_TIME_DELTA` has elapsed since push):
   - `_isStale(4600, 4600, 3600, 3600)` → `refTime == nowTs`, age = 0 ≤ 3600 → **not stale**. Price still accepted despite being 1 hour old.
5. At `T = 8200` (7200 seconds after push, `MAX_TIME_DELTA + FUTURE_TOLERANCE`):
   - `_isStale(4600, 8200, 3600, 3600)` → age = 3600 = maxDelta → **not stale** (boundary).
6. Price finally expires at `T = 8201` — a full extra hour beyond the intended `MAX_TIME_DELTA`.

During steps 4–6, the market price may have moved significantly. Any swap in that window executes at the stale bid/ask, extracting value from LPs. [1](#0-0) [5](#0-4) [6](#0-5)
