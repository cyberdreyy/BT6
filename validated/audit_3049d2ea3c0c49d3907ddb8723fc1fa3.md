### Title
`AnchoredPriceProvider` Uses L1-Only Staleness Check on L2 Chains, Causing Intermittent `FeedStalled` Reverts — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` is described as "the one standard provider for public pools" and is deployed on all supported chains including Base and HyperEVM. However, its `_isStale` function unconditionally treats any oracle `refTime` ahead of `block.timestamp` as stale — the L1 behavior. The protocol already recognized this problem for `PriceProvider` and created `PriceProviderL2` with a `FUTURE_TOLERANCE` window, but no equivalent `AnchoredPriceProviderL2` exists. On L2 chains where sequencer clock skew can push oracle timestamps slightly ahead of `block.timestamp`, every such oracle update causes `_readLeg` to return `ok = false`, which propagates to `getBidAndAskPrice()` reverting with `FeedStalled`, breaking all swaps.

---

### Finding Description

`AnchoredPriceProvider._isStale` is explicitly annotated as L1-only:

```solidity
/// @dev Pure staleness check (L1). Any future refTime is stale.
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // ← hard rejection of any future timestamp
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

The protocol already solved this for `PriceProvider` by shipping `PriceProviderL2`, which adds a `FUTURE_TOLERANCE` immutable and a two-branch staleness check:

```solidity
/// @dev L2 sequencer timestamp can lag behind oracle publication time.
///      Allows refTime up to FUTURE_TOLERANCE seconds ahead of block.timestamp.
uint256 public immutable FUTURE_TOLERANCE;
...
if (refTime > nowTs) {
    return (refTime - nowTs) > futureTol;   // ← tolerates clock skew
}
``` [2](#0-1) [3](#0-2) 

The contracts directory contains `PriceProviderL2.sol`, `ProtectedPriceProviderL2.sol`, and `PriceProviderFactoryL2.sol`, but **no** `AnchoredPriceProviderL2.sol`:



`AnchoredPriceProvider._readLeg` calls `_isStale` and returns `ok = false` on any stale result:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [4](#0-3) 

`_getBidAndAskPrice` propagates this failure immediately:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
if (!ok) return (0, type(uint128).max);
``` [5](#0-4) 

And `getBidAndAskPrice()` converts the sentinel into a revert:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
``` [6](#0-5) 

---

### Impact Explanation

Every swap through a pool whose price provider is `AnchoredPriceProvider` calls `getBidAndAskPrice()`. On Base or HyperEVM, whenever the oracle's `refTime` is even 1 second ahead of `block.timestamp` (a normal occurrence under sequencer clock skew), the call reverts with `FeedStalled`. This breaks the core swap flow for all pools using the "one standard provider for public pools" on L2 — a direct loss of protocol usability matching the "broken core pool functionality" impact gate.

---

### Likelihood Explanation

The protocol explicitly targets Base and HyperEVM alongside Ethereum. The development team already identified and fixed the L2 clock-skew problem for `PriceProvider` (creating `PriceProviderL2`), confirming the phenomenon is real and expected. `AnchoredPriceProvider` is the primary provider for public pools, so any L2 deployment of it is affected. The trigger requires no attacker — normal oracle operation on L2 is sufficient.

---

### Recommendation

Create an `AnchoredPriceProviderL2` that mirrors the `FUTURE_TOLERANCE` pattern from `PriceProviderL2`:

1. Add `uint256 public immutable FUTURE_TOLERANCE` (bounded to ≤ 1 hour, matching `PriceProviderL2`).
2. Replace the three-argument `_isStale` with the four-argument version from `PriceProviderL2` that tolerates `refTime` up to `FUTURE_TOLERANCE` seconds ahead of `block.timestamp`.
3. Deploy `AnchoredProviderFactory` on L2 chains pointing to the L2 variant.

---

### Proof of Concept

1. Deploy `AnchoredPriceProvider` on Base with `MAX_REF_STALENESS = 60` seconds.
2. The oracle pushes an update with `refTime = block.timestamp + 2` (2-second sequencer clock skew — well within the tolerance `PriceProviderL2` would allow).
3. Call `getBidAndAskPrice()` from a pool swap.
4. `_readLeg` calls `_isStale(refTime, block.timestamp, 60)` → `refTime > nowTs` → returns `true`.
5. `_readLeg` returns `ok = false`.
6. `_getBidAndAskPrice` returns `(0, type(uint128).max)`.
7. `getBidAndAskPrice` reverts with `FeedStalled`.
8. The pool swap reverts. All swaps remain broken until the oracle's `refTime` falls behind `block.timestamp`, which may not happen consistently on L2.
