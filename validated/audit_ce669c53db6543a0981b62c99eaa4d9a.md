### Title
Permissionless `register` unconditionally clears admin blacklist, allowing any blacklisted pool to bypass oracle abuse protection — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase.register` is a permissionless, fee-gated function that unconditionally clears the blacklist entry for a pool as a side effect of registration. Because the only requirements are paying the registration fee (default: 1 wei) and being recognized by an approved factory, any blacklisted pool whose factory entry is still live can self-un-blacklist by calling `register` again. The admin's sole runtime abuse-protection lever is thereby nullified by an unprivileged path.

---

### Finding Description

`OracleBase.register` contains the following logic:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

    if (blacklisted[pool]) {
        blacklisted[pool] = false;          // ← unconditional side-effect
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
``` [1](#0-0) 

The blacklist is the primary runtime guard in the attributed read path:

```solidity
function price(bytes32 feedId, address pool)
    external
    feedExists(feedId)
    notBlacklisted          // checks msg.sender (the provider)
    returns (...)
{
    require(pool != address(0) && IPool(pool).inSwap() == msg.sender, InvalidInSwap());
    require(!blacklisted[pool], Blacklisted(pool));   // checks the pool
    require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
    ...
}
``` [2](#0-1) 

The admin sets the blacklist exclusively through the privileged `setBlacklist`:

```solidity
function setBlacklist(address account, bool value) external onlyRole(ADMIN_ROLE) { ... }
``` [3](#0-2) 

But `register` is `external payable` with no role check. The registration fee defaults to 1 wei:

```solidity
registrationFee = 1 wei; // very cheap default; ADMIN tunes via setRegistrationFee
``` [4](#0-3) 

A pool that was blacklisted for oracle abuse was, by definition, previously a valid pool recognized by an approved factory. Those two conditions (`approvedFactories.contains(factory)` and `IPoolFactory(factory).isPool(pool)`) remain true unless the admin separately removes the pool from the factory — a distinct, non-atomic action. The blacklisted pool operator therefore retains the ability to call `register(feedId, pool, factory)` with 1 wei and atomically clear its own blacklist entry, restoring full oracle read access.

The analog to the external bug class is exact: a state update that should be privileged (blacklist clearing, analogous to credit-capacity recalculation) is reachable through an unprivileged path (permissionless `register`, analogous to the public `deposit` call), causing the downstream invariant (oracle abuse protection, analogous to `NoDelegatedCredit` guard) to be silently bypassed.

---

### Impact Explanation

Once un-blacklisted, the pool passes both the `notBlacklisted` modifier and the explicit `!blacklisted[pool]` check inside `price`, and it can again call `getBidAndAskPrice()` through any attached `AnchoredPriceProvider` or `ProtectedPriceProviderL2` during live swaps. If the pool was blacklisted because it was executing abusive or manipulated swaps, restoring its oracle access allows bad-price execution to resume — traders swapping through that pool receive prices the oracle admin had already determined were unsafe. This is a direct loss-of-funds path for pool users and a broken core invariant (oracle abuse protection is the only runtime circuit breaker for pool reads).

---

### Likelihood Explanation

- The registration fee is 1 wei by default; even if raised by the admin, the blacklisted pool operator is economically motivated to pay it.
- No privileged role is required — any EOA can call `register`.
- The factory and pool conditions are trivially satisfied for any pool that was legitimate before blacklisting.
- The admin and the pool operator are in a permanent cat-and-mouse loop: every `setBlacklist(pool, true)` can be immediately countered with `register(feedId, pool, factory)`.

---

### Recommendation

Decouple blacklist clearing from registration. Remove the unconditional blacklist side-effect from `register`:

```solidity
// Remove this block from register():
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
```

If clearing the blacklist on re-registration is intentional, gate it behind `ADMIN_ROLE` or require an explicit admin pre-approval step before `register` can clear it. Alternatively, make the blacklist sticky: once set, it can only be cleared by `ADMIN_ROLE` via `setBlacklist`, never as a side effect of a permissionless call.

---

### Proof of Concept

```solidity
// 1. Admin blacklists the pool for oracle abuse.
oracle.setBlacklist(address(pool), true);
assertTrue(oracle.blacklisted(address(pool)));

// 2. Pool operator (or anyone) pays 1 wei to re-register — atomically clears the blacklist.
oracle.register{value: 1}(feedId, address(pool), address(approvedFactory));

// 3. Blacklist is gone.
assertFalse(oracle.blacklisted(address(pool)));

// 4. Pool can now read oracle prices again during swaps.
pool.swap(...); // internally calls provider.getBidAndAskPrice() → oracle.price(feedId, pool) — succeeds
```
