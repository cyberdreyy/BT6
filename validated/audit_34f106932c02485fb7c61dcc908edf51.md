### Title
Pusher delegation consent signature is replayable within its deadline window, allowing a creator to re-establish a revoked delegation and redirect an automated pusher's price writes back into the creator's namespace — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original signature — with the same `deadline` — to re-establish `namespaceRemapping[pusher] = creator`. The code comment at lines 186–191 acknowledges that the deadline is the only guard against this, but the deadline only bounds the replay *window*; it does not prevent replay *within* that window.

---

### Finding Description

`allowPushers` signs over `(chainid, oracle, deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = msg.sender` on success: [1](#0-0) 

`revokePusher` clears the mapping to `address(0)`: [2](#0-1) 

There is no nonce, no used-signature bitmap, and no state change that invalidates the signature after first use. `_ensureDeadline` only checks `block.timestamp <= deadline`: [3](#0-2) 

The code comment itself acknowledges the gap — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — but treats the deadline as the fix. The deadline is not a fix; it is a time-bounded replay window: [4](#0-3) 

The `fallback` push path resolves the namespace at call time from `namespaceRemapping[msg.sender]`, so re-establishing the mapping immediately redirects all subsequent pushes from the pusher's bot back into the creator's namespace: [5](#0-4) 

---

### Impact Explanation

If the pusher is an automated price-feed bot that continues to emit slot words after calling `revokePusher()` (e.g., because the operator has not yet stopped the process, or because the pusher is a contract that cannot be halted), the creator can replay the old signature to redirect those pushes back into the creator's namespace. The creator's oracle feeds — identified by `feedIdOf(creator, slotIndex, positionIndex)` — are then overwritten with the pusher's data. Any pool consuming the creator's oracle via a `PriceProvider` will execute swaps at those prices. If the pusher's bot is emitting stale or otherwise undesirable data at the time of revocation, the creator can lock in that data stream, constituting bad-price execution reaching live pool swaps.

---

### Likelihood Explanation

The creator always retains the original signature bytes (they submitted them in the first `allowPushers` call). The only requirement is that `block.timestamp <= deadline`. Deadlines are typically set days or weeks in the future. The pusher has no on-chain mechanism to invalidate the signature before the deadline expires; their only recourse is to stop the bot off-chain, which may not be instantaneous.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) public usedConsentSignatures` and mark each signature hash as spent on first use in `allowPushers`. Alternatively, include a per-pusher nonce in the signed message and increment it on every successful `allowPushers` call; `revokePusher` should also increment the nonce so that all prior signatures are invalidated atomically with the revocation.

---

### Proof of Concept

```
// Step 1 — pusher signs consent (deadline = T + 7 days)
sig = sign(keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)))

// Step 2 — creator establishes delegation
creator calls allowPushers(deadline, [pusher], [sig])
→ namespaceRemapping[pusher] = creator  ✓

// Step 3 — pusher revokes
pusher calls revokePusher()
→ namespaceRemapping[pusher] = address(0)  ✓

// Step 4 — creator replays the SAME sig (deadline not yet expired)
creator calls allowPushers(deadline, [pusher], [sig])
→ _ensureDeadline passes (block.timestamp ≤ deadline)
→ ECDSA.recover returns pusher  ✓
→ namespaceRemapping[pusher] = creator  ← delegation silently re-established

// Step 5 — pusher's bot continues to push slot words via fallback
//          namespaceRemapping[pusher] == creator → prices land in creator's namespace
//          → feedIdOf(creator, slot, pos) updated with pusher's data
//          → pool swap reads creator's oracle → bad-price execution
```

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
```

```text
    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
```
