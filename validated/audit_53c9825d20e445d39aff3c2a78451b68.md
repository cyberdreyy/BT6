### Title
Creator Can Replay Pusher Consent Signature to Silently Re-Establish Delegation After `revokePusher()` — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts any valid EIP-191 pusher-consent signature whose deadline has not yet expired. There is no per-signature nonce or used-signature registry. A creator who holds a pusher's original consent bytes can call `allowPushers` again at any time before the deadline, re-writing `namespaceRemapping[pusher] = creator` immediately after the pusher cleared it with `revokePusher()`. The pusher's self-revocation is therefore not final: the creator can silently restore the delegation within the same deadline window, causing the pusher's subsequent fallback pushes — which the pusher believes are landing in their own namespace — to overwrite the creator's live feed slots instead.

---

### Finding Description

`allowPushers` is the EOA-pusher delegation entry point:

```solidity
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← unconditional write
    ...
}
``` [1](#0-0) 

The only replay guard is `_ensureDeadline`, which checks `block.timestamp <= deadline`: [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

Because `allowPushers` carries no nonce and no used-signature set, the creator can immediately call `allowPushers(sameDeadline, [pusher], [originalSig])` again. The deadline check passes (deadline is still in the future), the ECDSA recovery succeeds (the signature is cryptographically valid), and `namespaceRemapping[pusher]` is restored to `creator`. The pusher's revocation is silently undone.

The code comment on `allowPushers` acknowledges the risk of undated signatures but incorrectly concludes that the deadline alone solves it:

> "an undated signature could re-establish a delegation AFTER the pusher revoked it" [4](#0-3) 

The deadline prevents replay only after it expires; it does nothing to prevent replay within the validity window.

After the creator re-establishes the delegation, the pusher's next fallback push resolves the namespace as:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So the pusher's data — which the pusher believes is going to their own namespace — overwrites the creator's live feed slot. The `getOracleData` read path then returns this data with the pusher's timestamp, which the price provider accepts as fresh: [6](#0-5) 

---

### Impact Explanation

A pool whose price provider reads from the creator's compressed feed will receive whatever the revoked pusher pushes next. If the pusher revoked because they are compromised, want to push a different asset, or simply want to stop, their subsequent pushes carry a fresh timestamp and pass the staleness check in the price provider. The pool's `swap` call then executes against a bid/ask derived from the pusher's unintended or manipulated data, causing direct loss to traders or LPs.

The `_isStale` check in `ProtectedPriceProvider._computeBidAsk` only rejects data older than `MAX_TIME_DELTA`; it does not detect that the data came from a pusher whose revocation was silently overridden: [7](#0-6) 

---

### Likelihood Explanation

The creator receives the pusher's consent signature during the original `allowPushers` call and can store it off-chain. Replaying it costs only a single transaction. The window is as long as the original deadline — which can be days or weeks. Any creator who wants to retain a pusher's data feed after the pusher revokes can exploit this with no additional permissions or setup.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` mapping. After `revokePusher()` clears `namespaceRemapping[pusher]`, any previously issued signature for that `(pusher, creator, deadline)` tuple must be invalidated. One concrete approach: include a per-pusher nonce in the signed message and increment it on every successful `allowPushers` or `revokePusher` call, so old signatures become invalid immediately after revocation.

---

### Proof of Concept

```
1. block.timestamp = T0
2. Pusher signs: keccak256(abi.encode(chainid, oracle, T0+7days, pusher, creator))
3. Creator calls allowPushers(T0+7days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
4. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)
5. Creator immediately calls allowPushers(T0+7days, [pusher], [sig])  ← same sig, same deadline
   → _ensureDeadline passes (T0 < T0+7days)
   → ECDSA.recover returns pusher  ← signature still valid
   → namespaceRemapping[pusher] = creator  ← delegation silently restored
6. Pusher pushes fallback data (thinking it goes to their own namespace)
   → namespace resolves to creator's namespace
   → creator's feed slot is overwritten with pusher's unintended data
7. Pool swap reads creator's feed → bad price → trader/LP loss
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
