### Title
`allowPushers` Delegation Signature Replayable Within Deadline Window After Pusher Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` does not invalidate a pusher's consent signature upon revocation. A creator can replay the original EIP-191 signature within the deadline window to silently re-establish a delegation the pusher explicitly revoked, forcing the pusher's price updates to land in the creator's namespace without current consent and starving any pool that relies on the pusher's own namespace of fresh prices.

---

### Finding Description

`allowPushers` binds a pusher to a creator's namespace by verifying an EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)`: [1](#0-0) 

The code's own NatSpec comment at lines 186–191 acknowledges the replay risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The comment claims the deadline is the mitigation. However, the deadline only prevents replay **after** it expires. Within the deadline window, the same `(deadline, sig)` tuple can be submitted to `allowPushers` an unlimited number of times. There is no nonce, no used-signature set, and no revocation counter included in the signed message.

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But the original signature remains cryptographically valid until `deadline`. The creator can immediately call `allowPushers` again with the identical arguments, writing `namespaceRemapping[pusher] = creator` again. The pusher's revocation is silently undone.

The `feedIdOf` function encodes the **creator** address into the feed id: [4](#0-3) 

The `fallback` push path resolves the effective namespace through `namespaceRemapping`: [5](#0-4) 

So when the delegation is re-established without the pusher's current consent, every subsequent push by the pusher lands in the creator's namespace, not the pusher's own. Any pool registered against the pusher's own feed id (`feedIdOf(pusher, slot, pos)`) receives no new updates and its price becomes stale.

---

### Impact Explanation

A pusher operating as an automated price-pushing bot for multiple namespaces cannot effectively revoke their delegation to a creator within the deadline window. The creator replays the old signature, the bot's pushes continue to land in the creator's namespace, and the pusher's own namespace (used by a separate pool) stops receiving updates. Stale prices from the pusher's own namespace then reach pool swaps via `AnchoredPriceProvider` / `price(feedId, pool)`, constituting bad-price execution. The pool's `maxRefStaleness` guard is the only downstream check, and it fires only after the staleness threshold is crossed — until then, the last stale price is served as live.

---

### Likelihood Explanation

Requires a creator to deliberately call `allowPushers` a second time with the same arguments after the pusher revokes. The deadline window can be arbitrarily long (e.g., 7 days is a natural choice for off-chain tooling). The creator is a semi-trusted, unprivileged actor — not the oracle admin — so this is a reachable, non-privileged trigger. The pusher can revoke again, but the creator can replay again, making the revocation effectively unenforceable for the duration of the deadline.

---

### Recommendation

Track consumed signatures. Add a `mapping(bytes32 => bool) private _usedDelegationSigs` and mark each signature hash as used on first acceptance. Alternatively, include a per-pusher revocation nonce in the signed message:

```solidity
mapping(address => uint256) public pusherRevocationNonce;

// In revokePusher / removePushers:
pusherRevocationNonce[pusher]++;

// In the signed digest:
keccak256(abi.encode(
    block.chainid, address(this), deadline,
    pusher, msg.sender,
    pusherRevocationNonce[pusher]   // ← invalidates all prior sigs on revoke
))
```

This mirrors the fix recommended in the external report (reset the accumulator on re-add): here, incrementing the nonce on revocation makes every prior signature invalid, so re-delegation requires fresh pusher consent.

---

### Proof of Concept

```
1. Pusher signs:
   sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+7days, pusher, creator)))

2. Creator calls allowPushers(T+7days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓

4. Creator calls allowPushers(T+7days, [pusher], [sig])  ← SAME args, deadline still valid
   → namespaceRemapping[pusher] = creator  ← revocation silently undone

5. Pusher's bot pushes slot word → lands in creator's namespace (not pusher's own)
   → feedIdOf(pusher, slot, pos) receives no new data → timestampMs stale

6. Pool registered on feedIdOf(pusher, slot, pos) calls price() → returns last stale price
   → swap executes at a stale bid/ask
```

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
```
