### Title
`revokePusher()` Revocation Is Ineffective Within the Deadline Window Due to Signature Replay in `allowPushers` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never invalidates it after use. A creator who holds a still-valid signature can replay it to re-establish a delegation immediately after the pusher calls `revokePusher()`. The deadline only caps the replay window; it does not prevent replay within that window. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the signed deadline, and any pushes the pusher makes after revoking — believing they are writing to their own namespace — continue to land in the creator's namespace and can feed wrong prices to a live pool.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no used-signature bitmap, and no check that the pusher is not already delegated. The function simply overwrites `namespaceRemapping[pusher]` with `msg.sender` on every successful call. [2](#0-1) 

`revokePusher()` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But the creator still holds the original signature. As long as `block.timestamp <= deadline`, the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments and the mapping is restored to `msg.sender` (the creator). The code comment itself acknowledges the risk — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — but the deadline only bounds the replay window; it does not prevent replay within it. [4](#0-3) 

The `fallback` push path resolves the effective namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after revoking — while the creator has silently re-established the mapping — writes into the creator's namespace, not the pusher's own.

---

### Impact Explanation

The creator's namespace is the source of truth for `feedIdOf(creator, slotIndex, positionIndex)`, which is the `feedId` bound to a live pool's `AnchoredPriceProvider` or `PriceProvider`. If the pusher, believing they are now writing to their own namespace, begins pushing prices for a different asset pair, those prices overwrite the creator's feed. The pool's price provider reads the creator's feed via `offchainOracle.price(feedId, pool)` and computes bid/ask from the corrupted mid price. [6](#0-5) 

A pool that was quoting ETH/USD would receive BTC/USD prices (or any other pair the pusher switched to), producing an inverted or wildly wrong bid/ask. Swappers executing against that quote receive more output than the oracle permits (swap conservation failure) or the pool becomes insolvent relative to LP claims.

---

### Likelihood Explanation

Medium. The creator must actively replay the signature in the same block or shortly after the pusher's `revokePusher()` transaction. The pusher must then continue pushing (e.g., they switched to a new feed and are unaware the delegation was silently restored). The pusher could detect the re-establishment by monitoring `PusherAuthorized` events, but there is no on-chain enforcement that prevents the creator from doing this repeatedly for the full deadline window (which in tests is set to 1 day, and in production could be longer). [7](#0-6) 

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` bitmap. Increment the nonce in the signed digest so that each consent signature is single-use:

```solidity
// In the signed digest, include a per-pusher nonce:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce[pusher]++))
```

Alternatively, record the revocation timestamp and reject any signature whose deadline predates the revocation, making `revokePusher()` permanently effective against all previously issued signatures.

---

### Proof of Concept

```
1. Pusher P signs consent for creator C:
   digest = keccak256(abi.encode(chainid, oracle, deadline=T+1day, P, C))
   sig = sign(P_key, digest)

2. Creator C calls allowPushers(T+1day, [P], [sig])
   → namespaceRemapping[P] = C  ✓

3. Pusher P calls revokePusher()
   → namespaceRemapping[P] = address(0)  ✓ (pusher believes delegation is gone)

4. Creator C immediately calls allowPushers(T+1day, [P], [sig])  ← SAME sig, still valid
   → namespaceRemapping[P] = C  ← delegation silently restored

5. Pusher P, unaware, begins pushing BTC/USD prices via fallback():
   fallback resolves creator = namespaceRemapping[P] = C
   → prices land in feedIdOf(C, slotIndex, positionIndex)

6. Creator C's ETH/USD pool reads feedIdOf(C, slotIndex, positionIndex)
   → pool receives BTC/USD mid price
   → bid/ask computed from wrong price → bad-price execution
   → swappers extract excess output or pool becomes insolvent
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
