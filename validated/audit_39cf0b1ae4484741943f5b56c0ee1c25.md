### Title
Pusher Consent Signature Not Consumed on Use — Creator Can Replay Expired Revocation to Re-Delegate Compromised Key - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()` to stop a compromised key from writing prices, the creator can immediately replay the original signature (before its deadline) to re-establish the delegation. The attacker holding the compromised key then regains write authority over the creator's namespace and can push arbitrary prices that flow through `price()` → `getBidAndAskPrice()` into live pool swaps.

### Finding Description

`allowPushers` signs a message over `(block.chainid, address(this), deadline, pusher, msg.sender)` and writes `namespaceRemapping[pusher] = msg.sender` on success. [1](#0-0) 

There is no nonce, no used-signature bitmap, and no on-chain record that the signature was already accepted. The only expiry gate is the deadline timestamp. [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`, but it cannot invalidate the off-chain signature the pusher already issued. [3](#0-2) 

The code comment itself acknowledges the gap: *"the signed consent has no data timestamp, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The same acknowledgement appears in the slot-structure documentation. [5](#0-4) 

Once re-delegated, the `fallback` push path resolves the namespace from `namespaceRemapping[msg.sender]` and writes directly into the creator's storage slot with no further authorization check beyond timestamp monotonicity. [6](#0-5) 

The written price is then returned verbatim by `getOracleData` / `price()` and consumed by `AnchoredPriceProvider._getBidAndAskPrice()` (or `PriceProviderL2._getBidAndAskPrice()`), which applies staleness and spread checks but cannot detect that the price was written by a re-authorized compromised key. [7](#0-6) 

### Impact Explanation

A compromised pusher key that was revoked can be silently re-authorized by the creator replaying the original consent signature. The attacker then pushes an arbitrary price into the creator's namespace. That price propagates through `price()` → `getBidAndAskPrice()` → pool swap, producing bad-price execution: the pool executes swaps at an attacker-controlled bid/ask, causing direct loss of LP principal or enabling the attacker to extract value from the pool at the expense of LPs and traders.

### Likelihood Explanation

The scenario requires: (1) a pusher key is compromised, (2) the pusher calls `revokePusher()`, and (3) the creator — unaware of the compromise — calls `allowPushers` again with the same signature before the deadline. Step 3 is plausible because the creator may have automated re-delegation logic or may simply not monitor revocation events. The deadline window (set by the creator at delegation time) determines the exposure window. Likelihood is **Medium**.

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` bitmap. The simplest fix is to include a per-pusher nonce in the signed message and increment it on each successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++   // <-- nonce
    ))
);
```

After `revokePusher()` the nonce is already incremented, so any previously issued signature is permanently invalid regardless of its deadline.

### Proof of Concept

```
1. Creator calls allowPushers(deadline=T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

2. Pusher's key is compromised; pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓

3. Creator (unaware) calls allowPushers(deadline=T+1day, [pusher], [SAME sig])
   → signature still valid (deadline not expired, no nonce)
   → namespaceRemapping[pusher] = creator  ← re-established

4. Attacker (holding compromised key) calls oracle.fallback() with
   crafted slot word: price = 1 (near-zero), timestamp = now
   → fallback resolves namespaceRemapping[attacker_key] = creator
   → writes price=1 into creator's slot

5. Pool calls getBidAndAskPrice()
   → AnchoredPriceProvider reads price=1 from oracle
   → staleness check passes (timestamp = now)
   → spread check passes (attacker sets valid spread indexes)
   → pool executes swap at attacker-controlled price
   → LP principal drained
``` [2](#0-1) [3](#0-2) [6](#0-5)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

        address creator = namespaceRemapping[msg.sender];
