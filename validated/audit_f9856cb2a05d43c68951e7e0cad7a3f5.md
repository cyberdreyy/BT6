### Title
Permissionless `register()` unconditionally clears pool blacklist, bypassing oracle abuse protection — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase.register()` is a permissionless, paid function that simultaneously whitelists a pool for a feed and **unconditionally clears the pool's blacklist entry**. Because the registration fee defaults to 1 wei and any caller may invoke it, any blacklisted pool (or a third party acting on its behalf) can erase its own blacklist status for the cost of 1 wei, permanently nullifying the admin's only runtime abuse-protection lever over pool-level oracle reads.

---

### Finding Description

`OracleBase` enforces two independent blacklist checks before returning a live oracle price: [1](#0-0) 

```
notBlacklisted          → msg.sender (the price provider) must not be blacklisted
!blacklisted[pool]      → the pool forwarded by the provider must not be blacklisted
```

The admin sets blacklist entries via `setBlacklist`, which is gated on `ADMIN_ROLE`. [2](#0-1) 

However, `register()` is **permissionless** and **unconditionally clears** the pool's blacklist entry as a side-effect of registration: [3](#0-2) 

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);          // 1 wei default
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;                       // ← unconditional clear
        emit BlacklistUpdated(pool, false);
    }
    registeredPool[feedId][pool] = true;
    ...
}
```

The registration fee is initialized to **1 wei** in the constructor: [4](#0-3) 

```solidity
registrationFee = 1 wei; // very cheap default; ADMIN tunes via setRegistrationFee
```

The only preconditions are that `factory` is an approved factory and `pool` is recognized by that factory as a legitimate pool — both of which are true for any pool that was already deployed and operating (including the blacklisted one). There is no check that the pool is not already blacklisted before clearing it, no timelock, and no ADMIN_ROLE requirement.

**Analog to Vether4 M-12**: In Vether4, paying 128 VETH to `addExcluded()` allowed a one-time setup of a middleman contract that all users could route through to avoid transfer fees. Here, paying 1 wei to `register()` allows a one-time call that clears the blacklist for any legitimate pool, bypassing the admin's runtime abuse-protection control. Both involve a permissionless paid path that permanently undermines a security invariant.

---

### Impact Explanation

The blacklist is the only mechanism by which the oracle admin can halt a specific pool's access to live price data at runtime — for example, to stop a pool that is being exploited, that is consuming prices without authorization, or that is behaving abusively. Once cleared via `register()`, the pool regains full access to `price(feedId, pool)` and can execute swaps using live oracle quotes. The admin must re-blacklist the pool, but the attacker can immediately clear it again for 1 wei, making the blacklist permanently ineffective against a motivated actor.

**Impact category**: Admin-boundary break — an unprivileged path bypasses an oracle admin role check (`setBlacklist` / `ADMIN_ROLE`), allowing a blacklisted pool to consume live oracle prices and execute swaps that the admin intended to halt.

---

### Likelihood Explanation

- **Unprivileged**: any EOA or contract can call `register()`.
- **Cost**: 1 wei (the default `registrationFee`; even if raised, the attacker pays once per admin blacklist action).
- **Precondition**: the pool must be a legitimate pool in an approved factory — satisfied by definition for any pool that was previously operating.
- **Repeatability**: the attacker can re-clear the blacklist immediately after every admin re-blacklist, making the control permanently ineffective.

Likelihood: **High**.

---

### Recommendation

Separate blacklist clearing from registration. Remove the unconditional `blacklisted[pool] = false` side-effect from `register()`. If clearing the blacklist on re-registration is desired, gate it on `ADMIN_ROLE` or require an explicit separate admin call:

```solidity
// Remove from register():
// if (blacklisted[pool]) {
//     blacklisted[pool] = false;
//     emit BlacklistUpdated(pool, false);
// }
```

Blacklist status should only be modifiable through `setBlacklist()`, which is already correctly gated on `ADMIN_ROLE`. [2](#0-1) 

---

### Proof of Concept

```
1. Admin deploys oracle, pool P is registered and operating normally.
2. Admin detects abuse: oracle.setBlacklist(P, true)
   → blacklisted[P] == true
   → P.swap() → provider.getBidAndAskPrice() → oracle.price(feedId, P) reverts Blacklisted(P)

3. Attacker (or P itself, or any third party) calls:
   oracle.register{value: 1 wei}(anyFeedId, P, approvedFactory)
   Preconditions:
     - approvedFactory is already approved (public state)
     - approvedFactory.isPool(P) == true (P is a legitimate pool)
   Effect:
     - blacklisted[P] = false  ← blacklist cleared
     - registeredPool[anyFeedId][P] = true

4. P.swap() → provider.getBidAndAskPrice() → oracle.price(feedId, P) succeeds
   → live oracle price returned, swap executes at oracle-anchored bid/ask

5. Admin re-blacklists: oracle.setBlacklist(P, true)
   Attacker immediately repeats step 3 for 1 wei → blacklist cleared again.
   The control is permanently ineffective.
``` [3](#0-2) [1](#0-0) [4](#0-3)

    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
    }
