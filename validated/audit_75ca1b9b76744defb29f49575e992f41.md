### Title
Pusher Consent Signature Never Invalidated, Allowing Creator to Replay Delegation After Pusher Self-Revokes — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature and checks that `block.timestamp <= deadline`, but **never marks the signature as consumed**. After a pusher calls `revokePusher()` to withdraw consent, the creator can immediately replay the identical signature (while the deadline is still in the future) to silently re-establish the delegation. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the deadline window — a direct analog to the spend-limit bypass in the reference report, where a cap is checked but never decremented.

---

### Finding Description

`allowPushers` in `CompressedOracle.sol` accepts a `deadline`, a list of pusher addresses, and their EIP-191 signatures. The signed message binds `(chainid, address(this), deadline, pusher, creator)`. [1](#0-0) 

The only replay guard is the deadline check in `_ensureDeadline`: [2](#0-1) 

This check only prevents replay **after** the deadline expires. There is no nonce, no used-signature bitmap, and no per-pusher revocation epoch. The code comment explicitly acknowledges the problem but incorrectly concludes the deadline solves it: [3](#0-2) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [4](#0-3) 

Because the signature is never consumed, the creator can call `allowPushers` again with the same `(deadline, pusher, signature)` tuple immediately after the pusher's revocation, restoring `namespaceRemapping[pusher] = creator`. This cycle can repeat indefinitely until the deadline expires.

---

### Impact Explanation

Every push from the pusher's EOA is routed through `namespaceRemapping[msg.sender]` in the `fallback`: [5](#0-4) 

While the delegation is active, the pusher's slot writes land in the **creator's namespace**, not the pusher's own. Pools and price providers reading `feedIdOf(creator, slotIndex, positionIndex)` consume those prices. If the pusher revokes because they want to stop supplying data (e.g., their key is compromised, or they want to redirect pushes to their own namespace for their own pools), the creator can override that decision and keep the pusher's data flowing into the creator's namespace — and into any pool reading from it — for the full remaining deadline window.

The concrete oracle-price impact: a pusher who revokes to stop bad or stale prices from reaching the creator's namespace cannot do so; the creator replays the signature and the pusher's next push again overwrites the creator's slot with whatever price the pusher sends.

---

### Likelihood Explanation

- The creator is a semi-trusted party who already controls their namespace; the motivation to replay exists whenever a pusher revokes against the creator's interest.
- The pusher must have signed a consent with a future deadline (standard practice — the test suite uses `block.timestamp + 1 days`).
- No special privileges or setup beyond holding the original signature are required; the creator simply calls `allowPushers` again with the same calldata.
- The window is bounded by the deadline, but deadlines of hours or days are typical.

---

### Recommendation

Track consumed signatures with a per-pusher revocation nonce or a `usedSignatures` bitmap. The simplest fix is to add a per-pusher nonce that is incremented on every successful `allowPushers` and included in the signed message:

```solidity
mapping(address => uint256) public pusherNonce;

// in allowPushers, sign over: (chainid, address(this), deadline, pusher, creator, pusherNonce[pusher])
// after writing namespaceRemapping[pusher] = msg.sender:
pusherNonce[pusher]++;
```

When the pusher calls `revokePusher()`, the nonce is already incremented, so any previously issued signature (which committed to the old nonce) is permanently invalid. Alternatively, record the deadline at which each pusher was last delegated and reject any `allowPushers` call that presents a deadline ≤ the recorded revocation timestamp.

---

### Proof of Concept

```
1. Pusher signs consent: (chainid, oracle, deadline=T+1day, pusher, creator)
2. Creator calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)
4. Creator immediately calls allowPushers(T+1day, [pusher], [sig])  ← SAME sig, no nonce
   → _ensureDeadline passes (T+1day > now)
   → ECDSA.recover returns pusher  ← signature still valid
   → namespaceRemapping[pusher] = creator  ← revocation silently overridden
5. Pusher's next fallback push lands in creator's namespace again
6. Steps 3–5 repeat until T+1day elapses
```

The test helper `_signConsent` and `_allowPusher` in the existing test suite confirm the signature domain and that a single signature is sufficient to establish delegation; no test covers the replay-after-revoke scenario. [6](#0-5)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
