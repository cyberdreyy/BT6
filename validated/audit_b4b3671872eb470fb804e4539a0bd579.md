### Title
Pusher Consent Signature Replayable Within Deadline Window, Nullifying `revokePusher()` — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` accepts any valid EIP-191 consent signature as long as `block.timestamp <= deadline`. No nonce or "used" flag is recorded. A creator who holds a pusher's signed consent can call `allowPushers` repeatedly with the same signature, re-establishing the delegation immediately after the pusher calls `revokePusher()`. The pusher's self-revocation is therefore ineffective for the entire duration of the deadline window, and any price pushes the pusher makes continue to land in the creator's namespace rather than the pusher's own.

### Finding Description

`allowPushers` constructs a hash over `(chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's ECDSA signature against it: [1](#0-0) 

The only replay guard is `_ensureDeadline`, which simply checks `block.timestamp <= deadline`: [2](#0-1) 

There is no nonce, no per-signature consumed flag, and no check that `namespaceRemapping[pusher]` is currently `address(0)` before writing. The code's own NatSpec comment acknowledges the concern:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The comment treats the deadline as the fix, but the deadline only prevents use *after* it expires — it does not prevent the creator from replaying the same signature any number of times *within* the window. `revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [4](#0-3) 

But the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple and restore `namespaceRemapping[pusher] = creator`, overwriting the revocation.

### Impact Explanation

The `fallback` push path resolves the namespace at call time: [5](#0-4) 

If the pusher is running an automated price feed and revokes their delegation intending to push into their own namespace (so that pools using `feedIdOf(pusher, slot, pos)` receive live prices), the creator's replay of the consent signature silently redirects every subsequent push back into the creator's namespace. The pusher's own oracle slots receive no further updates, causing stale prices (`timestampMs` frozen at the last pre-revocation value) to be returned to any pool or provider that reads `feedIdOf(pusher, ...)`. This matches the allowed impact gate: **stale bid/ask quote reaching a pool swap**.

### Likelihood Explanation

- The creator already holds the pusher's signed consent (they used it once to establish delegation).
- Replaying it costs only a single transaction and requires no additional off-chain material.
- The pusher has no on-chain mechanism to invalidate the signature before the deadline expires; their only recourse is to wait out the deadline window.
- Likelihood is **medium**: requires the pusher to have their own oracle feeds consumed by pools, but the delegation pattern is the normal operational setup described in the protocol.

### Recommendation

Record a per-pusher-per-creator nonce or a consumed-signature bitmap so that each consent signature can only establish the delegation once. The simplest fix is to add a `mapping(address => uint256) public pusherNonce` and include the nonce in the signed hash, incrementing it on every successful `allowPushers` call for that pusher. Alternatively, require that `namespaceRemapping[pusher] == address(0)` before accepting the signature, so a re-delegation after revocation always requires a fresh signature from the pusher.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator, deadline = now + 1 day
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — no new pusher consent needed
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation silently restored

// 5. Pusher's subsequent fallback pushes land in creator's namespace,
//    leaving feedIdOf(pusher, slot, pos) with a stale timestamp.
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
