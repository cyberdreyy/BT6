### Title
Slot-Level Timestamp Shared Across All Four Positions Allows a Stale Per-Position Price to Bypass the Staleness Guard — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1` packs four independent price positions into one 256-bit storage slot with a **single shared timestamp**. When a pusher updates any one position (carrying the old values of the other three), the slot timestamp advances for all four positions simultaneously. A consumer reading a position whose price data has not been refreshed since before the staleness window will see a fresh timestamp and pass the `MAX_REF_STALENESS` / `MAX_TIME_DELTA` guard in `AnchoredPriceProvider._readLeg()` or `PriceProvider._getBidAndAskPrice()`, allowing a stale price to reach pool swap math.

---

### Finding Description

**Slot layout** (256 bits, one EVM storage word):

```
bits 255…208 : oracle[0] (48 bits)
bits 207…160 : oracle[1] (48 bits)
bits 159…112 : oracle[2] (48 bits)
bits 111… 64 : oracle[3] (48 bits)
bits  63…  8 : timestamp (uint56, unix milliseconds)   ← ONE timestamp for all four
bits   7…  0 : reserved
``` [1](#0-0) 

`getOracleData(feedId)` unconditionally returns the slot-level timestamp for every position:

```solidity
data.timestampMs = _layout.timestampMs;   // same value for pos 0, 1, 2, 3
``` [2](#0-1) 

Both push paths (`fallback` and `updateBySignature`) overwrite the **entire** 256-bit slot word atomically — there is no per-position write:

```solidity
_writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
``` [3](#0-2) [4](#0-3) 

The design is explicitly acknowledged in the test suite:

> *"A sibling lane in the SAME slot shares the slot's 56-bit timestamp word … the timestamp is per-slot, not per-lane."* [5](#0-4) 

**The stale-bypass path:**

A pusher who needs to update only position 0 (e.g., only that asset's price moved) will:

1. Read the current slot: `[P0 | P1 | P2 | P3 | T0 | slotId]`
2. Modify position 0 and advance the timestamp: `[P0′ | P1 | P2 | P3 | T1 | slotId]`
3. Write the new word — the monotonicity guard passes because `T1 > T0`

After this write, `getOracleData` for positions 1, 2, and 3 returns `timestampMs = T1`, even though their price bytes `P1`, `P2`, `P3` are unchanged since `T0`. If `T0 < T1 − MAX_REF_STALENESS`, those positions are economically stale but the guard passes.

`AnchoredPriceProvider._readLeg()` consumes `refTime` directly from `oracle.price(feedId, pool)`:

```solidity
(mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (..., false);
``` [6](#0-5) 

`PriceProvider._getBidAndAskPrice()` has the identical pattern:

```solidity
(uint256 mid, uint256 spread, , uint256 refTime) =
    IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA)) { return (0, type(uint128).max); }
``` [7](#0-6) 

Both providers receive `refTime = T1` for a position whose price is from `T0`, so the staleness guard is silently bypassed and the stale mid price flows into `_computeBidAsk` / `_getBidAskFrom`, producing incorrect bid/ask quotes that the pool executes against.

**Analog to H-02:** Just as `playerOwnedPenguins[owner].length` (all-time count, including dead penguins) was used instead of `alivePenguins[owner].length()` (current alive count), the slot-level timestamp (refreshed by updating *any* position) is used instead of a per-position timestamp (reflecting only *that* position's last update). In both cases the wrong data structure is queried for a freshness/validity check, and the discrepancy is invisible to the consumer.

---

### Impact Explanation

A pool whose `AnchoredPriceProvider` or `PriceProvider` is backed by a `CompressedOracleV1` feed will quote bid/ask prices derived from a stale mid. Traders can arb the gap between the stale on-chain quote and the real market price, extracting value from LPs. The magnitude scales with `MAX_REF_STALENESS` (up to 7 days by constructor bound) and the asset's volatility during that window. In the worst case the pool quotes a price that is many percent away from fair value, causing direct loss of LP principal — a pool-insolvency / swap-conservation failure matching the allowed impact gate. [8](#0-7) 

---

### Likelihood Explanation

Medium. The trigger is a **non-malicious, operationally normal** pusher action: updating only the positions whose prices have moved materially (a standard gas-optimisation pattern). No signature forgery, no privileged escalation, and no malicious setup is required. Any delegated pusher — or the creator themselves — who updates a subset of a slot's four positions will inadvertently refresh the timestamp for the untouched positions. The `allowPushers` / `allowContractPushers` delegation paths make this reachable by any address the creator has authorised. [9](#0-8) 

---

### Recommendation

Add a **per-position timestamp** to the slot layout. The current 4 × 48-bit lane + 56-bit timestamp + 8-bit reserved = 256 bits leaves no room, so one option is to split the slot into two 128-bit halves, each holding two 48-bit lanes and one 32-bit timestamp (second-resolution, sufficient for staleness windows up to ~136 years). Alternatively, enforce at the push layer that a slot word may only be written when **all four lanes carry fresh data** (i.e., the pusher must update all positions together), and document this as a hard protocol invariant. Either change eliminates the case where a sibling-lane update silently refreshes the staleness clock for positions whose prices have not changed.

### Proof of Concept

```
T0  : Pusher writes slot 5 for creator C:
        [ETH_P0 | BTC_P1 | SOL_P2 | AVAX_P3 | T0 | slotId=5]
      All four feeds are fresh; staleness check passes for all.

T1 = T0 + MAX_REF_STALENESS + 1:
      ETH price moves; pusher reads current slot, updates only lane 0:
        [ETH_P0' | BTC_P1 | SOL_P2 | AVAX_P3 | T1 | slotId=5]
      Monotonicity guard: T1 > T0 → write accepted.

Pool backed by feedIdOf(C, 5, 1)  [BTC/USD]:
  oracle.price(feedId, pool)
    → getOracleData → _layout.timestampMs = T1   ← stale price, fresh timestamp
    → refTime = T1 / 1000
  AnchoredPriceProvider._readLeg:
    _isStale(T1/1000, block.timestamp, MAX_REF_STALENESS) → false  ← guard bypassed
    mid = BTC_P1  (price from T0, now MAX_REF_STALENESS+1 seconds old)
  _computeBidAsk(BTC_P1, ...) → incorrect bid/ask quoted to traders
``` [2](#0-1) [10](#0-9) [6](#0-5)

    function _loadSlotLayout(bytes32 slotIndex) internal view returns (SlotLayout memory _layout) {
        uint256 slotValue;
        assembly ("memory-safe") {

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
