### Title
`priceGuard` Bounds Configured but Never Enforced in `_readPrice` — Out-of-Bounds Oracle Prices Reach Pools Unchecked - (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase` exposes `setPriceGuard` to let admins or `stateGuard` addresses set absolute `min`/`max` price bounds per feed. Those bounds are stored in `priceGuard[feedId]` but are **never read back** inside `_readPrice`, the single internal function that both `price()` (pool path) and `integratorPrice()` (integrator path) call. Any price — including one that is zero, inverted, or wildly out of range — is returned verbatim to the pool's price provider and ultimately drives swap execution.

---

### Finding Description

`setPriceGuard` writes a `PriceGuard{min, max}` struct: [1](#0-0) 

The guard is stored in the public mapping: [2](#0-1) 

`_readPrice`, the sole internal read path, ignores it entirely: [3](#0-2) 

Both public read surfaces delegate to `_readPrice` without any guard check: [4](#0-3) [5](#0-4) 

The `priceGuard` mapping is therefore dead state: it is written, emitted, and publicly visible, but it never gates any price that leaves the contract.

---

### Impact Explanation

A price that falls outside the configured `[min, max]` window — whether from a momentarily corrupted Pyth/Chainlink payload, a compromised pusher, or a stale extreme value — is forwarded unchanged to the pool's price provider. The pool then computes bid/ask from that unclamped mid-price and executes swaps against it. Depending on the magnitude of the deviation, traders can receive more output than the oracle curve permits (swap conservation failure) or LPs suffer losses because the pool settles at a price that does not reflect fair value.

---

### Likelihood Explanation

The `priceGuard` feature is explicitly documented as a safety mechanism against oracle manipulation. Admins are expected to configure it for production feeds. Any event that pushes a price outside the guard window — a flash-corrupted Pyth update, a Chainlink DON anomaly, or a compromised pusher key — would normally be caught by the guard. Because the guard is never applied, every such event silently propagates to pools. The trigger does not require any privileged action by the attacker; it only requires the upstream oracle to emit an out-of-range value, which is precisely the scenario the guard was designed to handle.

---

### Recommendation

Enforce the guard inside `_readPrice`:

```solidity
function _readPrice(bytes32 feedId)
    internal
    view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = _oracleDataRaw(feedId);
    PriceGuard memory guard = priceGuard[feedId];
    if (guard.min != 0 || guard.max != 0) {
        require(
            data.price >= guard.min && data.price <= guard.max,
            PriceOutOfBounds(feedId, data.price, guard.min, guard.max)
        );
    }
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
```

This mirrors the DAO fix pattern: the threshold that was already being stored must also be checked at the point of consumption, not merely at configuration time.

---

### Proof of Concept

1. Admin calls `setPriceGuard(feedId, 1e8, 2e8)` — setting a valid price window of `[1.00, 2.00]` (8-decimal).
2. A corrupted Pyth payload pushes `price = 0` (or `price = 1e12`) for that feed. The monotonicity check passes because the timestamp is newer.
3. A pool calls `price(feedId, pool)`. Execution reaches `_readPrice`, which reads `data.price = 0` and returns it without consulting `priceGuard`.
4. The pool's price provider receives `mid = 0`, computes `bid = 0`, `ask = 0`, and the pool executes swaps at a zero price — draining LP assets. [3](#0-2) [1](#0-0)
