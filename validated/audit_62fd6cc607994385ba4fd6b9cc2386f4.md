### Title
`allowPushers` delegation signature is replayable after `revokePusher`, permanently re-hijacking a pusher's namespace writes until the deadline — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts a pusher's EIP-191 consent signature that is bound to `(chainid, address(this), deadline, pusher, creator)` but contains **no nonce**. The same signature is therefore replayable an unlimited number of times before the deadline. A creator who holds a pusher's consent signature can call `allowPushers` again immediately after the pusher calls `revokePusher()`, silently re-establishing the delegation. The pusher's revocation is completely ineffective for the entire window between the revocation and the deadline.

---

### Finding Description

`allowPushers` in `CompressedOracle.sol` verifies the pusher's EIP-191 signature and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The signed payload contains no per-use nonce. The code comment itself acknowledges the partial mitigation:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

But the deadline only prevents re-establishment **after** the deadline expires. Before the deadline, the creator can replay the identical signature any number of times. `revokePusher` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

There is no invalidation of the outstanding signature. The creator immediately calls `allowPushers` with the same bytes to restore `namespaceRemapping[pusher] = creator`.

The `fallback` push path reads `namespaceRemapping[msg.sender]` at the start of every push and routes the slot write to whichever namespace is currently set:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after their revocation — believing they are writing to their own namespace — is silently redirected to the creator's namespace.

---

### Impact Explanation

The pusher's price data continues to be attributed to the creator's namespace against the pusher's will. Any pool reading `feedIdOf(creator, slotIndex, positionIndex)` via an `AnchoredPriceProvider` or `CompressedOracle.price()` will consume the pusher's data as if the delegation were still active. If the pusher has been repurposed (e.g., now pushing prices for a different asset or with different spread parameters), the creator's pool receives a feed that was never intended for it — a bad-price execution path. The pusher has no on-chain mechanism to stop this before the deadline expires.

---

### Likelihood Explanation

The creator already holds the pusher's signed consent (they used it to establish the delegation). No additional off-chain action is required. The creator simply calls `allowPushers` again with the same calldata immediately after observing the `PusherRevoked` event. The only precondition is that the deadline has not yet passed, which is the normal case for any delegation with a multi-day or multi-week deadline.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` bitmap. The simplest fix is to add a `mapping(address => uint256) public pusherNonce` and include the nonce in the signed payload:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusherNonce[pusher]++, pusher, msg.sender))
```

Alternatively, invalidate the delegation slot on revocation by storing a per-pusher revocation epoch and requiring the signature to commit to the current epoch. Either approach ensures that a revoked signature cannot be replayed.

---

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
