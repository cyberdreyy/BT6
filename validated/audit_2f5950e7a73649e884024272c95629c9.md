### Title
Single-Step `transferProviderOwnership` Permanently Locks `setSource` Control, Enabling Unrecoverable Pool Swap Halt - (`AnchoredProviderFactory.sol`)

---

### Summary

`AnchoredProviderFactory.transferProviderOwnership` transfers provider ownership in a single atomic step with only a zero-address check. If the current owner accidentally supplies a wrong address (e.g., a typo or a dead EOA), ownership is irrecoverably lost. The most critical consequence is that `setSource` — the only path to switch an `AnchoredPriceProvider` between reference mode and source mode — becomes permanently inaccessible, leaving pools that depend on that provider unable to recover from a broken source oracle.

---

### Finding Description

`AnchoredProviderFactory.transferProviderOwnership` immediately overwrites `providerOwner[provider]` with `newOwner` after a single zero-address check:

```solidity
// AnchoredProviderFactory.sol lines 230-240
function transferProviderOwnership(address provider, address newOwner)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    providerOwner[provider] = newOwner;          // ← immediate, irreversible
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);

    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
``` [1](#0-0) 

There is no pending-owner stage and no acceptance step. The same single-step pattern is present in `PriceProviderFactory.transferProviderOwnership` and `PriceProviderFactoryL2.transferProviderOwnership`. [2](#0-1) [3](#0-2) 

The `providerOwner` role is the exclusive gatekeeper for `setSource` in `AnchoredProviderFactory`:

```solidity
// AnchoredProviderFactory.sol lines 224-228
function setSource(address provider, address newSource)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    AnchoredPriceProvider(provider).setSource(newSource);
    ...
}
``` [4](#0-3) 

Inside `AnchoredPriceProvider`, `setSource` is callable only by the factory:

```solidity
// AnchoredPriceProvider.sol lines 178-182
function setSource(address newSource) external {
    require(msg.sender == factory, OnlyFactory());
    source = newSource;
    ...
}
``` [5](#0-4) 

The factory is immutable in the provider, so there is no alternative path to call `setSource`.

Notably, the codebase already implements the correct two-step pattern for `stateGuard` transfers in both `OracleBase` contracts (`setPendingStateGuardRole` / `acceptStateGuardRole`), demonstrating developer awareness of the pattern — but it was not applied to `providerOwner`. [6](#0-5) 

---

### Impact Explanation

When `source != address(0)`, `AnchoredPriceProvider._computeBidAsk` reads bid/ask from the source contract and clamps it to the reference band. If the source reverts, runs out of gas, or returns malformed data, `_readSource` returns `(false, 0, 0)`, and `_computeBidAsk` returns the `(0, type(uint128).max)` sentinel:

```solidity
// AnchoredPriceProvider.sol lines 320-327
if (_source != address(0)) {
    bool ok;
    (ok, cBid, cAsk) = _readSource(_source);
    if (!ok) {
        return (0, type(uint128).max);
    }
}
``` [7](#0-6) 

`getBidAndAskPrice` then reverts with `FeedStalled`:

```solidity
// AnchoredPriceProvider.sol lines 214-217
function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
``` [8](#0-7) 

Every swap through any pool using this provider is permanently blocked. The only recovery — switching back to reference mode via `setSource(provider, address(0))` — requires `onlyProviderOwner`, which is now an uncontrolled address. The pool's swap functionality is permanently broken with no on-chain recovery path.

---

### Likelihood Explanation

Provider ownership transfer is a routine operational action. A single address typo, use of a hardware wallet address that has been lost, or transfer to a contract that cannot call `transferProviderOwnership` (e.g., a multisig that is later bricked) is sufficient to trigger the condition. The function is callable by any provider owner without admin involvement, making the surface area proportional to the number of deployed providers. Likelihood is **Medium**.

---

### Recommendation

Implement a two-step transfer pattern identical to the existing `stateGuard` mechanism already present in `OracleBase`:

1. Add a `mapping(address provider => address) public pendingProviderOwner` storage variable.
2. Replace the immediate write in `transferProviderOwnership` with a write to `pendingProviderOwner[provider]`.
3. Add an `acceptProviderOwnership(address provider)` function that requires `msg.sender == pendingProviderOwner[provider]`, then promotes the pending owner to `providerOwner` and clears the pending slot.

Apply this pattern to `AnchoredProviderFactory`, `PriceProviderFactory`, and `PriceProviderFactoryL2`.

---

### Proof of Concept

1. Alice deploys an `AnchoredPriceProvider` in source mode (pointing to a live source contract) via `AnchoredProviderFactory.createAnchoredProvider(...)`. A pool is registered against this provider.
2. Alice calls `transferProviderOwnership(provider, 0xDEAD...BEEF)` with a mistyped address. The call succeeds; `providerOwner[provider]` is immediately set to the dead address.
3. The source contract is later upgraded and its `getBidAndAskPrice()` selector changes, causing every staticcall from `_readSource` to return `success = false`.
4. `_computeBidAsk` returns `(0, type(uint128).max)`. `getBidAndAskPrice()` reverts with `FeedStalled` on every call.
5. Alice attempts `setSource(provider, address(0))` to fall back to reference mode. The call reverts with `NotProviderOwner()` because `msg.sender != providerOwner[provider]`.
6. All swaps through the pool are permanently halted. There is no on-chain recovery path. [1](#0-0) [8](#0-7)
