### Title
`allowPushers` Signature Replay Within Deadline Window Bypasses Pusher Revocation, Routing Prices to Wrong Namespace - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

`CompressedOracle.allowPushers` does not invalidate a pusher's consent signature after it is consumed. A creator who holds a valid, unexpired signature can replay it an unlimited number of times within the deadline window, silently re-establishing a delegation the pusher has already revoked via `revokePusher()`. This is the direct analog of M-01: just as `addIncentive` could re-add an incentive whose `secondsClaimed` was already non-zero, `allowPushers` can re-add a delegation whose `namespaceRemapping` was already cleared, causing the pusher's subsequent price pushes to land in the wrong namespace and leaving the pusher's own feeds stale.

### Finding Description

`allowPushers` verifies the pusher's EIP-191 signature and then unconditionally writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The code comment explicitly acknowledges the revocation-replay risk and names the deadline as the mitigation: [2](#0-1) 

However, the deadline only prevents replay *after* it expires. Within `[0, deadline]` the same `(deadline, pusher, creator, sig)` tuple is accepted on every call because no used-signature set or per-pusher nonce is maintained. `revokePusher` clears `namespaceRemapping[pusher]` to `address(0)`: [3](#0-2) 

But the creator can immediately call `allowPushers` again with the identical calldata, restoring the mapping. The pusher has no on-chain mechanism to invalidate the outstanding signature before the deadline expires.

The fallback push path resolves the namespace at call time: [4](#0-3) 

So every push the pusher makes after their revocation — intending to write to their own namespace — is silently redirected to the creator's namespace instead.

### Impact Explanation

The pusher's own namespace (`feedIdOf(pusher, slotIndex, positionIndex)`) receives no updates after the replay. Any pool registered against those feeds reads a stale price. The `feedExists` / staleness guards in the provider layer treat a zero or old `timestampMs` as a stall: [5](#0-4) 

A stalled feed causes `getBidAndAskPrice` to return the `(0, type(uint128).max)` sentinel, halting swaps in any pool that relies on the pusher's own feeds. This is a direct bad-price / unusable-swap impact.

### Likelihood Explanation

The trigger is unprivileged: any creator who legitimately obtained a pusher's consent signature (a normal operational step) can replay it. No special role or admin access is required. The window is bounded only by the deadline the pusher chose when signing — commonly hours to days in practice. The pusher has no way to close the window early.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

        address creator = namespaceRemapping[msg.sender];
