### Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Ratio Feeds ŌĆö (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (e.g., BTC/USD ├Ę ETH/USD ŌåÆ BTC/ETH). Both the `baseFeedId` and `quoteFeedId` legs are validated through the same `_readLeg()` helper, which applies the single immutable `MAX_REF_STALENESS` to both feeds. The `AnchoredProviderFactory` envelope that bounds `maxRefStaleness` is keyed exclusively on the **base** feed's class, with no consideration for the quote feed's natural update frequency. This is the direct structural analog of M-12.

---

`MAX_REF_STALENESS` is a single immutable set at construction: [3](#0-2) 

The `AnchoredProviderFactory` validates `maxRefStaleness` only against the **base** feed's class envelope. The factory comment explicitly acknowledges the quote feed is treated differently: *"The envelope is keyed on `baseFeedId` (the provider's class); the ref feed only contributes its uncertainty."* [4](#0-3) 

The `ChainlinkOracle` (Data Streams) ingests both v3 crypto feeds (high-frequency, seconds resolution) and v4 RWA feeds (low-frequency, market-hours only). A synthetic provider combining a v3 crypto `baseFeedId` with a v4 RWA `quoteFeedId` ŌĆö or vice versa ŌĆö will have a `MAX_REF_STALENESS` calibrated to one feed's class envelope that is wholly inappropriate for the other feed's natural update cadence. ---

### Impact Explanation

Two failure modes, mirroring M-12 exactly:

1. **Constant DoS**: If `MAX_REF_STALENESS` is calibrated to the faster-updating base feed (e.g., 60 s for a crypto feed), the slower quote feed (e.g., an RWA feed updating once per trading day) will fail `_isStale()` for the entire interval between its updates. `getBidAndAskPrice()` reverts `FeedStalled`, making the pool unusable for swaps.

2. **Stale-price execution**: If `MAX_REF_STALENESS` is set wide enough to accommodate the slower quote feed (e.g., 86 400 s), the faster base feed's price can be up to 24 hours stale before being rejected. A trader observing a stale base price can execute swaps at a price that no longer reflects market reality, extracting value from LPs.

Both outcomes satisfy the Allowed Impact Gate: broken core pool functionality and bad-price execution reaching a pool swap.

---

### Likelihood Explanation

- The synthetic ratio mode is permissionlessly accessible via `createAnchoredProvider` with any non-zero `quoteFeedId`.
- The factory envelope enforces staleness bounds only for the base feed's class; no analogous guard exists for the quote feed.
- The `ChainlinkOracle` already stores both v3 (crypto, ~seconds) and v4 (RWA, ~daily) feeds in the same contract, making a mismatched synthetic pair a realistic deployment scenario.
- The `AnchoredProviderFactory` comment explicitly notes the quote feed is not envelope-validated for staleness, confirming the gap is structural, not accidental. [6](#0-5) 

---

### Recommendation

Introduce a separate `MAX_QUOTE_STALENESS` immutable (set at construction alongside `MAX_REF_STALENESS`) and pass it to `_readLeg()` when reading the quote leg:

```solidity
// _readLeg(feedId, maxStaleness)
if (_isStale(refTime, block.timestamp, maxStaleness)) return ...;

// caller:
_readLeg(baseFeedId,  MAX_REF_STALENESS);
_readLeg(quoteFeedId, MAX_QUOTE_STALENESS);
```

The `AnchoredProviderFactory` should validate `maxQuoteStaleness` against the quote feed's class envelope (or at minimum require it to be set independently of the base feed's envelope). This mirrors the fix applied in the JOJO audit: two separate heartbeat/staleness parameters, one per feed.

---

### Proof of Concept

1. Admin configures a `MAJORS` envelope with `stalenessMin = 1`, `stalenessMax = 60` (appropriate for crypto feeds).
2. Curator calls `createAnchoredProvider(oracle, BTC_USD_FEED, RWA_FEED, ..., maxRefStaleness=60, ...)`. The factory validates `60` against the `MAJORS` envelope (keyed on `BTC_USD_FEED`) ŌĆö passes.
3. The RWA quote feed (`RWA_FEED`) updates once per trading day (~86 400 s).
4. At `t = 61 s` after the last RWA update, a swap triggers `getBidAndAskPrice()`.
5. `_readLeg(RWA_FEED)` calls `_isStale(refTime, block.timestamp, 60)` ŌåÆ `(now ŌłÆ refTime) = 61 > 60` ŌåÆ returns `ok = false`.
6. `_getBidAndAskPrice()` returns `(0, type(uint128).max)` ŌåÆ `getBidAndAskPrice()` reverts `FeedStalled`.
7. The pool is DoS'd for the remaining ~86 339 s of the RWA feed's update interval.

Alternatively, with `maxRefStaleness = 86400` to avoid DoS: the BTC/USD base feed price can be up to 24 hours stale, allowing a trader to exploit the divergence between the stale on-chain price and the current market price. [1](#0-0) [2](#0-1) [7](#0-6)

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference ŌåÆ not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity ŌĆö mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
