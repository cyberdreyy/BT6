### Title
Unprivileged caller can un-blacklist any pool via `OracleBase.register` for 1 wei, permanently defeating the admin's blacklist — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
`OracleBase.register` is a public payable function with no role check. When called with `msg.value >= registrationFee` (default: 1 wei) for a blacklisted pool that belongs to an approved factory, it unconditionally sets `blacklisted[pool] = false`. This lets any unprivileged caller permanently undo the admin's `setBlacklist` decision at negligible cost.

### Finding Description
`setBlacklist` is `onlyRole(ADMIN_ROLE)` and is the intended mechanism to revoke a pool's oracle-read access. However, `register` (lines 201–214) has no role check and contains:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;          // ← unconditional clear
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
```

The only preconditions are:
1. `msg.value >= registrationFee` (1 wei by default)
2. `factory` is in `approvedFactories` (public knowledge)
3. `IPoolFactory(factory).isPool(pool)` returns true (the pool is already a legitimate pool — it was blacklisted *after* creation)

All three are trivially satisfiable by any caller for any pool that was blacklisted post-registration. The NatDoc on `register` even documents this side-effect: *"Paying also clears any blacklist on the pool."*

### Impact Explanation
The blacklist is the oracle's primary runtime security control. If the admin blacklists a pool (e.g., because it is being used to drain liquidity via oracle manipulation, or because a vulnerability was discovered), an attacker can immediately restore the pool's oracle-read access for 1 wei. The pool can then call `price(feedId, pool)` during `beforeSwap` (via `PriceVelocityGuardExtension` or any other oracle-consuming extension), receiving live oracle prices and continuing swaps. This is a direct admin-boundary break: an unprivileged path overrides an explicit privileged security decision, enabling bad-price execution or continued exploitation of a pool the admin intended to halt.

### Likelihood Explanation
Trivially exploitable. The cost is 1 wei. The attacker only needs to know the pool address and any approved factory address (both are on-chain public state). No special permissions, no off-chain data, no complex setup required.

### Recommendation
Remove the blacklist-clearing logic from `register`. Blacklist removal should only be possible via `setBlacklist` (ADMIN_ROLE). If re-registration of a previously blacklisted pool must be allowed, require an explicit admin un-blacklist first, then allow registration separately:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(!blacklisted[pool], Blacklisted(pool));   // ← reject, don't clear
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

### Proof of Concept
```solidity
// 1. Admin blacklists the pool
oracle.setBlacklist(pool, true);
assert(oracle.blacklisted(pool) == true);

// 2. Attacker (any address) re-registers for 1 wei
vm.prank(attacker);
oracle.register{value: 1 wei}(feedId, pool, approvedFactory);

// 3. Blacklist is cleared
assert(oracle.blacklisted(pool) == false);
assert(oracle.registeredPool(feedId, pool) == true);

// 4. Pool can now call price() during beforeSwap
// vm.prank(provider); oracle.price(feedId, pool); // succeeds
```
