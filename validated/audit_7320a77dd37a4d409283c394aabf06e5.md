The vulnerability is real. Let me trace it precisely.

**The core issue:** `allowPushers` has no used-signature tracking or nonce. The only replay protection is the deadline, which only prevents use *after* expiry — not re-use *before* expiry.

The code comment in `allowPushers` even acknowledges the problem it was trying to solve:

> "the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it"

But the deadline does not solve this. A deadline-bounded signature is replayable by the creator any number of times until `block.timestamp > deadline`.

**Exact attack trace:**

1. Pusher signs: `keccak256(abi.encode(block.chainid, address(this), D1, pusher, creator))` with `D1 = now + 30 days`
2. Creator calls `allowPushers(D1, [pusher], [sig])` → `namespaceRemapping[pusher] = creator` [1](#0-0) 
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)` [2](#0-1) 
4. Creator calls `allowPushers(D1, [pusher], [sig])` again with the **identical** signature — `_ensureDeadline` passes (D1 still future), ECDSA recovery succeeds (same hash), `namespaceRemapping[pusher] = creator` is written again [3](#0-2) 
5. `_ensureDeadline` only checks `block.timestamp <= deadline` — no nonce, no consumed-signature set [4](#0-3) 

**Consequence in the fallback push path:**

After re-delegation, the pusher's fallback pushes resolve `namespaceRemapping[msg.sender]` = creator, so every push lands in the creator's namespace — not the pusher's own. [5](#0-4) 

The pusher, believing they have revoked, may now push data intended for their own namespace (different asset, different scale, different slot layout). That data overwrites the creator's live feed slots. If those feeds are consumed by a pool via the provider chain, the pool receives a corrupted price → bad-price execution → fund loss.

### Title
Signature Replay in `allowPushers` Allows Creator to Re-Establish Revoked Pusher Delegation Without New Consent — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` uses a deadline-bounded EIP-191 signature as the sole replay guard. After a pusher calls `revokePusher()`, the creator can immediately re-call `allowPushers` with the original signature (still valid until deadline) to silently restore delegation. The pusher's revocation is ineffective for the entire lifetime of the original deadline.

### Finding Description
`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

There is no nonce and no consumed-signature registry. `_ensureDeadline` only enforces `block.timestamp <= deadline`. Once a pusher signs consent with a future deadline, that signature remains valid and replayable by the creator for the entire deadline window, regardless of how many times `revokePusher` is called in between. [6](#0-5) 

### Impact Explanation
After replay, `namespaceRemapping[pusher] = creator` is restored. The pusher's subsequent fallback pushes — intended for their own namespace — are silently redirected into the creator's namespace. If the creator's feeds are consumed by a pool, the pool receives prices from a pusher who has explicitly revoked consent, potentially with data for a different asset or scale, causing bad-price execution and direct fund loss for traders.

### Likelihood Explanation
Requires a creator who is willing to act adversarially against their own pusher. The pusher must continue pushing after revoking (e.g., pushing for their own namespace). The window is bounded by the original deadline, which can be set to weeks or months. Likelihood is **medium**.

### Recommendation
Track consumed signatures. Add a `mapping(bytes32 => bool) private _usedSignatures` and mark each digest as used on first acceptance:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedSignatures[hash], "signature already used");
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedSignatures[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, include a per-pusher nonce in the signed digest so the pusher can invalidate prior signatures by incrementing their nonce on revocation.

### Proof of Concept

    uint256 deadline = block.timestamp + 30 days;

    // Step 1: pusher signs consent
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 2: creator delegates
    vm.prank(creator);

    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 3: pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 4: creator replays the SAME signature — should revert, but does not
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);

    // Delegation is re-established without new pusher consent
    assertEq(oracle.namespaceRemapping(pusher), creator, "INVARIANT VIOLATED: revoked pusher re-delegated via replay");
}
