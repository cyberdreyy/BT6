### Title
Pusher revocation is ineffective within the deadline window — creator can replay the original consent signature to silently re-establish delegation after `revokePusher()` - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature (while the deadline is still valid) to re-establish the delegation, making the revocation completely ineffective within the deadline window.

---

### Finding Description

The `allowPushers` function requires a pusher-signed consent message bound to `(chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

The code's own NatSpec acknowledges the exact threat:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The deadline is the intended mitigation, but it is insufficient. The signature is **never invalidated after first use**. The attack sequence is:

1. Pusher signs consent with `deadline = now + N days`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`. [3](#0-2) 

4. Creator immediately calls `allowPushers(deadline, [pusher], [sig])` again with the **identical signature** — the deadline has not expired, the signature is cryptographically valid, and there is no used-signature registry. `namespaceRemapping[pusher] = creator` is restored.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So every subsequent push by the pusher continues to land in the creator's namespace, not the pusher's own.

The analog to the SoftClay bug is exact: the "claimable" resource (the pusher's signed consent) is not zeroed/consumed after first use, so the creator can re-claim it unlimited times within the deadline window.

---

### Impact Explanation

A pusher who discovers a creator is using their oracle data maliciously (e.g., to feed manipulated prices into a pool backed by `AnchoredPriceProvider`) cannot effectively revoke their delegation within the deadline window. The creator can loop `allowPushers` with the same signature on every block, keeping the pusher's data flowing into the creator's namespace indefinitely. The pusher's only recourse is to stop pushing entirely, which may itself cause stale-price harm to the pool. If the pusher's data is the live price source for a production pool, the creator retains write authority over that feed despite the pusher's explicit revocation.

---

### Likelihood Explanation

Medium. Requires a creator who is willing to act adversarially against their own pusher. The window of exposure equals the remaining lifetime of the original deadline (which can be set to days or weeks). The replay call is a single cheap transaction the creator can automate.

---

### Recommendation

Maintain a `mapping(bytes32 => bool) private _usedSignatures` keyed on the signature hash (or a `mapping(address => uint256) private _pusherNonce` incremented on each `allowPushers` and included in the signed payload). On `revokePusher` or `removePushers`, increment the nonce or mark the signature hash as spent so that any previously issued consent is immediately invalidated regardless of its deadline.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent (deadline = now + 7 days)
bytes memory sig = pusher.sign(
    keccak256(abi.encode(chainid, oracle, deadline, pusherAddr, creatorAddr))
);

// 2. Creator establishes delegation
oracle.allowPushers(deadline, [pusherAddr], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher revokes
vm.prank(pusherAddr);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature — no used-signature check exists
vm.prank(creatorAddr);
oracle.allowPushers(deadline, [pusherAddr], [sig]);
// namespaceRemapping[pusher] == creator again — revocation nullified ✓

// 5. Pusher's next push lands in creator's namespace, not their own
vm.prank(pusherAddr);
(bool ok,) = address(oracle).call(encodedSlotWord);
assertTrue(ok);
assertEq(oracle.getOracleData(oracle.feedIdOf(creatorAddr, slot, pos)).price, pushedPrice);
assertEq(oracle.getOracleData(oracle.feedIdOf(pusherAddr,  slot, pos)).price, 0);
```

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
