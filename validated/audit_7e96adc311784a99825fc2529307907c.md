### Title
Pool Admin Can Frontrun User Swaps by Instantly Increasing Fees Without Timelock — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolAdminFees()` allows a pool admin to instantly raise the admin spread fee and notional fee to the maximum allowed caps with no timelock. A malicious pool admin can observe a pending swap in the mempool, front-run it by calling `setPoolAdminFees()` to push fees to their ceiling, and then restore them afterward — extracting value from every targeted swap.

---

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees()` enforces only a cap check against `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` before immediately writing the new fees to `poolFeeConfig` and calling `pool.setPoolFees()`: [1](#0-0) 

There is no pending/execute two-step pattern and no time delay. Compare this to the price-provider change path, which explicitly enforces a per-pool `priceProviderTimelock`: [2](#0-1) 

The hard caps themselves (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%; `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000`, i.e. 1%) are set at construction and can never be exceeded: [3](#0-2) 

But the factory owner can set `maxAdminSpreadFeeE6` all the way up to `HARD_MAX_SPREAD_FEE_E6` via `setFeeCaps()`: [4](#0-3) 

So the pool admin's effective ceiling is up to 20% spread fee — changeable in a single transaction with no delay.

The attack sequence:

1. Pool admin monitors the mempool for a large swap against their pool.
2. Pool admin front-runs the swap with `setPoolAdminFees(pool, maxAdminSpreadFeeE6, maxAdminNotionalFeeE8)`.
3. The user's swap executes at the inflated fee; the pool admin's fee destination receives the excess.
4. Pool admin immediately calls `setPoolAdminFees(pool, 0, 0)` to restore normal fees.

Note that `setPoolAdminFees()` does call `collectFees()` first (to settle accrued fees at the old rate before the change), but this does not prevent the front-run — it only ensures the admin collects previously accrued fees before the rate jumps. [5](#0-4) 

---

### Impact Explanation

Every swap through the pool reads the live `spreadFeeE6` and `notionalFeeE8` stored in the pool at execution time. A front-run fee increase directly widens the effective bid/ask spread the user trades against, causing them to receive fewer output tokens than they expected. At the maximum admin spread cap of 20%, a user swapping a large position could lose a material fraction of their principal to fees they never agreed to. This is a direct, quantifiable loss of user principal — matching the "direct loss of user principal or owed LP assets" impact gate.

---

### Likelihood Explanation

The pool admin role is assigned at pool creation by the pool creator and is not the protocol-level factory owner. It is a semi-trusted role. Any pool whose admin is a profit-motivated or malicious actor can execute this attack on every swap. The attack requires no special on-chain conditions beyond a pending swap in the mempool and is executable on any EVM chain where the factory is deployed. Likelihood is **Medium**: it requires a malicious pool admin, but that assumption is no stronger than the M-26 analog's "malicious admin" premise, which the C4 judge explicitly accepted.

---

### Recommendation

Apply the same two-step timelock pattern already used for price-provider changes to admin fee changes:

1. Introduce a `pendingAdminFeeConfig` mapping and a `pendingAdminFeeExecuteAfter` mapping (analogous to `pendingPriceProvider` / `pendingPriceProviderExecuteAfter`).
2. Split `setPoolAdminFees()` into `proposePoolAdminFees()` (records the pending change and a `block.timestamp + feeTimelock` deadline) and `executePoolAdminFees()` (applies the change only after the deadline).
3. Expose a per-pool `feeTimelock` (set at creation, similar to `priceProviderTimelock`) so users can verify the maximum delay before a fee change takes effect.

This mirrors the existing guard for price-provider changes and eliminates the front-run window.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.35;

// Assume: pool deployed with adminSpreadFeeE6 = 0, maxAdminSpreadFeeE6 = 200_000 (20%)
// Attacker = pool admin

contract FrontRunFee {
    IMetricOmmPoolFactory factory;
    address pool;
    address victim;

    // Step 1: pool admin sees victim's swap in mempool (e.g. exactInputSingle)
    // Step 2: pool admin submits this tx with higher gas to land first
    function frontRun() external {
        // Instantly raise admin spread fee to 20%
        factory.setPoolAdminFees(pool, 200_000, 0);
        // victim's swap now executes at 20% spread fee
    }

    // Step 3: after victim's swap confirms, restore fee
    function restore() external {
        factory.setPoolAdminFees(pool, 0, 0);
        // collect the inflated fees
        factory.collectPoolFees(pool);
    }
}
```

No special permissions beyond the pool admin role are required. The attack is repeatable on every swap.

  function setPoolAdminFees(address pool, uint24 newAdminSpreadFeeE6, uint24 newAdminNotionalFeeE8)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();

    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool)
      .collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
      );

    c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
    c.adminNotionalFeeE8 = newAdminNotionalFeeE8;
    poolFeeConfig[pool] = c;

    IMetricOmmPoolFactoryActions(pool)
      .setPoolFees(c.protocolSpreadFeeE6 + c.adminSpreadFeeE6, c.protocolNotionalFeeE8 + c.adminNotionalFeeE8);
    emit PoolAdminSpreadFeeUpdated(pool, newAdminSpreadFeeE6);
    emit PoolAdminNotionalFeeUpdated(pool, newAdminNotionalFeeE8);
  }
