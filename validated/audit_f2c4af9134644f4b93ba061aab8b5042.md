### Title
Pusher delegation consent signature is replayable after `revokePusher()`, allowing a creator to silently restore revoked write authority and redirect price pushes into their namespace — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` accepts a pusher's EIP-191 consent signature but records no evidence that the signature has been consumed. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original signature — provided `block.timestamp < deadline` — to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is therefore not permanent for the remaining lifetime of the deadline.

### Finding Description
`allowPushers` verifies the pusher's EIP-191 signature and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

The signed message commits to `(chainid, oracle, deadline, pusher, creator)`. There is no nonce, no used-signature bitmap, and no on-chain record that the pusher has revoked.

`revokePusher()` sets `namespaceRemapping[msg.sender] = address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

The creator still holds the original signature. As long as `block.timestamp < deadline`, the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, restoring `namespaceRemapping[pusher] = creator`.

The code comment explicitly acknowledges the deadline's purpose: *"an undated signature could re-establish a delegation AFTER the pusher revoked it."* But the deadline only prevents replay after expiry — it does not prevent replay within the deadline window. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the deadline, which can be up to any future timestamp the pusher agreed to when signing.

This is the direct analog of the AirdropBroker `phase4Users` bug: just as `phase4Users[user]` is not cleared between epochs (allowing re-participation), `namespaceRemapping[pusher]` is not permanently cleared after revocation (allowing re-delegation using the same stale consent).

### Impact Explanation
After the creator replays the signature, every subsequent push from the pusher's address lands in the **creator's namespace** instead of the pusher's own namespace (the `fallback` push path resolves the namespace via `namespaceRemapping[msg.sender]`). If the pusher is simultaneously operating their own feeds — pushing data they intend for their own namespace — those pushes are silently redirected to the creator's namespace. The creator's price feeds receive data the creator did not intend, and the pusher's own feeds receive no updates (stale). If the creator's feeds are consumed by pools via `PriceProvider._getBidAndAskPrice()` or `AnchoredPriceProvider._readLeg()`, those pools execute swaps against corrupted oracle prices — a bad-price execution impact that satisfies the allowed impact gate.

### Likelihood Explanation
The creator is a semi-trusted party who retains the original signature after delegation. Replaying it requires no special privilege — `allowPushers` is permissionless. The pusher may not monitor `PusherAuthorized` events and may be unaware that their revocation has been silently undone. The window of exposure equals the remaining time until the deadline the pusher originally agreed to, which could be days or weeks. Likelihood is Medium: it requires the creator to act adversarially and the pusher to continue pushing data after revoking.

### Recommendation
Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` bitmap. Increment or mark the nonce/hash on every successful `allowPushers` call, and include the nonce in the signed message. A revoked pusher can then only be re-delegated with a **new** signature (a new nonce), which requires fresh consent from the pusher — not a replay of the old one.

Alternatively, bind the signature to a monotonic per-pusher delegation counter stored on-chain, so that `revokePusher()` increments the counter and invalidates all prior signatures.

### Proof of Concept
```
1. Creator calls allowPushers(deadline=T+7days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

2. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓

3. Creator calls allowPushers(deadline=T+7days, [pusher], [sig])  ← SAME sig, SAME deadline
   → _ensureDeadline passes (deadline not expired)
   → ECDSA.recover succeeds (sig is still valid)
   → namespaceRemapping[pusher] = creator  ← revocation silently undone

4. Pusher pushes data intending to update their own namespace (slot 0, pos 0):
   fallback: creator = namespaceRemapping[pusher] = creator (not address(0))
   → push lands in creator's namespace, not pusher's

5. Creator's feeds (consumed by registered pools) now carry the pusher's
   unintended data; pools swap against a corrupted oracle price.
```
