### Title
Pusher delegation consent signature is replayable within the deadline window, allowing a creator to re-establish a revoked delegation — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracle.sol` verifies a pusher's EIP-191 consent signature and sets `namespaceRemapping[pusher] = creator`. The deadline is the only replay guard. However, the deadline only prevents replay **after** it expires. Within the deadline window, a creator can call `allowPushers` again with the **identical signature** to silently re-establish a delegation that the pusher explicitly revoked via `revokePusher()`. There is no nonce, no used-signature bitmap, and no on-chain record that a given `(pusher, creator, deadline)` tuple has already been consumed.

---

### Finding Description

The `allowPushers` function constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

After verifying the signature, it unconditionally writes `namespaceRemapping[pusher] = msg.sender` with no record that this signature was consumed. [2](#0-1) 

`revokePusher()` clears the mapping to `address(0)`: [3](#0-2) 

But because the original signature is still cryptographically valid (the deadline has not passed), the creator can immediately call `allowPushers` again with the same `deadline` and `signature` bytes, writing `namespaceRemapping[pusher] = creator` again. The pusher's revocation is silently undone.

The code's own NatSpec comment acknowledges the deadline is the intended guard against this exact scenario:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The comment correctly identifies the threat but the deadline alone does not close it — it only closes it after expiry. Within the window `[now, deadline]`, the signature is fully replayable.

The `_ensureDeadline` check in `OracleBase` only enforces `block.timestamp <= deadline`: [5](#0-4) 

No nonce, no used-hash set, and no other state prevents the same `(deadline, pusher, creator, sig)` tuple from being submitted multiple times.

---

### Impact Explanation

When a pusher is re-delegated without their knowledge, their subsequent `fallback` pushes are silently routed into the creator's namespace:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [6](#0-5) 

A pusher who revoked because their key was compromised, or because they no longer trust the creator, will continue writing price data into the creator's namespace. Any `AnchoredPriceProvider` or pool consuming that namespace feed will receive prices attributed to a pusher who believed they had stopped contributing. This is a **bad-price execution** path: stale or adversarially-chosen prices reach pool swaps through a feed the pusher thought they had detached from.

---

### Likelihood Explanation

The trigger requires a creator who is willing to act adversarially and who retained the original signature bytes (trivially available from the original `allowPushers` calldata on-chain). The pusher must continue pushing after revoking — a realistic assumption since off-chain push bots typically run continuously and do not check `namespaceRemapping` before each push. The deadline window is creator-chosen and can be set to days or weeks, giving a large replay window.

---

### Recommendation

Mark each `(pusher, creator, deadline)` consent as consumed on first use. The simplest approach is a `mapping(bytes32 => bool) private _usedConsents` keyed on the full digest:

```solidity
mapping(bytes32 => bool) private _usedConsents;

// inside allowPushers, after recovering the signer:
require(!_usedConsents[hash], "consent already used");
_usedConsents[hash] = true;
```

Alternatively, introduce a per-pusher nonce that the pusher signs and the contract increments on each successful delegation, making every consent one-time by construction.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — still valid, deadline not expired
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);           // succeeds — no revert
assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation silently restored

// 5. Pusher's next push (e.g. from an automated bot) lands in creator's namespace
uint56 tsMs = uint56(block.timestamp * 1000);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, _packRaw(9_000_000, 4, 2), tsMs));
assertTrue(ok);
// Creator's feed now contains the pusher's data — pusher believed they had detached
assertGt(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, 0);
```
