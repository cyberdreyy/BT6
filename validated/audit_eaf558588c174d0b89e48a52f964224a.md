### Title
Permissionless `register()` Unconditionally Clears Admin Blacklist, Allowing Any Actor to Reinstate a Banned Pool for 1 Wei - (File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol)

---

### Summary

`OracleBase.register()` is a permissionless, payable function whose stated purpose is to whitelist a pool for a specific `feedId`. However, it contains an unconditional side-effect: it clears `blacklisted[pool]` for any pool that passes the factory/isPool check. Because the default `registrationFee` is **1 wei**, any actor can pay 1 wei to reinstate a pool that the oracle admin explicitly blacklisted, completely nullifying the admin's abuse-protection action.

---

### Finding Description

The oracle admin's only on-chain tool to stop an abusive pool from reading prices is `setBlacklist(pool, true)` (ADMIN_ROLE gated). The `price(feedId, pool)` read path enforces this:

```solidity
// OracleBase.sol (providers) – price()
require(!blacklisted[pool], Blacklisted(pool));
```

However, `register()` — which is **fully permissionless** — contains:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);          // default: 1 wei
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;                       // ← unconditional blacklist clear
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    ...
}
``` [1](#0-0) 

The only preconditions are:
1. `factory` is in `approvedFactories` (a trusted, already-deployed factory — not a malicious setup assumption).
2. `factory.isPool(pool)` returns true (the pool is a legitimate, already-deployed pool from that factory).
3. `msg.value >= 1 wei`.

All three conditions are trivially satisfiable by the pool operator or any third party. The admin's `setBlacklist` action is therefore not a durable security control — it can be reversed by anyone at negligible cost.

The analog to the external report is exact: just as `Timelock.executeTransaction` was callable by anyone and bypassed the Governor's atomic-execution guarantee, `register()` is callable by anyone and bypasses the oracle admin's blacklist guarantee. In both cases a privileged protective action (Governor atomic execution / admin blacklist) is silently undone by an unprivileged external call.

---

### Impact Explanation

The blacklist is the oracle's last-resort abuse-protection gate. Once a pool is blacklisted, it cannot call `price(feedId, pool)` through any provider (Pyth or Chainlink), so its swaps stall. If the blacklist can be cleared for 1 wei:

- A pool blacklisted for price-read abuse (e.g., sandwich-attack patterns, excessive reads, or a compromised pool contract) is immediately reinstated.
- The pool resumes reading oracle prices and executing swaps, potentially causing direct loss of user principal through manipulated bid/ask quotes reaching the pool.
- The admin has no recourse other than re-blacklisting, which the attacker can undo again in the same block for 1 wei.

This is an **admin-boundary break**: an oracle-role protection is bypassed by an unprivileged path, matching the allowed impact gate.

---

### Likelihood Explanation

**High.** The registration fee defaults to 1 wei. The only inputs required (`feedId`, `pool`, an approved `factory`) are all public on-chain. The pool operator whose pool was blacklisted has a direct financial incentive to call `register()`. No special role, signature, or off-chain coordination is needed.

---

### Recommendation

Separate blacklist management from pool registration. Do not clear `blacklisted[pool]` inside `register()`. If un-blacklisting is a desired side-effect of re-registration, gate it on `ADMIN_ROLE` or require an explicit admin co-signature:

```solidity
// Remove the blacklist-clearing side-effect entirely:
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);
    require(pool != address(0));
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);
    // ← do NOT touch blacklisted[pool] here
    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

Un-blacklisting should remain exclusively in `setBlacklist(account, false)` which is already gated on `ADMIN_ROLE`. [2](#0-1) 

---

### Proof of Concept

1. Oracle admin calls `setBlacklist(poolX, true)` — `poolX` is an abusive pool from an approved factory.
2. `poolX` can no longer call `price(feedId, poolX)` — swaps revert with `Blacklisted`.
3. Attacker (or pool operator) calls:
   ```solidity
   oracleBase.register{value: 1 wei}(feedId, poolX, approvedFactory);
   ```
   - `approvedFactories.contains(approvedFactory)` → true (already approved).
   - `IPoolFactory(approvedFactory).isPool(poolX)` → true (legitimate pool).
   - `blacklisted[poolX]` is `true` → branch executes: `blacklisted[poolX] = false`.
4. `poolX` is now un-blacklisted and registered for `feedId`.
5. `poolX` resumes calling `price(feedId, poolX)` through its price provider, reading oracle prices and executing swaps — the admin's protective action is fully nullified. [3](#0-2) [4](#0-3)
