### Title
Pusher Revocation Bypassed by Signature Replay Within Deadline Window — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` includes no nonce in the signed consent. A creator who holds a pusher's previously issued signature can call `allowPushers` again after the pusher has self-revoked, silently re-establishing delegation for as long as the original deadline has not expired. The pusher's `revokePusher()` call is therefore not final: the creator can unilaterally undo it, keeping the pusher's future slot writes landing in the creator's namespace without the pusher's current knowledge or consent.

### Finding Description

`allowPushers` requires the pusher to supply an EIP-191 signature over `(block.chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

There is no nonce, no used-signature registry, and no check that `namespaceRemapping[pusher]` is currently `address(0)` before writing. The only replay guard is the deadline: [2](#0-1) 

`revokePusher` clears the mapping: [3](#0-2) 

Because the same signature remains cryptographically valid until the deadline expires, the creator can immediately call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, writing `namespaceRemapping[pusher] = creator` a second time. The revocation is silently undone.

The code comment in `allowPushers` explicitly acknowledges the concern — *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* — and states the deadline was added as the fix: [4](#0-3) 

The deadline prevents replay only after it expires. Within the window it provides no protection at all.

The `fallback` push path resolves the namespace from `namespaceRemapping[msg.sender]` at call time: [5](#0-4) 

So any push the pusher makes after believing they have revoked lands in the creator's namespace, not their own.

### Impact Explanation

A pusher that revokes intending to stop feeding the creator's namespace (e.g., because it detected a compromise, is shutting down, or wants to push only into its own namespace) continues to overwrite the creator's slots. Those slots are the raw source for `getOracleData` / `price`, which `AnchoredPriceProvider._readLeg` and `PriceProvider._getBidAndAskPrice` consume at swap time: [6](#0-5) [7](#0-6) 

If the pusher's post-revocation data is stale, incorrect, or from a compromised key, those bad prices reach live pool swaps. The staleness guard (`_isStale`) and price guard are the only downstream defences; they do not detect that the pusher's consent was revoked.

### Likelihood Explanation

- The creator retains the pusher's original signed bytes off-chain (they needed them to call `allowPushers` the first time).
- Re-calling `allowPushers` costs one transaction and requires no new signature.
- Deadlines are typically set days or weeks in the future (a short deadline defeats the purpose of delegation).
- A pusher bot that revokes but keeps running (e.g., pushing into what it believes is its own namespace) is a realistic operational pattern.

### Recommendation

Add a per-pusher nonce or a used-signature bitmap so each consent can be consumed at most once:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on each allowPushers or revoke

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusherNonce[pusher],   // <-- add nonce
        pusher, msg.sender
    ))
);
pusherNonce[pusher]++;
```

Alternatively, increment the nonce inside `revokePusher` so that any previously issued signature is immediately invalidated on revocation.

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 30-day deadline
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = pusher.sign(keccak256(abi.encode(
    block.chainid, address(oracle), deadline, pusher, creator
)));

// 2. Creator delegates pusher
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator);

// 3. Pusher self-revokes — believes delegation is gone
vm.prank(pusher);
oracle.revokePusher();
assert(oracle.namespaceRemapping(pusher) == address(0));

// 4. Creator replays the SAME signature — no new consent needed
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator); // revocation undone

// 5. Pusher pushes (thinking it writes to its own namespace)
vm.prank(pusher);
(bool ok,) = address(oracle).call(slotWord); // lands in CREATOR namespace
// Creator's pool now reads the pusher's data through AnchoredPriceProvider

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
