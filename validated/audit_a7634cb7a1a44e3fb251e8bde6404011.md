### Title
Pool admin can unfairly increase swap fees without a timelock to front-run users — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFees` allows the pool admin to raise `adminSpreadFeeE6` and `adminNotionalFeeE8` to their configured caps at any time with no delay. A malicious pool admin can front-run pending swaps by spiking fees to the maximum, then resetting them, extracting more revenue from users than they expected — a direct structural analog of the Napier `setFeeParameter` vulnerability.

---

### Finding Description

`setPoolAdminFees` is gated only by `onlyPoolAdmin(pool)` and a cap check; there is no timelock, no minimum delay, and no event-based warning window before the new rates take effect on the pool. [1](#0-0) 

The function first calls `collectFees` at the **old** rates, then immediately writes the new (higher) rates and calls `setPoolFees` on the pool: [2](#0-1) 

The pre-collection step only crystallises accrued fees at the old rate; it does **not** prevent the admin from then setting a higher rate that applies to the very next swap. Because `setPoolFees` is called in the same transaction, the pool's stored `spreadFeeE6` / `notionalFeeE8` are updated atomically and are visible to the next swap in the same block.

The same pattern exists for the factory owner's `setPoolProtocolFee`, which also has no timelock: [3](#0-2) 

The per-bin variant `setPoolBinAdditionalFees` (also pool-admin-only, no timelock) compounds the risk: a pool admin can target a specific bin that a pending swap will traverse and spike only that bin's fee: [4](#0-3) 

The interface confirms no timelock is specified for any of these fee-change paths: [5](#0-4) 

Compare with the **price-provider rotation**, which explicitly enforces a timelock (`priceProviderTimelock`) before the new oracle takes effect — demonstrating the protocol already understands the need for delays on sensitive parameter changes, but did not apply the same discipline to fee changes: [6](#0-5) 

---

### Impact Explanation

A malicious pool admin can:

1. Advertise low fees (`adminSpreadFeeE6 = 0`) to attract swap volume.
2. Observe a pending swap in the mempool and front-run it with `setPoolAdminFees(pool, maxAdminSpreadFeeE6, maxAdminNotionalFeeE8)`.
3. The victim's swap executes at the maximum admin fee rate.
4. The admin immediately calls `setPoolAdminFees` back to low values.

Users suffer a direct, unannounced loss of principal on every front-run swap. The admin collects the excess fee delta as revenue. This matches the **"Critical/High/Medium direct loss of user principal"** impact gate.

---

### Likelihood Explanation

**Medium.** The pool admin is a semi-trusted role (docs recommend a multisig but do not enforce it). Any EOA or contract holding `poolAdmin[pool]` can execute this attack permissionlessly. The attack requires no special on-chain conditions — only mempool visibility of a pending swap, which is trivially available on any public EVM chain.

---

### Recommendation

Apply a mandatory timelock to `setPoolAdminFees` and `setPoolBinAdditionalFees`, mirroring the existing `priceProviderTimelock` pattern already used for oracle rotation:

```solidity
// Proposed: two-step fee update
function proposePoolAdminFees(address pool, uint24 newSpread, uint24 newNotional) external onlyPoolAdmin(pool) {
    pendingAdminFees[pool] = PendingFee(newSpread, newNotional, block.timestamp + FEE_TIMELOCK);
}

function executePoolAdminFees(address pool) external onlyPoolAdmin(pool) {
    PendingFee memory p = pendingAdminFees[pool];
    require(block.timestamp >= p.executeAfter, TimelockNotElapsed());
    // apply fees
}
```

Alternatively, cap the magnitude of a single fee change per time window (rate-limiting).

---

### Proof of Concept

```solidity
function test_poolAdmin_frontRuns_swap_with_higher_fees() public {
    // Setup: pool admin sets low fees to attract users
    vm.prank(poolAdmin);
    factory.setPoolAdminFees(pool, 0, 0); // 0% admin fee

    // Alice prepares a swap
    uint256 amountIn = 1000e18;
    // ... Alice's tx is in the mempool ...

    // Pool admin sees Alice's swap and front-runs it
    vm.prank(poolAdmin);
    factory.setPoolAdminFees(pool, maxAdminSpreadFeeE6, maxAdminNotionalFeeE8); // max fees

    // Alice's swap executes at max fees — she receives less than expected
    vm.prank(alice);
    uint256 amountOut = router.swap(pool, amountIn, minAmountOut, ...);

    // Pool admin resets fees to attract more users
    vm.prank(poolAdmin);
    factory.setPoolAdminFees(pool, 0, 0);

    // Admin collects the inflated fee revenue
    factory.collectPoolFees(pool);
}
```
