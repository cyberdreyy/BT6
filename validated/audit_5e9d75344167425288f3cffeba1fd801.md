### Title
Pusher Consent Signature Is Not Invalidated on Revocation, Allowing Creator to Replay It and Restore Delegation Against the Pusher's Will — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()`, the creator can immediately call `allowPushers` again with the exact same signature and deadline, atomically restoring `namespaceRemapping[pusher] = creator`. The pusher has no on-chain mechanism to permanently cancel a still-valid consent signature before its deadline expires. A compromised pusher key therefore retains write authority over the creator's namespace for the full deadline window, and every fallback push from that key lands in the creator's namespace and is later consumed by price providers and pools.

### Finding Description

The `allowPushers` function constructs a hash over `(block.chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's signature against it.

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The code comment explicitly acknowledges the risk:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

The deadline check (`_ensureDeadline`) only rejects signatures whose deadline has already passed. It does **not** prevent the same signature from being submitted a second time within the deadline window. There is no `usedSignatures` bitmap, nonce, or any other consumed-signature tracking anywhere in the contract.

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But the creator can immediately call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, and the mapping is restored. The pusher's revocation is silently undone in the same block.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push from the pusher's address after the re-established delegation continues to overwrite the creator's namespace slots with attacker-controlled prices.

### Impact Explanation

A compromised pusher key that has been self-revoked can be forced back into the creator's namespace by the creator replaying the original consent. The attacker holding the compromised key then pushes arbitrary `U64x32`-encoded prices and codebook spread indices into the creator's namespace. `getOracleData` / `price` return these values directly to `AnchoredPriceProvider` and any pool that has registered against this oracle feed. The result is bad-price execution: swappers receive more output than the true oracle price permits, or the pool receives less input than owed, causing direct loss of LP principal and protocol fees.

### Likelihood Explanation

Pusher keys are long-lived off-chain signing keys. Consent signatures are typically issued with multi-day or multi-week deadlines (the test suite uses `block.timestamp + 1 days`). Key compromise is a realistic operational event. A creator who is unaware of the compromise, or who is racing to keep their feed live, will naturally replay the cached signature. The replay requires no special privilege — only the creator's EOA and the original calldata, which is publicly visible on-chain from the first `allowPushers` transaction.

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedConsentHashes` and set the entry to `true` inside `allowPushers` before writing `namespaceRemapping`. Reject any signature whose hash is already present:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedConsentHashes[hash], "consent already consumed");
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedConsentHashes[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

This ensures each signed consent can establish delegation exactly once. A pusher who wants to revoke and prevent re-establishment simply revokes; the creator must obtain a fresh signature with a new deadline.

### Proof of Concept

1. Pusher signs consent: `sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+30days, pusher, creator)))`.
2. Creator calls `allowPushers(T+30days, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher's key is compromised. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator (or attacker watching mempool) calls `allowPushers(T+30days, [pusher], [sig])` with the identical arguments → `namespaceRemapping[pusher] = creator` again. No revert.
5. Attacker (holding the compromised pusher key) calls `oracle.fallback()` with a crafted 32-byte slot word encoding an extreme price (e.g., `U64x32`-encoded `9_000_000` with sentinel spread indices).
6. The write lands in the creator's namespace because `namespaceRemapping[attacker_key] == creator`.
7. `getOracleData(feedIdOf(creator, slotId, positionIndex))` returns the attacker's price.
8. `AnchoredPriceProvider.getBidAndAskPrice` consumes this value and passes it to `MetricOmmPool.swap`.
9. Swappers execute against the manipulated bid/ask, draining LP reserves beyond what the true oracle price permits.
