### Title
Pusher Delegation Consent Signature Lacks Nonce, Enabling Replay to Override Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` verifies an EIP-191 consent signature whose payload is `(chainid, oracle_address, deadline, pusher, creator)`. There is no nonce. Once a pusher's consent signature has been submitted on-chain, the creator can replay it any number of times before the deadline expires — including after the pusher has called `revokePusher()` — silently re-establishing the delegation the pusher intended to terminate.

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The signed tuple `(chainid, oracle, deadline, pusher, creator)` is identical for every invocation of `allowPushers` with the same arguments. There is no per-delegation nonce, no consumed-flag, and no on-chain record that the signature has already been used.

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

But nothing prevents the creator from immediately calling `allowPushers` again with the same `deadline`, `pusher`, and original signature bytes — all of which are permanently visible in the original transaction's calldata — to write `namespaceRemapping[pusher] = creator` again.

The code comment itself acknowledges the partial risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The comment treats the deadline as the complete fix, but the deadline only bounds the outer replay window; it does not prevent replay within that window. A deadline set one year in the future leaves a one-year window during which the creator can replay the signature after every revocation.

### Impact Explanation

Every fallback push from the pusher resolves its namespace through `namespaceRemapping`:

```
fallback push → namespace resolution (namespaceRemapping[msg.sender]) → slot write in creator namespace
```

If the delegation is forcibly re-established after the pusher revokes:

1. The pusher's subsequent pushes continue to land in the **creator's namespace**, not the pusher's own. Any pool consuming the creator's compressed feed continues to receive the pusher's data even though the pusher intended to stop contributing.
2. If the pusher simultaneously tries to operate their own namespace (e.g., for their own pool), their pushes are silently swallowed by the creator's namespace. The pusher's own pool receives **no price data**, breaking its swap path.
3. The creator's pool may receive **stale or mismatched price data** if the pusher's intent was to stop updating that feed — the pusher may reduce push frequency or push data calibrated for a different pool, causing bad-price execution in the creator's pool.

The pusher's only effective escape is to stop pushing entirely, which itself breaks any pool that legitimately depends on their data.

### Likelihood Explanation

- The original consent signature is permanently on-chain in the `allowPushers` calldata of the first delegation transaction. Any party who can read chain history (including the creator) can extract and replay it.
- The creator is a semi-trusted but unprivileged actor (not a protocol admin). The replay requires no special access — only the ability to call a public function with already-public arguments.
- Deadlines are typically set days to months in the future, giving a long replay window.
- The pusher has no on-chain mechanism to invalidate the old signature short of waiting for the deadline to expire.

### Recommendation

Add a per-pusher nonce to the signed payload and increment it on every successful `allowPushers` call (and optionally on `revokePusher`):

```solidity
mapping(address => uint256) public pusherNonces;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonces[pusher]++
    ))
);
```

This mirrors the fix applied to `SpokePoolPeriphery` (adding `permitNonces` and calling `_validateAndIncrementNonce` before signature verification). After revocation, the nonce is stale and the old signature is permanently invalid.

### Proof of Concept

```
1. Pusher signs consent: hash = keccak256(chainid, oracle, deadline=T+365d, pusher, creator)
   → sig = ECDSA.sign(pusherKey, hash)

2. Creator calls allowPushers(T+365d, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes delegation is gone)

4. Creator extracts sig from step-2 calldata (public on-chain).
   Creator calls allowPushers(T+365d, [pusher], [sig])  ← same args, same sig
   → _ensureDeadline passes (T+365d > now)
   → ECDSA.recover returns pusher  ← same hash, same sig
   → namespaceRemapping[pusher] = creator  ✓ (delegation silently restored)

5. Pusher's next fallback push lands in creator namespace, not pusher's own.
   Pusher's own pool receives no price data → swap path broken.
   Creator's pool receives pusher's data even though pusher revoked consent.
``` [4](#0-3)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
