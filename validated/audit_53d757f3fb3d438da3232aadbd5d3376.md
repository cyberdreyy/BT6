### Title
Pusher consent signature in `allowPushers` carries no nonce, allowing a creator to silently re-establish a delegation the pusher has revoked — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but does not invalidate it after first use. A creator who holds a pusher's valid (not-yet-expired) consent signature can call `allowPushers` repeatedly to re-establish `namespaceRemapping[pusher] = creator` every time the pusher calls `revokePusher()`. Because the `fallback()` push path resolves the target namespace from `namespaceRemapping[msg.sender]` at push time, the pusher's subsequent price updates are silently redirected into the creator's namespace against the pusher's will, potentially feeding wrong prices into pools that read from that namespace.

### Finding Description

`allowPushers` signs over `(chainid, address(this), deadline, pusher, msg.sender)`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;   // re-established unconditionally
``` [1](#0-0) 

There is no nonce, no used-signature bitmap, and no check that the pusher is not already revoked. The only replay barrier is the `deadline` field, but the deadline is chosen by the creator and can be set far in the future. The code comment itself acknowledges the deadline is needed to prevent re-establishment *after* the pusher revoked it, yet the deadline only limits the window — it does not prevent replay *within* the window:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But the creator can immediately replay the original signature to restore the mapping. The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after revocation — intending to write into their own namespace — lands in the creator's namespace instead, as long as the creator keeps replaying the signature.

### Impact Explanation

The creator's pool reads prices from `feedIdOf(creator, slotIndex, positionIndex)`. If the pusher revokes because they are now pushing prices for a different asset or market (into their own namespace), and the creator replays the consent signature, the pusher's new-asset prices are written into the creator's namespace. The pool's `getBidAndAskPrice()` → `PriceProvider._getBidAndAskPrice()` → `oracle.price(offchainFeedId, pool)` path then returns the wrong asset's price to the pool: [5](#0-4) 

Swaps execute at the wrong bid/ask, causing traders to receive more tokens than the oracle curve permits or the pool to receive fewer input tokens than owed — direct loss of user principal or LP assets. The staleness check does not help because the pusher is actively pushing fresh data; only the namespace destination is wrong.

### Likelihood Explanation

The trigger is unprivileged: any creator who holds a pusher's consent signature (which they received legitimately during the original `allowPushers` call) can replay it. The pusher has no on-chain mechanism to invalidate the signature before the deadline expires. The pusher must monitor `PusherAuthorized` events continuously to detect silent re-establishment. The window is bounded only by the deadline the creator chose when requesting the signature — which can be days or weeks in the future.

### Recommendation

Track a per-pusher nonce in the consent signature and increment it on every successful `allowPushers` or `revokePusher` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);
pusherNonce[pusher]++;        // invalidate after use

// In revokePusher:
pusherNonce[msg.sender]++;    // invalidate any outstanding consent
namespaceRemapping[msg.sender] = address(0);
```

This ensures each consent signature is single-use and that revocation permanently invalidates all previously issued signatures.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline = now + 30 days.
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.

// 3. Pusher revokes — intends to push into own namespace.
vm.prank(pusher);
oracle.revokePusher();

// 4. Creator replays the SAME signature — no revert, delegation restored.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // pusher's revocation undone

// 5. Pusher pushes BTC/USD price thinking it goes to their own namespace.
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(slotId, pos, btcUsdRaw, tsMs));
assertTrue(ok);

// 6. Creator's ETH/USD pool reads the BTC/USD price from creator's namespace.
IOffchainOracle.OracleData memory data =
    oracle.getOracleData(oracle.feedIdOf(creator, slotId, pos));
// data.price == BTC/USD price — wrong asset, pool executes swaps at wrong rate.
