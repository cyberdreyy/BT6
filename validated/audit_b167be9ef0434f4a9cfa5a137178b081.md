### Title
Creator Can Replay Pusher Consent Signature to Override `revokePusher()` Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()` to clear their delegation, the original consent signature remains cryptographically valid until its deadline. The creator can immediately replay that same signature to re-establish the delegation, rendering the pusher's self-revocation permanently ineffective for the lifetime of the deadline.

---

### Finding Description

`allowPushers` builds a hash over `(chainid, oracle_address, deadline, pusher, creator)` and recovers the signer, but stores no record that the signature was used: [1](#0-0) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [2](#0-1) 

Because no used-signature bitmap or per-pusher nonce exists, the original consent bytes are still a valid input to `allowPushers` at any point before `deadline`. The creator (who already holds the signature from the initial delegation call) can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments and the `require(pusher == ECDSA.recover(...))` check passes, writing `namespaceRemapping[pusher] = msg.sender` again.

The code comment explicitly acknowledges the deadline is the only replay guard: [3](#0-2) 

But the deadline only prevents replay *after* expiry. Within the deadline window, the signature is unconditionally reusable.

The analog to the `prepareForMigration` bug is exact: just as that function lacked a `used[salt]` check allowing signatures to be replayed across functions, `allowPushers` lacks a `usedSignature[hash]` check allowing the same consent to re-establish a delegation the pusher has explicitly revoked.

---

### Impact Explanation

A pusher whose private key is compromised calls `revokePusher()` to stop the attacker from writing bad prices into the creator's namespace. The creator (or the attacker if they also control the creator key) replays the original consent signature via `allowPushers`. The delegation is immediately restored. The compromised pusher key continues to push arbitrary `(price, spread0, spread1, timestampMs)` tuples into the creator's slot storage via the `fallback` push path: [4](#0-3) 

Those values are consumed verbatim by `getOracleData` and surfaced through `price()` to any pool or `AnchoredPriceProvider` reading the feed. The pusher's self-revocation is the only unilateral escape hatch available to them; if it can be overridden, bad prices reach pools for the full remaining deadline duration.

---

### Likelihood Explanation

Medium. The creator already holds the consent signature (they submitted it in the original `allowPushers` call). Replaying it requires a single additional transaction. The scenario is realistic whenever: (a) a pusher key is compromised and the creator is either unaware or malicious, or (b) a creator deliberately wants to retain a pusher who has tried to exit. Deadlines are expected to be long-lived (the comment notes undated signatures are the risk being mitigated, implying deadlines of days to months are normal).

---

### Recommendation

Track each consent signature as consumed immediately after first use:

```solidity
mapping(bytes32 => bool) public usedConsentHash;

function allowPushers(...) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(!usedConsentHash[hash], "signature already used");
    require(pusher == ECDSA.recover(hash, signatures[i]));
    usedConsentHash[hash] = true;
    namespaceRemapping[pusher] = msg.sender;
    ...
}
```

Alternatively, maintain a per-pusher revocation nonce that is incremented by `revokePusher` and embedded in the signed message, so any prior signature is invalidated on revocation.

### Proof of Concept

```solidity

oracle.allowPushers(deadline, _arr(pusher), _arr(sig));

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
