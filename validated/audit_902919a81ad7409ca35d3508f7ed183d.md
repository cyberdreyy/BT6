### Title
`allowPushers` Delegation Signature Lacks Nonce — Creator Can Replay Pusher Consent to Nullify `revokePusher` and Sustain Bad-Price Injection - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowPushers` signs pusher consent over `(chainid, oracle, deadline, pusher, creator)` with no nonce. Within the deadline window the creator can replay the identical signature after the pusher has called `revokePusher()`, silently re-establishing the delegation. If the pusher's key is compromised, the pusher's only on-chain remedy is revocation; the creator's replay nullifies it, allowing the attacker to continue pushing arbitrary prices into the creator's namespace and from there into live pool swaps.

---

### Finding Description

`allowPushers` in `CompressedOracleV1` verifies an EIP-191 signature whose preimage is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no used-signature bitmap, and no state that changes on each call. The only replay barrier is the deadline. `_ensureDeadline` only checks `block.timestamp <= deadline`: [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

Because `allowPushers` writes `namespaceRemapping[pusher] = msg.sender` unconditionally on every successful call, the creator can immediately call `allowPushers` again with the **same** signature and the **same** deadline to restore the mapping. The protocol's own comment acknowledges the concern ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but the deadline alone does not close the window — within the deadline the signature is permanently replayable. [4](#0-3) 

The fallback push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So once the mapping is restored, every subsequent fallback push from the compromised pusher key lands in the creator's namespace again.

The `CompressedOracle.price()` is a permissionless `view` — it ignores the `pool` parameter and applies no in-swap binding: [6](#0-5) 

Price providers (`PriceProvider`, `ProtectedPriceProvider`, `AnchoredPriceProvider`) read through this path. A fresh timestamp on the pushed slot passes the staleness check in every provider variant: [7](#0-6) 

---

### Impact Explanation

An attacker who has obtained a pusher's private key can push arbitrary prices into the creator's namespace. The pusher's only on-chain remedy — `revokePusher()` — is immediately undone by the creator replaying the original consent signature. The attacker's pushes continue to land in the creator's namespace, are decoded by `getOracleData`, forwarded through the price provider, and consumed by pool swaps. Pools execute at the attacker-controlled bid/ask, causing direct loss of user principal (swap conservation failure).

---

### Likelihood Explanation

- Pusher key compromise is a realistic operational risk for any off-chain price-feed operator.
- The creator may replay the signature in good faith (not knowing the key is compromised) or maliciously.
- The deadline is typically set days or weeks in the future (the test uses `block.timestamp + 1 days`), giving a wide replay window.
- No special privilege is required: `allowPushers` is permissionless for any creator. [8](#0-7) 

---

### Recommendation

Add a per-pusher nonce or a used-signature set so each consent can only be consumed once:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on each allowPushers call

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusherNonce[pusher]++,   // <-- nonce
        pusher, msg.sender
    ))
);
```

Alternatively, after `revokePusher()` increments the pusher's nonce, any previously signed consent becomes invalid even if the deadline has not expired.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
