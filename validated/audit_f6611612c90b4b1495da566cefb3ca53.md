### Title
`create`-based deployment in oracle provider factories enables reorg-hijack of pool price provider — (`File: smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`, `AnchoredProviderFactory.sol`)

---

### Summary

All three oracle provider factories deploy new provider contracts using the plain `new` opcode (`create`), whose address is determined solely by the factory's address and its current nonce. On chains where Metric OMM is deployed (Base, HyperEVM), block reorgs can occur. A reorg allows an attacker to race the factory's nonce and deploy a malicious provider at the same address a victim expected their legitimate provider to occupy. If the victim's pool was created with `immutablePriceProvider = true` pointing to that address, the pool is permanently bound to the attacker's oracle, which can return arbitrary prices and drain LP funds through bad-price swaps.

---

### Finding Description

`PriceProviderFactory.createPriceProvider()` deploys a `PriceProvider` with a plain `new`:

```solidity
// PriceProviderFactory.sol L49
PriceProvider p = new PriceProvider(
    address(this), _oracle, _feedId, _marginStep, _maxTimeDelta, _baseToken, _quoteToken
);
```

`PriceProviderFactoryL2.createPriceProvider()` does the same:

```solidity
// PriceProviderFactoryL2.sol L50
PriceProviderL2 p = new PriceProviderL2(
    address(this), _oracle, _feedId, _marginStep, _maxTimeDelta, _futureTolerance, _baseToken, _quoteToken
);
```

`AnchoredProviderFactory.createAnchoredProvider()` also uses plain `new`:

```solidity
// AnchoredProviderFactory.sol L182
AnchoredPriceProvider p = new AnchoredPriceProvider(
    address(this), oracle, baseFeedId, quoteFeedId, ...
);
```

By contrast, `MetricOmmPoolDeployer.deploy()` correctly uses `create2`:

```solidity
// MetricOmmPoolDeployer.sol L62
pool = address(new MetricOmmPool{salt: params.salt}(...));
```

The provider factories are fully permissionless — any caller may supply any `_oracle` address to `PriceProviderFactory.createPriceProvider()`. There is no oracle allow-list in `PriceProviderFactory` or `PriceProviderFactoryL2`.

**Attack sequence on Base (reorg scenario):**

1. Victim calls `PriceProviderFactory.createPriceProvider(legitimateOracle, ...)` in block N. Factory nonce is `k`. Provider lands at address `X = keccak256(rlp(factory, k))[12:]`.
2. Victim immediately creates a pool with `priceProvider = X` and `immutablePriceProvider = true` (also in block N or N+1).
3. A reorg drops block N (and N+1). Factory nonce resets to `k`.
4. Attacker calls `PriceProviderFactory.createPriceProvider(maliciousOracle, ...)` — factory nonce is still `k`, so the new provider lands at the same address `X`.
5. Victim's pool-creation transaction is re-included from the mempool. The pool is now permanently bound to the attacker's provider at `X`.
6. Attacker's `maliciousOracle` returns arbitrary bid/ask prices. Every swap through the pool executes at attacker-controlled prices, draining LP principal.

The `immutablePriceProvider = true` flag makes the compromise permanent — the pool admin cannot update the price provider after deployment.

---

### Impact Explanation

A pool permanently bound to an attacker-controlled price provider will execute all swaps at arbitrary prices. LPs suffer direct loss of principal: the attacker can set a price that allows them to extract token0 or token1 at far below market value, leaving the pool insolvent relative to LP claims. This satisfies the impact gate criteria of **bad-price execution** and **pool insolvency**.

---

### Likelihood Explanation

Base is an optimistic rollup explicitly listed as a Metric OMM deployment target. Base reorgs, while infrequent, have been observed in practice (e.g., the April 2024 Base reorg). The attack requires:
- A reorg of at least 1 block (observed on Base).
- The victim to create a pool with `immutablePriceProvider = true` in the same block or shortly after provider creation (common scripted deployment pattern).
- The attacker to monitor the mempool and re-submit with a malicious oracle before the victim's tx is re-included.

All three conditions are achievable by a well-resourced attacker with mempool monitoring. The `PriceProviderFactory` is permissionless, so no privileged access is required.

---

### Recommendation

Deploy all provider contracts via `create2` with a deterministic salt that commits to the caller and the constructor parameters:

```solidity
// Example fix for PriceProviderFactory.createPriceProvider()
bytes32 salt = keccak256(abi.encode(msg.sender, _oracle, _feedId, _marginStep, _maxTimeDelta, _baseToken, _quoteToken));
PriceProvider p = new PriceProvider{salt: salt}(
    address(this), _oracle, _feedId, _marginStep, _maxTimeDelta, _baseToken, _quoteToken
);
```

Apply the same pattern to `PriceProviderFactoryL2.createPriceProvider()` and `AnchoredProviderFactory.createAnchoredProvider()`. This makes the deployed address a function of the parameters, not the factory nonce, so a reorg cannot cause a different provider to land at the same address.

---

### Proof of Concept

```
Block N (Base):
  tx1: victim → PriceProviderFactory.createPriceProvider(legitimateOracle, feedId, ...)
       → factory nonce k → PriceProvider deployed at X
  tx2: victim → MetricOmmPoolFactory.createPool(token0, token1, priceProvider=X, immutablePriceProvider=true, ...)
       → pool P created, permanently bound to X

[Base reorg: block N dropped, factory nonce resets to k]

Block N' (attacker wins):
  tx1: attacker → PriceProviderFactory.createPriceProvider(MaliciousOracle, feedId, ...)
       → factory nonce k → MaliciousProvider deployed at X  ← same address!
  tx2: victim's pool-creation tx re-included from mempool
       → pool P created, permanently bound to X = MaliciousProvider

Post-reorg:
  attacker calls MaliciousOracle to return price = 1 wei
  attacker swaps token1 → token0 at 1 wei price
  → pool pays out token0 at near-zero cost → LP principal drained
```
