### Title
Permissionless `register` Unconditionally Clears Admin-Set Blacklist, Bypassing Abuse Protection — (File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol)

---

### Summary

The `register` function in the providers `OracleBase` is permissionless and clears the `blacklisted` flag for any pool as a side effect of re-registration. An admin-set blacklist can be bypassed by any unprivileged caller — including the blacklisted pool itself — by paying the registration fee (default: 1 wei) and supplying an approved factory that recognizes the pool. The admin's abuse-protection control is rendered ineffective.

---

### Finding Description

The `register` function contains the following logic:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
``` [1](#0-0) 

There is no check whether the pool was already registered, and no check whether the blacklist entry was intentionally set by an admin. The only preconditions are:

- `msg.value >= registrationFee` (default: 1 wei)
- `approvedFactories.contains(factory)`
- `IPoolFactory(factory).isPool(pool)` [2](#0-1) 

Since the pool was already registered before being blacklisted, it already satisfies the factory/pool checks. Any caller — including the pool itself — can pay 1 wei to re-register and clear the blacklist.

**Analog to "Duplicate Dividends":** In the external bug, the only precondition for adding an address to the `users` list was `balance == 0`, with no check for existing membership. Here, the only precondition for clearing the blacklist is paying the fee, with no check for whether the blacklist was intentionally set. Just as an address could re-enter the dividend list by minting after transferring, a blacklisted pool re-enters the "authorized reader" state by re-registering.

The `price` function enforces the blacklist at read time:

```solidity
require(!blacklisted[pool], Blacklisted(pool));
require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
``` [3](#0-2) 

But this guard is bypassed the moment `register` clears the flag.

The admin's only other recourse is `setBlacklist`, which is `ADMIN_ROLE`-gated: [4](#0-3) 

This creates an asymmetry: the admin can set the blacklist, but an unprivileged path can clear it immediately.

---

### Impact Explanation

The blacklist is the oracle's primary runtime abuse-protection control. If a pool is blacklisted (e.g., for price manipulation, unauthorized reads, or protocol abuse), the blacklisted pool or any third party can pay 1 wei to re-register and restore full oracle read access. The admin cannot durably enforce the blacklist without continuously re-blacklisting after every re-registration, which is not a viable security posture. A pool that was blacklisted for feeding bad prices into swaps can immediately resume doing so.

**Impact class:** Admin-boundary break — factory/oracle role checks bypassed by an unprivileged path.

---

### Likelihood Explanation

- The `register` function is fully permissionless.
- The default `registrationFee` is 1 wei.
- The pool was already recognized by an approved factory before blacklisting, so all preconditions are trivially satisfied.
- No special role, signature, or off-chain coordination is required. [5](#0-4) 

---

### Recommendation

Separate the blacklist-clearing side effect from the registration path. Two options:

1. **Remove blacklist clearing from `register` entirely.** Blacklist removal should be an explicit admin action only (`setBlacklist(pool, false)`).
2. **Guard the blacklist clear with an admin check.** If clearing on re-registration is intentional, require `ADMIN_ROLE` to authorize it, or revert if the pool is currently blacklisted.

---

### Proof of Concept

```
1. Admin calls setBlacklist(pool, true)
   → blacklisted[pool] = true
   → pool.price(feedId, pool) reverts with Blacklisted(pool)

2. Attacker (or pool itself) calls register(feedId, pool, approvedFactory) with msg.value = 1 wei
   → approvedFactories.contains(approvedFactory) ✓
   → IPoolFactory(approvedFactory).isPool(pool) ✓  (pool was already registered)
   → blacklisted[pool] = false  ← blacklist cleared
   → registeredPool[feedId][pool] = true

3. Pool calls price(feedId, pool) again
   → notBlacklisted: blacklisted[msg.sender] = false ✓
   → !blacklisted[pool] ✓
   → registeredPool[feedId][pool] ✓
   → oracle returns live price — abuse protection defeated
```

    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
    }
```

        require(account != address(0));
        if (blacklisted[account] == value) return;
        blacklisted[account] = value;
        emit BlacklistUpdated(account, value);
    }
