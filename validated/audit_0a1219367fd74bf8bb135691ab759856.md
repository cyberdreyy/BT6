The `allowPushers` function in `CompressedOracleV1` has no guard against overwriting an existing `namespaceRemapping` entry. Let me confirm the exact code path.

### Title
`allowPushers` Allows Unconditional Overwrite of Existing Pusher Delegation, Enabling Silent Feed Hijack — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` writes `namespaceRemapping[pusher] = msg.sender` with **no guard against an already-set mapping**. Any third-party creator C who holds a valid signed consent from pusher B can call `allowPushers` and silently overwrite B's existing delegation from creator A to creator C. After the overwrite, every subsequent push from B lands in C's namespace instead of A's, A's feeds go stale, and A cannot even remove B (because `removePushers` checks `namespaceRemapping[pusher] == msg.sender`, which is now C, not A).

---

### Finding Description

The vulnerable write is unconditional:

```solidity
// CompressedOracle.sol L209
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no `require(namespaceRemapping[pusher] == address(0))` or equivalent before the assignment. The signed consent message binds `(chainid, oracle, deadline, pusher, creator)`:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [2](#0-1) 

The signature proves only that pusher B consented to delegate to `msg.sender` (C). It proves nothing about whether B is already delegated to A. The function accepts any valid, non-expired consent and overwrites whatever was there.

**Attack sequence:**

| Step | Actor | Action | State after |
|------|-------|--------|-------------|
| 1 | B | Signs consent for A, deadline D1 | — |
| 2 | A | `allowPushers(D1, [B], [sig_B→A])` | `namespaceRemapping[B] = A` |
| 3 | B | Signs consent for C, deadline D2 > D1 | — |
| 4 | C | `allowPushers(D2, [B], [sig_B→C])` | `namespaceRemapping[B] = C` ← **overwrite** |
| 5 | B | Pushes slot data | Lands in C's namespace, not A's |

After step 4, A's feeds stop receiving B's updates. A cannot recover via `removePushers` because that function checks `namespaceRemapping[pusher] == msg.sender`:

```solidity
if (namespaceRemapping[pusher] == msg.sender) {
    namespaceRemapping[pusher] = address(0);
    ...
} else {
    revert InvalidManager(msg.sender);   // A hits this
}
``` [3](#0-2) 

B's only recourse is `revokePusher()`, which sets the mapping to `address(0)` (not back to A), permanently breaking A's delegation without A's involvement. [4](#0-3) 

---

### Impact Explanation

- **Feed hijack**: B's pushes silently redirect to C's namespace. C receives oracle data it was never entitled to.
- **Oracle data integrity loss for A**: A's feeds stop updating. Any pool or provider reading A's feeds receives stale price data.
- **Stale price → bad execution**: The `fallback` push path uses monotonicity as the only freshness gate. Once B stops pushing to A's namespace, A's slot timestamps freeze. Consumers (AnchoredPriceProvider, pools) will eventually reject A's feeds as stale, causing swap failures or bad-price execution depending on the consumer's `maxTimeDrift` configuration.
- **A cannot self-heal**: `removePushers` is blocked by the `InvalidManager` revert. A must wait for B to voluntarily self-revoke. [5](#0-4) 

---

### Likelihood Explanation

B signing consent for multiple creators is an expected operational pattern (shared pusher services, key rotation, multi-creator setups). C does not need to compromise B — C only needs to obtain a valid signed consent from B, which B may have issued legitimately for a different purpose. The deadline check only prevents replaying an expired signature; it does not prevent a fresh, valid signature from overwriting an existing delegation.

---

### Recommendation

Add an existence check before writing the remapping:

```solidity
address existing = namespaceRemapping[pusher];
require(existing == address(0), AlreadyDelegated(pusher, existing));
namespaceRemapping[pusher] = msg.sender;
```

This enforces the single-namespace-binding invariant: a pusher can only be delegated to one creator at a time, and re-delegation requires the pusher to first self-revoke via `revokePusher()`.

// Foundry integration test sketch
function testNamespaceHijack() public {
    uint256 D1 = block.timestamp + 1 days;
    uint256 D2 = block.timestamp + 2 days;

    // Step 1-2: A registers B
    bytes memory sigBA = _signConsent(PUSHER_KEY, D1, pusher, creatorA);
    vm.prank(creatorA);
    oracle.allowPushers(D1, _arr(pusher), _sigs(sigBA));
    assertEq(oracle.namespaceRemapping(pusher), creatorA);

    // Step 3-4: C overwrites with B's consent for C
    bytes memory sigBC = _signConsent(PUSHER_KEY, D2, pusher, creatorC);
    vm.prank(creatorC);
    oracle.allowPushers(D2, _arr(pusher), _sigs(sigBC));

    // Invariant violated: B now maps to C, not A
    assertEq(oracle.namespaceRemapping(pusher), creatorC);

    // B's next push lands in C's namespace
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, _packRaw(1_000_000, 3, 3), uint56(block.timestamp * 1000)));
    assertTrue(ok);
    assertGt(oracle.getOracleData(oracle.feedIdOf(creatorC, 0, 0)).price, 0); // C gets data
    assertEq(oracle.getOracleData(oracle.feedIdOf(creatorA, 0, 0)).price, 0);  // A gets nothing

    // A cannot remove B — reverts with InvalidManager
    vm.prank(creatorA);
    vm.expectRevert(abi.encodeWithSelector(ICompressedOracleV1.InvalidManager.selector, creatorA));
    oracle.removePushers(_arr(pusher));
}
