### Title
Permissionless `register()` Irrevocably Clears Admin Blacklist, Preventing Permanent Pool Exclusion from Oracle Price Reads — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary

`OracleBase.register()` is fully permissionless and unconditionally clears `blacklisted[pool]` whenever the caller pays `>= registrationFee`. Because the default fee is `1 wei` and the function accepts payment from **any** address, the admin's only tool to permanently block a pool from reading oracle prices — the blacklist — can be undone by the pool operator (or any third party) for a trivial cost. There is no `unregister()` path; `registeredPool[feedId][pool]` is write-once-true and can never be set back to `false`.

### Finding Description

`OracleBase.sol` implements an abuse-protection layer whose stated security model is:

> *"Misuse → maintainer blacklists the pool → recovery requires paying the fee again."*

The admin calls `setBlacklist(pool, true)` to revoke a pool's access. However, `register()` is unconditionally permissionless:

```solidity
// OracleBase.sol lines 201-213
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);   // default: 1 wei
    require(pool != address(0));
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;          // ← unconditional blacklist clear
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;    // ← write-once, never cleared
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
``` [1](#0-0) 

The `registeredPool` mapping has no inverse operation anywhere in the contract. [2](#0-1) 

The test suite explicitly confirms that a **third party** (not the pool, not the admin) can pay the fee to clear the blacklist and re-enable reads:

```solidity
// test: test_register_thirdParty_redeems_andEnablesRead
oracle.setBlacklist(pool, true);
vm.prank(thirdParty);
oracle.register{value: 1}(FEED, pool, address(factory));
assertFalse(oracle.blacklisted(pool));   // blacklist cleared by stranger
``` [3](#0-2) 

The admin's only alternative mitigation — raising `registrationFee` via `setRegistrationFee()` — is a **global** parameter that raises the cost for every pool, not just the targeted one. It cannot selectively and permanently exclude a single pool. [4](#0-3) 

### Impact Explanation

The abuse-protection layer's security guarantee — that the admin can revoke a pool's oracle access — is broken. If a pool is found to be exploiting the oracle read path (e.g., a pool whose price provider is feeding manipulated bid/ask into swaps, or a pool draining LP assets via oracle-anchored pricing), the admin blacklists it. The pool operator (or any colluding address) immediately pays `1 wei` to `register()`, clears the blacklist, and the pool resumes reading oracle prices. The admin is in a permanent reactive loop with no permanent resolution short of removing the approved factory — which blocks **all** pools from that factory, not just the malicious one.

This breaks the invariant: *"admin blacklist = durable access revocation."* The oracle's price-read path (`price(feedId, pool)`) continues to serve the blacklisted pool, allowing it to consume oracle prices and execute swaps that may drain LP principal or protocol fees.

### Likelihood Explanation

- The registration fee defaults to `1 wei` — trivially cheap for any attacker.
- `register()` is fully permissionless; no role, signature, or timelock is required.
- The factory's `isPool()` check is the only gate, and the pool already passed it on initial registration.
- The test `test_register_thirdParty_redeems_andEnablesRead` demonstrates this path works with a stranger address. [5](#0-4) 

### Recommendation

Add an admin-only `unregister(bytes32 feedId, address pool)` function that sets `registeredPool[feedId][pool] = false` and cannot be overridden by `register()` while a permanent-ban flag is set. Alternatively, separate the blacklist into two tiers: a soft blacklist (clearable by paying the fee, as today) and a hard ban (admin-only, not clearable by `register()`). The `register()` function should check the hard-ban flag before clearing the soft blacklist:

```solidity
mapping(address => bool) public permanentlyBanned;

function register(...) external payable {
    require(!permanentlyBanned[pool], PermanentlyBanned(pool));
    // ... existing logic
}

function permanentBan(address pool) external onlyRole(ADMIN_ROLE) {
    permanentlyBanned[pool] = true;
    blacklisted[pool] = true;
}
```

### Proof of Concept

1. Admin calls `setBlacklist(pool, true)` after detecting pool abuse.
2. `price(feedId, pool)` now reverts `Blacklisted(pool)` — pool is blocked.
3. Attacker (pool operator or any address) calls `oracle.register{value: 1}(feedId, pool, factory)`.
4. Inside `register()`: `blacklisted[pool] = false` executes unconditionally.
5. `price(feedId, pool)` succeeds again — pool reads oracle prices and resumes swaps.
6. Admin re-blacklists; attacker pays `1 wei` again. Loop repeats indefinitely.

The admin's only escalation path — raising `registrationFee` — harms all legitimate pools equally and still does not permanently exclude the targeted pool. [6](#0-5) [7](#0-6)
