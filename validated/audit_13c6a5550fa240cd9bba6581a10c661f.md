### Title
Metamorphic/Upgradeable Contract Pusher Bypasses One-Time Consent Check in `allowContractPushers`, Enabling Arbitrary Price Injection into Creator Namespace — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowContractPushers` validates pusher consent via a single live `isPusher(creator)` staticcall, then permanently writes `namespaceRemapping[pusher] = creator`. If the registered pusher contract is an upgradeable proxy or metamorphic contract, its implementation can be silently replaced after registration. The new implementation can push arbitrary prices into the creator's namespace, which flow directly into `AnchoredPriceProvider` and downstream pool swaps with no further authorization check.

---

### Finding Description

`allowContractPushers` (lines 217–234) performs a one-time `isPusher(creator)` staticcall to verify consent:

```solidity
(bool ok, bytes memory res) = pusher.staticcall(
    abi.encodeWithSignature("isPusher(address)", msg.sender)
);
require(ok);
bool allowed = abi.decode(res, (bool));
require(allowed);

namespaceRemapping[pusher] = msg.sender;   // permanent write
``` [1](#0-0) 

After this single check, the pusher contract has permanent, unconditional write authority over the creator's namespace via the oracle's `fallback`. There is no mechanism to verify that the pusher contract's code remains unchanged after registration.

The `allowPushers` (EOA) path explicitly defends against the analogous replay risk with a signed, deadline-bound consent — the comment at line 186–191 explains that the deadline prevents re-establishing delegation after the pusher revokes. But `allowContractPushers` carries no equivalent protection against implementation changes: [2](#0-1) 

If the pusher contract is:

1. **An upgradeable proxy** (UUPS, Transparent Proxy, etc.) — the proxy admin can upgrade the implementation after registration without any on-chain signal to the oracle or the creator.
2. **A metamorphic contract** (CREATE2 + SELFDESTRUCT) — the contract can self-destruct and be redeployed at the same address with a different implementation.

In either case, the new implementation calls the oracle's `fallback` to push arbitrary slot words into the creator's namespace: [3](#0-2) 

The `fallback` resolves the namespace from `namespaceRemapping[msg.sender]`, which still maps to the creator. The only gate is timestamp monotonicity — a fresh timestamp on the malicious push passes it trivially.

The manipulated price then flows through `AnchoredPriceProvider._readLeg`, which reads from `CompressedOracle.price`: [4](#0-3) 

Note that `CompressedOracle.price` ignores the `pool` parameter entirely — there is no `inSwap` binding or registration check (the oracle is intentionally open). The manipulated mid then enters `_computeBidAsk`, which computes the reference band **around the attacker-controlled mid**: [5](#0-4) 

The band clamp at lines 342–346 clips the source quote to the band, but the band itself is derived from the manipulated mid — so the clamp provides no protection when the anchor itself is corrupted. [6](#0-5) 

---

### Impact Explanation

A manipulated mid price (e.g., 2× the real market price) produces a manipulated bid/ask band. Pool swaps execute at this band, causing traders to receive wrong amounts or the pool to become insolvent. This matches the **bad-price execution** impact gate: an unclamped bid/ask quote derived from an attacker-controlled anchor reaches a live pool swap, causing direct loss of user principal.

---

### Likelihood Explanation

The attack requires the creator to delegate to an upgradeable or metamorphic contract pusher. This is realistic when:

- A creator uses a third-party off-chain pusher service that operates via an upgradeable proxy (common in production infrastructure).
- The creator audits the current implementation and sees it is legitimate — exactly as described in the external report: "Staking users could audit the current implementation themselves and would not see this possibility in the current code, and assume it is safe."
- The proxy admin (a different party from the creator) later upgrades the implementation.

The `allowContractPushers` function's own comment ("there is nothing to replay and no deadline is needed") signals that the designers considered replay but not implementation mutability — making this a latent, non-obvious trust assumption with no on-chain guard.

---

### Recommendation

1. **Codehash binding at registration:** Store `pusher.codehash` at `allowContractPushers` time and re-verify it on every fallback push from that contract. Any implementation change invalidates the delegation.

2. **Explicit non-upgradeability assertion:** Require the pusher contract to implement a `isUpgradeable() returns (bool)` function that returns `false`, and verify it at registration. While not foolproof against metamorphic contracts, it raises the bar and creates an explicit on-chain commitment.

3. **Revocation on codehash mismatch:** Add a public `verifyContractPusher(address pusher)` function that anyone can call; if `pusher.codehash` no longer matches the stored value, it clears `namespaceRemapping[pusher]` and emits a `PusherRevoked` event.

---

contract LegitImpl {

    // No isPusher needed — delegation is already stored
    function pushBadPrice(address oracle, address creator) external {
        // Craft a slot word with manipulated price (e.g., 2× real price)
        // packed as [data0:6][data1:6][data2:6][data3:6][ts:7][slotId:1]

        word |= uint256(raw) << 208;
        // Push into creator's namespace (namespaceRemapping[this] == creator)
        (bool ok,) = oracle.call(abi.encodePacked(word));
        require(ok);
    }
}

        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle

        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;

    function _computeBidAsk(uint256 mid, uint256 spreadBps)
        internal view returns (uint128, uint128)
    {
        // Circuit breaker: extreme (combined) uncertainty means the feed is clearly broken.
        if (spreadBps > MAX_SPREAD_BPS) {
            return (0, type(uint128).max);
        }

        // Reference band: mid ± (spreadBps + minMargin), bid rounded down, ask rounded up.

        uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
        uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
