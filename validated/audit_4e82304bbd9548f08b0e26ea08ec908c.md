### Title
Atomic fee delivery to both admin and protocol recipients in `collectFees` allows pool admin to permanently lock protocol fees via USDC/USDT blacklist — (File: metric-core/contracts/MetricOmmPool.sol)

### Summary
`MetricOmmPool.collectFees` delivers fees to both `adminFeeDestination_` and `FACTORY` atomically in a single transaction using `SafeERC20.safeTransfer`. If `adminFeeDestination_` is blacklisted by USDC or USDT, every transfer to that address reverts, permanently blocking the protocol from collecting its own share of fees. The pool admin controls `adminFeeDestination_` and the protocol has no independent collection path.

### Finding Description

`collectFees` in `MetricOmmPool.sol` performs four sequential `safeTransfer` calls inside one atomic block:

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // (1)
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);   // (2)
}
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);             // (3)
}
if (totalFee1ToProtocol > 0) {
    transferToken1(FACTORY, totalFee1ToProtocol);             // (4)
}

notionalFeeToken0Scaled = 0;   // only reached if all four succeed
notionalFeeToken1Scaled = 0;
``` [1](#0-0) 

`transferToken0` / `transferToken1` are thin wrappers around `IERC20.safeTransfer`: [2](#0-1) 

If transfer (1) or (2) reverts (e.g., USDC blacklists `adminFeeDestination_`), the entire call reverts. Transfers (3) and (4) — the protocol's share — never execute, and `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` are never cleared.

The pool admin sets `adminFeeDestination_` via `setPoolAdminFeeDestination`, which only validates non-zero: [3](#0-2) 

All three factory entry points that trigger fee collection pass `poolAdminFeeDestination[pool]` directly:

- `collectPoolFees` (permissionless): [4](#0-3) 
- `setPoolAdminFees` (pool admin): [5](#0-4) 
- `setPoolProtocolFee` (protocol owner): [6](#0-5) 

The protocol (factory owner) has no path to collect its fees that bypasses `adminFeeDestination_`. Only the pool admin can change `adminFeeDestination_`, and `setPoolAdminFeeDestination` does not call `collectFees` first, so the admin can freely rotate to a blacklisted address at any time without triggering a collection.

### Impact Explanation

Both spread fees (surplus-based) and notional fees (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) are permanently locked in the pool for the affected token. The protocol cannot collect its revenue from that pool. Additionally, `setPoolProtocolFee` — which calls `collectFees` before updating rates — is also bricked, preventing the protocol from adjusting fee parameters on the pool. LP principal is unaffected, but all accrued protocol fees are unrecoverable without pool admin cooperation.

### Likelihood Explanation

USDC and USDT are explicitly in scope. A pool admin's treasury address can be blacklisted by USDC/USDT after pool creation (e.g., regulatory action, sanctions). A malicious pool admin can also deliberately set a known-blacklisted address. The pool admin role is transferable, so a compromised admin key is a realistic trigger. `collectPoolFees` is permissionless, so any caller will continuously hit the revert once the destination is blacklisted.

### Recommendation

Decouple admin and protocol fee collection so each recipient can be served independently. Concretely:

1. Add a `recipient` selector parameter to `collectFees` (or split into `collectAdminFees` / `collectProtocolFees`).
2. Maintain separate cleared-flags or sub-counters for admin and protocol notional balances so a failed admin transfer does not prevent the protocol counter from being zeroed.
3. Alternatively, wrap the admin transfer in a `try/catch` and emit an event on failure, allowing the protocol share to proceed regardless of the admin destination's token-level state.

### Proof of Concept

1. Deploy a pool with `token0 = USDC`, `token1 = WETH`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, blacklistedAddr)` where `blacklistedAddr` is on the USDC blacklist.
3. Swaps accumulate spread and notional fees; `notionalFeeToken0Scaled > 0`.
4. Anyone calls `collectPoolFees(pool)`.
5. Inside `collectFees`, `transferToken0(blacklistedAddr, totalFee0ToAdmin)` calls `USDC.safeTransfer(blacklistedAddr, ...)` which reverts.
6. The entire `collectFees` transaction reverts; `notionalFeeToken0Scaled` is **not** cleared.
7. Protocol fees remain locked. Every subsequent call to `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee` for this pool reverts identically.
8. The protocol has no independent path to recover its fees without the pool admin changing `adminFeeDestination_` to a non-blacklisted address.
