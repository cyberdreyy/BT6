### Title
Pool admin can frontrun swaps by instantly raising per-bin additional fees with no cap and no timelock — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` allows the pool admin to instantly set per-bin buy/sell fees to any value up to `uint16` max (65,535 in E6 = **6.5535%**) with no cap check and no timelock. A malicious pool admin can observe a pending swap in the mempool, frontrun it by inflating the per-bin fee on the active bin, collect the excess fee, then reset the fee to zero — all in adjacent blocks or the same block on chains with private mempools.

---

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` enforces explicit caps before updating global spread/notional fees: [1](#0-0) 

By contrast, `setPoolBinAdditionalFees` passes the caller-supplied values directly to the pool with **no cap check**: [2](#0-1) 

The pool's `setBinAdditionalFees` similarly performs no cap validation — only a bin-index range check: [3](#0-2) 

The per-bin fees are consumed directly in swap math, added on top of the oracle-derived base spread fee: [4](#0-3) 

The only bound is the `uint16` type itself (max 65,535 in E6 = 6.5535%). There is no timelock, no fee-collection trigger (unlike `setPoolAdminFees`), and no hard ceiling analogous to `maxAdminSpreadFeeE6`. [5](#0-4) 

---

### Impact Explanation

A malicious pool admin can:

1. Observe a pending swap targeting bin `B` in the mempool.
2. Frontrun it with `setPoolBinAdditionalFees(pool, B, 65535, 65535)` — instantly raising the per-bin fee to 6.5535% with no delay.
3. The victim's swap executes at the inflated fee; the excess accrues as pool spread surplus claimable by the admin via `collectPoolFees`.
4. Admin resets the fee to 0 in the next transaction.

This is a direct loss of user principal on every swap through the targeted bin. The global spread fee has a hard cap of 20% (`maxOwnerSpreadCapE6`), but per-bin fees have no such ceiling, making the total effective fee `globalSpread + 6.5535%` — potentially far above what users expect. [6](#0-5) 

---

### Likelihood Explanation

The pool admin is a per-pool role set at creation and can be any address, including an EOA. Unlike the factory owner (protocol), the pool admin is not subject to governance timelocks. The attack requires only a single `setPoolBinAdditionalFees` call — no special setup, no multi-step exploit. On any chain with a public mempool the frontrun is trivially executable; on chains with private mempools the admin can simply call both transactions in the same block. [2](#0-1) 

---

### Recommendation

Apply the same cap discipline to per-bin fees that already exists for global admin fees:

1. **Add a hard cap constant** (e.g., `uint16 public constant MAX_BIN_ADDITIONAL_FEE_E6 = 50_000` for 5%) and enforce it inside `setPoolBinAdditionalFees` before forwarding to the pool.
2. **Alternatively**, introduce a timelock for per-bin fee changes analogous to the price-provider timelock, giving users time to exit before elevated fees take effect.
3. **Emit a predictable event** at proposal time so off-chain monitors can alert users before the change is applied. [7](#0-6) 

---

### Proof of Concept

```
Setup:
  - Pool deployed with bin 0 as the active bin, globalSpreadFeeE6 = 3000 (0.3%)
  - Alice submits swap tx: buy token0 through bin 0, amountIn = 100,000 USDC

Attack sequence (same block, admin frontruns):
  1. Admin calls: factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)
     → bin 0 addFeeBuyE6 = 65535 (6.5535%)
     → No revert: no cap check exists

  2. Alice's swap executes:
     → effectiveBuyFee = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
     → Alice pays ~6.85% total fee instead of expected 0.3%
     → Excess ~6.55% of 100,000 USDC = ~6,550 USDC lost to pool surplus

  3. Admin calls: factory.setPoolBinAdditionalFees(pool, 0, 0, 0)
     → Fee reset to 0

  4. Admin (or anyone) calls: factory.collectPoolFees(pool)
     → Admin's share of the inflated surplus is transferred to adminFeeDestination
``` [3](#0-2) [2](#0-1)
