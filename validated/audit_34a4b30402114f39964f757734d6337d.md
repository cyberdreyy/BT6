### Title
`allowContractPushers` Silently Overwrites `namespaceRemapping` Without Existing-Mapping Check, Enabling Persistent Feed-Staleness DoS on Pools — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracleV1.allowContractPushers()` unconditionally overwrites `namespaceRemapping[pusher]` with `msg.sender` without first checking whether the pusher is already mapped to a different creator. A second creator that the same pusher contract approves can silently redirect all future pushes to their own namespace, permanently abandoning the first creator's feeds. Pools whose `AnchoredPriceProvider` reads those feeds then halt with `FeedStalled` and cannot swap.

### Finding Description

`CompressedOracleV1` stores a **single** creator per pusher in `mapping(address => address) public namespaceRemapping`. Every `fallback()` push resolves the target namespace from this mapping:

```solidity
// CompressedOracle.sol L315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

`allowContractPushers` proves consent via a live `isPusher(creator)` staticcall and then unconditionally writes:

```solidity
// CompressedOracle.sol L231
namespaceRemapping[pusher] = msg.sender;   // no guard on existing value
```

There is no check that `namespaceRemapping[pusher] == address(0)` before the write. This is the structural analog of the `tokenIdsToPoolIds` bug: a single-slot mapping that can be overwritten by any second valid registration, silently discarding the first.

A shared contract-pusher service — a realistic production pattern where one pusher contract serves multiple creators and returns `true` from `isPusher()` for each of them — is the trigger. `creatorB` calls `allowContractPushers([sharedPusher])`, the oracle staticcalls `sharedPusher.isPusher(creatorB)` which returns `true`, and the mapping is overwritten from `creatorA` to `creatorB`. All subsequent pushes from `sharedPusher` land in `creatorB`'s namespace; `creatorA`'s feeds receive no further updates.

Compounding the issue, `removePushers` reverts with `InvalidManager` when the caller is not the current mapped creator:

```solidity
// CompressedOracle.sol L253-258
if (namespaceRemapping[pusher] == msg.sender) {
    namespaceRemapping[pusher] = address(0);
    ...
} else {
    revert InvalidManager(msg.sender);
}
```

After the hijack `namespaceRemapping[sharedPusher] == creatorB`, so `creatorA` cannot call `removePushers` to reclaim the pusher. `creatorA` can call `allowContractPushers` again to re-map, but `creatorB` can immediately overwrite again — a persistent griefing loop with no on-chain resolution.
