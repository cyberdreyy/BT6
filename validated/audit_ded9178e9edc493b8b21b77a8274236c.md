### Title
Unexpired `pendingStateGuard` Allows Stale Guard Acceptance to Bypass Feed Price-Guard, Enabling Bad-Price Execution in Pool Swaps - (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, `smart-contracts-poc/contracts/oracles/compressed/OracleBase.sol`)

---

### Summary

Both `OracleBase` contracts implement a two-step `stateGuard` transfer pattern (`setStateGuardRole` / `setPendingStateGuardRole` → `acceptStateGuardRole`) with no expiration deadline on the pending nomination. A stale `pendingStateGuard` entry can be accepted at any future time — even after the original intent has changed or the nominated address has become adversarial — granting the acceptor full control over `setPriceGuard` for the affected feed. A malicious new guard can then widen or remove a previously tight price guard, allowing out-of-range oracle prices to reach pool swaps.

---

### Finding Description

In both oracle bases the guard-transfer flow is:

**Providers oracle** (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`):

```solidity
function setStateGuardRole(bytes32 feedId, address newGuard) external checkRole(feedId) {
    pendingStateGuard[feedId] = newGuard;          // no deadline stored
    emit StateGuardPending(feedId, newGuard);
}

function acceptStateGuardRole(bytes32 feedId) external {
    require(pendingStateGuard[feedId] == msg.sender, InvalidGuard(msg.sender));
    // ← no block.timestamp ≤ deadline check
    delete pendingStateGuard[feedId];
    stateGuard[feedId] = msg.sender;
    emit StateGuardUpdated(feedId, msg.sender);
}
``` [1](#0-0) 

The identical pattern exists in the compressed oracle: [2](#0-1)

---

### Impact Explanation

A malicious actor who is the `pendingStateGuard` for a feed can call `acceptStateGuardRole` at any future time — months or years after the nomination — and become the `stateGuard`. They then call `setPriceGuard(feedId, 1, type(uint128).max - 1)`, effectively disabling a previously tight guard (e.g., a stablecoin guard of `[0.99e8, 1.01e8]`). From that point forward, any oracle price — including a severely manipulated or stale one — passes the guard check and reaches the pool's bid/ask computation, satisfying the "bad-price execution" impact criterion.

Because `stateGuard` is the exclusive authority once set, the ADMIN cannot override or re-tighten the guard without the malicious guard's cooperation.

---

### Likelihood Explanation

The trigger requires:
1. A legitimate guard to have called `setStateGuardRole` / `setPendingStateGuardRole` nominating an address.
2. The current guard to have forgotten (or become unable) to call `purgePendingStateGuardRole`.
3. The nominated address to have become adversarial (key compromise, social engineering, or deliberate delayed acceptance).

Steps 1–2 are realistic operational oversights; step 3 is a known threat model for time-unlimited pending roles. The `allowPushers` path in the same codebase explicitly requires a deadline for exactly this reason (replay / stale-consent prevention), making the absence of a deadline on `pendingStateGuard` an inconsistency the codebase itself acknowledges as dangerous:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [7](#0-6) 

The same rationale applies to `pendingStateGuard` but is absent.

---

### Recommendation

Add an expiration timestamp to the pending nomination and enforce it in `acceptStateGuardRole`:

```solidity
mapping(bytes32 => address)  public pendingStateGuard;
mapping(bytes32 => uint256)  public pendingStateGuardExpiry;  // NEW

function setStateGuardRole(bytes32 feedId, address newGuard, uint256 deadline) external checkRole(feedId) {
    require(block.timestamp <= deadline, DeadlineExceeded());
    pendingStateGuard[feedId]       = newGuard;
    pendingStateGuardExpiry[feedId] = deadline;   // NEW
    emit StateGuardPending(feedId, newGuard);
}

function acceptStateGuardRole(bytes32 feedId) external {
    require(pendingStateGuard[feedId] == msg.sender, InvalidGuard(msg.sender));
    require(block.timestamp <= pendingStateGuardExpiry[feedId], DeadlineExceeded()); // NEW
    delete pendingStateGuard[feedId];
    delete pendingStateGuardExpiry[feedId];
    stateGuard[feedId] = msg.sender;
    emit StateGuardUpdated(feedId, msg.sender);
}
```

Apply the same fix to both `OracleBase` contracts.

---

### Proof of Concept

```
T=0   : Guard A (legitimate) calls setStateGuardRole(feedId, guardB)
         → pendingStateGuard[feedId] = guardB  (no expiry stored)

T=30d : Guard A changes their mind; forgets to call purgePendingStateGuardRole.
         Guard A's key is rotated / lost.

T=90d : guardB (now adversarial) calls acceptStateGuardRole(feedId)
         → stateGuard[feedId] = guardB  (accepted with no deadline check)

T=90d : guardB calls setPriceGuard(feedId, 1, type(uint128).max - 1)
         → priceGuard[feedId] = {min: 1, max: type(uint128).max - 1}
         (previously tight guard, e.g. [0.99e8, 1.01e8], is now effectively disabled)

T=90d : Oracle reports a manipulated price of 5e8 (5× the stablecoin peg).
         _computeBidAsk / _readLeg: price 5e8 passes the guard (1 ≤ 5e8 ≤ max).
         Pool swap executes at the bad price → trader receives more than the
         oracle/bin curve permits, or LP assets are drained at the wrong rate.
```
