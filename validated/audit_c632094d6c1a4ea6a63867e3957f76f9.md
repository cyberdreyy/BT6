### Title
Permissionless `updateReport` Drains Oracle ETH Balance via Unpaid Verification Fees, Causing Stale-Price Execution in Pools — (`smart-contracts-poc/contracts/oracles/providers/ChainlinkOracle.sol`)

---

### Summary

`ChainlinkOracle.updateReport` and `updateReports` are permissionless external functions that pay `VERIFICATION_FEE` from the **oracle contract's own ETH balance** to the Chainlink VerifierProxy on every call. Because the oracle accumulates ETH from pool registration fees (and any operational reserve), an attacker can replay any valid DON-signed Chainlink Data Streams report repeatedly at zero personal cost to drain the oracle's balance. Once drained, all future price-update calls revert, the oracle serves permanently stale bid/ask prices to pools, and swaps execute against those stale prices.

---

### Finding Description

`ChainlinkOracle.updateReport` is declared `external` (not `payable`) and performs no access-control check:

```solidity
// ChainlinkOracle.sol lines 68-70
function updateReport(bytes calldata fullReport) external {
    _store(_verifyReport(fullReport));
}
``` [1](#0-0) 

The internal helper pays the fee from the oracle's own balance:

```solidity
// ChainlinkOracle.sol lines 81-83
function _verifyReport(bytes calldata fullReport) internal virtual returns (bytes memory reportData) {
    return verifierProxy.verify{value: VERIFICATION_FEE}(fullReport, abi.encode(feeToken));
}
``` [2](#0-1) 

The oracle contract holds ETH because:
1. `register{value}(feedId, pool, factory)` banks registration fees and explicitly does **not** refund overpayment.
2. `receive()` accepts arbitrary ETH for operational reserves. [3](#0-2) 

The test suite confirms the fee-from-balance pattern and that the oracle holds a non-zero balance after each update:

```
// test comment: "setUp pushed one update: the oracle balance = funding (2) - verification fee (1) = 1 wei."
``` [4](#0-3) 

The same pattern exists in `updateReports` (batch variant): [5](#0-4) 

Neither function checks that `msg.value >= VERIFICATION_FEE`, nor that the caller is authorized. The caller contributes **zero ETH**; the oracle pays the VerifierProxy from its own balance on every invocation.

---

### Impact Explanation

Once the oracle's ETH balance reaches zero, every subsequent `updateReport` call reverts (the `{value: VERIFICATION_FEE}` sub-call fails). The oracle can no longer ingest new Chainlink Data Streams reports. All pools whose `PriceProvider` reads from this oracle will serve the last stored (now stale) bid/ask price indefinitely. Swaps execute against stale prices, enabling:

- Traders to receive more output than the current market price permits (swap conservation failure).
- LPs to suffer losses from arbitrageurs exploiting the stale quote.

This matches the **bad-price execution** and **broken core pool functionality** impact categories.

---

### Likelihood Explanation

- `updateReport` is permissionless and callable by any EOA or contract.
- Valid Chainlink Data Streams reports are publicly observable on-chain (emitted by the DON); an attacker can replay any previously seen report.
- The Chainlink VerifierProxy verifies the DON signature but does **not** enforce report freshness — that check lives in `_store`. The fee is consumed by the VerifierProxy regardless of whether `_store` ultimately updates the stored price.
- The oracle's ETH balance grows with every pool registration (fee default is 1 wei but configurable upward via `setRegistrationFee`). A protocol with many registered pools accumulates a meaningful balance.
- No existing guard prevents this call path.

---

### Recommendation

Two complementary fixes, mirroring the remediation in the referenced report:

1. **Require the caller to cover the verification fee.** Make `updateReport` / `updateReports` `payable` and assert `msg.value >= VERIFICATION_FEE` (or `>= VERIFICATION_FEE * reports.length` for the batch). Forward `msg.value` to the VerifierProxy instead of the oracle's own balance.

2. **Uphold the invariant that the oracle holds no ETH beyond what is explicitly deposited as an operational reserve.** If an operational reserve is intentional, gate `updateReport` to `ADMIN_ROLE` or a trusted keeper, so only authorized parties can spend it.

---

### Proof of Concept

```solidity
// Attacker contract
contract Drain {
    ChainlinkOracle oracle;
    bytes validReport; // any previously observed DON-signed report

    constructor(address _oracle, bytes memory _report) {
        oracle = ChainlinkOracle(_oracle);
        validReport = _report;
    }

    // Call until oracle.balance == 0
    function drain() external {
        while (address(oracle).balance >= VERIFICATION_FEE) {
            oracle.updateReport(validReport);
            // Each call: oracle pays VERIFICATION_FEE to VerifierProxy, caller pays 0
            // _store() silently ignores the old timestamp (no revert), fee is gone
        }
        // oracle.balance == 0; all future updateReport calls revert
        // Pools now read permanently stale bid/ask prices
    }
}
```

Step-by-step:
1. Attacker observes any valid Chainlink Data Streams report on-chain.
2. Calls `drain()` — each iteration costs only gas; the oracle pays `VERIFICATION_FEE` to the VerifierProxy.
3. `_store` does not revert on a replayed (old-timestamp) report — it simply skips the write — so the loop never reverts until the oracle's balance is exhausted.
4. Oracle balance reaches zero; `updateReport` reverts on all future calls.
5. Pools read the last stored (stale) price; swaps execute at the wrong rate.
