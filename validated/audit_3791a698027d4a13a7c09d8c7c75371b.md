### Title
`revokePusher()` Is Ineffective Before Deadline Expiry — Creator Can Replay Consent Signature to Silently Re-Delegate a Pusher and Starve Its Own-Namespace Pools of Prices - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowPushers` signs consent with `(chainid, oracle, deadline, pusher, creator)` but includes **no nonce**. The same signature is therefore replayable an unlimited number of times before the deadline expires. A creator who holds a pusher's original consent signature can call `allowPushers` again immediately after the pusher calls `revokePusher()`, atomically re-establishing the delegation. The pusher's revocation is silently undone, and every subsequent push the pusher makes lands in the creator's namespace instead of the pusher's own — starving any pool that reads from the pusher's own namespace of fresh prices.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

The only replay barrier is `_ensureDeadline(deadline)`, which passes as long as `block.timestamp <= deadline`. There is no per-use nonce, no consumed-flag, and no check for whether the pusher has already revoked.

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

After this call `namespaceRemapping[pusher] == address(0)`, so the pusher's next push via `fallback()` would correctly land in the pusher's own namespace:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;   // own namespace
```

But the creator still holds the original `(deadline, pusher, creator)` signature. As long as `block.timestamp <= deadline`, the creator can call `allowPushers` again with the identical arguments and signature, writing `namespaceRemapping[pusher] = creator` again. The revocation is undone in the same block or any subsequent block before the deadline.

The code comment at lines 186–191 explicitly acknowledges the replay concern ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but misidentifies the deadline as the fix. The deadline only prevents replay **after** it expires; it does nothing to prevent replay **before** it expires. Creators routinely set deadlines days or weeks in the future, leaving a large window.

---

### Impact Explanation

The `CompressedOracleV1` is registrationless: any address is simultaneously a potential pusher **and** a potential namespace creator whose feeds are consumed by pools. A pusher that also operates its own pools (reading from `feedIdOf(pusher, slot, pos)`) is the directly affected party:

1. Pusher P is delegated to creator C's namespace; P's pushes land in C's namespace.
2. P decides to exit the delegation and serve its own pools. P calls `revokePusher()`.
3. C immediately replays the original consent signature via `allowPushers(deadline, [P], [sig])`. `namespaceRemapping[P]` is set back to C.
4. P's subsequent pushes continue to land in C's namespace, not P's own.
5. P's own namespace (`feedIdOf(P, slot, pos)`) receives no updates. Its timestamp stays at the last pre-revocation value (or zero if never independently pushed).
6. Any pool whose `PriceProvider` reads `feedIdOf(P, slot, pos)` sees a stale or zero price. The provider's staleness check (`refTime < block.timestamp - maxTimeDrift`) fires, reverting with `FeedStalled`. Swaps through that pool are broken for the entire duration of the deadline window.

The pusher has no on-chain recourse: it cannot invalidate the already-signed consent, cannot shorten the deadline, and cannot force the creator to call `removePushers`. The only escape is waiting for the deadline to expire — which can be arbitrarily far in the future since the creator sets it.

---

### Likelihood Explanation

- The creator holds the original signature bytes from the initial `allowPushers` call; no additional off-chain action is needed.
- The attack requires only one additional on-chain transaction (replay `allowPushers`) after the pusher revokes.
- Deadlines are set by the creator and are typically days to weeks in the future (the test suite uses `block.timestamp + 1 days`), giving a large replay window.
- The pusher has no way to detect the re-delegation on-chain before their next push is silently redirected.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on every successful `allowPushers` call. Alternatively, record a "revoked-at" timestamp per pusher and reject any consent signature whose deadline predates the revocation. The simplest fix is a mapping `uint256 public pusherNonce[address]` incremented in `revokePusher` and included in the hash:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))
```

`revokePusher` increments `pusherNonce[pusher]`, invalidating all previously signed consents for that pusher.

### Proof of Concept

```solidity
// 1. Creator holds pusher's consent signed for deadline = T + 365 days.
//    Initial delegation:
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator);

// 2. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 3. Creator replays the IDENTICAL call — deadline has not expired.
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);            // succeeds: no nonce check
assertEq(oracle.namespaceRemapping(pusher), creator);    // re-delegated

// 4. Pusher pushes, believing it is writing to its own namespace.
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw  = _packRaw(500_000, 3, 3);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(slotId, pos, raw, tsMs));
assertTrue(ok);

// 5. Push landed in CREATOR namespace, not pusher's own.
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, slotId, pos)).price,
         U64x32.decode(uint32(raw >> 16)));              // creator got the price
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, slotId, pos)).price, 0);

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
