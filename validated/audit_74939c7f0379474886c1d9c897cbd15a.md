### Title
Delegated pusher can lock a `CompressedOracleV1` slot with a future timestamp, causing bad-price execution or swap DoS in pools — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1` accepts slot words whose embedded timestamp is up to `MAX_TIME_DRIFT` seconds in the future. Once such a word is stored, the per-slot monotonicity guard blocks every subsequent push that carries a current-time timestamp for the entire drift window. During that window `AnchoredPriceProvider` treats the future `refTime` as stale and halts swaps, while `PriceProviderL2` (whose `FUTURE_TOLERANCE ≥ MAX_TIME_DRIFT`) accepts the attacker-chosen price as fresh, feeding a manipulated bid/ask into pool swaps.

---

### Finding Description

**Push path — future timestamp accepted and stored**

Both the `fallback()` and `updateBySignature()` paths call:

```solidity
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
``` [1](#0-0) 

which expands to:

```solidity
require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
``` [2](#0-1) 

When `MAX_TIME_DRIFT > 0` (e.g. 5 s for L2 clock-skew), a pusher may embed `ts = (block.timestamp + MAX_TIME_DRIFT) * 1000` ms and the word is accepted and written to storage.

**Monotonicity guard freezes the slot**

After the write, the freshness gate:

```solidity
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;
``` [3](#0-2) 

rejects every subsequent push whose timestamp is ≤ the stored future value. No legitimate pusher — including the creator — can overwrite the slot until real time surpasses the stored timestamp.

**`AnchoredPriceProvider` — swap DoS**

`AnchoredPriceProvider._isStale` treats any `refTime > nowTs` as stale:

```solidity
if (refTime > nowTs) return true;
``` [4](#0-3) 

`_readLeg` therefore returns `ok = false`, `_getBidAndAskPrice` returns the `(0, type(uint128).max)` sentinel, and `getBidAndAskPrice` reverts with `FeedStalled`:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
``` [5](#0-4) 

Every pool swap is frozen for the entire drift window.

**`PriceProviderL2` — bad-price execution**

`PriceProviderL2._isStale` is L2-aware and explicitly tolerates future timestamps within `FUTURE_TOLERANCE`:

```solidity
if (refTime > nowTs) {
    // refTime in the future: tolerate only within futureTol
    return (refTime - nowTs) > futureTol;
}
``` [6](#0-5) 

When `MAX_TIME_DRIFT ≤ FUTURE_TOLERANCE` (the normal L2 configuration: e.g. drift = 5 s, tolerance = 60 s), the stored future timestamp satisfies `refTime − nowTs ≤ FUTURE_TOLERANCE`, so the feed is **not** stale. The attacker-chosen price — which can be arbitrarily low or high within the `U64x32` encoding range — is returned as the live mid price, and the pool executes swaps against it.

**Who can trigger this**

The `fallback()` resolves the effective namespace via:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [7](#0-6) 

A delegated pusher (authorized by the creator via `allowPushers` or `allowContractPushers`) writes into the creator's namespace — the same namespace that `AnchoredPriceProvider` / `PriceProviderL2` read from. The pusher can self-revoke after the attack; the frozen slot persists regardless.

---

### Impact Explanation

- **`AnchoredPriceProvider` pools**: all swaps revert with `FeedStalled` for up to `MAX_TIME_DRIFT` seconds — broken core swap functionality.
- **`PriceProviderL2` pools** (when `FUTURE_TOLERANCE ≥ MAX_TIME_DRIFT`): the attacker-chosen price is fed into `_getBidAndAskPrice`, producing an unbounded or inverted bid/ask. Traders execute swaps at the manipulated price, suffering direct loss of principal. The attacker can be on the opposite side of those swaps to extract the difference.

Both outcomes match the allowed impact gate: broken swap flows and bad-price execution causing direct loss of user principal.

---

### Likelihood Explanation

The attacker must hold a valid pusher delegation for the target creator namespace. Delegations are granted via `allowPushers` (EIP-191 signature) or `allowContractPushers` (live `isPusher` call). A pusher is a semi-trusted operational role, not a privileged admin. The attack requires a single `fallback()` call with a crafted slot word — no special setup beyond the existing delegation.

---

### Recommendation

1. **Reject future timestamps unconditionally in `CompressedOracleV1`**: set `MAX_TIME_DRIFT = 0` or remove the drift parameter. L2 clock skew should be absorbed by the *consumer* (provider), not the oracle store.
2. **Alternatively**, add a creator-only emergency-reset path that can overwrite a slot regardless of the monotonicity check, so a frozen slot can be cleared without waiting for real time to catch up.
3. **In `PriceProviderL2`**, align the staleness check with `AnchoredPriceProvider`: treat any `refTime > nowTs` as stale, applying `FUTURE_TOLERANCE` only as a leniency on the *age* check, not as a window in which future-dated prices are accepted as live.

---

### Proof of Concept

```solidity
// Setup
CompressedOracleV1 oracle = new CompressedOracleV1(owner, 5); // MAX_TIME_DRIFT = 5 s
// PriceProviderL2 deployed with FUTURE_TOLERANCE = 60 s, reading oracle/feedId

// Creator authorizes malicious pusher
uint256 deadline = block.timestamp + 1 days;
oracle.allowPushers(deadline, [maliciousPusher], [pusherSig]);

// Attack: push manipulated price with timestamp 5 s in the future
uint56 futureTs = uint56((block.timestamp + 5) * 1000);
uint48 manipulatedRaw = packRaw(1 /* near-zero price */, 3, 3);
bytes memory payload = wordAt(slotId, positionIndex, manipulatedRaw, futureTs);
vm.prank(maliciousPusher);
(bool ok,) = address(oracle).call(payload);
assert(ok); // accepted — within MAX_TIME_DRIFT

// For the next 5 seconds:
// 1. Legitimate pushes with current timestamp fail (monotonicity check).
// 2. PriceProviderL2._isStale: refTime - nowTs = 5 <= FUTURE_TOLERANCE = 60 → NOT stale.
// 3. Pool swaps execute at the near-zero manipulated price.
// 4. Attacker profits by being on the opposite side of those swaps.
```
