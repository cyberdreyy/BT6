### Title
Creator Re-Establishes Revoked Pusher Delegation Using Unexpired Signature, Routing Wrong-Asset Price Data Into Production Feeds — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` in `CompressedOracleV1` does not prevent a creator from re-establishing a pusher delegation that the pusher explicitly revoked via `revokePusher()`. As long as the original signed consent's deadline has not expired, the creator can replay the same signature to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is therefore not final until the deadline elapses. If the pusher has already begun pushing data for a different namespace after revoking, those pushes are silently redirected back into the creator's namespace, feeding wrong-asset prices into any pool that reads from those feeds.

### Finding Description

`revokePusher()` clears the delegation by writing `namespaceRemapping[msg.sender] = address(0)`. [1](#0-0) 

`allowPushers` then re-establishes it unconditionally: it checks only that the deadline is in the future and that the pusher's EIP-191 signature is valid. There is no check that the pusher has previously revoked, and no nonce or consumed-signature registry. [2](#0-1) 

The code comment explicitly acknowledges the re-establishment risk but claims the deadline mitigates it:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The comment is incorrect: the deadline prevents re-establishment only after the deadline expires, not after the pusher revokes. A creator can call `allowPushers` with the original signature at any moment before the deadline, even seconds after `revokePusher()` succeeds.

Once the delegation is restored, the `fallback()` push path resolves the namespace from `namespaceRemapping[msg.sender]` and routes every subsequent push from the pusher into the creator's namespace: [4](#0-3) 

Because the slot word layout is identical across all namespaces (same slotId, same positionIndex), data the pusher intended for their own feeds (e.g., BTC/USD at slot 5, position 0) overwrites the creator's feeds at the same coordinates (e.g., ETH/USD at slot 5, position 0), with a fresh timestamp that passes every staleness guard downstream. [5](#0-4) 

### Impact Explanation

Any `PriceProvider` or `AnchoredPriceProvider` bound to the creator's feed reads the wrong-asset price through the standard `price(feedId, pool)` path. The price is fresh (passes `MAX_REF_STALENESS`), non-zero (passes `mid == 0`), and within the codebook spread range (passes `spread >= ORACLE_BPS`). The anchored band clamp cannot correct a price that is wrong by an order of magnitude (e.g., BTC/USD ≈ 60 000 landing in an ETH/USD feed ≈ 3 000). Every swap executed against that pool during the contamination window receives a materially wrong bid/ask, constituting direct bad-price execution and loss of LP principal above contest thresholds. [6](#0-5) 

### Likelihood Explanation

The trigger requires a creator who re-calls `allowPushers` after the pusher revokes, and a pusher who continues pushing (for their own namespace or a new creator) after revoking. The creator is not a privileged party (not the factory owner or oracle admin); `allowPushers` is a public function callable by any namespace owner. The pusher's original signature is already on-chain (emitted via `PusherAuthorized`), so the creator needs no new off-chain material. The window of vulnerability equals the remaining lifetime of the original deadline, which the code encourages to be long (the comment warns only against *undated* signatures, implying long deadlines are acceptable).

### Recommendation

Track consumed delegation consents with a per-pusher revocation nonce or a `usedConsents` mapping keyed on `keccak256(signature)`. In `allowPushers`, revert if the consent has already been used or if `namespaceRemapping[pusher]` was previously set and then cleared (indicating a prior revocation). Alternatively, require the pusher to re-sign a fresh consent after every revocation, making the deadline irrelevant to the re-establishment attack.

### Proof of Concept

```
1. Creator A calls allowPushers(deadline = now + 7 days, [pusher], [sig])
   → namespaceRemapping[pusher] = creatorA

2. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)
   Pusher begins pushing BTC/USD data intended for their own namespace.

3. Creator A calls allowPushers(deadline = now + 7 days, [pusher], [sig])
   (same signature, deadline still valid)
   → namespaceRemapping[pusher] = creatorA  ← revocation undone

4. Pusher's next fallback push (BTC/USD, slotId=5, pos=0, ts=T+1)
   → namespace resolves to creatorA
   → overwrites creatorA's ETH/USD feed at (slot=5, pos=0) with BTC/USD price

5. Pool bound to creatorA's ETH/USD feed reads BTC/USD price (~60 000)
   instead of ETH/USD price (~3 000) → bad-price execution, LP loss.
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
