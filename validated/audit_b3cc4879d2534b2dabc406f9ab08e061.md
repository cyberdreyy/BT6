### Title
Immutable `FACTORY` recipient in `collectFees` causes protocol fees permanently stuck when blacklisted by USDC/USDT — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.collectFees` transfers the protocol share of accrued fees directly to the immutable `FACTORY` address via `safeTransfer`. If `FACTORY` is blacklisted by USDC or USDT, every call to `collectFees` reverts permanently, locking all accumulated protocol and admin fees in the pool with no on-chain rescue path. This is a structural analog to the Sparkn `STADIUM_ADDRESS` bug: an immutable fee-recipient address whose blacklisting freezes the entire fee-collection flow.

---

### Finding Description

`FACTORY` is set once at construction and stored as an EVM immutable: [1](#0-0) 

Inside `collectFees`, the protocol share of both spread and notional fees is pushed to that address with `safeTransfer`: [2](#0-1) 

`transferToken0` / `transferToken1` are thin wrappers around `IERC20.safeTransfer`: [3](#0-2) 

The fee-state reset (`notionalFeeToken0Scaled = 0`) happens **after** the transfers: [4](#0-3) 

If `FACTORY` is blacklisted by the USDC or USDT operator, `safeTransfer` reverts, the state reset is never reached, and the fees remain permanently locked in the pool.

The blast radius extends beyond fee collection. Both `setPoolProtocolFee` and `setPoolAdminFees` in the factory call `collectFees` as their first action before updating fee parameters: [5](#0-4) [6](#0-5) 

If `collectFees` always reverts, these admin functions are permanently broken — the pool's fee configuration is frozen.

The factory's only token-rescue function (`collectTokens`) rescues tokens held **by the factory**, not by pools: [7](#0-6) 

There is no pool-level rescue path.

---

### Impact Explanation

- All accumulated protocol fees (`notionalFeeToken0Scaled`, `notionalFeeToken1Scaled`, and spread surplus) are permanently stuck in the pool.
- Admin fees are equally stuck because `collectFees` handles both legs atomically.
- `setPoolProtocolFee` and `setPoolAdminFees` become permanently unusable for every pool whose `FACTORY` is blacklisted, freezing fee configuration.
- Core swap and liquidity flows are unaffected (they do not call `collectFees`), so LP principal is safe — but owed protocol and admin fee revenue is irrecoverable.

---

### Likelihood Explanation

Low. USDC/USDT blacklisting a deployed DEX factory contract is an unusual event, but it is technically possible (e.g., OFAC-linked activity touching the factory address). USDC and USDT blacklisting is explicitly in scope per the contest rules. The immutability of `FACTORY` means there is no administrative path to substitute a non-blacklisted recipient without redeploying every affected pool.

---

### Recommendation

1. **Decouple the protocol fee destination from `FACTORY`.** Store a mutable `protocolFeeRecipient` address (updatable by the factory owner) and transfer fees there instead of to the factory contract itself.
2. **Wrap the transfer in a try/catch or use a pull model.** Accumulate protocol fees in a claimable balance rather than pushing them in `collectFees`, so a blacklisted recipient cannot freeze the entire function.
3. **Add a pool-level rescue path.** Allow the factory owner to redirect stuck fee balances to an alternative address without requiring a full pool redeployment.

---

### Proof of Concept

1. Deploy `MetricOmmPool` with USDC as `TOKEN0`; `FACTORY` is set immutably at construction.
2. Execute swaps; `notionalFeeToken0Scaled` and spread surplus accumulate in the pool.
3. USDC operator blacklists the `FACTORY` address.
4. Anyone calls `collectPoolFees(pool)` on the factory → internally calls `pool.collectFees(...)` → `transferToken0(FACTORY, amount)` → `IERC20(USDC).safeTransfer(FACTORY, amount)` → **reverts** (blacklisted recipient).
5. `notionalFeeToken0Scaled` is never cleared; fees remain in the pool.
6. Factory owner calls `setPoolProtocolFee(pool, newFee, 0)` → same revert path → fee configuration is frozen.
7. No on-chain function exists to rescue the stuck USDC from the pool.
