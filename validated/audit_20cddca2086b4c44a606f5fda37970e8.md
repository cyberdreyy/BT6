### Title
Pusher consent signature replay after `revokePusher()` re-establishes delegation without current consent — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers()` verifies an EIP-191 signature whose hash covers `(chainid, address(this), deadline, pusher, creator)` but does **not** include any state that changes when a pusher revokes. After a pusher calls `revokePusher()`, the creator can replay the original consent signature — as long as the deadline has not expired — to silently re-establish the delegation without the pusher's current consent.

---

### Finding Description

The `allowPushers` function in `CompressedOracleV1` accepts a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The signed message contains no nonce and no representation of the current `namespaceRemapping[pusher]` value. The code's own NatSpec comment acknowledges the deadline is meant to prevent re-establishment after revocation:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [2](#0-1) 

However, the deadline only prevents replay **after** it expires. Within the deadline window, the signature remains valid regardless of how many times `revokePusher()` has been called, because `revokePusher()` only zeroes `namespaceRemapping[msg.sender]` — it does not invalidate any outstanding signatures:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

This is the direct analog of the `IdRegistry` bug: just as `changeRecoveryAddress()` did not invalidate a pending `changeRecoveryAddressFor()` signature, `revokePusher()` does not invalidate a pending `allowPushers()` signature.

---

### Impact Explanation

After the pusher revokes, the creator replays the old signature to re-establish `namespaceRemapping[pusher] = creator`. All subsequent `fallback()` pushes from the pusher then resolve the namespace as the creator's:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

The pusher, believing they are writing to their own namespace, instead overwrites the creator's live feed slots. Any `PriceProvider` or pool reading from the creator's `feedIdOf(creator, slotIndex, positionIndex)` will receive the pusher's unintended data as the canonical price. This constitutes bad-price execution reaching live pool swaps.

The creator can repeat the replay every time the pusher revokes, for the full duration of the deadline window (which can be set to any future timestamp the creator chooses when calling `allowPushers`).

---

### Likelihood Explanation

- The signature is broadcast on-chain when `allowPushers` is first called; anyone can observe and store it.
- The creator controls when to call `allowPushers` again — no privileged access is required.
- The pusher has no on-chain mechanism to cancel the signature short of waiting for the deadline to expire.
- Deadlines can be set arbitrarily far in the future (no cap is enforced), so the replay window can be months or years.

---

### Recommendation

Include the current `namespaceRemapping[pusher]` value (or a per-pusher nonce that increments on every delegation/revocation) in the signed message. This mirrors the fix applied to `IdRegistry`: binding the signature to the current state means any state change automatically invalidates outstanding signatures.

Example nonce approach:

```solidity
mapping(address => uint256) public pusherNonce;

// in allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// in revokePusher / removePushers:
pusherNonce[pusher]++;
```

---

### Proof of Concept

```
1. Pusher signs consent for creatorA with deadline = block.timestamp + 30 days.
2. creatorA calls allowPushers(deadline, [pusher], [sig])
   → namespaceRemapping[pusher] = creatorA  ✓
3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓
4. creatorA calls allowPushers(deadline, [pusher], [sig])  ← SAME signature, still valid
   → namespaceRemapping[pusher] = creatorA  ← re-established without pusher consent
5. Pusher calls fallback() intending to push to own namespace.
   → namespaceRemapping[pusher] resolves to creatorA
   → data lands in creatorA's feed slots
   → any pool reading feedIdOf(creatorA, slotIndex, positionIndex) receives pusher's
      unintended price data as the live oracle quote
```
