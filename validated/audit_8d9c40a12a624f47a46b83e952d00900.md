### Title
Creator Can Re-Establish Revoked EOA Pusher Delegation Using Unexpired Signature, Feeding Unauthorized Prices Into Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` contains no nonce, no used-signature bitmap, and no check on the current state of `namespaceRemapping[pusher]`. A creator who holds a pusher's still-valid (deadline not yet expired) EIP-191 consent signature can call `allowPushers` again at any time after the pusher has self-revoked via `revokePusher()`, silently re-establishing the delegation. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the signed deadline, and any data the pusher subsequently pushes — intended for their own namespace — is silently redirected into the creator's namespace and consumed by downstream price providers and pools.

---

### Finding Description

`CompressedOracleV1.allowPushers` signs consent as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

The signed tuple contains no nonce and no "already-used" marker. The only freshness gate is `_ensureDeadline(deadline)`, which only rejects calls made **after** the deadline timestamp. [1](#0-0) 

After a pusher calls `revokePusher()`, which sets `namespaceRemapping[pusher] = address(0)`: [2](#0-1) 

…the creator can immediately call `allowPushers` again with the **identical** `(deadline, pusher, signature)` tuple. Because the deadline has not expired, `_ensureDeadline` passes. Because the signature is cryptographically valid, `ECDSA.recover` passes. The function unconditionally overwrites `namespaceRemapping[pusher] = msg.sender`, restoring the delegation without any new consent from the pusher.

The code's own NatSpec acknowledges this exact risk but misidentifies the deadline as the fix:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline prevents re-use only **after** it expires; it does nothing to prevent re-use **before** expiry. There is no guard that checks whether `namespaceRemapping[pusher]` was previously cleared by the pusher.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after revocation — believing they are writing to their own namespace — is silently redirected into the creator's namespace once the creator re-establishes the delegation.

---

### Impact Explanation

The creator's namespace is the data source for `feedIdOf(creator, slotIndex, positionIndex)` feeds. These feeds are consumed by `AnchoredPriceProvider` (and `PriceProvider`) via `offchainOracle.price(feedId, pool)`: [5](#0-4) 

If the pusher, after revoking, begins pushing data for a different asset pair or at a different price scale into what they believe is their own namespace, those values land in the creator's namespace and are returned as the mid price, spread, and timestamp for the creator's pool feeds. The `AnchoredPriceProvider` clamps the bid/ask to the reference band, but the reference band itself is derived from the same corrupted mid price — so the clamp does not protect against a wrong mid. A pool swap that executes against a wrong mid price violates swap conservation: the trader receives more (or less) than the oracle curve permits.

---

### Likelihood Explanation

The trigger requires:
1. A pusher who signed consent with a deadline far in the future (common for long-lived infrastructure pushers).
2. The pusher calling `revokePusher()` (e.g., to exit the relationship or switch to their own namespace).
3. The creator calling `allowPushers` again with the same signature before the deadline expires.
4. The pusher continuing to push data (to what they believe is their own namespace).

Steps 1–3 are fully on-chain and require no off-chain coordination. Step 4 is the condition that converts the broken invariant into a bad-price event. A pusher who revokes in order to start pushing a different feed (e.g., switching from BTC/USD to ETH/USD in their own namespace) satisfies step 4 automatically.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` bitmap, and reject any `allowPushers` call whose signature has already been accepted:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

// inside allowPushers, after recovering the signer:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedDelegationSigs[sigHash], "signature already used");
_usedDelegationSigs[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed message and increment it on every successful delegation, making each signature single-use regardless of deadline.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
