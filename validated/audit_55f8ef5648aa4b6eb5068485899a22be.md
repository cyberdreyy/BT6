### Title
`setPriceGuard()` bounds are stored but never enforced in `CompressedOracle.price()` / `getOracleData()`, allowing any pushed price to reach pools unchecked — (`smart-contracts-poc/contracts/oracles/compressed/OracleBase.sol`, `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`compressed/OracleBase.sol` exposes `setPriceGuard(feedId, minPrice, maxPrice)` so a feed's `stateGuard` (or creator) can set absolute price boundaries. The protocol documentation states that a price outside those bounds must cause the read to fail. However, neither `CompressedOracle.getOracleData()` nor `CompressedOracle._price()` / `price()` ever reads or enforces the `priceGuard` mapping. Any price a pusher writes into a slot — including values far outside the configured guard — is returned verbatim to every pool that calls `price()`.

---

### Finding Description

**State set, invariant never checked — exact analog of the LybraFinance boost-lock bypass.**

In LybraFinance, `setLockStatus()` records a lock duration that earns a higher boost, but `withdraw()` never reads that duration, so the user keeps the boost while withdrawing freely. Here, `setPriceGuard()` records `min`/`max` price bounds that are supposed to gate every read, but the entire `CompressedOracle` read path never reads those bounds.

**Step 1 — Guard is set.**

`compressed/OracleBase.setPriceGuard` writes the guard:

```solidity
// compressed/OracleBase.sol
function setPriceGuard(bytes32 feedId, uint128 minPrice, uint128 maxPrice)
    external checkRole(feedId)
{
    require(minPrice < maxPrice);
    priceGuard[feedId] = PriceGuard({min: minPrice, max: maxPrice});
    emit PriceGuardUpdated(feedId, minPrice, maxPrice);
}
``` [1](#0-0) 

**Step 2 — Pusher writes an out-of-bounds price.**

`CompressedOracle.fallback()` accepts any slot word whose timestamp is strictly newer than the stored one. There is no price-range check:

```solidity
// CompressedOracle.sol – fallback push path
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;
_writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
``` [2](#0-1) 

**Step 3 — Read path ignores `priceGuard` entirely.**

`getOracleData()` decodes the U64x32 price and codebook spreads and returns them with no guard check:

```solidity
data.price = U64x32.decode(compressed.p);
data.spread0 = _decodeCodebookIndex(compressed.s0);
data.spread1 = _decodeCodebookIndex(compressed.s1);
data.timestampMs = _layout.timestampMs;
``` [3](#0-2) 

`_price()` calls `getOracleData()` and forwards the result directly:

```solidity
function _price(bytes32 feedId) internal view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = getOracleData(feedId);
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
``` [4](#0-3) 

`price(feedId, pool)` — the function pools call — is a thin wrapper over `_price()` with no additional guard:

```solidity
function price(bytes32 feedId, address /* pool */)
    external view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    return _price(feedId);
}
``` [5](#0-4) 

The `priceGuard` mapping is populated but **never read** anywhere in the `CompressedOracle` contract or its base.

---

### Impact Explanation

`CompressedOracle.price()` is the oracle entry point consumed by `AnchoredPriceProvider` (and any pool wired directly to the compressed oracle). If the mid-price returned is outside the guard bounds — e.g., an order-of-magnitude spike or crash — the `AnchoredPriceProvider` safety band is computed around that corrupted anchor, producing a shifted bid/ask corridor. Every swap executed against that pool during the window trades at the bad price: traders receive more output than the true market permits (swap conservation failure) or the pool receives less input than owed, directly draining LP principal.

---

### Likelihood Explanation

The `CompressedOracle` is permissionless for pushes from the creator and all addresses in `namespaceRemapping`. A delegated pusher whose key is compromised, or a contract pusher whose `isPusher` authorization has been silently revoked at the contract level (but whose `namespaceRemapping` entry was never cleared by the creator), can push an arbitrary price. Because `price()` is a `view` function with no `inSwap` binding or registration check, the bad price is immediately visible to any pool reading the feed. No admin action is required to trigger the impact once the bad slot word is written.

---

### Recommendation

Enforce `priceGuard` inside `getOracleData()` (or `_price()`) before returning data to callers:

```solidity
function getOracleData(bytes32 feedId) public view override returns (OracleData memory data) {
    // ... existing decode logic ...
    data.price = U64x32.decode(compressed.p);
    data.spread0 = _decodeCodebookIndex(compressed.s0);
    data.spread1 = _decodeCodebookIndex(compressed.s1);
    data.timestampMs = _layout.timestampMs;

    // ADD: enforce priceGuard if configured
    PriceGuard memory guard = priceGuard[feedId];
    if (guard.min != 0 || guard.max != 0) {
        require(data.price >= guard.min && data.price <= guard.max, PriceOutOfGuard(feedId, data.price));
    }
}
```

### Proof of Concept

1. **Creator** calls `setPriceGuard(feedId, 1_000e8, 2_000e8)` — expecting prices to be bounded between $1 000 and $2 000.
2. **Authorized pusher** (via `allowPushers` or `allowContractPushers`) constructs a slot word encoding a U64x32 price of `50_000e8` (far above the guard max) with a timestamp strictly greater than the current stored timestamp.
3. Pusher calls `CompressedOracle.fallback()` with that slot word. The monotonicity check passes; the slot is written. No price-range check occurs.
4. **Pool** calls `price(feedId, pool)` → `_price()` → `getOracleData()`. The function returns `mid = 50_000e8` without consulting `priceGuard`.
5. `AnchoredPriceProvider._computeBidAsk()` builds its safety band around the corrupted anchor mid. The resulting bid/ask is shifted by ~25×.
6. A swap executes at the inflated ask: the trader pays far less input than the true market rate, draining the pool's token reserves beyond what LP shares can cover — pool insolvency. [6](#0-5) [7](#0-6) [8](#0-7)
