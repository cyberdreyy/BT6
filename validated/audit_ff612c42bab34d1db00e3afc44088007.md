### Title
Silent Namespace Hijack via `allowContractPushers` Overwrites Existing Delegation Without Prior Creator's Consent — (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowContractPushers` grants a **permanent, unbounded write delegation** to a contract pusher based solely on a point-in-time `isPusher(creator)` consent check. Because there is no exclusive-ownership guard on the `namespaceRemapping` slot, any second creator B can silently overwrite Creator A's existing delegation if the pusher contract returns `isPusher(B) = true`. After the overwrite, A's feeds stop being updated, pools consuming A's oracle receive stale prices, and bad-price execution becomes reachable.

---

### Finding Description

`allowContractPushers` (lines 217–234) sets `namespaceRemapping[pusher] = msg.sender` unconditionally after a live `isPusher(msg.sender)` check:

```solidity
function allowContractPushers(address[] calldata pushers) external {
    ...
    (bool ok, bytes memory res) = pusher.staticcall(
        abi.encodeWithSignature("isPusher(address)", msg.sender)
    );
    require(ok);
    bool allowed = abi.decode(res, (bool));
    require(allowed);

    namespaceRemapping[pusher] = msg.sender;   // ← unconditional overwrite
    emit PusherAuthorized(pusher, msg.sender);
}
``` [1](#0-0) 

There is **no check** that `namespaceRemapping[pusher]` is already zero (or already equals `msg.sender`). Any creator B whose address satisfies `pusher.isPusher(B) == true` can call this function and atomically redirect the pusher's future writes from Creator A's namespace to their own.

This is the direct analog to the external bug's unlimited-approval pattern: the delegation is granted once, persists indefinitely with no expiry, and can be silently overwritten by any actor the pusher contract considers authorized — without Creator A's knowledge or consent.

The EOA path (`allowPushers`) avoids a related problem by requiring a **deadline** in the signed message:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The contract-pusher path carries the same structural weakness in a different form: the consent is checked live, but the resulting delegation slot has no exclusive-ownership invariant, so a second authorized caller can overwrite it at any time.

---

### Impact Explanation

After Creator B overwrites `namespaceRemapping[P] = B`:

- All subsequent fallback pushes from P land in **B's namespace**, not A's.
- A's feeds receive **no further updates** — they become permanently stale from A's perspective.
- Any `AnchoredPriceProvider` or `PriceProvider` bound to A's feed IDs will return the last stale mid/spread values.
- Pools executing swaps against those providers consume a **stale bid/ask quote**, satisfying the "bad-price execution" impact gate: traders can receive more than the oracle/bin curve permits, or the pool fails to receive the owed input amount.

---

### Likelihood Explanation

The attack requires a pusher contract P whose `isPusher()` function returns `true` for more than one creator address. This is realistic for:

- **Shared oracle adapters** (e.g., a single Chainlink or Pyth relay contract that multiple protocol teams register against).
- **Upgradeable proxies** whose `isPusher()` logic is later broadened by an admin.
- **Public/permissive pusher contracts** that return `true` for any caller (analogous to `MockPusherAllowed` in the test suite).

The trigger is fully **unprivileged relative to Creator A**: Creator B needs no special role in A's namespace — only a pusher contract that consents to B.

---

### Recommendation

Add an exclusive-ownership guard before writing `namespaceRemapping`:

```solidity
address existing = namespaceRemapping[pusher];
if (existing != address(0) && existing != msg.sender) {
    revert PusherAlreadyDelegated(pusher, existing);
}
namespaceRemapping[pusher] = msg.sender;
```

This ensures an existing delegation can only be overwritten by the **same creator** who established it, or after the previous creator has explicitly revoked via `removePushers`. Alternatively, mirror the EOA path and require the original creator to call `removePushers` before a new creator can claim the pusher.

---

### Proof of Concept

```
Setup:
  - CompressedOracle deployed
  - SharedPusherContract P: isPusher(A) = true, isPusher(B) = true
  - Creator A calls allowContractPushers([P])
    → namespaceRemapping[P] = A  ✓

Attack:
  - Creator B calls allowContractPushers([P])
    → isPusher(B) = true  → passes
    → namespaceRemapping[P] = B  (A's delegation silently overwritten)

Result:
  - P's next fallback push resolves namespace = B → writes to B's slots
  - A's feedId(A, slotId, posId) receives no further updates
  - AnchoredPriceProvider reading A's feed returns last stale price
  - Pool swap against A's provider executes at stale bid/ask
    → swap conservation failure / bad-price execution
```
