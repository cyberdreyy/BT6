### Title
`allowPushers` re-establishes a revoked delegation using a still-valid deadline signature, redirecting the pusher's feed writes and causing dependent pools to halt — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` contains no guard against re-establishing a delegation that the pusher has already revoked. A creator who retains a still-valid (deadline not yet expired) consent signature can call `allowPushers` a second time after the pusher calls `revokePusher()`, silently overwriting `namespaceRemapping[pusher]` back to the creator's address. The pusher's subsequent `fallback()` pushes then land in the creator's namespace instead of the pusher's own namespace, causing the pusher's own feeds to go stale and halting every pool that reads those feeds.

---

### Finding Description

The NatSpec comment on `allowPushers` explicitly acknowledges the risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The comment claims the deadline is the mitigation. It is not. `_ensureDeadline` only checks `block.timestamp <= deadline` — it prevents using the signature *after* the deadline expires, but it does nothing to prevent re-use *during* the deadline window after the pusher has revoked. [1](#0-0) 

The write is unconditional once the signature passes:

```solidity
namespaceRemapping[pusher] = msg.sender;   // no check: was this pusher already revoked?
``` [2](#0-1) 

`_ensureDeadline` only enforces expiry, not revocation state: [3](#0-2) 

`revokePusher` clears the mapping to `address(0)`, but nothing prevents the creator from writing it back: [4](#0-3) 

The `fallback()` push path resolves the namespace from `namespaceRemapping[msg.sender]`, so after the re-establishment the pusher's writes silently land in the creator's namespace again: [5](#0-4) 

---

### Impact Explanation

Once the delegation is re-established without the pusher's knowledge:

1. Every `fallback()` call from the pusher writes into the **creator's** namespace, not the pusher's own namespace.
2. The pusher's own feeds (in the pusher's namespace) receive no further updates — their `timestampMs` freezes.
3. `AnchoredPriceProvider._readLeg` checks staleness against `MAX_REF_STALENESS`: [6](#0-5) 

4. Once the pusher's feeds age past `MAX_REF_STALENESS`, every `getBidAndAskPrice()` call on any pool backed by those feeds reverts with `FeedStalled`, making swaps and liquidity operations permanently unusable until the pusher notices and re-routes their pushes.

This matches the allowed impact gate: **broken core pool functionality causing unusable swap/liquidity flows**.

---

### Likelihood Explanation

- The pusher must have signed a consent with a deadline that has not yet expired at the time of the attack. Long-lived deadlines (days to months) are operationally common for automated pusher bots.
- The creator needs only to replay the original `allowPushers` calldata — no new signature is required.
- The pusher has no on-chain signal that the delegation was re-established; the only observable effect is that their own feeds silently go stale.
- The creator is a semi-trusted party (not a factory owner or oracle admin), so this is an unprivileged-enough trigger under the contest scope.

---

### Recommendation

Add a per-pusher revocation nonce and include it in the EIP-191 signature domain. Increment the nonce in `revokePusher` and `removePushers`. This invalidates all previously signed consents after revocation:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers signature domain:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
```

Alternatively, add an explicit check `require(namespaceRemapping[pusher] == address(0), AlreadyDelegated())` before writing, which prevents any re-delegation without a fresh revocation-and-re-consent cycle.

---

### Proof of Concept

```
1. Pusher P signs consent for Creator A:
   sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+365days, P, A)))

2. Creator A calls allowPushers(T+365days, [P], [sig])
   → namespaceRemapping[P] = A  ✓

3. Pusher P calls revokePusher()
   → namespaceRemapping[P] = address(0)  ✓ (P believes they are free)

4. Creator A calls allowPushers(T+365days, [P], [sig])  ← SAME calldata, deadline still valid
   → _ensureDeadline passes (block.timestamp < T+365days)
   → ECDSA.recover passes (same valid signature)
   → namespaceRemapping[P] = A  ← re-established without P's knowledge

5. P's fallback() pushes now land in A's namespace.
   P's own feeds (feedIdOf(P, slotIndex, positionIndex)) receive no updates.

6. After MAX_REF_STALENESS seconds, AnchoredPriceProvider._readLeg returns ok=false.
   getBidAndAskPrice() reverts FeedStalled on every swap attempt.
   Pool is bricked until P manually detects and re-routes.
``` [7](#0-6) [4](#0-3) [8](#0-7)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
