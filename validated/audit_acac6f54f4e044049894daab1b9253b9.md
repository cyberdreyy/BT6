After a thorough review of all oracle provider, compression, pusher delegation, codebook, timestamp, and abuse-protection code in the Metric OMM codebase, I traced every decimal-scaling operation against the bug class from the external report.

**Key scaling paths examined:**

1. **ChainlinkOracle `_toMid8()`** — divides 18-decimal Data Streams price by `PRICE_SCALE = 1e10` to produce 8-decimal `OracleData.price`. `18 − 10 = 8`. Correct. [1](#0-0) [2](#0-1) 

2. **`_spreadFromBidAsk()`** — `half` and `price` are both 18-decimal; the ratio `BPS_BASE * half / price` is dimensionless bps. Correct. [3](#0-2) 

3. **LazerConsumer `_normalize()`** — `totalExpo = expo + TARGET_DECIMALS (8)` rescales the feed's native price to 8 decimals. Spread uses `BPS_BASE * conf / pU`; both `conf` and `pU` are in the feed's native scale, so the ratio is dimensionless. Correct. [4](#0-3) 

4. **PriceProvider `_applyStepAdjustment()`** — `STEP_DENOM = ORACLE_DECIMALS (1e8) × BPS_BASE_U (1e18) = 1e26` correctly cancels both the oracle decimal scale and the 1e18-scale step factor, yielding a Q64 output. Correct. [5](#0-4)

    function _spreadFromBidAsk(int192 price, int192 bid, int192 ask) private pure returns (uint16) {
        require(price > 0 && ask >= bid, InvalidReportPrice());
        uint256 half = uint256(int256(ask) - int256(bid)) / 2;
        uint256 spread = Math.ceilDiv(uint256(BPS_BASE) * half, uint256(int256(price)));
