### Title
Signature Replay in `allowPushers` Lets Creator Permanently Undo Pusher Revocation Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` signs pusher consent with no nonce. The same EIP-191 signature is valid for every call within the deadline window. After a pusher calls `revokePusher()`, the creator can immediately replay the original `allowPushers` calldata to re-establish the delegation, nullifying the revocation. The pusher has no on-chain remedy until the deadline expires.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no consumed-flag, and no per-call counter. The code's own comment acknowledges the deadline is the only replay guard:

> "the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

`revokePusher` zeroes `namespaceRemapping[msg.sender]`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But `allowPushers` unconditionally overwrites it back:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [4](#0-3) 

Because the signature is stateless, the creator can call `allowPushers` again with the identical `(deadline, pushers, signatures)` tuple at any point before `deadline`, re-establishing the mapping the pusher just cleared. This is the direct analog of the CREATE2 salt-reuse bug: the same parameters produce the same outcome, and the victim's corrective action is silently undone.

---

### Impact Explanation

`namespaceRemapping[pusher]` determines which storage namespace receives every push from that address:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

If the pusher is a legitimate price-feed bot whose own namespace (`feedIdOf(pusher, slot, pos)`) is consumed by a pool oracle, and the creator replays the delegation after revocation, all subsequent pushes land in the creator's namespace instead. The pusher's own namespace receives no new data, its stored timestamp goes stale, and any pool reading that feed will see a stale price — matching the "bad-price execution: stale bid/ask quote reaches a pool swap" impact class. Alternatively, if the pool reads the creator's namespace, the creator can keep a pusher they no longer trust feeding live prices into that namespace, enabling price manipulation.

---

### Likelihood Explanation

The creator already possesses the original `allowPushers` calldata (they submitted it). No additional privilege, off-chain data, or special timing is required — they simply resubmit the same transaction after the pusher's `revokePusher` is mined. The window is as wide as the deadline the pusher originally signed (commonly hours to days). The trigger is a valid semi-trusted action (creator re-calling a public function with a still-valid signature), not a privileged admin operation.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` set. Mark the digest as used on first acceptance and revert on any subsequent call with the same digest:

```solidity
mapping(bytes32 => bool) private _usedDelegationHashes;

// inside allowPushers, after recovering the signer:
require(!_usedDelegationHashes[hash], "signature already used");
_usedDelegationHashes[hash] = true;
```

This mirrors the recommended mitigation in the external report (ever-increasing nonce counter) and ensures that a pusher's revocation is permanent regardless of the deadline.

---

### Proof of Concept

```
1. pusher signs: keccak256(abi.encode(chainid, oracle, deadline=T+1day, pusher, creator))
   → sig

2. creator calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)

4. creator calls allowPushers(T+1day, [pusher], [sig])   // identical calldata, no revert
   → namespaceRemapping[pusher] = creator  ← revocation silently undone

5. pusher's next push lands in creator's namespace, not pusher's own.
   Pools reading feedIdOf(pusher, slot, pos) now see a stale timestamp → bad-price path.
```

Steps 3→4 can be repeated indefinitely until `block.timestamp > T+1day`, giving the creator full control over the pusher's namespace routing for the entire deadline window despite the pusher's explicit revocation.
