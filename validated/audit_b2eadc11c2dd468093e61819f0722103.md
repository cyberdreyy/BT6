### Title
`revokePusher` / `removePushers` delegation cleanup is bypassable via signature replay — creator can silently re-establish a revoked pusher's write authority over their namespace feed data - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` accepts a pusher's EIP-191 consent signature that is bound to `(chainid, oracle_address, deadline, pusher, creator)` but carries **no nonce and no used-signature flag**. After a pusher calls `revokePusher()` (or a creator calls `removePushers()`), the creator can immediately replay the original, still-valid signature to re-establish the delegation — without any fresh consent from the pusher. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the deadline window, and the pusher's fallback pushes continue to land in the creator's namespace rather than their own.

### Finding Description

`allowPushers` hashes and verifies the pusher's consent as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no per-signature consumed flag, and no check that `namespaceRemapping[pusher]` is currently `address(0)` before writing. The only replay guard is the deadline, which only prevents using an *expired* signature — it does not prevent replaying a still-valid signature after revocation.

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

But it does not invalidate the pusher's original signature. The creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple — the signature still verifies, and `namespaceRemapping[pusher]` is set back to `creator`. This cycle can repeat until the deadline expires.

The code's own comment acknowledges the risk but incorrectly claims the deadline is sufficient:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The deadline limits the *window* of replay but does not close it. Within that window the pusher's revocation is completely ineffective.

### Impact Explanation

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

After the creator replays the signature, every push the pusher makes — including pushes the pusher believes are going to their own namespace — lands in the creator's namespace. If the creator's namespace feeds a production pool via an `AnchoredPriceProvider`, the pool's `getQuotes()` call consumes those writes as the live bid/ask. A pusher who has revoked because their key is compromised, their data is stale, or they are no longer a valid data source continues to write into the creator's production feed, delivering bad prices to live swaps.

### Likelihood Explanation

The creator holds the original signature (they submitted it in the first `allowPushers` call). Replaying it requires a single on-chain transaction with no additional off-chain material. The pusher has no on-chain mechanism to prevent this within the deadline window. Deadlines are typically set days in the future (the test suite uses `block.timestamp + 1 days`), giving the creator a large replay window. [5](#0-4) 

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` set. Alternatively, check that `namespaceRemapping[pusher] == address(0)` before accepting a delegation call, so a revoked pusher must produce a fresh signature to be re-delegated:

```solidity
require(namespaceRemapping[pusher] == address(0), AlreadyDelegated());
```

This ensures that once a pusher revokes, the creator cannot silently re-establish the delegation without the pusher signing a new consent message.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — no fresh consent needed
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs); // succeeds with old sig
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation silently restored

// 5. Pusher's next fallback push lands in creator's namespace, not their own
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(999_999, 2, 2); // attacker-chosen price
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);
// Creator's namespace (which feeds the pool) now contains the pusher's data
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
         U64x32.decode(uint32(raw >> 16)));
// Pusher's own namespace is empty — pusher had no idea
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0)).price, 0);
```
