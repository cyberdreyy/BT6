### Title
`allowPushers` Consent Signature Has No Post-Revocation Replay Guard — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` performs only a deadline check and an ECDSA signature verification. It contains no nonce, no used-signature bitmap, and no check against the pusher's current revocation state. After `revokePusher()` clears `namespaceRemapping[pusher]`, the creator can immediately call `allowPushers` again with the identical signature and the same still-valid deadline, unconditionally re-writing `namespaceRemapping[pusher] = creator`. The REVOCATION_FINALITY_INVARIANT is violated.

### Finding Description

`allowPushers` is the only EOA-pusher delegation entry point:

```solidity
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);                          // only: block.timestamp <= deadline
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;            // unconditional write
    emit PusherAuthorized(pusher, msg.sender);
}
``` [1](#0-0) 

`revokePusher` only zeroes the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

`_ensureDeadline` is a single timestamp comparison with no state mutation:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [3](#0-2) 

A grep across all `.sol` files for `usedSignature`, `signatureUsed`, `nonce`, `revokedPusher`, or `REVOKED` returns zero hits in the oracle contracts — there is no post-revocation replay guard anywhere. [1](#0-0) 

The code comment itself acknowledges the design intent:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The deadline limits the *window* but does not prevent replay *within* that window. A signature signed for `deadline = now + 1 day` remains cryptographically valid for the full 24 hours regardless of how many times `revokePusher()` is called in between.

**Exact replay sequence:**

1. Pusher signs consent: `sig = sign(chainid, oracle, deadline=D+1day, pusher, creator)`
2. Creator calls `allowPushers(D+1day, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`
4. Creator calls `allowPushers(D+1day, [pusher], [sig])` again (same calldata, deadline still valid) → `namespaceRemapping[pusher] = creator` again

Step 4 succeeds unconditionally. The pusher's self-revocation is overridden.

### Impact Explanation

After step 4, any fallback push from the pusher's address is attributed to the creator's namespace via `namespaceRemapping`. The `feedIdOf(creator, slotIndex, positionIndex)` storage key is overwritten. Every pool or provider reading that feed via `getOracleData` / `price` receives the corrupted value. [5](#0-4) 

The concrete bad-price path: if the pusher's key was compromised and the pusher revoked to protect the creator's namespace, the creator (unaware of the compromise) re-establishes delegation; the attacker holding the compromised key then pushes crafted slot words into the creator's namespace, producing an arbitrary mid price consumed by pools at swap time.

### Likelihood Explanation

Medium-low. Requires the creator to call `allowPushers` a second time after the pusher revoked. This is a realistic operational mistake (e.g., automated re-delegation scripts, or a creator who does not monitor `PusherRevoked` events). The pusher has no on-chain mechanism to prevent it.

### Recommendation

Track each consumed consent by its hash and revert on reuse:

```solidity
mapping(bytes32 => bool) private _usedConsents;

function allowPushers(...) external {
    ...
    bytes32 hash = ...;
    require(!_usedConsents[hash], "consent already used");
    _usedConsents[hash] = true;
    namespaceRemapping[pusher] = msg.sender;
}
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful delegation, so a revoked pusher can issue a new consent only with a fresh nonce.
