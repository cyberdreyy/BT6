### Title
Cross-Leg Timestamp Inconsistency in Synthetic Ratio Mode Feeds Bad Price to Pool — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (`quoteFeedId != 0`) where the mid price is computed as `price(baseFeedId) / price(quoteFeedId)`. Each leg is staleness-checked independently against the single `MAX_REF_STALENESS` bound, but there is **no cross-leg timestamp consistency check**. The two legs can therefore be read from prices that are up to `MAX_REF_STALENESS` apart in time. The resulting synthetic ratio is computed from temporally inconsistent data, producing a bad price that reaches the pool swap.

---

### Finding Description

In `_getBidAndAskPrice`, the two legs are read sequentially via `_readLeg`:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
...
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
...
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // ratio from two independent timestamps
spreadBps += spreadBps2;
``` [1](#0-0) 

Inside `_readLeg`, the only temporal guard is a per-leg staleness check against `block.timestamp`:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (..., false);
``` [2](#0-1) 

The `refTime` returned by each leg is **never compared against the other leg's `refTime`**. Both legs independently pass their staleness check as long as each is within `MAX_REF_STALENESS` of `block.timestamp`. The constructor allows `MAX_REF_STALENESS` up to 7 days:

```solidity
if (_maxRefStaleness > 7 days) revert MaxRefStalenessOutOfBounds();
``` [3](#0-2) 

This is the direct analog to the IonPool bug: just as `_accrueInterestForIlk` updates shared state (`supplyFactor`) for one ILK while leaving others stale — so that a later accrual for a second ILK applies the already-shifted `supplyFactor` to a different time window — here the synthetic ratio applies a fresh base-leg price against a stale quote-leg price (or vice versa), producing a ratio that does not correspond to any real market moment.

---

### Impact Explanation

The pool executes swaps at the synthetic bid/ask derived from this ratio. If the base feed was updated at time T and the quote feed was last updated at time T − Δ (where Δ ≤ `MAX_REF_STALENESS`), the synthetic mid is:

```
mid_synthetic = price_base(T) / price_quote(T − Δ)
```

A trader who observes that one leg is stale can swap at the stale synthetic price and immediately arbitrage the difference against the real market, extracting value from LPs. The maximum mispricing is bounded by how much the quote asset moved during Δ, which for a 7-day staleness window on a volatile asset can be tens of percent. This constitutes **bad-price execution** and **direct loss of LP principal**.

---

### Likelihood Explanation

The oracle's pusher infrastructure (CompressedOracleV1 / pusher delegation) pushes updates per feed-id independently. A pusher that updates only the base feed — whether due to a network partition, a selective push, or simply different update cadences for the two feeds — naturally creates the temporal gap. No privileged or malicious setup is required; the gap arises from normal independent feed operation. Any swap executed while the gap exists hits the bad price.

---

### Recommendation

After reading both legs, compare their `refTime` values and reject if the inter-leg gap exceeds a configurable `MAX_LEG_DRIFT` (e.g., a small multiple of the expected push interval):

```solidity
(uint256 mid,  uint256 spreadBps,  uint256 refTime1, bool ok1) = _readLeg(baseFeedId);
(uint256 mid2, uint256 spreadBps2, uint256 refTime2, bool ok2) = _readLeg(_quote);

// Cross-leg consistency: reject if legs are too far apart in time.
uint256 legDrift = refTime1 > refTime2 ? refTime1 - refTime2 : refTime2 - refTime1;
if (legDrift > MAX_LEG_DRIFT) return (0, type(uint128).max);
```

`MAX_LEG_DRIFT` should be set as an immutable at construction, much smaller than `MAX_REF_STALENESS`.

---

### Proof of Concept

1. Deploy `AnchoredPriceProvider` in synthetic mode (`quoteFeedId != 0`) with `MAX_REF_STALENESS = 1 hour`.
2. Push a fresh update to `baseFeedId` (e.g., BTC/USD = 100,000) at block timestamp T.
3. Do **not** push an update to `quoteFeedId` (ETH/USD); its last stored price is 2,000 from T − 59 minutes.
4. Both legs pass `_isStale` independently (each within 1 hour of T).
5. Synthetic mid = 100,000 / 2,000 = 50 BTC/ETH.
6. Actual ETH/USD at T = 2,500 → actual BTC/ETH = 40.
7. Trader calls `swap` on the pool, buying BTC at the stale synthetic price of 50 (25% below real value), immediately selling on a real market at 40, extracting the 25% difference from LP reserves. [1](#0-0) [2](#0-1) [3](#0-2)

        (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
        if (!ok) return (0, type(uint128).max);

        bytes32 _quote = quoteFeedId;
        if (_quote != bytes32(0)) {

            mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
            spreadBps += spreadBps2;
        }

    }

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
```
