### Title
Pusher consent signature has no invalidation mechanism, allowing creator to replay it and override `revokePusher()` within the deadline window — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts a deadline-bounded EIP-191 signature as the pusher's consent to be delegated into a creator's namespace. The signature is never invalidated after first use and carries no nonce. Within the deadline window, the creator can replay the identical signature any number of times to re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher calls `revokePusher()`, permanently overriding the pusher's revocation until the deadline expires.

---

### Finding Description

`allowPushers` constructs the consent hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

and verifies `pusher == ECDSA.recover(hash, signatures[i])`, then unconditionally writes:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no consumed-signature registry, and no guard that checks whether the pusher is already delegated or has previously revoked. The same `(chainid, oracle, deadline, pusher, creator)` tuple produces the same hash every time it is presented, so the signature remains valid for the entire deadline window.

When the pusher calls `revokePusher()`, `namespaceRemapping[pusher]` is cleared to `address(0)`: [2](#0-1) 

The creator can immediately call `allowPushers` again with the exact same bytes to restore `namespaceRemapping[pusher] = creator`. This cycle repeats indefinitely until the deadline timestamp passes.

The code's own NatSpec comment acknowledges the problem but claims the deadline solves it:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The claim is incorrect: the deadline only prevents replay *after* it expires. Within the window it provides no protection against re-establishment after revocation.

The analog to the KintoID bug is exact: in KintoID the nonce is read from `nonces[signer]` but incremented for `nonces[account]`, so the same signature is accepted repeatedly. Here there is no nonce at all, so the same signature is accepted repeatedly for the full deadline duration.

---

### Impact Explanation

**Namespace hijack / feed-attribution DoS.** The `fallback` push path resolves the effective namespace as:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

If the creator keeps the pusher re-delegated against the pusher's will, every push the pusher makes lands in the creator's namespace, not the pusher's own or a new creator's namespace. Two concrete loss paths follow:

1. **Stale-feed DoS on a pool.** The pusher revokes to stop contributing to creator C1 and re-delegates to creator C2 (whose feeds a production pool reads). C1 replays the old signature to override `namespaceRemapping[pusher] = C2` back to C1. The pusher, unable to push to C2's namespace, stops pushing entirely. C2's feeds go stale. Any pool whose `PriceProvider` reads C2's `CompressedOracleV1` feeds will have its `getSafePrice` call revert on the staleness check, making swaps and liquidity operations permanently unusable until the deadline expires — matching the "Broken core pool functionality" impact gate.

2. **Forced attribution of live price data.** If the pusher continues pushing (unaware of the re-delegation), their data lands in C1's namespace. C1 can use this to keep a production feed alive with data the pusher no longer intends to provide to C1, bypassing the pusher's explicit revocation of write authority.

---

### Likelihood Explanation

The attack requires:
- A creator who obtained a valid pusher consent signature (normal operational step)
- The pusher to subsequently call `revokePusher()` (a normal operational step)
- The creator to replay the old signature before the deadline expires (trivially automatable)

No privileged role, no malicious initial setup, and no non-standard token behavior is required. The creator is a semi-trusted party (namespace owner), not a fully trusted admin, so this is an unprivileged-relative-to-the-pusher attack path. The deadline window in production is expected to be hours to days, giving ample time to execute the replay.

---

### Recommendation

Add a per-pusher nonce that is incremented on every successful `allowPushers` call and included in the signed hash:

```solidity
mapping(address => uint256) public pusherNonces;

// in allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonces[pusher]++
    ))
);
```

Alternatively, record consumed signatures in a `mapping(bytes32 => bool) usedHashes` and revert on reuse. Either approach ensures that a revoked pusher's old consent cannot be replayed to re-establish delegation.

---

### Proof of Concept

```
1. Pusher signs: sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+1day, pusher, C1)))
2. C1 calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = C1  ✓
3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓
4. C1 calls allowPushers(T+1day, [pusher], [sig])  ← SAME sig, still valid
   → namespaceRemapping[pusher] = C1  (revocation overridden)
5. Pusher calls revokePusher() again → cleared
6. C1 replays again → re-delegated
   ... repeats until block.timestamp > T+1day
7. During this window, pusher cannot push to own namespace or C2's namespace;
   any push lands in C1's namespace. If pusher stops pushing, C2's feeds go stale
   and pools consuming C2's feeds cannot execute swaps.
```
