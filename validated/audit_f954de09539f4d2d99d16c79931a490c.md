### Title
`revokePusher()` does not invalidate the pusher's signed consent — creator can silently re-establish delegation after self-revocation, redirecting pushes into the creator's namespace and feeding bad prices to pools - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`revokePusher()` clears `namespaceRemapping[pusher]` but does not invalidate the pusher's previously signed EIP-191 consent. A creator who holds an unexpired consent signature can immediately call `allowPushers` again with the same signature to re-establish the delegation, bypassing the pusher's revocation. If the pusher is now pushing data intended for their own namespace (a different asset/context), that data is silently redirected into the creator's namespace, feeding wrong prices to any pool that relies on the creator's `feedId`.

---

### Finding Description

The `allowPushers` function requires a pusher's EIP-191 signature as proof of consent. The signed message is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The signature is bound to `(chainid, oracle, deadline, pusher, creator)`. There is no nonce, no revocation registry, and no per-pusher invalidation mechanism.

When a pusher calls `revokePusher()`, the function only clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

The signed consent is not invalidated. As long as the deadline has not expired, the creator can call `allowPushers` again with the exact same signature to re-establish `namespaceRemapping[pusher] = creator`.

The code comment acknowledges this partially:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

However, the deadline only prevents re-establishment **after** the deadline expires. During the entire deadline window (which can be days or weeks), the creator can re-establish the delegation at any time after the pusher revoked — the pusher has no on-chain mechanism to prevent this.

Once re-established, the `fallback()` push path resolves the namespace from `namespaceRemapping[msg.sender]`:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So the pusher's subsequent pushes — now intended for their own namespace (a different asset) — are silently written into the creator's namespace instead.

---

### Impact Explanation

A pool registered for `feedIdOf(creatorA, slot, pos)` reads prices from creator A's namespace. If creator A re-establishes pusher P's delegation after P revoked, P's pushes (now intended for a different asset in P's own namespace) land in creator A's namespace. The pool receives wrong prices — e.g., BTC/USD prices on an ETH/USD feed — and executes swaps at those prices, causing direct loss of user principal through bad-price execution.

The `AnchoredPriceProvider` staleness, spread, and price-guard checks do not protect against this: P's data is fresh, has valid spread indexes, and may pass any price guard set before the attack. The wrong price reaches `_computeBidAsk` and is returned as a valid bid/ask to the pool. [5](#0-4) 

---

### Likelihood Explanation

- The creator is a semi-trusted actor (not a protocol admin) who set up the feed the pool relies on.
- The creator already holds the pusher's signed consent (they used it to establish the delegation originally).
- Pushers commonly sign consents with multi-day or multi-week deadlines to avoid frequent re-signing.
- The pusher has no on-chain way to invalidate the old signature before the deadline expires.
- The attack requires no additional privileged access beyond what the creator already possesses.

---

### Recommendation

Add a per-pusher revocation nonce. Include the nonce in the signed message hash:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher:
namespaceRemapping[msg.sender] = address(0);
pusherNonce[msg.sender]++;  // invalidates all prior signed consents
```

Alternatively, store a per-pusher revocation timestamp and reject any `allowPushers` call whose deadline predates the last revocation.

---

### Proof of Concept

```
1. Pusher P signs consent for creator A:
   sig = sign(keccak256(chainid, oracle, deadline=T+30days, P, A))

2. Creator A calls allowPushers(T+30days, [P], [sig])
   → namespaceRemapping[P] = A

3. P calls revokePusher()
   → namespaceRemapping[P] = 0
   P now pushes BTC/USD prices to their own namespace.

4. Creator A calls allowPushers(T+30days, [P], [sig])  ← SAME signature, still valid
   → namespaceRemapping[P] = A  (re-established without P's current consent)

5. P pushes a BTC/USD slot word via fallback():
   → namespace resolves to A (not P)
   → BTC/USD price written to feedIdOf(A, slot, pos)

6. Pool registered for feedIdOf(A, slot, pos) (expecting ETH/USD) calls getBidAndAskPrice()
   → AnchoredPriceProvider reads BTC/USD price from creator A's namespace
   → Staleness/spread/guard checks pass (data is fresh and valid)
   → Pool executes ETH/USD swap at BTC/USD price → loss of user funds
``` [6](#0-5) [2](#0-1) [7](#0-6)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

    fallback() override external {
        uint256 end;
        uint256 namespace;

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

        assembly ("memory-safe") {
            end := calldatasize()
            namespace := shl(96, creator) // [creator:20][zeros:12]
        }

        // 4 * 6 + 7 + 1 = 32 bytes per slot
        if (end == 0 || end % 32 != 0) revert BadCalldataLength();

        for (uint256 ptr = 0; ptr < end; ptr += 32) {
            uint256 word;
            assembly ("memory-safe") {
                word := calldataload(ptr)
            }
            // casting to 'uint8' is safe we want LSB
            // forge-lint: disable-next-line(unsafe-typecast)
            uint8 slotId = uint8(word);
            TimeMs timestampMs = toTimeMs(word >> 8 & X56);
            timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
            bytes32 key = bytes32(namespace | uint256(slotId));
            uint256 old = uint256(_loadStorage(key));
            TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

            bool newer = timestampMs.isAfter(oldTimestampMs);
            if (!newer) continue;

            _writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
        }

        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
