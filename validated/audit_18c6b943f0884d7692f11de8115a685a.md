### Title
Pusher consent signature replay allows creator to re-instate delegation after pusher self-revokes — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but never invalidates it after first use. A creator can replay the same signature — within the deadline window — to re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher has self-revoked via `revokePusher()`. This is the direct analog of the `confidentialApprove` bug: the revocation state is not bound into the signature, so the same bytes that originally granted delegation can restore it after revocation.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The digest binds: `chainid`, `address(this)`, `deadline`, `pusher`, and `creator` (`msg.sender`). It does **not** bind any revocation nonce, a "grant vs revoke" action flag, or a used-signature registry. There is no state that marks a signature as consumed.

`revokePusher()` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

But the original consent bytes remain cryptographically valid until `deadline` passes. The creator can immediately call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, and the contract re-writes `namespaceRemapping[pusher] = creator`, silently undoing the revocation.

The code comment itself acknowledges the risk but only partially addresses it:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The deadline caps the *outer* replay window but does nothing to prevent replay *within* that window. A pusher who signs a 7-day consent and revokes on day 1 remains captive for the remaining 6 days.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after the creator's replay lands in the creator's namespace, not the pusher's own. The pusher cannot reclaim their own namespace until the deadline expires or they stop pushing entirely.

---

### Impact Explanation

**Oracle namespace hijack / bad-price execution path:**

1. A pusher is a legitimate off-chain data provider. They revoke delegation because they no longer consent to feed the creator's namespace (e.g., the creator is using their data for a pool the pusher considers unsafe).
2. The creator replays the original consent signature. `namespaceRemapping[pusher]` is restored to `creator`.
3. Every subsequent fallback push by the pusher — intended for the pusher's own feeds — is silently redirected into the creator's namespace.
4. The creator's oracle slots receive live price updates from a pusher who explicitly withdrew consent, and the pusher's own feeds receive nothing.
5. Any `AnchoredPriceProvider` or pool consuming the creator's `CompressedOracleV1` feeds now reads prices the pusher did not intend to authorize, potentially stale or mismatched relative to the pusher's current view.

The pusher can stop pushing entirely as a last resort, but that leaves the creator's namespace with the last-written (potentially stale) values — which consumers will eventually reject as stale — while the pusher's own namespace is also starved. The pusher has no on-chain remedy until the deadline expires.

---

### Likelihood Explanation

- The creator already holds the pusher's consent signature (they submitted it in the original `allowPushers` call).
- Replaying it costs one transaction and zero additional off-chain work.
- The window is as long as the deadline the pusher originally signed — commonly hours to days in oracle delegation flows.
- No privileged access is required beyond being the creator who originally called `allowPushers`.

---

### Recommendation

**Short term:** Track consumed signatures with a `mapping(bytes32 => bool) usedConsents` and revert if the digest has already been accepted. A single `sstore` per delegation prevents all within-window replays.

**Medium term:** Include a per-pusher revocation nonce in the signed digest — `keccak256(abi.encode(chainid, address(this), deadline, pusher, creator, revocationNonce[pusher]))` — and increment `revocationNonce[pusher]` inside `revokePusher()`. Any previously issued signature becomes invalid the moment the pusher revokes.

**Long term:** Adopt a pull-consent model: the pusher must submit an on-chain transaction to activate delegation rather than signing an off-chain message that the creator can replay.

### Proof of Concept

```
// Step 1: pusher signs consent for creator, deadline = now + 7 days
bytes memory sig = pusher.sign(keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)));

// Step 2: creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator  ✓

// Step 3: pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0)  ✓

// Step 4: creator replays the IDENTICAL signature — no new pusher action required
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);   // same deadline, same sig
// namespaceRemapping[pusher] == creator  ← revocation silently undone

// Step 5: pusher's next fallback push lands in creator's namespace, not pusher's own
vm.prank(pusher);
address(oracle).call(slotWord);
assertEq(oracle.getOracleData(feedIdOf(creator, slot, pos)).price, pushedPrice);  // creator gets it
assertEq(oracle.getOracleData(feedIdOf(pusher,   slot, pos)).price, 0);           // pusher's own is empty
```
