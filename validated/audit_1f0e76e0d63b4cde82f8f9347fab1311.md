### Title
Creator Can Permanently Re-Trap a Pusher After Self-Revocation via Signature Replay in `allowPushers` — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts a pusher's EIP-191 consent signature but neither tracks used signatures nor invalidates them when the pusher calls `revokePusher()`. A malicious creator who holds a valid, unexpired signature can replay it immediately after every self-revocation, keeping the pusher permanently redirected into the creator's namespace until the deadline expires. While trapped, the pusher's own oracle feeds receive no updates, go stale, and any pool or price-provider that reads those feeds cannot execute swaps.

---

### Finding Description

`allowPushers` signs over `(block.chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

There is no nonce, no per-signature consumed flag, and no revocation timestamp. The only freshness gate is `_ensureDeadline`: [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` back to `address(0)`: [3](#0-2) 

Because the signature is not consumed or invalidated on revocation, the creator can immediately replay the identical bytes to `allowPushers` and restore `namespaceRemapping[pusher] = creator`: [4](#0-3) 

This cycle — pusher revokes → creator replays → pusher revokes → … — can repeat at zero cost until the deadline timestamp passes. The `fallback` push path reads `namespaceRemapping[msg.sender]` and routes every push word into the creator's namespace instead of the pusher's own: [5](#0-4) 

So while trapped, every byte the pusher sends enriches the creator's feeds and leaves the pusher's own slot storage at price `0` / timestamp `0`.

---

### Impact Explanation

A never-pushed or stale feed returns `price = 0`, `timestampMs = 0`. Every consumer (price providers, `AnchoredPriceProvider`, pool swap path) already rejects timestamp-zero data as stale. Any pool whose `IPriceProvider` is backed by the pusher's own feed IDs will therefore be unable to obtain a valid bid/ask quote, making swaps revert for the entire duration of the griefing window. This satisfies the **broken core pool functionality / stale bid/ask reaching a pool swap** impact gate.

---

### Likelihood Explanation

The creator must have obtained a valid signature from the pusher — which is the normal, expected flow when a pusher willingly delegates. Once the creator holds that signature, the replay requires only a single on-chain call per revocation cycle, costs only gas, and can be scripted. The pusher has no on-chain escape until the deadline expires; they cannot shorten the deadline retroactively. Likelihood is **Medium**: the attacker must be a creator who was legitimately delegated to, but the exploit itself is trivially automatable.

---

### Recommendation

Add a per-pusher revocation nonce or a consumed-signature bitmap. The simplest fix is a `uint256 public revokeNonce` mapping per pusher address, incremented on every `revokePusher()` call, and included in the signed digest:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusher, msg.sender, revokeNonce[pusher]))
```

After `revokePusher()` increments `revokeNonce[pusher]`, all previously issued signatures become cryptographically invalid, mirroring the FrankenDAO recommendation of an emergency-eject with a cooldown.

---

### Proof of Concept

```
t=0   Pusher signs consent: sig = sign(chainid, oracle, deadline=t+1day, pusher, creator)
t=1   Creator calls allowPushers(deadline, [pusher], [sig])
        → namespaceRemapping[pusher] = creator  ✓

t=2   Pusher calls revokePusher()
        → namespaceRemapping[pusher] = address(0)  ✓

t=3   Creator calls allowPushers(deadline, [pusher], [sig])   // SAME sig, deadline still valid
        → namespaceRemapping[pusher] = creator  ✓  (pusher re-trapped)

t=4   Pusher pushes slot word → lands in CREATOR namespace, own namespace stays at price=0/ts=0
      Pool reads feedIdOf(pusher, slot, pos) → timestampMs=0 → staleness check fails → swap reverts

      ... cycle repeats until block.timestamp > deadline ...
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
