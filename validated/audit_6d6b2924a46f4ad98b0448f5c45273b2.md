### Title
`revokePusher()` Is Ineffective Against Creator Signature Replay — Pusher's Own Namespace Feeds Become Permanently Stale Until Deadline Expiry - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` stores no nonce and marks no signature as consumed. After a pusher calls `revokePusher()` to restore their own namespace, the creator can immediately replay the original EIP-191 consent signature (which remains valid until its deadline) to re-establish the delegation. The pusher is trapped: every push they make lands in the creator's namespace, their own namespace feeds go stale, and any pool anchored to those feeds halts with `FeedStalled`.

---

### Finding Description

**`allowPushers` signature domain and storage write:**

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)`, which checks `block.timestamp <= deadline`. [2](#0-1) 

**`revokePusher` clears the mapping but cannot invalidate the signature:**

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

There is no nonce, no per-`(pusher, creator)` used-deadline tracking, and no on-chain record that the pusher revoked. The creator holds the original bytes and can call `allowPushers` again in the same block as the revocation, writing `namespaceRemapping[pusher] = creator` back.

**The code's own comment acknowledges the partial problem but does not close it:**

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The deadline prevents *indefinite* replay, but within the deadline window the creator can replay the signature an unlimited number of times, negating every `revokePusher` call the pusher makes.

**Fallback namespace resolution — the pusher's pushes keep landing in the creator's namespace:**

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
...
namespace := shl(96, creator)
``` [5](#0-4) 

While the delegation is active the pusher cannot write to `feedIdOf(pusher, slotIndex, positionIndex)`. Those slots retain their last-written timestamp (or zero if never pushed). The slot-level timestamp is the sole freshness signal consumed by every price provider.

**Price provider staleness gate — stale pusher feeds halt pools:**

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [6](#0-5) 

```solidity
function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
``` [7](#0-6) 

Any pool whose `IPriceProvider` is bound to a feed in the pusher's own namespace will revert `FeedStalled` on every swap once the feed ages past `MAX_REF_STALENESS`.

---

### Impact Explanation

A pusher who operates both as a delegated price updater for creator C and as an independent feed operator for their own pools calls `revokePusher()` intending to stop serving C and resume updating their own namespace. The creator replays the original consent signature before the deadline, re-establishing the delegation. The pusher's pushes continue to land in C's namespace; the pusher's own namespace feeds go stale. Every pool anchored to `feedIdOf(pusher, *, *)` halts with `FeedStalled`, making swaps, liquidity additions, and withdrawals that depend on a live price quote unusable for the remainder of the deadline window — potentially up to the maximum deadline the pusher signed.

---

### Likelihood Explanation

The creator holds the pusher's signed bytes off-chain from the original `allowPushers` call. Monitoring the chain for `PusherRevoked` events and immediately re-submitting the same calldata is a single transaction requiring no special privilege. The window of exposure equals the remaining time until the deadline the pusher originally signed, which could be hours or days.

---

### Recommendation

Track a per-`(pusher, creator)` revocation nonce or a "minimum acceptable deadline" that advances on each successful `revokePusher`. Reject any `allowPushers` call whose deadline does not exceed the stored minimum:

```solidity
mapping(address => uint256) public pusherRevokedAt; // pusher => timestamp of last revoke

function revokePusher() external {
    ...
    pusherRevokedAt[msg.sender] = block.timestamp;
    namespaceRemapping[msg.sender] = address(0);
}

// In allowPushers, after _ensureDeadline:
require(deadline > pusherRevokedAt[pusher], "signature predates revocation");
```

This ensures a signature signed before the pusher's last revocation cannot be replayed.

---

### Proof of Concept

```
T=0:  Pusher P signs consent:
        sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+1day, P, C)))

T=1:  Creator C calls allowPushers(T+1day, [P], [sig])
        → namespaceRemapping[P] = C
        → P's pushes land in C's namespace

T=2:  P calls revokePusher()
        → namespaceRemapping[P] = address(0)
        → P pushes to own namespace: feedIdOf(P, 0, 0) updated ✓

T=3:  C calls allowPushers(T+1day, [P], [sig])  ← SAME signature, still before deadline
        → namespaceRemapping[P] = C  (re-established)
        → P's next push lands in C's namespace again

T=4+: P's own namespace feeds (feedIdOf(P, *, *)) receive no further updates.
      After MAX_REF_STALENESS seconds, every pool using those feeds reverts FeedStalled.
      C can repeat step T=3 on every block until T+1day.
```

The `testRevokePusherRestoresOwnNamespace` test confirms revocation works in isolation but does not test the creator-replay scenario, leaving the invariant unverified. [8](#0-7)
