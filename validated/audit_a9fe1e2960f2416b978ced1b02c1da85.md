### Title
Stale Updater Permissions Persist After Provider Ownership Transfer, Enabling Unauthorized Confidence Manipulation and Pool DoS — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `smart-contracts-poc/contracts/AnchoredProviderFactory.sol`)

### Summary
`transferProviderOwnership` updates `providerOwner` and `_providersByCreator` but never clears the `isUpdater` mappings for the previous owner's granted updaters. After transfer, those stale updaters retain the ability to call `setConfidence` on the provider, allowing them to set `confidenceParam = 0` and halt pool swaps via `FeedStalled`.

### Finding Description
In both `PriceProviderFactory` and `AnchoredProviderFactory`, `transferProviderOwnership` performs three state mutations: [1](#0-0) 

```solidity
providerOwner[provider] = newOwner;
_providersByCreator[previousOwner].remove(provider);
_providersByCreator[newOwner].add(provider);
```

It does **not** clear `isUpdater[provider][*]` for any updater the previous owner had granted. The factory's `_requireUpdater` check passes for any address where `isUpdater[provider][addr] == true`, regardless of who the current owner is: [2](#0-1) 

So after ownership transfer, old updaters can still call `factory.setConfidence([provider], [value])` successfully, because `_requireUpdater` still sees their stale `isUpdater` entry as `true`. [3](#0-2) 

The same flaw exists in `AnchoredProviderFactory`: [4](#0-3) [5](#0-4) 

### Impact Explanation
A stale updater sets `confidenceParam = 0` on a `PriceProvider` with `marginStep = 0`. Inside `_getBidAndAskPrice`: [6](#0-5) 

With `confidenceParam = 0`: `adjustedSpread = 0`, so `bid = mid` and `ask = mid`. After `_applyBidAdjustments` / `_applyAskAdjustments` with `marginStep = 0`, `stepBidFactor = stepAskFactor = BPS_BASE_U`, yielding `bidOut == askOut`. The guard `if (bidOut >= askOut)` fires and returns `(0, type(uint128).max)`. The pool's `getBidAndAskPrice()` then reverts with `FeedStalled`, making all swaps through that pool unusable until the new owner discovers the stale updater, revokes them, and resets confidence — a broken core pool functionality impact. [7](#0-6) 

### Likelihood Explanation
Medium. The previous owner must have granted at least one updater before transferring ownership, and the new owner must not immediately enumerate and revoke all existing updaters. There is no on-chain mechanism or event that alerts the new owner to inherited updater permissions. The `CONFIDENCE_COOLDOWN` (1 minute) limits attack frequency but does not prevent it. [8](#0-7) 

### Recommendation
In `transferProviderOwnership`, either:
1. Enumerate and delete all `isUpdater[provider][*]` entries for the provider before updating `providerOwner`, or
2. Key updater authorization on `(provider, owner, updater)` so that permissions are implicitly scoped to the current owner and become void on transfer.

At minimum, emit an event listing all active updaters at transfer time so the new owner can revoke them.

### Proof of Concept
1. Alice calls `PriceProviderFactory.createPriceProvider(...)` → receives `provider`.
2. Alice calls `grantUpdater(provider, Bob)` → `isUpdater[provider][Bob] = true`.
3. Alice calls `transferProviderOwnership(provider, Carol)` → `providerOwner[provider] = Carol`; `isUpdater[provider][Bob]` is **not** cleared.
4. Bob (no longer authorized by Carol) calls `factory.setConfidence([provider], [0])`.
   - `_requireUpdater`: `msg.sender != Carol` but `isUpdater[provider][Bob] == true` → passes.
   - `PriceProvider.setConfidenceParam(0)` executes; `confidenceParam = 0`.
5. Any pool using `provider` calls `getBidAndAskPrice()`:
   - `adjustedSpread = spread * 0 = 0` → `bid = ask = mid` → `bidOut >= askOut` → returns `(0, type(uint128).max)`.
   - Pool reverts with `FeedStalled`; all swaps are blocked. [9](#0-8) [10](#0-9)
