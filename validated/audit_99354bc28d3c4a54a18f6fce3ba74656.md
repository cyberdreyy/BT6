### Title
Signature Replay in `allowPushers` Allows Creator to Silently Re-Establish Revoked Delegation Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` has no nonce or used-signature tracking. The only replay guard is `_ensureDeadline`, which only rejects signatures whose deadline has already passed. A creator who holds a pusher's original consent signature can call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple after the pusher has called `revokePusher()`, re-writing `namespaceRemapping[pusher] = creator` without any fresh consent from the pusher.

---

### Finding Description

The signed consent message is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no per-pusher revocation counter, and no "signature consumed" bitmap. The only guard is:

```solidity
_ensureDeadline(deadline);
``` [2](#0-1) 

`revokePusher` clears the mapping to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But it does not invalidate the original signature. Because the signature is stateless and the deadline has not expired, the creator can immediately call `allowPushers` again with the same arguments, and the `require(pusher == ECDSA.recover(...))` check passes identically, writing `namespaceRemapping[pusher] = creator` again.

The code comment on `allowPushers` explicitly acknowledges the concern:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The comment identifies the exact attack but the chosen mitigation (deadline) is insufficient: it only prevents replaying an *expired* signature, not replaying a *still-valid* one after revocation.

The `fallback` push path reads `namespaceRemapping[msg.sender]` at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every fallback push the pusher makes after the silent re-delegation lands in the creator's namespace, not the pusher's own.

---

### Impact Explanation

After the pusher self-revokes and begins pushing into their own namespace, the creator silently re-routes those writes back into the creator's namespace. The pusher's own namespace stays empty (their intended destination), while the creator's namespace receives data the pusher did not intend to provide there. If the creator's namespace feeds back a pool's `AnchoredPriceProvider`, those feeds receive pusher data under a namespace the pusher explicitly withdrew consent from, which can produce stale, misattributed, or manipulated oracle quotes reaching live pool swaps.

---

### Likelihood Explanation

The attack requires only that the deadline on the original consent signature has not yet expired. Consent signatures are typically issued with multi-hour or multi-day deadlines (the test suite uses `block.timestamp + 1 days`). [6](#0-5) 

The creator holds the signature (they received it to call `allowPushers` the first time), so no additional capability is needed. The replay is a single public transaction.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) usedConsents` keyed on the signature hash (or a per-pusher nonce). In `allowPushers`, after recovering the signer, assert `!usedConsents[sigHash]` and then set `usedConsents[sigHash] = true`. Alternatively, include a per-pusher monotonic nonce in the signed payload and increment it on each successful delegation, making every prior signature invalid after use.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME sig + deadline (still valid) — no revert
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));

// 5. Delegation silently re-established without pusher's fresh consent
assertEq(oracle.namespaceRemapping(pusher), creator); // REVOCATION_FINALITY_INVARIANT violated

// 6. Pusher's next fallback push lands in creator's namespace, not their own
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, _packRaw(999_000, 1, 1), uint56(block.timestamp * 1000)));
assertTrue(ok);
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, U64x32.decode(999_000 >> 16));
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher,  0, 0)).price, 0); // pusher's own ns empty
```
