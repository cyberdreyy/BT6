### Title
`purgeStateGuardRole` Does Not Clear `pendingStateGuard`, Allowing a Stale Pending Guard to Claim Feed Authority - (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary

`purgeStateGuardRole` deletes `stateGuard[feedId]` but leaves `pendingStateGuard[feedId]` intact. A previously-nominated pending guard can call `acceptStateGuardRole` after the active guard has been purged, silently claiming authority over a feed's price-guard configuration.

### Finding Description

In `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, two cleanup functions exist for the two-step guard-transfer flow:

- `purgePendingStateGuardRole(feedId)` — deletes `pendingStateGuard[feedId]` only.
- `purgeStateGuardRole(feedId)` — deletes `stateGuard[feedId]` only. [1](#0-0) 

`acceptStateGuardRole` correctly clears `pendingStateGuard` when a pending guard accepts: [2](#0-1) 

But `purgeStateGuardRole` — the function intended to strip the active guard and return authority to ADMIN — does **not** clear `pendingStateGuard`. After it executes, `stateGuard[feedId] == address(0)` (authority reverts to ADMIN), yet `pendingStateGuard[feedId]` still holds the previously-nominated address. That address can immediately call `acceptStateGuardRole` and become the new active guard, bypassing the ADMIN's expectation that the feed's guard state was fully reset.

This is the direct analog of the external M-02 bug: `timeUnlockERC721` cleaned `timelockERC721s` while `transferERC721` did not; here, `acceptStateGuardRole` cleans `pendingStateGuard` while `purgeStateGuardRole` does not.

The compressed oracle's `OracleBase` does not expose `purgeStateGuardRole` at all, so the bug is isolated to the providers oracle path. [3](#0-2) 

### Impact Explanation

The `stateGuard` for a feed controls:
1. `setPriceGuard` — sets the `(min, max)` price bounds that clamp raw oracle output before it reaches `AnchoredPriceProvider` and pool swaps.
2. `setStateGuardRole` / `purgePendingStateGuardRole` — controls future guard nominations.

A stale pending guard that accepts after a purge can call `setPriceGuard(feedId, 0, type(uint128).max)`, effectively disabling the price clamp. With the guard removed, stale, inverted, or otherwise bad oracle prices can propagate unclamped into pool bid/ask quotes, satisfying the "bad-price execution" and "pool insolvency" impact criteria. [4](#0-3) 

### Likelihood Explanation

The trigger requires the current guard (or ADMIN, when no guard is set) to call `purgeStateGuardRole` while a `pendingStateGuard` entry is live. This is a realistic operational mistake: an admin who wants to cancel a pending nomination may call `purgeStateGuardRole` instead of `purgePendingStateGuardRole` (the names are similar and the distinction is non-obvious). Alternatively, a guard who is rotating out can intentionally nominate a colluding address before purging themselves. The stale pending guard then calls `acceptStateGuardRole` — a permissionless call — to complete the takeover.

### Recommendation

Add a `delete pendingStateGuard[feedId]` inside `purgeStateGuardRole` so that purging the active guard atomically cancels any outstanding nomination:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];
    delete pendingStateGuard[feedId];   // ← add this line
    emit StateGuardDeleted(feedId);
}
```

This mirrors the cleanup already performed by `acceptStateGuardRole` and closes the stale-state window.

### Proof of Concept

```
1. ADMIN calls setStateGuardRole(feedId, guardA)
   → pendingStateGuard[feedId] = guardA

2. ADMIN calls purgeStateGuardRole(feedId)   // intending a full reset
   → stateGuard[feedId]  = address(0)        // cleared
   → pendingStateGuard[feedId] = guardA       // NOT cleared ← bug

3. guardA calls acceptStateGuardRole(feedId)
   → require(pendingStateGuard[feedId] == msg.sender) passes
   → stateGuard[feedId] = guardA             // guardA is now the guard

4. guardA calls setPriceGuard(feedId, 0, type(uint128).max)
   → price clamp disabled; any oracle price passes through to pool swaps
``` [5](#0-4)
