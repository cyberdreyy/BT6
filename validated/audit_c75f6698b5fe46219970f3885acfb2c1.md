### Title
`PriceProvider` with negative `marginStep` always reverts with `FeedStalled()` at default `confidenceParam = 0`, making pool swaps permanently unusable â€” (`smart-contracts-poc/contracts/PriceProvider.sol`)

### Summary

A `PriceProvider` deployed with `marginStep < 0` is immediately broken at deployment because the zero-initialized default `confidenceParam = 0` causes `bidOut > askOut`, triggering the inversion guard and returning the stall sentinel on every call, even when the oracle has a valid, fresh price.

### Finding Description

In `PriceProvider._getBidAndAskPrice()`, the bid/ask computation proceeds as follows: [1](#0-0) 

When `confidenceParam = 0` (the zero-initialized storage default), `adjustedSpread = spread * 0 = 0`. `_getBidAskFrom(mid, 0)` then returns `bid = ask = mid` (delta is zero): [2](#0-1) 

The two step factors are set at construction: [3](#0-2) 

When `marginStep < 0`:
- `stepBidFactor = BPS_BASE_U - marginStep` **> BPS_BASE_U** (subtracting a negative value)
- `stepAskFactor = BPS_BASE_U + marginStep` **< BPS_BASE_U**

Since `bid = ask = mid`, applying the larger `stepBidFactor` to bid and the smaller `stepAskFactor` to ask yields `bidOut > askOut`. The hard invariant check then fires: [4](#0-3) 

This returns `(0, type(uint128).max)`, causing `getBidAndAskPrice()` to revert with `FeedStalled()`: [5](#0-4) 

The code comment at line 227 explicitly acknowledges this: *"Can be violated when marginStep < 0 and confidence is too small."* â€” but provides no constructor guard to prevent deploying into this broken state.

The constructor only bounds `marginStep` to the open interval `(-BPS_BASE, BPS_BASE)`, so any negative value is accepted: [6](#0-5) 

### Impact Explanation

Every swap through a pool whose price provider is a `PriceProvider` with `marginStep < 0` reverts with `FeedStalled()` from the moment of deployment until the factory calls `setConfidenceParam` with a sufficiently large value. LPs who deposit liquidity into such a pool cannot exit via swaps. This is broken core pool functionality â€” unusable swap/liquidity flows â€” matching the allowed impact gate.

### Likelihood Explanation

Negative `marginStep` is an explicitly documented and constructor-accepted deployment choice (it tightens the pre-clamp shaped quote). Any integrator or factory operator who deploys with `marginStep < 0` and relies on the pool being immediately usable will find it permanently stalled. The default `confidenceParam = 0` is the trigger; no attacker action is required â€” the broken state is the initial state.

### Recommendation

Add a constructor check that rejects the combination of `marginStep < 0` and `confidenceParam = 0` being the only valid initial state, or require `confidenceParam` to be passed as a constructor argument (with a minimum floor) when `marginStep < 0`. At minimum, document the invariant as a hard deployment requirement and enforce it in the factory's deployment path.

```solidity
// In constructor, after setting marginStep:
if (_marginStep < 0) {
    // A negative marginStep inverts bid/ask when confidenceParam == 0.
    // Require a non-zero initial confidenceParam to be set before deployment.
    revert NegativeMarginStepRequiresConfidence();
}
```
