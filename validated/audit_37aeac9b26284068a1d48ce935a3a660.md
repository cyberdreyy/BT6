### Title
`allowContractPushers` Missing Existing-Mapping Guard Enables Silent Namespace Hijacking — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `allowContractPushers` function does not check whether a contract pusher is already mapped to a different creator. Any caller for whom the pusher contract's `isPusher()` returns `true` can silently overwrite an existing delegation, redirecting all future price pushes to their own namespace and causing the original creator's feeds to go permanently stale, making every pool that depends on those feeds revert with `FeedStalled` on every swap.

---

### Finding Description

The EOA delegation path (`allowPushers`) requires a pusher-signed EIP-191 message that encodes the specific creator (`msg.sender`), the deadline, the chain id, and the oracle address: [1](#0-0) 

This signature binding ensures a pusher can only be mapped to the one creator they explicitly consented to at signing time. The deadline further prevents a stale consent from re-establishing a delegation after the pusher revoked it.

The contract delegation path (`allowContractPushers`) replaces the signature with a live `isPusher(creator)` staticcall: [2](#0-1) 

Critically, **there is no check that `namespaceRemapping[pusher]` is already set to a different creator before overwriting it**. If a contract pusher's `isPusher()` returns `true` for more than one caller — a realistic scenario for shared infrastructure contracts, multi-sig-controlled pushers, or any contract that authorises a set of creators — any of those callers can call `allowContractPushers` at any time and silently overwrite the existing mapping.

This is the direct structural analog to the external bug: just as `shareKey` and `grantKey` bypass the `notSoldOut` cap check that `purchase` enforces via the `notSoldOut` modifier, `allowContractPushers` bypasses the creator-binding invariant that `allowPushers` enforces via the EIP-191 signature. In both cases, a shared helper (`_assignNewTokenId` / `namespaceRemapping[pusher] = msg.sender`) executes an unchecked state mutation that a parallel path guards correctly.

The comment in the code acknowledges the asymmetry but does not address the missing guard: [3](#0-2) 

---

### Impact Explanation

Once the mapping is overwritten:

1. All future pushes from the contract pusher land in the attacker's namespace, not the original creator's.
2. The original creator's feeds receive no further updates and become stale.
3. `PriceProvider._getBidAndAskPrice` calls `_isStale` and returns `(0, type(uint128).max)`. [4](#0-3) 

4. `ProtectedPriceProvider.getBidAndAskPrice` reverts with `FeedStalled`. [5](#0-4) 

5. Every swap, liquidity add, and withdrawal that triggers a price read on the affected pool reverts permanently. The pool is bricked for as long as the hijack persists.

This satisfies the impact gate: **broken core pool functionality causing unusable swap/liquidity flows**.

---

### Likelihood Explanation

The trigger is fully unprivileged: any EOA can call `allowContractPushers`. The only precondition is that the targeted contract pusher's `isPusher(attacker)` returns `true`. This is realistic for:

- Shared infrastructure pushers that authorise a set of creators (e.g., a DAO-controlled pusher).
- Contract pushers whose `isPusher` logic is permissive or has a separate bug.
- Any pusher contract that the attacker themselves deployed and can make return `true` for themselves.

The test suite already demonstrates a `MockPusherAllowed` that returns `true` for any caller, confirming the scenario is within the design space of the system: [6](#0-5) 

---

### Recommendation

Add an existing-mapping guard inside the `allowContractPushers` loop before writing `namespaceRemapping[pusher]`:

```solidity
address existing = namespaceRemapping[pusher];
if (existing != address(0) && existing != msg.sender) {
    revert AlreadyMapped(pusher, existing);
}
```

This mirrors the protection that the EIP-191 signature provides in `allowPushers`: a pusher can only be claimed by one creator at a time, and changing that mapping requires the pusher's explicit consent (either a new signature or the pusher calling `revokePusher` first).

---

### Proof of Concept

```
1. Creator A deploys contract pusher P where P.isPusher(creatorA) == true
   AND P.isPusher(creatorB) == true (shared/permissive pusher).

2. creatorA calls allowContractPushers([P])
   → namespaceRemapping[P] = creatorA  ✓

3. P begins pushing price data; feeds land in creatorA's namespace.
   Pools are configured with PriceProvider pointing at feedIdOf(creatorA, slotId, posId).

4. creatorB calls allowContractPushers([P])
   → isPusher(creatorB) returns true
   → namespaceRemapping[P] = creatorB  (overwrites, no revert)

5. All subsequent pushes from P land in creatorB's namespace.
   creatorA's feeds receive zero further updates.

6. After MAX_TIME_DELTA seconds, PriceProvider._isStale returns true.
   _getBidAndAskPrice returns (0, type(uint128).max).
   getBidAndAskPrice reverts with FeedStalled.

7. Every pool swap/liquidity operation that calls getBidAndAskPrice reverts.
   Pools are permanently unusable until creatorA finds an alternative pusher.
``` [7](#0-6) [8](#0-7)

    function allowContractPushers(address[] calldata pushers) external {
        uint256 l = pushers.length;
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            (bool ok, bytes memory res) = pusher.staticcall(abi.encodeWithSignature("isPusher(address)", msg.sender));
            require(ok);
            bool allowed = abi.decode(res, (bool));
            require(allowed);

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
