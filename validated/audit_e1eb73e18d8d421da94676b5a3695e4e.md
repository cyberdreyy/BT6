### Title
Pusher self-revocation is silently bypassed by deadline-window signature replay — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`revokePusher()` clears `namespaceRemapping[msg.sender]` but does not invalidate the pusher's previously signed consent. Because `allowPushers` has no nonce or used-signature tracking, the creator can immediately replay the original EIP-191 signature — as long as its `deadline` has not yet expired — to re-establish the delegation. The code comment at lines 186-191 explicitly acknowledges the deadline is the only guard against post-revocation replay, but the deadline only blocks replay *after* it expires, not within the window.

---

### Finding Description

`allowPushers` hashes `(chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's signature against that hash: [1](#0-0) 

No nonce, no used-signature bitmap, and no per-pusher revocation epoch is stored. `revokePusher` only zeroes the mapping: [2](#0-1) 

The code comment directly above `allowPushers` states the deadline is required precisely to prevent re-establishing delegation after revocation: [3](#0-2) 

But `_ensureDeadline` only checks `block.timestamp <= deadline`: [4](#0-3) 

Within the deadline window the same bytes are a valid, replayable credential. The protection the developers intended is incomplete.

---

### Impact Explanation

The `fallback` push path resolves the namespace from `namespaceRemapping[msg.sender]`: [5](#0-4) 

If a pusher's private key is compromised, the pusher calls `revokePusher()` to stop the attacker from writing bad prices into the creator's namespace. The creator — unaware of the compromise — calls `allowPushers` again with the same cached signature to "restore" the delegation. The attacker (holding the compromised key) immediately resumes pushing manipulated slot words into the creator's namespace. Any `AnchoredPriceProvider` or `PriceProvider` reading that namespace will serve the corrupted mid/spread to a live pool swap, producing bad-price execution and potential swap conservation failure.

---

### Likelihood Explanation

Pusher keys are off-chain hot wallets used by automated bots. Key compromise is a realistic operational event. Creators routinely cache delegation signatures for operational convenience. The window between a typical delegation deadline (e.g., 1 day, 7 days) and the moment the creator learns of a compromise is wide enough for the attacker to push many bad price updates. The creator's re-delegation call requires no special privilege — it is a normal public transaction.

---

### Recommendation

Track a per-pusher revocation epoch or a used-signature set. The simplest fix is to add a `mapping(address => uint256) public pusherNonce` incremented by `revokePusher()`, and include `pusherNonce[pusher]` in the signed digest inside `allowPushers`. After revocation the old signature (which committed to the previous nonce) becomes permanently invalid, matching the protection the comment describes.

---

### Proof of Concept

```
// 1. Pusher signs consent with deadline = T + 1 day.
bytes memory sig = pusher.sign(keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)));

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator  ✓

// 3. Pusher's key is compromised; pusher self-revokes.
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0)  ✓

// 4. Creator (unaware) replays the SAME sig — deadline still valid.
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);   // succeeds — no revert
// namespaceRemapping[pusher] == creator  ← revocation silently undone

// 5. Attacker (with compromised pusher key) pushes a manipulated price.
vm.prank(attacker_with_pusher_key);
(bool ok,) = address(oracle).call(maliciousSlotWord);
// Price lands in creator's namespace → consumed by price provider → bad swap execution.
```

The existing test suite confirms `revokePusher` clears the mapping but never tests that the old signature is rejected afterward: [6](#0-5)

        oracle.revokePusher();
