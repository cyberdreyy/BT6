### Title
Pusher Consent Signature Replay Allows Creator to Nullify `revokePusher` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks the signature as consumed. A creator can replay the same valid signature after the pusher calls `revokePusher`, immediately re-establishing the delegation the pusher just cancelled. The pusher's only escape is to wait for the original deadline to expire.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

After verifying the signature it writes `namespaceRemapping[pusher] = msg.sender` and emits an event. There is no mapping of consumed signature hashes, no per-pusher nonce, and no state flag that records "this consent was already used." [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because the original signature is still cryptographically valid (same `chainid`, same `address(this)`, same `deadline`, same `pusher`, same `creator`), the creator can immediately call `allowPushers` again with the identical bytes, pass the `_ensureDeadline` check, pass the `ECDSA.recover` check, and overwrite `namespaceRemapping[pusher]` back to themselves. [4](#0-3) 

The code comment acknowledges the deadline is meant to prevent exactly this scenario ("an undated signature could re-establish a delegation AFTER the pusher revoked it"), but the deadline only prevents re-establishment *after* the deadline timestamp — within the deadline window the replay is unconditionally accepted. [5](#0-4) 

---

### Impact Explanation

The `namespaceRemapping` table determines which storage namespace a pusher's `fallback()` writes land in. If a pusher's key is compromised, or the pusher decides to stop providing prices, they call `revokePusher`. A creator who holds the original consent bytes can replay it in the same block, restoring the mapping. The pusher's prices — potentially stale, wrong, or from a compromised key — continue to flow into the creator's namespace and into every pool that reads from it via `getOracleData` / `price`. This satisfies the "bad-price execution" and "broken core pool functionality" impact gates.

---

### Likelihood Explanation

The attack requires only that the creator saved the original `allowPushers` calldata (trivially available from the mempool or chain history). No special privilege is needed beyond being the creator who originally called `allowPushers`. The pusher has no on-chain recourse until the deadline passes. Deadlines are typically set days or weeks in the future, so the window is large.

---

### Recommendation

Track consumed consent hashes in a `mapping(bytes32 => bool) private _usedConsents` and revert if the hash was already used:

```solidity
mapping(bytes32 => bool) private _usedConsents;

// inside allowPushers, after ECDSA.recover succeeds:
require(!_usedConsents[hash], "consent already used");
_usedConsents[hash] = true;
```

Alternatively, include a per-pusher nonce in the signed message and increment it on every successful `allowPushers` call, so any previously issued signature becomes invalid after first use.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator, deadline = now + 30 days.
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — no revert, delegation restored.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // re-delegated despite revoke

// 5. Pusher's fallback pushes now land in creator's namespace again,
//    feeding potentially bad prices into any pool reading that namespace.
```
