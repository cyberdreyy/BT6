### Title
Permissionless `register()` Unconditionally Clears Admin Blacklist, Defeating the Emergency Oracle-Read Stop — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary

`OracleBase.register()` is intentionally permissionless: any caller may pay `registrationFee` (defaulting to 1 wei) to whitelist a valid pool for a feed. The function also unconditionally clears any existing blacklist on the pool. Because the blacklist is the only on-chain mechanism the admin has to stop a pool from consuming oracle prices during a security incident, any unprivileged actor can instantly neutralize that emergency stop by calling `register()` with 1 wei, re-enabling oracle reads and swaps for the blacklisted pool.

### Finding Description

`OracleBase.register()` at lines 201–214 of `OracleBase.sol` performs two actions atomically:

1. It whitelists `pool` for `feedId` in `registeredPool`.
2. It clears `blacklisted[pool]` if it was set.

```solidity
// OracleBase.sol L201-214
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

    if (blacklisted[pool]) {
        blacklisted[pool] = false;          // ← clears admin emergency stop
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

The `registrationFee` defaults to `1 wei` (set in the constructor at line 53). The function has no `onlyRole` guard. The only validation is that `factory` is approved and `factory.isPool(pool)` returns true — both of which remain true for a legitimately deployed pool even during a security incident.

The `price()` function enforces the blacklist at lines 163 and 167:

```solidity
// OracleBase.sol L160-172
function price(bytes32 feedId, address pool)
    external
    feedExists(feedId)
    notBlacklisted          // ← checks msg.sender (the price provider)
    returns (...)
{
    require(!blacklisted[pool], Blacklisted(pool));   // ← checks the pool
    require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
    (mid, spread, spread1, refTime) = _readPrice(feedId);
    emit PriceRead(pool, feedId);
}
```

When the admin calls `setBlacklist(pool, true)` to halt a pool's oracle reads during an incident, any actor can immediately call `register(feedId, pool, approvedFactory)` with 1 wei to clear `blacklisted[pool]` and restore `registeredPool[feedId][pool] = true`, fully re-enabling the pool's swap path.

The test `test_register_thirdParty_redeems_andEnablesRead` in `OracleBaseAbuseProtection.t.sol` explicitly confirms this behavior is reachable by an unprivileged third party.

### Impact Explanation

The blacklist is the admin's sole on-chain lever to stop a pool from reading oracle prices and executing swaps during a security incident (e.g., oracle manipulation, pool exploit, or strategy drain). Because `register()` is permissionless and clears the blacklist unconditionally, the emergency stop provides no real security guarantee: an attacker or any third party can restore the pool's oracle-read access in the same block the admin blacklists it, for a cost of 1 wei. This allows swaps to continue executing at potentially manipulated or stale prices during the incident window, directly enabling loss of user principal and pool assets.

**Impact: High** — the admin's emergency protection mechanism is fully bypassable by an unprivileged actor, allowing continued bad-price swap execution during a security incident.

### Likelihood Explanation

**Likelihood: High.**

- `registrationFee` defaults to 1 wei; the cost to bypass is negligible.
- The call requires no special role, no signature, and no privileged access — only a valid `(feedId, pool, approvedFactory)` triple, all of which are observable on-chain from prior `PoolRegistered` events.
- An attacker actively exploiting a pool has strong incentive to keep swaps running and can front-run or same-block the admin's `setBlacklist` transaction.
- Even if the admin raises `registrationFee` reactively, the attacker only needs to pay the new fee — there is no mechanism to make the blacklist irrevocable.

### Recommendation

Separate the blacklist-clearing logic from the permissionless registration path. Two options:

1. **Remove the automatic blacklist-clear from `register()`**. Require a separate, ADMIN-only `clearBlacklist(address pool)` call. Registration and blacklist recovery become independent operations; the admin retains exclusive control over the emergency stop.

2. **Add an ADMIN-only `forceRegister()` that clears the blacklist**, while the permissionless `register()` reverts if the pool is currently blacklisted (`require(!blacklisted[pool], Blacklisted(pool))`). This preserves the economic recovery path for non-emergency blacklists while making the emergency stop irrevocable by unprivileged actors.

### Proof of Concept

```
1. Admin observes exploit on Pool P; calls oracle.setBlacklist(P, true).
   → blacklisted[P] = true; Pool P's swap() now reverts Blacklisted(P).

2. Attacker (or anyone) calls:
       oracle.register{value: 1}(feedId, P, approvedFactory)
   Preconditions satisfied:
     - msg.value (1 wei) >= registrationFee (1 wei)  ✓
     - approvedFactory is in approvedFactories         ✓  (observable from ApprovedFactoryAdded events)
     - approvedFactory.isPool(P) == true               ✓  (P was legitimately deployed; isPool is immutable)

3. register() executes:
       blacklisted[P] = false;          // emergency stop cleared
       registeredPool[feedId][P] = true;

4. Pool P's swap() → getBidAndAskPrice() → ProtectedPriceProvider → oracle.price(feedId, P)
   now passes all checks and returns live oracle prices.
   Swaps resume during the active security incident.
```

Confirmed by existing test `test_register_thirdParty_redeems_andEnablesRead` in `smart-contracts-poc/test/oracles/OracleBaseAbuseProtection.t.sol` (lines 467–487), which asserts that a third party paying 1 wei clears the blacklist and re-enables reads.
