### Title
Fallback push path in `CompressedOracle` lacks the future-timestamp drift guard enforced by `updateBySignature`, allowing a pusher to permanently lock a feed at a manipulated price — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`updateBySignature` explicitly calls `timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT)` before accepting any slot write. The permissionless fallback push path performs only a monotonicity check ("is the new timestamp newer than the stored one?") with no equivalent drift guard. Any pusher — including a delegated pusher writing into a creator's namespace — can submit a far-future timestamp via the fallback, permanently locking the feed at an arbitrary price. Because the monotonicity check then rejects all subsequent legitimate updates (including those via `updateBySignature`, which cannot exceed `block.timestamp + MAX_TIME_DRIFT`), the feed is irrecoverably frozen. Pools consuming the feed via `ProtectedPriceProvider` receive the locked price as a perpetually "fresh" quote, enabling bad-price execution on every swap.

### Finding Description

`CompressedOracle.updateBySignature` enforces a drift guard at line 285:

```solidity
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
``` [1](#0-0) 

This prevents any signed update from embedding a timestamp beyond `block.timestamp + MAX_TIME_DRIFT`.

The fallback push path, by contrast, is documented as performing only a **"per-word timestamp check → packed slot overwrite if newer"** — a pure monotonicity check with no drift guard: [2](#0-1) 

The control hint for the fallback path explicitly states: *"slot packing and timestamp monotonicity must be airtight"* — confirming that monotonicity is the **only** protection, with no mention of drift clamping. [3](#0-2) 

**Attack flow:**

1. A pusher (any EOA in their own namespace, or a delegated pusher in a creator's namespace) crafts a fallback payload with:
   - `slotId` matching the target feed slot
   - `timestampMs` = `uint56(type(uint56).max)` (e.g., year ~4 million in milliseconds)
   - `price` = attacker-chosen manipulated value
2. The fallback accepts the write: the timestamp is newer than the current stored value, and no drift check is applied.
3. All subsequent `updateBySignature` calls for the same slot fail the monotonicity check — no legitimate timestamp can exceed `block.timestamp + MAX_TIME_DRIFT`, which is far less than `type(uint56).max`.
4. All subsequent fallback pushes for the same slot also fail monotonicity for the same reason.
5. The feed is **permanently frozen** at the attacker-chosen price.
6. The price provider's staleness check (`maxTimeDrift`) does **not** catch this: the stored timestamp appears perpetually "fresh" relative to `block.timestamp`.

The two push paths are the direct analog of `safeTransferFrom` vs. `safeBatchTransferFrom` in the external report: one enforces the drift guard, the other silently omits it.

### Impact Explanation

Pools registered to consume the affected feed via `ProtectedPriceProvider → oracle.price(feedId, pool)` receive the frozen manipulated price on every swap: [4](#0-3) 

Traders execute at a bad bid/ask derived from the locked price, causing direct loss of user principal or protocol fees. The `priceGuard` (min/max price bounds) does not mitigate this if the attacker's chosen price falls within the guard range. The feed cannot be recovered without a contract upgrade or redeployment, as there is no admin override for stored slot data.

### Likelihood Explanation

- **Own-namespace attack**: Any EOA whose namespace feeds are consumed by a registered pool can execute this. The attacker sacrifices their own feed but permanently harms pool users and LPs.
- **Delegated-pusher attack**: A creator who delegates push authority via `allowPushers` or `allowContractPushers` exposes their namespace to this attack. The delegated pusher can lock the creator's feed before the creator can call `removePushers`. This is a semi-trusted but fully reachable path — delegation is explicitly described as "intentionally permissionless." [5](#0-4) 

### Recommendation

Apply the same drift guard to the fallback push path that `updateBySignature` already enforces. After extracting `timestampMs` from each packed word in the fallback handler, add:

```diff
+ timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
  bool newer = timestampMs.isAfter(oldTimestampMs);
  if (!newer) { /* skip or continue */ }
```

This ensures no push path — signed or unsigned — can embed a timestamp beyond `block.timestamp + MAX_TIME_DRIFT`, preserving the monotonicity invariant's usefulness and preventing permanent feed lockout.

### Proof of Concept

```
1. Deploy CompressedOracle. Creator establishes a feed at feedIdOf(creator, slotId, posIdx).
2. Pool registers for this feed: oracle.register{value: fee}(feedId, pool, factory).
3. Attacker (delegated pusher or creator) crafts a fallback payload:
     - slotId = target slot
     - timestampMs = uint56(type(uint56).max)   // far-future timestamp
     - price = manipulated value within priceGuard bounds
4. Attacker calls: address(oracle).call(payload)
   → Fallback accepts: timestamp is newer than stored value, no drift check applied.
5. Attempt legitimate update via updateBySignature with timestamp = block.timestamp + MAX_TIME_DRIFT:
   → isAfter(type(uint56).max) returns false → returns false, no write.
6. Attempt fallback push with timestamp = block.timestamp * 1000:
   → isAfter(type(uint56).max) returns false → no write.
7. Feed is permanently frozen at attacker's price.
8. Pool.swap() → ProtectedPriceProvider.getBidAndAskPrice() → oracle.price(feedId, pool)
   → Returns manipulated price as "fresh" on every swap indefinitely.
```
