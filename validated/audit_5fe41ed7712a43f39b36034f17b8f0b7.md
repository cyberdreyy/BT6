### Title
`allowPushers` Consent Signature Is Replayable After Pusher Self-Revocation, Allowing Bad Prices Into Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts a pusher's EIP-191 consent signature that is bound only to `(chainid, oracle, deadline, pusher, creator)`. There is no nonce and no "used" flag. A creator can replay the same valid (non-expired) signature to re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher calls `revokePusher()`. The documentation explicitly states the deadline is the mechanism that prevents re-establishment after revocation, but this claim is false: the deadline only blocks *expired* signatures, not *valid* ones. Any pusher whose key is compromised cannot achieve a durable revocation while the creator holds a non-expired consent signature, and pools consuming feeds in that namespace are exposed to attacker-controlled prices.

---

### Finding Description

`allowPushers` sets `namespaceRemapping[pusher] = msg.sender` (the creator) after verifying the pusher's EIP-191 signature and that `block.timestamp <= deadline`: [1](#0-0) 

`revokePusher` clears the mapping: [2](#0-1) 

Because `allowPushers` contains no nonce, no per-signature consumed flag, and no check that the pusher has not previously revoked, the creator can call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple at any point before the deadline expires. The mapping is restored to `creator` in the same block as the revocation.

The inline documentation for `allowPushers` states:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

This claim is incorrect. The deadline prevents using *expired* signatures; it does not prevent replaying a *valid* (non-expired) signature. A pusher who signed consent with `deadline = now + 1 year` cannot achieve a durable revocation for the entire remaining year.

The slot-structure documentation confirms that a delegated pusher's writes land in the creator's namespace, not the pusher's own: [4](#0-3) 

---

### Impact Explanation

Once the creator replays the consent signature, the compromised pusher key can push arbitrary slot words into the creator's namespace. Each slot word carries a `U64x32`-encoded price and two `Codebook256` spread indices: [5](#0-4) 

The fallback path overwrites the entire storage slot on every accepted push (timestamp must only be strictly increasing): [6](#0-5) 

A price provider reading from the creator's feed via `price(feedId, pool)` or `getOracleData(feedId)` will decode the attacker-supplied `U64x32` price and `Codebook256` spread indices and return them as the live bid/ask to the pool. The pool's `swap` path consumes this quote directly: [7](#0-6) 

An inverted, extreme, or stale price reaching `MetricOmmPool.swap` causes traders to receive wrong output amounts and LPs to suffer direct principal loss — matching the "bad-price execution" and "pool insolvency" impact categories.

---

### Likelihood Explanation

The trigger is semi-trusted (the creator), not fully unprivileged. However, the scenario is realistic:

- Pusher services commonly sign long-lived consent (days to months) to avoid repeated on-chain setup.
- A creator operating an automated delegation-management bot may re-establish delegation upon detecting a revocation, assuming it was accidental.
- A malicious creator can deliberately replay the signature to weaponize a compromised pusher key against pools that trust their feeds.

The broken invariant is documented as a guarantee ("the deadline prevents re-establishment after revocation"), so integrators and pusher operators have a reasonable expectation that `revokePusher` is durable.

---

### Recommendation

Add a per-pusher revocation nonce or a consumed-signature set. The simplest fix is a `mapping(address pusher => uint256 nonce) public pusherNonce` that the pusher can increment via `revokePusher`, and require the consent signature to commit to the current nonce:

```solidity
// In the signed message:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher:
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

This makes every previously issued consent signature invalid the moment the pusher revokes, regardless of the deadline.

---

### Proof of Concept

```
1. Creator signs nothing; Pusher signs consent:
   sig = EIP191( keccak256(chainid, oracle, deadline=T+365days, pusher, creator) )

2. Creator calls allowPushers(T+365days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher's key is compromised. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (revocation recorded)

4. Creator calls allowPushers(T+365days, [pusher], [sig])  ← SAME sig, same deadline
   → _ensureDeadline passes (deadline still in future)
   → ECDSA.recover returns pusher  ← same hash, same sig
   → namespaceRemapping[pusher] = creator  ← revocation silently undone

5. Attacker (holding compromised pusher key) calls oracle fallback with:
   slotWord = [U64x32(price=0) | s0=0xFF | s1=0xFF | ts=now_ms | slotId=X]
   → timestamp > stored timestamp → accepted
   → creator's slot X overwritten with price=0, spread=10_000

6. Pool calls provider.getBidAndAskPrice()
   → provider calls oracle.price(feedId, pool)
   → returns mid=0, spread=10_000
   → bid/ask computed from corrupted values → bad-price execution
```

    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

## Updates (Fallback / Signature)

Updates overwrite the entire storage slot value.

- `fallback()` accepts one or more 32-byte slot words (no prefix; calldata length must
  be a non-zero multiple of 32).
- `updateBySignature(feedCreator, newSlotValue, signature)` accepts a slot word signed
  by the feed creator over `keccak256(abi.encode(chainid, oracleAddress, feedCreator,
  newSlotValue))`, and is callable by anyone.

There is **no deadline** on either push path: each word carries its own timestamp and
the per-slot monotonicity check neutralizes replay (a replayed word is "not newer" and
is skipped).
