### Title
Pusher Self-Revocation Is Ineffective Within Deadline Window Due to Signature Replay in `allowPushers` — (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

The `allowPushers` delegation signature commits to `(chainid, address(this), deadline, pusher, creator)` but contains **no nonce**. A creator who holds a valid, unexpired signature can replay it to re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher calls `revokePusher()`. The pusher's self-revocation is therefore completely ineffective within the deadline window, and the code comment that claims the deadline prevents this is incorrect.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no one-time-use flag, and no on-chain record that the signature was consumed. The only expiry is `deadline`. The code comment at lines 186–191 explicitly claims the deadline solves the replay-after-revoke problem:

> *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

This claim is **wrong**. The deadline only prevents replay *after* the deadline expires. Within the deadline window the creator can call `allowPushers` with the identical signature an unlimited number of times, each time overwriting the `address(0)` that `revokePusher` wrote:

```solidity
// revokePusher clears the mapping …
namespaceRemapping[msg.sender] = address(0);   // line 241

// … but allowPushers immediately overwrites it with the same old sig
namespaceRemapping[pusher] = msg.sender;        // line 209
``` [3](#0-2) [4](#0-3) 

The fallback push path resolves the effective namespace at push time:

```solidity
// namespace = namespaceRemapping[msg.sender] if set, else msg.sender
``` [5](#0-4) 

So every push the pusher makes after a "successful" revoke still lands in the **creator's namespace** as long as the creator keeps replaying the signature.

---

### Impact Explanation

The `CompressedOracle` is the upstream data source for `AnchoredPriceProvider`, which is the standard provider for public pools. The provider reads `mid`, `spread0/spread1`, and `refTime` from the oracle and uses them to compute the bid/ask band that governs every swap: [6](#0-5) 

Attack scenario that feeds bad prices into a pool:

1. A pusher's off-chain data source becomes compromised or stale. The pusher calls `revokePusher()` to stop writing to the creator's namespace and halt the bad-price flow.
2. The creator (or any party holding the original signature) immediately calls `allowPushers(deadline, [pusher], [oldSig])` — the same signature, no new consent required.
3. `namespaceRemapping[pusher]` is restored to `creator`. The pusher, believing they have revoked, continues pushing prices (e.g., to what they believe is their own namespace for a different purpose). Every push still lands in the creator's namespace.
4. The creator's namespace now contains stale or manipulated `(mid, spread, timestamp)` tuples. `AnchoredPriceProvider.getBidAndAskPrice()` reads these and computes a bad bid/ask band. Swaps execute at the wrong price, causing direct loss to LPs or traders.

The loop (revoke → replay → revoke → replay) can repeat until `block.timestamp >= deadline`. Tests show deadlines of `block.timestamp + 1 days` are typical: [7](#0-6) 

---

### Likelihood Explanation

- `allowPushers` is a **public, permissionless** function — anyone who holds the original calldata (trivially recoverable from the transaction that established the delegation) can replay it.
- The creator has a direct economic incentive to keep the pusher delegated (their pool's price feed depends on it).
- The pusher has no on-chain way to verify that their revocation "stuck" without reading `namespaceRemapping` in the same block, and even then the creator can front-run the next push.
- Deadlines of hours-to-days are the natural choice, giving a large replay window.

---

### Recommendation

Add a per-pusher nonce to the signed digest and track consumed nonces on-chain:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // ← add nonce
    ))
);
pusherNonce[pusher]++;        // ← consume it
```

Alternatively, allow the pusher to increment their own nonce (a `invalidatePusherSignatures()` function), so they can unilaterally invalidate any outstanding delegation signatures before calling `revokePusher()`.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
