### Title
No Slippage Protection on `removeLiquidity()` Exposes LPs to Oracle-Driven Bin Composition Shifts - (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.removeLiquidity()` accepts only a share quantity per bin and returns however many tokens those shares are worth at execution time. There is no minimum-output parameter. Because the pool's bin balances are continuously reshaped by oracle-price-driven swaps, the token amounts an LP receives can be materially lower than what they observed when they submitted the transaction.

### Finding Description

`removeLiquidity` signature: [1](#0-0) 

The function accepts `LiquidityDelta` (arrays of `binIdxs` and `shares`) and returns `(amount0Removed, amount1Removed)`. There is no `minAmount0Out`, `minAmount1Out`, or any equivalent deadline/price-limit guard. The returned amounts are computed from the live bin token balances at the moment of execution.

Bin balances are updated by every swap. Swaps are priced by the oracle provider (`_getBidAndAskPriceX64` → `IPriceProvider.getBidAndAskPrice()`): [2](#0-1) 

The oracle mid-price is read live from `PriceProvider` or `AnchoredPriceProvider` on every swap call: [3](#0-2) 

A price move between the LP's off-chain preview and block inclusion triggers arbitrage swaps that drain one token side of the affected bins. The LP's shares then redeem for a heavily skewed composition — potentially all of the cheaper token and none of the appreciated one.

`addLiquidity` has the symmetric problem: no maximum-input guard means the token cost can exceed what the LP budgeted. [4](#0-3) 

### Impact Explanation

An LP submitting `removeLiquidity` for shares worth approximately $10,000 at observed prices can receive substantially less if the oracle price moves and arbitrageurs rebalance the bins before the transaction is mined. The loss is permanent and unrecoverable — the LP has already burned their shares. This is a direct loss of LP principal with no on-chain recourse, satisfying the Critical/High/Medium direct-loss gate.

### Likelihood Explanation

Any LP interacting with any pool is exposed on every `removeLiquidity` call. No special role or privileged access is required. The risk is elevated on high-throughput chains (Base, HyperEVM) where oracle updates and arbitrage bots operate within the same block. The `PriceProvider` staleness window (`MAX_TIME_DELTA`) can be up to 7 days: [5](#0-4) 

meaning the oracle can legitimately deliver a price significantly different from what the LP saw when composing the transaction.

### Recommendation

Add `minAmount0Out` and `minAmount1Out` parameters to `removeLiquidity`, reverting if the computed output falls below either threshold:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,   // ← new
    uint256 minAmount1Out,   // ← new
    bytes calldata extensionData
) external returns (uint256 amount0Removed, uint256 amount1Removed) {
    ...
    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out)
        revert SlippageExceeded();
}
```

Similarly, add `maxAmount0In` / `maxAmount1In` to `addLiquidity`. Expose a view/simulate function so UIs can quote expected outputs before submission.

### Proof of Concept

1. ETH/USDC pool; oracle mid = $3,000. LP holds shares worth ~$10,000 (≈ 1.67 ETH + 5,000 USDC) across bin 0.
2. LP submits `removeLiquidity(owner, salt, {binIdxs:[0], shares:[N]}, extensionData)`.
3. Before the LP's tx lands, the oracle updates to $2,000. An arbitrageur calls `swap(zeroForOne=false, ...)`, buying all ETH from bin 0 at the new oracle price, leaving bin 0 holding only USDC.
4. LP's `removeLiquidity` executes: shares redeem for ~$6,667 in USDC only (at the new price), a ~33% loss with no on-chain protection.
5. The oracle recovers to $3,000 seconds later; the LP has already received the degraded payout. [1](#0-0)
