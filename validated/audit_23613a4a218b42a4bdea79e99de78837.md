### Title
Creator can replay a non-invalidated `allowPushers` consent signature to silently re-establish a revoked pusher delegation, rendering `revokePusher()` ineffective until the deadline expires — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` uses a deadline-scoped EIP-191 signature as the sole replay barrier, but there is no nonce, no used-signature bitmap, and no check that `namespaceRemapping[pusher]` is already cleared. After a pusher calls `revokePusher()`, the creator can immediately replay the original consent signature (while the deadline is still live) to overwrite the cleared mapping and re-establish the delegation the pusher just revoked. The code comment explicitly claims the deadline prevents this, but it does not.

---

### Finding Description

`allowPushers` signs consent as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only freshness gate is `_ensureDeadline`, which checks `block.timestamp <= deadline`. [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because `allowPushers` has no nonce, no used-signature set, and no guard against overwriting a zero mapping, the creator can call it again with the identical `(deadline, pusher, sig)` tuple at any point before the deadline. The mapping is silently restored to `creator`, and the pusher's revocation is undone.

The code comment directly contradicts this behavior:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The comment claims the deadline prevents re-establishment after revocation. It does not — it only prevents re-establishment after the deadline itself expires. Before expiry, the creator can replay the signature an unlimited number of times.

The `fallback` push path resolves the namespace as:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So while the pusher is forcibly re-delegated, every push they make lands in the creator's namespace, not their own. The pusher's own namespace (`feedIdOf(pusher, ...)`) receives no updates.

---

### Impact Explanation

A pusher who operates their own pool backed by `feedIdOf(pusher, slotIndex, positionIndex)` revokes their delegation to escape a creator relationship. The creator replays the original signature before the deadline. The pusher's only options are:

1. **Keep pushing** — all writes land in the creator's namespace; the pusher's own pool feeds remain at price 0 / timestamp 0 (stale), causing every swap in the pusher's pool to revert on the staleness check.
2. **Stop pushing** — the creator's pool also goes stale, but the pusher's pool is still DoS'd.

In either branch the pusher's pool is unusable for the full remaining lifetime of the deadline (which can be set to any future timestamp the creator chooses when they first call `allowPushers`). This maps directly to the H-4 class: a revocation mechanism that is supposed to restore a safe state is silently bypassed by replaying a previously valid authorization, leaving the affected party's pool frozen.

---

### Likelihood Explanation

The creator already holds the pusher's signed consent from the original `allowPushers` call. No additional off-chain material is needed. The replay requires a single on-chain transaction and succeeds as long as `block.timestamp <= deadline`. Deadlines are typically set to hours or days in the future, giving the creator a wide window. The attack is fully permissionless and requires no privileged role.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedConsents` keyed on the full consent hash. Alternatively, invalidate the consent on first use by recording the hash and reverting on any second submission of the same hash. Either approach closes the replay window regardless of the deadline.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator, deadline = now + 1 day
bytes memory sig = sign(PUSHER_KEY, abi.encode(chainid, oracle, deadline, pusher, creator));

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 4. Creator replays the SAME signature — no revert, mapping restored
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // re-established

// 5. Pusher's own pool feed is now permanently stale until deadline expires
IOffchainOracle.OracleData memory d = oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0));
assertEq(d.price, 0);        // zero — never updated
assertEq(TimeMs.unwrap(d.timestampMs), 0); // stale — pool swap reverts
```
