### Title
Permissionless `register` Unconditionally Clears Admin Blacklist, Allowing Any Pool to Bypass Oracle Read Restrictions — (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary
The `OracleBase.register` function is permissionless and unconditionally clears the blacklist for any pool that pays the registration fee (default: 1 wei). This allows any unprivileged caller to immediately undo an admin blacklist, rendering the `setBlacklist` security mechanism permanently ineffective and allowing a blacklisted pool to resume reading oracle prices.

---

### Finding Description

`OracleBase` exposes two state-changing paths for the pool blacklist:

**Admin-only setter** (ADMIN_ROLE): [1](#0-0) 

**Permissionless `register`** (anyone, 1 wei default): [2](#0-1) 

The critical block inside `register`:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
```

There is no access-control check on who may clear the blacklist. Any caller who supplies `msg.value >= registrationFee` (1 wei by default), an approved factory, and a pool that factory recognises can atomically un-blacklist the pool and re-register it for a feedId.

The `price` read path enforces the blacklist at call time: [3](#0-2) 

Because `register` clears `blacklisted[pool]` and sets `registeredPool[feedId][pool] = true` in the same transaction, the pool immediately satisfies both guards and can read prices again.

The admin can re-invoke `setBlacklist(pool, true)`, but the pool owner (or any third party) can front-run or immediately follow with another `register` call. This is the exact griefing pattern from the external report: a state transition that should be gated is reachable by an unprivileged caller an unlimited number of times, leaving the system in a bad state the admin cannot durably enforce.

---

### Impact Explanation

The blacklist is the oracle's last-resort mechanism to cut off a pool that is malicious, exploited, or otherwise unsafe. Once bypassed, the pool can call `price(feedId, pool)` through `ProtectedPriceProvider` or `PriceProvider`, receiving live Pyth/Chainlink mid-prices and spreads that it then uses to execute swaps. A pool blacklisted because it was draining LP funds or manipulating swap accounting can resume doing so immediately after any 1-wei `register` call. This satisfies the **admin-boundary break** impact gate: an oracle-admin role check (`blacklisted`) is bypassed by an unprivileged path.

---

### Likelihood Explanation

- `registrationFee` is initialised to `1 wei` and is only raised by an explicit admin call. [4](#0-3) 
- The pool must come from an approved factory, but the same factory that originally deployed the pool qualifies — no new setup is required.
- The call is a single transaction with no time delay, so it can be submitted in the same block as the admin's `setBlacklist`.

---

### Recommendation

Remove the blacklist-clearing side-effect from `register`. The blacklist should be exclusively clearable by `ADMIN_ROLE` via `setBlacklist`. If the protocol intends a "pay-to-rehabilitate" path, it should be a separate function gated on `ADMIN_ROLE` or at minimum require a fee large enough to be economically prohibitive:

```diff
 function register(bytes32 feedId, address pool, address factory) external payable {
     require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
     require(pool != address(0));
     require(approvedFactories.contains(factory), FactoryNotApproved(factory));
     require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

-    if (blacklisted[pool]) {
-        blacklisted[pool] = false;
-        emit BlacklistUpdated(pool, false);
-    }
+    require(!blacklisted[pool], Blacklisted(pool));

     registeredPool[feedId][pool] = true;
     emit PoolRegistered(feedId, pool, msg.sender, msg.value);
 }
```

### Proof of Concept

```solidity
// 1. Admin blacklists a compromised pool
oracle.setBlacklist(pool, true);
// oracle.blacklisted(pool) == true
// oracle.price(feedId, pool) now reverts with Blacklisted

// 2. Attacker (or pool owner) pays 1 wei to re-register
oracle.register{value: 1 wei}(feedId, pool, approvedFactory);
// oracle.blacklisted(pool) == false  ← blacklist silently cleared
// oracle.registeredPool[feedId][pool] == true

// 3. Pool reads prices again — blacklist has no effect
(uint256 mid, uint256 spread,,) = oracle.price(feedId, pool);
// succeeds; pool resumes swaps at live oracle prices

// 4. Admin re-blacklists; attacker immediately replays step 2 — cycle repeats indefinitely
```
