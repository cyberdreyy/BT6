### Title
`revokePusher()` is Ineffective Against Signature Replay — Creator Can Immediately Re-Establish Delegation Using the Original Consent Signature - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

A pusher who calls `revokePusher()` to exit a delegation cannot prevent the creator from immediately re-establishing it by replaying the original `allowPushers` consent signature. Because there is no nonce or used-signature tracking, the same EIP-191 consent is valid for any number of `allowPushers` calls until the deadline expires. This breaks the invariant that `revokePusher()` gives the pusher effective control over their own namespace, and can cause the pusher's own feeds to become stale, halting swaps in pools that depend on them.

---

### Finding Description

`allowPushers` accepts a pusher's EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The function is permissionless and has no nonce, no used-signature set, and no per-pusher revocation counter. The only expiry is the deadline embedded in the signature itself.

`revokePusher()` clears the mapping: [2](#0-1) 

But immediately after the pusher calls `revokePusher()`, the creator can call `allowPushers` again with the **identical** `(deadline, pusher, signature)` tuple. Because `_ensureDeadline` only checks `block.timestamp < deadline` and the signature is still cryptographically valid, the call succeeds and `namespaceRemapping[pusher]` is restored to the creator's address. This cycle can be repeated indefinitely until the deadline expires.

The code comment explicitly acknowledges the deadline as the protection against re-establishment after revocation: [3](#0-2) 

However, the deadline only prevents re-establishment *after* it expires — it does not prevent re-establishment *before* it expires. A pusher who signed a long-lived consent (e.g., 1-year deadline) and then calls `revokePusher()` after 1 day remains effectively locked for the remaining 364 days.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So as long as `namespaceRemapping[pusher] = creator`, every push from the pusher lands in the creator's namespace, not the pusher's own.

---

### Impact Explanation

If the pusher is also a creator of their own feeds (used by registered pools), they cannot push to their own namespace while the delegation is active. After calling `revokePusher()`, the creator immediately replays the signature, and the pusher's pushes continue to be redirected. The pusher's own feeds receive no updates and become stale. Providers reading those feeds return `(0, type(uint128).max)` (the `FeedStalled` sentinel): [5](#0-4) 

Pools depending on those feeds cannot execute swaps — core swap functionality is broken for the duration of the deadline window.

---

### Likelihood Explanation

The trigger requires a creator who actively replays the signature after the pusher revokes. This is a semi-trusted but adversarial creator. The pusher must also be an independent creator of feeds used by live pools. Both conditions are plausible in a production deployment where the same operator runs multiple roles. The attacker action (calling `allowPushers` with an already-seen signature) is a single, cheap, permissionless transaction.

---

### Recommendation

Add a per-pusher revocation nonce or a used-signature set to `allowPushers`. One minimal fix: maintain a `mapping(address => uint256) public pusherNonce` and include it in the signed digest. When the pusher calls `revokePusher()`, increment their nonce. Any previously issued signature (which committed to the old nonce) becomes permanently invalid.

```solidity
// In the signed digest:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher():
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

### Proof of Concept

```
1. Pusher P signs consent for creator A with deadline = block.timestamp + 365 days.
   sig = sign(keccak256(abi.encode(chainid, oracle, deadline, P, A)))

2. Creator A calls allowPushers(deadline, [P], [sig])
   → namespaceRemapping[P] = A
   → P's fallback pushes land in A's namespace

3. P calls revokePusher()
   → namespaceRemapping[P] = address(0)
   → P expects to push to own namespace

4. Creator A immediately calls allowPushers(deadline, [P], [sig])  ← same sig, same deadline
   → _ensureDeadline passes (deadline still in future)
   → ECDSA.recover returns P ✓
   → namespaceRemapping[P] = A  ← restored

5. P's pushes continue to land in A's namespace.
   P's own feeds (feedIdOf(P, slotId, posId)) receive no updates → timestamp stays stale.

6. Pools registered to P's own feeds call getBidAndAskPrice():
   _isStale(refTime=0, ...) → true → returns (0, type(uint128).max) → FeedStalled revert.
   Swaps are blocked for the remaining 364 days.

Steps 3–4 can be repeated by A on every block at negligible cost.
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
