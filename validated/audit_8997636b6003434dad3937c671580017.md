### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Unlisted Caller to Deposit Into a Restricted Pool — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor) and gates only on `owner` (the position recipient). Any unlisted caller can bypass the allowlist by supplying an allowlisted address as `owner`.

---

### Finding Description

The `IMetricOmmExtensions.beforeAddLiquidity` interface passes two identity parameters:

```
beforeAddLiquidity(address sender, address owner, ...)
```

`sender` is `msg.sender` of the pool's `addLiquidity` call — the actual party initiating and paying for the deposit. `owner` is the position recipient. [1](#0-0) 

The pool correctly passes both:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [3](#0-2) 

However, `DepositAllowlistExtension.beforeAddLiquidity` **discards `sender` entirely** (unnamed first parameter) and only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [4](#0-3) 

The allowlist mapping is keyed `allowedDepositor[pool][depositor]`, where the pool admin intends `depositor` to mean the party actually depositing funds. But the check uses `owner` — the LP share recipient — not the actual caller.

---

### Impact Explanation

An unlisted caller can:

1. Call `pool.addLiquidity(allowlistedAddress, salt, deltas, ...)` directly, **or**
2. Call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, allowlistedAddress, ...)`. [5](#0-4) 

In both cases, the extension checks `allowedDepositor[pool][allowlistedAddress]` = `true` and does not revert. The unlisted caller's tokens are pulled via callback, LP shares are minted to `allowlistedAddress`. The allowlist is completely ineffective.

Consequences:
- **Allowlist fully bypassed**: any unlisted party can deposit into a restricted pool by naming any allowlisted address as `owner`.
- **Forced LP positions**: allowlisted addresses receive LP shares they never requested, which can affect their accounting, tax treatment, or compliance posture.
- **Pool state manipulation by unauthorized parties**: unlisted actors can freely shape bin liquidity distribution in a pool designed to restrict participation.

---

### Likelihood Explanation

The bypass requires only knowing one allowlisted address (which is public on-chain via `AllowedToDepositSet` events) and calling a public function. No privileged access, no special tokens, no malicious pool setup required. Likelihood is **high**.

---

### Recommendation

Check `sender` (the actual depositor/caller), not `owner` (the position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [4](#0-3) 

---

### Proof of Concept

```solidity
// Setup: pool uses DepositAllowlistExtension; only address(0xALLOWED) is allowlisted.
// extension.setAllowedToDeposit(pool, address(0xALLOWED), true);

// Attacker (address(0xATTACKER), NOT on allowlist) calls:
liquidityAdder.addLiquidityExactShares(
    pool,
    address(0xALLOWED),   // owner = allowlisted address
    salt,
    deltas,
    maxAmount0,
    maxAmount1,
    extensionData
);
// beforeAddLiquidity checks allowedDepositor[pool][address(0xALLOWED)] == true → passes
// Attacker's tokens are pulled; LP shares minted to address(0xALLOWED)
// Allowlist invariant violated: unlisted caller successfully deposited
``` [6](#0-5) [5](#0-4)
