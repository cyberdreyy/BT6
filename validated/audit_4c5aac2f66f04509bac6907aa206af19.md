### Title
Signature Replay in `allowPushers` Defeats `revokePusher()` Within the Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` verifies an EIP-191 pusher-consent signature but never marks it as consumed. The same signature can be submitted an unlimited number of times before `deadline` expires. A creator can therefore re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher calls `revokePusher()`, making self-revocation ineffective for the entire deadline window.

### Finding Description

`allowPushers` performs three checks before writing `namespaceRemapping[pusher] = msg.sender`:

1. `_ensureDeadline(deadline)` — rejects calls after the deadline.
2. Signature recovery — confirms the pusher signed `(chainid, address(this), deadline, pusher, msg.sender)`.
3. Self-remapping guard — rejects `pusher == msg.sender`.

There is **no nonce, no used-signature set, and no single-use enforcement** of any kind. [1](#0-0) 

`revokePusher` simply zeroes the mapping: [2](#0-1) 

The code's own NatSpec acknowledges the risk — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — and claims the deadline is the fix. But the deadline only bounds the outer window; it does not prevent the creator from replaying the same signature multiple times **within** that window. [3](#0-2) 

The slot-structure documentation repeats the same claim: [4](#0-3) 

### Impact Explanation

A pusher who is actively streaming price data (e.g., a bot) and calls `revokePusher()` to stop contributing to the creator's namespace cannot actually stop within the deadline window. The creator replays the original signature, `namespaceRemapping[pusher]` is restored to `creator`, and every subsequent fallback push from the pusher lands in the creator's namespace again — without the pusher's current consent. If the pusher revoked because they detected a data quality issue or a compromise, the creator can silently keep their data flowing into the pool's feed namespace, potentially delivering stale or bad prices to any pool consuming that feed.

### Likelihood Explanation

The pusher must have signed a consent with a future deadline (standard practice), and the creator must be adversarial or negligent. The pusher must continue pushing after revoking (common for automated bots). All three conditions are realistic in a live deployment.

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedDelegations` keyed on the full signature hash (or a `(pusher, creator, deadline)` tuple). Mark the entry `true` on first use and revert on any subsequent call with the same key. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful delegation or revocation.

```solidity
// Foundry unit test (pseudo-code)
function testSignatureReplayAfterRevoke() public {
    uint256 deadline = block.timestamp + 1 hours;

    // 1. Pusher signs consent
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator delegates
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher self-revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature — succeeds, no revert
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // should revert but does not

    // 5. Revocation is undone
    assertEq(oracle.namespaceRemapping(pusher), creator); // pusher re-delegated without consent
}
```

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
