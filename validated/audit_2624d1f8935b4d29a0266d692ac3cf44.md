### Title
`allowPushers` consent-signature replay re-establishes a revoked pusher delegation, routing price writes away from the pusher's own namespace and leaving any pool reading from it permanently stale — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = creator`. The signed message is bound to `(chainid, oracle, deadline, pusher, creator)` but there is **no nonce and no used-signature tracking**. A creator can replay the exact same consent signature — any number of times before the deadline — to silently re-establish a delegation the pusher already revoked via `revokePusher()`. The pusher's own namespace stays at timestamp 0, and every price provider that reads from it treats that as stale, reverting `FeedStalled` on every swap.

---

### Finding Description

`allowPushers` sets `namespaceRemapping[pusher] = msg.sender` after verifying the pusher's signature:

```solidity
// CompressedOracle.sol L192-211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← no nonce, no "used" flag
    ...
}
```

`revokePusher` clears the mapping:

```solidity
// CompressedOracle.sol L238-243
function revokePusher() external {
    ...
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

Because `allowPushers` never records that a signature was consumed, the creator can call it again with the identical `(deadline, [pusher], [sig])` tuple immediately after the pusher revokes. The mapping is restored to `creator` and the pusher's revocation is silently undone. The code's own comment acknowledges the risk but presents the deadline as the only mitigation:

> *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The deadline only bounds the window; it does not prevent replay within that window. A pusher who signed with a long deadline (days, weeks, or months — common for long-running bots) has no on-chain way to permanently stop the creator from re-establishing the delegation until the deadline expires.

The `fallback` push path resolves the namespace at call time:

```solidity
// CompressedOracle.sol L315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

So every push the pusher makes after revoking — intending to write to their own namespace — still lands in the creator's namespace as long as the creator keeps replaying the consent.

---

### Impact Explanation

When the pusher's own namespace is never written (timestamp 0), every price provider that reads from it treats the feed as stale. `AnchoredPriceProvider._isStale` and `PriceProvider._isStale` both return `true` when `refTime == 0`:

```solidity
// AnchoredPriceProvider.sol L227
if (refTime == 0) return true;
```

`_readLeg` propagates `ok = false`, `_getBidAndAskPrice` returns `(0, type(uint128).max)`, and `getBidAndAskPrice` reverts `FeedStalled`. Any pool whose price provider reads from the pusher's namespace becomes completely unusable for swaps — satisfying the "unusable withdraw/swap/liquidity flows" impact gate.

---

### Likelihood Explanation

Medium. The trigger requires a creator who is unwilling to respect the pusher's revocation (malicious or uncooperative). However:

- The creator is not a privileged protocol role; any address can be a creator.
- Pushers who sign with long deadlines (a natural choice for long-running bots) are fully exposed for the entire remaining deadline window.
- The attack requires only a single on-chain call (`allowPushers` with the cached signature) and costs only gas.
- No test in the suite covers the "creator replays after pusher revokes" scenario, confirming the gap is unguarded.

---

### Recommendation

Track consumed consent signatures. Add a `mapping(bytes32 => bool) private _usedConsents` and mark each signature hash as used on first acceptance:

```solidity
bytes32 consentKey = keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender));
require(!_usedConsents[consentKey], "consent already used");
_usedConsents[consentKey] = true;
```

Alternatively, add a per-pusher revocation nonce to the signed message so that a post-revocation consent requires a fresh signature the pusher has not yet issued.

---

### Proof of Concept

```
1. Pusher P signs: keccak256(abi.encode(chainid, oracle, deadline=T+30days, P, C))
   → sig_P

2. Creator C calls allowPushers(T+30days, [P], [sig_P])
   → namespaceRemapping[P] = C  ✓

3. Pusher P calls revokePusher()
   → namespaceRemapping[P] = address(0)  ✓

4. Creator C calls allowPushers(T+30days, [P], [sig_P])  ← same call, same sig
   → _ensureDeadline passes (deadline still in future)
   → ECDSA.recover returns P  ← same hash, same sig
   → namespaceRemapping[P] = C  ← delegation silently re-established

5. P pushes prices (intending own namespace):
   fallback: creator = namespaceRemapping[P] = C  → writes to C's namespace

6. feedIdOf(P, slot, pos) → timestamp 0
   AnchoredPriceProvider._isStale(0, ...) → true
   getBidAndAskPrice() → revert FeedStalled
   → P's pool is permanently unusable for swaps
