### Title
Pusher delegation consent signature in `allowPushers()` carries no nonce, making `revokePusher()` ineffective within the deadline window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers()` verifies a pusher's EIP-191 consent signature but includes no nonce or used-signature tracking. The deadline field only prevents replay *after* it expires; within the window the identical signature can be submitted again. A creator can therefore re-establish a delegation that the pusher explicitly cancelled via `revokePusher()`, silently redirecting the pusher's subsequent slot writes into the creator's namespace and potentially feeding unintended prices into any pool that reads those feeds.

---

### Finding Description

`allowPushers()` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The NatSpec comment above the function explicitly acknowledges the replay concern:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The comment's reasoning is incomplete. The deadline is checked by `_ensureDeadline`:

```solidity
require(block.timestamp <= deadline, DeadlineExceeded());
``` [3](#0-2) 

This only rejects calls made *after* the deadline. It does nothing to prevent the creator from submitting the same `(deadline, pusher, sig)` tuple a second time *before* the deadline expires. There is no nonce, no per-pusher revocation counter, and no used-signature bitmap. `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [4](#0-3) 

But the creator can immediately call `allowPushers()` again with the original signature, writing `namespaceRemapping[pusher] = creator` back, as long as `block.timestamp <= deadline`.

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after their revocation — believing they are writing into their own namespace — is silently redirected into the creator's namespace.

---

### Impact Explanation

The `namespaceRemapping` table is the sole authority that determines which storage slots a pusher's calldata overwrites. Pools consume prices from the creator's feeds via `price(feedId, pool)` → `getOracleData(feedId)` → slot decode. If the creator replays the delegation after the pusher revokes:

1. The pusher's own namespace stays at price 0 / timestamp 0 (stale, rejected by any consumer).
2. The creator's namespace receives whatever the pusher pushes — data the pusher may now intend for a completely different feed or asset pair.
3. Any pool whose `IPriceProvider` is backed by the creator's `CompressedOracleV1` feeds will execute swaps against those unintended prices.

This satisfies the **bad-price execution** impact gate: an unintended bid/ask quote reaches a live pool swap because the oracle's namespace routing was silently overridden.

---

### Likelihood Explanation

Medium. The creator must be willing to act against the pusher's explicit revocation (semi-trusted, not an oracle admin). The window of opportunity equals the remaining time before the original deadline expires — which can be arbitrarily long if the pusher signed a consent with a multi-day or multi-week deadline. The pusher has no on-chain way to invalidate the old signature before the deadline; their only mitigation is to stop pushing entirely until the deadline lapses, which may not be operationally feasible.

---

### Recommendation

Add a per-pusher revocation nonce to the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// in allowPushers():
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);
// after successful delegation:
pusherNonce[pusher]++;
```

Alternatively, increment the nonce inside `revokePusher()` so that any previously issued signature is immediately invalidated on revocation, regardless of its deadline.

---

### Proof of Concept

```
T=0   Pusher signs: hash(chainid, oracle, deadline=T+1day, pusher, creator)
T=0   Creator calls allowPushers(deadline, [pusher], [sig])
         → namespaceRemapping[pusher] = creator  ✓

T=1h  Pusher calls revokePusher()
         → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)

T=1h  Creator calls allowPushers(deadline, [pusher], [sig])  ← SAME sig, deadline still valid
         → namespaceRemapping[pusher] = creator  ← re-established without pusher consent

T=2h  Pusher pushes slot data (intending own namespace)
         fallback: creator = namespaceRemapping[pusher] = creator
         → data written to creator's namespace, not pusher's own

      Pool reads creator's feed → receives pusher's unintended price data
      Pool executes swap at wrong price
```
