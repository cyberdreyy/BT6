### Title
`PriceVelocityGuardExtension` Measures Price Velocity in Blocks Instead of Seconds, Allowing Guard Bypass on Berachain's Non-Constant Block Time — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` caps how fast the oracle mid-price can move between swaps using `block.number` as the time unit. On Berachain, block time is explicitly non-constant (it varies with network congestion). When blocks are produced faster than the rate the pool admin assumed when setting `maxChangePerBlockE18`, the guard's allowed-change window expands proportionally in real time, letting a price movement that should have been blocked pass through to the pool.

---

### Finding Description

The extension stores the block number of the last observed mid-price and computes the allowed price change as:

```
allowedSq = maxChange² × (1 + blockDiff)
```

where `blockDiff = block.number - prevBlock`. [1](#0-0) 

The pool admin calibrates `maxChangePerBlockE18` against a current average block time `T`. If Berachain later produces blocks at a faster rate `T' < T`, then for any fixed real-world interval `Δt`:

- **Expected** `blockDiff` = `Δt / T`
- **Actual** `blockDiff` = `Δt / T'` > `Δt / T`

The `allowedSq` budget grows by the ratio `T / T'`. A price movement that covers `Δt` seconds of real time is now permitted to be `sqrt(T/T')` times larger than the admin intended. The guard's real-time velocity cap silently inflates without any admin action or on-chain signal.

The state update writes `block.number` unconditionally before the check, so every swap refreshes the baseline: [2](#0-1) 

The rest of the oracle stack — `CompressedOracle`, `ChainlinkOracle`, `AnchoredPriceProvider` — all use `block.timestamp` / `timestampMs` for staleness and drift checks: [3](#0-2) [4](#0-3) 

`PriceVelocityGuardExtension` is the only time-sensitive guard in the stack that still uses `block.number`.

---

### Impact Explanation

**Bad-price execution reaches the pool.** The velocity guard is the last in-swap circuit breaker before a bid/ask quote is accepted by `MetricOmmPool`. If the guard is miscalibrated upward (faster blocks → larger `blockDiff` → larger `allowedSq`), a price that moved too fast in real time — e.g., a stale or manipulated oracle push that jumps the mid-price by 2× the intended per-second cap — passes the `actualSq > allowedSq` check and the swap executes at the bad price. LPs absorb the loss as adverse selection.

The inverse direction (slower blocks → smaller `blockDiff` → tighter guard) causes legitimate swaps to revert with `PriceVelocityExceeded`, breaking core pool swap functionality. [5](#0-4) 

---

### Likelihood Explanation

Berachain's block time is explicitly documented as non-constant (it depends on network congestion, as cited in the external report). The protocol is deployed on Berachain. Any sustained shift in block production rate — which is a normal operational condition, not an attack — silently miscalibrates the guard. No privileged action, malicious setup, or non-standard token is required. Any user can trigger a swap that exercises the miscalibrated guard.

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Store `lastUpdateTimestamp` (seconds) instead of `lastUpdateBlock`, and express `maxChangePerBlockE18` as `maxChangePerSecondE18`. The velocity formula becomes:

```
uint256 timeDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This makes the guard invariant to block-time fluctuations, consistent with how every other time-sensitive check in the stack (`revertIfAfterBlockTimeWithDrift`, `_isStale`, `MAX_REF_STALENESS`) already operates. [6](#0-5) 

---

### Proof of Concept

1. Pool admin deploys `PriceVelocityGuardExtension` on Berachain and sets `maxChangePerBlockE18` assuming a 2-second average block time, intending to allow at most 0.5% price movement per second.
2. Berachain network congestion drops and blocks begin arriving every 0.5 seconds (4× faster).
3. A pusher submits a stale `CompressedOracle` slot with a price 3% above the last observed mid (within `MAX_TIME_DRIFT` so it is accepted by the oracle).
4. A swap is triggered. `blockDiff = block.number - prevBlock` reflects 4 blocks elapsed (2 seconds of real time at the new rate), so `allowedSq = maxChange² × 5`.
5. At the original 2-second block time, only 1 block would have elapsed, giving `allowedSq = maxChange² × 2` — the 3% jump would have been rejected.
6. At the faster block rate, `allowedSq` is 2.5× larger; the 3% jump satisfies `actualSq ≤ allowedSq` and the swap executes at the manipulated price, with LPs receiving less than fair value for the tokens they provide. [7](#0-6)
