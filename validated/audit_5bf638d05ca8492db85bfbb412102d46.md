### Title
Pusher consent signature replay in `allowPushers` lets creator permanently override a pusher's revocation, causing stale namespace feeds - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowPushers` accepts a deadline-bounded EIP-191 signature from a pusher but neither tracks consumed signatures nor increments any nonce. A creator who holds a valid consent signature can call `allowPushers` again after the pusher calls `revokePusher()`, silently re-establishing the delegation within the same deadline window. The pusher cannot permanently revoke their delegation until the deadline expires; their own namespace feeds remain empty while their pushes are redirected to the creator's namespace.

---

### Finding Description

The `allowPushers` function resolves consent purely from the signature and a deadline: [1](#0-0) 

The signed digest is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [2](#0-1) 

There is no nonce, no per-pusher revocation counter, and no `_usedSignatures` mapping. The function can be called with the **same** `(deadline, pusher, sig)` tuple any number of times before `block.timestamp > deadline`.

`revokePusher` clears the mapping: [3](#0-2) 

But because the original signature is still valid (the deadline has not expired), the creator can immediately call `allowPushers` again with the identical signature, writing `namespaceRemapping[pusher] = creator` back into storage. The code comment acknowledges the replay concern but incorrectly claims the deadline resolves it: [4](#0-3) 

The deadline only prevents replay **after** it expires. Within the window it provides zero protection against re-establishment.

The `fallback` push path resolves the namespace once at entry from `namespaceRemapping[msg.sender]`: [5](#0-4) 

So every push the pusher makes after the creator's replay lands in the **creator's** namespace, not the pusher's own. The pusher's own namespace slots receive no updates and their timestamps remain at whatever value they held before the delegation was first established.

---

### Impact Explanation

1. The pusher's own namespace feeds (`feedIdOf(pusher, slotIndex, positionIndex)`) receive no new data; their timestamps freeze.
2. Any pool or `AnchoredPriceProvider` configured to read from the pusher's own namespace feeds will observe a stale `timestampMs`, which every consumer is documented to reject as stale — producing a zero-price / max-spread quote.
3. A zero-price or max-spread quote reaching `MetricOmmPool.swap` via `getBidAndAskPrice` constitutes bad-price execution: traders receive incorrect fill prices or swaps revert, breaking core pool swap functionality.
4. The creator can sustain this condition for the entire remaining lifetime of the deadline by replaying the signature each time the pusher revokes — zero additional cost per replay.

---

### Likelihood Explanation

- The creator must have previously obtained a valid pusher consent signature (a normal, legitimate setup step).
- After that, replaying `allowPushers` is a single public call with no additional prerequisites.
- The creator can monitor the mempool for `PusherRevoked` events and front-run or immediately follow each revocation with a replay — the attack is atomic and repeatable.
- Any creator who has ever delegated a pusher and later had that pusher revoke is a potential attacker.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedDelegations` and mark each digest as used on first acceptance:

```solidity
mapping(bytes32 => bool) private _usedDelegations;

function allowPushers(...) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(!_usedDelegations[hash], "signature already consumed");
    require(pusher == ECDSA.recover(hash, signatures[i]));
    _usedDelegations[hash] = true;
    namespaceRemapping[pusher] = msg.sender;
    ...
}
```

Alternatively, add a per-pusher revocation nonce to the signed digest so that each revocation invalidates all prior signatures for that pusher.

---

### Proof of Concept

```
Step 1 – Setup (legitimate):
  Pusher P signs: keccak256(abi.encode(chainid, oracle, deadline=T, pusher=P, creator=C))
  Creator C calls allowPushers(T, [P], [sig])
  → namespaceRemapping[P] = C  ✓

Step 2 – Pusher revokes:
  P calls revokePusher()
  → namespaceRemapping[P] = address(0)  ✓

Step 3 – Creator replays (same sig, same deadline, T not yet expired):
  C calls allowPushers(T, [P], [sig])   ← identical calldata
  → _ensureDeadline(T) passes
  → ECDSA.recover returns P  (signature still valid)
  → namespaceRemapping[P] = C  ← revocation silently undone

Step 4 – P pushes data (intending own namespace):
  P calls fallback with slot word
  → creator = namespaceRemapping[P] = C  (not address(0))
  → namespace = C << 96
  → data lands in C's namespace, P's own namespace stays stale

Step 5 – Pool reads feedIdOf(P, slotIndex, pos):
  → timestampMs == 0 (never updated)
  → consumer rejects as stale → zero price / max spread → bad-price execution

Steps 2–5 repeat indefinitely until block.timestamp > T.
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
