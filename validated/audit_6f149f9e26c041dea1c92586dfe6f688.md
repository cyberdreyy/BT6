### Title
Creator Can Re-Establish Pusher Delegation After Revocation by Replaying EIP-191 Consent Signature — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracleV1.allowPushers` does not invalidate a pusher's EIP-191 consent signature when the pusher self-revokes via `revokePusher()`. A creator who retains the original signature can replay it within the deadline window to silently re-establish the delegation, bypassing the pusher's unilateral revocation. This is the direct analog of the ENS fuse-bypass: a constraint that is supposed to be permanent (`revokePusher` is documented as a final, unilateral action) can be undone by the other party through a state-ordering trick.

### Finding Description

`allowPushers` verifies a pusher's EIP-191 signature over the tuple `(chainid, oracle, deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = creator`. [1](#0-0) 

`revokePusher` clears that mapping to `address(0)`: [2](#0-1) 

There is no nonce, no per-pusher revocation counter, and no on-chain record that the pusher ever revoked. The signed message contains only `(chainid, oracle, deadline, pusher, creator)`. As long as `block.timestamp < deadline`, the exact same bytes are a valid input to `allowPushers` again. The creator calls `allowPushers` a second time with the identical signature and deadline; `_ensureDeadline` passes, ECDSA recovery succeeds, and `namespaceRemapping[pusher]` is written back to `creator`.

The code's own comment acknowledges the risk but claims the deadline is the mitigation: [3](#0-2) 

The deadline prevents replay *after* it expires, but within the deadline window the creator can re-establish the delegation an unlimited number of times, each time immediately after the pusher revokes. The pusher's revocation is therefore not final.

The `fallback()` push path resolves the namespace at call time from `namespaceRemapping[msg.sender]`: [4](#0-3) 

While the pusher is re-delegated, every push lands in the creator's namespace. The pusher's own namespace (`feedIdOf(pusher, slot, pos)`) receives no updates and its stored timestamp freezes.

### Impact Explanation

Pools are configured with a specific `feedId` at deployment. If a pool's `feedId` encodes the pusher's own namespace (`feedIdOf(pusher, slot, pos)`), and the creator keeps the pusher re-delegated, the pusher's namespace timestamp never advances. The `AnchoredPriceProvider` reads through `offchainOracle.price(feedId, pool)`, which calls `_readPrice` and checks staleness via `MAX_REF_STALENESS`. Once the frozen timestamp exceeds `MAX_REF_STALENESS`, `_getBidAndAskPrice` returns `(0, type(uint128).max)` and `getBidAndAskPrice` reverts with `FeedStalled`. [5](#0-4) 

Every swap through any pool backed by the pusher's feed halts. LPs cannot exit via swap; the pool's core swap flow is broken for the duration of the attack (until the deadline expires or the pusher obtains a new consent with a shorter deadline).

### Likelihood Explanation

- The creator must have retained the original signature bytes — trivially true since the creator submitted the transaction and can log calldata.
- The deadline is chosen by the creator when calling `allowPushers`; there is no maximum deadline enforced on-chain, so a creator can request a signature over a deadline months in the future.
- The pusher must have signed the consent at least once (they were willing to be a pusher at some point). The attack is that the creator can prevent the pusher from ever revoking within the deadline window.
- No privileged role is required; the creator is a semi-trusted party (they own their own namespace and legitimately call `allowPushers`).

### Recommendation

Include a per-pusher revocation nonce in the signed message. Increment the nonce on every `revokePusher` or `removePushers` call. Old signatures become invalid immediately upon revocation:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on revoke

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[msg.sender]++;   // or pusherNonce[pusher]++
namespaceRemapping[...] = address(0);
```

Alternatively, store the deadline at which the delegation was established and reject re-delegation with the same or an earlier deadline.

### Proof of Concept

```
// Step 1: Pusher signs consent with a 7-day deadline
deadline = block.timestamp + 7 days
sig = pusher.sign(keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)))

// Step 2: Creator establishes delegation
oracle.allowPushers(deadline, [pusher], [sig])
// namespaceRemapping[pusher] == creator ✓

// Step 3: Pusher revokes
oracle.revokePusher()   // called by pusher
// namespaceRemapping[pusher] == address(0) ✓

// Step 4: Creator immediately replays the SAME sig (deadline still valid)
oracle.allowPushers(deadline, [pusher], [sig])
// namespaceRemapping[pusher] == creator again ✓  ← revocation bypassed

// Step 5: Pusher's pushes continue landing in creator's namespace.
//         feedIdOf(pusher, slot, pos) timestamp freezes.
//         After MAX_REF_STALENESS seconds, any pool backed by
//         feedIdOf(pusher, slot, pos) reverts FeedStalled on every swap.
```

The creator can repeat Step 4 indefinitely until the deadline expires, making the pusher's revocation permanently ineffective within the deadline window.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

    fallback() override external {
        uint256 end;
        uint256 namespace;

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

        assembly ("memory-safe") {
            end := calldatasize()
            namespace := shl(96, creator) // [creator:20][zeros:12]
        }

        // 4 * 6 + 7 + 1 = 32 bytes per slot
        if (end == 0 || end % 32 != 0) revert BadCalldataLength();

        for (uint256 ptr = 0; ptr < end; ptr += 32) {
            uint256 word;
            assembly ("memory-safe") {
                word := calldataload(ptr)
            }
            // casting to 'uint8' is safe we want LSB
            // forge-lint: disable-next-line(unsafe-typecast)
            uint8 slotId = uint8(word);
            TimeMs timestampMs = toTimeMs(word >> 8 & X56);
            timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);

            uint256 old = uint256(_loadStorage(key));
            TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

            bool newer = timestampMs.isAfter(oldTimestampMs);
            if (!newer) continue;

            _writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
