### Title
`allowPushers` Signature Replay Allows Creator to Permanently Re-Delegate Pusher After Self-Revocation, Enabling Bad-Price Injection into Pools - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

The `allowPushers` function in `CompressedOracleV1` accepts an EIP-191 pusher-consent signature and writes `namespaceRemapping[pusher] = creator`. The function contains no nonce, no used-signature bitmap, and no revocation flag. A creator can call `allowPushers` with the exact same signature an unlimited number of times as long as `block.timestamp <= deadline`. This means that after a pusher calls `revokePusher()`, the creator can immediately replay the original signature to silently re-establish the delegation. The pusher's self-revocation is rendered permanently ineffective for the entire lifetime of the deadline, which the code's own comment claims the deadline is supposed to prevent.

### Finding Description

`allowPushers` verifies the pusher's EIP-191 consent and writes the namespace remapping: [1](#0-0) 

The signed message covers `(chainid, oracle, deadline, pusher, creator)`. There is no nonce, no consumed-signature set, and no check against `namespaceRemapping`'s current state. The function unconditionally overwrites `namespaceRemapping[pusher] = msg.sender` on every valid call.

`revokePusher` clears the mapping: [2](#0-1) 

Because `allowPushers` has no replay guard, the creator can call it again with the identical `(deadline, [pusher], [sig])` tuple immediately after the pusher's revocation, restoring `namespaceRemapping[pusher] = creator`. The pusher can call `revokePusher()` again, and the creator can replay again — an infinite loop that the pusher can only escape by waiting for the deadline to expire.

The code's own NatSpec comment explicitly acknowledges this threat and claims the deadline is the fix: [3](#0-2) 

But the deadline only prevents replay *after* it expires. Within the deadline window — which can be set to `type(uint256).max` since `_ensureDeadline` imposes no upper bound — the creator can replay indefinitely. The stated invariant is broken.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So every push the pusher makes after a silently-undone revocation lands in the creator's namespace, not the pusher's own.

### Impact Explanation

**Bad-price execution reaching a pool swap.**

Attack path:

1. Pusher signs consent with `deadline = now + 1 year` for creator's ETH/USD feed.
2. Creator calls `allowPushers` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`. Pusher believes they are now pushing to their own namespace.
4. Creator immediately calls `allowPushers` with the identical `(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator` restored. Pusher is unaware.
5. Pusher, believing they are in their own namespace, begins pushing BTC/USD prices (a new feed they are operating for a different creator).
6. Every push lands in the original creator's ETH/USD slot. The ETH/USD feed now contains BTC/USD prices — a price that is orders of magnitude wrong.
7. `AnchoredPriceProvider._readLeg` reads the corrupted mid price. The staleness check passes (timestamp is fresh). The `mid == 0` check passes (BTC/USD price is non-zero). The `spreadBps >= ORACLE_BPS` check passes (spread is normal). The price guard passes if the creator has not set tight bounds (or has set none).
8. `_computeBidAsk` produces a bid/ask anchored to the BTC/USD price for an ETH/USD pool.
9. Pool swap executes at the wrong price — traders receive far more or far less than the oracle curve permits, draining LP reserves. [5](#0-4) 

### Likelihood Explanation

**Low-Medium.** The creator must be malicious and must hold the pusher's original consent signature (which they received legitimately during the initial `allowPushers` call). The pusher must continue pushing after revoking — specifically pushing data intended for a different feed. This is realistic in a multi-creator ecosystem where pushers serve multiple creators and rotate their delegations. The pusher has no on-chain signal that their revocation was undone unless they explicitly re-query `namespaceRemapping` before every push batch.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(

            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
