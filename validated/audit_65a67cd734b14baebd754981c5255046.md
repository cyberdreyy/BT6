### Title
Missing L2 Sequencer Uptime Check Allows Bad-Price Execution After Sequencer Restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly L2-aware price providers (they carry a `FUTURE_TOLERANCE` immutable to handle L2 sequencer clock skew) but neither checks a Chainlink sequencer uptime feed. After an L2 sequencer outage, the oracle's `refTime` staleness check passes immediately upon restart, allowing pool swaps to execute against prices that have not undergone proper post-restart price discovery.

---

### Finding Description

Both L2 providers implement `_isStale` with a `futureTol` branch specifically to accommodate L2 sequencer clock lag:

```solidity
// PriceProviderL2.sol  lines 135-150
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

The identical logic appears in `ProtectedPriceProviderL2`: [2](#0-1) 

This staleness check is the **only** temporal guard. There is no sequencer uptime feed consulted anywhere in `_getBidAndAskPrice`:

```solidity
// PriceProviderL2.sol  lines 208-248
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spread, , uint256 refTime) =
        IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

    if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
        return (0, type(uint128).max);
    }
    ...
}
``` [3](#0-2) 

**Sequencer downtime scenario:**

1. L2 sequencer goes offline at real-world time `T`. `block.timestamp` on L2 freezes near `T`.
2. During the outage, Pyth/Chainlink prices continue updating off-chain but cannot be pushed to L2. The on-chain `refTime` stays at `T`.
3. Sequencer restarts at `T + D`. The first L2 block's `block.timestamp` jumps to `≈ T + D`.
4. A keeper immediately pushes a fresh oracle update; `refTime` becomes `T + D`.
5. `_isStale` evaluates `(T+D) − (T+D) = 0 ≤ MAX_TIME_DELTA` → **not stale, passes**.
6. No grace period exists. Pool swaps execute immediately against prices that reflect the moment of restart, before the market has re-established liquidity and price discovery on L2.

During the outage window, large price moves may have occurred on other venues. The first post-restart oracle push captures the new price, but arbitrageurs who were waiting for the sequencer to come back can immediately drain the pool at the pre-outage price (if the oracle update is delayed by even one block) or at an unvalidated post-restart price (if the oracle is updated in the same block as the swap).

The `ProtectedPriceProviderL2` has the identical gap: [4](#0-3) 

`PriceProvider.sol` (L1) and `AnchoredPriceProvider.sol` (L1) are **not** affected — their `_isStale` rejects any `refTime > nowTs` outright and they are not deployed on L2. [5](#0-4) 

---

### Impact Explanation

A pool swap routed through `PriceProviderL2` or `ProtectedPriceProviderL2` calls `getBidAndAskPrice()`, which calls `_getBidAndAskPrice()`. If the sequencer just restarted, the bid/ask returned is derived from an oracle price that has not been validated against sequencer liveness. A trader can receive more output tokens than the true market price permits (bad-price execution), or LP assets are sold at a discount, constituting a direct loss of LP principal. This matches the allowed impact gate: **bad-price execution — stale or unvalidated bid/ask quote reaches a pool swap**.

---

### Likelihood Explanation

L2 sequencer outages are documented historical events (Arbitrum, Optimism, Base have all experienced them). The providers are explicitly designed for L2 deployment (the `FUTURE_TOLERANCE` immutable exists solely for L2 clock skew). Any sequencer outage lasting longer than the oracle's normal update cadence creates the exploitable window. No privileged access is required — any user can submit a swap transaction in the first block after restart.

---

### Recommendation

Add a Chainlink sequencer uptime feed check with a configurable grace period inside `_getBidAndAskPrice` of both L2 providers, following the standard pattern:

```solidity
// Immutable set at construction:
AggregatorV2V3Interface public immutable sequencerUptimeFeed;
uint256 public immutable GRACE_PERIOD; // e.g. 3600 seconds

function _checkSequencer() internal view {
    (, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0 means sequencer is up
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`. If no Chainlink sequencer feed exists for the target L2, the providers should not be deployed there, or an alternative liveness signal must be used.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 60` (60-second staleness window).
2. Arbitrum sequencer goes offline for 30 minutes. On-chain oracle `refTime` freezes.
3. During outage, ETH/USD moves from $3,000 to $2,500 on CEXes.
4. Sequencer restarts. Keeper pushes Pyth price update: `mid = 250000000000` (8-dec), `refTime = block.timestamp`.
5. Attacker calls pool swap in the same block. `_isStale` returns `false` (age = 0). `getBidAndAskPrice()` returns a valid bid/ask at $2,500.
6. If the pool's internal reserves still reflect $3,000 (no rebalancing occurred during outage), the attacker buys the underpriced token and extracts the $500/ETH difference from LP reserves.
7. Alternatively, if the oracle update is delayed one block, the stale $3,000 price is served to the pool while the attacker knows the true price is $2,500 — they sell the overpriced token into the pool. [6](#0-5) [7](#0-6)
