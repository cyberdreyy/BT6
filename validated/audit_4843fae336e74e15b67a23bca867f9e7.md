### Title
`revokePusher()` Revocation Is Silently Nullified by Signature Replay in `allowPushers` — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

A pusher who calls `revokePusher()` to exit a creator's namespace can have that revocation immediately undone by the creator replaying the original consent signature through `allowPushers`, as long as the deadline has not yet expired. The code comment explicitly states the deadline is the mechanism that prevents re-establishment after revocation, but the implementation contains no nonce or used-signature guard, so the deadline only blocks *expired* signatures — not *replayed* valid ones.

---

### Finding Description

`allowPushers` accepts a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The NatSpec comment is explicit about the design intent:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

`revokePusher()` clears the mapping: [3](#0-2) 

However, `allowPushers` has **no nonce, no used-signature bitmap, and no check that the pusher is not already revoked**. The only guard is `_ensureDeadline(deadline)`, which only rejects signatures whose deadline timestamp is in the past. A signature with `deadline = now + 1 day` remains cryptographically valid for the entire day regardless of how many times `revokePusher()` is called in between.

The signed message binds `(chainid, address(this), deadline, pusher, msg.sender)`: [4](#0-3) 

Nothing in that tuple changes between the first `allowPushers` call and a replay. The creator holds the calldata from the original transaction and can resubmit it at any time before the deadline.

The analog to the staking bug is direct:

| Staking bug | Oracle analog |
|---|---|
| `delete stakers[msg.sender]` erases `owedRewards` | `revokePusher()` clears `namespaceRemapping[pusher]` |
| Creator re-stakes to restore the deleted struct | Creator replays `allowPushers` to restore the cleared mapping |
| Pending rewards are silently lost | Pending revocation is silently lost |

---

### Impact Explanation

After a pusher calls `revokePusher()`, the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments. The mapping is restored to `namespaceRemapping[pusher] = creator` and any subsequent fallback push from the pusher's address lands in the creator's namespace again, not the pusher's own namespace.

The pusher's only recourse is to stop pushing entirely and wait for the deadline to expire. During that window — which can be set to any future timestamp the creator chose when first calling `allowPushers` — the pusher cannot redirect their pushes to their own namespace. If the pusher is a live price-feed bot that continues to push (e.g., because it is unaware the revocation was undone), all its updates continue to write into the creator's namespace and are consumed by any pool or provider reading that namespace.

The `fallback` push path resolves the namespace at call time: [5](#0-4) 

So the namespace used for every subsequent push is determined by the live `namespaceRemapping` state, which the creator controls via replay.

---

### Likelihood Explanation

The creator already holds the original `allowPushers` calldata from the on-chain transaction. Replaying it costs one additional transaction. No special privilege, off-chain access, or timing constraint is required beyond the deadline not having expired. The creator is the only party who can trigger this (the signature binds `msg.sender` to the creator), so this is not an arbitrary third-party attack, but it is a unilateral action the creator can take against a pusher who has explicitly opted out.

---

### Recommendation

Add a per-pusher revocation nonce or a used-signature mapping so that once a pusher revokes, the original signature cannot re-establish the delegation:

```solidity
// Option A: per-pusher nonce incremented on every revoke
mapping(address => uint256) public pusherNonce;

// include nonce in the signed message
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// in revokePusher():
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);

// Option B: mark the exact (pusher, deadline) pair as consumed
mapping(bytes32 => bool) private _usedConsents;
bytes32 consentKey = keccak256(abi.encode(pusher, msg.sender, deadline));
require(!_usedConsents[consentKey]);
_usedConsents[consentKey] = true;
```

Either approach ensures that a revocation is permanent for the lifetime of the signed consent.

### Proof of Concept

```
// Setup
deadline = block.timestamp + 1 days
sig = pusher.sign(keccak256(chainid, oracle, deadline, pusher, creator))

// Step 1: creator establishes delegation
creator → oracle.allowPushers(deadline, [pusher], [sig])
// namespaceRemapping[pusher] == creator ✓

// Step 2: pusher revokes
pusher → oracle.revokePusher()
// namespaceRemapping[pusher] == address(0) ✓

// Step 3: creator replays the SAME calldata (deadline still valid)
creator → oracle.allowPushers(deadline, [pusher], [sig])
// namespaceRemapping[pusher] == creator again — revocation nullified

// Step 4: pusher's next fallback push lands in creator's namespace
pusher → oracle.call(slotWord)
// fallback reads namespaceRemapping[pusher] == creator
// write goes to creator's namespace, not pusher's own
```

The test suite confirms `revokePusher` clears the mapping and that post-revoke pushes land in the pusher's own namespace: [6](#0-5) 

But no test covers the replay path — calling `allowPushers` a second time with the same signature after `revokePusher` — which is the missing invariant check.

    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
