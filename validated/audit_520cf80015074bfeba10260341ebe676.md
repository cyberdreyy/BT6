### Title
Excess `msg.value` Sent With Non-WETH Swaps Is Permanently Stuck In The Router And Stealable By Any Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

Every payable entry point in `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` accepts `msg.value` unconditionally, but the internal `pay()` helper only consumes native ETH when `tokenIn == WETH`. ETH sent alongside any non-WETH swap is silently retained by the router. Because `refundETH()` transfers the entire contract balance to whoever calls it first — not to the original depositor — any third party can front-run the victim and steal the stuck ETH. Additionally, residual ETH from a prior non-WETH call is silently consumed by the next WETH swap, transferring the loss to the original depositor.

---

### Finding Description

All six user-facing entry points are declared `payable`: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The internal `pay()` helper only consumes native ETH when `token == WETH`: [7](#0-6) 

When `tokenIn != WETH`, execution falls into the final `else` branch (`safeTransferFrom`). The ETH carried in `msg.value` is never touched and accumulates in the router.

The recovery function sends the **entire** balance to whoever calls it first: [8](#0-7) 

There is no ownership check — any EOA can call `refundETH()` and drain the balance before the original depositor can recover it.

**Secondary drain path:** If the router holds residual ETH from a prior non-WETH call, the next WETH swap triggers the `nativeBalance >= value` branch: [9](#0-8) 

The router wraps and forwards the stuck ETH to the pool **without pulling from the new payer**. The new swapper receives a free (or subsidized) swap; the original depositor's ETH is silently consumed.

The `receive()` guard confirms the only non-`msg.value` ETH source is WETH itself: [10](#0-9) 

---

### Impact Explanation

Direct loss of user principal. A user who sends `msg.value > 0` alongside a non-WETH swap (e.g., USDC→DAI) loses that ETH either to a front-runner calling `refundETH()` or to the next WETH swap that silently consumes the router's native balance. No privileged access is required to trigger either loss path.

---

### Likelihood Explanation

Any user who includes `msg.value` with a non-WETH swap triggers the stuck-ETH condition. This is a realistic mistake for users interacting directly with the ABI, using generic multicall wrappers, or porting scripts from ETH-native routers. The `refundETH()` theft requires no special access — any EOA can call it in the same block.

---

### Recommendation

Add an explicit guard in each payable entry point when the input token is not WETH:

```solidity
// In exactInputSingle, exactInput, exactOutputSingle, exactOutput,
// addLiquidityExactShares, addLiquidityWeighted:
if (tokenIn != WETH) {
    require(msg.value == 0, "msg.value stuck in router");
}
```

Alternatively, automatically invoke `refundETH()` at the end of each swap to return any unused ETH to `msg.sender` within the same transaction.

---

### Proof of Concept

1. User calls `exactInputSingle({tokenIn: USDC, amountIn: 1000e6, ...})` with `msg.value = 1 ether`.
2. `pay(USDC, user, pool, 1000e6)` fires the `else` branch — USDC is pulled via `safeTransferFrom`; the 1 ETH is untouched.
3. 1 ETH sits in the router after the transaction.
4. Attacker observes the router balance (on-chain or via mempool) and calls `refundETH()`.
5. `_transferETH(attacker, 1 ether)` executes; attacker receives 1 ETH.
6. User loses 1 ETH with no recourse.

**Alternative (silent drain):** If the router already holds 1 ETH from step 3, the next user who calls `exactInputSingle({tokenIn: WETH, amountIn: 0.5 ether})` triggers `nativeBalance (1 ETH) >= value (0.5 ETH)` — the router wraps 0.5 ETH of the victim's funds and delivers it to the pool without charging the new swapper, consuming the original depositor's ETH silently.

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
