### Title
Pusher Cannot Permanently Invalidate Consent Signature Before Deadline — Creator Can Re-Establish Delegation After Revocation - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary
`allowPushers` in `CompressedOracleV1` includes no nonce in the signed consent message. After a pusher calls `revokePusher()`, the creator can immediately call `allowPushers` again with the identical signature (which remains valid until the deadline) to re-establish the delegation. The pusher has no on-chain mechanism to permanently invalidate their consent before the deadline expires.

### Finding Description

The consent message signed by a pusher is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no used-signature bitmap, and no per-pusher revocation counter in this digest. The signature is valid for every call to `allowPushers` that supplies the same `(deadline, pusher, creator)` tuple before the deadline, regardless of how many times the pusher has revoked.

`revokePusher()` only clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

It does not burn or record the consent signature. The creator can immediately replay the same bytes against `allowPushers` and restore `namespaceRemapping[pusher] = creator`.

The code comment itself acknowledges the deadline is the only guard against post-revocation re-establishment:

> "an undated signature could re-establish a delegation AFTER the pusher revoked it" [3](#0-2) 

But the deadline only bounds the window — it does not prevent the creator from cycling `allowPushers → pusher revokes → allowPushers → pusher revokes …` for the entire duration of the deadline.

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every time the creator re-establishes delegation, the pusher's subsequent `fallback` calls write into the creator's namespace instead of the pusher's own namespace.

### Impact Explanation

A pusher who operates their own namespace feeds (used by a production pool) signs a consent for a creator. Later the pusher revokes. The creator replays the same signature to re-establish delegation. The pusher's `fallback` pushes now land in the creator's namespace, leaving the pusher's own namespace feeds stale. Any `PriceProvider` / pool that reads from the pusher's own namespace `feedId` receives a stale timestamp, which every consumer treats as an invalid quote. This is a bad-price execution path: the pool either halts (if it rejects stale data) or, if the stale value happens to pass a lenient drift check, executes at a wrong price.

The creator can sustain this indefinitely until the deadline, making the pusher's own namespace permanently stale for the deadline window.

### Likelihood Explanation

- The trigger is a single public call to `allowPushers` with a signature the creator already holds.
- No privileged role, no special setup beyond having obtained the pusher's consent signature once.
- The pusher has no on-chain countermeasure other than waiting for the deadline to expire.
- Likelihood is medium: requires the creator to act adversarially after the pusher revokes, but the mechanism is fully permissionless and costs only gas.

### Recommendation

Include a per-pusher nonce in the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);
```

Expose an external `invalidateConsent()` (or `incrementNonce()`) that the pusher can call to bump their nonce, burning all outstanding consent signatures without waiting for the deadline. This is the exact pattern recommended in the referenced external report and applied by the Farcaster fix.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator, deadline = now + 30 days
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes — intends to push to own namespace again
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 4. Creator replays the SAME signature — no nonce, still within deadline
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored

// 5. Pusher's fallback push now lands in creator's namespace, not pusher's own
vm.prank(pusher);
(bool ok,) = address(oracle).call(wordPayload);
assertTrue(ok);
// pusher's own feedId has price == 0 (stale) — pool reads bad price
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, slotId, pos)).price, 0);
```
