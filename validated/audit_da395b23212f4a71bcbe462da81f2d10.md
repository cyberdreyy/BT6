### Title
Pusher revocation is bypassable via signature replay in `allowPushers` — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature over `(chainid, address(this), deadline, pusher, creator)`. There is no nonce and no per-signature invalidation. After a pusher calls `revokePusher()`, the creator can replay the **identical** signature (before the deadline) to silently re-establish the delegation. The code's own NatSpec claims the deadline prevents exactly this, but it does not — it only prevents replay after the deadline expires.

---

### Finding Description

`allowPushers` constructs the signed hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The NatSpec comment directly above the function states:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

This claim is false. The deadline only blocks replay **after** it expires. Before expiry, the same bytes can be submitted to `allowPushers` an unlimited number of times. `revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

But `allowPushers` unconditionally overwrites `namespaceRemapping[pusher] = msg.sender` on any valid signature, with no check that the mapping was previously zero or that the pusher consented again: [4](#0-3) 

The fallback push path resolves the write namespace at call time from `namespaceRemapping[msg.sender]`:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So after the creator replays the signature, every subsequent push from the pusher's address lands in the **creator's namespace**, not the pusher's own — silently, with no on-chain signal to the pusher.

---

### Impact Explanation

The pusher cannot effectively revoke their delegation for the entire remaining lifetime of the deadline. Any pushes the pusher makes after revoking — believing they are writing to their own namespace — are silently redirected to the creator's namespace. The pusher's own namespace receives nothing, so any pool or price provider configured to read `feedIdOf(pusher, slotIndex, positionIndex)` receives stale or zero data. Meanwhile, the creator's namespace continues to receive fresh prices from a pusher who explicitly withdrew consent. This is a silent feed hijack: the creator maintains unauthorized write authority over their namespace via a pusher who has revoked.

---

### Likelihood Explanation

The creator holds the pusher's original signed bytes off-chain. Replaying them costs one transaction and requires no privileged access. The window is the full remaining deadline duration — which can be days or weeks depending on what the pusher originally signed. The pusher has no on-chain mechanism to close this window short of waiting for the deadline to expire.

---

### Recommendation

Add a per-pusher-per-creator nonce that is incremented on each successful `allowPushers` call and included in the signed hash:

```solidity
mapping(address pusher => mapping(address creator => uint256)) public delegationNonce;

// in allowPushers:
uint256 nonce = delegationNonce[pusher][msg.sender]++;
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce))
);
```

Alternatively, invalidate all outstanding signatures for a pusher when `revokePusher` is called by incrementing the nonce at revocation time. Either approach ensures a revoked pusher's prior signed consent cannot be replayed.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 30 days
bytes memory sig = pusher.sign(
    keccak256(abi.encode(chainid, oracle, deadline, pusherAddr, creatorAddr))
);

// 2. Creator establishes delegation
vm.prank(creatorAddr);
oracle.allowPushers(deadline, [pusherAddr], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher revokes
vm.prank(pusherAddr);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature — no new consent from pusher
vm.prank(creatorAddr);
oracle.allowPushers(deadline, [pusherAddr], [sig]);
// namespaceRemapping[pusher] == creator again ✓ (revocation bypassed)

// 5. Pusher pushes, believing they write to their own namespace
vm.prank(pusherAddr);
(bool ok,) = address(oracle).call(slotWord);
assertTrue(ok);

// Push lands in CREATOR namespace, not pusher's own
assertEq(oracle.getOracleData(oracle.feedIdOf(creatorAddr, slotIdx, posIdx)).price, pushedPrice);
assertEq(oracle.getOracleData(oracle.feedIdOf(pusherAddr,  slotIdx, posIdx)).price, 0); // stale
```
