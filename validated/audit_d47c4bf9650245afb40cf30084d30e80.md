### Title
Pusher consent signature is replayable within its deadline window, silently re-establishing delegation after `revokePusher()` — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` has no nonce or used-signature registry. A pusher's EIP-191 consent signature is bound only to `(chainid, address(this), deadline, pusher, creator)`. After a pusher calls `revokePusher()`, the creator can immediately replay the original consent signature — as long as `block.timestamp < deadline` — to silently re-establish the delegation without any fresh consent from the pusher. The code comment claims the deadline prevents this, but the deadline only blocks expired signatures; it does not prevent replay of a still-valid signature within the deadline window.

---

### Finding Description

`allowPushers` verifies the pusher's EIP-191 consent and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
```

There is no nonce, no `usedSignatures` set, and no per-pusher revocation counter. The only guard is `_ensureDeadline`, which checks `block.timestamp <= deadline`. A signature that was valid when first used remains cryptographically valid for any future call to `allowPushers` with the same `deadline` argument, as long as the deadline has not yet passed.

The code comment at line 186–191 explicitly acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and claims the deadline is the fix. This is incorrect: the deadline limits the vulnerability window but does not close it. Within that window the same bytes can be submitted an unlimited number of times.

---

### Impact Explanation

After a pusher revokes (e.g., because their signing key was compromised), the creator calls `allowPushers` with the original signature. `namespaceRemapping[pusher]` is restored to `creator`. The compromised pusher key can now push arbitrary price data into the creator's namespace. `CompressedOracleV1.price()` and `getOracleData()` return that data to any `AnchoredPriceProvider` or pool that reads the creator's feeds. The result is bad-price execution: a pool swap consumes a manipulated bid/ask quote, causing the trader to receive more than the oracle curve permits or the pool to receive less than owed.

---

### Likelihood Explanation

The trigger requires two conditions: (1) a pusher revokes within the deadline window of a previously issued consent signature, and (2) the creator (or an attacker who controls the creator address) replays the old signature. Condition (1) is a normal operational event (key rotation, key compromise). Condition (2) is a single permissionless on-chain call with publicly available calldata from the original `allowPushers` transaction. Both conditions are reachable without any privileged access.

---

### Recommendation

**Short term:** Track consumed consent signatures. Add a `mapping(bytes32 => bool) private _usedConsentSigs` and mark each signature hash as used on first acceptance. Reject any replay.

**Long term:** Replace the deadline-based scheme with a per-pusher nonce: `mapping(address => uint256) public pusherNonce`. Include the nonce in the signed digest and increment it on every successful `allowPushers` and `revokePusher` call. This makes every consent single-use and invalidates all prior signatures on revocation, exactly as the Dharma report recommends for key-change operations.

---

### Proof of Concept

```
// Step 1 – pusher signs consent with a 30-day deadline
deadline = block.timestamp + 30 days
sig = sign(keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)))

// Step 2 – creator establishes delegation
oracle.allowPushers(deadline, [pusher], [sig])
// namespaceRemapping[pusher] == creator  ✓

// Step 3 – pusher revokes (e.g., key compromised)
vm.prank(pusher)
oracle.revokePusher()
// namespaceRemapping[pusher] == address(0)  ✓

// Step 4 – creator (or attacker) replays the SAME sig, same deadline
oracle.allowPushers(deadline, [pusher], [sig])
// namespaceRemapping[pusher] == creator  ← revocation nullified

// Step 5 – attacker (holding compromised pusher key) pushes bad price
vm.prank(attacker_with_pusher_key)
oracle.call(encodedSlotWord(badPrice, slotId, tsMs))
// oracle.getOracleData(feedIdOf(creator, slotId, pos)).price == badPrice
// Pool swap now consumes the manipulated quote
```

The replay at Step 4 succeeds because `_ensureDeadline` only checks `block.timestamp <= deadline` and there is no record that the signature was already consumed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
