### Title
`MetricOmmPoolFactory._validatePriceProvider` Does Not Verify the Provider Was Deployed by `PriceProviderFactory`, Allowing a Fake Price Provider to Feed Arbitrary Bid/Ask Prices into a Pool â€” (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory._validatePriceProvider` only checks that the candidate price provider returns the correct `token0()`/`token1()` pair. It never calls `PriceProviderFactory.isProvider()` to confirm the provider was deployed through the sanctioned factory. Because pool creation is permissionless, an attacker can deploy a malicious `IPriceProvider` that satisfies the token-pair check but returns arbitrary bid/ask prices, create a real factory pool backed by it, and drain any LP who adds liquidity to that pool.

---

### Finding Description

`_validatePriceProvider` is the sole guard executed at pool creation time and at every price-provider update:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 541-546
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [1](#0-0) 

The check is called unconditionally from `_validatePoolParameters` (pool creation) and from `proposePoolPriceProvider` / `executePoolPriceProviderUpdate`: [2](#0-1) [3](#0-2) 

Meanwhile, `PriceProviderFactory` tracks every legitimately deployed provider in an enumerable set and exposes `isProvider(address)`: [4](#0-3) 

`_validatePriceProvider` never calls this predicate. A contract that hard-codes the correct `token0`/`token1` return values but returns any value from `getBidAndAskPrice()` passes the check unconditionally.

`PriceProviderFactory.createPriceProvider` is itself permissionless â€” any address can deploy a tracked provider â€” but the attack does not even require going through the factory: [5](#0-4) 

---

### Impact Explanation

A pool created with a malicious price provider is indistinguishable from a legitimate pool at the factory level. Every swap and liquidity operation in `MetricOmmPool` calls `getBidAndAskPrice()` on the stored provider. If that provider returns an inverted, stale, or unbounded quote, the pool's `SwapMath` will compute incorrect output amounts, causing:

- **Swap conservation failure**: traders receive more output tokens than the oracle curve permits.
- **LP principal loss / pool insolvency**: the pool's reserves are drained below what LP shares represent.
- **Bad-price execution**: every swap in the pool executes at an attacker-controlled bid/ask.

The `OracleBase` abuse-protection layer (`price(feedId, pool)`, `registeredPool`, `inSwap()` binding) is entirely bypassed because the malicious provider never calls `OracleBase` at all. [6](#0-5) 

---

### Likelihood Explanation

Pool creation is permissionless (any address can call `createPool` and set itself as pool admin). The only on-chain barrier is the token-pair check, which a two-line stub satisfies. No signature, no fee, no whitelist is required. An attacker can:

1. Deploy a `MaliciousPriceProvider` stub that returns the correct tokens and an arbitrary price.
2. Call `factory.createPool(params)` â€” passes `_validatePriceProvider`.
3. Advertise the pool (same factory, same token pair as a legitimate pool).
4. Wait for LPs to deposit, then flip the price to drain them.

The attack requires no privileged role and no cooperation from any existing participant.

---

### Recommendation

Add a factory-registry check inside `_validatePriceProvider`. The `PriceProviderFactory` (and `AnchoredProviderFactory`) already expose `isProvider(address)`. The pool factory should store an approved set of price-provider factories and require the candidate to be tracked by at least one of them:

```solidity
// Pseudocode addition to _validatePriceProvider
require(
    approvedProviderFactory.isProvider(priceProvider),
    NotARegisteredProvider()
);
```

Alternatively, mirror the pattern already used in `OracleBase.register`, which validates the pool factory via an approved-factory set before whitelisting a pool: [7](#0-6) 

The same `approvedFactories` / `isProvider` pattern should be applied symmetrically on the AMM side.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {IPriceProvider} from "metric-core/contracts/interfaces/IPriceProvider.sol";

/// @dev Fake provider: passes _validatePriceProvider, returns attacker-chosen prices.
contract MaliciousPriceProvider is IPriceProvider {
    address public immutable token0;
    address public immutable token1;
    uint128 public bid;
    uint128 public ask;

    constructor(address _t0, address _t1) { token0 = _t0; token1 = _t1; }

    function setBidAsk(uint128 _bid, uint128 _ask) external { bid = _bid; ask = _ask; }

    function getBidAndAskPrice() external view returns (uint128, uint128) {
        return (bid, ask);
    }
}

contract Attack {
    function run(
        IMetricOmmPoolFactory factory,
        address token0,
        address token1,
        /* ... other pool params ... */
    ) external {
        // 1. Deploy fake provider â€” satisfies token-pair check, returns arbitrary prices.
        MaliciousPriceProvider fake = new MaliciousPriceProvider(token0, token1);
        fake.setBidAsk(type(uint128).max, 1); // extreme spread

        // 2. Create a real factory pool backed by the fake provider.
        PoolParameters memory p;
        p.token0        = token0;
        p.token1        = token1;
        p.priceProvider = address(fake); // _validatePriceProvider passes
        p.admin         = address(this);
        // ... fill remaining params ...
        address pool = factory.createPool(p);

        // 3. LPs add liquidity to the pool (attracted by the legitimate factory address).
        // 4. Attacker flips prices â†’ drains LP reserves via swap at manipulated bid/ask.
        fake.setBidAsk(1, type(uint128).max);
        // execute swap â€” pool uses fake.getBidAndAskPrice() â†’ LP funds drained.
    }
}
```
