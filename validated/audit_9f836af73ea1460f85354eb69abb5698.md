### Title
Pusher self-revocation is ineffective during the deadline window — creator can replay the original consent signature to immediately re-delegate a pusher who just called `revokePusher()` - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.revokePusher()` clears `namespaceRemapping[msg.sender]`, but the creator retains the original EIP-191 consent signature and can call `allowPushers()` again with the identical signature before the deadline expires, atomically re-establishing the delegation. The pusher cannot permanently escape the creator's namespace until the deadline timestamp passes.

---

### Finding Description

`allowPushers` binds a pusher's consent to `(chainid, address(this), deadline, pusher, creator)` and stores `namespaceRemapping[pusher] = creator`. [1](#0-0) 

The code comment explicitly acknowledges the risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

However, the deadline only prevents re-establishment **after** the deadline timestamp. During the entire deadline window the creator holds a still-valid, already-verified signature. `allowPushers` contains no nonce, no used-signature bitmap, and no check that `namespaceRemapping[pusher] == address(0)` before overwriting it.

`revokePusher` simply zeroes the mapping: [3](#0-2) 

Because the signature is not consumed or invalidated on revocation, the creator can call `allowPushers` again in the same block with the identical `(deadline, [pusher], [sig])` arguments and the mapping is restored to `creator`. This cycle can be repeated indefinitely until `block.timestamp >= deadline`.

---

### Impact Explanation

The `fallback()` push path resolves the namespace at call time: [4](#0-3) 

If the creator re-delegates the pusher after revocation, every subsequent `fallback` push from the pusher's address lands in the **creator's** namespace, not the pusher's own. A pusher running an automated off-chain bot that does not re-check `namespaceRemapping` before each push will continue writing price data into the creator's namespace against its will. The creator's namespace feeds are consumed by `price(feedId, pool)` and ultimately by pool swaps via `PriceProvider.getBidAndAskPrice()`. A pusher who intended to stop contributing to a potentially manipulated feed cannot do so; the creator can keep the pusher's live price stream flowing into their oracle namespace for the full duration of the deadline window, sustaining a feed that the pusher believed it had detached from.

---

### Likelihood Explanation

- The creator already holds the signature (they submitted it in the original `allowPushers` call).
- The deadline is a future timestamp chosen by the creator at delegation time; typical values are hours to days.
- No privileged role, no special setup, and no additional cost beyond gas is required to replay the call.
- The pusher has no on-chain mechanism to invalidate the signature before the deadline expires.

---

### Recommendation

Track consumed consent signatures with a per-pusher nonce that is incremented on every `revokePusher()` call, and include the nonce in the signed message:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher:
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

After revocation the nonce increments, making every previously issued signature invalid. A pusher who wants to re-delegate must sign a new message over the updated nonce.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 1 day
bytes memory sig = pusher.sign(
    keccak256(abi.encode(chainid, oracle, deadline, pusher, creator))
);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator  ✓

// 3. Pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0)  ✓

// 4. Creator replays the SAME signature — still valid, deadline not expired
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator  ← revocation undone

// 5. Pusher's next fallback push lands in creator's namespace, not its own
vm.prank(pusher);
(bool ok,) = address(oracle).call(slotWord);
// oracle.getOracleData(feedIdOf(creator, slot, pos)).price != 0  ← pusher's data in creator ns
// oracle.getOracleData(feedIdOf(pusher,   slot, pos)).price == 0 ← pusher's own ns empty
```

Steps 3–4 can be repeated in every block until `block.timestamp >= deadline`, making `revokePusher()` a no-op for the entire delegation window.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
