### Title
`CompressedOracleV1#allowPushers` Allows Creator to Replay Revoked Consent Signature Within Deadline Window, Permanently Re-establishing Pusher Delegation - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1#allowPushers` contains no nonce or consumed-signature tracking. After a pusher self-revokes via `revokePusher()`, the creator who holds the original EIP-191 consent signature can replay it — as many times as desired — to re-establish `namespaceRemapping[pusher] = creator` for the entire remaining lifetime of the deadline. The pusher's revocation is therefore ineffective within the deadline window, and any subsequent re-delegation to a different creator can be silently hijacked by the original creator replaying the stale signature.

---

### Finding Description

`allowPushers` verifies a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
// CompressedOracle.sol L192-L211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← unconditional overwrite, no revocation check
    emit PusherAuthorized(pusher, msg.sender);
}
```

`_ensureDeadline` (in `OracleBase`) only checks `block.timestamp <= deadline`:

```solidity
// compressed/OracleBase.sol L124-L126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
```

There is no nonce, no per-pusher revocation epoch, and no check that `namespaceRemapping[pusher]` is currently zero (i.e., that the pusher has not already revoked). The signed message commits only to `(chainid, oracle, deadline, pusher, creator)` — it carries no revocation state.

`revokePusher` clears the mapping:

```solidity
// CompressedOracle.sol L238-L243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

But the creator still holds the original signature. Because `allowPushers` performs no revocation-state check, the creator can immediately replay the identical `(deadline, [pusher], [sig])` call to restore `namespaceRemapping[pusher] = creator`. This cycle can repeat indefinitely until the deadline expires.

The code comment at L186-L191 explicitly acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and states the deadline is the mitigation — but the deadline only prevents replay *after* it expires, not *within* the window.

The `fallback()` push path reads `namespaceRemapping[msg.sender]` at call time:

```solidity
// CompressedOracle.sol L315-L316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

So every push the pusher makes while the replayed delegation is active lands in the creator's namespace, not the pusher's own.

---

### Impact Explanation

**Namespace hijack / feed starvation leading to pool halt:**

1. Pusher P signs consent for creator A with a long deadline (e.g., 7 days). Creator A calls `allowPushers` → `namespaceRemapping[P] = A`.
2. P discovers A is malicious and calls `revokePusher()` → `namespaceRemapping[P] = 0`.
3. P re-delegates to creator B (a legitimate pool operator) by signing a new consent. Creator B calls `allowPushers` → `namespaceRemapping[P] = B`.
4. Creator A replays the original signature (deadline still valid) → `namespaceRemapping[P] = A` again.
5. All of P's subsequent pushes land in A's namespace, not B's. Creator B's feeds receive no updates.
6. `AnchoredPriceProvider._readLeg` calls `offchainOracle.price(feedId, pool)` which reads B's namespace. With no fresh data, `refTime` is stale → `_isStale` returns `true` → `ok = false` → `getBidAndAskPrice` returns `(0, type(uint128).max)` → `FeedStalled` revert.
7. Every swap through B's pool reverts. LPs cannot exit via swaps; the pool is effectively bricked until B sources a replacement pusher.

The corrupted value is `namespaceRemapping[P]`, which should remain `address(0)` (or `B`) after revocation but is forced back to `A` by the replayed signature.

---

### Likelihood Explanation

- **Trigger is semi-trusted and unprivileged**: the creator is a normal participant who already holds the signature from the initial `allowPushers` call. No special role is required to replay it.
- **Deadline window is long by design**: production deployments are expected to use multi-day deadlines (the test suite uses `block.timestamp + 1 days`). The attack window is the full remaining deadline duration.
- **No on-chain signal**: the replay emits `PusherAuthorized` again, which is indistinguishable from a fresh delegation. The pusher has no on-chain way to detect the re-establishment without monitoring events.
- **Repeatable**: the creator can replay on every block until the deadline expires, making the pusher's revocation permanently ineffective within the window.

---

### Recommendation

Add a per-pusher revocation nonce (or a monotonic epoch counter) to the signed message and track it on-chain. Increment the nonce on every `revokePusher` / `removePushers` call. `allowPushers` must verify the signature against the *current* nonce and reject any signature whose nonce is stale:

```solidity
mapping(address => uint256) public pusherNonce;

// In revokePusher / removePushers:
pusherNonce[pusher]++;

// In allowPushers signature hash:
keccak256(abi.encode(
    block.chainid, address(this), deadline,
    pusher, msg.sender,
    pusherNonce[pusher]   // ← binds to current revocation epoch
))
```

Alternatively, check that `namespaceRemapping[pusher] == address(0)` before writing, so a re-delegation after revocation requires a *new* signature (with a new deadline) rather than allowing replay of the old one.
