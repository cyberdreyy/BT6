### Title
Signature Replay in `allowPushers` Bypasses `revokePusher()` Within the Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
A creator who holds a pusher's EIP-191 consent signature can call `allowPushers` repeatedly with the same signature until the deadline expires, re-establishing delegation after the pusher has called `revokePusher()`. The pusher's self-revocation is silently undone without any fresh consent.

### Finding Description
`allowPushers` verifies the pusher's signature over `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))` and writes `namespaceRemapping[pusher] = msg.sender`. [4](#0-3) 

There is no nonce, no `usedSignatures` bitmap, and no check that the pusher has previously revoked. `revokePusher()` only clears the mapping entry: [5](#0-4) 

The original signature remains cryptographically valid until `block.timestamp > deadline`. The creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments at any point before the deadline, restoring `namespaceRemapping[pusher] = creator` without the pusher's knowledge or fresh consent.

### Impact Explanation
After re-delegation, the pusher's fallback pushes are routed into the creator's namespace instead of the pusher's own namespace. The creator can simultaneously push arbitrary price data into the same namespace slots. Any pool configured to read feeds from the creator's namespace will receive creator-controlled prices, enabling bad-price execution (manipulated bid/ask quotes reaching live swaps). This satisfies the "bad-price execution" impact gate.

### Likelihood Explanation
The attacker must be the creator who originally called `allowPushers` (not a random third party — the signature binds `msg.sender`). The window is bounded by the deadline the pusher chose when signing. A pusher who signs a 24-hour deadline and revokes after 1 hour leaves a 23-hour replay window. This is a realistic scenario for any pusher who revokes due to a key compromise or relationship change.

### Recommendation
Track consumed signatures. Add a `mapping(bytes32 => bool) private _usedDelegationSigs` and mark the signature hash as used on the first successful `allowPushers` call:

```solidity
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedDelegationSigs[sigHash], "signature already used");
_usedDelegationSigs[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed message and increment it on each successful delegation or revocation.

### Proof of Concept
```solidity
function testRevokeBypassViaSignatureReplay() public {
    uint256 deadline = block.timestamp + 1 hours;

    // 1. Pusher signs consent
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator delegates
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature before deadline
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // succeeds — no revert

    // 5. Invariant violated: pusher is re-delegated without fresh consent
    assertEq(oracle.namespaceRemapping(pusher), creator);
}
```

```text
        uint256 l = pushers.length;
        require(l == signatures.length);
```
