### Title
Delegated pusher can lock `CompressedOracleV1` feed with a future timestamp, DoS-ing pool swaps for up to `MAX_TIME_DRIFT` seconds - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.fallback()` accepts slot-word timestamps up to `MAX_TIME_DRIFT` seconds in the future. Because the only freshness gate is a strict monotonicity check (`isAfter`), a future-timestamped write permanently blocks any honest pusher from overwriting the slot until real time catches up. When the pool reads the feed through `PriceProvider` (or `PriceProviderL2` with a smaller `FUTURE_TOLERANCE`), the stored future `refTime` fails the staleness check and the pool halts with `FeedStalled` for the entire lock-out window — up to 60 seconds in the production configuration.

---

### Finding Description

**Oracle ingestion gate — allows future timestamps up to `MAX_TIME_DRIFT`**

`revertIfAfterBlockTimeWithDrift` permits any timestamp satisfying `t0.toSeconds() <= block.timestamp + drift`: [1](#0-0) 

With the production value `MAX_TIME_DRIFT = 60` (seconds), a pusher may embed `tsMs = (block.timestamp + 60) * 1000` in a slot word and the oracle will accept it.

**Fallback push path — stores the future-timestamped word**

Inside `fallback()`, after the drift check passes, the slot is overwritten unconditionally if the new timestamp is strictly greater than the stored one: [2](#0-1) 

The same logic applies in `updateBySignature()`: [3](#0-2) 

**Monotonicity check locks out honest pushers**

Once the slot holds `tsMs = (block.timestamp + 60) * 1000`, any honest push carrying the real current time (`tsMs ≈ block.timestamp * 1000`) fails `isAfter` and is silently skipped: [4](#0-3) 

The creator cannot correct the slot for the full `MAX_TIME_DRIFT` window even after revoking the pusher, because the stored timestamp is already in the future.

**Price provider staleness check — rejects the future `refTime`, stalling the pool**

`CompressedOracleV1.price()` returns `refTime = timestampMs.toSeconds()`, which equals `block.timestamp + 60`. `PriceProvider._isStale()` (L1 variant) treats any future `refTime` as stale: [5](#0-4) 

`PriceProviderL2._isStale()` tolerates future timestamps only within `FUTURE_TOLERANCE` (production value: 5 s). A pusher using any timestamp between `FUTURE_TOLERANCE + 1` and `MAX_TIME_DRIFT` seconds ahead (i.e., 6–60 s) also triggers the stale path: [6](#0-5) 

In both cases `_getBidAndAskPrice()` returns `(0, type(uint128).max)` and `getBidAndAskPrice()` reverts with `FeedStalled`, making the pool's swap path completely unusable for the lock-out window. [7](#0-6) 

**Production parameters confirm the gap**

`networks.json` sets `oracle.maxTimeDrift = 60` and (for L2 providers) `futureTolerance = 5`, leaving a 55-second window where a stored timestamp is accepted by the oracle but rejected as stale by the provider: [8](#0-7) 

---

### Impact Explanation

Every swap through a pool whose `PriceProvider` / `PriceProviderL2` reads the affected `CompressedOracleV1` feed reverts with `FeedStalled` for up to `MAX_TIME_DRIFT = 60` seconds. Liquidity providers cannot execute swaps on their behalf; arbitrageurs cannot rebalance the pool; any time-sensitive trade fails. The attack is repeatable: a new future-timestamped push can be submitted immediately after the previous lock-out expires, sustaining the DoS indefinitely as long as the pusher retains delegation.

This satisfies the "broken core pool functionality / unusable swap flows" criterion in the impact gate.

---

### Likelihood Explanation

The trigger is a **delegated pusher** — a semi-trusted actor explicitly authorized by the feed creator via `allowPushers`. The creator may delegate multiple pushers for redundancy or automation; any one of them can execute this attack. The creator can call `removePushers()` to revoke the delegation, but the 60-second lock-out is already in effect and the corrective push cannot land until the stored future timestamp expires. The attack requires no special on-chain state beyond an active delegation and costs only one transaction.

---

### Recommendation

Clamp the stored timestamp to `block.timestamp` (in milliseconds) before writing, regardless of what the pusher supplied:

```diff
- _writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
+ // Clamp timestamp to current block time before storing
+ uint256 clampedTs = block.timestamp * 1000;
+ uint256 clampedWord = (word & ~(X56 << 8)) | (clampedTs << 8);
+ _writeStorage(key, bytes32(clampedWord & ~uint256(0xff)));
```

Alternatively, reduce `MAX_TIME_DRIFT` to be no larger than `FUTURE_TOLERANCE` on every chain where a `PriceProviderL2` reads the oracle, so that any timestamp the oracle accepts is also accepted by the provider's staleness check. This eliminates the gap that enables the DoS variant while preserving the intended clock-skew tolerance.

### Proof of Concept

```solidity
// Assume: creator has delegated `maliciousPusher` via allowPushers().
// Assume: a MetricOmmPool is configured with PriceProvider → CompressedOracleV1 feed.

// Step 1 — malicious pusher locks the slot with a future timestamp
uint56 futureTs = uint56((block.timestamp + 60) * 1000); // MAX_TIME_DRIFT = 60
uint48 badPrice = _packRaw(1, 3, 3); // arbitrary manipulated price
bytes memory payload = _wordAt(slotId, positionIndex, badPrice, futureTs);
vm.prank(maliciousPusher);
(bool ok,) = address(oracle).call(payload);
assertTrue(ok); // accepted — within MAX_TIME_DRIFT

// Step 2 — creator tries to correct immediately, but is locked out
uint56 nowTs = uint56(block.timestamp * 1000);
uint48 correctPrice = _packRaw(1_000_000, 3, 3);
bytes memory correction = _wordAt(slotId, positionIndex, correctPrice, nowTs);
vm.prank(creator);
(ok,) = address(oracle).call(correction);
assertTrue(ok); // no revert, but silently skipped — nowTs < futureTs

// Verify: bad price is still stored
IOffchainOracle.OracleData memory d = oracle.getOracleData(feedId);
assertEq(TimeMs.unwrap(d.timestampMs), futureTs); // future timestamp persists

// Step 3 — pool swap reverts for the next 60 seconds
// PriceProvider._isStale: refTime = block.timestamp + 60 > block.timestamp → true
vm.expectRevert(PriceProvider.FeedStalled.selector);
pool.swap(...); // unusable for MAX_TIME_DRIFT seconds
```
