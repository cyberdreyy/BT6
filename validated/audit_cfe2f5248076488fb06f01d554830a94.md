### Title
Single `MAX_REF_STALENESS` Applied to Both Legs in Synthetic Mode Allows Stale Price from Faster Feed to Reach Pool — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a synthetic ratio mode where `price(baseFeedId) / price(quoteFeedId)` is computed (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). Both legs are staleness-checked with the same single immutable `MAX_REF_STALENESS`. Because the two feeds can have different natural update frequencies, the deployer is forced to set `MAX_REF_STALENESS` to the slower feed's heartbeat, which silently permits the faster feed to be stale by up to that full window. A stale base-leg price produces a wrong synthetic ratio that is quoted directly to the pool as bid/ask.

---

### Finding Description

`_readLeg()` applies `MAX_REF_STALENESS` identically to every feed it is called with:

```solidity
// AnchoredPriceProvider.sol L277-L295
function _readLeg(bytes32 feedId)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

    // Stale reference → not ok.
    if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
    ...
    ok = true;
}
``` [1](#0-0) 

In synthetic mode `_getBidAndAskPrice()` calls `_readLeg` twice — once for `baseFeedId`, once for `quoteFeedId` — both against the same `MAX_REF_STALENESS`:

```solidity
// AnchoredPriceProvider.sol L258-L272
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
        ...
        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
        spreadBps += spreadBps2;
    }
    return _computeBidAsk(mid, spreadBps);
}
``` [2](#0-1) 

`MAX_REF_STALENESS` is a single immutable set at construction: [3](#0-2) 

The `AnchoredProviderFactory` validates `maxRefStaleness` only against the envelope of `baseFeedId`. The factory comment explicitly acknowledges that `quoteFeedId` is not envelope-validated:

> "The envelope is keyed on `baseFeedId` (the provider's class); the ref feed only contributes its uncertainty and is validated for existence at provider construction." [4](#0-3) 

The factory's `createAnchoredProvider` enforces the envelope only on `baseFeedId`'s class: [5](#0-4) 

**Concrete scenario:**

| Feed | Natural heartbeat | Role |
|---|---|---|
| BTC/USD (Chainlink HFS) | ~1 second | `baseFeedId` |
| ETH/USD (Chainlink v3) | ~1 hour | `quoteFeedId` |

The deployer must set `MAX_REF_STALENESS ≥ 3600 s` or the ETH/USD leg will revert on every swap. With `MAX_REF_STALENESS = 3600 s`, the BTC/USD leg is accepted as fresh even if it is 3599 seconds old. The synthetic ratio `BTC/ETH = stale_BTC_USD / fresh_ETH_USD` is wrong by however much BTC moved in that window. The band in `_computeBidAsk` is centered on this stale ratio, so the clamp does not correct it — it only clips a custom source quote, not the reference band itself. [6](#0-5) 

---

### Impact Explanation

A stale `baseFeedId` price produces a wrong synthetic mid. `_computeBidAsk` derives the reference band from that wrong mid, so the pool's bid/ask are both shifted in the direction of the stale price. A trader who observes the stale price can:

- **Buy** the synthetic asset at the stale (lower) bid if the true price has risen, extracting value from LPs.
- **Sell** the synthetic asset at the stale (higher) ask if the true price has fallen, again extracting value from LPs.

This is a direct loss of LP principal through bad-price execution. The `AnchoredPriceProvider` is the standard provider for public pools, so any synthetic pool is affected.

---

### Likelihood Explanation

- The synthetic mode is explicitly documented, supported, and permissionlessly deployable via `AnchoredProviderFactory.createAnchoredProvider()`.
- Any two feeds with different natural heartbeats create the mismatch. Chainlink Data Streams offers both HFS (millisecond-resolution, near-continuous) and standard v3/v4 (second-resolution, heartbeat-driven) feeds on the same oracle contract.
- The deployer has no mechanism to set per-leg staleness; they must pick one value that accommodates the slower feed.
- The oracle push model means the faster feed can legitimately lag behind its natural heartbeat during any pusher slowdown, making the staleness window reachable without any attacker action.

---

### Recommendation

Add a separate `maxQuoteStaleness` immutable for the quote leg in synthetic mode, passed at construction alongside `maxRefStaleness`. In `_readLeg`, pass the appropriate bound depending on which feed is being read:

```solidity
// In _getBidAndAskPrice():
(uint256 mid, uint256 spreadBps, , bool ok) = _readLegWithStaleness(baseFeedId, MAX_REF_STALENESS);
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLegWithStaleness(_quote, MAX_QUOTE_STALENESS);
    ...
}
```

The `AnchoredProviderFactory` should accept and validate `maxQuoteStaleness` against the `quoteFeedId`'s class envelope (or a separate quote-feed envelope), and pass it through to the provider constructor.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

// Setup:
// - baseFeedId  = BTC/USD, natural heartbeat 1 s
// - quoteFeedId = ETH/USD, natural heartbeat 3600 s
// - MAX_REF_STALENESS = 3600 s (forced by the slow quote feed)

// 1. At T=0: pusher stores BTC/USD = 100_000e8, ETH/USD = 3_000e8.
//    Synthetic BTC/ETH = 100_000/3_000 ≈ 33.33.

// 2. At T=3599: BTC rises to 110_000e8 but the pusher has not yet updated BTC/USD.
//    ETH/USD was refreshed at T=3600 (just expired), so it is still fresh.

// 3. Attacker calls pool.swap() → provider.getBidAndAskPrice():
//    _readLeg(BTC/USD): refTime = 0, nowTs = 3599, delta = 3599 ≤ 3600 → NOT stale → mid1 = 100_000e8
//    _readLeg(ETH/USD): refTime = 0, nowTs = 3599, delta = 3599 ≤ 3600 → NOT stale → mid2 = 3_000e8
//    synthetic mid = 100_000e8 / 3_000e8 * 1e8 = 33.33e8  (stale; true = 36.67e8)

// 4. Pool quotes bid/ask around 33.33 BTC/ETH.
//    Attacker buys BTC/ETH at the stale ask (≈33.33), immediately worth ≈36.67 on the market.
//    LP loss per unit ≈ (36.67 - 33.33) / 36.67 ≈ 9.1%.
```

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }

    function _computeBidAsk(uint256 mid, uint256 spreadBps)
        internal view returns (uint128, uint128)
    {
        // Circuit breaker: extreme (combined) uncertainty means the feed is clearly broken.
        if (spreadBps > MAX_SPREAD_BPS) {
            return (0, type(uint128).max);
