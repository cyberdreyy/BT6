### Title
`allowPushers` deadline-only replay allows creator to silently re-delegate a revoked pusher, enabling namespace hijack and bad-price injection — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `allowPushers` function in `CompressedOracleV1` uses a deadline-based EIP-191 signature to authorize pushers into a creator's namespace. The code comment explicitly claims the deadline prevents re-establishment of a delegation after the pusher revokes it. This claim is incorrect: the deadline only blocks replay *after* it expires, not *before*. A creator can replay a pusher's old signature at any time before the deadline — including after the pusher has called `revokePusher()` — to silently re-establish the delegation without fresh consent. This redirects the pusher's future updates into the creator's namespace, corrupting feeds with wrong price data and feeding bad prices into pools.

---

### Finding Description

`allowPushers` verifies a pusher's EIP-191 signature over the tuple `(block.chainid, address(this), deadline, pusher, msg.sender)` and writes `namespaceRemapping[pusher] = msg.sender` (the creator): [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)`, which checks `block.timestamp <= deadline`: [2](#0-1) 

There is **no nonce** and **no used-signature tracking**. The same `(deadline, pusher, creator)` tuple can be submitted to `allowPushers` an unlimited number of times before the deadline expires.

When a pusher calls `revokePusher()`, the mapping is cleared to `address(0)`: [3](#0-2) 

But the creator can immediately call `allowPushers` again with the **identical signature** (deadline still in the future), passing both the deadline check and the ECDSA check, and writing `namespaceRemapping[pusher] = creator` again. The revocation is silently undone.

The code comment at line 191–192 explicitly claims the deadline prevents this:

> *"the deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

This claim is wrong. The deadline prevents replay only after it expires. Within the deadline window, the signature is unconditionally reusable.

Once the delegation is re-established, the `fallback()` push path resolves the namespace from `namespaceRemapping[msg.sender]`: [5](#0-4) 

Every subsequent push from the pusher — even pushes the pusher intends for its own namespace — is written into the creator's namespace instead.

---

### Impact Explanation

The `fallback()` push writes the entire 256-bit slot word (all four lanes + shared timestamp) into the storage key `namespace | slotId`: [6](#0-5) 

If pusher P is simultaneously operating its own namespace (e.g., pushing ETH/USD data at slot 0, position 0 of its own feeds), those pushes — after the silent re-delegation — land in creator A's slot 0, position 0, which may be a BTC/USD feed. The `getOracleData` path decodes the corrupted slot and returns the wrong price: [7](#0-6) 

Any pool whose `PriceProvider` reads creator A's BTC/USD `feedId` will receive ETH/USD prices, executing swaps at a completely wrong bid/ask. This is a direct bad-price execution impact: traders receive more output than the oracle curve permits, or the pool receives less input than owed, causing direct loss of LP principal.

---

### Likelihood Explanation

The preconditions are:

1. **Creator A is malicious or economically motivated** — they replay the signature to profit from price manipulation in a pool that uses their feeds. Creator A is a semi-trusted namespace owner, not a protocol admin, so this is within the contest-relevant semi-trusted trigger scope.
2. **Pusher P continues to push after revocation** — realistic when P is a multi-creator price bot that revokes from creator A but continues pushing for its own namespace or other creators.
3. **Deadline has not expired** — delegation deadlines are set by the creator, who can choose a long window (e.g., 30 days) to maximize the replay window.

All three conditions are independently plausible and can be arranged by a single malicious creator. Likelihood is **Medium**.

---

### Recommendation

Add a per-pusher nonce to the signed message and increment it on each successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(
                block.chainid, address(this), deadline,
                pusher, msg.sender,
                pusherNonce[pusher]++   // <-- nonce invalidates old signatures
            ))
        );
        require(pusher == ECDSA.recover(hash, signatures[i]));

        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

`revokePusher()` should also increment the nonce so that any outstanding signed-but-not-yet-submitted consent is invalidated at revocation time.

### Proof of Concept

```
t=0:  Creator A calls allowPushers(deadline=T, [pusherP], [sig])
      → namespaceRemapping[pusherP] = creatorA
      → sig covers (chainid, oracle, T, pusherP, creatorA)

t=1:  Pusher P calls revokePusher()
      → namespaceRemapping[pusherP] = address(0)
      → Pusher P believes it now pushes to its own namespace

t=2:  Creator A calls allowPushers(deadline=T, [pusherP], [sig])
      with the IDENTICAL sig from t=0
      → _ensureDeadline(T): block.timestamp(2) <= T  ✓
      → ECDSA.recover(hash, sig) == pusherP           ✓
      → namespaceRemapping[pusherP] = creatorA        ← revocation silently undone

t=3:  Pusher P pushes ETH/USD data to its own slot 0
      (calldata word: [ETH_price][ETH_spread0][ETH_spread1][ts][slotId=0])
      → fallback() reads namespaceRemapping[pusherP] = creatorA
      → key = creatorA_namespace | 0
      → Creator A's BTC/USD slot 0 is overwritten with ETH/USD data

t=4:  Pool calls getBidAsk() → PriceProvider reads creator A's feedId
      → getOracleData returns ETH/USD price for BTC/USD feed
      → Swap executes at wrong price → LP loss
```
