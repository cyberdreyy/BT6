Looking at the `allowPushers` and `revokePusher` logic in `CompressedOracle.sol`:

The vulnerability is **real and exploitable**. Here is the complete analysis:

### Title
Signature Replay in `allowPushers` Silently Undoes `revokePusher` Within Deadline Window â€” (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` performs no used-signature tracking. A creator who holds a pusher's EIP-191 consent signature can call `allowPushers` repeatedly with the **same signature** any number of times before the deadline expires. This means a pusher's `revokePusher()` call can be silently overwritten by the creator re-submitting the original signature, forcing the pusher back into the creator's namespace without renewed consent.

### Finding Description

`allowPushers` validates the pusher's signature and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The only freshness gate is `_ensureDeadline(deadline)`. There is no nonce, no `usedSignatures` mapping, and no check that the pusher is not currently revoked. [2](#0-1) 

`revokePusher` sets `namespaceRemapping[msg.sender] = address(0)`: [3](#0-2) 

Because `allowPushers` does not record that the signature was consumed, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple, re-writing `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone.

The code comment itself acknowledges the concern but only addresses the post-deadline case:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The deadline prevents replay **after** expiry, but does nothing to prevent replay **within** the deadline window â€” which is exactly when the pusher's revocation is most likely to occur.

### Impact Explanation

The `fallback()` push path resolves the namespace at call time: [5](#0-4) 

After the creator silently re-establishes the delegation:

1. **Pusher's pushes land in the creator's slots**, not the pusher's own namespace. The pusher believes they are writing to their own feeds (e.g., for a different asset or a different consumer), but the data is written into the creator's price slots.
2. **Creator's production feeds are corrupted** with data the pusher did not intend to provide for the creator's namespace.
3. **Pusher's own feeds go stale** â€” their pushes never reach their own namespace, so any pool consuming the pusher's own feeds receives stale/zero prices.
4. The creator can repeat this indefinitely within the deadline window, making the pusher's revocation effectively unenforceable for up to the full deadline duration (e.g., 1 hour).

Both corrupted creator feeds and stale pusher feeds directly affect production oracle quotes consumed by pools.

### Likelihood Explanation

- The creator already holds the pusher's signature (they used it in the first `allowPushers` call).
- No additional privileges or off-chain coordination are needed.
- The attack is a single on-chain transaction (`allowPushers` with the cached signature).
- The pusher has no on-chain mechanism to invalidate the signature before the deadline; they can only call `revokePusher` again, which the creator can immediately undo again.

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedSignatures` and mark each signature hash as used on first consumption:

```solidity
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedSignatures[sigHash], "signature already used");
_usedSignatures[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed payload so the pusher can invalidate outstanding signatures by incrementing their nonce via a dedicated `invalidatePusherNonce()` function.
