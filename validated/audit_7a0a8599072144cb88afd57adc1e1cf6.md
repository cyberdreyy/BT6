### Title
`allowPushers` Signature Replay Lets Creator Silently Re-Establish Revoked Pusher Delegation, Feeding Wrong Prices Into Pool Feeds — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracleV1.allowPushers` signs consent over `(chainid, oracle, deadline, pusher, creator)` with no nonce. The same signature is replayable an unlimited number of times before the deadline expires. A creator can therefore call `allowPushers` again with the original signature immediately after a pusher calls `revokePusher()`, silently re-mapping the pusher's namespace back to the creator's. The pusher, believing their delegation is cleared, may then push price data intended for their own namespace; those pushes land in the creator's namespace instead, feeding incorrect prices into any pool that reads the creator's feeds.

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no "used-signature" bitmap, and no on-chain record that the signature was already consumed. `_ensureDeadline` only checks `block.timestamp <= deadline`; it does not prevent the same `(deadline, pusher, creator, sig)` tuple from being submitted repeatedly. [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But nothing prevents the creator from immediately calling `allowPushers` again with the identical signature, writing `namespaceRemapping[pusher] = creator` back. The contract's own NatSpec acknowledges the risk but incorrectly claims the deadline is the fix:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The deadline only bounds the replay window to `[now, deadline]`; it does not prevent replay within that window. `revokePusher` is therefore not a unilateral right — the creator can nullify it at will for the entire lifetime of the signed consent.

The `fallback()` push path reads `namespaceRemapping[msg.sender]` at call time to determine the target namespace: [5](#0-4) 

If the mapping has been silently re-established, every subsequent push by the pusher lands in the creator's namespace, not the pusher's own.

### Impact Explanation

A pusher who has revoked and begun pushing prices for their own namespace (e.g., different asset pairs, different slot layout) will have those pushes silently redirected into the creator's namespace. The creator's feeds — which pools read via `price(feedId, pool)` — receive prices that were never intended for those feeds. This constitutes bad-price execution: the pool's bid/ask is derived from an oracle value that does not correspond to the pool's actual asset pair. Depending on the magnitude of the price mismatch, swappers receive more output than the correct price permits, or the pool accumulates an incorrect fee, both of which are direct loss-of-funds scenarios above Sherlock thresholds.

### Likelihood Explanation

The creator holds the original signature and can replay it in a single transaction at any time before the deadline. A pusher who signs a long-lived consent (e.g., 30 days, 1 year — common for operational convenience) and later revokes has no on-chain mechanism to prevent the creator from re-establishing the delegation. The pusher has no way to detect the re-establishment without monitoring chain state. The scenario is reachable by any creator who holds a valid, unexpired pusher signature.

### Recommendation

Add a per-`(pusher, creator)` nonce to the signed digest and increment it on every successful `allowPushers` call. Alternatively, record each consumed `(deadline, pusher, creator, sigHash)` tuple in a `mapping(bytes32 => bool) usedSignatures` and revert on reuse. Either approach makes each signed consent single-use, so `revokePusher` becomes truly final within the deadline window.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 30 days
bytes32 hash = keccak256(abi.encode(chainid, oracle, deadline, pusher, creator));
bytes memory sig = sign(PUSHER_KEY, hash);

// 2. Creator establishes delegation
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature — no nonce, no used-sig check
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);   // succeeds
// namespaceRemapping[pusher] == creator again ✗

// 5. Pusher pushes prices for their own namespace (slot 0, ETH/USD)
//    — but the push lands in the creator's namespace (slot 0, BTC/USD)
vm.prank(pusher);
(bool ok,) = address(oracle).call(wordAt(slotId=0, pos=0, ethUsdPrice, ts));
// creator's BTC/USD feed now holds ETH/USD price → bad-price execution in pool
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```

        address creator = namespaceRemapping[msg.sender];
