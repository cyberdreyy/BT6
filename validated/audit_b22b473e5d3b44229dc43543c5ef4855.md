### Title
Creator Can Replay `allowPushers` Signature Within Deadline Window to Nullify Pusher Self-Revocation, Routing Stale or Wrong-Asset Data Into Creator's Price Feeds — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` accepts an EIP-191 signature from a pusher consenting to push into a creator's namespace. The signature commits to `(chainid, oracle, deadline, pusher, creator)` but carries **no revocation nonce**. After a pusher self-revokes via `revokePusher()`, the creator can replay the identical, still-valid signature to silently re-establish delegation before the deadline expires. The pusher's subsequent pushes — now intended for their own namespace — are transparently redirected into the creator's namespace, potentially overwriting live price slots with data for the wrong asset and feeding bad bid/ask prices into pools.

---

### Finding Description

`allowPushers` sets `namespaceRemapping[pusher] = msg.sender` after verifying the pusher's EIP-191 signature:

```solidity
// CompressedOracle.sol lines 204-209
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

`revokePusher` clears the mapping:

```solidity
// CompressedOracle.sol lines 238-243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

Because the signature tuple `(chainid, oracle, deadline, pusher, creator)` is **stateless** — it contains no nonce, no revocation counter, and no "first-use" flag — the creator can call `allowPushers` a second time with the **exact same signature** at any point before `deadline`. The only guard is `_ensureDeadline`:

```solidity
// compressed/OracleBase.sol line 124-126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
```

This check only prevents replay **after** the deadline, not before. Within the deadline window the creator can replay the signature an unlimited number of times, each time overwriting `namespaceRemapping[pusher]` back to themselves.

The code comment acknowledges the risk but incorrectly claims the deadline fully mitigates it:

> *"the deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The comment describes the post-deadline case; the pre-deadline replay is unaddressed.

**Attack path:**

1. Creator C calls `allowPushers(deadline=T+7days, [P], [sig])` — `namespaceRemapping[P] = C`.
2. Pusher P calls `revokePusher()` — `namespaceRemapping[P] = 0`.
3. P begins pushing to **their own** namespace (e.g., slot 0 = BTC/USD).
4. Creator C calls `allowPushers(T+7days, [P], [sig])` again with the **same signature** — `namespaceRemapping[P] = C` is restored.
5. P's BTC/USD pushes are now silently written into C's namespace slot 0, which C's pools treat as ETH/USD.
6. Pools reading C's ETH/USD feed receive BTC/USD prices → bad-price execution on every swap.

The `fallback()` push path resolves the target namespace at call time:

```solidity
// CompressedOracle.sol lines 315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

P has no visibility into this remapping and cannot detect that their pushes are being hijacked.

---

### Impact Explanation

Pools anchored to the creator's compressed-oracle feeds call `price(feedId, pool)` → `_price(feedId)` → `getOracleData(feedId)`, which reads the slot written by the redirected pusher. If the pusher is writing prices for a different asset (e.g., BTC/USD into an ETH/USD slot), every swap in the affected pool executes at a grossly wrong mid-price. Depending on pool depth and the price ratio between the two assets, traders can drain LP principal or receive far more output than the oracle permits — a direct loss of user principal and LP assets above Sherlock thresholds.

---

### Likelihood Explanation

The creator is a semi-trusted namespace owner, not the privileged ADMIN. A creator who wants to retain a pusher against the pusher's will (e.g., to maintain a feed that the pusher is now populating with different data) has a clear economic incentive. The replay requires only a single additional transaction with already-held calldata; no new cryptographic material is needed. Deadlines in practice are set days to weeks in the future to accommodate operational windows, leaving a large replay window.

---

### Recommendation

Introduce a per-pusher revocation nonce that is incremented on every `revokePusher` or `removePushers` call and include it in the signed message:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline,
                         pusher, msg.sender, pusherNonce[pusher]))
);

// In revokePusher / removePushers:
pusherNonce[pusher]++;
namespaceRemapping[pusher] = address(0);
```

This makes every previously issued signature invalid the moment the pusher revokes, regardless of whether the deadline has passed.

         → namespaceRemapping[P] = C  ✓

t=2:   P calls revokePusher()
         → namespaceRemapping[P] = 0  ✓ (P believes they are free)

       fallback(): namespaceRemapping[P] == C → creator = C
       Slot written: C's namespace, slot 0 ← BTC/USD price
