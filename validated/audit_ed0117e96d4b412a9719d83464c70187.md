### Title
`allowPushers` signed consent is replayable within the deadline window, silently re-establishing delegation after a pusher's explicit revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` includes a `deadline` in the EIP-191 signature domain specifically to prevent a pusher's signed consent from being replayed after the pusher has revoked their delegation. However, the deadline only blocks replay **after it expires**. Within the deadline window, the creator can call `allowPushers` again with the exact same `(deadline, pusher, signature)` triple to silently re-establish `namespaceRemapping[pusher] = creator` immediately after the pusher calls `revokePusher()`. There is no used-signature registry, no nonce, and no per-revocation counter — the only guard is the deadline itself.

### Finding Description

`allowPushers` verifies the pusher's EIP-191 signature over `(block.chainid, address(this), deadline, pusher, msg.sender)` and then unconditionally writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The code's own comment acknowledges the invariant: *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* — and claims the deadline solves this: [2](#0-1) 

But `_ensureDeadline` only rejects calls **after** the deadline timestamp. Within the window `[now, deadline]`, the same signature is valid on every call. `revokePusher` clears the mapping: [3](#0-2) 

Nothing prevents the creator from immediately calling `allowPushers` again with the identical `(deadline, pusher, sig)` to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So every push the pusher makes after their revocation — believing they are writing to their own namespace — is silently redirected into the creator's namespace.

### Impact Explanation

After the creator replays the delegation:

1. The pusher's fallback pushes land in the creator's namespace, not the pusher's own. Any `AnchoredPriceProvider` or pool consuming `feedIdOf(creator, slot, pos)` receives prices from a pusher who explicitly opted out.
2. The pusher's own namespace (`feedIdOf(pusher, slot, pos)`) is never updated, so any consumer of that feed sees a stale or zero price — triggering `FeedStalled` halts or stale-price execution depending on `MAX_REF_STALENESS`.
3. If the pusher revoked because their key was compromised, the creator's re-delegation keeps the compromised key writing live prices into the creator's feed, which flows through `AnchoredPriceProvider._getBidAndAskPrice()` into pool swaps. [5](#0-4) 

### Likelihood Explanation

The trigger is a single additional `allowPushers` call by the creator — no special privilege, no new signature needed, no on-chain cost beyond gas. Any creator who wants to retain a pusher's write authority against the pusher's will can do so for the entire deadline window (up to the maximum deadline the pusher agreed to). Deadlines of hours or days are operationally normal.

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedConsents` keyed on the full signature hash (or the signed digest). Mark it `true` on first use and revert on replay. Alternatively, include a per-pusher nonce in the signed domain so each consent is single-use regardless of deadline.

```solidity
mapping(bytes32 => bool) private _usedConsents;

// inside allowPushers, after ECDSA.recover:
require(!_usedConsents[hash], "consent already used");
_usedConsents[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

### Proof of Concept

```
T=0:  pusher signs sig = sign(keccak256(chainid, oracle, deadline=T+1day, pusher, creator))
T=1:  creator calls allowPushers(deadline, [pusher], [sig])
        → namespaceRemapping[pusher] = creator  ✓

T=2:  pusher calls revokePusher()
        → namespaceRemapping[pusher] = address(0)  (pusher believes delegation is gone)

T=3:  creator calls allowPushers(deadline, [pusher], [sig])  // SAME sig, deadline still valid
        → namespaceRemapping[pusher] = creator  (re-established without pusher's knowledge)

T=4:  pusher pushes slot word via fallback
        → namespace resolved to creator (line 315-316)
        → feedIdOf(creator, slot, pos) updated with pusher's data
        → feedIdOf(pusher, slot, pos) stays at 0 / stale
        → AnchoredPriceProvider consuming creator's feed receives pusher's price
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
