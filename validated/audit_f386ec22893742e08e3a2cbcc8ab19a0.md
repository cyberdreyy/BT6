### Title
Precision loss in `_getBidAskFrom` collapses bid/ask spread to zero for low-priced assets, allowing swaps at oracle mid with no spread protection — (`smart-contracts-poc/contracts/PriceProvider.sol`, `ProtectedPriceProvider.sol`)

---

### Summary

`PriceProvider` and `ProtectedPriceProvider` compute the confidence-adjusted spread delta as:

```solidity
uint256 delta = midPrice * confidence / CONFIDENCE_BASE;
```

where `CONFIDENCE_BASE = 1e10`. When `midPrice` is small (low-priced assets) and `confidence = spread * confidenceParam` is modest, integer division truncates `delta` to zero. This collapses `bid = ask = midPrice`, bypassing the intended spread protection. The only downstream guard (`bidOut >= askOut`) fails to catch this for the common case where `midPrice` is not exactly divisible by `1e8`, allowing the pool to execute swaps at the oracle mid price with a spread of 1 Q64 unit — effectively zero — instead of the intended non-zero spread.

---

### Finding Description

In `PriceProvider._getBidAskFrom` (line 138) and the identical copy in `ProtectedPriceProvider._getBidAskFrom` (line 143):

```solidity
function _getBidAskFrom(uint256 midPrice, uint256 confidence) internal pure returns (uint256 bid, uint256 ask) {
    uint256 delta = midPrice * confidence / CONFIDENCE_BASE;   // ← truncates to 0
    bid = delta >= midPrice ? 0 : midPrice - delta;
    ask = midPrice + delta;
}
```

`confidence` is computed upstream as `spread * confidenceParam` where `spread` is the oracle's BPS value (0–9999) and `confidenceParam` is the factory-set multiplier (0–1,000,000). `CONFIDENCE_BASE = 1e10`.

When `midPrice * spread * confidenceParam < 1e10`, `delta = 0`, so `bid = ask = midPrice`.

These equal values then flow into `_applyBidAdjustments` (Floor rounding) and `_applyAskAdjustments` (Ceil rounding):

```solidity
bidOut = Math.mulDiv(midPrice, Q64 * stepBidFactor, STEP_DENOM, Floor);
askOut = Math.mulDiv(midPrice, Q64 * stepAskFactor, STEP_DENOM, Ceil);
```

With `marginStep = 0` (the default, `stepBidFactor = stepAskFactor = BPS_BASE_U = 1e18`):

```
bidOut = floor(midPrice × Q64 / 1e8)
askOut = ceil(midPrice  × Q64 / 1e8)
```

For any `midPrice` not exactly divisible by `1e8`, `bidOut < askOut` by exactly **1 Q64 unit** (≈ 5.4 × 10⁻²⁰ in price terms). The guard at line 228 (`if (bidOut >= askOut) return (0, type(uint128).max)`) does **not** trigger, and the pool executes with a spread of 1 Q64 unit — functionally zero.

The `AnchoredPriceProvider` has the same precision loss in `_shapedQuote` (line 363), but it is **not vulnerable** because its `_computeBidAsk` applies a band clamp (`Math.min(refBid, cBid)` / `Math.max(refAsk, cAsk)`) that restores the full reference spread regardless of the shaped-quote delta. `PriceProvider` and `ProtectedPriceProvider` have no such clamp. ### Impact Explanation

When `delta = 0` due to truncation, `bid ≈ ask ≈ mid`. The pool quotes at the oracle mid price with no spread margin:

- **Traders buying token0** pay `ask ≈ mid` instead of the intended `ask > mid` → they receive more value than the oracle curve permits.
- **Traders selling token0** receive `bid ≈ mid` instead of the intended `bid < mid` → the pool pays out more than it should.

LPs lose the spread income they are supposed to earn on every swap. Within the `MAX_TIME_DELTA` staleness window, a trader can exploit a slightly stale oracle price with zero spread buffer, extracting value from LP reserves. This is a direct loss of LP assets and a swap conservation failure.

---

### Likelihood Explanation

The truncation condition is `midPrice * spread * confidenceParam < 1e10`.

With a representative configuration of `confidenceParam = 10_000` (1× multiplier) and `spread = 1` (1 bps from the oracle), the threshold is `midPrice < 1e6`, i.e., any asset priced below **$0.01** in 8-decimal format. Tokens such as SHIB, PEPE, and many other ERC-20s routinely trade below this threshold. No malicious action is required — valid oracle data and a reasonable factory configuration are sufficient. The trigger is fully unprivileged. [5](#0-4) [6](#0-5) 

---

### Recommendation

Replace the truncating division with `Math.ceilDiv` in both `PriceProvider` and `ProtectedPriceProvider`, mirroring the fix recommended in the external report:

```solidity
// Before (truncates to 0 for small midPrice):
uint256 delta = midPrice * confidence / CONFIDENCE_BASE;

// After (rounds up, preserving a non-zero delta whenever confidence > 0):
uint256 delta = Math.ceilDiv(midPrice * confidence, CONFIDENCE_BASE);
```

This ensures that any non-zero `confidence` always produces a non-zero `delta`, preserving the intended spread separation between bid and ask even for low-priced assets. [1](#0-0) [2](#0-1)

### Proof of Concept

**Setup:**
- Asset: low-priced token, oracle 8-decimal mid price `mid = 1e5` ($0.001)
- Oracle spread: `spread = 1` (1 bps)
- Factory sets `confidenceParam = 10_000` (1× multiplier — reasonable)
- `marginStep = 0` (default)

**Execution trace:**

```
adjustedSpread = spread * confidenceParam = 1 * 10_000 = 10_000
delta = midPrice * confidence / CONFIDENCE_BASE
      = 1e5 * 10_000 / 1e10
      = 1e9 / 1e10
      = 0                          ← truncated to zero

bid = midPrice - 0 = 1e5
ask = midPrice + 0 = 1e5          ← bid == ask

bidOut = floor(1e5 * Q64 / 1e8)  = floor(Q64 / 1000) = 18_446_744_073_709_551
askOut = ceil (1e5 * Q64 / 1e8)  = ceil (Q64 / 1000) = 18_446_744_073_709_552

bidOut < askOut  →  guard does NOT trigger
```

**Result:** The pool executes with a spread of **1 Q64 unit** (≈ 5.4 × 10⁻²⁰ in price terms) instead of the intended 1 bps ($0.00001). Every swap against this pool during the staleness window extracts value from LPs with no spread buffer.

**Intended behavior:** `delta` should be `ceil(1e5 * 10_000 / 1e10) = 1`, giving `bid = 99_999`, `ask = 100_001`, and a proper 2-unit spread in 8-decimal space. [7](#0-6) [8](#0-7)
