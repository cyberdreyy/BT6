### Title
Unprivileged `register` Unconditionally Clears Admin Blacklist, Permanently Undermining Pool Oracle Access Control — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
`OracleBase.register` is a permissionless payable function that clears `blacklisted[pool]` whenever `msg.value >= registrationFee`. Because `registrationFee` defaults to 1 wei and `register` carries no role check, any caller can undo an admin blacklist decision for a trivial cost, restoring the pool's ability to call `price(feedId, pool)`.

### Finding Description
`setBlacklist` is protected by `onlyRole(ADMIN_ROLE)` and is the intended mechanism for the oracle admin to revoke a pool's oracle read access. However, `register` — which is `external payable` with no role restriction — contains the following unconditional clear:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;          // ← no role check
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
```

The only preconditions are payment of `registrationFee` (default 1 wei), an approved factory, and `isPool` membership. All three are satisfiable by any unprivileged caller for any pool that was legitimately deployed through an approved factory. The admin cannot permanently blacklist a specific pool without either removing the entire factory from `approvedFactories` (collateral damage to all other pools) or raising `registrationFee` globally (affects all future registrations).

### Impact Explanation
This is a direct admin-boundary break: an ADMIN-gated security control (`setBlacklist`) is bypassed by an unprivileged path (`register`). A pool the admin blacklisted — e.g., because it was being used to extract oracle prices for manipulation, or because a vulnerability was discovered in that pool — can immediately resume calling `price(feedId, pool)` and feeding those prices into `PriceVelocityGuardExtension.beforeSwap` or any other oracle-consuming swap path. The admin cannot win a re-blacklist race because the attacker can re-register in the same block for 1 wei.

### Likelihood Explanation
The attack requires only: (1) knowledge that a pool was blacklisted (observable on-chain via the `BlacklistUpdated` event), (2) access to 1 wei, and (3) calling `register` with the pool's existing approved factory. No special role, no privileged access, no complex setup. Any EOA can execute this.

### Recommendation
Remove the blacklist-clearing logic from `register`, or gate it behind `ADMIN_ROLE`. The registration path should only add `registeredPool[feedId][pool] = true`; clearing a blacklist should remain an explicit, admin-only action. If the intent is to allow "rehabilitation" via payment, it should require an explicit admin co-signature or a separate admin-gated `unblacklist` function, not a side-effect of permissionless registration.

### Proof of Concept
```solidity
// 1. Admin blacklists the pool
oracle.setBlacklist(pool, true);
assertEq(oracle.blacklisted(pool), true);

// 2. Unprivileged attacker re-registers for 1 wei
vm.prank(attacker);
oracle.register{value: 1 wei}(feedId, pool, approvedFactory);

// 3. Blacklist is cleared
assertEq(oracle.blacklisted(pool), false);

// 4. Pool can now read oracle prices again
vm.prank(provider);
(uint256 mid,,,) = oracle.price(feedId, pool); // succeeds
```

```text
    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
    }
```
