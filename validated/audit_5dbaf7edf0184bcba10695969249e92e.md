### Title
Permissionless `register()` Unconditionally Clears Admin Blacklist, Allowing Any Actor to Re-enable a Banned Pool's Oracle Access - (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

The `register()` function in `OracleBase.sol` is permissionless and unconditionally removes a pool from the `blacklisted` mapping as a side-effect of registration. Because the registration fee defaults to 1 wei and any caller may invoke `register()` for any valid pool, an admin's blacklisting decision can be trivially reversed by any third party — including the pool operator — restoring the pool's ability to consume oracle prices and execute swaps.

---

### Finding Description

`OracleBase.sol` implements an abuse-protection layer whose enforcement relies on two independent controls: (1) a `blacklisted` mapping that the `ADMIN_ROLE` sets via `setBlacklist()`, and (2) a `registeredPool` mapping that gates the `price()` read path. The `price()` function enforces both:

```solidity
require(!blacklisted[pool], Blacklisted(pool));
require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
``` [1](#0-0) 

However, the `register()` function — which is fully permissionless — contains the following logic:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
``` [2](#0-1) 

There is no `ADMIN_ROLE` check, no `notBlacklisted` guard, and no check that `msg.sender` is the pool or its authorized operator. The only prerequisites are:

- `msg.value >= registrationFee` (defaults to **1 wei**)
- `factory` is in `approvedFactories`
- `factory.isPool(pool)` returns `true` [3](#0-2) 

All three conditions are satisfiable by any external actor for any legitimately-deployed pool. The `registrationFee` is initialized to 1 wei: [4](#0-3) 

This creates a direct inconsistency: `setBlacklist()` is an `ADMIN_ROLE`-only operation meant to permanently bar a pool, but `register()` silently undoes it without any privileged authorization.

---

### Impact Explanation

Once a blacklisted pool is re-registered, it satisfies both checks in `price()` and can again call the oracle during swaps. This restores the pool's ability to read live oracle prices and execute swaps against those prices. The abuse-protection layer — the entire purpose of the `price(feedId, pool)` attributed read model — is rendered ineffective for any pool that an admin attempts to blacklist. If the blacklisting was triggered by detected price manipulation, regulatory action, or protocol-level abuse, the attacker can immediately restore access for the cost of 1 wei.

---

### Likelihood Explanation

The attack requires no special privilege, no signature, and no cooperation from the pool. Any EOA can call `register(feedId, blacklistedPool, approvedFactory)` with 1 wei. The pool must be recognized by an approved factory (`isPool(pool) == true`), which is true for all legitimately deployed pools — the exact pools the admin would want to blacklist. The default `registrationFee` of 1 wei makes the cost negligible. [4](#0-3) 

---

### Recommendation

Remove the automatic blacklist-clearing side-effect from `register()`. The blacklist and registration should be orthogonal controls: registration grants fee-based access, while blacklisting is an admin override that must not be reversible by an unprivileged caller. If clearing the blacklist on re-registration is intentional, it must require `ADMIN_ROLE` authorization, not a permissionless payment.

```solidity
// Remove these lines from register():
// if (blacklisted[pool]) {
//     blacklisted[pool] = false;
//     emit BlacklistUpdated(pool, false);
// }
```

Additionally, the `price()` function should check the blacklist on `msg.sender` (the `PriceProvider`) as well as the `pool`, since a blacklisted provider could route through a non-blacklisted pool.

---

### Proof of Concept

1. Admin calls `setBlacklist(poolA, true)` to bar `poolA` from oracle reads due to detected abuse.
2. Attacker (any EOA) calls `register(feedId, poolA, approvedFactory)` with `msg.value = 1 wei`.
3. `register()` executes `blacklisted[poolA] = false` unconditionally.
4. `poolA` now passes both checks in `price()`:
   - `!blacklisted[poolA]` → `true`
   - `registeredPool[feedId][poolA]` → `true`
5. `poolA` resumes consuming oracle prices and executing swaps, fully bypassing the admin's blacklist. [5](#0-4) [6](#0-5)

    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
```
