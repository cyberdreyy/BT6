### Title
Pusher consent signature is replayable before deadline expiry, allowing creator to silently re-establish revoked delegation and redirect feed writes — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature but stores no record of used signatures and performs no nonce check. A creator can replay the identical pusher signature any number of times before the deadline expires, re-establishing delegation immediately after the pusher calls `revokePusher()`. The pusher's self-revocation is silently nullified, their subsequent pushes are redirected back into the creator's namespace, and their own namespace stops receiving updates — causing staleness that propagates to every pool whose price provider reads from that namespace.

---

### Finding Description

`allowPushers` signs over `(block.chainid, address(this), deadline, pusher, msg.sender)` and the only replay guard is `_ensureDeadline(deadline)`, which only rejects calls made *after* the deadline has passed. [1](#0-0) 

When a pusher calls `revokePusher()`, the mapping is cleared to `address(0)`: [2](#0-1) 

But because no used-signature bitmap or nonce exists, the creator can immediately call `allowPushers` again with the **same** signature and the **same** deadline (still in the future), writing `namespaceRemapping[pusher] = msg.sender` again. The pusher has no on-chain mechanism to detect or prevent this before the deadline expires.

The `fallback` push path resolves the namespace at call time: [3](#0-2) 

So every push the pusher makes after their revocation — intending to write to their own namespace — is silently redirected to the creator's namespace for the entire remaining lifetime of the deadline.

The code comment itself acknowledges the partial nature of the protection: *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* — but the deadline only closes the window after expiry, not before. [4](#0-3) 

---

### Impact Explanation

Two concrete bad-price paths reach pools:

**Path 1 — Pusher's own namespace goes stale.**
After re-delegation, the pusher's writes land in the creator's namespace. The pusher's own namespace receives no further updates. Any `PriceProvider` or `PriceProviderL2` whose `offchainFeedId` points to the pusher's namespace will eventually fail the staleness check: [5](#0-4) 

returning `(0, type(uint128).max)`, causing `getBidAndAskPrice` to revert with `FeedStalled` and making swaps on every pool backed by that provider permanently unusable until the pusher can re-establish their own namespace.

**Path 2 — Creator's namespace receives wrong-asset data.**
If the pusher has revoked because they are now pushing data for a different asset pair (e.g., they moved to a new creator), the creator's namespace is overwritten with prices for the wrong asset. Pools whose `PriceProvider` reads from the creator's oracle then execute swaps against a stale or inverted bid/ask, satisfying the bad-price execution impact gate. [6](#0-5) 

---

### Likelihood Explanation

Deadlines in production deployments are typically set weeks or months in the future (the test fixture uses `block.timestamp + 1 days`). Any creator whose off-chain infrastructure does not explicitly handle pusher revocation events — or any creator who acts adversarially — can replay the signature for the entire window. The pusher receives no on-chain signal that their revocation was undone; the attack is silent.

---

### Recommendation

Track consumed signatures to make each consent one-time-use. The simplest fix is a `mapping(bytes32 => bool) private usedConsents` keyed on the signature hash, set to `true` on first use and checked before writing `namespaceRemapping`. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, so a revoked pusher can invalidate all outstanding signatures by incrementing their own nonce via a dedicated function.

---

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
