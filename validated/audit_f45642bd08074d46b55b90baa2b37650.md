### Title
`updateBySignature` Lacks Submission Deadline, Allowing Irrevocable Signed Price Updates to Feed Stale Prices into Pools — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.updateBySignature` accepts a creator-signed slot word and writes it to storage with no on-chain deadline. The signed message commits only to `(chainid, oracleAddress, feedCreator, newSlotValue)`. Anyone who holds a valid signed update can submit it at any future moment as long as the embedded price timestamp is strictly newer than the currently stored slot timestamp. There is no expiry that the creator can set to make a previously issued signature unsubmittable. This is the direct analog of the RFQ "free option" bug: a relay or any party that received the signed update off-chain can withhold it and submit it precisely when the embedded price is favorable relative to the current market, causing a pool to execute swaps at a stale bid/ask.

---

### Finding Description

`updateBySignature` in `CompressedOracleV1` has the following signature:

```solidity
function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
    external override returns (bool)
```

The signed digest is:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
);
```

No deadline field is included in the signed message and no deadline parameter is accepted by the function. The only temporal guard is:

1. `timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT)` — rejects timestamps that are in the future beyond the drift window.
2. `timestampMs.isAfter(oldTimestampMs)` — rejects updates whose embedded timestamp is not strictly newer than the stored slot timestamp.

Guard (2) is the monotonicity check. It prevents replay of an update **after** a newer one has been stored, but it does **not** prevent submission of a signed update that was created before the current stored timestamp was pushed, as long as the signed update's timestamp is still newer than whatever is currently stored.

This is in direct conflict with the top-level specification. `smart-contracts-poc/docs/en/oracle-packet-structure.md` documents the intended interface as:

```
updateBySignature(feedCreator, deadline, newSlotValue, signature)
```

with the signed message including `deadline` and the guard:

> `deadline` must be in the future (`DeadlineExceeded` otherwise).

The contract-registry ABI (`smart-contracts-poc/contract-registry/versions/registry.json`) likewise lists `deadline` as the second parameter of `updateBySignature`. The actual deployed contract omits it entirely.

The `allowPushers` function in the same contract explicitly explains why a deadline is necessary for signed consent:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The same logic applies to `updateBySignature`: the embedded price timestamp T is the **price's own freshness marker**, not a submission expiry. Without a separate deadline, a signed update with price timestamp T can be submitted at any wall-clock time, as long as T is still newer than the stored slot timestamp.

---

### Impact Explanation

**Attack path (bad-price execution in a pool):**

1. Creator signs slot word A encoding price P at price-timestamp T and hands it to a relay for submission.
2. The market moves; the true price becomes P′ (e.g., P′ > P, so P is now a favorable buy price for a trader).
3. The relay (or any party that intercepted the signed update) withholds slot word A.
4. The creator pushes a newer update B (price P′, timestamp T+1) via `fallback()`. Stored timestamp is now T+1.
5. The relay cannot submit A now (T < T+1, monotonicity rejects it). **But** if the relay acts before step 4, or if the creator has not yet pushed a newer update:
   - Relay submits A at wall-clock time `now`, where `now − T/1000 < MAX_TIME_DELTA`.
   - `CompressedOracleV1.price()` returns `(P, spread, spread1, T/1000)`.
   - `PriceProvider._isStale(T/1000, block.timestamp, MAX_TIME_DELTA)` passes because the age is within the allowed window.
   - `PriceProvider.getBidAndAskPrice()` returns bid/ask derived from stale price P.
   - `MetricOmmPool.swap()` executes at the stale price, giving the trader a better rate than the current market.
   - LPs and the pool absorb the loss.

The attack window is bounded by `MAX_TIME_DELTA` (up to 7 days per the constructor check), which is the price provider's staleness tolerance. Within that window, the signed update is both valid (not stale) and irrevocable (no deadline to expire it).

---

### Likelihood Explanation

The `updateBySignature` path is explicitly designed for relay services: the creator signs off-chain and a third party submits on-chain. Any relay that receives signed updates has the material to execute this attack. The creator's only mitigation is to push a newer update via `fallback()` before the relay submits the old one — a race condition that the relay controls. The attack is permissionless (no special role required beyond possessing the signed update), and the signed update is valid for up to `MAX_TIME_DELTA` seconds after the price timestamp.

---

### Recommendation

Add a `deadline` parameter to `updateBySignature` and include it in the signed digest, matching the specification in `smart-contracts-poc/docs/en/oracle-packet-structure.md` and the registry ABI:

```solidity
function updateBySignature(
    address feedCreator,
    uint256 deadline,
    uint256 newSlotValue,
    bytes calldata signature
) external override returns (bool) {
    _ensureDeadline(deadline);   // revert if block.timestamp > deadline
    // ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), feedCreator, deadline, newSlotValue))
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    // ...
}
```

This matches the pattern already used in `allowPushers` and aligns the contract with its own top-level documentation and registry ABI. Creators can then set short deadlines (e.g., 60 seconds) so that any withheld signed update expires before it can be used to inject a stale price.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {Test} from "forge-std/Test.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {IOffchainOracle} from "contracts/interfaces/IOffchainOracle.sol";
import {U64x32} from "contracts/oracles/utils/U64x32.sol";
import {TimeMs} from "contracts/oracles/utils/TimeMs.sol";

contract StaleSignedUpdatePoC is Test {
    CompressedOracleV1 oracle;
    uint256 constant CREATOR_KEY = 0xC0FFEE;
    address creator;

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 60); // 60s drift
        creator = vm.addr(CREATOR_KEY);
        vm.warp(1_700_000_000);
    }

    function testIrrevocableSignedUpdateFeedsStalePrice() public {
        // 1. Creator signs update A at price P = 1_000_000, timestamp T = now
        uint56 T = uint56(block.timestamp * 1000);
        uint48 rawA = (uint48(1_000_000) << 16) | (uint48(3) << 8) | uint48(3);
        uint256 slotValueA = (uint256(T) << 8) | uint256(11); // slotId=11
        slotValueA |= uint256(rawA) << 208;

        bytes32 digestA = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), creator, slotValueA))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(CREATOR_KEY, digestA);
        bytes memory sigA = abi.encodePacked(r, s, v);

        // 2. Price moves; creator pushes update B (price P' = 2_000_000, timestamp T+1)
        //    via fallback — but relay withholds A and submits it BEFORE B lands.
        //    Simulate: relay submits A first (stored timestamp = 0 < T).
        oracle.updateBySignature(creator, slotValueA, sigA);

        // 3. Advance time by MAX_TIME_DELTA - 1 (still within staleness window)
        vm.warp(block.timestamp + 3599); // assume MAX_TIME_DELTA = 3600 at provider

        // 4. Price provider reads the oracle — refTime = T/1000 = original timestamp
        IOffchainOracle.OracleData memory data =
            oracle.getOracleData(oracle.feedIdOf(creator, 11, 0));

        // Price P is stored, timestamp is T (old), but within MAX_TIME_DELTA of now
        assertEq(data.price, U64x32.decode(uint32(rawA >> 16)));
        // refTime returned by price() = T/1000, age = 3599 < MAX_TIME_DELTA → accepted as fresh
        // Pool executes swap at stale price P instead of current P'
    }
}
```

```text
    /// @notice Single-slot update authorized by the creator's signature. The signed slot
    ///         word carries its own 56-bit timestamp, so replay is neutralized by the
    ///         monotonicity check below — no deadline is needed.
    function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
        external
        override
        returns (bool)
    {
        require(feedCreator != address(0), InvalidNamespace());

        uint256 namespace;
        assembly ("memory-safe") {
            namespace := shl(96, feedCreator) // [creator:20][zeros:12]
        }

        uint8 slotId = uint8(newSlotValue); // LSB
        TimeMs timestampMs = toTimeMs(newSlotValue >> 8 & X56);
        timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
        bytes32 key = bytes32(namespace | uint256(slotId));
        uint256 old = uint256(_loadStorage(key));
        TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

        bool newer = timestampMs.isAfter(oldTimestampMs);
        if (!newer) {
            return false;
        }

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
        );
        require(feedCreator == ECDSA.recover(hash, signature));

        _writeStorage(key, bytes32(newSlotValue & ~uint256(0xff)));

        return true;
    }
```

## Required Guards

- `deadline` must be in the future (`DeadlineExceeded` otherwise).
- `timestamp` must not be in the future (`FutureTimestamp` otherwise).
- `timestamp` must be strictly increasing per slot (older updates are ignored).
- For the creator’s *current* (partially filled) slot, uncreated lanes must be zero (`UninializedPush` otherwise).
