### Title
Future-Dated Timestamp Injection via `MAX_TIME_DRIFT` Drift Window Allows Rogue Pusher to Lock Manipulated Price as Fresh in CompressedOracle — (`smart-contracts-poc/contracts/oracles/utils/TimeMs.sol`, `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `revertIfAfterBlockTimeWithDrift` guard in `CompressedOracle` permits pushed timestamps up to `block.timestamp + MAX_TIME_DRIFT` seconds in the **future**. A delegated pusher can exploit this drift window to: (1) store a manipulated price with a future-dated timestamp, (2) freeze out all competing pushers via the monotonicity check for the entire drift window, and (3) have that manipulated price accepted as **fresh** by any downstream `PriceProvider` exactly when the future timestamp becomes current — enabling bad-price execution against pool swaps.

---

### Finding Description

**Drift window in `TimeMs.revertIfAfterBlockTimeWithDrift`** [1](#0-0) 

```solidity
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}
```

This check is the **only** upper-bound guard on pushed timestamps. It explicitly permits any timestamp up to `block.timestamp + MAX_TIME_DRIFT` seconds in the future.

**Both push paths invoke this guard with the full drift allowance:**

`fallback()` (bulk push path): [2](#0-1) 

`updateBySignature()`: [3](#0-2) 

**Monotonicity check freezes the slot for the entire drift window:** [4](#0-3) 

```solidity
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;
```

Once a future-dated timestamp `T_push + MAX_TIME_DRIFT` is stored, every subsequent push with a current timestamp `T_now < T_push + MAX_TIME_DRIFT` fails this check and is silently skipped. No other pusher — including the creator — can overwrite the slot until `MAX_TIME_DRIFT` seconds have elapsed, because their current-time timestamps are all numerically smaller.

**The manipulated price becomes "fresh" exactly at `T_push + MAX_TIME_DRIFT`:**

The `PriceProvider` staleness check: [5](#0-4) 

```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;          // future → stale
    return (nowTs - refTime) > maxDelta;
}
```

The oracle returns `refTime = timestampMs.toSeconds()`: [6](#0-5) 

At time `T_push + MAX_TIME_DRIFT`, `block.timestamp == refTime`, so:
- `refTime > nowTs` → **false** (not future)
- `(nowTs - refTime) > maxDelta` → `0 > maxDelta` → **false** (not stale)

The manipulated price passes all staleness gates and is delivered to the pool as a fresh quote.

**Attack path summary:**

| Step | Time | Action |
|------|------|--------|
| 1 | T | Rogue pusher submits `timestampMs = (T + MAX_TIME_DRIFT) × 1000` with manipulated price P |
| 2 | T → T+drift | Monotonicity blocks all competing pushes; price reads return stale (DoS) |
| 3 | T+drift | `refTime == block.timestamp`; staleness check passes; P delivered to pool |
| 4 | T+drift | Attacker executes swap at manipulated price P |

---

### Impact Explanation

A rogue delegated pusher can feed an arbitrary price (within `priceGuard` bounds) to any pool whose `PriceProvider` or `AnchoredPriceProvider` is backed by the `CompressedOracle`. At the moment the future timestamp becomes current, the pool executes swaps at the manipulated bid/ask, causing direct loss of user principal or LP assets. The `AnchoredPriceProvider` band clamp and `priceGuard` limit the magnitude but do not prevent the attack — any price within those bounds can be injected.

---

### Likelihood Explanation

Requires a delegated pusher (semi-trusted actor) to go rogue. Pusher delegation is a normal operational feature (`allowPushers`, `allowContractPushers`). The attack requires no special on-chain conditions, no flash loans, and no privileged admin access — only the ability to call `fallback()` with crafted calldata. The timing is deterministic and MEV-executable.

---

### Recommendation

Remove the drift allowance from the upper-bound check. Pushed timestamps must not exceed the current block time:

```solidity
// Before (TimeMs.sol):
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}

// After:
function revertIfAfterBlockTime(TimeMs t0) view {
    require(t0.toSeconds() <= block.timestamp, FutureTimestamp());
}
```

Update both call sites in `CompressedOracle.fallback()` and `updateBySignature()` accordingly. If a small clock-skew tolerance is genuinely needed for network jitter, cap it at a few seconds (e.g., 15 s), not at `MAX_TIME_DRIFT` which can be set to minutes.

---

### Proof of Concept

```solidity
// Assume MAX_TIME_DRIFT = 30 minutes (1800 s), set at construction.
// Attacker is a delegated pusher for creator Alice.

// Step 1: At block T, attacker crafts a slot word with:
//   timestampMs = (T + 1800) * 1000   (30 min in the future)
//   price       = manipulated_price    (e.g., 50% of true price)
//   slotId      = 0

uint256 manipulatedPrice = ...; // packed U64x32 encoding of 50% true price
uint256 futureTs = (block.timestamp + 1800) * 1000; // 56-bit ms timestamp
uint8 slotId = 0;

// Pack: [data0:6][data1:6][data2:6][data3:6][ts:7][slotId:1]
uint256 word = (manipulatedPrice << 64) | (futureTs << 8) | slotId;

// Step 2: Push via fallback — passes revertIfAfterBlockTimeWithDrift(1800)
(bool ok,) = address(compressedOracle).call(abi.encodePacked(word));
require(ok);

// Step 3: For the next 30 minutes, all legitimate pushes are blocked by monotonicity.
// Price reads return stale (refTime > block.timestamp) → pool swaps revert.

// Step 4: At block T + 1800, refTime == block.timestamp → staleness check passes.
// Attacker executes a swap through the pool at the manipulated price.
pool.swap(...); // executes at 50% of true price → LP suffers loss
```
