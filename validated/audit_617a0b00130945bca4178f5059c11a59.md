### Title
EIP-191 Pusher-Consent Signature Replay Reinstates Revoked Delegation Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()`, the creator can replay the original on-chain signature (which is permanently visible in calldata) to silently reinstate the delegation before the deadline expires. The pusher's pushes then land in the creator's namespace again, feeding potentially wrong prices into any pool backed by that namespace.

---

### Finding Description

`allowPushers` builds a hash over `(chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's signature, then unconditionally writes `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

The only replay guard is `_ensureDeadline`, which checks `block.timestamp <= deadline`. [2](#0-1) 

There is no `usedSignature` mapping, nonce, or any other per-signature invalidation. A grep across all production contracts in `smart-contracts-poc/contracts/` confirms zero occurrences of any such tracking.

When a pusher calls `revokePusher()`, the mapping is cleared to `address(0)`: [3](#0-2) 

But the original consent signature — already broadcast in the first `allowPushers` transaction's calldata — remains cryptographically valid for any `block.timestamp <= deadline`. The creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, pass every check, and restore `namespaceRemapping[pusher] = creator`.

The code comment itself acknowledges the deadline is the only guard against post-revocation replay: [4](#0-3) 

But the deadline only bounds the outer time window; it does not prevent the signature from being replayed multiple times within that window.

---

### Impact Explanation

After the creator replays the signature, the pusher's `fallback()` pushes are silently redirected into the creator's namespace: [5](#0-4) 

If the pusher, believing they have revoked, begins pushing data for a different feed (e.g., their own namespace for a different asset pair), those slot words overwrite the creator's namespace slots. Any pool whose `PriceProvider` reads `feedIdOf(creator, slotIndex, positionIndex)` now receives the wrong price. This is a direct bad-price execution path: the pool's bid/ask is set from corrupted oracle data, causing traders to receive more output than the curve permits or the pool to receive less input than owed.

---

### Likelihood Explanation

- The consent signature is permanently visible in the first `allowPushers` transaction's calldata; no secret extraction is needed.
- The creator is the only party who can replay it (the hash commits to `msg.sender = creator`), so the trigger requires a malicious or opportunistic creator.
- The pusher must continue pushing data after revoking (e.g., redirecting to their own namespace for a different asset). This is the normal post-revocation workflow.
- The deadline window is creator-chosen and can be set to the maximum the protocol allows, giving a wide replay window.

Likelihood: **Medium** — requires a malicious creator and a pusher who continues pushing after revoking, both of which are realistic in a permissionless oracle system.

---

### Recommendation

Track each consent signature as consumed on first use. The minimal fix is a `mapping(bytes32 => bool) private _usedConsentSig` keyed on the full digest, set to `true` after the first successful `allowPushers` call for that `(deadline, pusher, creator)` tuple, and checked before writing `namespaceRemapping`. Alternatively, include a per-pusher nonce in the signed payload and increment it on every successful delegation or revocation, making every new consent signature structurally distinct from any prior one.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator, deadline = now + 1 days
bytes memory sig = sign(PUSHER_KEY,
    keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator)));

vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME sig before deadline — succeeds, no revert
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
// Revocation is nullified:
assertEq(oracle.namespaceRemapping(pusher), creator);

// 5. Pusher pushes BTC/USD data thinking it goes to their own namespace;
//    it lands in creator's ETH/USD slot → bad price reaches pools.
vm.prank(pusher);
(bool ok,) = address(oracle).call(encodeBtcSlotWord(slotId, btcPrice, tsMs));
assertTrue(ok);
// creator's ETH/USD feed now contains BTC/USD price
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, slotId, 0)).price, btcPrice);
```

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
```
