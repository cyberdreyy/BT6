### Title
`purgeStateGuardRole()` Does Not Clear `pendingStateGuard`, Allowing Revoked Guard to Reclaim Price-Bound Control — (`File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary

`purgeStateGuardRole()` clears `stateGuard[feedId]` but leaves `pendingStateGuard[feedId]` intact. A previously nominated pending guard can call `acceptStateGuardRole()` after the purge to silently reclaim the `stateGuard` role, locking ADMIN out and gaining unilateral control over `setPriceGuard()` — the on-chain price-bound that gates every bid/ask quote delivered to pools.

### Finding Description

`OracleBase` (providers) implements a two-step guard-transfer pattern:

1. `setStateGuardRole(feedId, newGuard)` — writes `pendingStateGuard[feedId]`
2. `acceptStateGuardRole(feedId)` — promotes `pendingStateGuard` → `stateGuard` and clears the pending slot

The renounce path is:

```solidity
// smart-contracts-poc/contracts/oracles/providers/OracleBase.sol L120-124
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];          // ← clears active guard
    // pendingStateGuard[feedId] is NOT cleared
    emit StateGuardDeleted(feedId);
}
```

After `purgeStateGuardRole` executes, `stateGuard[feedId] == address(0)`, so `checkRole` falls back to `ADMIN_ROLE`. However, `pendingStateGuard[feedId]` still holds whatever address was written by the last `setStateGuardRole` call. That address can immediately call `acceptStateGuardRole(feedId)` — which only checks `pendingStateGuard[feedId] == msg.sender` — and become the new `stateGuard`, locking ADMIN out again.

Two concrete trigger paths exist:

**Path A — Malicious guard plants a backdoor before renouncing:**
1. Guard calls `setStateGuardRole(feedId, attackerAddress)` → `pendingStateGuard[feedId] = attacker`
2. Guard calls `purgeStateGuardRole(feedId)` → `stateGuard[feedId] = 0`, pending untouched
3. ADMIN observes `stateGuard == 0` and believes control is restored
4. Attacker calls `acceptStateGuardRole(feedId)` → `stateGuard[feedId] = attacker`

**Path B — ADMIN cancels a pending transfer by purging the active guard:**
1. ADMIN calls `setStateGuardRole(feedId, newGuard)` to initiate a transfer
2. Before `newGuard` accepts, ADMIN decides to abort and calls `purgeStateGuardRole(feedId)` (intending full removal)
3. `purgeStateGuardRole` clears `stateGuard` but not `pendingStateGuard`
4. `newGuard` can still call `acceptStateGuardRole` and take over

In both cases the attacker ends up as the sole authority over `setPriceGuard(feedId, minPrice, maxPrice)`.

### Impact Explanation

`setPriceGuard` is the on-chain circuit-breaker that `PriceProvider.getBidAndAskPrice()` enforces before returning bid/ask quotes to pools. A rogue `stateGuard` can:

- **Disable bounds entirely** by calling `setPriceGuard(feedId, 0, 0)` — the provider treats `guardMax == 0` as "unlimited", so any price (stale, manipulated, extreme) passes through to pool swaps.
- **Brick the feed** by setting bounds that exclude the real market price, causing `FeedStalled` on every swap and making the pool unusable.

Both outcomes satisfy the allowed impact gate: bad-price execution reaching pools, and broken core pool swap functionality causing loss of funds or unusable withdraw/swap flows.

### Likelihood Explanation

- Path A requires a semi-trusted guard to act maliciously — plausible given the guard role can be granted to external parties.
- Path B requires only an ADMIN mistake in the transfer-cancel flow, which is a normal operational scenario with no malicious intent required.
- `acceptStateGuardRole` is permissionless and can be called at any time after the purge; there is no time window that closes the vulnerability.

### Recommendation

`purgeStateGuardRole` must also clear `pendingStateGuard`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];
    delete pendingStateGuard[feedId];   // ← add this
    emit StateGuardDeleted(feedId);
}
```

Symmetrically, `purgePendingStateGuardRole` already exists for clearing only the pending slot; the active-guard purge must clear both.

### Proof of Concept

```
1. Guard is active: stateGuard[feedId] = guard
2. guard calls setStateGuardRole(feedId, attacker)
   → pendingStateGuard[feedId] = attacker
3. guard calls purgeStateGuardRole(feedId)
   → stateGuard[feedId] = 0
   → pendingStateGuard[feedId] = attacker  ← NOT cleared
4. ADMIN observes stateGuard[feedId] == 0, believes guard is gone
5. attacker calls acceptStateGuardRole(feedId)
   → require(pendingStateGuard[feedId] == msg.sender) passes
   → stateGuard[feedId] = attacker
6. attacker calls setPriceGuard(feedId, 0, 0)
   → price bounds disabled; any oracle price passes to pools
7. Pool swaps execute against unbounded/stale prices
```
