### Title
`revokePusher()` is permanently defeatable within the deadline window via nonce-less consent-signature replay — pusher's own namespace feeds go stale, feeding bad prices to pools - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowPushers` accepts an EIP-191 consent signature whose domain is `(chainid, oracle, deadline, pusher, creator)` with **no nonce and no revocation flag**. After a pusher calls `revokePusher()`, the creator can immediately replay the identical signature — as many times as needed — to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is therefore completely ineffective for the entire remaining lifetime of the deadline. Every push the pusher makes continues to land in the creator's namespace, leaving the pusher's own namespace feeds at price 0 / timestamp 0, which every downstream `PriceProvider` and pool rejects as `FeedStalled`.

---

### Finding Description

`CompressedOracleV1.allowPushers` constructs the consent hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no per-pusher revocation counter, and no "already-revoked" flag checked before the signature is accepted. The only replay guard is `_ensureDeadline(deadline)`, which only rejects calls made *after* the deadline — it does nothing to prevent re-use of the same signature *before* the deadline. [2](#0-1) 

`revokePusher()` simply writes `namespaceRemapping[msg.sender] = address(0)`: [3](#0-2) 

Because `allowPushers` performs no check against a revocation state, the creator can call it again with the same `(deadline, [pusher], [sig])` arguments in the very next block, restoring `namespaceRemapping[pusher] = creator`. The cycle repeats indefinitely until `block.timestamp > deadline`.

The code comment at lines 186–191 explicitly acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and claims the deadline is the mitigation. But the deadline only *bounds* the replay window — it does not *close* it. [4](#0-3) 

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after a replayed delegation still lands in the creator's namespace, and `feedIdOf(pusher, slotIndex, positionIndex)` remains at the zero-initialized state (price = 0, timestampMs = 0). [6](#0-5) 

---

### Impact Explanation

A pusher operating their own namespace to serve their own pool (e.g., a data-provider bot that pushes to `feedIdOf(pusher, slot, pos)`) cannot redirect their pushes away from a creator who holds a still-valid consent signature. Every push the bot makes is silently swallowed into the creator's namespace. The pusher's own feeds remain at timestamp 0, which `getOracleData` returns as `timestampMs = 0`. Any `PriceProvider` reading those feeds will see a stale timestamp and revert with `FeedStalled`, making every swap on the pool that depends on those feeds permanently unusable until the deadline expires.

This satisfies:
- **Bad-price execution**: stale (timestamp 0) bid/ask quote reaches a pool swap.
- **Broken core pool functionality**: swap/liquidity flows revert on `FeedStalled` for the entire deadline window.

---

### Likelihood Explanation

The creator needs only to hold a consent signature that has not yet expired — a normal operational state for any active delegation. The creator monitors the mempool for `revokePusher` transactions and front-runs each one with a replay of `allowPushers`. This is a permissionless, zero-cost action. Deadlines are typically set days to weeks in the future (the test suite uses `block.timestamp + 1 days`), giving the creator a large window to sustain the attack. [7](#0-6) 

---

### Recommendation

Include a per-pusher revocation nonce in the consent signature domain. Increment it on every `revokePusher()` or `removePushers()` call so that any previously issued signature is immediately invalidated:

```solidity
mapping(address => uint256) public pusherNonce;

// In revokePusher() / removePushers():
pusherNonce[pusher]++;

// In allowPushers() hash:
keccak256(abi.encode(
    block.chainid, address(this), deadline,
    pusher, msg.sender, pusherNonce[pusher]
))
```

After revocation the nonce advances, the old signature no longer recovers to `pusher`, and replay is impossible regardless of the deadline.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
