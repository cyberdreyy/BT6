### Title
Signature Replay in `allowPushers` Permanently Defeats Pusher Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` includes a `deadline` in the signed hash but has no nonce and no used-signature tracking. A creator who holds a pusher's signature with a far-future (or `type(uint256).max`) deadline can replay it an unlimited number of times to re-establish delegation immediately after the pusher calls `revokePusher()`. The pusher's revocation is permanently defeatable, and their price pushes continue to flow into the creator's namespace against their will — a silent feed hijack.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only replay barrier is `_ensureDeadline(deadline)`, which checks `block.timestamp <= deadline`. [2](#0-1) 

There is no nonce, no per-signature consumed flag, and no mapping that records which `(pusher, creator, deadline)` tuples have already been used. The code comment explicitly acknowledges the replay risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but the mitigation it describes — requiring a deadline — does not actually prevent replay; it only bounds the replay window to the deadline's expiry. [3](#0-2) 

`revokePusher()` sets `namespaceRemapping[pusher] = address(0)`: [4](#0-3) 

But `allowPushers` unconditionally overwrites that mapping back to `msg.sender` whenever the signature verifies and the deadline has not expired. Because the same `(chainid, oracle, deadline, pusher, creator)` tuple produces the same hash every time, the original signature is valid for every replay until `block.timestamp > deadline`.

---

### Impact Explanation

After a pusher revokes, the creator replays the original `allowPushers` call. `namespaceRemapping[pusher]` is set back to the creator's address. Every subsequent `fallback()` push from the pusher is silently routed into the creator's slot namespace instead of the pusher's own namespace. The pusher believes they have exited the delegation; the oracle state says otherwise. Any pool whose `PriceProvider` reads a feed in the creator's namespace continues to receive the pusher's data, and the pusher has no reliable way to stop it. If the pusher signed with `deadline = type(uint256).max` (the direct analog of the Across `_deposit` bug), the hijack is permanent for the lifetime of the contract.

---

### Likelihood Explanation

The pusher must have previously signed a delegation with a deadline that has not yet expired. This is the normal operating condition — pushers are expected to sign delegations with reasonable or long-lived deadlines. A creator who wants to retain a pusher's data feed after the pusher exits has a clear, low-cost path: simply replay the original `allowPushers` calldata. No privileged access, no special setup, and no on-chain cost beyond gas.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on each successful `allowPushers` call, or record each consumed signature hash in a `mapping(bytes32 => bool) usedSignatures` and revert on reuse:

```solidity
// Option A: nonce
mapping(address => uint256) public pusherNonce;

bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);

// Option B: consumed-signature bitmap
mapping(bytes32 => bool) private _usedSigs;
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedSigs[sigHash]);
_usedSigs[sigHash] = true;
```

Either approach ensures that once a pusher revokes, the original signature cannot re-establish the delegation.

---

### Proof of Concept

```
1. Pusher P signs:
     digest = keccak256(abi.encode(chainid, oracle, type(uint256).max, P, Creator))
   and hands the signature to Creator.

2. Creator calls allowPushers(type(uint256).max, [P], [sig]).
   → namespaceRemapping[P] = Creator.  ✓ delegation active.

3. P calls revokePusher().
   → namespaceRemapping[P] = address(0).  ✓ P believes delegation is gone.

4. Creator calls allowPushers(type(uint256).max, [P], [sig])  ← same calldata.
   _ensureDeadline: block.timestamp <= type(uint256).max  → passes.
   ECDSA.recover(hash, sig) == P                          → passes.
   → namespaceRemapping[P] = Creator.  ✗ delegation silently restored.

5. P's next fallback() push is routed to Creator's namespace.
   P can call revokePusher() again; Creator replays again. Loop is infinite.
``` [5](#0-4) [4](#0-3) [6](#0-5)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
