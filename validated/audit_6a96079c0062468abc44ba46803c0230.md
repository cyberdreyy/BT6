### Title
Missing Read-Side Staleness Check Allows Arbitrarily Old Prices to Reach Pool Swaps — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `CompressedOracleV1.price()` function returns stored oracle data to pools without checking whether the stored timestamp is within an acceptable staleness window. The `MAX_TIME_DRIFT` guard is enforced only on the **write path** (preventing future timestamps), but there is no symmetric check on the **read path** to reject prices that are too old. If a pusher goes offline or is revoked, the last stored price remains permanently valid for pool swaps, enabling bad-price execution at arbitrarily stale quotes.

---

### Finding Description

The external bug's invariant is: *"the remaining lock period must be longer than the epoch time."* The voting system checked initial lock duration on entry but never checked remaining validity at the moment of use. The direct analog here is: *"the remaining freshness of a stored oracle price must be within the acceptable drift window."* The oracle checks timestamp validity on write but never on read.

**Write path** — both `fallback()` and `updateBySignature()` call `revertIfAfterBlockTimeWithDrift`, which enforces:

```
t0.toSeconds() <= block.timestamp + MAX_TIME_DRIFT
```

This prevents a pusher from storing a timestamp that is too far in the **future**. [1](#0-0) [2](#0-1) 

**Read path** — `price()` calls `_price()` which calls `getOracleData()`, which loads the slot and returns the stored `timestampMs` with **no check** that `block.timestamp - storedTimestamp` is within any bound:

```solidity
function _price(bytes32 feedId)
    internal view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = getOracleData(feedId);
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
``` [3](#0-2) 

The external `price()` entry point is permissionless (the `pool` argument is explicitly ignored):

```solidity
function price(bytes32 feedId, address /* pool */)
    external view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    return _price(feedId);
}
``` [4](#0-3) 

The same gap exists in the providers `OracleBase._readPrice()`:

```solidity
function _readPrice(bytes32 feedId) internal view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = _oracleDataRaw(feedId);
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
``` [5](#0-4) 

`TimeMs` provides no utility for checking past-staleness — only `revertIfAfterBlockTimeWithDrift` (future guard) and `isAfter` (monotonicity) exist: [6](#0-5) 

There is no `revertIfBeforeBlockTimeWithDrift` or equivalent. The `MAX_TIME_DRIFT` immutable is never consulted during a read. [7](#0-6) 

---

### Impact Explanation

A pool executing a swap calls `price(feedId, pool)` on `CompressedOracleV1`. If the pusher for that feed has gone offline (network failure, key rotation, revocation via `revokePusher`), the oracle returns the last stored price — which may be hours or days old — without any revert or staleness signal. A trader who knows the true market price can exploit the stale bid/ask to extract value from LPs: buying at a stale low ask or selling at a stale high bid. This is a direct **bad-price execution** impact: the pool receives less input than the current market rate permits, or pays out more output than it should, causing LP principal loss.

---

### Likelihood Explanation

The trigger requires only that a pusher stops updating a feed — a realistic operational event (key compromise, infrastructure failure, intentional revocation). No privileged action by the attacker is needed; any user can call `swap()` on the pool. The `revokePusher()` self-revocation path is unprivileged and callable by the pusher itself, immediately creating a stale feed with no on-chain staleness guard to protect pools. [8](#0-7) 

---

### Recommendation

Add a symmetric staleness check on the read path. In `CompressedOracleV1._price()` and `OracleBase._readPrice()`, after loading `timestampMs`, enforce:

```solidity
require(
    block.timestamp <= data.timestampMs.toSeconds() + MAX_TIME_DRIFT,
    StalePrice(feedId)
);
```

Add a corresponding `revertIfStale(TimeMs t, uint256 maxAgeSeconds)` helper to `TimeMs.sol` to mirror the existing `revertIfAfterBlockTimeWithDrift`.

---

### Proof of Concept

1. Pusher `P` is authorized for creator `C`'s namespace via `allowPushers`.
2. `P` pushes a valid slot at `block.timestamp = T`, storing `timestampMs = T * 1000`.
3. `P` calls `revokePusher()` (unprivileged) or simply goes offline.
4. At `block.timestamp = T + 86400` (24 hours later), a pool calls `price(feedId, pool)`.
5. `CompressedOracleV1.price()` → `_price()` → `getOracleData()` returns the 24-hour-old price with no revert.
6. The pool executes a swap at the stale price. A trader who knows the true price profits at LP expense.

The stale-marker sentinel (`s0 == 0xFF && s1 == 0xFF`) only applies when a pusher explicitly encodes it; a pusher that simply stops pushing leaves the last valid price in storage indefinitely with no automatic staleness signal. [9](#0-8)
