### Title
`CompressedOracle` `fallback()` / `updateBySignature()` compare pusher's real-world timestamp against L2 `block.timestamp`, blocking valid price updates and enabling stale prices to reach pool swaps when L2 `block.timestamp` lags — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.fallback()` and `updateBySignature()` both call `timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT)`, which enforces `pusherTimestamp_seconds ≤ block.timestamp + MAX_TIME_DRIFT`. On L2 chains where `block.timestamp` is not linked to real-world time and can lag significantly (Arbitrum during sequencer downtime, Blast at launch), a pusher using wall-clock time will have every update rejected with `FutureTimestamp`. When `PriceProviderL2.FUTURE_TOLERANCE > CompressedOracle.MAX_TIME_DRIFT`, the previously stored (now stale) price still passes `_isStale()`, and the pool executes swaps at that stale bid/ask.

---

### Finding Description

**Push-time guard (the broken check):**

`TimeMs.revertIfAfterBlockTimeWithDrift` is:

```solidity
// smart-contracts-poc/contracts/oracles/utils/TimeMs.sol
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}
```

Both push paths call it unconditionally:

```solidity
// CompressedOracle.sol – fallback(), line 335
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);

// CompressedOracle.sol – updateBySignature(), line 285
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
```

The pusher is an off-chain bot that stamps each slot word with wall-clock time (milliseconds). On an L2 where `block.timestamp` lags real-world time by Δ seconds, the arithmetic becomes:

```
wall_clock_seconds  ≤  (wall_clock_seconds − Δ) + MAX_TIME_DRIFT
```

which simplifies to `Δ ≤ MAX_TIME_DRIFT`. The deployed value is `MAX_TIME_DRIFT = 60 s` (networks.json). Any lag exceeding 60 s causes every push to revert.

**Staleness check in the price provider (the second half of the bug):**

`PriceProviderL2._isStale()` uses a separate, independently configured `FUTURE_TOLERANCE` (capped at 1 hour):

```solidity
// PriceProviderL2.sol, lines 143-149
if (refTime > nowTs) {
    // refTime in the future: tolerate only within futureTol
    return (refTime - nowTs) > futureTol;
}
```

`refTime` is the oracle's stored timestamp (seconds), returned by `CompressedOracle.price()` as `data.timestampMs.toSeconds()`. If the stored timestamp is T₁ (real-world time) and L2's `block.timestamp` is T_l2 = T₁ − Δ, then `_isStale` returns `false` whenever `Δ ≤ FUTURE_TOLERANCE`.

**The combined window:**

| Condition | Effect |
|---|---|
| `Δ > MAX_TIME_DRIFT` | New pushes rejected → oracle cannot be updated |
| `Δ ≤ FUTURE_TOLERANCE` | Old stored price passes staleness → pool accepts it |
| `MAX_TIME_DRIFT < Δ ≤ FUTURE_TOLERANCE` | **Both hold simultaneously** → stale price reaches swap |

With `MAX_TIME_DRIFT = 60 s` and `FUTURE_TOLERANCE` up to `3600 s`, the window is 61 s – 3600 s of L2 lag.

**Same pattern in `LazerConsumer`** (used by `PythOracle`):

```solidity
// LazerConsumer.sol, line 162
ts.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
```

The Pyth Lazer `FeedUpdateTimestamp` (property 12) is also real-world time; the same rejection occurs.

---

### Impact Explanation

**Primary – bad-price execution:** When `MAX_TIME_DRIFT < Δ ≤ FUTURE_TOLERANCE`, the pool's `getBidAndAskPrice()` returns bid/ask derived from a price that is up to `FUTURE_TOLERANCE` seconds stale. A trader can swap against a price that no longer reflects the market, extracting value from LPs or the protocol.

**Secondary – pool DoS:** When `Δ > FUTURE_TOLERANCE`, the stored price also fails staleness, `getBidAndAskPrice()` reverts with `FeedStalled`, and all swaps, withdrawals, and liquidity operations that require a price quote are blocked until the sequencer catches up.

---

### Likelihood Explanation

The `CompressedOracle` is deployed on Blast and other L2s (networks.json). The Renzo report's empirical measurement showed Blast's `block.timestamp` lagging Ethereum's by ~107 days at launch. Even under normal Arbitrum operation the sequencer can set timestamps up to 24 hours behind real-world time per the Arbitrum docs. The `MAX_TIME_DRIFT = 60 s` tolerance is orders of magnitude smaller than these documented bounds. Any sequencer pause or chain-specific timestamp initialization that exceeds 60 s triggers the rejection path.

---

### Recommendation

1. **Remove the future-timestamp guard from the push paths** (as the Renzo judge recommended for the analogous check). Monotonicity (`isAfter`) already prevents replay; the `FutureTimestamp` check adds no security benefit and breaks L2 compatibility.
2. If the guard must be retained, set `MAX_TIME_DRIFT ≥ FUTURE_TOLERANCE` so that any lag that allows the old price to pass staleness also allows a new push to succeed, eliminating the bad-price window.
3. Add a sequencer-uptime check (Chainlink L2 Sequencer Feed) before accepting or serving prices, consistent with the `PriceProviderL2` L2-awareness intent.

### Proof of Concept

```
Setup:
  CompressedOracle deployed on Blast, MAX_TIME_DRIFT = 60 s
  PriceProviderL2 pointing at CompressedOracle, FUTURE_TOLERANCE = 3600 s

State at T=0 (real-world):
  Blast block.timestamp = T_l2 = T_real − 120 s   (sequencer 120 s behind)
  Oracle stores price P1 with timestampMs = T_real * 1000
  _isStale(T_real, T_l2, MAX_TIME_DELTA, 3600):
      T_real > T_l2  →  (T_real − T_l2) = 120 ≤ 3600  →  NOT stale ✓

Price moves to P2 (e.g., +5%):

Pusher attempts to push P2 with timestamp T_real + 1:
  revertIfAfterBlockTimeWithDrift:
      (T_real + 1) > T_l2 + 60 = T_real − 120 + 60 = T_real − 60
      T_real + 1 > T_real − 60  →  TRUE  →  FutureTimestamp revert ✗

Oracle still holds P1.
Pool calls getBidAndAskPrice() → _isStale passes → bid/ask from P1 returned.
Trader swaps at P1 while market is at P2.
