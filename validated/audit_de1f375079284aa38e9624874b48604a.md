### Title
`revokePusher()` does not invalidate the pusher's consent signature, allowing the creator to replay it and re-establish delegation within the deadline window — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers()` accepts an EIP-191 pusher-consent signature that is bound to `(chainid, oracle, deadline, pusher, creator)`. No nonce or "used" flag is recorded when the signature is consumed. After a pusher calls `revokePusher()` to clear `namespaceRemapping[pusher]`, the creator can immediately replay the original signature (while the deadline is still live) to re-establish the delegation. The pusher's revocation is therefore ineffective for the entire remaining deadline window, and the pusher's fallback pushes continue to land in the creator's namespace against the pusher's explicit intent.

---

### Finding Description

`allowPushers()` verifies the pusher's EIP-191 consent and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The only replay guard is a deadline check: [2](#0-1) 

`revokePusher()` clears the mapping but does **not** consume or invalidate the signature: [3](#0-2) 

The code's own NatSpec comment acknowledges the exact risk but claims the deadline is the solution: [4](#0-3) 

The deadline is **not** a solution — it only limits the window. Within that window, the creator can call `allowPushers()` again with the identical `(deadline, pusher, sig)` tuple after every `revokePusher()` call, cycling the mapping back to `creator` indefinitely until the deadline expires. There is no nonce, no consumed-signature mapping, and no revocation-timestamp check.

---

### Impact Explanation

The `fallback()` push path resolves the namespace at call time: [5](#0-4) 

When the creator re-establishes delegation after revocation, every subsequent push from the pusher lands in the **creator's** namespace instead of the pusher's own. Any pool or price provider reading from `feedIdOf(pusher, slotIndex, positionIndex)` receives stale or zero data. `ProtectedPriceProvider._computeBidAsk()` treats a zero price or a zero timestamp as a stall condition and returns `(0, type(uint128).max)`: [6](#0-5) 

This freezes swaps on any pool whose price provider is bound to the pusher's feed namespace, constituting broken core pool functionality and bad-price execution.

---

### Likelihood Explanation

The creator already holds a valid, unexpired signature — they used it in the initial `allowPushers()` call. No new off-chain material is needed. The replay requires only a single on-chain transaction after each `revokePusher()`. The window is as long as the deadline the pusher originally signed (commonly hours to days). Any creator who wishes to retain namespace control over a pusher who has revoked can exploit this at zero cost.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) public usedSignatures` and mark each signature hash as used on first acceptance:

```diff
+ mapping(bytes32 => bool) public usedSignatures;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
        );
        require(pusher == ECDSA.recover(hash, signatures[i]));
+       require(!usedSignatures[hash], SignatureAlreadyUsed());
+       usedSignatures[hash] = true;

        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

Alternatively, include a per-pusher nonce in the signed message and increment it on every `revokePusher()` call, so any previously issued signature becomes cryptographically invalid after revocation.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = block.timestamp + 1 days
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1);
pushers[0] = pusher;

vm.prank(creator);
