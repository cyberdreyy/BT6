### Title
`revokePusher()` bypass via `allowPushers()` signature replay — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` contains no guard against re-establishing delegation after a pusher has self-revoked. A creator who holds a valid (not-yet-expired) consent signature can call `allowPushers` again with the **same signature** immediately after the pusher calls `revokePusher()`, restoring `namespaceRemapping[pusher] = creator`. This makes `revokePusher()` completely ineffective for the entire duration of the deadline window — the exact analog of the Shelter.sol M-17 bug where `activate()` could be called again to reset the timestamp and bypass the grace-period check.

---

### Finding Description

`allowPushers` checks two things: (1) the deadline has not expired, and (2) the pusher's EIP-191 signature is valid over `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`. [1](#0-0) 

There is **no nonce, no revocation flag, and no check that `namespaceRemapping[pusher]` is currently zero** before writing. After a pusher calls `revokePusher()` — which sets `namespaceRemapping[msg.sender] = address(0)` — the creator can immediately call `allowPushers` again with the **identical** signature and deadline to restore `namespaceRemapping[pusher] = creator`. [2](#0-1) 

The code's own NatSpec comment acknowledges the threat model precisely:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The deadline only prevents using an **expired** signature. It does not prevent replaying a **valid, not-yet-expired** signature to re-establish delegation after revocation. The mitigation the comment claims the deadline provides does not actually exist.

The slot-structure documentation repeats the same incorrect claim: [4](#0-3) 

---

### Impact Explanation

`revokePusher()` is the only self-protection mechanism a pusher has. Once bypassed, the pusher's fallback pushes continue to land in the creator's namespace: [5](#0-4) 

If the pusher is an automated oracle bot or contract that continues to push price data (believing it is now writing to its own namespace), every push is silently redirected to the creator's namespace. The creator's oracle feeds remain live and attributed to the pusher's data stream even after the pusher explicitly tried to stop. Any pool consuming that feed via `AnchoredPriceProvider` will continue to receive prices from a pusher who has revoked consent. [6](#0-5) 

The broken invariant is: **"A pusher can self-revoke their delegation."** This invariant is documented in `revokePusher()`'s NatSpec and is the stated purpose of the function. The creator — a semi-trusted party — can bypass it indefinitely until the deadline expires.

---

### Likelihood Explanation

- The creator already holds the pusher's consent signature (they used it to call `allowPushers` the first time).
- No front-running is required. The creator can call `allowPushers` at any time after the pusher revokes, as long as the deadline has not expired.
- Deadlines are typically set days or weeks in the future (the test suite uses `block.timestamp + 1 days`).
- The loop `revokePusher → allowPushers(same sig)` can be repeated indefinitely within the deadline window. [7](#0-6) 

No test covers the scenario where the creator re-establishes delegation after the pusher revokes, confirming the gap.

---

### Recommendation

Include a per-pusher revocation nonce in the signed message so that each revocation invalidates all prior consent signatures:

```solidity
mapping(address => uint256) public pusherRevocationNonce;

function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    pusherRevocationNonce[msg.sender]++;          // invalidates all prior sigs
    emit PusherRevoked(msg.sender, creator);
}

// In allowPushers, include the nonce in the signed digest:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherRevocationNonce[pusher]   // <-- added
    ))
);
```

After `revokePusher()` increments the nonce, any previously issued signature (which was signed over the old nonce) will fail ECDSA recovery, making re-establishment impossible without a fresh signature from the pusher.

### Proof of Concept

```
1. Pusher signs: sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+1day, pusher, creator)))
2. Creator calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)

4. Creator calls allowPushers(T+1day, [pusher], [sig])  ← SAME signature, SAME deadline
   → namespaceRemapping[pusher] = creator  ✓ (revocation silently undone)

5. Pusher's automated pushes continue to land in creator's namespace.
   Steps 3–4 repeat indefinitely until T+1day expires.
```

This is the direct analog of Shelter.sol M-17: just as `activate()` could be called again to reset `activated[_token]` and bypass the grace-period check in `deactivate()`, `allowPushers()` can be called again with the same signature to reset `namespaceRemapping[pusher]` and bypass `revokePusher()`. [8](#0-7)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
