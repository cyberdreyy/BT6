### Title
`allowPushers` Delegation Signature Lacks Nonce, Enabling Creator to Replay Revoked Pusher Consent — (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

The `allowPushers` function in `CompressedOracle.sol` signs pusher consent over a fixed tuple `(chainid, oracle, deadline, pusher, creator)` with no nonce and no used-signature tracking. A creator who holds a valid, not-yet-expired pusher signature can call `allowPushers` repeatedly with the identical signature, re-establishing a delegation that the pusher explicitly cleared via `revokePusher()`. The pusher's revocation is therefore ineffective for the entire remaining deadline window.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The five-field tuple `(chainid, oracle, deadline, pusher, creator)` is **identical on every call** for the same pusher/creator pair and deadline. There is no per-use nonce, no used-signature bitmap, and no single-use enforcement anywhere in the contract. A grep across all production `.sol` files for `nonce`, `usedSignature`, `signatureUsed`, or `replayProtect` returns zero hits inside the oracle contracts, confirming the absence of any replay guard.

The pusher's self-revocation path clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

But because the creator still holds the original signature and `block.timestamp <= deadline` remains true, the creator can immediately call `allowPushers` again with the same bytes, restoring `namespaceRemapping[pusher] = creator`. This cycle repeats until the deadline expires.

The `fallback()` push path resolves the effective namespace at push time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [3](#0-2) 

So any push that arrives while the mapping has been silently re-established routes into the creator's namespace, not the pusher's own namespace.

---

### Impact Explanation

- **`revokePusher()` is rendered a no-op** for the entire remaining deadline window. The pusher's explicit consent withdrawal is overridden by the creator on every block.
- Any automated pusher (off-chain bot, keeper, or contract that uses `allowContractPushers` is immune, but EOA-based automated pushers are not) that does not re-query `namespaceRemapping` before each push will continue routing price updates into the creator's namespace.
- The creator's namespace feeds directly into pools via `getOracleData` → `price()` → `getBidAsk()`. A malicious creator who has lured a reputable price pusher into their namespace can maintain that routing indefinitely, causing the pool to consume prices the pusher no longer intends to publish.
- This satisfies **bad-price execution** (stale or unintended bid/ask quote reaches a pool swap) and **admin-boundary break** (oracle role check — pusher revocation — bypassed by an unprivileged replay).

---

### Likelihood Explanation

- Requires a creator who holds a valid, not-yet-expired pusher signature. Deadlines are set by the creator and can be arbitrarily far in the future.
- Automated price-pushing infrastructure (bots, keepers) is the norm in oracle systems; such systems rarely re-check `namespaceRemapping` on every push cycle.
- The replay requires zero additional privileges — the creator simply re-submits the same calldata to `allowPushers`.
- **Medium likelihood**: the attacker must have previously obtained a pusher's consent signature, but once obtained, the attack is trivially repeatable.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on each successful `allowPushers` call, so that a revoked pusher can only be re-authorized with a freshly signed message:

```solidity
mapping(address => uint256) public pusherNonce;

// inside allowPushers loop:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

Alternatively, record used signature hashes in a `mapping(bytes32 => bool) private _usedSigs` and revert on reuse. Either approach ensures that `revokePusher()` is final: the creator cannot re-establish the delegation without a new, post-revocation signature from the pusher.

---

### Proof of Concept

```
1. Pusher P signs off-chain:
       sig = sign(keccak256(abi.encode(chainid, oracle, T_far_future, P, C)))

2. Creator C calls allowPushers(T_far_future, [P], [sig])
       → namespaceRemapping[P] = C  ✓

3. P calls revokePusher()
       → namespaceRemapping[P] = address(0)  (P believes delegation is gone)

4. C calls allowPushers(T_far_future, [P], [sig])  ← SAME signature, no revert
       → namespaceRemapping[P] = C  (delegation silently restored)

5. P's automated pusher fires its next update via fallback():
       creator = namespaceRemapping[P]  // = C, not P
       key = C_namespace | slotId
       _writeStorage(key, newSlotValue)   // price lands in C's feed, not P's

6. Pool consuming C's feedId reads the price P pushed, which C now controls
   the routing of — bad-price execution path is open.
```
