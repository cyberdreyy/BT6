### Title
Revoked Pusher Re-Authorization via Replayed Consent Signature in `allowPushers` — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
The `allowPushers` delegation mechanism accepts EIP-191 consent signatures that commit to `(chainid, oracle, deadline, pusher, creator)` but include **no nonce and no revocation counter**. After a creator calls `revokePusher`, the original on-chain consent signature remains cryptographically valid until its deadline expires. Anyone who observed the original `allowPushers` calldata can replay it to silently re-authorize the revoked pusher, restoring their ability to write arbitrary prices into the oracle slot.

### Finding Description

`CompressedOracleV1` is a registrationless oracle where a feed's identity is its location: `feedId = creator << 96 | chainid << 16 | slotIndex << 8 | positionIndex`. [1](#0-0) 

The `fallback()` push path resolves the writing namespace via `namespaceRemapping[msg.sender]`, falling back to `msg.sender` itself: [2](#0-1) 

Pusher delegation is established through `allowPushers`, which requires each pusher to supply an EIP-191 consent signature. The test helper reveals the exact signed digest:

```solidity
keccak256(abi.encode(block.chainid, address(oracle), deadline, _pusher, _creator))
``` [3](#0-2) 

The signed message commits to `(chainid, oracle, deadline, pusher, creator)`. There is **no nonce, no revocation epoch, and no "issued-at" timestamp** in the signed payload. The protocol documentation explicitly acknowledges this gap:

> "the signed consent has no data timestamp, so an undated signature could re-establish a delegation after the pusher revoked it" [4](#0-3) 

**Attack path:**

1. Creator authorizes pusher `P` with `deadline = block.timestamp + 30 days`. The `allowPushers` calldata (including `P`'s consent signature) is permanently visible on-chain.
2. Pusher `P`'s key is compromised. Creator calls `revokePusher(P)`, clearing `namespaceRemapping[P]`.
3. Attacker (or `P` itself) replays the original `allowPushers` calldata before `deadline` expires. Because the signature is still cryptographically valid (deadline has not passed, no nonce was consumed), the call succeeds and `namespaceRemapping[P]` is restored.
4. `P` calls `fallback()` and writes a manipulated slot word — arbitrary price `p`, spread indices `s0`/`s1` — into the creator's namespace: [5](#0-4) 

5. The only freshness gate is monotonic timestamp: `timestampMs.isAfter(oldTimestampMs)`. The attacker supplies a timestamp strictly greater than the stored one (but not in the future), so the write succeeds.
6. `PriceProvider` / `AnchoredPriceProvider` reads the corrupted slot via `getOracleData`, which decodes the attacker-controlled `p` through `U64x32.decode` and the spread indices through `Codebook256`: [6](#0-5) 

7. The pool's `_getBidAndAskPriceX64()` consumes the corrupted bid/ask, and `SwapMath` executes swaps at the attacker-controlled price, draining the pool or giving the attacker a free arbitrage. [7](#0-6) 

### Impact Explanation

A compromised-and-revoked pusher can be silently re-authorized by replaying the original on-chain consent signature. Once re-authorized, the pusher writes an arbitrary price into the oracle slot. This price propagates through `PriceProvider → pool.swap()`, causing:

- **Bad-price execution**: traders receive more output than the true oracle price permits, or LPs receive less input than owed.
- **Pool insolvency**: repeated bad-price swaps drain token reserves below LP claims.

This is a direct loss of user principal and LP assets, meeting the Critical/High impact gate.

### Likelihood Explanation

- The original `allowPushers` calldata is permanently on-chain and trivially extractable.
- The replay window is the full remaining lifetime of the original `deadline` (which could be weeks or months).
- No special privilege is required beyond submitting a transaction before the deadline.
- The creator has no on-chain mechanism to invalidate the old signature short of waiting for the deadline to expire.

Likelihood: **Medium–High** (requires a compromised pusher key, but the replay window is long and the replay itself is unprivileged).

### Recommendation

Include a per-creator **revocation nonce** (or per-pusher nonce) in the signed consent message:

```solidity
keccak256(abi.encode(
    block.chainid,
    address(oracle),
    deadline,
    _pusher,
    _creator,
    revocationNonce[_creator]   // ← increment on every revokePusher call
))
```

Increment `revocationNonce[creator]` inside `revokePusher`. This invalidates all outstanding consent signatures for that creator's namespace in a single transaction, regardless of their deadlines.

        uint256 end;
        uint256 namespace;

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

            end := calldatasize()
            namespace := shl(96, creator) // [creator:20][zeros:12]

        // 4 * 6 + 7 + 1 = 32 bytes per slot
        if (end == 0 || end % 32 != 0) revert BadCalldataLength();

        for (uint256 ptr = 0; ptr < end; ptr += 32) {
            uint256 word;
            assembly ("memory-safe") {
                word := calldataload(ptr)
