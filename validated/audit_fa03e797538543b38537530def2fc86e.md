### Title
`allowPushers` Delegation-Consent Signature Replay Silently Re-Establishes Revoked Pusher Delegation, Feeding Bad Prices Into Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts an EIP-191 consent signature from the pusher and maps `namespaceRemapping[pusher] = creator`. The only replay guard is a deadline check. There is no nonce and no used-signature registry. After a pusher calls `revokePusher()` to clear the mapping, the creator can immediately call `allowPushers` again with the **identical** signature and deadline to silently re-establish the delegation — as long as the deadline has not yet expired. The pusher has no on-chain way to detect or prevent this within the deadline window.

---

### Finding Description

`allowPushers` computes the consent hash as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only guard before writing `namespaceRemapping[pusher] = msg.sender` is `_ensureDeadline(deadline)`, which checks `block.timestamp <= deadline`. [2](#0-1) 

`revokePusher()` clears the mapping to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because the signature is not invalidated or recorded as used, the creator can call `allowPushers` again with the same `(deadline, pusher, signature)` tuple at any point before `deadline`. The mapping is restored to `msg.sender` (the creator) without any new consent from the pusher.

The NatSpec comment on `allowPushers` itself acknowledges the structural gap:

> "the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The comment frames the deadline as the mitigation, but the deadline only bounds the replay window — it does not prevent replay within that window. A pusher who signs a 30-day consent and revokes on day 1 remains forcibly re-delegatable for the remaining 29 days.

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after the silent re-delegation lands in the creator's namespace, not the pusher's own namespace.

---

### Impact Explanation

The `feedIdOf` encoding ties a feed to a specific creator address:

```
feedId = creator << 96 | block.chainid << 16 | slotIndex << 8 | positionIndex
``` [6](#0-5) 

A pool's `IPriceProvider` is configured against a specific `feedId` derived from the creator's address. After silent re-delegation, the pusher's slot words — which the pusher believes are updating their own feeds — overwrite the creator's storage slot. The pool then reads the pusher's data as its authoritative bid/ask quote.

If the pusher is pushing data for a different asset pair, a different price scale, or simply stale data (because they believe they have no active delegation and are not maintaining the creator's feeds), the pool receives an incorrect or stale price. This satisfies the "bad-price execution" impact gate: a stale, inverted, or mismatched bid/ask quote reaches a live pool swap.

---

### Likelihood Explanation

- The creator is a semi-trusted actor (they deployed the pool and configured the oracle), so a malicious creator is within the contest's semi-trusted threat model.
- The pusher must have signed a consent with a deadline that has not yet expired. Long-lived deadlines (days to weeks) are the natural operational choice for off-chain pusher infrastructure.
- The pusher has no on-chain signal that their revocation was undone. `namespaceRemapping` is a public mapping, but off-chain pushers are not expected to poll it before every push.
- The pusher continuing to push after revoking is the normal case: they revoked to stop feeding the creator's pool, but they continue maintaining their own feeds — which now silently land in the creator's namespace.

---

### Recommendation

Track used consent signatures with a per-pusher nonce or a `usedSignatures` mapping, and invalidate the signature on first use. Alternatively, require the pusher to sign a fresh consent on every `allowPushers` call by including a monotonically increasing nonce in the hash:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline,
                         pusherNonce[pusher]++, pusher, msg.sender))
);
```

This ensures that once a pusher revokes, any previously issued consent signature is permanently invalidated regardless of the deadline.

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 30-day deadline.
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = sign(PUSHER_KEY,
    keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)));

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes — believes they are free.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — no new consent required.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds
assertEq(oracle.namespaceRemapping(pusher), creator);   // re-delegated

// 5. Pusher pushes data believing it goes to their own namespace.
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(WRONG_PRICE, 5, 5);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(slotId, pos, raw, tsMs));
assertTrue(ok);

// 6. Data lands in CREATOR's namespace, not pusher's.
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, slotId, pos)).price,
         U64x32.decode(uint32(raw >> 16)));  // creator's pool reads WRONG_PRICE
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher,  slotId, pos)).price, 0);
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
