### Title
Creator can replay pusher consent signature within deadline window to silently re-establish revoked delegation, feeding wrong prices into pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` uses a deadline-bounded EIP-191 signature for pusher consent but includes **no nonce**. Within the deadline window the creator can replay the identical signature to re-establish a delegation the pusher already revoked via `revokePusher()`. The pusher's subsequent pushes — now intended for their own namespace — silently land in the creator's namespace instead, feeding wrong prices into every pool that reads from the creator's feeds.

---

### Finding Description

`allowPushers` hashes `(chainid, oracle_address, deadline, pusher, creator)` and requires the pusher's ECDSA signature over that digest: [1](#0-0) 

There is no nonce, no "signature-consumed" bitmap, and no on-chain record that the signature was already used. The code comment explicitly acknowledges the replay risk and claims the deadline mitigates it: [2](#0-1) 

The claim is incorrect. The deadline prevents replay **after** it expires; it does nothing to prevent the creator from calling `allowPushers` a second time with the same signature **before** the deadline. `revokePusher` clears `namespaceRemapping[pusher]` to `address(0)`: [3](#0-2) 

But the original signature is still valid until `deadline`, so the creator can immediately call `allowPushers` again with the same bytes to restore `namespaceRemapping[pusher] = creator`. The pusher has no on-chain way to detect this.

After re-establishment the `fallback` push path resolves the namespace as: [4](#0-3) 

Every subsequent push by the pusher — now intended for their own namespace and a different asset — overwrites the creator's storage slot. Pools reading `feedIdOf(creator, slotIndex, positionIndex)` consume those wrong prices.

---

### Impact Explanation

A pool whose `PriceProvider` reads `CompressedOracleV1` at `feedIdOf(creator, slot, pos)` will receive prices that were pushed for a completely different asset (the pusher's own feed). This is a **bad-price execution** path: the bid/ask returned to `MetricOmmPool.swap` is derived from an unrelated price, allowing traders to extract value from LPs or causing the pool to reject valid swaps. The corrupted value is the decoded `U64x32` mid-price and `Codebook256` spread pair stored in the creator's slot: [5](#0-4) 

---

### Likelihood Explanation

- The pusher must have signed a consent with a future deadline (normal operational practice).
- The pusher must revoke within the deadline window (e.g., after a business dispute or key rotation).
- The creator needs only to re-submit the already-broadcast signature — no new cryptographic material is required.
- The creator is not trusted by the pusher; the pusher's only recourse is to wait for the deadline to expire, during which the creator can keep replaying.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on each successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// in allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]   // ← add nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;                             // ← consume
namespaceRemapping[pusher] = msg.sender;
```

This makes every consent signature single-use: once the pusher revokes, the old signature is permanently invalidated regardless of the deadline.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator; deadline = now + 1 day
bytes memory sig = _signConsent(PUSHER_KEY, block.timestamp + 1 days, pusher, creator);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(block.timestamp + 1 days, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes — expects to push into own namespace from now on
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — no revert, delegation restored
vm.prank(creator);
oracle.allowPushers(block.timestamp + 1 days, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // ← re-established

// 5. Pusher pushes prices for a different asset (their own feed)
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 wrongPrice = _packRaw(9_999_999, 8, 8); // attacker-asset price
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, wrongPrice, tsMs));
assertTrue(ok);

// 6. Creator's pool reads the wrong price from creator's namespace
IOffchainOracle.OracleData memory data =
    oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
// data.price == U64x32.decode(wrongPrice >> 16) — wrong asset price
// pools using this feedId now execute swaps at the wrong rate
```
