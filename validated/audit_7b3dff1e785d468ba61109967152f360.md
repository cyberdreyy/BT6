### Title
`AnchoredPriceProvider` synthetic mode with `baseFeedId == quoteFeedId` collapses every swap price to a constant 1.0, feeding a bad price into any pool using this provider — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

### Summary
When `AnchoredPriceProvider` is deployed with `baseFeedId == quoteFeedId`, the synthetic ratio in `_getBidAndAskPrice` always resolves to `ORACLE_DECIMALS = 1e8` (i.e., price = 1.0) regardless of the actual market price. The constructor guards `_baseToken != _quoteToken` but has no analogous guard for the feed IDs. The permissionless `AnchoredProviderFactory.createAnchoredProvider` also lacks this check, so any caller can deploy a misconfigured provider that silently feeds a constant 1.0 price to a live pool.

### Finding Description

In `_getBidAndAskPrice`:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
if (!ok) return (0, type(uint128).max);

bytes32 _quote = quoteFeedId;
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
    if (!ok2 || mid2 == 0) return (0, type(uint128).max);
    mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // X * 1e8 / X = 1e8
    spreadBps += spreadBps2;                          // 2 × original spread
}
return _computeBidAsk(mid, spreadBps);
``` [1](#0-0) 

When `baseFeedId == quoteFeedId`, both `_readLeg` calls return the same oracle value `X`. The ratio `X * ORACLE_DECIMALS / X = ORACLE_DECIMALS = 1e8` is always exactly 1.0 in 8-decimal format, regardless of the actual market price. The spread is doubled.

The constructor only guards against equal tokens:

```solidity
require(_baseToken != address(0) && _quoteToken != address(0) && _baseToken != _quoteToken);
``` [2](#0-1) 

There is no guard against `_baseFeedId == _quoteFeedId`. The factory's `createAnchoredProvider` is permissionless and passes both feed IDs directly to the constructor without any equality check: [3](#0-2) 

The circuit breaker `spreadBps > MAX_SPREAD_BPS` only fires if the doubled spread exceeds the configured cap. For feeds with small spreads (e.g., `spreadBps = 3`, `MAX_SPREAD_BPS = 150`), `2 × 3 = 6 ≤ 150`, so the circuit breaker does **not** trigger and the constant 1.0 price passes through to `_computeBidAsk` and then to the pool. [4](#0-3) 

### Impact Explanation

Any pool using this provider executes all swaps at a price of 1.0 (1e8) instead of the actual market price. If the actual price differs significantly from 1.0 (e.g., ETH/USD = $3 000), arbitrageurs drain the pool by trading at the wrong price, causing direct loss of LP principal. This is a bad-price execution impact: an unclamped, incorrect bid/ask quote reaches a live pool swap.

### Likelihood Explanation

`createAnchoredProvider` is permissionless — any caller can deploy a provider with `baseFeedId == quoteFeedId`. A pool deployer may do this accidentally (confusing single-feed and synthetic-feed modes) or intentionally. The missing guard is the only protection, and it is absent at both the constructor and factory layers. Once deployed, the provider is immutable (`baseFeedId` and `quoteFeedId` are `immutable`), so the misconfiguration cannot be corrected without redeployment.

### Recommendation

Add a guard in the `AnchoredPriceProvider` constructor immediately after the feed ID assignments:

```solidity
require(
    _quoteFeedId == bytes32(0) || _quoteFeedId != _baseFeedId,
    "SameFeedId"
);
```

Add the same check in `AnchoredProviderFactory.createAnchoredProvider` before deploying the provider:

```solidity
require(
    quoteFeedId == bytes32(0) || quoteFeedId != baseFeedId,
    "SameFeedId"
);
```

### Proof of Concept

1. Deploy `AnchoredPriceProvider` with `baseFeedId = quoteFeedId = FEED_ID` (same feed ID for both legs).
2. Push oracle data for `FEED_ID`: `mid = 3_000_00000000` (ETH/USD = $3 000, 8-decimal), `spreadBps = 3`.
3. Call `getBidAndAskPrice()` through a registered pool.
4. Inside `_getBidAndAskPrice`:
   - `_readLeg(baseFeedId)` → `mid = 3_000_00000000`, `spreadBps = 3`
   - `_readLeg(quoteFeedId)` → `mid2 = 3_000_00000000`, `spreadBps2 = 3` (same feed)
   - `mid = mulDiv(3_000_00000000, 1e8, 3_000_00000000) = 1e8` ← constant 1.0
   - `spreadBps = 3 + 3 = 6` ← doubled but below `MAX_SPREAD_BPS`
5. `_computeBidAsk(1e8, 6)` produces bid/ask around $1.00 instead of $3 000.
6. The pool executes all swaps at $1.00; arbitrageurs drain LP funds at $2 999 profit per ETH.

This is directly analogous to the Unlock `transferFrom(_from, _recipient)` bug where `_from == _recipient`: in both cases a self-referential identity (same source and destination feed / same sender and recipient) causes the two-step operation to collapse into a corrupted result — key destruction there, constant-1.0 price here — because the code validates the token-level identity (`_baseToken != _quoteToken`) but omits the feed-level identity check (`_baseFeedId != _quoteFeedId`).
