### Title
Pusher Consent Signature Has No Invalidation Mechanism — Creator Can Replay Original Signature to Re-Establish Delegation After Pusher Revokes - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` accepts an EIP-191 consent signature from the pusher but stores no record that the signature was consumed. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original consent signature (same `deadline`, same `pusher`, same `creator`) to re-establish the delegation. The pusher has no on-chain mechanism to invalidate the already-signed consent, so `revokePusher()` is not a durable security boundary when the creator still holds a valid, unexpired signature.

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)`, which checks `block.timestamp <= deadline`. There is no nonce, no used-signature bitmap, and no per-pusher revocation counter. The code's own comment acknowledges the problem but frames the deadline as the solution:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

The deadline limits the *window* but does not close it. Within that window, the same signature bytes can be submitted to `allowPushers` an unlimited number of times. `revokePusher()` only writes `namespaceRemapping[msg.sender] = address(0)`: [3](#0-2) 

There is nothing that marks the original consent as spent, so the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` tuple and the mapping is immediately restored to `msg.sender` (the creator).

### Impact Explanation

Once re-delegated, the pusher's `fallback()` push path resolves `namespaceRemapping[msg.sender]` to the creator and writes into the creator's namespace: [4](#0-3) 

Feeds in the creator's namespace are consumed by `AnchoredPriceProvider` / pool `getQuotes()` calls. If the pusher's key was compromised (the reason the pusher revoked), the attacker controlling that key can push arbitrary `U64x32`-encoded prices and `Codebook256` spread indices into the creator's live feed slots. Those values flow directly to pool swaps, enabling bad-price execution and direct loss of trader principal.

### Likelihood Explanation

The scenario requires the creator to replay the signature — either maliciously (to keep a compromised pusher active) or unknowingly (believing the revocation was a mistake). Consent signatures are broadcast on-chain in the original `allowPushers` calldata, so any observer can reconstruct the exact bytes. Deadlines are caller-chosen and can be set arbitrarily far in the future (the tests use `block.timestamp + 1 days` but nothing enforces a maximum). The longer the deadline, the larger the replay window after revocation.

### Recommendation

Record each consumed consent signature in a `mapping(bytes32 => bool) usedConsents` keyed on the digest, and revert if the digest has already been used. Alternatively, include a per-pusher nonce in the signed message and increment it on each successful delegation, so a revoked pusher can advance their nonce to invalidate all prior signatures:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline,
                         pusher, msg.sender, pusherNonce[pusher]++))
);
```

### Proof of Concept

```
1. Pusher signs consent for creator with deadline = block.timestamp + 365 days.
2. Creator calls allowPushers(deadline, [pusher], [sig]).
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher discovers their key is compromised; calls revokePusher().
   → namespaceRemapping[pusher] = address(0)  ✓
4. Creator (or attacker who observed step 2 on-chain) calls
   allowPushers(deadline, [pusher], [sig])  ← identical calldata.
   _ensureDeadline passes (deadline still in future).
   ECDSA.recover returns pusher  ← same sig, same hash.
   → namespaceRemapping[pusher] = creator  ← delegation restored.
5. Attacker (holding compromised pusher key) calls oracle fallback with
   a crafted slot word encoding an extreme price.
   → Creator's namespace feed is overwritten with bad price.
6. Pool swap reads the feed via getQuotes() → bad bid/ask → trader loss.
``` [5](#0-4) [6](#0-5)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

            end := calldatasize()

        // 4 * 6 + 7 + 1 = 32 bytes per slot

                word := calldataload(ptr)
