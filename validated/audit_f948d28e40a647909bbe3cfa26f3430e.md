### Title
Pusher self-revocation is silently bypassable via consent-signature replay before deadline — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`revokePusher()` clears `namespaceRemapping[pusher]` but the original EIP-191 consent signature carries no nonce and is never invalidated. As long as the deadline encoded in the signature has not yet expired, the creator can call `allowPushers` again with the exact same signature to immediately re-establish the delegation. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the signed consent.

### Finding Description

`allowPushers` requires a pusher-signed EIP-191 message that commits to `(chainid, oracle, deadline, pusher, creator)`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

The code comment explicitly acknowledges the threat:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The deadline is checked only at call time via `_ensureDeadline(deadline)` (`block.timestamp <= deadline`). There is no nonce, no per-pusher revocation flag, and no on-chain record that the pusher ever revoked. After `revokePusher()` sets `namespaceRemapping[pusher] = address(0)`, the creator can immediately call `allowPushers` again with the identical `(deadline, [pusher], [sig])` tuple — the check passes and `namespaceRemapping[pusher]` is restored to `creator`.

The protection the deadline provides is only against replay *after* the deadline expires. It provides zero protection against replay *before* the deadline expires, which is the exact window in which a pusher would revoke.

**Attack path:**

1. Pusher signs consent with a long-lived deadline (e.g., `block.timestamp + 365 days`).
2. Creator calls `allowPushers` — delegation established.
3. Pusher's key is compromised; pusher calls `revokePusher()` to protect the creator's namespace.
4. Creator (or an automated keeper that stores the original signature) calls `allowPushers` again with the same `(deadline, pusher, sig)` — delegation is immediately re-established.
5. Attacker (holding the compromised pusher key) calls the oracle `fallback` and pushes arbitrary prices into the creator's namespace.
6. `AnchoredPriceProvider` / pool reads the corrupted feed and executes swaps at the attacker-controlled bid/ask.

### Impact Explanation

Arbitrary prices written into the creator's namespace flow directly into any pool whose `IPriceProvider` reads that feed. A pool swap executed against a manipulated bid or ask allows the attacker to extract value from LPs or traders — a direct loss of user principal matching the bad-price execution impact class.

### Likelihood Explanation

The creator must call `allowPushers` again with the old signature. This is realistic because:
- Creators commonly store signed consents off-chain for operational re-use.
- Automated keeper scripts may re-establish delegations whenever `namespaceRemapping[pusher]` reads as zero.
- The creator may not know the pusher revoked for security reasons and may treat the revocation as an error to correct.

The pusher has no on-chain mechanism to signal "do not re-delegate" — the only signal is the cleared mapping, which the creator can overwrite.

### Recommendation

Introduce a per-pusher nonce or a revocation tombstone. The simplest fix is a monotonically incrementing nonce per pusher that is included in the signed digest and incremented on every successful `revokePusher` / `removePushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
namespaceRemapping[pusher] = address(0);
```

After revocation the nonce advances, making all previously issued signatures invalid regardless of their deadline.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

    /// @notice Allows a pusher to self-revoke their delegation. After revocation the
    ///         wallet pushes into its OWN namespace again (the registrationless default).
    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
