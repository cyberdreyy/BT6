### Title
Permissionless `register()` Unconditionally Clears Admin Blacklist, Bypassing Abuse-Protection — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
`OracleBase.register()` is a permissionless, fee-gated function that unconditionally removes a pool from the admin blacklist as a side-effect of registration. Any user can pay the registration fee (default: 1 wei) to un-blacklist a pool the admin has emergency-stopped, permanently defeating the abuse-protection mechanism.

### Finding Description
The `price(feedId, pool)` read path in `OracleBase` enforces three guards before returning oracle data to a pool:

```solidity
require(pool != address(0) && IPool(pool).inSwap() == msg.sender, InvalidInSwap());
require(!blacklisted[pool], Blacklisted(pool));
require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
``` [1](#0-0) 

The blacklist check (`!blacklisted[pool]`) is the admin's emergency lever to halt a pool from consuming oracle prices — for example, if the pool is identified as an MEV bot, is being used for oracle manipulation, or has a critical bug being exploited.

However, `register()` is permissionless and unconditionally clears the blacklist:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;          // ← clears admin blacklist
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    ...
}
``` [2](#0-1) 

The `registrationFee` is initialized to `1 wei`: [3](#0-2) 

The admin can raise the fee via `setRegistrationFee()`, but the bypass remains permissionless at any fee level — anyone with sufficient ETH can call it. There is no mechanism to make a blacklist entry permanent.

The analog to the external LpPool insolvency bug is direct: just as the LpPool's available-liquidity calculation omits unrealized short-position losses (a missing accounting term that lets the pool over-commit funds), `OracleBase`'s blacklist omits the invariant that a blacklisted pool must never be re-admitted through an unprivileged path. In both cases, a guard that is supposed to bound a critical resource (available liquidity / oracle access) is silently bypassed.

### Impact Explanation
The blacklist is the sole on-chain emergency brake for oracle abuse. Once bypassed:

- A pool blacklisted for oracle manipulation can immediately resume reading prices through the attributed `price(feedId, pool)` path.
- The `PriceRead` event emitted on each read provides no signal that the pool was previously blacklisted.
- The admin enters an unwinnable cat-and-mouse loop: every `setBlacklist(pool, true)` can be countered in the same block with `register(feedId, pool, factory)` for 1 wei.
- If the pool was blacklisted because it was extracting value from LPs via stale or manipulated prices, the extraction continues unimpeded.

This matches the allowed impact gate: **Admin-boundary break — factory/oracle role checks are bypassed by an unprivileged path.**

### Likelihood Explanation
**High.** The registration fee is 1 wei by default. The only precondition is that `factory` is an approved factory and `pool` is recognized by it as a valid pool — both conditions are trivially satisfied for any legitimate (but blacklisted) pool. The pool operator or any third party can execute the bypass in a single transaction.

### Recommendation
Separate blacklist clearing from registration. Add an explicit guard:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);
    require(!blacklisted[pool], Blacklisted(pool));   // ← reject, do not clear
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);
    registeredPool[feedId][pool] = true;
    ...
}
```

If clearing the blacklist on re-registration is intentional, it must be gated on `ADMIN_ROLE`, not left permissionless.

### Proof of Concept

1. Admin calls `setBlacklist(poolP, true)` — pool `P` is blacklisted for oracle abuse.
2. Attacker (or pool operator) calls `register(feedId, poolP, approvedFactory)` with `msg.value = 1 wei`.
3. Inside `register()`: `blacklisted[poolP]` is `true`, so the branch executes `blacklisted[poolP] = false`.
4. `registeredPool[feedId][poolP]` is set to `true`.
5. Pool `P` initiates a swap, marks itself in-swap with the provider, and calls `getBidAndAskPrice()`.
6. The provider calls `oracle.price(feedId, poolP)`.
7. All three guards in `price()` pass: `inSwap() == provider` ✓, `!blacklisted[poolP]` ✓ (cleared in step 3), `registeredPool[feedId][poolP]` ✓.
8. Oracle returns live price data to the previously-blacklisted pool — abuse-protection is fully defeated. [2](#0-1) [1](#0-0) [4](#0-3)
