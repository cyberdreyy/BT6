### Title
Creator Can Replay Pusher Consent Signature Within Deadline Window to Nullify `revokePusher()`, Enabling Continued Bad-Price Injection into Pool Feeds - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracle.sol` uses a deadline-bound EIP-191 signature as the sole replay guard. After a pusher calls `revokePusher()`, the creator can immediately re-call `allowPushers` with the **same signature and same deadline** — because no used-signature record exists — re-establishing delegation before the deadline expires. A compromised pusher whose legitimate owner tried to revoke can therefore continue writing arbitrary prices into the creator's namespace, which flows directly into pool swaps.

---

### Finding Description

`allowPushers` accepts a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`. The signed message commits to `(block.chainid, address(this), deadline, pusher, msg.sender)`. The only replay guard is the deadline check:

```solidity
// OracleBase.sol line 124-126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
```

When the pusher calls `revokePusher()`, it clears `namespaceRemapping[msg.sender] = address(0)`. But the original consent signature is **not invalidated** — there is no used-signature bitmap, no nonce, and no on-chain record that the signature was consumed. As long as `block.timestamp <= deadline`, the creator can call `allowPushers` again with the identical `(deadline, pushers[], signatures[])` arguments, immediately restoring `namespaceRemapping[pusher] = creator`.

The code's own NatSpec comment acknowledges the risk but misstates the protection:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The deadline prevents replay only **after** it expires. It provides zero protection against replay **before** expiry, which is exactly the window during which a compromised pusher is active.

The fallback push path resolves the namespace at call time:

```solidity
// CompressedOracle.sol lines 315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

So every time the creator replays `allowPushers`, the attacker (holding the compromised pusher key) regains write access to the creator's namespace slots.

---

### Impact Explanation

A compromised pusher can write any `U64x32`-encoded price and any codebook spread index into the creator's namespace via the `fallback()` push path. Those values are read by `getOracleData()` → `price()` → `AnchoredPriceProvider.getBidAndAskPrice()` → `MetricOmmPool.swap()`. Arbitrary bid/ask prices reaching a live swap cause:

- Traders receiving more output than the oracle permits (swap conservation failure).
- The pool receiving less input than owed (pool insolvency over time).
- LP claims becoming undercollateralised.

The creator's `removePushers` call is the correct remediation, but a creator who does not know the pusher is compromised will instead replay `allowPushers` to "restore" what they perceive as a broken feed — directly re-enabling the attacker.

---

### Likelihood Explanation

Medium. The attack requires:

1. A pusher's private key to be compromised.
2. The legitimate pusher to call `revokePusher()`.
3. The creator to replay `allowPushers` with the original calldata (which is permanently on-chain and trivially extractable from transaction history).

Step 3 is realistic: creators running automated feed-management scripts will re-submit the delegation transaction when they observe `namespaceRemapping[pusher] == address(0)`, without necessarily diagnosing the root cause. The deadline window can be days (typical operational practice is 1–7 day deadlines), giving the attacker a sustained window.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) public usedSignatures` and mark each signature hash as used on first acceptance:

```solidity
bytes32 sigHash = keccak256(signatures[i]);
require(!usedSignatures[sigHash], "signature already used");
usedSignatures[sigHash] = true;
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, include a per-pusher nonce in the signed message and increment it on every `revokePusher` / `removePushers` call, so any previously issued signature becomes invalid after revocation.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
