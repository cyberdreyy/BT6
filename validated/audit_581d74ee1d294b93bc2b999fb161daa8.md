### Title
`allowPushers` Signed Consent Is Never Invalidated, Allowing Creator to Replay Signature and Override Pusher's Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 signature but never marks it as consumed. A creator who holds a valid, unexpired signature can call `allowPushers` repeatedly with the same bytes to re-establish a delegation immediately after the pusher calls `revokePusher()`. The pusher's revocation is silently overwritten, and any subsequent pushes from that wallet continue to land in the creator's namespace — feeding the creator's registered pool with prices the pusher no longer intends to provide.

---

### Finding Description

`allowPushers` builds a hash over `(chainid, oracle, deadline, pusher, creator)` and recovers the signer: [1](#0-0) 

There is no nonce, no per-signature bitmap, and no state write that would prevent the same `(deadline, signature)` pair from being accepted a second time. After the creator calls `allowPushers` once, the only on-chain record is `namespaceRemapping[pusher] = creator`.

`revokePusher` clears that mapping: [2](#0-1) 

But the creator can immediately replay the original signature to restore it: [3](#0-2) 

The code comment explicitly acknowledges the replay concern ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and claims the deadline mitigates it. The deadline only caps the outer window — it does **not** prevent the creator from replaying the signature any number of times **within** that window. [4](#0-3) 

The `fallback` push path resolves the namespace at call time: [5](#0-4) 

So every push the wallet makes after the creator's replay lands in the creator's namespace, not the pusher's own.

---

### Impact Explanation

A pool registered against the creator's feedId reads prices from the creator's namespace via `getOracleData`: [6](#0-5) 

If the pusher revokes because their key is compromised, or because they no longer wish to serve the creator's pool, the creator can keep the delegation alive for the full deadline window. Any pushes the wallet makes during that window — whether by the legitimate pusher or by an attacker who has stolen the key — continue to update the creator's pool price. This is a **bad-price execution** path: the pool consumes prices from a source the pusher has explicitly tried to disconnect, potentially causing swaps to execute at stale, manipulated, or attacker-controlled mid/spread values.

---

### Likelihood Explanation

The attack requires only that the creator retained the original `(deadline, signature)` calldata — trivially available from the transaction that established the delegation. No privileged role is needed. The creator is a semi-trusted but unprivileged actor in the registrationless model. The deadline window can be set to days or weeks by the pusher at signing time, giving the creator a long replay window.

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` bitmap. Increment or set the flag on first acceptance, and reject any replay of the same hash:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

// inside allowPushers, after hash is computed:
require(!_usedDelegationSigs[hash], SignatureAlreadyUsed());
_usedDelegationSigs[hash] = true;
```

Alternatively, include a per-pusher nonce in the signed payload and store `pusherNonce[pusher]`, incrementing it on each successful `allowPushers` call so old signatures are permanently invalidated.

---

### Proof of Concept

```
1. Pusher signs: hash = keccak256(abi.encode(chainid, oracle, deadline=T+7days, pusher, creator))
2. Creator calls allowPushers(T+7days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)
4. Creator calls allowPushers(T+7days, [pusher], [sig])  ← SAME sig, still before deadline
   → namespaceRemapping[pusher] = creator  ← revocation silently overwritten
5. Pusher pushes a slot word via fallback()
   → creator = namespaceRemapping[pusher] = creator (not address(0))
   → data written to creator's namespace slot
6. Creator's pool calls getOracleData(feedIdOf(creator, slotIndex, positionIndex))
   → returns the pusher's (possibly stale/attacker-controlled) price
   → pool executes swap at that price → loss of funds
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
