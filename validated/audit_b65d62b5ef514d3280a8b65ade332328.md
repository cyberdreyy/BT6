### Title
Pusher delegation signature in `allowPushers` lacks a nonce, allowing a creator to replay a revoked consent and forcibly re-delegate a pusher within the deadline window — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies an EIP-191 signature that commits to `(chainid, oracle, deadline, pusher, creator)` but contains **no nonce**. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original, still-valid signature to re-establish `namespaceRemapping[pusher] = creator`, nullifying the revocation. The pusher's subsequent `fallback()` pushes are then silently redirected back into the creator's namespace, feeding oracle data the pusher no longer consented to provide into any pool that reads the creator's feeds.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The five-tuple `(chainid, oracle, deadline, pusher, creator)` is **static** for the lifetime of the deadline. There is no per-use counter, no consumed-flag, and no state change that would invalidate the signature after a single use.

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

But because the signature is not consumed, the creator can call `allowPushers` again with the **identical** `(deadline, pusher, signature)` tuple while `block.timestamp <= deadline`, and the check passes again, writing `namespaceRemapping[pusher] = creator` a second time. The pusher's revocation is silently undone.

The code comment acknowledges the deadline is needed to prevent post-revocation replay, but the deadline only prevents replay **after** it expires — it does nothing to prevent replay **before** it expires:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The comment describes the exact attack but incorrectly concludes the deadline alone prevents it.

---

### Impact Explanation

After the creator replays the signature, the pusher's `fallback()` pushes land in the creator's namespace again:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
// ... writes to creator's namespace
``` [4](#0-3) 

Any pool whose `PriceProvider` reads the creator's `feedId` via `price(feedId, pool)` will consume oracle data the pusher no longer authorized for that namespace. If the pusher is simultaneously pushing to their own namespace (believing their revocation succeeded), the creator's feeds receive data intended for a different context — a bad-price execution path for pools consuming those feeds. [5](#0-4) 

---

### Likelihood Explanation

- The pusher's original consent signature is a public on-chain transaction (calldata of the first `allowPushers` call), so the creator already holds it.
- The creator needs only to call `allowPushers` again with the same arguments before the deadline expires — a single transaction.
- The pusher has no on-chain mechanism to invalidate the signature short of waiting for the deadline to expire.
- Deadlines are typically set days in the future (tests use `block.timestamp + 1 days`). [6](#0-5) 

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on every successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
uint256 nonce = pusherNonce[pusher]++;
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce))
);
```

This ensures each consent signature is single-use. After `revokePusher()`, the nonce has already been incremented, so the old signature is permanently invalid.

---

### Proof of Concept

```solidity

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
