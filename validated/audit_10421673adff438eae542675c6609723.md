### Title
Missing Deadline in `updateBySignature` Allows Stale Signed Slot Words to Be Submitted at Any Future Time — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.updateBySignature` is missing the `deadline` parameter and deadline enforcement that the project's own docs and deployed ABI registry specify. Without a deadline, any party that obtains a creator-signed slot word can withhold it and submit it at a strategically chosen future moment, injecting a stale price that the downstream price providers will accept as fresh.

---

### Finding Description

**Documented design vs. actual code**

The project's own documentation at `smart-contracts-poc/docs/ru/oracle-packet-structure.md` specifies:

```
updateBySignature(feedCreator, deadline, newSlotValue, signature)
signature over: keccak256(abi.encode(chainid, oracleAddress, feedCreator, deadline, newSlotValue))
Required check: deadline must be in the future (DeadlineExceeded otherwise)
```

The deployed ABI in `smart-contracts-poc/contract-registry/versions/registry.json` also records the four-parameter form `updateBySignature(address,uint256,uint256,bytes)` with a `deadline` field.

The actual on-chain code, however, is:

```solidity
// CompressedOracle.sol L271-303
function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
    external override returns (bool)
{
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    _writeStorage(key, bytes32(newSlotValue & ~uint256(0xff)));
    return true;
}
```

No `deadline` parameter, no `_ensureDeadline` call, and no deadline field in the signed digest.

**Why the code's own justification is wrong**

The NatSpec comment at L268-270 claims:

> The signed slot word carries its own 56-bit timestamp, so replay is neutralized by the monotonicity check below — no deadline is needed.

The monotonicity check (`timestampMs.isAfter(oldTimestampMs)`) prevents *replay of the same slot word after a newer one has been stored*. It does **not** prevent a party from withholding a freshly-signed slot word and submitting it later while the embedded timestamp is still within the downstream staleness window. The two protections guard against different threats.

**The analog to the auction-unbundling bug class**

In the external report, bundled transactions pass the same signature checks as standalone ones because nothing asserts they must be included in an auction. Here, a signed slot word passes the same signature check whether it is submitted immediately or held for up to `MAX_REF_STALENESS` seconds, because nothing asserts it must be submitted before a deadline.

**Attack path**

1. Creator signs a slot word at time T with price P and hands it to a relayer (or broadcasts it to a public mempool).
2. Attacker observes the signed slot word (mempool, relayer API, or a compromised relayer).
3. Price moves from P to P′ within the next 30 seconds.
4. Attacker calls `updateBySignature(creator, slotWord_T_P, sig)` at time T+30.
5. The oracle accepts it: `timestampMs(T) > stored(0)` and `T ≤ (T+30) + MAX_TIME_DRIFT`.
6. Oracle stores price P with `refTime = T`.
7. `AnchoredPriceProvider._readLeg` calls `_isStale(T, T+30, 60)` → `(T+30 − T) = 30 ≤ 60` → **not stale**.
8. Pool swap executes at the stale price P instead of the current price P′.

The `maxTimeDrift` on the oracle (set to 60 in production config) only blocks *future* timestamps; it imposes no lower bound on how old a submitted timestamp may be.

---

### Impact Explanation

A stale bid/ask quote from `CompressedOracleV1` reaches `AnchoredPriceProvider.getBidAndAskPrice()` and is consumed by a live pool swap. Depending on the direction of price movement:

- Traders receive more output tokens than the current oracle price permits (swap conservation failure).
- LPs absorb the loss: their reserves are depleted at a price that no longer reflects market reality.

The window is bounded by `MAX_REF_STALENESS` (60 s in production), but 60 seconds is more than sufficient for meaningful price movement in volatile markets.

---

### Likelihood Explanation

`updateBySignature` is a permissionless, publicly callable function. Any signed slot word that transits a public mempool, a shared relayer, or any observable channel is replayable by any observer within the staleness window. The creator has no on-chain mechanism to cancel or expire a signed slot word once issued, because the only cancellation path is submitting a newer slot word first — which requires the creator to act before the attacker does.

---

### Recommendation

Restore the deadline parameter that the docs and ABI registry specify:

```solidity
function updateBySignature(
    address feedCreator,
    uint256 deadline,
    uint256 newSlotValue,
    bytes calldata signature
) external override returns (bool) {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), feedCreator, deadline, newSlotValue))
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    ...
}
```

Creators should set deadlines to the shortest interval consistent with their relayer's expected submission latency (e.g., 10–30 seconds), ensuring that a withheld slot word expires before the downstream staleness window closes.

    /// @notice Single-slot update authorized by the creator's signature. The signed slot
    ///         word carries its own 56-bit timestamp, so replay is neutralized by the
    ///         monotonicity check below — no deadline is needed.
    function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
        external
        override
        returns (bool)
    {
        require(feedCreator != address(0), InvalidNamespace());

        uint256 namespace;
        assembly ("memory-safe") {
            namespace := shl(96, feedCreator) // [creator:20][zeros:12]
        }

        uint8 slotId = uint8(newSlotValue); // LSB
        TimeMs timestampMs = toTimeMs(newSlotValue >> 8 & X56);
        timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
        bytes32 key = bytes32(namespace | uint256(slotId));
        uint256 old = uint256(_loadStorage(key));
        TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

        bool newer = timestampMs.isAfter(oldTimestampMs);
        if (!newer) {
            return false;
        }

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
        );
        require(feedCreator == ECDSA.recover(hash, signature));

        _writeStorage(key, bytes32(newSlotValue & ~uint256(0xff)));

        return true;
    }

    /// @dev Pure staleness check (L1). Any future refTime is stale.
    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta
    ) internal pure returns (bool) {
        if (refTime == 0) return true;
        if (refTime > nowTs) return true;
        return (nowTs - refTime) > maxDelta;
    }

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
