### Title
Pusher Cannot Invalidate Signed Consent — Creator Can Replay `allowPushers` Signature After `revokePusher()` to Restore Hijacked Write Authority — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1::allowPushers()` accepts a pusher's EIP-191 consent signature bound only to `(chainid, address(this), deadline, pusher, creator)`. There is no nonce in the signed payload. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately call `allowPushers()` again with the **identical old signature** (provided the deadline has not yet expired) to silently re-establish the mapping. The pusher has no mechanism to invalidate their already-signed consent hash within the deadline window.

---

### Finding Description

`allowPushers` constructs the signed hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no per-pusher nonce, no used-signature bitmap, and no on-chain record that the consent was already consumed. The only replay guard is the `deadline` timestamp check:

```solidity
_ensureDeadline(deadline);
``` [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But this does not consume or invalidate the original signature. The creator still holds the raw bytes of the pusher's consent and can call `allowPushers(deadline, [pusher], [oldSig])` again in the same block, restoring `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone.

The code comment itself acknowledges the risk but only addresses the post-deadline case:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [4](#0-3) 

The deadline prevents re-establishment **after** expiry, but not **within** the deadline window — which is exactly when revocation is most urgently needed (e.g., key compromise).

---

### Impact Explanation

The `fallback()` push path resolves the effective namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

If the pusher's private key is compromised and the pusher calls `revokePusher()`, the creator can replay the old consent signature to restore the mapping. The attacker controlling the compromised key then pushes arbitrary price data directly into the creator's namespace. Those prices are consumed by `AnchoredPriceProvider` / pool swap paths via `price(feedId, pool)`:

```solidity
function price(bytes32 feedId, address /* pool */)
    external view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    return _price(feedId);
}
``` [6](#0-5) 

The only freshness gate is monotonicity (timestamp must be newer than the stored slot). An attacker with the live key can trivially satisfy this by using `block.timestamp * 1000` as the timestamp, injecting a manipulated mid-price and spread into the feed that drives pool swaps — constituting bad-price execution and potential swap conservation failure.

---

### Likelihood Explanation

- The pusher must have previously signed a consent with a future deadline (normal operational practice for long-lived delegations).
- The creator must replay the signature after revocation. The creator is the namespace owner and holds the signature bytes from the original `allowPushers` call (on-chain calldata is public).
- The pusher's key compromise scenario is realistic for any hot-wallet pusher infrastructure.
- No privileged role beyond the creator (namespace owner) is required to replay.

---

### Recommendation

Add a per-pusher nonce to the signed consent payload and track consumed nonces on-chain:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

Alternatively, record each consumed `(pusher, deadline, creator)` tuple in a `usedConsents` mapping so that a revoked consent cannot be replayed even within its deadline window. This mirrors the standard EIP-2612 nonce pattern and directly closes the analog to the reported nonce-invalidation class.

---

### Proof of Concept

```
1. Pusher signs consent: sig = sign(chainid, oracle, deadline=T+1day, pusher, creator)
2. Creator calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher's key is compromised. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (revoked)
4. Creator calls allowPushers(T+1day, [pusher], [sig])  ← SAME sig, deadline still valid
   → namespaceRemapping[pusher] = creator  ✗ (re-established)
5. Attacker (holding compromised key) calls oracle fallback with:
     slotId=X, tsMs=block.timestamp*1000, price=MANIPULATED_VALUE
   → Passes monotonicity check (fresh timestamp)
   → Writes manipulated price into creator's namespace slot X
6. Pool reads price(feedIdOf(creator, X, pos), pool)
   → Returns manipulated mid/spread → bad-price swap execution
```
