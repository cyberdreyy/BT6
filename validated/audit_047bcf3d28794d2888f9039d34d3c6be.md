### Title
`revokePusher()` Is Ineffective Due to Nonce-less Signature Replay in `allowPushers` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts a pusher's EIP-191 consent signature that is bound to `(chainid, oracle, deadline, pusher, creator)` but carries **no nonce**. After a pusher calls `revokePusher()`, the creator can immediately re-call `allowPushers` with the identical, still-valid signature to silently re-establish the delegation. The revocation mechanism is therefore ineffective for the entire window between the revocation and the signature's deadline, allowing a compromised pusher key to continue writing arbitrary prices into the creator's namespace and feeding bad bid/ask quotes to pools.

---

### Finding Description

`allowPushers` constructs the signed hash as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no per-use invalidation, and no on-chain record that the signature was consumed. The code's own comment acknowledges the design intent:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The deadline prevents use **after** expiry, but it does nothing to prevent the creator from replaying the same bytes before expiry. `revokePusher()` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

Because the signature is not invalidated, the creator can call `allowPushers` again with the exact same `(deadline, pusher, sig)` tuple to restore `namespaceRemapping[pusher] = creator`, undoing the revocation in a single transaction.

The `fallback()` push path then accepts any calldata from the re-delegated (compromised) pusher address and writes it directly to the creator's storage namespace with no further authorization check beyond the monotonicity guard:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
...
_writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
``` [4](#0-3) 

The price written is decoded by `getOracleData` and returned verbatim through `price()` to any pool consuming that `feedId`: [5](#0-4) 

---

### Impact Explanation

A compromised pusher key can push arbitrary `U64x32`-encoded prices and codebook spread indices into the creator's namespace. Any pool registered against a `feedId` in that namespace will receive the manipulated bid/ask quote on the next swap, enabling:

- **Bad-price execution**: swapper receives more output than the true oracle price permits, draining pool reserves.
- **Pool insolvency**: repeated bad-price swaps cause LP claims to exceed pool balances.

---

### Likelihood Explanation

Medium. The attack requires two conditions: (1) a pusher key is compromised, and (2) the creator re-calls `allowPushers` with the original signature before the deadline. Condition (2) is realistic in operational practice — creators routinely store consent signatures for re-authorization after bot restarts or key rotations, and may not be immediately aware of a pusher compromise. Deadlines are commonly set months or years in the future, leaving a long replay window.

---

### Recommendation

Introduce a per-pusher nonce that is incremented on every successful `allowPushers` call and included in the signed hash:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

This ensures each consent signature is single-use: once consumed (or once the pusher revokes and the nonce advances), the old signature is permanently invalid.

### Proof of Concept

```
// Setup
creator calls allowPushers(deadline=T+365days, [pusherBot], [sig])
  → namespaceRemapping[pusherBot] = creator  ✓

// Pusher key is compromised; pusher tries to stop attacker
pusherBot calls revokePusher()
  → namespaceRemapping[pusherBot] = address(0)  ✓ (revoked)

// Creator re-uses the SAME sig (deadline not yet expired)
creator calls allowPushers(deadline=T+365days, [pusherBot], [sig])
  → namespaceRemapping[pusherBot] = creator  ← revocation undone

// Attacker (holding compromised pusherBot key) pushes manipulated price
attacker calls CompressedOracleV1.fallback() with crafted slot word:
  [manipulatedPrice:32][spread0:8][spread1:8] × 4 lanes | [ts:56] | [slotId:8]
  → _writeStorage(key, manipulatedSlotValue)

// Pool reads the bad price on next swap
pool calls price(feedId, pool)
  → getOracleData returns manipulated mid/spread
  → swap executes at attacker-controlled bid/ask
```
