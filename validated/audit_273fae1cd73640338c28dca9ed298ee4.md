### Title
`allowPushers` Signature Replay After `revokePusher` Re-Establishes Delegation Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` has no nonce or used-signature tracking. A creator who holds a pusher's still-valid EIP-191 consent signature can call `allowPushers` again after the pusher self-revoked, silently re-establishing `namespaceRemapping[pusher] = creator` and redirecting all subsequent fallback pushes back into the creator's namespace without the pusher's knowledge.

### Finding Description
The signature domain for `allowPushers` is `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`. [6](#0-5) 

Nothing in this domain or in the function body marks the signature as consumed after first use. `_ensureDeadline` only checks `block.timestamp <= deadline`. [2](#0-1) 

Attack sequence:
1. Pusher signs consent with `deadline = T + 1 day` and hands it to creator.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`. [3](#0-2) 
4. Creator calls `allowPushers(deadline, [pusher], [sig])` again with the **identical** signature (deadline not yet expired) → `namespaceRemapping[pusher] = creator` again.
5. Pusher, believing they have revoked, continues pushing prices intended for their own namespace. The fallback resolves `namespaceRemapping[pusher]` at call time and routes every push into the creator's namespace. [7](#0-6) 

### Impact Explanation
The pusher's prices — which may be calibrated for their own feed slots — silently overwrite the creator's namespace slots. Any pool consuming the creator's `feedId` will read the pusher's prices instead of the creator's intended prices. If the pusher's prices diverge from the creator's (different asset, different scale, or stale after the pusher stopped updating), this constitutes bad-price execution reaching live swaps. The pusher has no on-chain way to detect the re-delegation; `namespaceRemapping` is a public mapping but the pusher has no event or revert to alert them.

### Likelihood Explanation
Requires the creator to retain the pusher's original signature and act within the deadline window after the pusher revokes. Deadline windows of 1–7 days are typical in practice (as shown in the test suite). The creator is not a trusted oracle admin — they are an ordinary feed owner with no privileged role beyond their own namespace. The replay call is a single public transaction with no special access.

### Recommendation
Track consumed signatures with a `mapping(bytes32 => bool) usedSignatures` keyed on the full signature hash (or a `(pusher, nonce)` pair where the pusher increments their nonce on each consent). Alternatively, bind the signature to a monotonic per-pusher nonce stored on-chain so that `revokePusher` increments the nonce, invalidating all prior signatures regardless of deadline.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
