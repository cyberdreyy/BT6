### Title
Missing L2 Sequencer Uptime Check Allows Stale Pre-Downtime Prices to Reach Pool Swaps — (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`ProtectedPriceProviderL2` is deployed on L2 networks (Arbitrum, Optimism, Base, etc.) and is L2-aware via `FUTURE_TOLERANCE`, but it contains no sequencer uptime feed check. After a sequencer restart, the last pre-downtime oracle price stored in the offchain oracle remains in storage and passes `_isStale()` if the downtime duration was less than `MAX_TIME_DELTA`. A trader can immediately swap against this stale price before a fresh price is pushed, causing LP losses.

---

### Finding Description

`_computeBidAsk()` calls `_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)` as its sole freshness gate: [1](#0-0) 

The `_isStale()` implementation only checks the age of the oracle's stored `refTime` against `block.timestamp`: [2](#0-1) 

There is no `sequencerUptimeFeed` storage variable, no `GRACE_PERIOD` constant, and no call to a Chainlink `AggregatorV3Interface` sequencer uptime feed anywhere in the contract. The only L2-specific logic is `FUTURE_TOLERANCE` for clock skew: [3](#0-2) 

The Metric OMM offchain oracle (PythOracle / ChainlinkOracle data streams) is a **push-based** system. During sequencer downtime, authorized pushers cannot submit new prices. The last pre-downtime price remains in oracle storage with its original `refTime`. When the sequencer restarts, `block.timestamp` resumes from the restart point, and if `(restartTime - refTime) <= MAX_TIME_DELTA`, the stale price passes the freshness check and is returned as a valid bid/ask.

The protocol is aware of the sequencer uptime pattern: the registry shows a separate `ChainlinkVerifierL2` contract with both `GRACE_PERIOD` and `sequencerUptimeFeed`, confirming the design intent exists elsewhere but was not applied here: [4](#0-3) 

---

### Impact Explanation

After sequencer restart, there is a window — potentially minutes — before authorized pushers submit fresh prices. During this window, `getBidAndAskPrice()` returns the pre-downtime price. A trader who monitored the market during downtime knows the true current price diverges from the stored price. They can immediately execute a swap against the pool at the stale bid/ask, extracting value from LP reserves. This is **bad-price execution** leading to direct LP principal loss.

---

### Likelihood Explanation

Arbitrum, Optimism, and Base have all experienced sequencer outages. The attack requires: (1) a sequencer outage shorter than `MAX_TIME_DELTA`, (2) meaningful price movement during downtime, and (3) an attacker ready to transact at restart. All three conditions are realistic and have co-occurred historically. `MAX_TIME_DELTA` can be up to 7 days per the constructor bound, but even a tight deployment (e.g., 60 seconds) is exploitable if the sequencer is down for 59 seconds and the market moves. [5](#0-4) 

---

### Recommendation

Add a sequencer uptime check in `_computeBidAsk()` (or a dedicated internal helper) before the staleness check. The standard pattern:

```solidity
// immutable set at construction:
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600;

function _checkSequencer() internal view {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0 means sequencer is up; 1 means down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
}
```

Call `_checkSequencer()` at the top of `_computeBidAsk()` before `_isStale()`. This matches the pattern already present in `ChainlinkVerifierL2` in the registry.

---

### Proof of Concept

```solidity
// Foundry fork test (Arbitrum)
function test_stalePreDowntimePriceAccepted() public {
    // 1. Push a fresh price at T=0
    uint256 T0 = block.timestamp;
    oracle.push(feedId, price_before_downtime, T0); // refTime = T0

    // 2. Simulate sequencer downtime: warp forward by MAX_TIME_DELTA - 1
    vm.warp(T0 + MAX_TIME_DELTA - 1);

    // 3. No new price pushed (sequencer was down, pushers couldn't submit)
    // oracle still holds price_before_downtime with refTime = T0

    // 4. Call getBidAndAskPrice() — should revert FeedStalled but does NOT
    (uint128 bid, uint128 ask) = poolContract.getBidAndAskPrice();

    // 5. Pre-downtime price is accepted — invariant violated
    assertGt(bid, 0, "stale pre-downtime price accepted: invariant broken");
    // A trader can now swap at price_before_downtime while true price diverged
}
```

The `_isStale` check at line 207 evaluates `(MAX_TIME_DELTA - 1 - 0) > MAX_TIME_DELTA` → `false`, so the pre-downtime price passes and `getBidAndAskPrice()` returns valid bid/ask instead of reverting `FeedStalled`. [6](#0-5) [7](#0-6)

```text
    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```
