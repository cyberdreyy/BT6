### Title
Fallback Push Path Bypasses Future-Timestamp Guard, Allowing a Delegated Pusher to Permanently Lock a Creator's Feed and Feed Bad Prices to Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`updateBySignature` enforces a future-timestamp guard (reverts with `FutureTimestamp` if the slot's embedded 56-bit timestamp exceeds `block.timestamp`). The registrationless fallback push path enforces only a monotonicity check ("overwrite if newer than stored"). A delegated pusher can therefore push a slot carrying a far-future timestamp through the fallback path, which the oracle accepts. Every subsequent legitimate update — including the creator's own `updateBySignature` calls — is then rejected by the monotonicity check, permanently locking the feed until the forged future timestamp is reached. Because the forged slot also carries attacker-chosen price/spread data, and because the `maxTimeDrift` staleness guard in downstream providers only rejects timestamps that are *too old* (not too far in the future), the manipulated price propagates to any pool reading that feed.

---

### Finding Description

**The asymmetric guard (the direct analog to the mempool bug):**

| Path | Future-timestamp check | Monotonicity check |
|---|---|---|
| `updateBySignature` | ✅ reverts with `FutureTimestamp` | ✅ |
| `fallback` push | ❌ absent | ✅ |

`updateBySignature` is documented as: *"The signed slot word carries its own 56-bit timestamp, so replay is neutralized by the monotonicity check below — no deadline is needed."* The future-timestamp guard is an additional safety layer on top of monotonicity. The fallback path omits it. [1](#0-0) 

The test suite confirms `updateBySignature` rejects future timestamps: [2](#0-1) 

No equivalent test exists for the fallback path, and the audit pivot description for the fallback path reads only *"per-word timestamp check → packed slot overwrite if newer"* — monotonicity only, no future-timestamp bound. [3](#0-2) 

**Delegation surface that makes this reachable by a semi-trusted actor:**

`allowPushers` lets a creator delegate any EOA into their namespace. The delegated pusher can then call the fallback path and write into the creator's namespace. [4](#0-3) 

Once the pusher is delegated, their fallback call resolves the namespace to the creator's address and writes the slot. The creator has no on-chain mechanism to override a slot whose stored timestamp exceeds `block.timestamp` — every subsequent `updateBySignature` or fallback push with a current timestamp is rejected by the monotonicity check.

**Why `maxTimeDrift` does not save the pool:**

Downstream providers apply a staleness guard of the form `block.timestamp − refTime ≤ maxTimeDrift`. A future `refTime` makes `block.timestamp − refTime` underflow (wraps to a huge positive value in unchecked arithmetic, or is simply negative in checked math that treats the result as zero). Either way the staleness check passes, and the provider returns the attacker-chosen mid price and spread to the pool.

---

### Impact Explanation

A pool reading the compromised feed receives an attacker-controlled bid/ask quote for the entire duration between the current block and the forged future timestamp. Swaps execute against the manipulated price, causing:

- **Swap conservation failure**: the pool gives out more token than the true oracle price permits, or demands less input than owed.
- **Pool insolvency**: repeated swaps against the bad price drain one side of the pool faster than fees can compensate, leaving LP claims uncovered.
- **Permanent DoS of the feed**: the creator cannot push a corrective update until real wall-clock time reaches the forged timestamp.

---

### Likelihood Explanation

The trigger requires a delegated pusher to act maliciously. Delegation is intentionally permissionless — any creator can delegate any EOA that signs consent. The pusher is semi-trusted (trusted to push valid data, not trusted to lock the feed). The creator has no way to distinguish a future-timestamp push from a normal push before it lands on-chain, and the `removePushers` remedy only prevents *future* writes; it cannot undo the already-stored future timestamp. The attack is a single transaction and requires no special capital. [5](#0-4) 

---

### Recommendation

Add the same future-timestamp guard to the fallback push path that already exists in `updateBySignature`. Concretely, before accepting any per-word slot overwrite in the fallback handler, verify:

```solidity
if (newTimestampMs > block.timestamp * 1000) revert FutureTimestamp();
```

This mirrors the guard already present in `updateBySignature` and closes the asymmetry.

### Proof of Concept

```
Setup
─────
1. Creator deploys / uses CompressedOracleV1.
2. Creator calls allowPushers(deadline, [pusher], [sig]) to delegate `pusher`.
3. Pool is configured to read feedIdOf(creator, slotId, posIdx).

Attack (single tx, from `pusher`)
──────────────────────────────────
4. pusher constructs a fallback payload with:
     timestampMs = uint56((block.timestamp + 365 days) * 1000)   // 1 year in the future
     price       = <manipulated U64x32 value>
     spread0/1   = <attacker-chosen codebook indexes>
5. pusher calls address(oracle).call(payload).
   → namespace resolves to creator (delegated)
   → monotonicity check: storedTs (0 or any past value) < futureTs → passes
   → NO future-timestamp check → slot written

Result
──────
6. oracle.getOracleData(feedIdOf(creator, slotId, posIdx)) returns attacker price + future refTime.
7. Creator calls updateBySignature with current timestamp → monotonicity check fails → reverted.
8. Pool.swap() → provider.getBidAndAskPrice() → oracle.price() → returns manipulated mid/spread.
   maxTimeDrift check: block.timestamp − futureRefTime underflows → passes.
9. Swap executes at attacker-controlled price. Pool drains.
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
