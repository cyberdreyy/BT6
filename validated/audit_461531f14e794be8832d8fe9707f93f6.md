### Title
Pusher Self-Revocation Can Be Silently Replayed by Creator Within Deadline Window, Keeping Compromised Pusher Active — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` commits its EIP-191 signature to `(chainid, address(this), deadline, pusher, creator)` but includes **no nonce**. After a pusher calls `revokePusher()` to neutralize a compromised key, the creator can immediately replay the original signature — which remains cryptographically valid until `deadline` — to re-establish the delegation. This cycle can repeat indefinitely within the deadline window, keeping a compromised pusher active and feeding bad prices into any pool that consumes feeds from the creator's namespace.

---

### Finding Description

`allowPushers` validates the pusher's consent signature and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The signed message is:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

There is no nonce, no per-pusher revocation counter, and no on-chain flag that marks a signature as consumed. [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` back to `address(0)`: [3](#0-2) 

Because clearing the mapping does not invalidate the original signature, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple, re-writing `namespaceRemapping[pusher] = creator`. The cycle

```
revokePusher()  →  allowPushers(same sig)  →  revokePusher()  →  …
```

can repeat until `block.timestamp >= deadline`. The code comment acknowledges the replay concern but only addresses the post-deadline case, leaving the within-deadline window unguarded: [4](#0-3) 

---

### Impact Explanation

Pools that use `CompressedOracle` as their `IPriceProvider` call `price(feedId, pool)` during every swap to obtain the bid/ask spread. If a pusher key is compromised and the pusher calls `revokePusher()` to stop bad prices, but the creator (malicious or also compromised) immediately replays the original consent signature, the compromised pusher remains active. Bad prices (stale, inverted, or out-of-range values) continue to reach the pool's `_getBidAndAskPriceX64` path, causing swaps to execute at incorrect rates — direct loss of principal to LPs or traders, and potential pool insolvency if the price deviation is large enough.

---

### Likelihood Explanation

Requires the creator to be malicious or have their key compromised **and** the pusher key to be compromised simultaneously. However, the pusher consent signature is typically a long-lived off-chain artifact (deadline set days or weeks in the future), so a single leaked signature is sufficient for the creator to replay indefinitely. The pusher has no on-chain mechanism to neutralize the delegation before the deadline expires.

---

### Recommendation

Introduce a per-pusher revocation nonce or a boolean `revokedByPusher[pusher]` flag. When `revokePusher()` is called, increment the nonce (or set the flag). Include the nonce in the signed message so that any previously issued signature becomes invalid after revocation:

```solidity
// In the signed hash:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher():
pusherNonce[msg.sender]++;
delete namespaceRemapping[msg.sender];
```

This mirrors the fix applied to the external bug: just as `UnfrozenUntilUpgrade` prevents re-freezing after an unfreeze, a nonce prevents re-delegation after a self-revocation.

---

### Proof of Concept

```
1. Pusher signs consent: sig = sign(chainid || oracle || deadline=T+30days || pusher || creator)
2. Creator calls allowPushers(T+30days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher key is compromised; attacker pushes bad price into creator's feed.
4. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are safe)
5. Creator (or attacker with creator key) calls allowPushers(T+30days, [pusher], [sig])
   → same sig is still valid (deadline not reached, no nonce)
   → namespaceRemapping[pusher] = creator  ← delegation silently restored
6. Attacker continues pushing bad prices into creator's namespace.
7. Pool swap reads bad bid/ask from CompressedOracle → trader receives excess tokens or LP is drained.
8. Steps 4–7 repeat until block.timestamp >= T+30days.
``` [1](#0-0) [3](#0-2)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
