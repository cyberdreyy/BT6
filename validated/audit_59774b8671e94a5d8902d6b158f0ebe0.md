### Title
Pusher delegation consent signature can be replayed by creator to nullify `revokePusher()`, permanently redirecting pusher data into creator namespace — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracle.sol` verifies a pusher's EIP-191 consent signature but stores no nonce and marks no signature as consumed. A creator who holds a valid, non-expired consent signature can call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple immediately after the pusher calls `revokePusher()`, re-writing `namespaceRemapping[pusher] = creator` and silently undoing the revocation. The pusher cannot stop their fallback pushes from landing in the creator's namespace until the deadline timestamp passes.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

and then unconditionally writes `namespaceRemapping[pusher] = msg.sender` on a valid recovery. [1](#0-0) 

There is no nonce, no per-pusher used-signature bitmap, and no check that `namespaceRemapping[pusher]` is currently `address(0)` before writing. The only replay guard is the deadline, which is a future timestamp chosen by the creator at the time of the original delegation. [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But because the original signature is already on-chain (emitted in the `PusherAuthorized` event and visible in the original `allowPushers` calldata), the creator can immediately call `allowPushers` again with the same `deadline` and `signature`, restoring `namespaceRemapping[pusher] = creator`. This cycle can be repeated indefinitely until `block.timestamp > deadline`.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after their revocation still lands in the creator's namespace, not their own, as long as the creator keeps replaying the signature.

The code comment in `allowPushers` explicitly acknowledges the deadline is the only replay guard ("an undated signature could re-establish a delegation AFTER the pusher revoked it"), but the deadline alone is insufficient: it only bounds the window, it does not prevent the creator from replaying within that window. [5](#0-4) 

---

### Impact Explanation

The `CompressedOracle` namespace is the write surface for oracle price data consumed by `AnchoredPriceProvider` and ultimately by pool swaps. A creator whose namespace is backed by a pool can maintain unauthorized write attribution over a pusher's data stream even after the pusher has explicitly revoked consent. Concretely:

1. The pusher cannot redirect their pushes to their own namespace to build an independent feed — every push continues to overwrite the creator's slots.
2. If the pusher is a legitimate price provider who revoked because they discovered the creator is operating a manipulated pool, the creator can keep the data flowing into that pool's oracle feed, preventing the oracle from going stale and blocking the pool's natural circuit-breaker (stale timestamp rejection).
3. The pusher's own namespace remains at timestamp 0 (never-pushed), so any pool or provider that the pusher intended to serve with their own feed receives no data.

This maps to the allowed impact gate: **broken core oracle functionality** (revocation invariant) that can sustain bad-price execution by keeping a manipulated pool's oracle alive against the pusher's explicit intent.

---

### Likelihood Explanation

- The creator already possesses the signature: it was submitted on-chain in the original `allowPushers` calldata and is trivially recoverable from transaction history.
- No special privilege is required beyond being the original `msg.sender` (creator) in the delegation.
- The attack is a single additional `allowPushers` call, executable in the same block as the pusher's `revokePusher`.
- The window is bounded only by the deadline, which the creator chose and may have set far in the future (e.g., `block.timestamp + 365 days`).

Likelihood: **Medium** (requires a malicious creator, but the mechanism is fully permissionless and the signature is public).

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` bitmap so that each consent signature can only establish a delegation once:

```solidity
mapping(bytes32 => bool) private _usedConsentSignatures;

// inside allowPushers loop:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedConsentSignatures[sigHash], "signature already used");
_usedConsentSignatures[sigHash] = true;
```

Alternatively, include a per-pusher monotonic nonce in the signed digest and store the last-used nonce on-chain, so a pusher can invalidate all prior signatures by incrementing their nonce (analogous to EIP-2612 permit nonces).

A simpler partial fix: check that `namespaceRemapping[pusher] == address(0)` before writing, so a creator cannot re-establish a delegation that the pusher has revoked without the pusher signing a fresh consent.

---

### Proof of Concept

```
1. creator calls allowPushers(deadline=T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

2. pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)

3. creator calls allowPushers(deadline=T+1day, [pusher], [sig])  // SAME sig, on-chain
   → _ensureDeadline passes (T+1day > block.timestamp)
   → ECDSA.recover returns pusher  ✓
   → namespaceRemapping[pusher] = creator  ← revocation undone

4. pusher pushes via fallback():
   → namespaceRemapping[msg.sender] == creator  (not address(0))
   → push lands in creator namespace, not pusher's own namespace

5. Steps 2–4 repeat indefinitely until block.timestamp > T+1day.
   The pusher's own namespace stays at timestamp=0 (stale/rejected by consumers).
   The creator's namespace continues to receive live price data.
```
