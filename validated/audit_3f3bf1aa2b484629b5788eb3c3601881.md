### Title
`priceGuard` min/max bounds are stored but never enforced in any oracle read path, allowing out-of-bounds prices to reach pool swaps — (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, `smart-contracts-poc/contracts/oracles/compressed/OracleBase.sol`)

---

### Summary

Both oracle base contracts expose `setPriceGuard(feedId, minPrice, maxPrice)` which stores a `PriceGuard` struct intended to clamp oracle prices within safe bounds before they reach pool swaps. However, neither `_readPrice` in `providers/OracleBase` nor `getOracleData`/`_price` in `CompressedOracle` ever load or check `priceGuard[feedId]`. The stored bounds are dead code — the guard is configured but never fires.

---

### Finding Description

**Analog mapping:** The external report describes `_afterPublicDeposit` (the share-lock hook) being skipped on the destination chain. The native analog here is `priceGuard` (the price-bound hook) being skipped in every oracle read path.

`setPriceGuard` is callable by the admin or the feed's `stateGuard` on both oracle families: [1](#0-0) [2](#0-1) 

The stored `PriceGuard` struct holds `min` and `max` price bounds. The intent is clear from the name and the admin-only setter: these bounds are a safety gate.

However, every on-chain read path returns the raw price without consulting `priceGuard[feedId]`:

**`providers/OracleBase._readPrice`** — called by both `price(feedId, pool)` and `integratorPrice(feedId)`: [3](#0-2) 

**`CompressedOracle.getOracleData`** — called by `_price` and the public `price(feedId, pool)`: [4](#0-3) 

**`CompressedOracle._price`** — the final return to the pool: [5](#0-4) 

In none of these functions is `priceGuard[feedId]` loaded or compared against the returned price. The `priceGuard` mapping is written by `setPriceGuard` and emits `PriceGuardUpdated`, but is never read during price delivery.

The pool's attributed read path confirms the gap — `price(feedId, pool)` in `providers/OracleBase` performs the in-swap binding, blacklist, and registration checks, then calls `_readPrice` which returns the raw value: [6](#0-5) 

No priceGuard check exists between the raw oracle value and the value returned to the pool.

---

### Impact Explanation

Any price pushed by an authorized oracle source — including a Pyth Lazer flash-crash value, a Chainlink Data Streams anomaly, or a compromised CompressedOracle pusher — is delivered to `MetricOmmPool.getBidAsk()` without bound enforcement. A price below `min` causes the pool to quote an artificially low bid, allowing a trader to buy the base asset far below fair value and drain LP reserves. A price above `max` causes the pool to quote an artificially high ask, causing the pool to accept far less input than owed. Both cases are direct loss of LP principal and constitute bad-price execution under the allowed impact gate.

---

### Likelihood Explanation

Medium-High. The trigger does not require a malicious actor: any legitimate oracle update that falls outside the admin-configured bounds (e.g., during a flash crash, a Chainlink sequencer anomaly, or a stale Pyth update) will bypass the guard silently. For the CompressedOracle path, an authorized pusher (semi-trusted) can push any price within the timestamp window and the guard will not fire. The admin who called `setPriceGuard` has no on-chain indication that the guard is inoperative.

---

### Recommendation

In `providers/OracleBase._readPrice`, enforce the guard before returning:

```solidity
function _readPrice(bytes32 feedId) internal view returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime) {
    OracleData memory data = _oracleDataRaw(feedId);
    PriceGuard memory guard = priceGuard[feedId];
    if (guard.max != 0) {
        require(uint128(data.price) >= guard.min && uint128(data.price) <= guard.max, PriceOutOfBounds(feedId, data.price));
    }
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
```

Apply the equivalent check in `CompressedOracle.getOracleData` before returning `data.price`. Alternatively, enforce the guard at the `CompressedOracle._price` level so both the public `price()` and any future read paths inherit it automatically.

---

### Proof of Concept

1. Admin calls `setPriceGuard(ethFeedId, 1_000e8, 10_000e8)` on `PythOracle` — ETH/USD, bounds $1 000–$10 000.
2. Pyth Lazer delivers a flash-crash update: price = `100e8` ($100). `LazerConsumer._verifyAndStore` stores it (timestamp is newer, signature is valid).
3. Pool calls `price(ethFeedId, pool)` → `_readPrice` → returns `mid = 100e8`. `priceGuard[ethFeedId]` is never loaded.
4. Pool quotes a bid of ~$100 for ETH. A trader calls `swap`, receives ETH at $100, and immediately sells it at market price (~$3 000).
5. LP reserves are drained by the difference; the priceGuard that the admin believed was active never fired.

    function getOracleData(bytes32 feedId) public view override returns (OracleData memory data) {
        (address creator, uint8 slotIndex, uint8 positionIndex) = _unpackFeedId(feedId);

        SlotLayout memory _layout = _loadSlotLayout(_oracleSlot(creator, slotIndex));
        CompressedOracleData memory compressed = _selectCompressedData(_layout, positionIndex);

        if (compressed.s1 == 0xff && compressed.s0 == 0xff) {
            data.spread1 = BPS_BASE;
            data.spread0 = BPS_BASE;

        data.spread0 = _decodeCodebookIndex(compressed.s0);
        data.spread1 = _decodeCodebookIndex(compressed.s1);
        data.timestampMs = _layout.timestampMs;
    }
