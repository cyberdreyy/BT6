Now I have enough context to write the finding. The vulnerability is in `allowPushers` in `CompressedOracle.sol`. The code's own comment acknowledges the risk but the deadline mitigation is insufficient — it only prevents using an *expired* signature, not replaying a *still-valid* one after the pusher has revoked.

---

### Title
Pusher delegation re-established after self-revocation via signature replay within deadline window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracleV1.allowPushers` has no nonce and no revocation-state check. A creator who holds a pusher's consent signature can call `allowPushers` again with the same signature at any time before the deadline expires, silently overwriting the pusher's `revokePusher()` call and re-routing all subsequent pushes back into the creator's namespace. The pusher's explicit revocation is not permanent.

### Finding Description

`allowPushers` signs consent over `(chainid, address(this), deadline, pusher, creator)` and stores the delegation unconditionally:

```solidity
// CompressedOracle.sol:204-209
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;   // ← no nonce, no revocation check
``` [1](#0-0) 

`revokePusher` clears the mapping:

```solidity
// CompressedOracle.sol:238-242
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

Because the signature domain contains no nonce and no revocation epoch, the creator can call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple at any point before `deadline` expires. The `_ensureDeadline` check only rejects signatures whose deadline has already passed:

```solidity
// OracleBase (compressed).sol:124-126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [3](#0-2) 

The code comment on `allowPushers` explicitly acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but treats the deadline as the full mitigation. It is not: the deadline only bounds the replay window; it does not prevent the creator from replaying the same signature any number of times within that window. [4](#0-3) 

The analog to the Olympus M-05 bug is direct: in Olympus, `INSTR.store()` returns a counter-derived `proposalId` that can collide after a module upgrade, and `Governance.submitProposal` writes `getProposalMetadata[proposalId]` without checking whether the slot is already occupied. Here, `allowPushers` writes `namespaceRemapping[pusher]` without checking whether the pusher has since revoked — the "counter" is the deadline-scoped signature, and the "different contract" is the `namespaceRemapping` mapping that the pusher cleared via `revokePusher`.

### Impact Explanation

After the pusher revokes, they push into their **own** namespace (the registrationless default). If the creator replays the old signature, the pusher's subsequent pushes are silently re-routed into the **creator's** namespace. If the pusher has since changed what they are pushing (e.g., they now push BTC/USD prices for their own feeds, while the creator's slot was ETH/USD), the creator's slot receives the wrong asset's price. Any pool whose `IPriceProvider` reads that feed via `getBidAndAskPrice()` → `_readLeg()` → `oracle.price(feedId, pool)` will execute swaps against a corrupted mid price, causing bad-price execution and direct loss to LPs or traders. [5](#0-4) 

### Likelihood Explanation

The creator already holds the pusher's consent signature (they submitted it in the original `allowPushers` call). The only precondition is that the deadline has not yet expired. Deadlines are chosen by the creator, so a creator who wants to retain control can set a deadline far in the future (e.g., 1 year). The pusher has no on-chain way to invalidate the old signature short of waiting for the deadline to expire. The trigger is a single public transaction by the creator — no privileged role, no special setup.

### Recommendation

Add a per-pusher revocation nonce to the signature domain:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on every revoke

// in allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusherNonce[pusher], pusher, msg.sender))

// in revokePusher / removePushers:
pusherNonce[msg.sender]++;   // or pusherNonce[pusher]++
namespaceRemapping[...] = address(0);
```

This makes every previously issued consent signature immediately invalid once the pusher revokes, regardless of whether the deadline has expired.

### Proof of Concept

```solidity
// 1. Creator obtains pusher's consent for deadline = block.timestamp + 365 days
uint256 deadline = block.timestamp + 365 days;
bytes memory sig = pusher.sign(keccak256(abi.encode(
    block.chainid, address(oracle), deadline, pusher, creator)));

// 2. Creator establishes delegation
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator  ✓

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0)  ✓

// 4. Pusher now pushes BTC/USD into their OWN namespace (slot 0, position 0)
vm.prank(pusher);
oracle.call(wordAt(slotId=0, pos=0, btcUsdPrice, ts));
// oracle.getOracleData(feedIdOf(pusher, 0, 0)).price == btcUsdPrice  ✓

// 5. Creator replays the SAME signature — deadline still valid
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);   // ← replay, no revert
// namespaceRemapping[pusher] == creator  ← revocation silently undone

// 6. Pusher's next push (still BTC/USD) lands in CREATOR's ETH/USD slot
vm.prank(pusher);
oracle.call(wordAt(slotId=0, pos=0, btcUsdPrice2, ts+1));
// oracle.getOracleData(feedIdOf(creator, 0, 0)).price == btcUsdPrice2
// ← creator's ETH/USD feed now contains BTC/USD price → bad-price execution in pool
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
