### Title
`OracleBase::_readPrice` never enforces the configured `PriceGuard` bounds — out-of-range oracle prices reach pool swaps unclamped - (`File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase` (providers) exposes `setPriceGuard` to configure per-feed `[min, max]` price bounds intended to act as a circuit-breaker when an oracle reports an extreme or manipulated price. However, the internal `_readPrice` function — the single path through which every pool swap reads a price — never consults `priceGuard[feedId]`. The guard is stored but dead code. Any price, no matter how far outside the configured bounds, is returned verbatim to the `PriceProvider` and from there to the pool.

---

### Finding Description

`OracleBase` (providers) defines and stores a `PriceGuard` per feed:

```solidity
mapping(bytes32 => PriceGuard) public priceGuard;

function setPriceGuard(bytes32 feedId, uint128 minPrice, uint128 maxPrice)
    external checkRole(feedId)
{
    require(minPrice < maxPrice);
    priceGuard[feedId] = PriceGuard({min: minPrice, max: maxPrice});
    emit PriceGuardUpdated(feedId, minPrice, maxPrice);
}
``` [1](#0-0) [2](#0-1) 

The protocol documentation explicitly states the intent: *"If the oracle returns a price outside `[min, max]`, the system treats the feed as invalid/stalled."*

The entire on-chain read path for pools flows through `price(feedId, pool)` → `_readPrice(feedId)`:

```solidity
function _readPrice(bytes32 feedId)
    internal
    view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = _oracleDataRaw(feedId);
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
``` [3](#0-2) 

`_readPrice` is **not marked `virtual`**, so `PythOracle` and `ChainlinkOracle` cannot override it. It reads only `_oracleDataRaw(feedId)` and returns the raw stored price with zero reference to `priceGuard[feedId]`. The guard is never consulted at push time either — neither the Pyth fallback nor `ChainlinkOracle.updateReport` applies the guard before writing to `oracleData`.

The `price()` entry point that pools call:

```solidity
function price(bytes32 feedId, address pool)
    external feedExists(feedId) notBlacklisted
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    require(pool != address(0) && IPool(pool).inSwap() == msg.sender, InvalidInSwap());
    require(!blacklisted[pool], Blacklisted(pool));
    require(registeredPool[feedId][pool], NotRegistered(feedId, pool));

    (mid, spread, spread1, refTime) = _readPrice(feedId);   // ← PriceGuard never checked
    emit PriceRead(pool, feedId);
}
``` [4](#0-3) 

The `PriceProvider` receives the raw `mid` value and applies spread/margin math on top of it, then scales to Q64 for the pool. No downstream component re-checks the guard. [3](#0-2) 

---

### Impact Explanation

**High.** When a Pyth Lazer or Chainlink Data Streams report carries an extreme price (oracle bug, data-source manipulation, or a flash-crash spike that passes provider-level signature verification), the `PriceGuard` is the last on-chain line of defence. Because it is never applied, the extreme price propagates through `PriceProvider` → `getBidAndAskPrice()` → pool swap math. The pool will execute swaps at the corrupted bid/ask, causing:

- Traders to receive more output tokens than the true market price permits (swap conservation failure).
- LPs to suffer direct principal loss as the pool's reserves are drained at the wrong price.
- Pool insolvency if the price deviation is large enough that LP claims exceed remaining balances.

The `AnchoredPriceProvider` partially mitigates this for pools that use it (it clamps to a reference band), but pools using `PriceProvider`, `PriceProviderL2`, `ProtectedPriceProvider`, or `ProtectedPriceProviderL2` directly have no such backstop.

---

### Likelihood Explanation

**Low–Medium.** The trigger requires the upstream oracle (Pyth/Chainlink) to emit a price outside the admin-configured `[min, max]` window. This is rare under normal conditions but is a known failure mode: oracle data-source outages, stale aggregator fallbacks, and cross-chain bridge delays have historically produced extreme spikes in production. The `PriceGuard` was explicitly designed for exactly this scenario; its non-enforcement means the risk is permanently present for every pool using the provider oracles.

---

### Recommendation

Apply the `PriceGuard` inside `_readPrice`. Mark the function `virtual` so child contracts can extend it, and add the bounds check before returning:

```solidity
function _readPrice(bytes32 feedId)
    internal
    view
    virtual
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = _oracleDataRaw(feedId);
    PriceGuard memory guard = priceGuard[feedId];
    if (guard.min != 0 || guard.max != 0) {
        require(
            data.price >= guard.min && data.price <= guard.max,
            PriceOutOfBounds(feedId, data.price)
        );
    }
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
```

Alternatively, enforce the guard at push time (in `_store` / the Pyth/Chainlink write path) so that out-of-range data is never written to `oracleData` in the first place.

---

### Proof of Concept

1. Admin calls `setPriceGuard(feedId, 1_000e8, 100_000e8)` to bound a BTC/USD feed between $1,000 and $100,000.
2. A Pyth Lazer payload arrives carrying `price = 1e8` ($1 — a flash-crash spike that passes Pyth's own signature verification).
3. `PythOracle.fallback()` stores `oracleData[feedId].price = 1e8` without consulting `priceGuard`.
4. A pool swap calls `PriceProvider.getBidAndAskPrice()` → `oracle.price(feedId, pool)` → `_readPrice(feedId)`.
5. `_readPrice` returns `mid = 1e8` — no guard check occurs.
6. `PriceProvider` computes bid/ask around $1 and returns Q64-scaled values to the pool.
7. The pool executes the swap at $1/BTC, allowing a trader to drain BTC reserves at 1/100,000th of fair value, causing direct LP principal loss. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
