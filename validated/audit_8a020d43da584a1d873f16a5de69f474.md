### Title
`revokePusher` is ineffective within the deadline window: creator can replay the original `allowPushers` signature to silently re-establish delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` verifies a pusher-signed EIP-191 consent over `(chainid, oracle, deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = creator`. No nonce or used-signature bitmap is maintained. `revokePusher` only zeroes the mapping entry; it does not invalidate the original signature. As long as the original deadline has not expired, the creator can call `allowPushers` again with the identical `(deadline, sig)` tuple and the mapping is unconditionally restored.

### Finding Description
`allowPushers` in `CompressedOracleV1` performs three checks before writing the delegation:

1. `_ensureDeadline(deadline)` — passes if `block.timestamp <= deadline` [1](#0-0) 
2. ECDSA recovery over `(chainid, oracle, deadline, pusher, creator)` — the same signature is still cryptographically valid [2](#0-1) 
3. `pusher != msg.sender` — trivially satisfied [3](#0-2) 

None of these checks consult whether the pusher has previously revoked. `revokePusher` only zeroes `namespaceRemapping[msg.sender]`; it does not burn the signature or record a revocation epoch. [4](#0-3) 

The code comment at lines 186–191 explicitly states the deadline is the mechanism that prevents post-revocation replay, but the deadline only closes the window after it expires — not during it. [5](#0-4) 

### Impact Explanation
After the creator replays `allowPushers`, the pusher's subsequent oracle pushes are silently redirected into the creator's namespace without the pusher's fresh consent. [6](#0-5) 

If the pusher, believing they have revoked, then pushes data intended for their own namespace (e.g., a different asset pair or a stale/zeroed price), that data lands in the creator's namespace and is consumed by any pool whose oracle feed is keyed to `feedIdOf(creator, slot, pos)`. This constitutes bad-price execution reaching a live pool swap.

### Likelihood Explanation
The replay requires only that the creator retain the original calldata (deadline + signature) and submit it before the deadline expires. No privileged role, no off-chain oracle compromise, and no special token behavior is needed. The creator is a permissionless actor who already called `allowPushers` once. Deadlines are typically set hours to days in the future, giving a wide replay window.

### Recommendation
Track a per-pusher revocation nonce or a `revokedAt[pusher]` timestamp. In `allowPushers`, require that the signature was produced after the last revocation (e.g., embed a nonce in the signed message and increment it on `revokePusher`). Alternatively, maintain a `usedSignatures` bitmap keyed on the signature hash so each consent can only be consumed once.

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```

```text
    function revokePusher() external {
```
