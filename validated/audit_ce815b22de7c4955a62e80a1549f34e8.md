### Title
Missing L2 Sequencer Uptime Check Allows Stale Prices to Reach Pool Swaps After Sequencer Restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

Both L2 price providers (`PriceProviderL2` and `ProtectedPriceProviderL2`) are explicitly deployed on L2 (they carry `FUTURE_TOLERANCE` to handle sequencer clock skew) but contain no Chainlink sequencer uptime feed check. After an Arbitrum sequencer outage shorter than `MAX_TIME_DELTA`, the last pre-outage oracle price passes the staleness guard and is consumed by pool swaps at a price that no longer reflects the market.

---

### Finding Description

`PriceProviderL2._getBidAndAskPrice()` and `ProtectedPriceProviderL2._computeBidAsk()` both apply only a time-delta staleness guard:

```solidity
// PriceProviderL2.sol L215
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [1](#0-0) 

```solidity
// ProtectedPriceProviderL2.sol L207
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [2](#0-1) 

`_isStale` only checks whether `(nowTs - refTime) > MAX_TIME_DELTA`: [3](#0-2) 

Neither contract queries a Chainlink L2 sequencer uptime feed (e.g., `0x4da69F028a5790fA2...` on Arbitrum) nor enforces a post-restart grace period. The `FUTURE_TOLERANCE` immutable addresses only sequencer *clock skew* (refTime slightly ahead of block.timestamp), not sequencer *liveness*: [4](#0-3) 

The providers are deployed permissionlessly via `PriceProviderFactoryL2.createPriceProvider()`, which accepts any `_maxTimeDelta` up to 7 days: [5](#0-4) 

---

### Impact Explanation

**Bad-price execution — stale bid/ask quote reaches a pool swap.**

Scenario: Arbitrum sequencer goes down for *T* minutes where *T* < `MAX_TIME_DELTA`. The last Chainlink price published before the outage has `refTime = T_outage_start`. When the sequencer restarts, `nowTs - refTime = T < MAX_TIME_DELTA`, so `_isStale` returns `false`. The pre-outage price (which may be significantly off from the true market price after *T* minutes of volatility) is returned as a valid bid/ask and consumed by the pool swap. A trader who observes the price discrepancy can extract value from LPs by trading against the stale quote.

This matches the allowed impact gate: **bad-price execution — stale bid/ask quote reaches a pool swap**, and **direct loss of LP assets**.

---

### Likelihood Explanation

- Arbitrum sequencer outages have occurred historically (documented incidents in 2022 and 2023).
- The trigger is unprivileged: any user can call `getBidAndAskPrice()` through a normal pool swap during the vulnerable window after sequencer restart.
- The window of vulnerability is `MAX_TIME_DELTA` seconds after the sequencer restarts, which can be up to 7 days by construction.
- No special setup or malicious oracle data is required — the oracle is publishing correct data; the problem is the provider consuming pre-outage data as if it were current.

---

### Recommendation

Add a Chainlink sequencer uptime feed check with a post-restart grace period (Chainlink recommends ≥ 1 hour) inside `_getBidAndAskPrice()` / `_computeBidAsk()` of both L2 providers, before the staleness check:

```solidity
// Example pattern (Chainlink L2 sequencer uptime check)
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();                        // 0 = up, 1 = down
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

The sequencer uptime feed address should be an immutable set at construction time (alongside `offchainOracle`), and `GRACE_PERIOD` should be at least 3600 seconds.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 3600` (1 hour).
2. At `t=0`, Chainlink publishes price `P0 = $2000` for ETH/USD; `refTime = t`.
3. At `t=600`, Arbitrum sequencer goes offline. No new Chainlink updates are posted on L2.
4. At `t=2400` (40 minutes later), sequencer restarts. The last on-chain price is still `P0` with `refTime = t`.
5. True market price is now `P1 = $1800` (10% drop during outage).
6. Attacker calls pool swap. `PriceProviderL2._getBidAndAskPrice()` is invoked:
   - `_isStale(t, t+2400, 3600, futureTol)` → `2400 - 0 = 2400 < 3600` → **not stale** → passes.
   - `mid = P0 = $2000` passes guard checks.
   - Returns bid/ask around `$2000`.
7. Attacker buys ETH from the pool at the stale `$2000` price, immediately sells at true market `$1800` — net loss to LPs of ~10% per unit traded, bounded only by pool liquidity.

    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```
