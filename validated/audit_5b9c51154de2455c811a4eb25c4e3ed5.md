### Title
Pusher delegation signature replay bypasses `revokePusher` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but tracks no per-signature nonce or used-signature bitmap. A creator who holds a valid, unexpired signature can call `allowPushers` repeatedly with the same bytes, re-establishing a delegation that the pusher already cancelled via `revokePusher`. The pusher's subsequent pushes are silently redirected to the creator's namespace, starving the pusher's own feeds of updates and delivering stale prices to any pool that consumes them.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only replay barrier is `_ensureDeadline`, which rejects calls made **after** `deadline`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

There is no nonce, no used-signature set, and no check that `namespaceRemapping[pusher]` is currently `address(0)` before writing. The code's own comment acknowledges the risk but incorrectly treats the deadline as a complete fix:

> *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it"* [3](#0-2) 

The deadline prevents replay **after** expiry, but does nothing while the deadline is still live. A pusher who signs with `deadline = now + 365 days` and then calls `revokePusher` can have their revocation overwritten by the creator at any point during that year using the identical signature bytes.

`revokePusher` sets `namespaceRemapping[pusher] = address(0)`: [4](#0-3) 

A subsequent `allowPushers` call with the same signature overwrites that zero back to `creator`, with no on-chain record that the signature was already consumed.

The `fallback` push path reads `namespaceRemapping[msg.sender]` to decide which namespace receives the slot write: [5](#0-4) 

Once the creator replays the signature, every subsequent push from the pusher's key lands in the creator's namespace instead of the pusher's own namespace.

---

### Impact Explanation

- The pusher's own feeds (identified by `feedIdOf(pusher, slotIndex, positionIndex)`) stop receiving updates; their stored `timestampMs` freezes.
- Any `AnchoredPriceProvider` or pool configured to read those feeds will observe a timestamp that grows increasingly stale, eventually breaching `maxTimeDrift` and reverting, or — if the consumer does not enforce drift — consuming a frozen price as live.
- This maps directly to the allowed impact: **bad-price execution — stale bid/ask quote reaches a pool swap**.

---

### Likelihood Explanation

- The creator legitimately receives the signature as part of the normal delegation flow; no theft or front-running is required.
- The creator only needs to retain the calldata from the original `allowPushers` transaction (publicly visible on-chain) and replay it before `deadline` expires.
- Pushers routinely sign long-lived deadlines to avoid repeated signing ceremonies, making the replay window wide.
- The pusher has no on-chain way to invalidate the signature short of waiting for the deadline to expire.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedSignatures` and revert if the digest has already been seen:

```solidity
bytes32 digest = keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender));
require(!_usedSignatures[digest], SignatureAlreadyUsed());
_usedSignatures[digest] = true;
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on every successful `allowPushers` call, so that `revokePusher` implicitly invalidates all prior signatures by making the nonce stale.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 1-year deadline
uint256 deadline = block.timestamp + 365 days;
bytes32 digest = keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator));
bytes memory sig = sign(pusherKey, MessageHashUtils.toEthSignedMessageHash(digest));

// 2. Creator establishes delegation
address[] memory pushers = new address[](1); pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1); sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — revocation is overwritten
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);  // succeeds, no revert
assertEq(oracle.namespaceRemapping(pusher), creator); // pusher's revoke is gone

// 5. Pusher's subsequent pushes now land in creator's namespace;
//    pusher's own feeds freeze → stale prices reach any pool reading them.
```
