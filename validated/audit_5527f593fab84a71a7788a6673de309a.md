### Title
Permissionless `register` unconditionally clears admin blacklist, allowing any blacklisted pool to bypass oracle abuse protection — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
The `register` function in `providers/OracleBase.sol` conflates two concerns — pool-feed registration and blacklist clearing — in a single permissionless, payable call. Because `register` is callable by anyone (including the blacklisted pool itself) for as little as 1 wei, any pool the admin has blacklisted can immediately un-blacklist itself, bypassing the oracle's abuse-protection layer and regaining the ability to read live prices during swaps.

### Finding Description
`OracleBase.register` is the permissionless paid path that whitelists a pool for a specific feed:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);          // default: 1 wei
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;                       // ← unconditional clear
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    ...
}
``` [4](#0-3) 

The admin blacklists a pool via `setBlacklist(pool, true)` to cut off oracle access for an abusive or compromised pool. The `price` read path enforces this at call time:

```solidity
require(!blacklisted[pool], Blacklisted(pool));
require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
``` [5](#0-4) 

However, `register` clears `blacklisted[pool]` as a side effect of registration. Since `register` has no role check and the default `registrationFee` is 1 wei, the blacklisted pool itself (or any third party) can call `register(feedId, pool, approvedFactory)` with 1 wei to atomically clear the blacklist and re-register for the feed. The admin's safety decision is immediately undone.

The M-23 structural parallel is exact: in M-23, the fee collector's balance is shared with returnable fees — a valid withdrawal drains the shared pool and breaks a downstream invariant. Here, the blacklist state is shared with the registration path — a valid registration call clears the shared safety flag and breaks the admin-boundary invariant.

### Impact Explanation
Any pool blacklisted by the admin for oracle abuse, price manipulation, or compromise can un-blacklist itself for 1 wei and immediately resume reading live oracle prices during swaps. The admin's only recourse is to re-blacklist the pool, which the pool can again clear with another 1-wei `register` call — making the blacklist permanently ineffective against a motivated pool. This breaks the "Admin-boundary break" impact gate: an admin-controlled safety mechanism is bypassed by an unprivileged path.

### Likelihood Explanation
Likelihood is high. The trigger requires no special privileges: the blacklisted pool itself calls `register` with 1 wei and an approved factory address (both publicly known). The factory and `isPool` checks only confirm the pool is a real Metric pool — they do not prevent a blacklisted pool from registering. The attack is a single transaction.

### Recommendation
Separate blacklist clearing from registration. Two options:

1. **Remove the blacklist-clearing side effect from `register`**: a blacklisted pool should not be able to register at all — add `require(!blacklisted[pool])` before the registration write.
2. **Reserve blacklist clearing exclusively for `ADMIN_ROLE`**: remove the `if (blacklisted[pool]) { blacklisted[pool] = false; }` block from `register` entirely, and require the admin to explicitly call `setBlacklist(pool, false)` before a pool can re-register.

### Proof of Concept
```
1. Admin calls setBlacklist(poolA, true)
   → blacklisted[poolA] == true
   → poolA.price(feedId, poolA) reverts with Blacklisted(poolA)

2. poolA (or anyone) calls register(feedId, poolA, approvedFactory) with 1 wei
   → approvedFactories.contains(approvedFactory) == true  ✓
   → IPoolFactory(approvedFactory).isPool(poolA) == true  ✓
   → blacklisted[poolA] is cleared to false
   → registeredPool[feedId][poolA] = true

3. poolA calls price(feedId, poolA) during a swap
   → notBlacklisted: msg.sender (provider) not blacklisted ✓
   → !blacklisted[poolA] == true ✓  (blacklist bypassed)
   → registeredPool[feedId][poolA] == true ✓
   → live oracle price returned to the previously-blacklisted pool
```
