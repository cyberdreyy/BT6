### Title
Creator Can Replay `allowPushers` Signature Within Deadline Window to Nullify Pusher's `revokePusher()` Call — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The EIP-191 consent signature verified in `allowPushers` does not commit to any "action type" or per-delegation nonce. After a pusher calls `revokePusher()` to clear their namespace mapping, the creator can immediately replay the original, still-valid signature to re-establish the delegation — permanently blocking the pusher's ability to self-revoke within the deadline window.

---

### Finding Description

`allowPushers` verifies a pusher-signed EIP-191 message whose preimage is:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The hash commits to: chain id, oracle address, deadline, pusher address, and creator address. It does **not** commit to:
- a per-delegation nonce,
- a revocation counter, or
- any "action type" flag distinguishing "grant delegation" from "re-grant after revoke."

`revokePusher()` clears the mapping with a single storage write:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

Because the signature is still valid (the deadline has not expired), the creator can call `allowPushers` again in the very next transaction — or even in the same block — with the identical `deadline`, `pushers`, and `signatures` arrays. The `_ensureDeadline` check passes, ECDSA recovery succeeds, and `namespaceRemapping[pusher]` is set back to `msg.sender`. The pusher's revocation is silently undone.

The code's own NatSpec comment acknowledges the replay concern but misidentifies the deadline as the complete fix:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The deadline prevents replay **after** it expires. It does nothing to prevent replay **before** it expires, which is the entire window during which the pusher is trying to revoke.

The slot-structure documentation repeats the same incomplete reasoning:

> "Delegation (`allowPushers`) requires each pusher's EIP-191 signature (and a deadline: the signed consent has no data timestamp, so an undated signature could re-establish a delegation after the pusher revoked it)." [4](#0-3) 

This is the direct analog of the Ambire `[C-01]` bug: just as the `isCancellation` flag was absent from the Ambire recovery hash (allowing a RECOVER signature to be replayed as CANCEL), the "action type / revocation epoch" is absent from the Metric delegation hash, allowing an ALLOW signature to be replayed to undo a REVOKE.

---

### Impact Explanation

Once a pusher calls `revokePusher()`, they expect their fallback pushes to land in their **own** namespace again. If the creator immediately replays the consent signature, the pusher's pushes continue to land in the creator's namespace. The pusher's own pools — which read from `feedIdOf(pusher, slotId, positionIndex)` — receive no new price data and will eventually serve stale quotes to swaps. A creator can automate this replay (one cheap transaction per block) to permanently prevent the pusher from ever reclaiming their own namespace for the lifetime of the signed deadline.

This satisfies the "broken core pool functionality" criterion: pools backed by the pusher's own namespace become unable to obtain fresh oracle prices, which can cause swaps to execute against stale bid/ask quotes or revert entirely.

---

### Likelihood Explanation

- **No preconditions beyond holding the original signature.** The creator already possesses it — they submitted it in the original `allowPushers` call.
- **Fully unprivileged replay.** `allowPushers` is a public function with no access control beyond signature verification.
- **Automatable.** A simple script watching for `PusherRevoked` events can replay the delegation in the same block, making revocation practically impossible for the pusher.
- **Deadline window can be long.** Nothing in the protocol caps the deadline value; a pusher who signed a 30-day consent cannot revoke for 30 days.

---

### Recommendation

Include a per-pusher revocation nonce in the signed hash so that each delegation grant is cryptographically distinct from any re-grant after revocation:

```solidity
mapping(address pusher => uint256 nonce) public delegationNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid,
        address(this),
        deadline,
        pusher,
        msg.sender,
        delegationNonce[pusher]   // <-- add this
    ))
);
// ...
namespaceRemapping[pusher] = msg.sender;

// In revokePusher / removePushers, after clearing the mapping:
delegationNonce[msg.sender]++;   // invalidates all prior consent signatures
```

This mirrors the standard EIP-2612 / EIP-4494 pattern: incrementing the nonce on revocation makes every previously issued consent signature immediately invalid, so the creator cannot replay it.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import "forge-std/Test.sol";
import {CompressedOracleV1} from
    "smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract ReplayDelegationPoC is Test {
    CompressedOracleV1 oracle;

    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator = address(0xC0FFEE);

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 0);
        pusher = vm.addr(PUSHER_KEY);
        vm.warp(1_700_000_000);
    }

    function testRevokeIsNullifiedByReplay() public {
        uint256 deadline = block.timestamp + 30 days;

        // 1. Pusher signs consent once.
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // 2. Creator establishes delegation.
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // 3. Pusher revokes.
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

        // 4. Creator replays the SAME signature — no new consent from pusher needed.
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // succeeds!

        // 5. Delegation is re-established; pusher's revocation is nullified.
        assertEq(oracle.namespaceRemapping(pusher), creator,
            "revocation nullified: pusher still mapped to creator");

        // 6. Pusher's fallback pushes still land in creator's namespace, not their own.
        uint56 tsMs = uint56(block.timestamp * 1000);
        uint48 raw = (uint48(500_000) << 16) | (uint48(3) << 8) | uint48(1);
        vm.prank(pusher);
        (bool ok,) = address(oracle).call(abi.encodePacked(
            (uint256(tsMs) << 8) | uint256(0) | (uint256(raw) << 208)
        ));
        assertTrue(ok);
        // Price lands in creator namespace, not pusher's own — revocation had no effect.
    }
}
```
