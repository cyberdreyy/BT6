### Title
Signature Replay in `allowPushers` Lets Creator Re-Establish a Self-Revoked Pusher Delegation Within the Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but records no nonce and maintains no used-signature set. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature — with the identical `(deadline, pusher, creator)` tuple — to silently restore `namespaceRemapping[pusher] = creator`, defeating the pusher's self-revocation for the entire remaining deadline window.

---

### Finding Description

`allowPushers` signs over `(block.chainid, address(this), deadline, pusher, msg.sender)` and enforces only a deadline check:

```solidity
// CompressedOracle.sol lines 192-211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);          // only: block.timestamp <= deadline
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // re-writable with same sig
}
``` [1](#0-0) 

`_ensureDeadline` is a single timestamp comparison with no state mutation:

```solidity
// OracleBase.sol line 124-126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

`revokePusher` zeroes the mapping but stores nothing that would invalidate the old signature:

```solidity
// CompressedOracle.sol lines 238-243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

There is **no nonce, no used-signature bitmap, and no per-pusher revocation timestamp** anywhere in `CompressedOracleV1` or `OracleBase`. The grep across all production contracts returns zero matches for any such guard. The code comment at line 186-191 explicitly states the deadline exists to prevent exactly this replay, but the deadline only blocks replay *after* it expires — it does nothing within the window. [4](#0-3) 

The `fallback()` push path reads `namespaceRemapping[msg.sender]` at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So restoring the mapping immediately redirects all subsequent pushes from the pusher back into the creator's namespace, regardless of the pusher's intent.

---

### Impact Explanation

After the pusher revokes and begins pushing data intended for their own namespace (or a new creator's namespace), the replayed delegation silently routes that data into the original creator's namespace. The creator's oracle feeds now contain prices for a different asset pair or a different price level than intended. Every pool that consumes those `feedId`s via `price(feedId, pool)` receives corrupted mid/spread values. This satisfies the **bad-price execution** impact gate: a trader can swap against the pool at a price the oracle curve does not permit, causing the pool to receive less input than owed or pay out more output than it should, resulting in direct loss of LP principal.

---

### Likelihood Explanation

- The creator already holds the original signature bytes (they submitted them in step 1).
- The replay requires a single `allowPushers` call before the deadline — no privileged access, no special tooling.
- Deadlines are typically set hours to days in the future (the test suite uses `block.timestamp + 1 days`), giving a wide replay window.
- The pusher has no on-chain way to invalidate the signature short of waiting for the deadline to expire. [6](#0-5) 

---

### Recommendation

Add a per-pusher revocation nonce or a used-signature set. The minimal fix is a `mapping(address => uint256) public pusherNonce` incremented by `revokePusher()`, with the nonce included in the signed message:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusherNonce[pusher], pusher, msg.sender))
```

Alternatively, record the revocation timestamp and require `deadline < revocationTimestamp[pusher]` to be false, or simply store used signature hashes in a `mapping(bytes32 => bool) private _usedSigs`.

### Proof of Concept

```solidity
// Foundry unit test (pseudo-code)
function test_replayAfterRevoke() public {
    uint256 deadline = block.timestamp + 1 days;

    // 1. Creator calls allowPushers with pusher's consent sig
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 2. Pusher self-revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 3. Creator replays the SAME sig before deadline
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds — no revert

    // 4. Invariant violated: pusher is re-delegated without a new signature
    assertEq(oracle.namespaceRemapping(pusher), creator);
}
```

Step 3 succeeds because `block.timestamp < deadline` and the contract has no record that this signature was already consumed. The pusher's subsequent pushes now land in the creator's namespace, corrupting oracle feeds consumed by pools.
