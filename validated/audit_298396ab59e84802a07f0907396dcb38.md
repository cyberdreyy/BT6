### Title
Synthetic-Ratio `AnchoredPriceProvider` Combines Two Feeds Without Cross-Validating Their Timestamps, Enabling Bad-Price Execution - (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider._getBidAndAskPrice()` in synthetic-ratio mode (when `quoteFeedId != bytes32(0)`) reads two independent oracle feeds and divides their mid-prices to produce a cross-rate (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). Each leg's `refTime` is validated individually against `block.timestamp` within `_readLeg()`, but the two legs' timestamps are **never compared against each other**. The `refTime` return value is explicitly discarded at both call sites. With `MAX_REF_STALENESS` configurable up to 7 days, the two legs can reflect prices from arbitrarily different moments in time, producing a synthetic ratio that is materially wrong during volatile periods and feeding that bad price directly into pool swaps.

---

### Finding Description

In `_getBidAndAskPrice()`:

```solidity
(uint256 mid,  uint256 spreadBps,  , bool ok)  = _readLeg(baseFeedId);   // refTime discarded
...
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);        // refTime discarded
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
spreadBps += spreadBps2;
``` [1](#0-0) 

`_readLeg` returns `(mid, spreadBps, refTime, ok)` and validates `refTime` only against `block.timestamp`:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (..., false);
``` [2](#0-1) 

The third tuple slot (`refTime`) is silently dropped at both call sites in `_getBidAndAskPrice()` (the bare `,` in the destructuring). No code ever computes `|refTime1 - refTime2|` or enforces a maximum skew between the two legs.

`MAX_REF_STALENESS` is an immutable set at construction, bounded only by 7 days:

```solidity
if (_maxRefStaleness > 7 days) revert MaxRefStalenessOutOfBounds();
MAX_REF_STALENESS = _maxRefStaleness;
``` [3](#0-2) 

The `AnchoredProviderFactory.createAnchoredProvider()` is permissionless — any caller may deploy a synthetic-ratio provider with any `quoteFeedId` and any `maxRefStaleness` within the envelope: [4](#0-3) 

The envelope only bounds `maxRefStaleness` to `[stalenessMin, stalenessMax]` (both admin-set, with `stalenessMax` up to 7 days). There is no envelope field for inter-leg skew. [5](#0-4) 

The combined `spreadBps` is widened by adding both legs' uncertainties, but this widening reflects the oracle's own reported confidence interval — not the temporal divergence between the two feeds. The `MAX_SPREAD_BPS` circuit breaker is a static threshold set at construction; it does not adapt to inter-leg skew. [6](#0-5) 

---

### Impact Explanation

The synthetic ratio `mid1 / mid2` is computed from prices that may reflect market state at arbitrarily different times (up to `MAX_REF_STALENESS` apart). During a volatile period — e.g., ETH moves 5% while BTC is flat — the stale ETH/USD leg produces a BTC/ETH ratio that is 5% wrong. This bad ratio is passed directly to `_computeBidAsk` and then to the pool's `getSellAndBuyPrices()` / `swap()` path:

- **Overestimated ratio**: traders buying the base token receive it at a price below fair value — swap conservation failure; the pool receives less input than it should.
- **Underestimated ratio**: traders selling the base token receive less output than fair value — economic loss to the trader, but the pool is not insolvent.
- The band clamp (`refBid`/`refAsk`) is derived from the same bad mid, so it does not correct the error — it only enforces the spread shape around the wrong center.

The pool's `getSellAndBuyPrices()` is the direct consumer: [7](#0-6) 

---

### Likelihood Explanation

- **Trigger**: normal market operation. Any swap through a pool whose price provider is a synthetic-ratio `AnchoredPriceProvider` is affected whenever the two underlying feeds have different update cadences or one feed is temporarily delayed.
- **No privileged access required**: `createAnchoredProvider` is permissionless; any user can deploy a synthetic provider and attach it to a pool.
- **Frequency**: feed update-rate mismatches are routine (Chainlink heartbeats differ per feed; Pyth Lazer push cadence varies). The window of exposure equals `MAX_REF_STALENESS`, which can be hours or days.

---

### Recommendation

After both legs pass their individual staleness checks, enforce a maximum inter-leg skew:

```solidity
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid,  uint256 spreadBps,  uint256 refTime1, bool ok)  = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, uint256 refTime2, bool ok2) = _readLeg(_quote);
        if (!ok2 || mid2 == 0) return (0, type(uint128).max);

        // NEW: reject if the two legs' prices are from materially different times.
        uint256 skew = refTime1 > refTime2 ? refTime1 - refTime2 : refTime2 - refTime1;
        if (skew > MAX_LEG_SKEW) return (0, type(uint128).max);

        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
        spreadBps += spreadBps2;
    }
    return _computeBidAsk(mid, spreadBps);
}
```

`MAX_LEG_SKEW` should be an immutable set at construction (e.g., bounded to a few minutes for liquid pairs) and validated by the factory envelope alongside `maxRefStaleness`.

### Proof of Concept

1. Admin deploys a synthetic-ratio `AnchoredPriceProvider` with `baseFeedId = BTC/USD`, `quoteFeedId = ETH/USD`, `MAX_REF_STALENESS = 1 hour`.
2. At `T=0`: BTC/USD = 65,000, ETH/USD = 3,000 → ratio = 21.67. Both feeds fresh.
3. At `T=3500s` (just under 1 hour): ETH/USD feed stalls. Last ETH/USD price = 3,000 (from T=0). BTC/USD continues updating: BTC/USD = 65,000 (unchanged).
4. At `T=3599s`: ETH drops 10% in real market to 2,700, but the stale feed still reports 3,000. BTC/USD updates to 65,000.
5. `_readLeg(BTC/USD)` → `refTime = 3599`, fresh. `_readLeg(ETH/USD)` → `refTime = 0`, age = 3599s < 3600s = `MAX_REF_STALENESS`, passes.
6. Synthetic ratio = 65,000 / 3,000 = 21.67 (stale). True ratio = 65,000 / 2,700 = 24.07.
7. Pool quotes a bid/ask band centered on 21.67 instead of 24.07 — a ~10% underpricing of the base token.
8. A trader buys BTC (base) with ETH (quote) at the underpriced ask, receiving BTC at a ~10% discount relative to the true market rate. The pool receives less ETH than it should — swap conservation failure. [1](#0-0) [2](#0-1)

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }

    function _computeBidAsk(uint256 mid, uint256 spreadBps)
        internal view returns (uint128, uint128)
    {
        // Circuit breaker: extreme (combined) uncertainty means the feed is clearly broken.
        if (spreadBps > MAX_SPREAD_BPS) {
            return (0, type(uint128).max);
        }

        // Reference band: mid ± (spreadBps + minMargin), bid rounded down, ask rounded up.
