### Title
`CompressedOracleV1` drift-window timestamps accepted by oracle but permanently rejected as stale by non-L2 price providers, causing pool stalling — (`smart-contracts-poc/contracts/oracles/utils/TimeMs.sol`, `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`, `smart-contracts-poc/contracts/PriceProvider.sol`)

---

### Summary

`CompressedOracleV1` runs a **millisecond-based time system with a future-drift gate** for pushes, while `PriceProvider`, `AnchoredPriceProvider`, and `ProtectedPriceProvider` run a **second-based, zero-future-tolerance staleness check**. When `MAX_TIME_DRIFT > 0`, a legitimately pushed price whose timestamp falls inside the drift window is accepted and stored by the oracle, but is unconditionally rejected as stale by every non-L2 price provider. The pool then returns the sentinel `(0, type(uint128).max)` and reverts every swap with `FeedStalled` until the blockchain's `block.timestamp` overtakes the stored timestamp — a window of up to `MAX_TIME_DRIFT` seconds.

---

### Finding Description

**Oracle push gate — milliseconds with drift tolerance**

The `fallback()` and `updateBySignature()` push paths both call `revertIfAfterBlockTimeWithDrift`:

```solidity
// TimeMs.sol
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}

function toSeconds(TimeMs t) pure returns (uint56) {
    return TimeMs.unwrap(t) / 1000;   // integer division, truncates
}
``` [1](#0-0) 

So a slot word carrying timestamp `(block.timestamp + Δ) * 1000` ms (where `Δ ≤ MAX_TIME_DRIFT`) passes the gate and is written to storage.

**Oracle read path — converts back to seconds**

`CompressedOracleV1._price()` returns `refTime` in seconds:

```solidity
return (uint256(data.price), uint256(data.spread0), data.spread1,
        data.timestampMs.toSeconds());   // ÷ 1000
``` [2](#0-1) 

For the stored timestamp `(block.timestamp + Δ) * 1000` ms, `toSeconds()` returns `block.timestamp + Δ`.

**Price provider staleness check — zero future tolerance**

Every non-L2 provider (`PriceProvider`, `ProtectedPriceProvider`, `AnchoredPriceProvider`) uses:

```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // ← any future refTime is stale
    return (nowTs - refTime) > maxDelta;
}
``` [3](#0-2) [4](#0-3) [5](#0-4) 

`refTime = block.timestamp + Δ > block.timestamp` → `_isStale` returns `true` → provider returns `(0, type(uint128).max)` → `getBidAndAskPrice()` reverts with `FeedStalled`.

**The parallel time system**

| Layer | Time unit | Future tolerance |
|---|---|---|
| `CompressedOracleV1` push gate | milliseconds | `MAX_TIME_DRIFT` seconds |
| Non-L2 price provider staleness | seconds | **zero** |

The oracle and the price provider speak different time dialects. The drift window that the oracle deliberately opens is invisible to the price provider.

### Impact Explanation

When `MAX_TIME_DRIFT > 0` and the off-chain pusher's clock is ahead of `block.timestamp` by any amount `0 < Δ ≤ MAX_TIME_DRIFT`:

1. The push is accepted and stored (oracle side is satisfied).
2. Every subsequent `getBidAndAskPrice()` call returns the sentinel and reverts with `FeedStalled`.
3. All swaps against the pool are blocked for up to `MAX_TIME_DRIFT` seconds — until `block.timestamp` overtakes the stored timestamp.
4. During the stall window, LPs cannot execute swaps and traders cannot trade, constituting broken core pool functionality / unusable swap flows.

The pool cannot self-heal: the pusher cannot overwrite the stored future timestamp with a non-future one until `block.timestamp` reaches the stored value (the monotonicity guard rejects any equal-or-older timestamp). [7](#0-6) 

---

### Likelihood Explanation

The drift parameter exists precisely to handle off-chain clock skew. Any production deployment with `MAX_TIME_DRIFT > 0` paired with a non-L2 price provider (`PriceProvider`, `AnchoredPriceProvider`, `ProtectedPriceProvider`) is affected. A pusher whose system clock is even 1 second ahead of the chain's `block.timestamp` will naturally submit a future-millisecond timestamp, triggering the stall on every push cycle. No adversarial intent is required — normal clock drift is sufficient.

---

### Recommendation

Align the two time systems by one of:

1. **Preferred**: Give non-L2 `_isStale` the same `futureTolerance` parameter already present in `PriceProviderL2._isStale`, set equal to `MAX_TIME_DRIFT` of the backing oracle.
2. **Alternative**: In `CompressedOracleV1._price()`, clamp `refTime` to `min(toSeconds(timestampMs), block.timestamp)` before returning, so the price provider never sees a future `refTime`. This sacrifices sub-second precision but eliminates the mismatch.
3. **Deployment guard**: Document and enforce that `CompressedOracleV1` with `MAX_TIME_DRIFT > 0` must only be paired with L2 price providers that carry a matching `FUTURE_TOLERANCE`.

### Proof of Concept

```
Setup:
  CompressedOracleV1 deployed with MAX_TIME_DRIFT = 60 (seconds)
  PriceProvider deployed pointing at the above oracle
  Pool deployed pointing at the above PriceProvider

Step 1 — pusher's clock is 30 s ahead of block.timestamp:
  pusher submits slot word with timestampMs = (block.timestamp + 30) * 1000

Step 2 — oracle push gate check (TimeMs.sol:28-30):
  toSeconds((block.timestamp + 30) * 1000) = block.timestamp + 30
  block.timestamp + 30 <= block.timestamp + 60  ✓  → push accepted, stored

Step 3 — pool calls getBidAndAskPrice() during swap:
  oracle._price() returns refTime = block.timestamp + 30

Step 4 — PriceProvider._isStale(block.timestamp + 30, block.timestamp, MAX_TIME_DELTA):
  refTime > nowTs  →  true  →  stale  →  returns (0, type(uint128).max)

Step 5 — getBidAndAskPrice() reverts FeedStalled
  All swaps revert for the next ~30 seconds until block.timestamp catches up.

Step 6 — pusher cannot fix it immediately:
  Any new push with timestamp ≤ stored timestamp is rejected by isAfter() monotonicity guard.
  Pool remains stalled until block.timestamp > (block.timestamp_at_push + 30).
```
