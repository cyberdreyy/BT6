### Title
Shared Per-Slot Timestamp Allows Never-Pushed Lane to Bypass Staleness Check with Zero Price — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1` packs four price lanes into a single 256-bit storage slot, with **one shared 56-bit timestamp for all four lanes**. When any lane in a slot is pushed, the shared timestamp advances for the entire slot. A sibling lane that was never pushed inherits that fresh timestamp but retains `price = 0`. The staleness guard that downstream consumers rely on — "a never-pushed position reads as `price 0 / timestamp 0`, which every consumer already rejects as stale" — is **false** once any sibling lane in the same slot has been updated. The never-pushed lane now reads as `price = 0 / timestamp = fresh`, passing staleness checks and feeding a zero price into any pool bound to that feed.

---

### Finding Description

The slot word layout is:

```
bits 255…208 : oracle[0] (48-bit lane: 32-bit price | 8-bit s0 | 8-bit s1)
bits 207…160 : oracle[1]
bits 159…112 : oracle[2]
bits 111… 64 : oracle[3]
bits  63…  8 : timestamp (uint56, unix milliseconds) ← SHARED across all 4 lanes
bits   7…  0 : slotId (cleared in storage)
``` [1](#0-0) 

`_loadSlotLayout` reads the entire 256-bit slot word in one `sload` and extracts the shared timestamp for all four lanes: [2](#0-1) 

`getOracleData` then assigns this shared timestamp to whichever lane is requested: [3](#0-2) 

For a never-pushed lane (`p = 0, s0 = 0, s1 = 0`):
- The sentinel guard `s0 == 0xff && s1 == 0xff` is **not** triggered (values are `0`, not `0xff`).
- `data.price = U64x32.decode(0) = 0`
- `data.spread0 = _decodeCodebookIndex(0)` → a valid codebook spread (not rejected)
- `data.timestampMs = _layout.timestampMs` → **the shared slot timestamp, which is fresh if any sibling lane was recently pushed**

The protocol's own documentation states:

> "A never-pushed position reads as price 0 / timestamp 0, which every consumer already rejects as stale — no seeding or creation step is needed." [4](#0-3) 

This invariant is **broken** the moment any sibling lane in the same slot is pushed. The test `testZeroSetupPushThenPrice` explicitly confirms this behavior:

> "A sibling lane in the SAME slot shares the slot's 56-bit timestamp word, but its own 48-bit price lane was never written → price 0 (the timestamp is per-slot, not per-lane)." [5](#0-4) 

The `price()` view function propagates this zero price with a fresh `refTime` directly to any caller (pool or provider): [6](#0-5) 

---

### Impact Explanation

A pool bound to `feedId(creator, slot=N, position=1)` where only `position=0` of slot `N` has ever been pushed will receive `(mid=0, spread=codebook[0], refTime=fresh)` from the oracle. The staleness check passes because `refTime` is recent. The pool then computes `bid = 0` and `ask = 0` (or near-zero), allowing a trader to buy token0 for effectively nothing, draining LP principal. This is a **bad-price execution** impact: the oracle/bin curve permits a price of 0, and the pool cannot distinguish this from a legitimately pushed zero-price feed.

---

### Likelihood Explanation

The trigger requires a semi-trusted pusher to push to any lane in the same slot as the pool's configured lane, while the pool's lane remains unpushed. This can occur:

1. **Misconfiguration**: A pool admin configures `positionIndex=1` but the off-chain pusher only ever writes to `positionIndex=0` of the same slot. After the first push to position 0, position 1 appears fresh with price=0.
2. **Deliberate attack by an authorized pusher**: A pusher authorized for the creator's namespace pushes a slot word where the pool's configured lane carries `p=0, s0=0, s1=0` (not the sentinel `0xff/0xff`) while another lane carries valid data. The shared timestamp advances, the pool's lane passes staleness, and price=0 is delivered.

The `feedIdOf` encoding embeds `slotIndex` and `positionIndex` separately, making it easy to configure a pool for one position while pushes target another in the same slot. [7](#0-6) 

---

### Recommendation

The staleness check must be **per-lane**, not per-slot. Two options:

1. **Store a per-lane timestamp**: Expand the slot layout to include a timestamp per lane (requires a multi-slot layout or reduced lane count).
2. **Reject zero price at read time**: In `getOracleData`, treat `compressed.p == 0` (outside the `0xff/0xff` sentinel path) as an unpushed/invalid lane and return `timestampMs = 0` explicitly, so downstream staleness checks correctly reject it:

```solidity
data.price = U64x32.decode(compressed.p);
if (data.price == 0) {
    // Treat as never-pushed: zero timestamp forces staleness rejection
    data.timestampMs = toTimeMs(0);
    return data;
}
data.spread0 = _decodeCodebookIndex(compressed.s0);
data.spread1 = _decodeCodebookIndex(compressed.s1);
data.timestampMs = _layout.timestampMs;
```

This preserves the "never-pushed reads as stale" invariant even after sibling lanes are updated.

### Proof of Concept

```
Setup:
  creator = 0xALICE
  oracle = CompressedOracleV1(maxTimeDrift=60s)
  pool configured with feedId = feedIdOf(0xALICE, slot=5, position=1)

Step 1: Pusher pushes to slot=5, position=0 with valid price=1_000_000, fresh timestamp.
        Slot word: [lane0=valid | lane1=0x000000 | lane2=0x000000 | lane3=0x000000 | ts=now_ms | slotId=5]
        → Shared timestamp for slot 5 advances to now_ms.

Step 2: Pool calls oracle.price(feedId(0xALICE, 5, 1), pool).
        _loadSlotLayout reads slot 5: timestamp = now_ms (fresh).
        _selectCompressedData(layout, 1) → {p=0, s0=0, s1=0}.
        Sentinel check: s0==0xff && s1==0xff → FALSE (both are 0).
        data.price = U64x32.decode(0) = 0.
        data.timestampMs = now_ms → staleness check PASSES.
        Returns: (mid=0, spread=codebook[0], refTime=now).

Step 3: Pool computes bid = 0 * (1 - spread/BPS_BASE) = 0, ask = 0.
        Trader calls swap: buys token0 for 0 token1.
        LP principal drained.
```

The invariant "never-pushed position reads as stale" is violated as soon as any sibling lane in the same slot is pushed, directly analogous to `maxWithdraw()` returning a value inflated by unreleased rewards that the actual withdrawal path cannot honor.

        (address creator, uint8 slotIndex, uint8 positionIndex) = _unpackFeedId(feedId);

        SlotLayout memory _layout = _loadSlotLayout(_oracleSlot(creator, slotIndex));

    function _loadSlotLayout(bytes32 slotIndex) internal view returns (SlotLayout memory _layout) {
        uint256 slotValue;
        assembly ("memory-safe") {
            slotValue := sload(slotIndex)
        }

        _layout.timestampMs = toTimeMs(slotValue >> 8 & X56);

        _layout.oracle0 = _decodeCompressedOracleData(uint48((slotValue >> 208) & X48));
        _layout.oracle1 = _decodeCompressedOracleData(uint48((slotValue >> 160) & X48));
        _layout.oracle2 = _decodeCompressedOracleData(uint48((slotValue >> 112) & X48));
        _layout.oracle3 = _decodeCompressedOracleData(uint48((slotValue >> 64) & X48));
    }
```
