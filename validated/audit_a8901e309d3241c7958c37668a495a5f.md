### Title
`allowPushers` Signature Has No Nonce, Allowing Creator to Replay Consent and Permanently Block Pusher's Self-Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `allowPushers` delegation signature commits to `(chainid, oracle, deadline, pusher, creator)` but contains **no nonce**. The only replay protection is the deadline. A pusher who calls `revokePusher()` to stop bad prices from flowing into the creator's namespace can have their revocation immediately undone by the creator replaying the original consent signature. For the entire deadline window — which the creator can set to any future timestamp — the pusher cannot effectively revoke their delegation.

---

### Finding Description

`allowPushers` verifies each pusher's EIP-191 consent:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce in the signed payload. The NatSpec comment explicitly acknowledges the risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The deadline is the **only** guard. But the deadline does not prevent the creator from calling `allowPushers` again with the **same** signature before the deadline expires. `revokePusher()` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

But the creator can immediately replay the original consent signature (same `deadline`, same `pusher`, same `msg.sender`) to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone. This can be repeated indefinitely until the deadline expires.

The `_ensureDeadline` check only verifies `block.timestamp <= deadline`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [4](#0-3) 

There is no consumed-signature registry, no per-pusher revocation nonce, and no flag that marks a pusher as "self-revoked."

---

### Impact Explanation

The `CompressedOracleV1` is the price source for `AnchoredPriceProvider`, which feeds bid/ask quotes directly into pool swaps:

```solidity
(mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
``` [5](#0-4) 

If a pusher's key is compromised, the attacker pushes manipulated prices into the creator's namespace. The pusher calls `revokePusher()` to stop the flow. The creator (or an automated delegation-management system) replays the original consent signature via `allowPushers`, restoring `namespaceRemapping[pusher] = creator`. The attacker's pushes resume landing in the creator's namespace. The manipulated price passes through `_readLeg` (if within the `priceGuard` range and not stale) and reaches the pool swap as a bad bid/ask quote — direct loss to traders.

The `priceGuard` is not set by default (both `min` and `max` are zero, which the provider treats as unlimited):

```solidity
guardMax = guardMax == 0 ? type(uint128).max : guardMax;
if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);
``` [6](#0-5) 

---

### Likelihood Explanation

The trigger is the creator (semi-trusted) replaying the consent signature after the pusher revokes. This is reachable by:
- A malicious creator who wants to keep a compromised pusher active.
- An automated delegation-management system that re-establishes delegations whenever `PusherRevoked` is emitted.

The pusher signed a long-lived consent (deadline can be weeks or months in the future). The creator holds the signature off-chain and can replay it at any time before expiry. No privileged role is required — `allowPushers` is a public function.

---

### Recommendation

Add a per-pusher revocation nonce to the signed message and increment it on every `revokePusher()` or `removePushers()` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
```

After revocation, the old signature (which commits to the old nonce) is permanently invalid. The creator must obtain a fresh signature from the pusher to re-establish the delegation.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline 30 days out.
bytes memory sig = sign(PUSHER_KEY,
    keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)));

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher's key is compromised; attacker pushes bad price into creator's namespace.
vm.prank(attacker_with_pusher_key);
oracle.call(badPriceWord); // lands in creator namespace

// 4. Pusher self-revokes.
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 5. Creator replays the SAME signature (deadline still valid, no nonce).
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator again ✓

// 6. Attacker resumes pushing bad prices into creator's namespace.
// 7. AnchoredPriceProvider reads the bad price and delivers it to the pool swap.
```

The pusher cannot stop step 5 — `allowPushers` is public and the signature is permanently valid until the deadline expires. Steps 4–5 can be repeated indefinitely, making `revokePusher()` a no-op for the entire deadline window.
