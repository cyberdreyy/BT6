### Title
CompressedOracle `MAX_TIME_DRIFT` and L1 PriceProvider `_isStale` misalignment causes `FeedStalled` on valid pushes — (`smart-contracts-poc/contracts/PriceProvider.sol` / `AnchoredPriceProvider.sol`)

---

### Summary

The `CompressedOracleV1` accepts pushed timestamps up to `MAX_TIME_DRIFT` seconds **in the future** (production: 60 s). The L1 `PriceProvider` and `AnchoredPriceProvider` staleness checks unconditionally treat **any** `refTime > block.timestamp` as stale. A pusher whose wall-clock is ahead of the chain by up to `MAX_TIME_DRIFT` seconds will store a price the oracle considers valid, but every subsequent `getBidAndAskPrice()` call will return the `FeedStalled` sentinel, blocking all swaps on every pool that uses that provider for up to 60 seconds per push cycle.

---

### Finding Description

**Oracle push path — allows future timestamps**

`CompressedOracle.fallback()` and `updateBySignature()` both call:

```solidity
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
``` [1](#0-0) 

which resolves to:

```solidity
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}
``` [2](#0-1) 

With the production `MAX_TIME_DRIFT = 60`, a pusher may store `timestampMs = (block.timestamp + 60) * 1000`. The oracle writes this to storage and the stored `refTime` returned by `price()` is `timestampMs / 1000 = block.timestamp + 60`. [3](#0-2) 

**Provider read path — rejects any future `refTime`**

`PriceProvider._isStale()` (L1 variant):

```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;          // ← unconditional future-reject
    return (nowTs - refTime) > maxDelta;
}
``` [4](#0-3) 

The same guard is duplicated verbatim in `AnchoredPriceProvider`: [5](#0-4) 

and in `ProtectedPriceProvider`: [6](#0-5) 

When `refTime = block.timestamp + 60`, the check `refTime > nowTs` is `true`, so `_isStale` returns `true`, and `_getBidAndAskPrice` returns `(0, type(uint128).max)`: [7](#0-6) 

`getBidAndAskPrice()` then reverts with `FeedStalled`: [8](#0-7) 

**The gap**

| Layer | Interval | Behaviour |
|---|---|---|
| `CompressedOracleV1` push gate | `[0, block.timestamp + MAX_TIME_DRIFT]` | Accepts and stores the push |
| L1 `PriceProvider` read gate | `[0, block.timestamp]` | Rejects anything in the future |
| **Misaligned window** | `(block.timestamp, block.timestamp + 60]` | Oracle: valid. Provider: stale. |

This is structurally identical to the external report's two-interval misalignment: one gate (oracle push) unlocks earlier than the other (provider read), creating a window where the stored value is present but unusable.

The L2 variants (`PriceProviderL2`, `ProtectedPriceProviderL2`) explicitly carry a `FUTURE_TOLERANCE` parameter to close this gap: [9](#0-8) 

The L1 variants have no equivalent, confirming the omission is unintentional.

**Production configuration confirms the risk**

`networks.json` sets `"maxTimeDrift": 60` on Ethereum and Base: [10](#0-9) 

---

### Impact Explanation

Every pool whose `IPriceProvider` is an L1 `PriceProvider` or `AnchoredPriceProvider` backed by a `CompressedOracleV1` with `MAX_TIME_DRIFT > 0` will revert on `swap()` with `FeedStalled` for the entire duration of the misaligned window (up to 60 s per push). If the pusher's clock is consistently ahead — a normal condition for off-chain bots — every push cycle re-opens the window, making swaps continuously unavailable. This is broken core pool functionality causing loss of funds (missed trades, failed settlement) above Sherlock thresholds.

---

### Likelihood Explanation

The trigger is a **semi-trusted** actor (the feed creator or any wallet delegated via `allowPushers` / `allowContractPushers`) whose system clock is ahead of the chain's `block.timestamp` by any amount up to `MAX_TIME_DRIFT`. This is a routine condition for off-chain price-push bots. No malicious intent is required; normal operation with a 30–60 s clock lead suffices. The pusher does not need to know about the provider's staleness logic.

---

### Recommendation

Apply the same `FUTURE_TOLERANCE` pattern the L2 variants already use to the L1 `PriceProvider` and `AnchoredPriceProvider`:

```solidity
// In _isStale (L1 variant):
if (refTime > nowTs) {
    return (refTime - nowTs) > FUTURE_TOLERANCE; // mirror L2 logic
}
```

`FUTURE_TOLERANCE` should be set to at least `MAX_TIME_DRIFT` of the oracle it is paired with (60 s for the production `CompressedOracleV1`). Alternatively, enforce `MAX_TIME_DRIFT == 0` on any `CompressedOracleV1` instance that is paired exclusively with L1 providers, and document this constraint in the factory.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

// 1. Deploy CompressedOracleV1 with MAX_TIME_DRIFT = 60
CompressedOracleV1 oracle = new CompressedOracleV1(owner, 60);

// 2. Deploy PriceProvider backed by this oracle
PriceProvider provider = new PriceProvider(
    factory, address(oracle), feedId, 0, 10 /*maxTimeDelta*/, token0, token1
);

// 3. Pusher pushes with timestamp 30 s in the future
uint56 futureTsMs = uint56((block.timestamp + 30) * 1000);
uint48 raw = _packRaw(1_000_000, 3, 3);
bytes memory payload = _wordAt(0, 0, raw, futureTsMs);

vm.prank(creator);
(bool ok,) = address(oracle).call(payload);
assertTrue(ok, "oracle accepts future ts within MAX_TIME_DRIFT"); // PASSES

// 4. Provider rejects the stored price as stale
vm.expectRevert(PriceProvider.FeedStalled.selector);
provider.getBidAndAskPrice(); // REVERTS — refTime = block.timestamp+30 > block.timestamp

// 5. 31 seconds later the window closes and the price is now stale by age too
vm.warp(block.timestamp + 31);
// refTime = block.timestamp_original + 30 = now - 1 → not future, but age = 1s < maxTimeDelta
// Now it succeeds — but the 30-second swap blackout already occurred.
```
