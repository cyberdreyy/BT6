### Title
`priceProviderTimelock` has no minimum enforced, allowing a pool admin to atomically propose and execute a price-provider swap with zero LP reaction time — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory` exposes a two-step price-provider rotation guarded by `priceProviderTimelock[pool]`. Because no minimum value is enforced at pool creation or anywhere else, a pool can be deployed with `priceProviderTimelock = 0`. When the timelock is zero the propose and execute steps can be called in the same block (or even the same transaction via a multicall), giving liquidity providers zero time to exit before a malicious price provider begins feeding bad bid/ask quotes to the pool.

### Finding Description

`proposePoolPriceProvider` computes the execution deadline as:

```solidity
uint256 executeAfter = block.timestamp + timelock;   // timelock = priceProviderTimelock[pool]
``` [1](#0-0) 

`executePoolPriceProviderUpdate` enforces the delay with:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
``` [2](#0-1) 

When `priceProviderTimelock[pool] == 0`, `executeAfter == block.timestamp`. The guard condition `block.timestamp < block.timestamp` is always `false`, so the check never reverts and the update executes immediately.

`_validatePoolParameters` validates fees, tokens, price-provider token matching, and initial amounts, but imposes **no lower bound on `priceProviderTimelock`**: [3](#0-2) 

The `priceProviderTimelock` mapping is declared as a plain `mapping(address => uint256)` with no initialization floor: [4](#0-3) 

### Impact Explanation

A pool admin who created the pool with `priceProviderTimelock = 0` (or any value short enough to fit in one block) can atomically:

1. Call `proposePoolPriceProvider(pool, maliciousPriceProvider)`.
2. Call `executePoolPriceProviderUpdate(pool)` in the same transaction.

The malicious `IPriceProvider` can return any `(bid, ask)` pair that passes the `PriceProvider`'s staleness and guard checks — for example a price far from the true market. Every subsequent swap in the pool executes at the corrupted quote, draining LP token balances in favour of the swapper or the attacker. This is a direct bad-price execution path: LP principal is lost and pool balances fail to cover LP claims.

### Likelihood Explanation

Likelihood is **low**. The trigger requires a pool admin who either (a) intentionally set `priceProviderTimelock = 0` at pool creation, or (b) is a compromised key. However, the protocol's own timelock mechanism is the only on-chain protection LPs have against a rogue admin; when that mechanism is silently absent (zero), the protection disappears without any visible signal to depositors. The analog to the external report is exact: the timelock exists in code but is not long enough (here: not long enough = zero) to give users any reaction window.

### Recommendation

Enforce a protocol-level minimum in `_validatePoolParameters`:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// inside _validatePoolParameters:
if (params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK)
    revert PriceProviderTimelockTooShort();
``` [3](#0-2) 

Additionally, if a `setPriceProviderTimelock` setter is ever added, it must itself be guarded by the current timelock (analogous to how `OracleValueStopLossExtension` gates timelock reductions behind the existing timelock): [5](#0-4) 

### Proof of Concept

```
// Pool created with priceProviderTimelock = 0
// LPs deposit liquidity

// Attacker (pool admin) in one transaction:
factory.proposePoolPriceProvider(pool, address(maliciousPriceProvider));
// executeAfter = block.timestamp + 0 = block.timestamp
// block.timestamp < block.timestamp → false → no revert
factory.executePoolPriceProviderUpdate(pool);

// maliciousPriceProvider.getBidAndAskPrice() now returns
// an arbitrarily skewed (bid, ask) pair.
// Next swap drains LP reserves at the corrupted price.
```

The `_validatePriceProvider` call inside `executePoolPriceProviderUpdate` only checks that `token0()/token1()` match — it does not validate price sanity — so any contract that returns the correct token addresses passes: [6](#0-5)
