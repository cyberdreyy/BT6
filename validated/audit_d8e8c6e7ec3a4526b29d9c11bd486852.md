### Title
Pusher Revocation Bypassed by Creator Replaying Unexpired Consent Signature — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`revokePusher()` clears `namespaceRemapping[pusher]`, but the creator can immediately re-establish the delegation by replaying the pusher's original EIP-191 consent signature as long as the deadline has not expired. No nonce or used-signature tracking exists, so the revocation is not permanent within the deadline window. A compromised pusher key that was revoked can be silently re-delegated, allowing bad prices to continue reaching pools.

---

### Finding Description

`allowPushers` requires the pusher's EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)`: [1](#0-0) 

The code's own comment acknowledges the replay concern and states the deadline is the guard: [2](#0-1) 

However, the deadline only prevents replay **after** it expires. Within the deadline window, the same signature is unconditionally valid on every call. There is no nonce, no used-signature set, and no per-pusher revocation counter. After `revokePusher()` zeroes the mapping: [3](#0-2) 

the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple and the mapping is restored: [4](#0-3) 

The `fallback` push path resolves the effective namespace from `namespaceRemapping[msg.sender]`, so every subsequent push from the pusher's address lands in the creator's namespace again: [5](#0-4) 

---

### Impact Explanation

The invariant broken is: **after `revokePusher()`, the pusher's consent is withdrawn and their pushes must no longer reach the creator's namespace**. Because the creator can replay the signature, this invariant does not hold within the deadline window.

Concrete bad-price path:

1. Pusher's private key is compromised; attacker begins pushing manipulated prices into the creator's namespace.
2. Pusher calls `revokePusher()` to stop the attacker — `namespaceRemapping[pusher] = address(0)`.
3. Creator (or an attacker who also holds the creator key, or a creator who is colluding) calls `allowPushers(deadline, [pusher], [originalSig])` with the same signature — delegation is silently restored.
4. Attacker continues pushing inverted or unbounded prices into the creator's namespace.
5. Pools registered against the creator's feeds consume the bad prices during swaps, satisfying the **bad-price execution** impact gate.

---

### Likelihood Explanation

- The creator must have retained the original `(deadline, sig)` tuple — trivially true since it was submitted on-chain and is visible in calldata.
- The deadline is chosen by the creator and is typically set days or weeks in the future (the test suite uses `block.timestamp + 1 days`).
- No on-chain action by the pusher can prevent the replay within the deadline window.
- The trigger is a single `allowPushers` call by the creator — no special privilege beyond namespace ownership is required.

Likelihood: **Medium** (requires an active creator decision to replay, but the window is wide and the data is always available).

---

### Recommendation

Track consumed signatures with a per-pusher nonce or a `usedSignatures` mapping, and clear the nonce on revocation:

```diff
+ mapping(address => uint256) public pusherNonce;

  function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
      _ensureDeadline(deadline);
      ...
      for (uint256 i; i < l; i++) {
          address pusher = pushers[i];
+         uint256 nonce = pusherNonce[pusher]++;
          bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
-             keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
+             keccak256(abi.encode(block.chainid, address(this), deadline, nonce, pusher, msg.sender))
          );
          ...
      }
  }
```

Alternatively, require the pusher to sign a fresh consent after every revocation by incrementing the nonce inside `revokePusher()`.

---

### Proof of Concept

```
T=0:  pusher signs  hash(chainid, oracle, deadline=T+30d, pusher, creator)
T=1:  creator calls allowPushers(T+30d, [pusher], [sig])
        → namespaceRemapping[pusher] = creator  ✓

T=2:  pusher's key is compromised; attacker pushes bad prices into creator's namespace

T=3:  pusher calls revokePusher()
        → namespaceRemapping[pusher] = address(0)  ✓ (revocation recorded)

T=4:  creator calls allowPushers(T+30d, [pusher], [sig])  ← SAME signature, deadline still valid
        → namespaceRemapping[pusher] = creator  ← revocation silently undone

T=5:  attacker continues pushing bad prices via pusher address
        → fallback: creator = namespaceRemapping[attacker_addr] = creator
        → bad price stored in creator's slot
        → pool swap reads creator's feed → bad-price execution
```
