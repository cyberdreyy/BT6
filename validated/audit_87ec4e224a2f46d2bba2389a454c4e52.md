### Title
CREATE1 Deployment in `AnchoredProviderFactory` and `PriceProviderFactory` Allows Front-Running to Substitute a Malicious Price Provider at the Expected Address - (`smart-contracts-poc/contracts/AnchoredProviderFactory.sol`, `smart-contracts-poc/contracts/PriceProviderFactory.sol`)

---

### Summary

Both `AnchoredProviderFactory.createAnchoredProvider()` and `PriceProviderFactory.createPriceProvider()` deploy provider contracts using the Solidity `new` keyword (CREATE1). The deployed address is therefore a deterministic function of `(factory_address, factory_nonce)`. An attacker who monitors the mempool can front-run a legitimate provider deployment, consuming the factory nonce first and placing a malicious provider at the address the victim expected. Because the factory's own `_providers` set is the sole eligibility gate (`isProvider()`) checked by the pool factory, the malicious provider passes the public-pool admission check and feeds bad prices into every pool that references it.

---

### Finding Description

**Root cause — CREATE1 in both factories:**

`AnchoredProviderFactory.createAnchoredProvider()` deploys via:

```solidity
AnchoredPriceProvider p = new AnchoredPriceProvider(...);
``` [1](#0-0) 

`PriceProviderFactory.createPriceProvider()` deploys via:

```solidity
PriceProvider p = new PriceProvider(...);
``` [2](#0-1) 

Both are CREATE1: the address is `keccak256(rlp(factory, nonce))[12:]`. Anyone can compute the next deployment address off-chain by reading the factory's current nonce.

**Eligibility gate is address-only:**

After deployment the factory records the provider in `_providers` and sets `providerOwner`. The public-pool eligibility predicate is `isProvider(address)`, which is a pure set-membership check:

```solidity
function isProvider(address provider) external view returns (bool) {
    return _providers.contains(provider);
}
``` [3](#0-2) 

No feedId, no token-pair, no oracle binding is re-validated at pool-creation time — only whether the address is in `_providers`.

**`createAnchoredProvider` is permissionless:**

Any caller may invoke `createAnchoredProvider` as long as the chosen oracle is in the admin-curated allowlist and the parameters fall inside the envelope. Both conditions are satisfiable by an attacker using a legitimately allowlisted oracle with a *different* feedId (e.g. ETH/USD instead of BTC/USD) while supplying the same `baseToken`/`quoteToken` as the victim's intended provider. [4](#0-3) 

---

### Impact Explanation

A pool that is created (or reconfigured) to point at the pre-computed provider address will call `getBidAndAskPrice()` on the attacker's contract. The attacker's provider is a fully valid `AnchoredPriceProvider` — it passes `isProvider()` — but it reads a *different* feed (e.g. ETH/USD) for a pool whose tokens are WBTC/USDC. Every swap in that pool executes against the wrong oracle price, allowing the attacker (or any informed party) to drain LP capital via arbitrage. This is a direct loss of LP principal, matching the "bad-price execution" and "pool insolvency" impact categories.

---

### Likelihood Explanation

- `createAnchoredProvider` is permissionless; no special role is required.
- The next deployment address is trivially computable from the factory's on-chain nonce.
- Front-running is standard on EVM chains with a public mempool; chain reorgs (the secondary vector) are common on L2s.
- The attacker only needs one allowlisted oracle to exist (the allowlist is non-empty in any live deployment) and any envelope-valid parameter set.

Likelihood: **Medium** (requires mempool visibility or a reorg, but no privileged access).

---

### Recommendation

Replace `new Contract(...)` with `CREATE2` in both factories, using a salt derived from the full provider configuration (oracle address, feedId(s), tokens, margin parameters). This makes the deployment address a commitment to the config, so a front-runner deploying different parameters lands at a different address and cannot occupy the victim's expected slot.

```solidity
// Example salt for AnchoredProviderFactory
bytes32 salt = keccak256(abi.encode(
    oracle, baseFeedId, quoteFeedId,
    minMargin, maxRefStaleness, maxSpreadBps,
    mutableParams, marginStep,
    baseToken, quoteToken,
    msg.sender          // bind to creator to prevent cross-creator collisions
));
AnchoredPriceProvider p = new AnchoredPriceProvider{salt: salt}(...);
```

Apply the same pattern to `PriceProviderFactory.createPriceProvider`.

---

### Proof of Concept

1. **Setup**: `AnchoredProviderFactory` is live; oracle `O` (e.g. a Pyth oracle) is in the allowlist; envelope `E` covers both BTC/USD and ETH/USD feeds.

2. **Victim's intent**: Alice wants to deploy a BTC/USD `AnchoredPriceProvider` for her WBTC/USDC pool. She computes the expected address `P_expected = CREATE1(factory, nonce)` and submits two transactions to the mempool:
   - `Tx1`: `createAnchoredProvider(O, BTC_FEED, 0, …, WBTC, USDC)` → expected to land at `P_expected`
   - `Tx2`: `MetricOmmPoolFactory.createPool(…, P_expected)` → pool references `P_expected`

3. **Attack**: Bob sees `Tx1` in the mempool and front-runs it with a higher gas price:
   - `Tx_attack`: `createAnchoredProvider(O, ETH_FEED, 0, …, WBTC, USDC)` — same oracle, same tokens, but `baseFeedId = ETH/USD`. Parameters are within envelope `E`.
   - Bob's transaction is mined first; the malicious provider lands at `P_expected` and is registered in `_providers`.

4. **Outcome**: Alice's `Tx1` now deploys at `P_expected + 1` (different address). Alice's `Tx2` creates the pool pointing at `P_expected`. The pool factory calls `isProvider(P_expected)` → `true` (Bob's provider). The WBTC/USDC pool now prices swaps using the ETH/USD feed. Any swap that crosses the pool executes at the wrong price; an informed arbitrageur (Bob) extracts the difference from LP capital.

5. **Reorg variant**: Even without mempool visibility, a chain reorg on an L2 resets the factory nonce; Bob can replay the same attack in the reorganized chain, substituting his provider at the address Alice's pool already references on-chain.
