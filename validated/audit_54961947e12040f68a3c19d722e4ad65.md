### Title
Creator can replay a pusher's EIP-191 consent signature to silently re-establish a revoked delegation, routing wrong-namespace prices into pool-facing feeds — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts a pusher's EIP-191 signature that commits to `(chainid, oracle, deadline, pusher, creator)`. After a pusher calls `revokePusher()` to clear `namespaceRemapping[pusher]`, the creator can immediately replay the identical `(deadline, pusher, sig)` tuple — as long as `block.timestamp ≤ deadline` — to re-write `namespaceRemapping[pusher] = creator`. The pusher, believing the delegation is gone, continues pushing prices intended for their own namespace; those pushes land in the creator's namespace instead, feeding wrong prices to any pool that reads from the creator's feeds.

---

### Finding Description

`allowPushers` signs over `(block.chainid, address(this), deadline, pusher, msg.sender)` and stores `namespaceRemapping[pusher] = msg.sender` on success. [1](#0-0) 

`revokePusher` clears the mapping to `address(0)`: [2](#0-1) 

There is no nonce, no per-pusher revocation counter, and no on-chain record that a given `(deadline, pusher, creator)` tuple was already consumed and then revoked. The only freshness gate is `_ensureDeadline(deadline)`, which passes for any future deadline. The creator retains the original signature bytes and can call `allowPushers` again with the same arguments at any time before the deadline, atomically undoing the revocation.

The `fallback` push path resolves the namespace from `namespaceRemapping[msg.sender]`, falling back to `msg.sender` only when the mapping is zero: [3](#0-2) 

After the creator's replay, the pusher's subsequent `fallback` calls write into `creator`'s namespace slots, not the pusher's own. The pusher has no on-chain way to detect this; `revokePusher` will succeed again, but the creator can replay again immediately.

The code comment in `allowPushers` acknowledges the partial risk but frames the deadline as the mitigation: [4](#0-3) 

The slot-structure documentation repeats the same caveat without providing a code-level fix: [5](#0-4) 

The deadline is chosen by the pusher at signing time, not by the creator at delegation time. A pusher who signs with a 30-day or 1-year deadline (common for operational convenience) gives the creator a long replay window. The pusher has no mechanism to shorten or invalidate the deadline after the fact.

---

### Impact Explanation

After the creator replays the signature, the pusher's `fallback` writes land in the creator's namespace. A pool configured to read `feedIdOf(creator, slotIndex, positionIndex)` receives the pusher's prices, which are now intended for the pusher's own token pair or a different price range. This is a direct bad-price execution path:

- The `ProtectedPriceProvider` / `PriceProvider` reads the creator's feed via `oracle.price(feedId, pool)`.
- The pool's `getBidAndAskPrice` consumes the returned `(mid, spread0, spread1)` to compute bid/ask.
- Swappers execute against a bid/ask derived from the wrong price, receiving more output than the oracle permits or paying less input than owed. [6](#0-5) 

Pool solvency and swap conservation invariants are broken: the pool receives less input than the oracle-anchored curve requires, directly losing LP principal.

---

### Likelihood Explanation

- The creator already holds the pusher's signature bytes (they submitted them in the original `allowPushers` call).
- Any deadline longer than the time between delegation and revocation gives the creator a replay window; deadlines of days to months are operationally normal.
- The pusher has no on-chain signal that the revocation was undone; they will continue pushing, believing they are writing to their own namespace.
- The trigger is a single permissionless `allowPushers` call by the creator — no special role, no timelock, no admin approval.

---

### Recommendation

Track consumed `(deadline, pusher, creator)` tuples with a per-pusher revocation nonce or a `mapping(bytes32 => bool) usedConsents` keyed on `keccak256(abi.encode(deadline, pusher, creator))`. On `revokePusher`, mark the current consent as spent. `allowPushers` must reject any signature whose consent hash is already marked spent, regardless of the deadline.

Alternatively, include a monotonically increasing per-pusher nonce in the signed payload and store the last accepted nonce; `revokePusher` increments the nonce, invalidating all prior signatures.

---

### Proof of Concept

```
1. Creator calls allowPushers(deadline=now+30days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator

2. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)
   → Pusher believes delegation is gone; starts pushing to own namespace

3. Creator calls allowPushers(deadline=now+30days, [pusher], [sig])  ← SAME sig
   → _ensureDeadline passes (deadline still in future)
   → ECDSA.recover returns pusher (same hash, same sig)
   → namespaceRemapping[pusher] = creator  ← revocation silently undone

4. Pusher calls fallback() with slot word for (slotId=5, price=X, ts=T+1)
   → creator = namespaceRemapping[pusher] = creator  ← NOT address(0)
   → namespace = creator << 96
   → key = bytes32(creator_namespace | 5)
   → _writeStorage writes price X into creator's slot 5

5. Pool reads feedIdOf(creator, 5, 0) → receives price X (pusher's wrong-pair price)
   → bid/ask computed from X → swapper drains pool at wrong price
``` [7](#0-6) [2](#0-1) [8](#0-7)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
