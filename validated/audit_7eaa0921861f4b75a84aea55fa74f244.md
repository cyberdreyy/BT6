### Title
Pusher Revocation Bypassed by Signature Replay in `allowPushers` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original `allowPushers` call with the **identical EIP-191 signature** to re-establish the delegation — as long as the deadline has not expired. The code's own NatSpec acknowledges this exact concern but the deadline-only mitigation is insufficient within the deadline window. A compromised pusher key that the legitimate pusher tried to revoke can be silently re-authorized by the creator, allowing bad prices to flow into the creator's namespace and reach live pool swaps.

---

### Finding Description

`allowPushers` signs consent as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is **no nonce, no consumed-flag, and no one-time-use check**. The only replay guard is the deadline. After `revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

…the creator can call `allowPushers` again with the **exact same calldata and signature** to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is therefore completely ineffective for the entire remaining lifetime of the signed consent.

The code's own NatSpec explicitly acknowledges this concern:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The comment correctly identifies the threat but the deadline-only fix only prevents using an **expired** signature. Within the deadline window, the replay is fully valid and the revocation is nullified.

---

### Impact Explanation

The `CompressedOracle` is the upstream data source for `PriceProvider` / `AnchoredPriceProvider`, which pools call via `getBidAndAskPrice` during every swap. If a pusher's private key is compromised:

1. The legitimate pusher calls `revokePusher()` — their fallback pushes now land in their own namespace, not the creator's.
2. The creator (who may not know about the compromise, or who is acting on a stale automation script) calls `allowPushers` with the same signature — delegation is restored.
3. The attacker uses the compromised key to push arbitrary `(price, spread0, spread1, timestampMs)` tuples into the creator's namespace via the permissionless fallback path.
4. Pools consuming the creator's feed receive bad bid/ask quotes → bad-price execution → direct loss of trader or LP principal.

This is a **bad-price execution** impact: an unbounded or attacker-controlled quote reaches a live pool swap.

---

### Likelihood Explanation

**Medium.** The window is bounded by the deadline, but production deployments commonly use deadlines of days to weeks. The creator replaying `allowPushers` is realistic in any automated or scripted deployment that re-runs delegation setup after an outage or key rotation. The pusher has no on-chain mechanism to prevent the creator from replaying the signed consent before the deadline expires.

---

### Recommendation

Add a per-pusher nonce or a consumed-signature mapping to `allowPushers`. For example:

```solidity
mapping(address pusher => uint256 nonce) public pusherNonce;

// Include nonce in the signed message:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]++))
```

Alternatively, store a `mapping(bytes32 => bool) usedSignatureHashes` and mark each accepted signature hash as consumed, preventing any replay regardless of deadline.

---

### Proof of Concept

```
1. Pusher signs consent: keccak256(abi.encode(chainid, oracle, deadline=T+30d, pusher, creator))
2. Creator calls allowPushers(deadline, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
3. Pusher's key is compromised; legitimate pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (revocation appears successful)
4. Creator (unknowingly, via automation) calls allowPushers(deadline, [pusher], [sig]) — SAME calldata
   → namespaceRemapping[pusher] = creator  ← revocation nullified
5. Attacker uses compromised pusher key to call oracle fallback with crafted (price, spread, ts)
   → data lands in creator's namespace
6. Pool calls provider.getBidAndAskPrice() → oracle.price(feedId, pool)
   → returns attacker-controlled bid/ask
   → swap executes at bad price → loss of user funds
```
