### Title
Permissionless `register()` Unconditionally Clears Admin Blacklist, Bypassing Abuse-Protection Gate — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

The `register()` function in the providers `OracleBase` is permissionless (any caller, any pool) and unconditionally clears `blacklisted[pool]` whenever the pool is currently blacklisted. Because the ADMIN-only `setBlacklist()` is the only path to blacklist a pool, but `register()` is the unprivileged path to un-blacklist it, the abuse-protection invariant is broken: any actor can restore oracle read access for a blacklisted pool by paying the default registration fee of 1 wei.

---

### Finding Description

`OracleBase` (providers) exposes two asymmetric paths for the pool blacklist:

**Blacklist set — ADMIN only:** [1](#0-0) 

**Blacklist cleared — permissionless, inside `register()`:** [2](#0-1) 

The only guards on `register()` are:
1. `msg.value >= registrationFee` (default: **1 wei**)
2. `factory` must be in `approvedFactories`
3. `factory.isPool(pool)` must return `true`

None of these checks prevent a blacklisted pool's operator (or any third party) from calling `register()` to silently flip `blacklisted[pool]` back to `false`. The `price()` read path enforces the blacklist at swap time: [3](#0-2) 

Once `blacklisted[pool]` is cleared via `register()`, the `require(!blacklisted[pool], Blacklisted(pool))` check passes and the pool resumes reading oracle prices. The ADMIN can re-blacklist, but the attacker can immediately re-register for 1 wei, creating a permanent race the ADMIN cannot win on-chain.

The default fee is set at construction: [4](#0-3) 

Even if ADMIN raises the fee via `setRegistrationFee`, the pool operator (who has a financial incentive to keep the pool live) will simply pay it.

---

### Impact Explanation

The blacklist is the oracle's primary runtime abuse-protection mechanism. A blacklisted pool cannot call `price(feedId, pool)` and therefore cannot execute swaps that consume oracle prices. Bypassing the blacklist means:

- A pool flagged for price-feed abuse, sandwich attacks, or anomalous read patterns regains full oracle access immediately.
- The `PriceRead` event attribution that the oracle relies on for off-chain monitoring is restored, masking continued abuse.
- Any downstream `AnchoredPriceProvider` pool that was protected by the blacklist resumes receiving oracle-driven bid/ask quotes, re-exposing LPs to the original abuse vector.

This is a direct admin-boundary break: an oracle-admin role check (`onlyRole(ADMIN_ROLE)` on `setBlacklist`) is bypassed by an unprivileged, cheap, repeatable call.

---

### Likelihood Explanation

- **Cost:** 1 wei (the hardcoded default `registrationFee`).
- **Prerequisites:** The pool must be from an approved factory — but a pool that was already registered and then blacklisted trivially satisfies this (it was approved before blacklisting).
- **Repeatability:** Every time ADMIN re-blacklists, the attacker calls `register()` again. No signature, no timelock, no cooldown.
- **Trigger:** Any EOA or contract, including the pool operator themselves.

---

### Recommendation

Remove the blacklist-clearing side-effect from `register()`. Registration and blacklist management are orthogonal concerns:

```solidity
// Remove or guard this block in register():
if (blacklisted[pool]) {
    blacklisted[pool] = false;          // ← anyone can reach this
    emit BlacklistUpdated(pool, false);
}
```

Either:
1. **Delete the block entirely** — registration does not imply rehabilitation.
2. **Gate it on `ADMIN_ROLE`** — only ADMIN can clear a blacklist, even inside `register()`.
3. **Revert if blacklisted** — `require(!blacklisted[pool], Blacklisted(pool))` before registering, forcing ADMIN to explicitly un-blacklist first.

---

### Proof of Concept

```
1. ADMIN calls setBlacklist(pool, true)
   → blacklisted[pool] = true
   → pool.price(feedId, pool) reverts with Blacklisted(pool)

2. Attacker (or pool operator) calls:
   register{value: 1 wei}(feedId, pool, approvedFactory)
   
   Execution path:
   - msg.value (1 wei) >= registrationFee (1 wei)  ✓
   - approvedFactories.contains(approvedFactory)    ✓
   - approvedFactory.isPool(pool)                   ✓  (pool was deployed by this factory)
   - blacklisted[pool] == true → blacklisted[pool] = false  ← BLACKLIST CLEARED
   - registeredPool[feedId][pool] = true

3. pool.price(feedId, pool) now succeeds
   → oracle returns live bid/ask to the previously-blacklisted pool
   → AnchoredPriceProvider resumes quoting swaps through that pool

4. ADMIN re-blacklists → attacker pays 1 wei again → repeat indefinitely
``` [2](#0-1) [3](#0-2)

    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
    }
```
