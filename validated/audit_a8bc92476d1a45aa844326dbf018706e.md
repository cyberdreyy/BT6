### Title
`purgeStateGuardRole` Fails to Clear `pendingStateGuard`, Allowing Stale Pending Guard to Seize Feed Control and Manipulate `priceGuard` ‚Äî (`File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`purgeStateGuardRole` deletes `stateGuard[feedId]` but never clears `pendingStateGuard[feedId]`. A pending guard that was set before the purge can call `acceptStateGuardRole` after the purge and become the new `stateGuard`. That guard then controls `setPriceGuard`, which is the per-feed price-range gate consumed by every price provider (`PriceProvider`, `ProtectedPriceProvider`, `AnchoredPriceProvider`).

---

### Finding Description

In `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, `purgeStateGuardRole` is the function a current guard (or ADMIN when no guard is set) calls to remove guard authority from a feed:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];          // ‚Üê only this is cleared
    emit StateGuardDeleted(feedId);
    // pendingStateGuard[feedId] is NEVER touched
}
``` [1](#0-0) 

The two-step guard-transfer flow is:

1. `setStateGuardRole(feedId, newGuard)` ‚Üí writes `pendingStateGuard[feedId] = newGuard`
2. `newGuard` calls `acceptStateGuardRole(feedId)` ‚Üí promotes `pendingStateGuard` to `stateGuard` and clears `pendingStateGuard` [2](#0-1) 

The dedicated cancellation function is `purgePendingStateGuardRole`, which clears only `pendingStateGuard`: [3](#0-2) 

Because `purgeStateGuardRole` does not call `delete pendingStateGuard[feedId]`, the following sequence leaves the feed in an inconsistent state:

1. Guard (or ADMIN) calls `setStateGuardRole(feedId, pendingGuard)` ‚Äî sets `pendingStateGuard[feedId] = pendingGuard`.
2. Guard calls `purgeStateGuardRole(feedId)` ‚Äî intending to remove all guard authority and return control to ADMIN. `stateGuard[feedId]` is deleted; `pendingStateGuard[feedId]` is **not** deleted.
3. `pendingGuard` calls `acceptStateGuardRole(feedId)` ‚Äî the check `pendingStateGuard[feedId] == msg.sender` still passes, so `pendingGuard` becomes the new `stateGuard`.

The analog to the external report is exact: `removeLoanImpairment` removes the impairment but leaves `isImpaired = true`; `purgeStateGuardRole` removes the guard but leaves `pendingStateGuard` pointing at the pending address.

---

### Impact Explanation

`stateGuard` is the sole authority for `setPriceGuard` on a feed once it is set:

```solidity
modifier checkRole(bytes32 feedId) {
    address _guard = stateGuard[feedId];
    if (_guard != address(0)) {
        require(_guard == msg.sender, InvalidGuard(msg.sender));
    } else {
        _checkRole(ADMIN_ROLE);
    }
    _;
}
``` [4](#0-3) 

`priceGuard` is the per-feed price-range gate consumed by every price provider. In `PriceProvider._getBidAndAskPrice()`:

```solidity
(uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(offchainFeedId);
guardMax = guardMax == 0 ? type(uint128).max : guardMax;
if (mid < guardMin || mid > guardMax) {
    return (0, type(uint128).max);
}
``` [5](#0-4) 

The same check appears in `ProtectedPriceProvider._computeBidAsk()`: [6](#0-5) 

And in `AnchoredPriceProvider._readLeg()`: [7](#0-6) 

An attacker who seizes `stateGuard` via the stale `pendingStateGuard` can:

- **DoS swaps**: set `priceGuard` to a 1-unit range (e.g., `min = X, max = X+1`). Every oracle price outside that range causes the provider to return `(0, type(uint128).max)`, which the pool interprets as a stalled feed and reverts all swaps.
- **Disable the guard**: set `priceGuard` to `(0, 0)` (treated as unlimited) or `(1, type(uint128).max)`, removing the price-range protection entirely. Combined with any ability to influence oracle data, this allows out-of-range prices to reach the pool.

---

### Likelihood Explanation

The trigger requires two sequential privileged actions:

1. A current guard (or ADMIN) calls `setStateGuardRole` to nominate a pending guard.
2. The same actor calls `purgeStateGuardRole` believing it cancels the pending transfer (the correct cancellation function is `purgePendingStateGuardRole`).

This is a realistic operational mistake: the two functions have similar names and the documentation does not warn that `purgeStateGuardRole` leaves `pendingStateGuard` intact. A guard who wants to "undo everything" and return to ADMIN control would naturally call `purgeStateGuardRole` ‚Äî the function whose name most closely matches "remove the guard role" ‚Äî without realising the pending slot survives.

The pending guard is a semi-trusted actor (they were nominated by a privileged party), so the impact is not gated on a fully unprivileged path, but the broken invariant ‚Äî that purging the guard returns full control to ADMIN ‚Äî is the root cause.

---

### Recommendation

Clear `pendingStateGuard` inside `purgeStateGuardRole`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete pendingStateGuard[feedId];   // ‚Üê add this line
    delete stateGuard[feedId];
    emit StateGuardDeleted(feedId);
}
```

This mirrors the fix for the external report: just as `removeLoanImpairment` should reset `isImpaired = false`, `purgeStateGuardRole` must reset `pendingStateGuard[feedId] = address(0)`.
