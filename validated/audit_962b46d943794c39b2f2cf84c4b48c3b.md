### Title
`revokePusher()` is ineffective within the deadline window due to missing nonce in `allowPushers` signature domain — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` signs consent over `(chainid, oracle, deadline, pusher, creator)` with no nonce. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature to re-establish `namespaceRemapping[pusher] = creator`, nullifying the revocation for the entire remaining deadline window.

### Finding Description
`allowPushers` verifies a pusher-signed EIP-191 message and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The signature domain is `(chainid, address(this), deadline, pusher, creator)` — no nonce, no per-pusher sequence counter, no used-signature set.

`_ensureDeadline` only checks `block.timestamp <= deadline`: [2](#0-1) 

`revokePusher()` clears the mapping to `address(0)`: [3](#0-2) 

But nothing invalidates the already-issued signature. The creator can call `allowPushers(deadline, [pusher], [originalSig])` again immediately after the pusher's revocation, restoring `namespaceRemapping[pusher] = creator`. This can be repeated every block until the deadline expires.

The code comment at lines 186–191 explicitly claims the deadline prevents post-revocation re-delegation, but this is incorrect — the deadline only prevents replay *after* it expires, not *within* the window: [4](#0-3) 

### Impact Explanation
The pusher's `revokePusher()` call is immediately undone by the creator. The pusher cannot redirect their pushes to their own namespace until the deadline expires. Any fallback push the pusher makes while the creator keeps replaying the signature will land in the creator's namespace, not the pusher's own. If the pusher is trying to stop feeding data into a production feed (e.g., because they detected a compromise or disagreement), the creator can override this for the entire deadline window, keeping stale or unwanted prices flowing into the creator's namespace and any pool consuming it.

### Likelihood Explanation
The attack requires only that the creator retain the original signature bytes (trivially available from the original `allowPushers` calldata on-chain) and call `allowPushers` again after the pusher revokes. No special privilege, no off-chain coordination, and no new signature from the pusher is needed.

### Recommendation
Add a per-pusher revocation nonce or a used-signature set. The simplest fix is a `mapping(address => uint256) public pusherNonce` incremented by `revokePusher()`, and include `pusherNonce[pusher]` in the signed hash:

```solidity
// in signature domain:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))
// in revokePusher():
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

This makes every previously issued signature invalid the moment the pusher revokes.

    vm.prank(creator);

    assertEq(oracle.namespaceRemapping(pusher), creator);
