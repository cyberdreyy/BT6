### Title
Pusher Delegation Signature Replayable Within Deadline Window — Revocation Ineffective (`CompressedOracle.sol::allowPushers`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but never marks it as consumed. Any valid signature can be replayed an unlimited number of times before its `deadline` expires. A creator who holds a pusher's old consent signature can silently re-establish the delegation immediately after the pusher calls `revokePusher()`, making revocation completely ineffective within the deadline window and allowing the creator to keep routing the pusher's price writes into the creator's namespace without fresh consent.

---

### Finding Description

`allowPushers` constructs and verifies a signed hash:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The only replay guard is the deadline check in `_ensureDeadline`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

There is **no used-signature bitmap, nonce, or consumed-hash mapping**. The contract's own NatDoc acknowledges the replay concern but incorrectly concludes the deadline alone is sufficient:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline prevents replay **after** it expires, but within the validity window the same bytes can be submitted repeatedly. `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [4](#0-3) 

But the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple, writing `namespaceRemapping[pusher] = creator` again. The pusher's revocation is undone in the same block.

---

### Impact Explanation

`namespaceRemapping` is the sole authority that determines which namespace a pusher's `fallback()` writes land in:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
assembly ("memory-safe") {
    namespace := shl(96, creator)
}
``` [5](#0-4) 

A pusher whose key is compromised, or who wants to stop serving a creator, calls `revokePusher()` to stop their pushes from landing in the creator's namespace. If the creator replays the old signature, the pusher's subsequent pushes — including potentially bad or stale prices — continue to overwrite the creator's feed slots. Those slots are read by `AnchoredPriceProvider._readLeg()`:

```solidity
(mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
``` [6](#0-5) 

and returned to pools via `getBidAndAskPrice()`, directly setting the bid/ask prices used in swaps. A bad price injected into the creator's namespace through a re-delegated (but revoked) pusher reaches live pool swaps.

---

### Likelihood Explanation

- The creator already holds the pusher's signature (they used it to call `allowPushers` the first time).
- Replaying it requires a single public transaction with no additional privileges.
- The attack window is the entire remaining lifetime of the deadline, which callers are incentivised to set far in the future to avoid operational friction.
- The pusher has no on-chain way to invalidate the old signature short of waiting for the deadline to expire.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedDelegationHashes` and mark each hash as used on first acceptance:

```solidity
mapping(bytes32 => bool) private _usedDelegationHashes;

// inside allowPushers loop:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedDelegationHashes[hash], SignatureAlreadyUsed());
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedDelegationHashes[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, include a per-pusher nonce in the signed payload so each consent is unique and cannot be replayed even with a fresh deadline.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline = block.timestamp + 7 days.
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator delegates pusher.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — no new consent from pusher.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // re-delegated without pusher's knowledge

// 5. Pusher's next fallback push lands in creator's namespace again.
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, _packRaw(BAD_PRICE, 5, 5), tsMs));
assertTrue(ok);
// creator's feed slot now contains BAD_PRICE, consumed by AnchoredPriceProvider → pool swap.
```
