### Title
Pusher Delegation Signature Replay Allows Creator to Permanently Prevent `revokePusher()` Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers()` in `CompressedOracleV1` accepts an EIP-191 signature that commits a pusher wallet to push price data into a creator's namespace. No nonce or single-use guard exists on the signature. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature (with the same `deadline`) to re-establish the delegation. This cycle can repeat until the deadline expires, making `revokePusher()` ineffective for the entire deadline window.

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The code's own NatSpec comment acknowledges the replay risk and claims the deadline mitigates it:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

However, the deadline only bounds the outer window — it does **not** prevent the creator from replaying the same signature multiple times within that window. There is no nonce, no `usedSignatures` mapping, and no single-use invalidation. After `revokePusher()` zeroes out `namespaceRemapping[pusher]`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple, writing `namespaceRemapping[pusher] = creator` again:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [4](#0-3) 

The `fallback()` push path resolves the effective namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after the creator's replay goes into the creator's namespace, not the pusher's own namespace, regardless of how many times the pusher calls `revokePusher()`.

### Impact Explanation

The broken invariant is: **a pusher's call to `revokePusher()` must permanently terminate their delegation**. Instead, the creator can nullify every revocation attempt for the entire deadline window.

Concrete harm path:

1. Pusher P signs a delegation to Creator C with a deadline set days or weeks in the future (normal operational practice for a hot-wallet pusher).
2. P discovers their signing key is compromised or that C is feeding manipulated prices to a pool.
3. P calls `revokePusher()` to stop their pushes from reaching C's namespace and to signal key compromise.
4. C (or an attacker who obtained the original delegation calldata from the mempool/event history) immediately replays `allowPushers(deadline, [P], [sig])` — zero new signature required.
5. P's pushes continue to land in C's namespace. Any pool consuming C's `feedId` continues to receive prices attributed to P's key, including prices pushed by the attacker who compromised P.
6. Steps 3–4 repeat in a loop until the deadline expires. The pusher cannot stop the flow of (potentially attacker-controlled) prices into the pool.

This satisfies the "bad-price execution" and "oracle role checks bypassed by an unprivileged path" impact criteria: the creator (not a privileged admin) is using a replayed off-chain signature to override an on-chain revocation, keeping a potentially compromised price source active in the oracle that pools call via `getOracleData` / `price`.

### Likelihood Explanation

- The delegation signature and `deadline` are visible on-chain from the original `allowPushers` transaction.
- The creator needs only to re-submit the same calldata — no new cryptographic material is required.
- Pushers routinely sign delegations with multi-day or multi-week deadlines to avoid frequent re-signing; the replay window is therefore large in practice.
- The attack requires no special role: any address that was previously granted a delegation can be targeted by the creator who originally called `allowPushers`.

### Recommendation

Mark each signature as consumed after first use. The minimal fix is a `usedSignatures` mapping:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

// inside allowPushers loop:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedDelegationSigs[sigHash], "signature already used");
_usedDelegationSigs[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// signed digest:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]++))
```

This mirrors the fix recommended in the analogous Rigor H-04 report: "Use nonce to protect `setComplete()` and `changeOrder()` from signatures reuse."

### Proof of Concept

```solidity
// 1. Pusher signs delegation (off-chain):
bytes32 digest = keccak256(abi.encode(
    block.chainid, address(oracle), deadline, pusher, creator
));
bytes memory sig = sign(pusherKey, toEthSignedMessageHash(digest));

// 2. Creator establishes delegation:
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher revokes:
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature — no new sig needed:
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator again ✓
// revokePusher() had zero lasting effect.

// 5. Any push by the (now-compromised) pusher key lands in creator's namespace:
vm.prank(pusherAttacker);
(bool ok,) = address(oracle).call(slotWord); // goes to creator's feedId
// pool.getQuotes() reads creator's feedId → attacker-controlled price
```
