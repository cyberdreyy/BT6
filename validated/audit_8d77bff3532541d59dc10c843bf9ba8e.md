### Title
Pusher Revocation Bypass via Signature Replay in `allowPushers` Allows Creator to Force Stale/Wrong Prices into Pool Namespace — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` does not track consumed signatures or use a nonce. A creator can replay the same pusher consent signature any number of times within the deadline window to silently re-establish delegation after the pusher has explicitly called `revokePusher()`. The pusher's revocation is nullified, their subsequent pushes land in the creator's namespace instead of their own, and any pool anchored to that namespace can receive wrong or stale prices.

### Finding Description

`allowPushers` verifies a pusher's EIP-191 signature over `(block.chainid, address(this), deadline, pusher, msg.sender)` and writes `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

There is no nonce, no used-signature bitmap, and no check that the pusher is not already delegated. The only replay protection is the deadline, which is a future timestamp chosen by the creator. Nothing prevents the creator from calling `allowPushers` a second (or third) time with the identical `(deadline, pusher, sig)` triple.

`revokePusher` clears the mapping: [2](#0-1) 

But because the original signature is still cryptographically valid until `deadline`, the creator can immediately call `allowPushers` again with the same bytes to restore `namespaceRemapping[pusher] = creator`. The pusher's explicit revocation is overwritten in the same block.

The code's own comment acknowledges the deadline is the sole guard against post-revocation replay: [3](#0-2) 

The deadline prevents using a signature *after* it expires, but does nothing to prevent reuse *within* the window. There is no maximum deadline enforced by `_ensureDeadline`, so a creator can request a signature with `deadline = block.timestamp + 365 days`, giving a year-long replay window.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So every push the pusher makes after believing they have revoked will land in the creator's namespace, not their own.

### Impact Explanation

After revocation the pusher may begin pushing data to their own namespace for a different asset pair or a different pool. Because the creator has silently re-established delegation, those pushes overwrite the creator's namespace slots. Any pool whose `AnchoredPriceProvider` reads from the creator's namespace will receive the wrong mid price, wrong spread, or a stale timestamp, causing bad-price execution: traders receive more token output than the oracle permits, or LPs suffer losses from mispriced swaps. [5](#0-4) 

Even if the pusher simply stops pushing after revocation, the creator can re-establish delegation and then a third party who later obtains the pusher's key (or the pusher themselves, unaware of the re-delegation) can write arbitrary data into the creator's namespace.

### Likelihood Explanation

- The creator must be malicious or negligent (semi-trusted, within scope).
- The pusher's consent signature is public once submitted on-chain; the creator retains it.
- No special access or privileged role beyond being the namespace creator is required.
- The deadline window can be arbitrarily long (no cap enforced).
- The pusher has no on-chain way to detect that their revocation was overridden.

### Recommendation

Track consumed signatures to prevent reuse. The simplest fix is a `mapping(bytes32 => bool) private _usedConsents` keyed on `keccak256(signature)` (or on the full digest), set to `true` on first use and checked before writing `namespaceRemapping`. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful delegation, so a revoked pusher must sign a fresh consent with the new nonce before delegation can be re-established.

### Proof of Concept

```
// 1. Pusher signs consent for creator, deadline = now + 365 days.
bytes memory sig = pusher.sign(
    keccak256(abi.encode(chainid, oracle, deadline, pusher, creator))
);

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature — no revert, no nonce check.
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator again — revocation bypassed!

// 5. Pusher pushes data they intend for their own namespace (e.g. BTC/USD).
vm.prank(pusher);
oracle.call(wordAt(slotId=7, pos=0, btcUsdPrice, tsMs));

// 6. Data lands in creator's namespace (ETH/USD pool reads slot 7, pos 0).
//    Pool receives BTC/USD price as ETH/USD → bad-price execution.
assertEq(
    oracle.getOracleData(oracle.feedIdOf(creator, 7, 0)).price,
    btcUsdPrice   // wrong price in ETH/USD pool
);
``` [6](#0-5)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
