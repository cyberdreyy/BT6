### Title
`MAX_TIME_DRIFT` Buffer in `revertIfAfterBlockTimeWithDrift` Accepts Future-Dated Timestamps, Enabling Oracle Lock-Out and Stale Price Execution Against Pool Swaps — (`smart-contracts-poc/contracts/oracles/utils/TimeMs.sol`)

---

### Summary

The `revertIfAfterBlockTimeWithDrift` function in `TimeMs.sol` silently extends the accepted timestamp window by `MAX_TIME_DRIFT` seconds beyond `block.timestamp`. This is the direct analog of the external "expiry buffer" bug: just as that protocol extended order validity beyond user intent, the `CompressedOracleV1` extends the oracle's freshness acceptance window beyond the current block time. A malicious or compromised pusher can submit a price update with a timestamp `MAX_TIME_DRIFT` seconds in the future, lock out all legitimate price updates for that duration via the monotonicity gate, and then have a stale price consumed as "fresh" by the pool once the future timestamp catches up to real time.

---

### Finding Description

In `TimeMs.sol`, the freshness guard is:

```solidity
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}
``` [1](#0-0) 

This allows any submitted timestamp up to `block.timestamp + MAX_TIME_DRIFT` (in seconds). `CompressedOracleV1` calls this guard in both its `fallback()` push path and `updateBySignature`:

```solidity
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
``` [2](#0-1) [3](#0-2) 

After the drift check passes, the only other gate is the monotonicity check:

```solidity
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;
``` [4](#0-3) 

A pusher who submits a slot word with `timestampMs = (block.timestamp + MAX_TIME_DRIFT) * 1000` ms will:

1. **Pass the drift check** — the boundary condition `t0.toSeconds() <= block.timestamp + drift` is satisfied exactly.
2. **Win the monotonicity check** — the future timestamp is strictly greater than any currently stored timestamp, so the price is written.
3. **Lock out all legitimate pushers** for `MAX_TIME_DRIFT` seconds — any subsequent push with a real current timestamp `T_now < T_future` fails `isAfter` and is silently skipped.

During the lock-out window, `CompressedOracleV1.price()` returns `refTime = T_future` (in seconds). The consumer `AnchoredPriceProvider._readLeg()` then calls:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [5](#0-4) 

where `_isStale` contains:

```solidity
if (refTime > nowTs) return true;
``` [6](#0-5) 

So during the lock-out window the feed is stale, `getBidAndAskPrice()` reverts with `FeedStalled()`, and pool swaps are completely blocked. Once `block.timestamp` reaches `T_future`, the stored price transitions from "stale" to "fresh" — but the price data (mid, spread) is `MAX_TIME_DRIFT` seconds old. If the market moved during that window, the pool now executes swaps against a stale price.

The same `revertIfAfterBlockTimeWithDrift` pattern is also present in `ChainlinkOracle._store()`:

```solidity
d.timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
if (d.timestampMs.isAfter(oracleData[feedId].timestampMs)) {
    oracleData[feedId] = d;
``` [7](#0-6) 

However, for `ChainlinkOracle` the trust anchor is the DON's `VerifierProxy` signature, so a future-dated report requires a compromised DON — a much higher bar. The `CompressedOracleV1` path is the reachable surface: any delegated pusher (or the feed creator themselves) can craft the calldata freely.

---

### Impact Explanation

**Bad-price execution**: After the `MAX_TIME_DRIFT` lock-out window expires, the pool's `AnchoredPriceProvider` consumes a price that is `MAX_TIME_DRIFT` seconds old as if it were fresh. If the market moved significantly during that window, every swap executed in the first block after the window ends trades against a stale bid/ask. LPs suffer adverse selection losses and traders receive incorrect execution prices. The pool's oracle-anchored invariant — that the bid/ask reflects a current external price — is broken for the duration of the stale window.

Additionally, during the lock-out window itself, all swaps revert with `FeedStalled()`, rendering the pool completely unusable — broken core pool functionality.

---

### Likelihood Explanation

The trigger is a **semi-trusted pusher**: any address delegated by the feed creator via `allowPushers` or `allowContractPushers`, or the creator themselves. The `CompressedOracleV1` is registrationless and open — no admin approval is needed to push. The calldata to exploit this is trivial to construct (a single 32-byte slot word with a future timestamp in the low 7 bytes). A delegated pusher who turns malicious, or a creator who wants to manipulate their own pool, can execute this with a single transaction.

---

### Recommendation

Remove the `drift` tolerance from the freshness guard and treat the on-chain block timestamp as the exact upper boundary — mirroring the external report's recommendation to treat the expiry as an exact, user-controlled boundary:

```solidity
// Before (vulnerable):
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}

// After (fixed):
function revertIfAfterBlockTime(TimeMs t0) view {
    require(t0.toSeconds() <= block.timestamp, FutureTimestamp());
}
``` [1](#0-0) 

Update all call sites in `CompressedOracle.sol` and `ChainlinkOracle.sol` accordingly. If minor clock skew tolerance is genuinely required, bound it to a small constant (e.g., 2–3 seconds) rather than the full `MAX_TIME_DRIFT` value, and document it explicitly.

### Proof of Concept

```
Setup:
  - CompressedOracleV1 deployed with MAX_TIME_DRIFT = 60 (seconds)
  - AnchoredPriceProvider configured with baseFeedId = feedIdOf(creator, 0, 0)
  - MAX_REF_STALENESS = 300 seconds
  - Pool registered and active; current market price P1

Attack (single transaction by creator or delegated pusher):
  T = block.timestamp

  Craft slot word:
    timestampMs = (T + 60) * 1000   // 60 seconds in the future, in ms
    slotId      = 0
    price       = P1 (current price, encoded as U64x32)
    spread      = valid codebook indexes

  Call CompressedOracleV1 fallback with this 32-byte word.

  Check 1: revertIfAfterBlockTimeWithDrift
    (T + 60) <= T + 60  ✓  passes

  Check 2: monotonicity
    (T + 60) * 1000 > oldTimestampMs  ✓  passes (assuming no prior future-dated push)

  Price P1 with timestamp T+60 is now stored.

T to T+59:
  Any legitimate pusher submits timestamp T+k (k < 60):
    isAfter((T+k)*1000, (T+60)*1000) → false → skipped
  Pool swaps: AnchoredPriceProvider reads refTime = T+60 > block.timestamp → stale → FeedStalled()
  All swaps revert. Pool is unusable.

T+60:
  block.timestamp = T+60
  AnchoredPriceProvider reads refTime = T+60
  _isStale(T+60, T+60, 300) → refTime == nowTs → not stale → price P1 accepted as fresh
  But market has moved to P2 during the 60-second window.
  Pool executes swaps at stale price P1 → bad-price execution against LPs / traders.
```
