### Title
Pusher consent signature is not invalidated on revocation, allowing the creator to replay it within the deadline window to re-establish a revoked delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers()` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()` (or a creator calls `removePushers()`), the original signature remains cryptographically valid until the deadline expires. The creator can call `allowPushers()` again with the identical `(deadline, pusher, signature)` tuple to silently re-establish the delegation the pusher just revoked, causing the pusher's subsequent fallback pushes to land in the creator's namespace without the pusher's current consent.

---

### Finding Description

`allowPushers()` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only liveness check is `_ensureDeadline(deadline)`, which rejects calls after the deadline but accepts any number of calls before it. [2](#0-1) 

There is no `usedSignatures` bitmap, no per-pusher nonce, and no state change that would invalidate the signature after first use. The code's own NatSpec acknowledges the replay risk but claims the deadline is the sole mitigation:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The deadline prevents replay *after* it expires, but does nothing to prevent replay *within* the deadline window. `revokePusher()` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [4](#0-3) 

It does not burn the signature. The creator, who holds the original bytes, can immediately call `allowPushers()` again with the same arguments and the same signature, passing both the deadline check and the ECDSA recovery check, and writing `namespaceRemapping[pusher] = creator` again.

The fallback push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

So every push the pusher makes after their revocation — believing it goes to their own namespace — is silently redirected to the creator's namespace.

---

### Impact Explanation

The pusher's fallback pushes, which they believe are writing to their own namespace with their own price parameters, are instead written into the creator's namespace. If the pusher is operating with different price levels, spreads, or codebook indices than the creator's legitimate feed, the creator's oracle slots receive corrupted data. Any `PriceProvider` or pool consuming those feed IDs will read the wrong mid price, spread0, or spread1, producing bad-price execution in live swaps. The invariant broken is: **a pusher's revocation must be permanent until a new consent is signed**.

---

### Likelihood Explanation

The creator holds the original signature bytes (they submitted the first `allowPushers()` call). The replay requires only a single public transaction within the deadline window. Deadlines are typically set days in the future (the test suite uses `block.timestamp + 1 days`). [6](#0-5) 

The creator is a semi-trusted party (not a privileged oracle admin), and the pusher is an unprivileged valid trigger. No special setup is required beyond the normal delegation flow.

---

### Recommendation

Track each consent signature as consumed on first use:

```solidity
mapping(bytes32 => bool) private _usedConsentSignatures;

// inside allowPushers(), after ECDSA.recover:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedConsentSignatures[sigHash], "signature already used");
_usedConsentSignatures[sigHash] = true;
```

Alternatively, include a per-pusher monotonic nonce in the signed digest so that each consent is bound to exactly one delegation event and cannot be replayed even within the deadline window.

---

### Proof of Concept

```
1. Pusher signs consent:
   digest = keccak256(abi.encode(chainid, oracle, deadline=T+1day, pusher, creator))
   sig = ECDSA.sign(pusherKey, digest)

2. Creator calls allowPushers(deadline, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  (pusher believes delegation is gone)

4. Creator calls allowPushers(deadline, [pusher], [sig])  ← SAME sig, deadline still valid
   → _ensureDeadline passes (T+1day > now)
   → ECDSA.recover returns pusher  ← sig is still valid
   → namespaceRemapping[pusher] = creator  ← delegation silently re-established

5. Pusher calls fallback() with price data intended for their own namespace
   → namespaceRemapping[pusher] == creator  → push lands in creator's namespace
   → creator's oracle slots now contain the pusher's unintended price/spread values
   → pools reading creator's feedIds receive corrupted bid/ask quotes
```
