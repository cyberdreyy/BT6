### Title
Pusher delegation signature replay within deadline window nullifies `revokePusher()` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` accepts an EIP-191 signature that binds `(chainid, address(this), deadline, pusher, creator)`. There is no used-signature registry and no nonce. A creator who holds a valid signature can call `allowPushers` with the same signature an unlimited number of times before the deadline expires. This means a pusher's `revokePusher()` call is silently undone by the creator in the same transaction or the next block, making the revocation mechanism ineffective for the entire deadline window.

### Finding Description

The code comment on `allowPushers` explicitly states the deadline's purpose:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The deadline is intended to bound the window during which a signature can be replayed. However, the implementation only checks `block.timestamp <= deadline` via `_ensureDeadline`. There is no tracking of whether a signature has already been consumed, and no check that the pusher is not currently in a revoked state. The signature domain `(chainid, address(this), deadline, pusher, creator)` is identical on every replay call.

Attack sequence:

1. Pusher signs consent for `deadline = T + 1 day` and hands the signature to creator.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator immediately calls `allowPushers(deadline, [pusher], [sig])` again with the **same** signature → `namespaceRemapping[pusher] = creator` again.
5. Steps 3–4 can repeat indefinitely until `block.timestamp > deadline`.

The pusher has no on-chain mechanism to prevent step 4. The only escape is to wait for the deadline to expire.

### Impact Explanation

- The pusher's `revokePusher()` is rendered a no-op for the entire deadline window. Any fallback push the pusher makes after believing they have revoked still lands in the creator's namespace, not their own.
- The pusher's own namespace remains empty/stale. Any pool or price provider reading `feedIdOf(pusher, slot, pos)` receives a zero-price / zero-timestamp feed, which downstream staleness guards treat as an invalid quote.
- The creator can keep a pusher's live price data flowing into their namespace against the pusher's explicit intent, maintaining a feed that the pusher intended to withdraw from.
- This breaks the core oracle invariant that a pusher can unilaterally exit a delegation, which is the only abuse-protection mechanism available to a pusher who discovers their data is being misused.

### Likelihood Explanation

The trigger is fully unprivileged from the creator's side: the creator already holds the signature (they needed it to establish the delegation in the first place), and replaying it costs only gas. No special setup is required. The pusher has no on-chain defense. Likelihood is high whenever a creator has an incentive to keep a pusher's data flowing (e.g., to maintain a live feed for a pool they operate).

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) usedSignatures` and mark each signature hash as used on first acceptance. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, so that a revocation followed by a re-delegation requires a fresh signature from the pusher.

```solidity
// Option A: mark-used
bytes32 sigHash = keccak256(signatures[i]);
require(!usedSignatures[sigHash], SignatureAlreadyUsed());
usedSignatures[sigHash] = true;
```

The root cause is in `allowPushers` at lines 192–212: the function verifies the signature is well-formed and within the deadline but never records that the signature was consumed, so the same bytes can be submitted repeatedly. [1](#0-0) 

The `_ensureDeadline` helper only checks `block.timestamp <= deadline` and provides no replay protection: [2](#0-1) 

The `revokePusher` function clears the mapping but has no way to prevent the creator from immediately re-writing it with the original signature: [3](#0-2)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
