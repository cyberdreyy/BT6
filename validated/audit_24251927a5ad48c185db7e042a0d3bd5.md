### Title
Blacklisted Pool Can Self-Clear Its Blacklist via `register()`, Bypassing the Admin's Emergency Oracle-Read Guard — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

The `register()` function in `OracleBase.sol` (providers) unconditionally clears a pool's blacklist entry whenever the pool pays the registration fee. Because `register()` is permissionless and the default fee is `1 wei`, any pool that the admin has blacklisted can immediately re-enable its own oracle-price read access by calling `register()` again, rendering the blacklist protection ineffective.

---

### Finding Description

`OracleBase` (providers) enforces two blacklist checks on the `price()` read path:

```solidity
// OracleBase.sol (providers) – price()
modifier notBlacklisted() {
    require(!blacklisted[msg.sender], Blacklisted(msg.sender));
    _;
}

function price(bytes32 feedId, address pool)
    external
    feedExists(feedId)
    notBlacklisted          // ← checks msg.sender (price provider)
    returns (...)
{
    require(!blacklisted[pool], Blacklisted(pool));   // ← checks pool
    require(registeredPool[feedId][pool], ...);
    ...
}
``` [1](#0-0) [2](#0-1) 

The `register()` function is the bypass path. It is permissionless (any caller), requires only `msg.value >= registrationFee`, and **unconditionally clears the pool's blacklist** as a side-effect:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;          // ← blacklist silently cleared
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    ...
}
``` [3](#0-2) 

The default registration fee is `1 wei`:

```solidity
registrationFee = 1 wei; // very cheap default; ADMIN tunes via setRegistrationFee
``` [4](#0-3) 

Even if the admin raises the fee, the pool operator can still pay it. The admin has no way to permanently block a pool from clearing its own blacklist short of removing the pool's factory from the approved-factory list (a much more disruptive action).

---

### Impact Explanation

The blacklist is the protocol's emergency oracle-read kill-switch. If a pool is actively executing swaps using a manipulated or stale price feed, the admin blacklists it to stop further price reads. Because `register()` clears the blacklist for the cost of `1 wei` (default), the pool operator can immediately restore oracle access and continue executing swaps at bad prices. This breaks the admin-boundary invariant: an unprivileged path (`register()`) bypasses an admin-controlled guard (`blacklisted[pool]`), potentially allowing continued bad-price execution and direct loss of LP assets or swap settlement funds.

---

### Likelihood Explanation

- The bypass requires only a valid pool address from an approved factory and `>= 1 wei` — both trivially satisfied by any legitimate pool that has been blacklisted.
- No special role, signature, or off-chain coordination is needed.
- The pool operator is directly incentivized to clear the blacklist if they are being blocked from profitable (even if abusive) swaps.

---

### Recommendation

Remove the automatic blacklist-clearing side-effect from `register()`. Blacklist management should be exclusively controlled by `ADMIN_ROLE` via `setBlacklist()`. If re-registration of a previously blacklisted pool is desired, require an explicit admin approval step rather than silently clearing the blacklist on fee payment:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);
    require(pool != address(0));
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);
+   require(!blacklisted[pool], Blacklisted(pool)); // reject blacklisted pools

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

---

### Proof of Concept

1. Admin calls `setBlacklist(poolX, true)` to block `poolX` from reading oracle prices (e.g., due to detected price manipulation).
2. Pool operator calls `register(feedId, poolX, approvedFactory)` with `msg.value = 1 wei`.
3. Inside `register()`: `blacklisted[poolX]` is `true`, so the branch executes `blacklisted[poolX] = false`.
4. `poolX` is now un-blacklisted and `registeredPool[feedId][poolX] = true`.
5. `poolX` calls `price(feedId, poolX)` — both `notBlacklisted` and `!blacklisted[pool]` pass.
6. The oracle returns the price; the pool continues executing swaps at the previously-blocked feed, bypassing the admin's emergency intervention.

The admin can re-blacklist, but the pool can immediately re-call `register()` for `1 wei`, creating an unwinnable race.
