### Title
Single-Step `transferProviderOwnership` Allows Irrecoverable Loss of Provider Control — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`, `AnchoredProviderFactory.sol`)

---

### Summary

`PriceProviderFactory`, `PriceProviderFactoryL2`, and `AnchoredProviderFactory` all implement `transferProviderOwnership` as a single-step, immediately-effective operation. If the current owner supplies a wrong address (typo, zero-equivalent contract, or a contract that cannot call factory functions), ownership is irrecoverably lost in one transaction. The `MetricOmmPoolFactory` already uses a two-step `proposePoolAdminTransfer` / `acceptPoolAdmin` pattern for its own admin role, demonstrating the protocol is aware of the pattern — but the provider factories do not apply it.

---

### Finding Description

All three provider factories implement the same single-step ownership transfer:

```solidity
// PriceProviderFactory.sol L92-102
function transferProviderOwnership(address provider, address newOwner)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));          // only zero-address guard
    address previousOwner = providerOwner[provider];

    providerOwner[provider] = newOwner;       // immediate, no confirmation
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);

    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
```

The identical pattern appears in `PriceProviderFactoryL2` (L95-105) and `AnchoredProviderFactory` (L230-240). The only guard is `require(newOwner != address(0))`, which does not prevent transfer to an EOA the caller does not control, a contract with no relevant interface, or a mistyped address.

`providerOwner` is the sole authority for:
- `grantUpdater` / `revokeUpdater` — controls who may call `setConfidence` to update `confidenceParam` (the spread-shaping parameter consumed by live pool swaps).
- `setSource` (AnchoredProviderFactory only) — swaps the reference oracle the `AnchoredPriceProvider` reads from.
- `transferProviderOwnership` — the only recovery path.

By contrast, `MetricOmmPoolFactory` correctly uses a two-step pattern:

```solidity
// MetricOmmPoolFactory.sol L510-525
function proposePoolAdminTransfer(address pool, address newAdmin) external ... {
    if (newAdmin == address(0)) revert InvalidAdmin();
    if (newAdmin == poolAdmin[pool]) revert InvalidAdmin();
    pendingPoolAdmin[pool] = newAdmin;
    emit PoolAdminTransferProposed(pool, poolAdmin[pool], newAdmin);
}

function acceptPoolAdmin(address pool) external ... {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    poolAdmin[pool] = pending;
    ...
}
```

The provider factories have no equivalent `pendingProviderOwner` / `acceptProviderOwnership` mechanism.

---

### Impact Explanation

`providerOwner` controls `grantUpdater`, which in turn controls who may call `setConfidence` → `PriceProvider.setConfidenceParam`. The `confidenceParam` is a live spread-shaping input consumed during pool swaps. If ownership is transferred to a malicious or uncontrolled address:

1. **`PriceProviderFactory` / `PriceProviderFactoryL2`**: The new owner can grant arbitrary updaters who set `confidenceParam` to extreme values, widening or collapsing the bid/ask spread delivered to pools — constituting bad-price execution.
2. **`AnchoredProviderFactory`**: The new owner can call `setSource` to point the `AnchoredPriceProvider` at a manipulated reference oracle. Although immutable clamp bounds limit the worst-case deviation, the reference price itself shifts within those bounds, enabling systematic mispricing.

In both cases the original owner has no recovery path: there is no `pendingProviderOwner` slot and no cancel function.

---

### Likelihood Explanation

The trigger is an operational error by the provider owner (typo, copy-paste mistake, or sending to a contract address that cannot call factory functions). This is exactly the scenario described in the external report ("Bob calls `transfer_ownership` but does not set the `addr` parameter"). Provider owners are permissionless creators — not a small set of audited multisigs — so the population of potential callers is large and the probability of a mistake is non-negligible. The `require(newOwner != address(0))` guard catches only the trivially wrong case.

---

### Recommendation

Apply the same two-step pattern already used by `MetricOmmPoolFactory` to all three provider factories:

1. Add a `mapping(address provider => address) public pendingProviderOwner` storage slot.
2. Replace the current `transferProviderOwnership` with a `proposeProviderOwnershipTransfer` that writes `pendingProviderOwner[provider] = newOwner` and emits an event.
3. Add `acceptProviderOwnership(address provider)` that requires `msg.sender == pendingProviderOwner[provider]`, then promotes and clears the pending slot.
4. Add `cancelProviderOwnershipTransfer(address provider)` callable by the current owner.

This mirrors the `proposePoolAdminTransfer` / `acceptPoolAdmin` / `cancelPoolAdminTransfer` triad already present in `MetricOmmPoolFactory`.

---

### Proof of Concept

```solidity
// Scenario: owner mistypes newOwner address
address provider = factory.createPriceProvider(oracle, feedId, 0, 1 days, base, quote);
// owner intends to transfer to multisig 0xMULTISIG but pastes wrong address
factory.transferProviderOwnership(provider, address(0xDEAD_WRONG));
// providerOwner[provider] is now 0xDEAD_WRONG — immediately and irrecoverably

// Original owner can no longer:
factory.grantUpdater(provider, updater);       // reverts NotProviderOwner
factory.revokeUpdater(provider, updater);      // reverts NotProviderOwner
factory.transferProviderOwnership(provider, correctAddress); // reverts NotProviderOwner

// If 0xDEAD_WRONG is a malicious actor:
vm.prank(address(0xDEAD_WRONG));
factory.grantUpdater(provider, attacker);
// attacker can now call setConfidence with extreme values,
// feeding manipulated spreads into any pool using this provider

    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
