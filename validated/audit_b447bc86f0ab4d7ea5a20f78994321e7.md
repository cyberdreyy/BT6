### Title
`uint32` Overflow in `_afterTimelock` Allows Pool Admin to Bypass Stop-Loss Timelock Immediately - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

### Summary

The `_afterTimelock` helper in `OracleValueStopLossExtension` casts `block.timestamp + timelock` to `uint32` without overflow protection. When the pool admin sets `timelock = type(uint32).max`, the addition silently wraps and produces an `executeAfter` value that is already in the past, causing `_requireElapsed` to pass immediately. This lets the pool admin execute any pending stop-loss parameter change (drawdown, decay, watermarks) in the same block it was proposed, bypassing the delay that is supposed to give LPs time to exit.

### Finding Description

`_afterTimelock` computes the execution deadline as:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256` and `timelock` is `uint32`; the addition is performed in `uint256` space and then **truncated** to `uint32`. With `block.timestamp ≈ 1,753,000,000` (July 2026) and `timelock = type(uint32).max = 4,294,967,295`:

```
1,753,000,000 + 4,294,967,295 = 6,047,967,295
uint32(6,047,967,295) = 6,047,967,295 - 4,294,967,296 = 1,752,999,999
```

The resulting `executeAfter ≈ block.timestamp - 1`, which is already in the past.

`_requireElapsed` then checks:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

Since `block.timestamp (1,753,000,000) < 1,752,999,999` is **false**, no revert occurs and the proposal executes immediately.

**Attack path:**

1. Pool admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. This is gated by the **current** timelock (correct), so `executeAfter = block.timestamp + currentTimelock`.
2. Admin waits for the current timelock to elapse, then calls `executeOracleStopLossTimelock(pool)`. Now `oracleStopLossConfig[pool].timelock = type(uint32).max`.
3. Admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)` (100% drawdown = stop-loss disabled). `_afterTimelock` overflows → `executeAfter ≈ block.timestamp - 1`.
4. Admin immediately calls `executeOracleStopLossDrawdown(pool)` in the **same block**. `_requireElapsed` passes. `drawdownE6 = 1_000_000` is set.
5. In `_afterSwapOracleStopLoss`, `floorMultiplier = E6 - drawdown = 0`, so `metric < 0` is never true for `uint256` — the stop-loss never triggers again. [3](#0-2) [4](#0-3) 

There is no cap or validation on the `newTimelock` value in either `proposeOracleStopLossTimelock` or `initialize`: [5](#0-4) [6](#0-5) 

### Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain mechanism that blocks swaps causing excessive per-bin value drawdown. Its NatSpec explicitly states: *"Drawdown and decay changes are timelocked so LPs can react."* [7](#0-6) 

By bypassing the timelock, the pool admin can silently disable stop-loss protection in a single block. LPs have no opportunity to observe the pending change and exit. Subsequent swaps that would have been blocked by `OracleStopLossTriggered` now execute freely, draining LP principal at oracle-unfavorable prices. This is a direct loss of LP assets and an admin-boundary break (pool admin bypasses a timelock that is specifically designed to constrain their authority). [8](#0-7) 

### Likelihood Explanation

The pool admin is a semi-trusted role that already controls timelock proposals. The only prerequisite is waiting out the **current** timelock once to install `type(uint32).max`. After that, every subsequent proposal (drawdown, decay, watermarks) bypasses the delay permanently. The attack is invisible to LPs because the emitted `executeAfter` value in `OracleStopLossTimelockProposed` appears to be a very large future timestamp (it overflows only when used in `_afterTimelock` for the *next* proposal, not when stored).

### Recommendation

Add overflow protection in `_afterTimelock` and a maximum-timelock validation in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
// Option A: clamp in _afterTimelock
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    return result > type(uint32).max ? type(uint32).max : uint32(result);
}

// Option B: validate on write (e.g., max 365 days)
uint32 private constant MAX_TIMELOCK = 365 days;
function _validateTimelock(uint32 t) private pure {
    if (t > MAX_TIMELOCK) revert TimelockTooLarge(t);
}
```

Apply `_validateTimelock` in `initialize` and `proposeOracleStopLossTimelock`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {Test} from "forge-std/Test.sol";
import {OracleValueStopLossExtension} from
    "metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol";

contract TestTimelockOverflow is Test {
    OracleValueStopLossExtension ext;
    address pool  = address(0xBEEF);
    address admin = address(0xADDE);
    address factory;

    function setUp() public {
        // deploy with a stub factory that reports admin
        factory = address(this);
        ext = new OracleValueStopLossExtension(factory);
        // initialize with 1-day timelock and 50% drawdown
        ext.initialize(pool, abi.encode(uint32(500_000), uint32(0), uint32(1 days)));
        // wire admin
        // (stub factory exposes poolAdmin mapping)
    }

    function testOverflowBypassesTimelock() public {
        vm.warp(1_753_000_000);

        // Step 1: propose timelock = type(uint32).max (gated by current 1-day timelock)
        vm.prank(admin);
        ext.proposeOracleStopLossTimelock(pool, type(uint32).max);

        // Step 2: wait 1 day, execute
        vm.warp(block.timestamp + 1 days);
        vm.prank(admin);
        ext.executeOracleStopLossTimelock(pool);

        // Step 3: propose drawdown = 0 (disables stop-loss)
        vm.prank(admin);
        ext.proposeOracleStopLossDrawdown(pool, 0);

        // Step 4: execute IMMEDIATELY — no warp needed
        vm.prank(admin);
        ext.executeOracleStopLossDrawdown(pool); // must not revert

        (uint32 dd,,,) = ext.oracleStopLossConfig(pool);
        assertEq(dd, 0, "drawdown disabled without timelock delay");
    }
}
```
