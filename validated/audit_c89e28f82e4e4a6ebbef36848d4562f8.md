### Title
Pusher Self-Revocation Is Ineffective: Creator Can Replay Original Consent Signature to Re-Establish Delegation - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

`revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`, but the creator can immediately re-establish the delegation by replaying the original EIP-191 consent signature via `allowPushers()`. Because no nonce or used-signature registry exists, the same signature is valid for any number of calls until the deadline expires. A pusher's self-revocation is therefore not final: the creator can silently restore write authority over the creator's namespace at any time within the deadline window.

### Finding Description

`allowPushers()` verifies a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The signed message commits to `(block.chainid, address(this), deadline, pusher, msg.sender)`. There is no nonce and no record of consumed signatures. The same `(deadline, pusher, creator, signature)` tuple can be submitted to `allowPushers()` an unlimited number of times as long as `block.timestamp < deadline`.

`revokePusher()` clears the mapping: [2](#0-1) 

But immediately after the pusher calls `revokePusher()`, the creator calls `allowPushers()` with the identical signature that was used originally. `_ensureDeadline` passes (deadline still in the future), ECDSA recovery succeeds (same bytes), and `namespaceRemapping[pusher]` is set back to `creator`. The revocation is undone in a single transaction.

The code comment acknowledges the risk of re-establishment after revocation and claims the deadline is the mitigation: [3](#0-2) 

However, the deadline only prevents re-establishment *after* it expires. During the entire valid window (which operators routinely set to days or weeks), the creator can replay the signature an unlimited number of times, making `revokePusher()` functionally useless against a creator who retains the original signature bytes.

The `fallback()` push path resolves the namespace at call time: [4](#0-3) 

So every push the pusher makes after the creator's replay lands in the creator's namespace, not the pusher's own.

### Impact Explanation

After a pusher self-revokes, the creator replays the old consent to re-establish the delegation. The pusher's subsequent `fallback()` calls continue to overwrite the creator's namespace slots. If the pusher's key is compromised, or if the pusher intentionally pushes manipulated data (e.g., inverted or unbounded prices) in response to being unable to exit, those values land in the creator's live feed. Any `PriceProvider` or `AnchoredPriceProvider` backed by that `CompressedOracleV1` feed will serve the corrupted mid/spread to `MetricOmmPool.swap()`, causing bad-price execution against LP positions.

### Likelihood Explanation

The creator retains the original `signatures[]` bytes from the initial `allowPushers()` call (they submitted the transaction). Re-establishing requires one additional transaction with no new information. Any creator who wants to keep a pusher active against the pusher's will — or whose own account is compromised — can do so for the full deadline window. Deadlines of 1–30 days are operationally normal, giving a wide exploitation window.

### Recommendation

Track consumed consent signatures with a `mapping(bytes32 => bool) usedConsents` keyed on the signature hash (or a `(pusher, creator, deadline)` nonce). Mark the signature as used on the first successful `allowPushers()` call and reject any replay. Alternatively, introduce a per-pusher revocation nonce that is incremented on `revokePusher()` and included in the signed message, so any previously issued signature becomes invalid after revocation.

### Proof of Concept

```
1. Pusher signs consent: keccak256(abi.encode(chainid, oracle, deadline=now+7days, pusher, creator))
2. Creator calls allowPushers(deadline, [pusher], [sig])
   → namespaceRemapping[pusher] = creator
3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)   ← revocation appears complete
4. Creator calls allowPushers(deadline, [pusher], [sig])  ← SAME sig, deadline still valid
   → namespaceRemapping[pusher] = creator       ← delegation silently restored
5. Pusher's fallback() pushes land in creator's namespace
   → creator's feed receives pusher-controlled prices
6. PriceProvider reads creator's feed → bad mid/spread → pool swap executes at wrong price
```

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
```
