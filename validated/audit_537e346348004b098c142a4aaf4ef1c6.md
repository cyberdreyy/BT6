### Title
Delegated Pusher Can Lock Oracle Slot With Far-Future Timestamp, Blocking All Legitimate Price Updates — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

The `CompressedOracleV1.fallback()` push path enforces a strict monotonicity invariant: a slot update is only accepted if its embedded timestamp is strictly greater than the stored one. The only upper bound on the pushed timestamp is `revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT)`. A delegated pusher can therefore push a slot word whose timestamp equals `block.timestamp * 1000 + MAX_TIME_DRIFT` (the maximum allowed future value). Once stored, no subsequent push — including one from the creator themselves — can satisfy the monotonicity check until `block.timestamp` advances by `MAX_TIME_DRIFT / 1000` seconds. This is the direct analog of the external bug: after a time window is set by one party, the other party is locked out of any corrective action.

### Finding Description

`CompressedOracleV1.fallback()` processes each 32-byte calldata word as follows:

```
uint8 slotId = uint8(word);
TimeMs timestampMs = toTimeMs(word >> 8 & X56);
timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);   // upper bound
...
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;                                           // monotonicity gate
_writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
``` [1](#0-0) 

A delegated pusher (authorized via `allowPushers` or `allowContractPushers`) writes into the creator's namespace. If the pusher crafts a word with `timestampMs = block.timestamp * 1000 + MAX_TIME_DRIFT`, the slot is written with the maximum permissible future timestamp. From that point:

- Any push with `timestampMs ≤ stored` fails the `isAfter` monotonicity check and is silently skipped.
- Any push with `timestampMs > stored` would require `block.timestamp * 1000 + MAX_TIME_DRIFT > stored`, which is only true once `block.timestamp` has advanced by `MAX_TIME_DRIFT / 1000` seconds.

The creator cannot override this via `updateBySignature` either — it applies the identical monotonicity check:

```
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) { return false; }
``` [2](#0-1) 

Calling `removePushers` revokes the delegation going forward but does **not** clear the stored far-future timestamp. The slot remains locked for the full `MAX_TIME_DRIFT / 1000` seconds regardless. [3](#0-2) 

### Impact Explanation

**L1 pools** (`PriceProvider`, `AnchoredPriceProvider`, `ProtectedPriceProvider`): The staleness check `if (refTime > nowTs) return true` marks any future `refTime` as stale, causing `getBidAndAskPrice()` to revert with `FeedStalled`. All swaps through the affected pool are blocked for the lock duration. [4](#0-3) 

**L2 pools** (`PriceProviderL2`, `ProtectedPriceProviderL2`): The L2 staleness check tolerates timestamps slightly in the future (within `FUTURE_TOLERANCE`). If `MAX_TIME_DRIFT / 1000 ≤ FUTURE_TOLERANCE`, the locked-in price is accepted as fresh. The malicious pusher can therefore lock in an arbitrary price (within the codebook's representable range) and execute swaps against the pool at that price for the entire lock duration — a bad-price execution causing direct loss of LP principal. [5](#0-4) 

### Likelihood Explanation

The trigger is a delegated pusher — a semi-trusted entity explicitly authorized by the creator via `allowPushers` (EIP-191 signature) or `allowContractPushers` (live `isPusher` call). This is a valid semi-trusted trigger per the impact gate. The attack requires no special on-chain state beyond an active delegation and a non-zero `MAX_TIME_DRIFT`. The creator cannot remediate the locked slot immediately after revoking the pusher; they must wait for the lock to expire. [6](#0-5) 

### Recommendation

1. **Cap the stored timestamp at `block.timestamp * 1000`** (no future tolerance in the stored value). Accept pushes with timestamps up to `MAX_TIME_DRIFT` ahead for liveness, but clamp the written value to `min(timestampMs, block.timestamp * 1000)` before storage. This preserves clock-skew tolerance without allowing a pusher to advance the monotonicity cursor into the future.

2. **Alternatively**, add a creator-controlled emergency reset path that allows the creator (decoded from the feedId) to zero the slot's timestamp, breaking the monotonicity lock without requiring a full `MAX_TIME_DRIFT` wait.

### Proof of Concept

```solidity
// Assume: MAX_TIME_DRIFT = 60_000 ms (60 s), FUTURE_TOLERANCE = 60 s on L2
// Attacker = delegated pusher for creator's namespace

// Step 1: pusher crafts a slot word with timestamp = block.timestamp*1000 + 60_000
uint56 futureTs = uint56(block.timestamp * 1000 + 60_000);
uint48 raw = (uint48(manipulatedPrice) << 16) | (uint48(s0) << 8) | uint48(s1);
uint256 word = (uint256(futureTs) << 8) | uint256(slotId);
word |= uint256(raw) << (208 - uint256(pos) * 48);

// Step 2: push — passes revertIfAfterBlockTimeWithDrift(60_000), stored
(bool ok,) = address(oracle).call(abi.encodePacked(word));
assert(ok);

// Step 3: creator tries to push a corrected price — fails monotonicity
uint56 nowTs = uint56(block.timestamp * 1000); // < futureTs
// isAfter(nowTs, futureTs) == false → skipped silently

// Step 4 (L2): pool reads the locked-in price, FUTURE_TOLERANCE=60s accepts it as fresh
// Swaps execute at manipulatedPrice for 60 seconds
// LPs lose the spread between manipulatedPrice and true market price
``` [7](#0-6) 

**Caveat**: This finding is conditional on `MAX_TIME_DRIFT > 0` in the production deployment of `CompressedOracleV1`. The test suite deploys with `maxTimeDrift = 0`, which would make this attack impossible. If production deployments also use `maxTimeDrift = 0`, no vulnerability exists. The L2 bad-price variant additionally requires `MAX_TIME_DRIFT / 1000 ≤ FUTURE_TOLERANCE`. These parameter values could not be verified from the available indexed source.

    fallback() override external {
        uint256 end;
        uint256 namespace;

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

        assembly ("memory-safe") {
            end := calldatasize()
            namespace := shl(96, creator) // [creator:20][zeros:12]
        }

        // 4 * 6 + 7 + 1 = 32 bytes per slot
        if (end == 0 || end % 32 != 0) revert BadCalldataLength();

        for (uint256 ptr = 0; ptr < end; ptr += 32) {
            uint256 word;
            assembly ("memory-safe") {
                word := calldataload(ptr)
            }
            // casting to 'uint8' is safe we want LSB
            // forge-lint: disable-next-line(unsafe-typecast)
            uint8 slotId = uint8(word);
            TimeMs timestampMs = toTimeMs(word >> 8 & X56);
            timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
            bytes32 key = bytes32(namespace | uint256(slotId));
            uint256 old = uint256(_loadStorage(key));
            TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

            bool newer = timestampMs.isAfter(oldTimestampMs);
            if (!newer) continue;

            _writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
