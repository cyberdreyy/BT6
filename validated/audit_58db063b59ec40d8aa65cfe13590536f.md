After a thorough trace through the oracle provider, compression, pusher delegation, codebook, timestamp, and abuse-protection code, I find no valid analog.

**Encoding/decoding path (`_loadSlotLayout`, `_decodeCompressedOracleData`):** The 256-bit slot is correctly partitioned — oracle[0..3] at bits [255:208], [207:160], [159:112], [111:64], timestamp at bits [63:8], reserved at bits [7:0]. The bit-shift/mask arithmetic in `_loadSlotLayout` and `_decodeCompressedOracleData` is consistent with the documented layout. No lane bleeds into another. [1](#0-0) 

**Stalled-sentinel path (`getOracleData`):** When `s0 == 0xff && s1 == 0xff`, the early return leaves `data.timestampMs = TimeMs.wrap(0)` and `data.price = 0`. `_price()` therefore returns `refTime = 0`. Every downstream provider's `_isStale` check treats `refTime == 0` as unconditionally stale, so the sentinel correctly halts all consumers before any bid/ask is computed. [2](#0-1) [3](#0-2) 

**Never-pushed sibling lane:** A lane in a slot that was pushed for a different position reads `price = 0`, `spread0 = 0`, `spread1 = 0`, but inherits the slot's fresh timestamp. The `mid == 0` guard in every provider (`if (mid == 0 || spreadBps >= ORACLE_BPS)`) catches this before any bid/ask is produced. [4](#0-3) 

**Inline assembly (`memory-safe` annotation):** All assembly blocks (`_loadSlotLayout`, `_writeStorage`, `fallback`, `_oracleSlot`) perform only `sload`/`sstore`/`calldataload`/arithmetic — no memory reads or writes. The `memory-safe` annotation is correct; the optimizer-bug analog from the external report (memory side-effects of inline assembly) has no foothold here. [5](#0-4) 

**Delegation paths (`allowPushers`, `allowContractPushers`):** Signatures cover `(chainid, address(this), deadline, pusher, msg.sender)`, preventing cross-chain and cross-contract replay. The `allowContractPushers` live `isPusher` staticcall is checked at delegation time; subsequent pushes write only to the creator's namespace. No unprivileged path can redirect writes to a foreign namespace. [6](#0-5) [7](#0-6) 

**`updateBySignature` replay:** Monotonicity is checked before signature verification (gas optimisation), but the signature still covers `(chainid, address(this), feedCreator, newSlotValue)` including the embedded timestamp. A replayed word with an older timestamp is rejected by the monotonicity check; a replayed word with the same timestamp is also rejected (`isAfter` is strict). No stale data can be injected.
