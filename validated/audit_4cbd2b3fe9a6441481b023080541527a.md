### Title
Authorized Pusher Can Submit a Future-Timestamped Price to `CompressedOracleV1`, Locking Out Legitimate Updates and Causing Stale-Price Execution — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1`'s push path accepts timestamps up to `MAX_TIME_DRIFT` seconds in the future. Because the monotonicity guard compares raw millisecond values, a single push with the maximum-allowed future timestamp freezes the slot for the entire drift window. After that window elapses, the stale price data appears perfectly fresh to every downstream consumer (`AnchoredPriceProvider`, `PriceProvider`), bypassing their staleness checks and delivering a bad price to pool swaps.

---

### Finding Description

**Step 1 — Future timestamp is accepted.**

`revertIfAfterBlockTimeWithDrift` in `TimeMs.sol` converts the pushed millisecond timestamp to seconds via integer division before comparing it to `block.timestamp + drift`:

```solidity
function revertIfAfterBlockTimeWithDrift(TimeMs t0, uint256 drift) view {
    require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());
}
// toSeconds = TimeMs.unwrap(t) / 1000  (floor division)
``` [1](#0-0) 

A pusher may therefore submit `timestampMs = (block.timestamp + MAX_TIME_DRIFT) * 1000 + 999`. The floor division yields exactly `block.timestamp + MAX_TIME_DRIFT`, which satisfies the `<=` bound and the push is accepted.

**Step 2 — Monotonicity lock freezes the slot.**

The fallback loop in `CompressedOracleV1` compares raw millisecond values:

```solidity
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;
_writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
``` [2](#0-1) 

Any subsequent legitimate push carrying the actual current time (e.g., `T+1` seconds → `(T+1)*1000` ms) is strictly less than the stored `(T + MAX_TIME_DRIFT)*1000 + 999` ms and is silently skipped. The slot is frozen for the entire `MAX_TIME_DRIFT` window.

The same monotonicity check exists in `updateBySignature`: [3](#0-2) 

**Step 3 — Stale price appears fresh after the drift window.**

`_readPrice` in the providers `OracleBase` converts the stored millisecond timestamp to seconds:

```solidity
return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
``` [4](#0-3) 

`CompressedOracleV1._readPrice` does the same: [5](#0-4) 

So `refTime = floor(((T + MAX_TIME_DRIFT)*1000 + 999) / 1000) = T + MAX_TIME_DRIFT`.

`AnchoredPriceProvider._isStale` (and the identical copy in `PriceProvider`) is:

```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;          // future → stale
    return (nowTs - refTime) > maxDelta;
}
``` [6](#0-5) 

At time `T + MAX_TIME_DRIFT`, `nowTs = refTime`, so `refTime > nowTs` is false and `(nowTs − refTime) = 0 ≤ MAX_REF_STALENESS`. The price passes as fresh regardless of `MAX_REF_STALENESS`, even though the underlying market data is `MAX_TIME_DRIFT` seconds old and legitimate updates were blocked throughout that window.

---

### Impact Explanation

A malicious or compromised authorized pusher can:

1. Push attacker-chosen price data (mid, spread) with timestamp `T + MAX_TIME_DRIFT`.
2. Block every legitimate update for `MAX_TIME_DRIFT` seconds (monotonicity lock).
3. At `T + MAX_TIME_DRIFT`, the stale price is consumed by pool swaps as if it were current.

If the true market price moved during the lock window, the pool executes swaps at the wrong price: traders receive more output than the oracle curve permits (direct loss of LP principal) or the pool receives less input than owed (swap conservation failure). The `AnchoredPriceProvider` band clamp provides partial mitigation only if the stale price falls outside `MAX_SPREAD_BPS`; a small but deliberate price skew passes the clamp. [7](#0-6) 

---

### Likelihood Explanation

The trigger is an authorized pusher (creator or delegated wallet). Delegation is permissionless once the creator signs consent via `allowPushers`. A compromised pusher key, a malicious delegatee, or a creator acting adversarially can execute this with a single calldata push — no special on-chain setup beyond existing authorization. The attack is repeatable every `MAX_TIME_DRIFT` seconds. [8](#0-7) 

---

### Recommendation

Replace the `<=` bound in `revertIfAfterBlockTimeWithDrift` with a strict `<` bound, or compare in milliseconds rather than after floor-division, so that no timestamp strictly in the future can be stored:

```solidity
// current (vulnerable):
require(t0.toSeconds() <= block.timestamp + drift, FutureTimestamp());

// fixed: reject any ms value whose second-floor exceeds block.timestamp
require(t0.toSeconds() <= block.timestamp, FutureTimestamp());
// or, if a small drift is genuinely needed, keep drift but compare in ms:
require(TimeMs.unwrap(t0) <= (block.timestamp + drift) * 1000, FutureTimestamp());
```

This ensures the stored `refTime` can never exceed `block.timestamp` at push time, so the monotonicity lock cannot extend beyond the current block and the staleness check in downstream providers cannot be bypassed. [1](#0-0)

// Scenario (pseudocode):
// MAX_TIME_DRIFT = 60 seconds (as used in OracleBaseAbuseProtection.t.sol)
// MAX_REF_STALENESS = 30 seconds (AnchoredPriceProvider)

// 1. At block.timestamp = T, attacker (authorized pusher) crafts a slot word:
//    timestampMs = (T + 60) * 1000 + 999   ← 60 s 999 ms in the future
//    price data  = attacker-chosen bad price P_bad
//    slotId      = 0

uint256 T = block.timestamp;
uint256 badTimestampMs = (T + 60) * 1000 + 999;
uint256 badPrice = /* attacker-chosen value */;
uint256 word = (badPrice << 64) | (badTimestampMs << 8) | 0; // slotId=0

// 2. Attacker calls oracle fallback with this word.
//    revertIfAfterBlockTimeWithDrift: floor(badTimestampMs/1000) = T+60 <= T+60 ✓
//    Stored: timestampMs = (T+60)*1000+999, price = P_bad

(bool ok,) = address(oracle).call(abi.encodePacked(word));
assert(ok);

// 3. Legitimate pusher tries to update at T+1 with current price P_real:
//    timestampMs = (T+1)*1000 < (T+60)*1000+999 → isAfter = false → skipped silently.

// 4. At block.timestamp = T+60, AnchoredPriceProvider reads:
//    refTime = floor(((T+60)*1000+999)/1000) = T+60
//    nowTs   = T+60
//    _isStale: refTime > nowTs? No. (nowTs-refTime)=0 <= MAX_REF_STALENESS ✓
//    → P_bad is delivered to the pool swap as a fresh price.
//    → Pool executes at P_bad while true market price is P_real.
