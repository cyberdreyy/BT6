### Title
`allowPushers()` Does Not Consume the Pusher Signature After Use, Allowing Creator to Re-Establish Revoked Delegation Within the Deadline Window - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

`CompressedOracleV1.allowPushers()` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher self-revokes via `revokePusher()`, the creator can replay the original signature (while the deadline is still valid) to silently re-establish the delegation, routing the pusher's future fallback writes back into the creator's namespace against the pusher's will.

### Finding Description

`allowPushers()` enforces two guards: a deadline check and an ECDSA signature recovery. The signature domain is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

After the call succeeds, `namespaceRemapping[pusher] = msg.sender` is written and the function returns. The signature is **not invalidated, consumed, or tracked**. No nonce or "used-signature" bitmap exists anywhere in the contract. [2](#0-1) 

When the pusher later calls `revokePusher()`, `namespaceRemapping[pusher]` is cleared to `address(0)`: [3](#0-2) 

But the creator still holds the original signature. As long as `block.timestamp <= deadline`, the creator can call `allowPushers()` again with the **identical** `(deadline, [pusher], [sig])` arguments. The deadline check passes, the ECDSA recovery succeeds (the signature is still cryptographically valid), and `namespaceRemapping[pusher]` is set back to `msg.sender`. The revocation is silently undone.

The code comment itself acknowledges the deadline is the only replay barrier: [4](#0-3) 

But the deadline only prevents replays **after** it expires. Within the window it does not prevent the creator from re-establishing a delegation the pusher explicitly revoked.

This is the direct analog of the `diamondCut()` bug: `ds.acceptanceTimes[hash]` not being reset to zero after execution allows re-execution without delay. Here, the signature not being consumed after use allows re-delegation without the pusher's renewed consent.

### Impact Explanation

After a pusher revokes, their `fallback()` pushes are supposed to land in their **own** namespace: [5](#0-4) 

If the creator replays the old signature, those pushes are silently redirected back into the creator's namespace. Any pool whose `AnchoredPriceProvider` or `PriceProvider` reads from the creator's namespace will continue to receive the pusher's data — data the pusher intended to stop supplying. Conversely, the pusher's own namespace (which may be used by other pools) receives no new data, causing those pools to see a stale `refTime` and halt with `FeedStalled`. [6](#0-5) 

The broken invariant is: **a pusher who revokes their delegation must be able to stop contributing to the creator's namespace**. The creator can bypass this within the deadline window without any new consent from the pusher.

### Likelihood Explanation

- The creator already holds the signature (they submitted it in the original `allowPushers()` call).
- Deadlines are typically set days in the future (tests use `block.timestamp + 1 days`).
- The creator has a direct economic incentive to keep a live pusher feeding their namespace.
- No privileged access is required — the creator is a regular user, not an oracle admin.
- The attack is a single transaction replay with no additional cost. [7](#0-6) 

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) public usedSignatures` and mark each signature hash as used on first acceptance:

```solidity
bytes32 sigHash = keccak256(signatures[i]);
require(!usedSignatures[sigHash], "signature already consumed");
usedSignatures[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed domain so each consent is single-use:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]++))
```

This mirrors the fix recommended for `diamondCut()`: reset the acceptance state to zero after execution so the same approval cannot be replayed.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with deadline = now + 1 day
bytes memory sig = sign(PUSHER_KEY, keccak256(abi.encode(
    block.chainid, address(oracle), deadline, pusher, creator
)));

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher revokes — they no longer want to feed creator's namespace
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature (deadline still valid)
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator again — revocation bypassed ✗

// 5. Pusher's next fallback push lands in creator's namespace, not their own
vm.prank(pusher);
address(oracle).call(wordAt(slotId, pos, raw, tsMs));
// oracle.getOracleData(feedIdOf(creator, slotId, pos)).price != 0  ← creator's namespace updated
// oracle.getOracleData(feedIdOf(pusher,  slotId, pos)).price == 0  ← pusher's own namespace empty
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
