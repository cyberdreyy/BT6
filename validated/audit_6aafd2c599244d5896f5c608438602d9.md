The vulnerability is real. Let me trace through the exact code path.

**`allowPushers` has no nonce and no used-signature tracking:** [1](#0-0) 

The signed message is `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`. There is no nonce, no per-signature consumed flag, and no state that changes between the first and second call. The only guard is `_ensureDeadline(deadline)` — `block.timestamp <= deadline`.

**`revokePusher` zeroes the mapping but cannot invalidate the off-chain signature:** [2](#0-1) 

After `revokePusher()` sets `namespaceRemapping[pusher] = address(0)`, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple. `_ensureDeadline` passes (deadline is still in the future), `ECDSA.recover` returns the same pusher address, and `namespaceRemapping[pusher] = msg.sender` is written again. The revocation is silently undone.

The code's own comment acknowledges the intent of the deadline: [3](#0-2) 

> "an undated signature could re-establish a delegation AFTER the pusher revoked it"

But the deadline only prevents replay *after* it expires — it does nothing within the window.

**The `fallback` push path reads `namespaceRemapping` at call time:** [4](#0-3) 

So after the replay, any push from the pusher lands in the creator's namespace, not the pusher's own namespace.

### Title
Signature replay in `allowPushers()` lets a creator re-establish pusher delegation after `revokePusher()`, violating pusher-revocation finality and enabling oracle namespace corruption — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` accepts an EIP-191 signature that commits to `(chainid, oracle, deadline, pusher, creator)` but includes no nonce and tracks no consumed signatures. A creator who holds a valid signature can call `allowPushers` repeatedly within the deadline window, re-writing `namespaceRemapping[pusher] = creator` every time the pusher calls `revokePusher()`.

### Finding Description
1. Pusher signs consent for creator with `deadline = block.timestamp + N days`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator calls `allowPushers(deadline, [pusher], [sig])` again with the **identical arguments** → `namespaceRemapping[pusher] = creator` again.

Steps 3–4 can be repeated indefinitely until `block.timestamp > deadline`. The pusher has no on-chain mechanism to permanently revoke within the deadline window.

After the replay, the `fallback()` push path resolves `creator = namespaceRemapping[msg.sender]` and writes the pusher's slot data into the creator's namespace. If the pusher has since started pushing data for their own feeds (different assets or prices), that data is silently redirected into the creator's namespace, corrupting the oracle slots consumed by pools.

### Impact Explanation
Any pool configured to read a `feedId` from the creator's namespace receives the pusher's redirected data. If the pusher's data is for a different asset or at a different scale, the pool's `AnchoredPriceProvider` consumes a bad mid-price and spread, leading to bad-price execution: swaps settle at incorrect rates, causing direct loss of user principal or pool insolvency. The oracle data corruption is silent — no event distinguishes a legitimate push from a replayed-delegation push.

### Likelihood Explanation
The creator already holds the signature from the initial delegation. Re-calling `allowPushers` costs one transaction and requires no additional off-chain material. Any creator with an economic incentive to keep a pusher's data flowing (e.g., to maintain pool liquidity or avoid stale-price reverts) will exploit this. The pusher has no recourse until the deadline expires.

### Recommendation
Track consumed signatures with a `mapping(bytes32 => bool) usedSignatures` and revert if the hash has been seen before. Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, so `revokePusher` can also increment the nonce, permanently invalidating all prior signatures.

function test_replayAfterRevoke() public {
    uint256 deadline = block.timestamp + 1 days;
    // pusher signs (chainid, oracle, deadline, pusher, creator)
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(pusherKey, hash);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers = new address[](1); pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1); sigs[0] = sig;

    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    vm.prank(creator);

    vm.prank(pusher);
