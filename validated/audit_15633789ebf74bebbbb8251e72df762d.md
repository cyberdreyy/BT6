### Title
`protocolUnpausePool` Leaves Pool Swap-Paused at Level 1, Requiring Mandatory Extra Admin Action to Restore Swaps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

When the factory owner calls `protocolUnpausePool`, the pool transitions from pause level 2 to level 1 — not to level 0 (active). Because `swap()` guards on `pauseLevel != 0`, swaps remain blocked after the "unpause." If the pool was originally active (level 0) before the protocol pause, the pool admin is forced to take an extra action they never initiated, and swaps remain unavailable during that window.

---

### Finding Description

`MetricOmmPoolFactory.protocolUnpausePool` sets the pool to level 1, not 0: [1](#0-0) 

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← still paused
}
```

`MetricOmmPool.swap` carries `whenNotPaused`, which halts on any non-zero `pauseLevel`: [2](#0-1) [3](#0-2) 

So after `protocolUnpausePool`, every `swap()` call still reverts with `PoolPaused`. The only path back to level 0 is the pool admin calling `unpausePool`: [4](#0-3) 

The critical asymmetry is in `protocolPausePool`, which accepts **either** level 0 or level 1 as a valid starting state: [5](#0-4) 

```solidity
if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
```

This means a pool that was **never admin-paused** (level 0) can be protocol-paused to level 2, then "protocol-unpaused" to level 1 — a state the admin never set and must now actively clear. During the gap between `protocolUnpausePool` and the admin's `unpausePool`, all swaps are blocked.

The pause-level state machine is:

```
Active (0) ──[protocolPausePool]──► Protocol-Paused (2)
                                          │
                                  [protocolUnpausePool]
                                          │
                                          ▼
                                  Admin-Paused (1)   ← swaps still blocked
                                          │
                                    [unpausePool]
                                          │
                                          ▼
                                      Active (0)
```

---

### Impact Explanation

Swaps are a core pool function. After the factory owner calls `protocolUnpausePool` on a pool that was originally active, the pool remains swap-paused at level 1 until the pool admin separately calls `unpausePool`. During this window, no user can execute a swap, breaking the core trading flow. `removeLiquidity` is unaffected (no `whenNotPaused` guard), so LP principal is not at direct risk, but the pool is functionally unusable for trading. [6](#0-5) 

---

### Likelihood Explanation

Requires the factory owner (trusted but semi-privileged) to pause and then call `protocolUnpausePool`. This is a realistic operational scenario (emergency pause followed by a "clear" from the protocol side), making the likelihood low-to-medium. The admin may be unaware that a separate `unpausePool` call is required, especially if they never paused the pool themselves.

---

### Recommendation

When `protocolUnpausePool` is called on a pool that was originally at level 0 (i.e., the protocol pause overrode an active pool), restore it directly to level 0. One approach: record the pre-pause level in `protocolPausePool` and restore it in `protocolUnpausePool`. Alternatively, add a direct `protocolFullUnpausePool` path that transitions 2 → 0 when the pool was not admin-paused before the protocol pause.

---

### Proof of Concept

1. Pool is at `pauseLevel = 0` (active, never admin-paused).
2. Factory owner calls `protocolPausePool(pool)` → `pauseLevel = 2`.
3. Factory owner calls `protocolUnpausePool(pool)` → `pauseLevel = 1` (still paused).
4. Any user calls `pool.swap(...)` → reverts `PoolPaused` because `pauseLevel != 0`.
5. Pool admin must call `factory.unpausePool(pool)` → `pauseLevel = 0` to restore swaps.

Between steps 3 and 5, the pool is swap-dead despite the protocol owner having "unpaused" it.
