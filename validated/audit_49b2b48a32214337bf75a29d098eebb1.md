### Title
Pusher Delegation Replay: Creator Can Re-Establish Revoked Delegation Using Stale Signature — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `allowPushers` function in `CompressedOracleV1` uses an EIP-191 signature scheme with no nonce. A creator who holds a pusher's signed consent can replay that same signature any number of times before the deadline expires — including immediately after the pusher calls `revokePusher()`. This traps the pusher in the creator's namespace for the full lifetime of the deadline, preventing them from stopping their automated price pushes from landing in the creator's feed slots, which a malicious creator can then overwrite with bad prices.

---

### Finding Description

`allowPushers` verifies the pusher's consent over the tuple:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no used-signature bitmap, and no per-pusher revocation counter. The only replay barrier is the deadline, which only prevents use *after* expiry — it does nothing to prevent the creator from calling `allowPushers` again with the identical `(deadline, pusher, sig)` triple *before* expiry.

The code comment explicitly acknowledges the risk but misidentifies the deadline as the fix:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

`revokePusher()` sets `namespaceRemapping[pusher] = address(0)`: [3](#0-2) 

But the creator can immediately call `allowPushers` again with the same signature, writing `namespaceRemapping[pusher] = creator` again: [4](#0-3) 

The `fallback()` push path resolves the namespace from `namespaceRemapping[msg.sender]`, so every subsequent automated push from the pusher lands in the creator's slots: [5](#0-4) 

The creator can then overwrite those slots with a bad price at a newer timestamp, since the only write guard is monotonic timestamp ordering: [6](#0-5) 

The `CompressedOracleV1.price()` read path is fully permissionless — no pool registration, no blacklist, no in-swap binding — so any pool consuming this feed receives the bad price directly: [7](#0-6) 

---

### Impact Explanation

A pusher who discovers a creator is malicious and calls `revokePusher()` cannot stop their automated keeper from continuing to push into the creator's namespace. The creator replays the original signature in the same block, re-establishing the delegation. The creator then pushes a bad price (any value, any spread) with a timestamp one millisecond newer than the pusher's last legitimate update. Every pool reading that feed via `price()` receives the manipulated quote and executes swaps at the wrong rate, causing direct loss of user principal.

---

### Likelihood Explanation

- The pusher must have signed a delegation with a deadline far enough in the future that the creator still has replay window after the pusher tries to revoke. Deadlines of days or weeks are operationally normal for keeper setups.
- The creator is a semi-trusted actor (namespace owner), not a fully privileged oracle admin, so this is a valid semi-trusted trigger.
- No special on-chain state is required beyond the original `allowPushers` call having succeeded.
- The replay is a single public transaction with no gas barrier.

---

### Recommendation

Add a per-pusher nonce to the signed message and increment it on every successful `allowPushers` call. A used-signature bitmap keyed on the full hash also works. Either approach makes every consent single-use:

```solidity
mapping(address => uint256) public pusherNonce;

// inside allowPushers loop:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;        // <-- consume nonce
namespaceRemapping[pusher] = msg.sender;
```

The pusher must include the current nonce when signing. After `revokePusher()` the nonce has already been incremented, so the old signature is permanently invalid.

---

### Proof of Concept

```
T=0   Pusher signs: keccak256(chainid, oracle, deadline=T+30d, pusher, creator)
T=0   Creator calls allowPushers(T+30d, [pusher], [sig])
         → namespaceRemapping[pusher] = creator  ✓

T=1   Pusher's keeper pushes price P1 (legitimate) → lands in creator's slot

T=2   Pusher discovers creator is malicious
T=2   Pusher calls revokePusher()
         → namespaceRemapping[pusher] = address(0)

T=2   Creator calls allowPushers(T+30d, [pusher], [sig])  ← SAME sig, no revert
         → namespaceRemapping[pusher] = creator  ← delegation restored

T=3   Pusher's keeper pushes price P2 (legitimate) → still lands in creator's slot
T=3   Creator calls fallback() with price P_bad, timestamp = P2.ts + 1ms
         → creator's slot now holds P_bad

T=3   Pool calls price(feedId, pool) → receives P_bad → swap executes at wrong rate
         → user funds lost

Repeat T=2 cycle for the full 30-day deadline window.
```
