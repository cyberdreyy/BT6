### Title
`priceGuard` bounds are stored and configurable but never enforced in any push or read path — out-of-bounds prices reach pools unchecked - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`OracleBase` exposes `setPriceGuard(feedId, minPrice, maxPrice)` so a feed creator can bound the prices their feed may carry. However, neither the two write paths (`fallback`, `updateBySignature`) nor the three read paths (`getOracleData`, `_price`, `price`) ever consult the stored `priceGuard`. The guard is a dead feature: a compromised delegated pusher can write any price into the oracle and it will be returned verbatim to pools, enabling bad-price execution.

### Finding Description

`OracleBase` stores a per-feed price guard: [1](#0-0) 

It is set via `setPriceGuard`, gated by `checkRole` (creator or ADMIN): [2](#0-1) 

The guard is confirmed reachable by the creator in tests: [3](#0-2) 

**Write path 1 — `fallback()`**: checks only timestamp monotonicity and future-drift; no `priceGuard` read before `_writeStorage`: [4](#0-3) 

**Write path 2 — `updateBySignature()`**: checks timestamp, monotonicity, and creator signature; no `priceGuard` read before `_writeStorage`: [5](#0-4) 

**Read path — `getOracleData()`**: decodes and returns the raw stored price with no guard check: [6](#0-5) 

**Read path — `_price()` / `price()`**: calls `getOracleData` and returns the result directly to the pool's price provider: [7](#0-6) 

The same omission exists in the providers oracle: `OracleBase._readPrice()` returns `_oracleDataRaw(feedId)` with no guard check, and `ChainlinkOracle._store()` writes verified report data with no guard check: [8](#0-7) [9](#0-8) 

The analog to the ERC777 bug is exact: `transfer()`/`send()`/`burn()` checked the lock period but `operatorSend()`/`operatorBurn()`/`transferFrom()` did not. Here, `setPriceGuard` establishes the guard, but every path that reads or writes price data — `fallback`, `updateBySignature`, `getOracleData`, `_price`, `price` — bypasses it entirely.

### Impact Explanation

A delegated pusher whose key is compromised can call `fallback()` with a slot word encoding an arbitrarily low or high price. The timestamp monotonicity check is the only gate; if the attacker's timestamp is newer than the stored one, the write succeeds. The price is then returned verbatim by `price(feedId, pool)` to the pool's price provider, which passes it to `MetricOmmPool.swap()`. The pool executes bin traversal at the manipulated bid/ask, causing traders to receive more output than the oracle curve permits (swap conservation failure) or LPs to suffer insolvency if the pool pays out reserves at a price far from fair value. This matches the **bad-price execution** and **swap conservation failure** impact categories.

### Likelihood Explanation

The trigger is a compromised delegated pusher — a semi-trusted party. The creator controls delegation via `allowPushers` / `allowContractPushers` / `removePushers`, but the window between compromise and revocation is exploitable. The `priceGuard` was presumably added to bound exactly this risk. Since it is never enforced, the protection is illusory and the attack surface is as wide as the set of all delegated pushers across all feeds. [10](#0-9) 

### Recommendation

Enforce the guard in `CompressedOracle.getOracleData()` (or equivalently in `_price()`) immediately after decoding the price:

```solidity
PriceGuard memory guard = priceGuard[feedId];
if (guard.max > 0) {
    require(
        data.price >= guard.min && data.price <= guard.max,
        PriceOutOfGuard(feedId, data.price)
    );
}
```

Alternatively, enforce it at write time in both `fallback()` and `updateBySignature()` to reject out-of-bounds prices at ingestion, preventing stale bad data from persisting in storage.

### Proof of Concept

1. Creator calls `oracle.setPriceGuard(feedId, 1_000_000, 2_000_000)` — expecting prices to stay in `[1M, 2M]`.
2. Creator delegates a pusher via `allowPushers(deadline, [pusher], [sig])`.
3. Pusher key is compromised; attacker constructs a 32-byte slot word with `U64x32`-encoded price = `50` (far below `min`) and a timestamp newer than the stored one.
4. Attacker calls `address(oracle).call(slotWord)` — the `fallback()` checks only timestamp monotonicity, finds the timestamp newer, and calls `_writeStorage(key, ...)` with the manipulated price. No `priceGuard` check occurs.
5. Pool calls `oracle.price(feedId, pool)` → `_price(feedId)` → `getOracleData(feedId)` → returns `mid = 50`. No `priceGuard` check occurs.
6. `MetricOmmPool.swap()` uses `mid = 50` as the oracle price, executing swaps at a price 20,000× below the intended floor, draining pool reserves or delivering excess output to the attacker.

        (address creator, uint8 slotIndex, uint8 positionIndex) = _unpackFeedId(feedId);

        SlotLayout memory _layout = _loadSlotLayout(_oracleSlot(creator, slotIndex));
        CompressedOracleData memory compressed = _selectCompressedData(_layout, positionIndex);

            data.spread1 = BPS_BASE;
            data.spread0 = BPS_BASE;

        data.spread0 = _decodeCodebookIndex(compressed.s0);
        data.spread1 = _decodeCodebookIndex(compressed.s1);
        data.timestampMs = _layout.timestampMs;
    }

            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;

        }
