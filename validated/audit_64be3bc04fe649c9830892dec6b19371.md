### Title
`spread1` Oracle Uncertainty Component Silently Discarded in `AnchoredPriceProvider._readLeg()`, Producing Unclamped Narrow Bid/Ask Band — (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`CompressedOracleV1.price()` returns four values: `(mid, spread0, spread1, refTime)`. In `AnchoredPriceProvider._readLeg()`, the third return value — `spread1` — is silently discarded with a blank `,`. The band calculation in `_computeBidAsk()` therefore uses only `spread0` as `spreadBps`, ignoring the additional uncertainty component encoded in `spread1`. The resulting bid/ask band is narrower than the oracle's actual declared uncertainty, allowing swaps to execute at prices the oracle never sanctioned.

---

### Finding Description

`CompressedOracleV1` stores two spread components per feed lane: `s0` and `s1`, both decoded from `Codebook256` and surfaced through `getOracleData()`: [1](#0-0) 

The `_price()` internal function returns all four fields — `mid`, `spread0`, `spread1`, `refTime`: [2](#0-1) 

The `ICompressedOracleV1` interface explicitly defines both `s0` and `s1` as spread indices in the codebook: [3](#0-2) 

In `AnchoredPriceProvider._readLeg()`, the oracle is called and `spread1` (the third return value) is silently dropped: [4](#0-3) 

Specifically, line 280:
```solidity
(mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
```

`spreadBps` receives only `spread0`. `spread1` is never captured, never added to `spreadBps`, and never reaches `_computeBidAsk()`. The band is then computed as: [5](#0-4) 

```solidity
uint256 half = spreadBps * ONE_BPS_E18 + minMargin;
```

`spread1` contributes zero to `half`. The reference band is narrower than the oracle's full declared uncertainty.

The same discard pattern appears in `_getBidAndAskPrice()` for both the base and quote legs: [6](#0-5) 

`ProtectedPriceProvider` has the identical defect: [7](#0-6) 

This is the direct analog of the external report's bug class: a value is accumulated and returned by the oracle subsystem (`spread1`) but is never retrieved or utilized in any subsequent pricing logic, resulting in a systematically under-wide band.

---

### Impact Explanation

The `AnchoredPriceProvider` is described as "the one standard provider for public pools." Its core safety guarantee is that every quote is clamped to `mid ± (spreadBps + minMargin)`. When `spread1 > 0`, the actual oracle uncertainty is `spread0 + spread1`, but the band is computed with only `spread0`. The clamp is therefore tighter than the oracle intends.

**In reference mode** (no custom source): the pool quotes `bid = mid − spread0·bps − minMargin` and `ask = mid + spread0·bps + minMargin`. The missing `spread1` component means the spread is too narrow. Informed traders can execute at prices that are inside the oracle's true uncertainty band, extracting value from LPs at every swap — a direct loss of LP principal.

**In source mode**: the clamp `bidOut = min(refBid, cBid)` / `askOut = max(refAsk, cAsk)` uses the under-wide reference band. A custom source that correctly quotes the full `spread0 + spread1` uncertainty would be clipped inward to the narrower band, defeating the source's own risk management.

**In synthetic (ratio) mode**: both legs discard their respective `spread1` values, so the combined spread deficit is `spread1_base + spread1_quote`, compounding the under-pricing.

The pool's solvency invariant ("balances always cover LP claims") is not directly broken, but LP assets are systematically drained through adverse-selection swaps at prices the oracle never sanctioned — matching the "bad-price execution" and "direct loss of user principal / LP assets" impact categories.

---

### Likelihood Explanation

- Any oracle pusher who sets `s1 > 0` in their compressed feed triggers the defect. This is a normal, valid operation — the field exists precisely to carry a second spread component.
- The `CompressedOracleV1` is registrationless; any address can push feeds. A legitimate data provider encoding volatility or a second uncertainty source in `s1` will silently have that component ignored.
- All pools using `AnchoredPriceProvider` or `ProtectedPriceProvider` backed by `CompressedOracleV1` are affected whenever `spread1 ≠ 0`.
- No privileged action is required to trigger the defect; a normal oracle push with `s1 > 0` is sufficient.

---

### Recommendation

In `AnchoredPriceProvider._readLeg()`, capture `spread1` and add it to `spreadBps` before returning:

```solidity
function _readLeg(bytes32 feedId)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    uint256 spread1;
    (mid, spreadBps, spread1, refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
    spreadBps += spread1;   // ← incorporate the full oracle uncertainty
    ...
}
```

Apply the same fix to `ProtectedPriceProvider._getBidAndAskPrice()` and any other provider that calls `oracle.price()`.

---

### Proof of Concept

1. Deploy `CompressedOracleV1`. Push a feed with `s0 = 5` (5 bps) and `s1 = 10` (10 bps) at a mid price of `100_000_000` (1.00, 8-decimal).
2. Deploy `AnchoredPriceProvider` pointing at this oracle, `minMargin = 0`, `MAX_SPREAD_BPS = 500`.
3. Call `getBidAndAskPrice()`. Observe the band is computed with `spreadBps = 5` (only `spread0`), producing `half = 5 × 1e14`.
4. The correct band should use `spreadBps = 15` (`spread0 + spread1`), producing `half = 15 × 1e14` — a band 3× wider.
5. A trader can execute a swap at a price inside the oracle's true `±15 bps` uncertainty band but outside the pool's quoted `±5 bps` band, extracting value from LPs on every swap.

The `FaithfulAnchor` test mock independently confirms the pattern — it also discards `spread1` in its own `_readLeg`: [8](#0-7) 

```solidity
(mid, spreadBps,, refTime) = oracle.price(feedId, msg.sender);
```

This confirms the discard is a systemic pattern across the codebase, not an isolated oversight.

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }

    function _computeBidAsk(uint256 mid, uint256 spreadBps)
        internal view returns (uint128, uint128)
    {
        // Circuit breaker: extreme (combined) uncertainty means the feed is clearly broken.
        if (spreadBps > MAX_SPREAD_BPS) {
