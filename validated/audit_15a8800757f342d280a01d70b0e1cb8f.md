### Title
Pusher delegation can be re-established after `revokePusher()` by replaying the original `allowPushers` signature within its deadline window — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` uses a `deadline` as the sole replay guard for the pusher's EIP-191 consent signature. There is no per-signature invalidation (no nonce, no used-hash bitmap). After a pusher calls `revokePusher()`, the creator can replay the original `allowPushers` calldata with the same signature and the same unexpired deadline to silently re-establish the delegation — without any fresh consent from the pusher. The pusher's revocation is therefore ineffective for the entire remaining lifetime of the original deadline.

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only liveness check is `_ensureDeadline(deadline)`, which rejects calls where `block.timestamp > deadline`. [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)` but does nothing to invalidate the pusher's previously issued signature. [3](#0-2) 

Because the contract stores no used-signature set and no per-pusher nonce, the creator can call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple at any point before the deadline expires, writing `namespaceRemapping[pusher] = creator` a second time. The code comment acknowledges the replay risk and claims the deadline mitigates it, but the deadline only prevents replay *after* it expires — not within the window. [4](#0-3) 

The `fallback` push path resolves the namespace at call time from `namespaceRemapping[msg.sender]`, so any push the pusher makes after revocation — intending to write into their own namespace — is silently redirected into the creator's namespace once the delegation is replayed. [5](#0-4) 

### Impact Explanation

After the replay, every fallback push from the pusher lands in the creator's oracle namespace. If the pusher has begun pushing prices for a different asset or at a different scale into their own namespace, those values overwrite the creator's live feed slots. Any pool whose `AnchoredPriceProvider` or `PriceProvider` reads from the creator's `CompressedOracleV1` feed will consume these unintended prices during a swap, producing bad-price execution: the bid/ask spread delivered to `MetricOmmPool.swap` is derived from data the pusher never consented to place in the creator's namespace.

### Likelihood Explanation

The trigger requires the creator to act adversarially by replaying their own prior transaction. The pusher must have revoked and then started pushing into their own namespace before the original deadline expires. Deadlines are typically set days in the future (the test suite uses `block.timestamp + 1 days`), giving a wide replay window. The replay requires no special privilege — it is a plain public call with calldata already broadcast on-chain when the delegation was first established.

### Recommendation

Record each consumed signature hash in a `mapping(bytes32 => bool) private _usedDelegationHashes` and revert if the hash has already been used:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedDelegationHashes[hash], "signature already used");
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedDelegationHashes[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

This ensures each pusher consent signature can establish the delegation exactly once, so a `revokePusher` call permanently invalidates the original signature regardless of the remaining deadline.

### Proof of Concept

1. Pusher signs consent: `sig = sign(chainid, oracle, deadline=T+1day, pusher, creator)`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`. Delegation active.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`. Pusher believes they are now pushing into their own namespace.
4. Creator replays the identical call: `allowPushers(deadline, [pusher], [sig])` (same calldata, deadline still valid) → `namespaceRemapping[pusher] = creator` again. No revert.
5. Pusher pushes a price update intended for their own feed. The `fallback` resolves `namespaceRemapping[pusher] == creator` and writes the slot into the creator's namespace.
6. A pool reading `feedIdOf(creator, slotIndex, positionIndex)` via `AnchoredPriceProvider` receives the pusher's unintended price, executing a swap at a bad bid/ask.
