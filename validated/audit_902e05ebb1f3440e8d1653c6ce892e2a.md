### Title
Signature Replay in `allowPushers()` Lets a Creator Re-Establish a Revoked Pusher Delegation Within the Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers()` verifies a pusher's EIP-191 consent signature but stores no record of used signatures and tracks no per-pusher nonce. Any valid signature can be replayed by the creator an unlimited number of times before the deadline expires. A pusher who calls `revokePusher()` to stop feeding the creator's namespace can have that delegation silently re-established by the creator replaying the original signature, keeping the pusher's fallback push path locked into the creator's namespace and leaving the pusher's own namespace stale.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The signed payload binds `chainid`, `address(this)`, `deadline`, `pusher`, and `msg.sender` (the creator). There is no nonce and no "consumed signatures" mapping. After the signature is accepted once, it remains valid for any number of replays until `block.timestamp > deadline`.

The revocation path (`revokePusher`) simply zeroes `namespaceRemapping[msg.sender]`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

Because `allowPushers` does not invalidate the previously used signature, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple to restore `namespaceRemapping[pusher] = creator`. This cycle can be repeated indefinitely until the deadline passes.

The fallback push path resolves the target namespace as:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [3](#0-2) 

While the remapping is active, every slot word the pusher sends via the fallback lands in the **creator's** namespace, not the pusher's own. The pusher's own namespace receives no updates and its prices go stale.

---

### Impact Explanation

Pools that read price from the pusher's own namespace (via `feedIdOf(pusher, slotIndex, positionIndex)`) receive stale `timestampMs` values. Every consumer that enforces `maxTimeDrift` will reject the feed as stale, causing swaps to revert or fall back to a zero/sentinel price. This satisfies the **bad-price execution** and **broken core pool functionality causing loss of funds or unusable swap flows** impact gates.

Additionally, the creator's namespace receives the pusher's price updates without the pusher's consent, which can be used to anchor pools that reference the creator's namespace to prices the pusher did not intend to publish there.

---

### Likelihood Explanation

The preconditions are:
1. A pusher has signed a delegation consent with a non-trivial deadline (common for operational convenience — long deadlines reduce re-signing overhead).
2. The pusher later calls `revokePusher()` (e.g., after a dispute or key rotation).
3. The creator still holds the original `(deadline, signature)` tuple, which is on-chain from the first `allowPushers` call.

All three conditions are reachable in normal protocol operation. The creator is a semi-trusted but unprivileged actor with respect to the pusher's revocation right. No special access is required beyond having submitted the original `allowPushers` transaction.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) private _usedSignatures` and mark each digest as used on first acceptance:

```solidity
mapping(bytes32 => bool) private _usedSignatures;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
        );
        require(!_usedSignatures[hash], SignatureAlreadyUsed());
        require(pusher == ECDSA.recover(hash, signatures[i]));

        _usedSignatures[hash] = true;
        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, mirroring the pattern recommended in the reference report.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = block.timestamp + 1 days
bytes32 hash = keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator));
bytes memory sig = sign(PUSHER_KEY, MessageHashUtils.toEthSignedMessageHash(hash));

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — delegation is restored
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // revocation nullified

// 5. Pusher's fallback pushes now land in creator's namespace; pusher's own namespace goes stale
// Pools reading feedIdOf(pusher, slot, pos) receive timestampMs = 0 → rejected as stale
```
