### Title
Shared Slot-Level Timestamp in `CompressedOracleV1` Allows Stale Per-Lane Price Data to Bypass Staleness Checks, Feeding Bad Prices into Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1` packs four feed lanes into a single 256-bit storage slot with **one shared 56-bit timestamp**. When any lane in a slot is updated, the slot-level timestamp advances for **all four lanes simultaneously**, even those whose price data was not refreshed. A `PriceProvider` or `AnchoredPriceProvider` consuming a stale lane will receive a fresh `refTime` and pass the staleness guard, allowing stale prices to reach pool swaps.

---

### Finding Description

`CompressedOracleV1` stores four oracle feeds per EVM storage slot in the layout:

```
[Lane0:48][Lane1:48][Lane2:48][Lane3:48][timestamp:56][reserved:8]
``` [1](#0-0) 

The `_loadSlotLayout` function reads the single slot-level `timestampMs` and assigns it to the entire `SlotLayout` struct. `getOracleData` then returns this shared timestamp as `data.timestampMs` for whichever position is queried:

```solidity
_layout.timestampMs = toTimeMs(slotValue >> 8 & X56);
// ...
data.timestampMs = _layout.timestampMs;   // ← slot-level, not per-lane
``` [2](#0-1) 

The `fallback()` push path enforces only a **slot-level monotonicity check** — the incoming timestamp must exceed the stored slot timestamp — and then atomically overwrites the entire 256-bit word:

```solidity
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) continue;
_writeStorage(key, bytes32(bytes32(word & ~uint256(0xff))));
``` [3](#0-2) 

Because the slot is an atomic overwrite, a pusher who has fresh data for **Lane 0** but not for Lanes 1–3 must still supply values for all four lanes. The natural choice is to re-supply the currently stored (potentially stale) values for Lanes 1–3. After the write, all four lanes carry the new timestamp `T2`, even though Lanes 1–3 still hold price data from `T1`.

The `_price` function propagates this corrupted `refTime` directly to callers:

```solidity
function _price(bytes32 feedId) internal view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = getOracleData(feedId);
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
``` [4](#0-3) 

`PriceProvider._getBidAndAskPrice()` calls `IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender)` and uses the returned `refTime` as the sole freshness signal:

```solidity
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA)) {
    return (0, type(uint128).max);
}
``` [5](#0-4) 

Because `refTime` reflects the slot's last write (driven by Lane 0's update), the staleness check passes for Lanes 1–3 even though their price data is arbitrarily old. The same flaw exists in `AnchoredPriceProvider._readLeg`, which performs an identical staleness check on the returned `refTime`:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [6](#0-5) 

---

### Impact Explanation

A stale price for any of the three non-updated lanes passes the provider's staleness guard and is delivered as a valid bid/ask to the pool's swap math. Traders can exploit the price discrepancy between the stale on-chain quote and the true market price to extract value from LP positions. This constitutes **bad-price execution** (stale bid/ask quote reaches a pool swap) and **direct loss of LP principal**, both within the contest's allowed impact gate.

---

### Likelihood Explanation

This is a **structural design flaw**, not a malicious action. Any authorized pusher operating normally — updating one feed in a slot while carrying forward the stored values for the other three — triggers the condition. In a production deployment where feeds in the same slot update at different frequencies (e.g., a high-volatility asset alongside a stable asset), the divergence between the slot timestamp and individual lane freshness is routine and continuous. No special attacker capability is required beyond observing which lanes are stale and timing a swap accordingly.

---

### Recommendation

Introduce a **per-lane timestamp** within each 48-bit lane, replacing one of the two 8-bit spread indices with a compact relative timestamp (e.g., a delta from the slot timestamp), or expand the slot layout to 5 words (one per lane plus one for the shared slot header). Alternatively, require consumers to track the last-seen per-lane price and reject any read where the price value itself has not changed since the previous slot update, treating an unchanged price as implicitly stale after `MAX_TIME_DELTA`. At minimum, document that `refTime` is slot-granular and require all price providers backed by `CompressedOracleV1` to apply a **tighter** `MAX_TIME_DELTA` / `MAX_REF_STALENESS` that accounts for the worst-case inter-update gap of the slowest feed sharing the slot.

---

### Proof of Concept

**Setup:** A slot contains four feeds — `feedA` (ETH/USD, updates every 10 s) and `feedB` (low-volatility asset, updates every 10 min). Both are backed by the same `CompressedOracleV1` slot. A `PriceProvider` with `MAX_TIME_DELTA = 60 s` is configured for `feedB` and attached to a pool.

**Steps:**

1. At `T = 0`, pusher writes the slot with fresh data for all four lanes; slot timestamp = `T0`.
2. At `T = 590 s` (9 min 50 s), `feedB`'s price is now 590 s old — still within `MAX_TIME_DELTA = 60 s`? No — it would be stale. But the pusher now updates `feedA` with fresh data, carrying forward the 590-s-old `feedB` price. Slot timestamp advances to `T0 + 590 s`.
3. At `T = 600 s`, a trader calls the pool. `PriceProvider` calls `CompressedOracleV1.price(feedB, pool)`. The returned `refTime` is `T0 + 590 s` (10 s ago). `_isStale(T0+590, T0+600, 60)` → `false`. The 600-s-old price passes.
4. The trader swaps against the stale `feedB` price, extracting value from LPs.

The corrupted value is `data.timestampMs` returned by `getOracleData` for any lane that was not the trigger of the most recent slot write. [1](#0-0) [2](#0-1) [7](#0-6) [8](#0-7) [9](#0-8)

    function _loadSlotLayout(bytes32 slotIndex) internal view returns (SlotLayout memory _layout) {

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
