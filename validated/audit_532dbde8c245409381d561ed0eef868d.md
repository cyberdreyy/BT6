### Title
Temporary swap DoS: `ChainlinkOracle` accepts future-timestamped reports that `AnchoredPriceProvider._isStale` unconditionally rejects — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

### Summary
`ChainlinkOracle._store` admits any DON-verified report whose `observationsTimestamp` satisfies `observationsTimestamp ≤ block.timestamp + MAX_TIME_DRIFT`. `AnchoredPriceProvider._isStale` treats every `refTime > block.timestamp` as stale. Any report stored in the drift window `(block.timestamp, block.timestamp + MAX_TIME_DRIFT]` is therefore accepted by the oracle but rejected by the provider, stalling every pool that uses `AnchoredPriceProvider` until `block.timestamp` catches up.

### Finding Description
`ChainlinkOracle._store` calls `d.timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT)`, which passes when `observationsTimestamp ≤ block.timestamp + MAX_TIME_DRIFT` (inclusive). The stored `refTime` returned by `OracleBase._readPrice` is `data.timestampMs.toSeconds()` — the raw observation timestamp, unmodified.

`AnchoredPriceProvider._readLeg` calls `_isStale(refTime, block.timestamp, MAX_REF_STALENESS)`. `_isStale` returns `true` whenever `refTime > block.timestamp`, regardless of how small the excess is. The result is `ok = false`, which propagates through `_getBidAndAskPrice` to a `FeedStalled` revert in `getBidAndAskPrice`.

The L2-aware sibling `ProtectedPriceProviderL2._isStale` has an explicit `FUTURE_TOLERANCE` parameter that tolerates exactly this case. `AnchoredPriceProvider` has no such parameter, creating an incompatibility with any oracle that stores reports with future timestamps.

### Impact Explanation
Every pool using `AnchoredPriceProvider` backed by `ChainlinkOracle` is unable to execute swaps for up to `MAX_TIME_DRIFT` seconds (60 s on mainnet) after any report is stored with a future observation timestamp. The stall is self-resolving (no fund loss, no insolvency), but it constitutes a temporary break of the core swap flow. A griefing pusher can deliberately time a valid DON-verified push to maximise the stall window. Severity: **Medium**.

### Likelihood Explanation
DON nodes aggregate observations across multiple nodes; minor clock skew between DON nodes and the sequencer/validator is routine. The `MAX_TIME_DRIFT = 60 s` window was sized to accommodate this skew at ingestion time, but the provider-side check was not aligned. The condition can also be triggered deliberately by any permissionless caller who holds a valid DON-signed report and submits it one second before `block.timestamp` reaches `observationsTimestamp`.

### Recommendation
Add a `FUTURE_TOLERANCE` immutable to `AnchoredPriceProvider` (mirroring `ProtectedPriceProviderL2`) and update `_isStale` to tolerate `refTime ∈ (nowTs, nowTs + FUTURE_TOLERANCE]`:

```solidity
// AnchoredPriceProvider._isStale — aligned with ProtectedPriceProviderL2
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

Set `FUTURE_TOLERANCE` equal to `ChainlinkOracle.MAX_TIME_DRIFT` at deployment. Alternatively, enforce `MAX_TIME_DRIFT = 0` in `ChainlinkOracle` when paired with `AnchoredPriceProvider`, but this would break legitimate clock-skew tolerance at ingestion.
