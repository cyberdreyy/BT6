### Title
Synthetic-Mode `AnchoredPriceProvider` Accepts `baseFeedId == quoteFeedId`, Permanently Locking Pool Price at 1.0 — (`File: smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` enforces that `baseToken != quoteToken` but imposes **no analogous uniqueness check on `baseFeedId` vs `quoteFeedId`**. When both feed IDs are identical and non-zero (synthetic mode), the ratio computation always resolves to `ORACLE_DECIMALS` (1.0 in 8-decimal), regardless of the actual market price. Every swap through a pool bound to such a provider executes at a permanently wrong price.

---

### Finding Description

The constructor validates token uniqueness:

```solidity
require(_baseToken != address(0) && _quoteToken != address(0) && _baseToken != _quoteToken);
``` [1](#0-0) 

But it stores `baseFeedId` and `quoteFeedId` without any analogous guard:

```solidity
baseFeedId = _baseFeedId;
quoteFeedId = _quoteFeedId;
``` [2](#0-1) 

In `_getBidAndAskPrice()`, when `quoteFeedId != bytes32(0)`, the synthetic ratio path executes:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
spreadBps += spreadBps2;
``` [3](#0-2) 

When `baseFeedId == quoteFeedId`, both `_readLeg` calls hit the **same storage slot** in the oracle. Because `mid == mid2` identically (same feed, same block), the division always yields `ORACLE_DECIMALS = 1e8` — a hard-coded price of 1.0 — while `spreadBps` is doubled. The pool then computes bid/ask around this constant, completely decoupled from the real market price.

The `_readLeg` function reads through the attributed `price(feedId, pool)` path:

```solidity
(mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
``` [4](#0-3) 

No downstream guard in `_computeBidAsk` detects that the mid is synthetic-constant; the staleness, spread, and price-guard checks all pass normally because the underlying feed data is valid — only the ratio is wrong. [5](#0-4) 

---

### Impact Explanation

Any pool whose `IPriceProvider` is an `AnchoredPriceProvider` with `baseFeedId == quoteFeedId` will execute every swap at price ≈ 1.0 (in 8-decimal units), regardless of the actual market rate. Traders receive more output than the oracle curve permits (swap conservation failure) or far less, depending on which side of 1.0 the real price sits. LPs suffer direct principal loss because the pool's bin liquidity is consumed at the wrong price. This matches the "bad-price execution" and "swap conservation failure" impact gates.

---

### Likelihood Explanation

`AnchoredPriceProvider` is a plain contract; its constructor is callable by anyone. The `baseFeedId` and `quoteFeedId` immutables are set at construction time and can never be changed. A pool factory that does not independently validate feed-ID uniqueness will accept such a provider. Because the `CompressedOracleV1` feed namespace is registrationless — any address can push into its own namespace without setup — an attacker can also control the underlying feed data, compounding the manipulation surface. [6](#0-5) 

---

### Recommendation

Add a guard in the `AnchoredPriceProvider` constructor that rejects equal, non-zero feed IDs in synthetic mode:

```solidity
if (_quoteFeedId != bytes32(0)) {
    require(_baseFeedId != _quoteFeedId, "duplicate feed ids");
}
```

This mirrors the existing `_baseToken != _quoteToken` check and closes the asymmetric validation gap. [7](#0-6) 

---

### Proof of Concept

1. Attacker deploys `CompressedOracleV1` and pushes a real price (e.g., ETH/USD = 3000 × 1e8) into slot 0, position 0.
2. Attacker deploys `AnchoredPriceProvider` with:
   - `baseFeedId = feedIdOf(attacker, 0, 0)`
   - `quoteFeedId = feedIdOf(attacker, 0, 0)` ← same feed
   - `baseToken = WETH`, `quoteToken = USDC` (different tokens, passes the token check)
3. Attacker creates a pool using this provider (permissionless factory).
4. At swap time, `_getBidAndAskPrice()` calls `_readLeg(baseFeedId)` → `mid = 3000e8`, `spreadBps = S`. Then calls `_readLeg(quoteFeedId)` → same slot → `mid2 = 3000e8`, `spreadBps2 = S`.
5. `mid = mulDiv(3000e8, 1e8, 3000e8) = 1e8` (price = 1.0). `spreadBps = 2S`.
6. Pool quotes bid/ask around 1.0 instead of 3000. A trader swapping USDC→WETH receives ~3000× more WETH than the oracle permits, draining the pool's WETH reserves. LPs lose their entire WETH principal. [8](#0-7)

    function _getBidAndAskPrice() internal returns (uint128, uint128) {
        (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
        if (!ok) return (0, type(uint128).max);

        bytes32 _quote = quoteFeedId;
        if (_quote != bytes32(0)) {
            (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
            if (!ok2 || mid2 == 0) return (0, type(uint128).max);
            // Synthetic ratio (8-decimal): mid1 / mid2. Relative uncertainties of a ratio add.
            mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
            spreadBps += spreadBps2;
        }

        return _computeBidAsk(mid, spreadBps);
    }

    function _computeBidAsk(uint256 mid, uint256 spreadBps)
        internal view returns (uint128, uint128)

        // Circuit breaker: extreme (combined) uncertainty means the feed is clearly broken.
        if (spreadBps > MAX_SPREAD_BPS) {
            return (0, type(uint128).max);
        }

        // Reference band: mid ± (spreadBps + minMargin), bid rounded down, ask rounded up.
