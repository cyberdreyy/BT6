### Title
Permissionless `register` Unconditionally Clears Blacklist, Making Pool Blocking Irreversible — (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase.register` is callable by anyone and unconditionally clears the `blacklisted` flag for a pool when the registration fee (default: 1 wei) is paid. Because there is no `unregister` function and `registeredPool[feedId][pool]` can never be set back to `false`, the admin's only post-registration protection mechanism — the blacklist — can be bypassed by any unprivileged actor paying a trivial fee.

---

### Finding Description

`OracleBase` exposes two protection mechanisms for pools that read oracle prices:

1. `blacklisted[pool]` — set by `ADMIN_ROLE` via `setBlacklist`.
2. `registeredPool[feedId][pool]` — set permanently to `true` by `register`.

The `price()` function enforces both:

```solidity
require(!blacklisted[pool], Blacklisted(pool));
require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
``` [1](#0-0) 

However, `register` — which is **permissionless** and requires only `msg.value >= registrationFee` (default: 1 wei) — contains this logic:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
``` [2](#0-1) 

The only preconditions are that `factory` is in `approvedFactories` and `factory.isPool(pool)` returns true — both trivially satisfied for any legitimately deployed pool from an approved factory. [3](#0-2) 

There is **no `unregister` function** anywhere in the codebase. Once `registeredPool[feedId][pool] = true`, it is permanent. The blacklist is the only remaining lever, but it is defeatable by anyone paying 1 wei.

The `registrationFee` is initialized to 1 wei:

```solidity
registrationFee = 1 wei; // very cheap default; ADMIN tunes via setRegistrationFee
``` [4](#0-3) 

Even if ADMIN raises the fee, the attacker can still pay it — the fee is not a meaningful barrier for a motivated actor.

---

### Impact Explanation

If a pool is blacklisted by ADMIN (e.g., because a bug in the pool allows it to drain LP assets, or because it is executing swaps against a stale/manipulated price), any unprivileged actor can restore its oracle read access by calling `register(feedId, pool, factory)` with 1 wei. The pool then passes both checks in `price()` and resumes receiving live oracle bid/ask quotes, enabling continued bad-price execution or fund drainage.

This is a direct admin-boundary break: the oracle admin's ability to permanently block a pool from reading prices is bypassed by an unprivileged path, with no timelock, no role check, and no meaningful cost barrier.

---

### Likelihood Explanation

The trigger requires only:
- A pool that is registered with an approved factory (normal operational state).
- 1 wei (default fee).
- Knowledge of the `feedId`, `pool`, and `factory` addresses (all public on-chain).

Any actor — including the operator of the compromised pool — can execute this in a single transaction. The likelihood of exploitation in a scenario where ADMIN attempts to blacklist a pool is **high**.

---

### Recommendation

Separate the blacklist-clearing logic from the registration path. Do not clear the blacklist inside `register`. Instead, require an explicit ADMIN action to un-blacklist a pool, and add an `unregister` function so that oracle access can be permanently revoked:

```diff
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));
+   require(!blacklisted[pool], Blacklisted(pool));

-   if (blacklisted[pool]) {
-       blacklisted[pool] = false;
-       emit BlacklistUpdated(pool, false);
-   }

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}

+ function unregister(bytes32 feedId, address pool) external onlyRole(ADMIN_ROLE) {
+     registeredPool[feedId][pool] = false;
+     emit PoolUnregistered(feedId, pool);
+ }
```

### Proof of Concept

```
Setup:
  - CompromisedPool is deployed from ApprovedFactory.
  - CompromisedPool is registered: registeredPool[feedId][CompromisedPool] = true.
  - ADMIN discovers a drain bug in CompromisedPool and calls setBlacklist(CompromisedPool, true).
  - blacklisted[CompromisedPool] == true → price() reverts → pool cannot read oracle.

Attack (single tx, 1 wei):
  1. Attacker calls register(feedId, CompromisedPool, ApprovedFactory) with msg.value = 1 wei.
  2. register checks: msg.value >= 1 wei ✓, factory approved ✓, isPool(CompromisedPool) ✓.
  3. register executes: blacklisted[CompromisedPool] = false.
  4. registeredPool[feedId][CompromisedPool] = true (already true, no-op).

Result:
  - CompromisedPool passes both checks in price() again.
  - CompromisedPool resumes receiving live oracle bid/ask quotes.
  - Attacker exploits the drain bug against LP assets.
```
