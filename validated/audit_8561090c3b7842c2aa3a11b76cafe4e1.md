### Title
Silent Output Clamping in Exact-Out Swap Silently Delivers Less Than Requested Without Reverting — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

In `_swapToken1ForToken0SpecifiedOutput`, when the caller's requested exact-out amount exceeds the pool's total available token0, the code silently clamps the output to the available balance and continues execution instead of reverting. This is a structural analog to the Hashflow bug: a pre-execution cap silently reduces the effective amount, breaking the exact-out invariant and allowing the swap to "succeed" with a partial fill that the caller did not consent to.

---

### Finding Description

In `MetricOmmPool._swapToken1ForToken0SpecifiedOutput`: [1](#0-0) 

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    amountOutScaled = totalAvailableToken0Scaled;
}
```

When a caller invokes `swap(recipient, false, -X, ...)` (exact-out, requesting `X` token0), the pool internally computes `amountOutScaled = TOKEN_0_SCALE_MULTIPLIER * X`. If this exceeds `binTotals.scaledToken0`, the pool silently reduces `amountOutScaled` to the available balance and proceeds. The swap completes, the callback is invoked with the clamped `amount0Delta` and the correspondingly reduced `amount1Delta`, and no revert or error is raised. [2](#0-1) 

The outer `swap()` function then transfers only the clamped `|amount0Delta|` to the recipient and requests only the reduced `amount1Delta` from the callback — the caller has no on-chain signal that the requested amount was not honored. [3](#0-2) 

---

### Impact Explanation

Any integrator relying on exact-out semantics — a router enforcing a minimum output, an aggregator routing a fixed amount to a downstream protocol, or a flash-loan repayment flow — will silently receive fewer tokens than requested. The pool returns `(amount0Delta, amount1Delta)` reflecting the clamped amounts; if the caller's slippage guard is applied to the return value rather than the original request, the shortfall passes undetected. Downstream operations that depend on receiving exactly the requested amount (e.g., repaying a flash loan, seeding a subsequent hop) will fail or produce incorrect results. This constitutes broken core exact-out swap functionality.

---

### Likelihood Explanation

The condition triggers whenever a user requests more token0 out than is currently held across all bins (`binTotals.scaledToken0`). This is a realistic scenario during periods of concentrated demand, low liquidity, or after large prior swaps have drained the pool. No special permissions or adversarial setup are required — any ordinary exact-out swap call can hit this path.

---

### Recommendation

Replace the silent clamp with a revert:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    revert InsufficientLiquidity(); // new error
}
```

This preserves the exact-out invariant: either the caller receives exactly what they requested, or the transaction reverts. Callers that want best-effort partial fills should use exact-in (`amountSpecified > 0`) instead.

The symmetric function `_swapToken0ForToken1SpecifiedOutput` (not visible in the truncated file but expected by the code structure) should be audited for the same pattern.

---

### Proof of Concept

1. Pool holds 100 token0 scaled units (`binTotals.scaledToken0 = 100`).
2. Caller invokes `swap(recipient, false, -200, priceLimitX64, callbackData, "")` — requesting 200 token0 out.
3. `_executeSwap` computes `amountOutScaled = TOKEN_0_SCALE_MULTIPLIER * 200`.
4. `_swapToken1ForToken0SpecifiedOutput` clamps: `amountOutScaled = 100`.
5. Swap loop executes against the clamped amount; callback is invoked with `amount0Delta = -100` and `amount1Delta` = token1 cost for 100 token0.
6. Recipient receives 100 token0; caller pays token1 for 100 token0.
7. `swap()` returns `(-100, amount1Delta)` — no revert, no error. The caller's request for 200 token0 was silently halved. [4](#0-3)
