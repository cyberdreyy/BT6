### Title
Revoked pusher delegation can be silently re-established by replaying the original `allowPushers` signature within the deadline window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` verifies an EIP-191 pusher-consent signature and writes `namespaceRemapping[pusher] = msg.sender`, but **never marks the signature as consumed**. A pusher who self-revokes via `revokePusher()` can have their delegation silently re-established by anyone replaying the original `allowPushers` calldata — including the creator — as long as the deadline has not yet expired. This is the direct analog of the QV bug: a state variable (`namespaceRemapping`) is cleared by the revocation path, but the absence of a used-signature registry means the clearing is immediately reversible, making `revokePusher` ineffective within the deadline window.

---

### Finding Description

`allowPushers` constructs a hash over `(block.chainid, address(this), deadline, pusher, msg.sender)` and requires the pusher's ECDSA signature over that hash. After verification it writes the namespace mapping and emits an event — but it never records the hash as spent:

```solidity
// CompressedOracle.sol lines 192–212
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));

    namespaceRemapping[pusher] = msg.sender;   // ← written
    // ← NO: usedSignatures[hash] = true is never set
    emit PusherAuthorized(pusher, msg.sender);
}
```

`revokePusher` clears the mapping:

```solidity
// CompressedOracle.sol lines 238–243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

Because the original signature hash is never invalidated, the creator (or any on-chain observer who captured the original calldata) can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple, restoring `namespaceRemapping[pusher] = creator`. The code's own comment acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and cites the deadline as the mitigation — but the deadline only prevents replay **after** it expires, not **within** the window. ---

### Impact Explanation

Every push through the fallback path resolves the writer's namespace via `namespaceRemapping`:

```
fallback push → namespace resolution (namespaceRemapping[msg.sender]) → slot overwrite
```

If the pusher is an automated price-feed bot (a common deployment pattern for compressed oracles), and its key is compromised or it begins emitting stale/inverted prices, the legitimate pusher calls `revokePusher()` to stop those bad prices from landing in the creator's namespace. The creator (or any replayer) immediately re-establishes the delegation. The bot's subsequent pushes — carrying stale, inverted, or otherwise bad `U64x32` mid-prices and codebook spread indexes — continue to overwrite the creator's namespace slots. Price providers that consume `getOracleData(feedId)` for that creator's namespace then return corrupted bid/ask quotes to `MetricOmmPool.swap`, causing bad-price execution: traders receive more output than the oracle curve permits or the pool fails to receive owed input. [3](#0-2) 

---

### Likelihood Explanation

- The original `allowPushers` calldata (including the pusher's signature) is permanently visible on-chain from the first call.
- The creator has a direct incentive to replay it if they want to retain the pusher's data stream.
- No privileged role or special setup is required beyond possessing the original calldata — any EOA can replay it.
- Deadlines in production deployments are typically set days or weeks in the future, giving a large replay window.

---

### Recommendation

Track consumed signature hashes and reject replays:

```solidity
mapping(bytes32 => bool) public usedSignatures;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
        );
        require(!usedSignatures[hash], "signature already used");   // ← add
        require(pusher == ECDSA.recover(hash, signatures[i]));

        usedSignatures[hash] = true;                                // ← add
        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

Alternatively, introduce a per-pusher revocation nonce included in the signed message; incrementing it on `revokePusher` invalidates all prior signatures for that pusher without requiring a global hash registry.

### Proof of Concept

```
1. creator calls allowPushers(deadline=T+7days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator
   → calldata (including sig) is now permanently on-chain

2. pusher's key is compromised; attacker begins pushing stale prices via fallback
   → bad prices land in creator's namespace

3. pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)
   → pusher expects bad prices to stop flowing to creator's namespace

4. creator (or any observer) replays the original allowPushers calldata
   → _ensureDeadline passes (T+7days not yet reached)
   → ECDSA.recover returns pusher (same sig, same hash)
   → namespaceRemapping[pusher] = creator  ← restored
   → NO revert, NO new pusher consent required

5. attacker continues pushing stale/inverted prices via fallback
   → prices land in creator's namespace again
   → AnchoredPriceProvider / PriceProvider reads corrupted mid + spread
   → MetricOmmPool.swap executes at bad bid/ask → bad-price execution
``` [4](#0-3) [5](#0-4)
