### Title
`priceGuard` Validates Raw Oracle Mid Before Spread and MarginStep Transformations, Allowing Unclamped Ask/Bid to Reach Pool Swaps — (`smart-contracts-poc/contracts/PriceProvider.sol`, `ProtectedPriceProvider.sol`, `PriceProviderL2.sol`)

---

### Summary

`PriceProvider`, `ProtectedPriceProvider`, and `PriceProviderL2` check the raw oracle `mid` price against `priceGuard` bounds **before** applying the confidence-spread expansion and the `marginStep` step-factor. The final `askOut`/`bidOut` delivered to the pool can therefore exceed `guardMax` (or fall below `guardMin`) by a factor proportional to `(1 + marginStep/BPS_BASE_U) × (1 + adjustedSpread/CONFIDENCE_BASE)`, defeating the guard's purpose as a circuit-breaker.

---

### Finding Description

In all three providers the pricing pipeline is:

1. Read `(mid, spread, refTime)` from the oracle.
2. Staleness check.
3. Basic validity (`mid > 0`, `spread < ORACLE_BPS`).
4. **Price guard check** — `mid` vs `guardMin`/`guardMax`.
5. Compute `bid = mid − delta`, `ask = mid + delta` where `delta = mid × spread × confidenceParam / CONFIDENCE_BASE`.
6. Apply marginStep: `askOut = ask × Q64 × stepAskFactor / STEP_DENOM`, `bidOut = bid × Q64 × stepBidFactor / STEP_DENOM`.
7. Inversion guard (`bidOut >= askOut` → stall). [1](#0-0) 

The guard at step 4 checks only `mid`. Steps 5–6 then apply two multiplicative expansions to produce `askOut`. The final Q64 ask price delivered to the pool is:

```
askOut_Q64 = (mid + delta) × Q64 × (BPS_BASE_U + marginStep) / STEP_DENOM
```

Converting back to 8-decimal units for comparison with `guardMax`:

```
askOut_8dec_equiv = (mid + delta) × (BPS_BASE_U + marginStep) / BPS_BASE_U
```

For any positive `marginStep` and any non-zero `confidenceParam`, `askOut_8dec_equiv > mid`. When `mid` is near `guardMax`, `askOut_8dec_equiv` exceeds `guardMax` by the full spread+step expansion. The guard never sees this inflated value. [2](#0-1) 

The same pattern is present in `PriceProviderL2`: [3](#0-2) 

This is structurally identical to the Reality Cards M-07 bug: the limit check (`_maxSumOfPrices` / `priceGuard`) passes on the pre-transformation value, but the mandatory post-check transformation (10% forced increase / spread+marginStep) causes the final value to exceed the limit.

---

### Impact Explanation

The pool's swap math consumes `askOut` and `bidOut` directly as the bid/ask prices for every swap. When `askOut` exceeds the `guardMax` equivalent, the pool executes swaps at prices the feed creator explicitly declared out-of-range. This is unclamped bad-price execution reaching a live pool swap. Concretely:

- **Inflated ask** (positive `marginStep`, high spread): traders buying token0 pay above the guard's intended ceiling.
- **Deflated bid** (positive `marginStep`, high spread): traders selling token0 receive below the guard's intended floor.
- **Deflated ask** (negative `marginStep`, sufficient confidence): pool sells token0 below oracle mid, enabling arbitrage that drains pool token0 reserves and causes LP principal loss.

The `priceGuard` is the last sanity circuit-breaker before prices reach pools. Its bypass removes the only on-chain protection against extreme oracle readings being used for swap settlement.

---

### Likelihood Explanation

- `marginStep > 0` is the normal operational setting for any provider that widens the spread beyond the oracle's own uncertainty. It is set at construction time (immutable) and is always active.
- With `marginStep > 0` and `mid = guardMax` (the guard boundary), the guard passes but `askOut` always exceeds `guardMax` in equivalent 8-decimal terms — no additional trigger is required.
- A pusher (semi-trusted, authorized by the oracle creator) pushing any non-zero spread amplifies the bypass further. Pushers routinely push non-zero spreads during normal market operation.
- `confidenceParam` is set by the factory admin and can be up to `CONFIDENCE_MAX = 1_000_000` (100×). Once set, every subsequent price push by any authorized pusher triggers the bypass. [4](#0-3) [5](#0-4) 

---

### Recommendation

Move the price guard check to **after** the spread and marginStep transformations, comparing `askOut` and `bidOut` (converted back to 8-decimal units) against `guardMax` and `guardMin` respectively. Alternatively, tighten the guard bounds at the oracle level to account for the maximum possible expansion: `effectiveGuardMax = guardMax / (1 + marginStep/BPS_BASE_U) / (1 + maxAdjustedSpread/CONFIDENCE_BASE)`.

---

### Proof of Concept

**Setup** (`PriceProvider`):
- `marginStep = 5e17` (50%, within the `< BPS_BASE_U` bound)
- `confidenceParam = 1_000_000` (100×, the maximum)
- Oracle feed: `mid = 199_000_000`, `spread = 9_999` (max valid, just below `ORACLE_BPS` stall marker)
- `priceGuard`: `guardMin = 100_000_000`, `guardMax = 200_000_000`

**Execution**:

```
Step 4 guard: mid=199_000_000 ≤ guardMax=200_000_000 → PASSES

Step 5:
  adjustedSpread = 9_999 × 1_000_000 = 9_999_000_000
  delta = 199_000_000 × 9_999_000_000 / 1e10 ≈ 198_980_100
  ask = 199_000_000 + 198_980_100 = 397_980_100

Step 6:
  stepAskFactor = 1e18 + 5e17 = 1.5e18
  askOut_Q64 = 397_980_100 × Q64 × 1.5e18 / (1e8 × 1e18)
  askOut_8dec_equiv = 397_980_100 × 1.5 = 596_970_150

Result: pool receives ask ≈ 596_970_150 (8-dec equiv)
        guardMax =          200_000_000
        Excess: ~3× guardMax — guard completely bypassed
``` [6](#0-5) [7](#0-6)

    function _applyStepAdjustment(
        uint256        price,
        uint256        stepFactor,
        Math.Rounding  rounding
    ) private pure returns (uint256 out, bool ok) {
        if (price == 0) return (0, false);

        uint256 numerator = Q64 * stepFactor;

        out = Math.mulDiv(price, numerator, STEP_DENOM, rounding);

        if (out == 0) return (0, false);

        return (out, true);
    }
