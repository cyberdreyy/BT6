### Title
Pusher delegation consent has no invalidation mechanism — creator can replay a revoked signature within its deadline window - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` accepts any valid EIP-191 signature whose deadline has not yet expired. There is no nonce, no used-signature bitmap, and no on-chain record that a consent was already consumed. After a pusher calls `revokePusher()`, the creator can call `allowPushers()` again with the **identical signature** to silently re-establish the delegation — the pusher has no on-chain mechanism to permanently invalidate their signed consent before the deadline expires.

### Finding Description

`allowPushers` verifies the pusher's EIP-191 signature over `(block.chainid, address(this), deadline, pusher, msg.sender)` and writes `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

The function contains no nonce, no consumed-signature set, and no check that `namespaceRemapping[pusher]` is currently zero. After `revokePusher()` clears the mapping: [2](#0-1) 

…the creator can immediately call `allowPushers()` again with the same `(deadline, pusher, signature)` tuple. `_ensureDeadline` only checks `block.timestamp <= deadline`: [3](#0-2) 

The signature passes again because nothing has changed in the signed payload. The delegation is re-established without the pusher's knowledge or consent.

The code comment in `allowPushers` acknowledges the replay risk but incorrectly treats the deadline as a complete mitigation:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [4](#0-3) 

The deadline prevents replay *after* expiry, but does nothing to prevent replay *within* the deadline window — which is exactly the window during which a pusher who has revoked is still vulnerable.

### Impact Explanation

A pusher's `fallback()` writes land in `namespaceRemapping[msg.sender]`'s namespace: [5](#0-4) 

If the creator re-establishes delegation after revocation, the pusher's future pushes (or an attacker holding a compromised pusher key) write directly into the creator's namespace. `PriceProvider._getBidAndAskPrice()` reads from that namespace via `oracle.price(feedId, pool)`: [6](#0-5) 

A bad price that passes the staleness check and falls within the `priceGuard` range reaches `getBidAndAskPrice()` and is consumed by the pool's swap path, causing bad-price execution for traders.

### Likelihood Explanation

- Delegation is a normal operational pattern; creators are expected to hold pusher signatures for the full deadline window (which can be arbitrarily long).
- A creator running an automated monitoring script that re-establishes delegation whenever `namespaceRemapping[pusher]` is zero would trigger this silently.
- A pusher whose key is compromised has no on-chain recourse: they revoke, the creator (or the attacker themselves, if they also control the creator key) re-establishes, and bad prices flow.
- Deadlines in practice are set to days or weeks (the test suite uses `block.timestamp + 1 days`), giving a large replay window. [7](#0-6) 

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) usedConsents` keyed on the signature hash, and revert if the same hash is presented a second time. Alternatively, include a per-pusher nonce in the signed payload and increment it on every successful `allowPushers` call, so a revoked pusher can advance their nonce to invalidate all outstanding signatures.

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 30-day deadline.
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes (e.g., key compromise detected).
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — delegation re-established.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // no revert
assertEq(oracle.namespaceRemapping(pusher), creator);   // pusher is delegated again

// 5. Attacker (holding compromised pusher key) pushes a manipulated price.
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 badRaw = _packRaw(9_000_000, 4, 2); // extreme price
vm.prank(pusher); // attacker controls this key
(bool ok,) = address(oracle).call(_wordAt(0, 0, badRaw, tsMs));
assertTrue(ok);

// 6. Bad price now lives in creator's namespace and will be consumed by PriceProvider.
IOffchainOracle.OracleData memory d = oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
assertEq(d.price, U64x32.decode(uint32(badRaw >> 16))); // manipulated price in live feed
```
