### Title
`revokePusher()` Is Ineffective Within the Deadline Window Due to Missing Revocation Nonce — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` signs a payload that contains no nonce or revocation counter. After a pusher self-revokes via `revokePusher()`, the creator can replay the original signature before the deadline to silently re-establish the delegation. The `revokePusher()` guarantee is therefore broken for the entire duration of the original deadline window.

---

### Finding Description

The signed consent payload in `allowPushers` is:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no per-pusher revocation counter, and no "signature consumed" flag. The only replay guard is `_ensureDeadline`, which only checks `block.timestamp <= deadline`. [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender] = address(0)` but does nothing to invalidate the original signature bytes. [3](#0-2) 

The NatSpec comment on `allowPushers` explicitly acknowledges the risk — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — and claims the deadline is the mitigation. But the deadline only prevents replay **after** it expires; it does nothing to prevent replay **within** the deadline window. [4](#0-3) 

**Attack sequence:**
1. Pusher signs consent: `deadline = block.timestamp + 1 hour`
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`
4. Creator calls `allowPushers(deadline, [pusher], [sig])` again with the **same bytes** before the deadline → `namespaceRemapping[pusher] = creator` again — no new consent from the pusher required

---

### Impact Explanation

The pusher's `revokePusher()` right is the only self-sovereign exit from a delegation. Once bypassed:

- The pusher, believing they have revoked, continues pushing to what they think is their own namespace. Because `namespaceRemapping[pusher]` has been silently restored to `creator`, every fallback push lands in the **creator's namespace** instead.
- The pusher's own namespace feeds remain at zero/stale. Any pool or provider reading `feedIdOf(pusher, ...)` receives stale or zero price data, which can reach live swaps as a bad-price execution.
- The creator retains write authority over their namespace feeds using the pusher's data without the pusher's ongoing consent, violating the delegation monotonicity invariant the system is designed to enforce.

This satisfies the **admin-boundary break** and **bad-price execution** impact gates: an unprivileged path (the creator holding old signature bytes) bypasses the pusher's revocation right, and the resulting misdirected writes can drive stale or unintended prices into production pool reads.

---

### Likelihood Explanation

The creator already holds the signature bytes from the original `allowPushers` call (they submitted the transaction). No additional privilege is needed. The creator only needs to monitor the chain for a `PusherRevoked` event and replay before the deadline — a straightforward MEV/front-run or simple follow-up transaction. Any deadline longer than a few blocks (the typical case, e.g., 1 hour or 1 day) gives ample time.

---

### Recommendation

Track consumed signatures or maintain a per-pusher revocation nonce. The simplest fix is a `mapping(address => uint256) public pusherNonce` incremented on every successful `allowPushers` and `revokePusher`, and included in the signed payload:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))
```

`revokePusher()` increments `pusherNonce[pusher]`, immediately invalidating all prior signatures regardless of their deadline.

Alternatively, record a `mapping(bytes32 => bool) usedSignatureHashes` and mark each hash consumed on first use.

### Proof of Concept

```solidity
// Foundry test (pseudo-code)
function testRevokeReplayWithinDeadline() public {
    uint256 deadline = block.timestamp + 1 hours;

    // 1. Pusher signs consent
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    // 2. Creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher self-revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature before deadline — no new consent
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds
    assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation restored

    // 5. Pusher pushes, believing they are in their own namespace
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(1, 0, _packRaw(999_000, 4, 2), uint56(block.timestamp * 1000)));
    assertTrue(ok);

    // Data lands in CREATOR namespace, not pusher's own
    assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 1, 0)).price, U64x32.decode(999_000));
    assertEq(oracle.getOracleData(oracle.feedIdOf(pusher,  1, 0)).price, 0); // pusher's own feed is stale
}
```
