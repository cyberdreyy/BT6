### Title
Signature Replay in `allowPushers` Allows Creator to Re-Establish Delegation After Pusher Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`CompressedOracleV1.allowPushers` has no per-signature invalidation mechanism. Once a pusher signs a consent with deadline `D1`, that signature remains valid for any `allowPushers` call until `D1` expires — including calls made after the pusher has called `revokePusher()`. A creator can replay the original signature to silently re-establish delegation against the pusher's explicit revocation.

### Finding Description
`allowPushers` verifies an EIP-191 signature over `(chainid, address(this), deadline, pusher, creator)` and then unconditionally writes `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

`revokePusher` clears the mapping to `address(0)`: [2](#0-1) 

Because there is no nonce, no used-signature set, and no check that the current `namespaceRemapping[pusher]` is `address(0)` before writing, the creator can call `allowPushers` again with the exact same `(deadline, pusher, signature)` tuple after revocation. The `_ensureDeadline` check passes as long as `block.timestamp <= deadline`, which is unrelated to whether the pusher revoked.

The code comment at lines 186–191 explicitly states the deadline is the protection against post-revocation replay, but this is incorrect: the deadline only prevents using expired signatures, not replaying a still-valid one. [3](#0-2) 

### Impact Explanation
After re-delegation, the pusher's `fallback()` pushes are routed to the creator's namespace instead of the pusher's own: [4](#0-3) 

If the pusher is running an automated bot that pushes to their own namespace (serving pools that read from `feedIdOf(pusher, slotIndex, positionIndex)`), the re-delegation silently redirects those pushes to the creator's namespace. The pusher's own namespace feeds go stale. Any pool or `AnchoredPriceProvider` reading from the pusher's namespace will receive a zero/stale timestamp, which consumers reject — causing the pool to be unable to execute swaps until the pusher notices and calls `revokePusher()` again. The creator can immediately replay the signature to re-establish delegation, creating a persistent griefing loop for the duration of the original deadline.

### Likelihood Explanation
The attack requires a creator who is adversarial toward their own pusher — a semi-trusted relationship. The pusher must have signed a consent with a long-lived deadline (common in practice to avoid frequent re-signing). The creator needs only to retain the original calldata from the first `allowPushers` call (trivially available from on-chain history). No special privileges beyond being the original creator are required.

### Recommendation
Track consumed signatures with a `mapping(bytes32 => bool) usedConsents` keyed on the signature hash, and mark it used on first delegation. Alternatively, include a per-pusher nonce in the signed payload and increment it on each `revokePusher` call, so any prior signature becomes invalid after revocation.

function testSignatureReplayAfterRevoke() public {
    uint256 deadline = block.timestamp + 30 days;

    // 1. Pusher signs consent
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator delegates
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature — no new pusher consent
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // succeeds
    assertEq(oracle.namespaceRemapping(pusher), creator); // delegation re-established
}

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
