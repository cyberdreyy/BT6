### Title
Unbounded `allowPushers` deadline enables creator to replay pusher consent after self-revocation, permanently redirecting pusher writes into creator namespace — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts an arbitrary `deadline` with no upper-bound check. The NatSpec comment on the function explicitly states the deadline is the *only* mechanism preventing a creator from re-establishing a delegation after the pusher has revoked it. Because the deadline is unbounded, a pusher who signed consent with a far-future deadline (e.g. `type(uint256).max`) cannot permanently exit the delegation: the creator can replay the original signature indefinitely until the deadline expires, overwriting `namespaceRemapping[pusher]` back to the creator's address after every `revokePusher()` call. Subsequent fallback pushes from the pusher then land in the creator's namespace rather than the pusher's own, feeding the creator's oracle slots — and any pool reading those slots — with whatever prices the pusher is currently publishing.

---

### Finding Description

`allowPushers` signs over `(chainid, address(this), deadline, pusher, msg.sender)` with no nonce: [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)`: [2](#0-1) 

which only checks `block.timestamp <= deadline` — no upper bound, no nonce, no "already-revoked" check. The NatSpec comment on `allowPushers` itself acknowledges the exact risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

`revokePusher()` clears `namespaceRemapping[msg.sender] = address(0)`: [4](#0-3) 

But because the signed message contains no nonce, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple, restoring `namespaceRemapping[pusher] = creator`. The fallback push path resolves the namespace at call time: [5](#0-4) 

so every subsequent push from the pusher silently lands in the creator's namespace.

---

### Impact Explanation

A malicious creator who holds a pusher's far-future-deadline consent signature can:

1. Re-delegate the pusher after every `revokePusher()` call.
2. Force the pusher's slot writes into the creator's namespace indefinitely.
3. If the pusher has since switched to publishing prices for a different asset (e.g. BTC/USD instead of ETH/USD) and the creator's pool reads from the same slot index in the creator's namespace, the pool receives the wrong asset's price.
4. Swaps execute at the wrong oracle bid/ask, causing traders to receive more or less than the correct amount — direct loss of user principal or protocol fees.

The `price()` read path on `CompressedOracleV1` is permissionless and open: [6](#0-5) 

so any `PriceProvider` or `AnchoredPriceProvider` backed by the compressed oracle will consume the corrupted slot value without additional filtering beyond the staleness check.

---

### Likelihood Explanation

- Far-future deadlines are a natural choice for pushers who want to avoid re-signing frequently; `type(uint256).max` is a one-line off-chain choice.
- The creator is not the oracle admin (ADMIN_ROLE) — they are an unprivileged namespace owner. A malicious creator is a realistic semi-trusted adversary.
- The replay requires only a single on-chain transaction (calling `allowPushers` with the stored signature) and can be repeated after every revocation with no additional cost.
- No test in the suite covers the post-revocation replay scenario, confirming the gap is undetected.

---

### Recommendation

Add a per-pusher nonce to the signed message and increment it on every successful delegation or revocation:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]   // ← nonce added
    ))
);
pusherNonce[pusher]++;   // invalidate after use
```

Alternatively, enforce a maximum delegation window (e.g. `require(deadline <= block.timestamp + 30 days)`) so that any replayed signature expires within a bounded period after revocation.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with far-future deadline
uint256 deadline = type(uint256).max;
bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
);
(uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
bytes memory sig = abi.encodePacked(r, s, v);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1); pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1); sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 4. Creator replays the SAME signature — no revert, delegation restored
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // ← back to creator

// 5. Pusher's next push (e.g. BTC/USD prices) lands in creator's namespace
uint56 tsMs = uint56(block.timestamp * 1000 + 1);
vm.prank(pusher);
(bool ok,) = address(oracle).call(btcUsdSlotWord); // pushes to slot 5
assertTrue(ok);
// creator's pool reading feedIdOf(creator, 5, 0) now sees BTC/USD price
```
