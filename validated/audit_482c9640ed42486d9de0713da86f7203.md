### Title
Post-Revocation Signature Replay in `allowPushers` Breaks Revocation Finality — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` accepts any EIP-191 consent signature whose deadline has not yet expired. Because no nonce or used-signature record exists, the creator can replay the same signature after the pusher calls `revokePusher()`, silently re-establishing `namespaceRemapping[pusher] = creator` and redirecting all subsequent pusher fallback writes back into the creator's namespace.

### Finding Description
The signed consent message is `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`. Nothing in this message or in contract state changes when `revokePusher()` fires. The creator retains the original calldata (deadline + signature) and can call `allowPushers` again at any time before the deadline, passing the identical arguments. `_ensureDeadline` passes, ECDSA recovery succeeds, and `namespaceRemapping[pusher]` is overwritten back to `creator`.

The code comment explicitly states the deadline is the mechanism preventing this exact attack, but a deadline only blocks *expired* signatures — it does not block a still-valid signature replayed after revocation.

**Attack sequence:**
1. Pusher signs consent with `deadline = now + N` (any future time).
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator immediately calls `allowPushers(deadline, [pusher], [sig])` again (same args, deadline still valid) → `namespaceRemapping[pusher] = creator` re-established.
5. Pusher's subsequent fallback pushes (which the pusher believes land in their own namespace) are silently routed into the creator's namespace.

### Impact Explanation
After step 4, the pusher's fallback writes overwrite the creator's live oracle slots. If the pusher is now pushing data intended for their own namespace (different asset, different price scale), the creator's feeds — and any pools consuming them — receive corrupted mid prices and spreads. This constitutes bad-price execution reaching live swaps, satisfying the contest's fund-loss impact gate.

The pusher cannot escape the creator's namespace until the deadline expires, regardless of how many times they call `revokePusher()` — the creator can replay the signature in the same block.

### Likelihood Explanation
The creator holds the original `allowPushers` calldata from step 2 (it is on-chain and trivially retrievable). Replaying it requires a single additional transaction. Any creator who wants to retain a pusher against their will, or who wants to corrupt a pusher's own-namespace data by forcing it into their namespace, can do so at zero cost for the entire duration of the deadline window.

### Recommendation
Track consumed signatures. Add a `mapping(bytes32 => bool) public usedConsentHashes` and mark each `hash` as used after the first successful `allowPushers` call. Reject any replay of the same hash. Alternatively, add a per-pusher nonce to the signed message so that `revokePusher()` increments the nonce, invalidating all prior signatures.
