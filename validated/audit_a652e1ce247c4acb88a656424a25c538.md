### Title
`MetricOmmPoolFactory.createPool()` accepts arbitrary price provider addresses without verifying factory origin, enabling bad-price execution against LP principal — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool()` and `proposePoolPriceProvider()` validate a price provider only by checking that its `token0()`/`token1()` return values match the pool's tokens. There is no check that the provider was deployed by the official `PriceProviderFactory`, `PriceProviderFactoryL2`, or `AnchoredProviderFactory`. A malicious actor can deploy a contract that satisfies the token check but returns arbitrary bid/ask prices, create an official-factory pool with it, and drain LP principal through bad-price swaps.

---

### Finding Description

`_validatePriceProvider` is the sole gate between an arbitrary address and a live pool's price feed:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  L541-L546
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [1](#0-0) 

This check is called both at pool creation and at price-provider update time: [2](#0-1) [3](#0-2) 

The official oracle factories each expose an explicit **public-pool eligibility predicate** that is never consulted. `AnchoredProviderFactory` documents this intent directly in its NatSpec:

> "public-pool eligibility is then the machine-checkable predicate `recognizedFactory.isProvider(p)`" [4](#0-3) 

The `isProvider()` view exists on all three official factories but is never called by `MetricOmmPoolFactory`: [5](#0-4) [6](#0-5) 

The pool consumes the provider's output without any further validation:

```solidity
// metric-core/contracts/MetricOmmPool.sol  L804-L813
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
``` [7](#0-6) 

Any `(bid, ask)` pair where `bid > 0` and `bid < ask` passes directly into `SwapMath.midAndSpreadFeeX64FromBidAsk` and drives all swap execution. [8](#0-7) 

---

### Impact Explanation

A malicious price provider can return any valid-looking `(bid, ask)` pair. The pool's swap math uses this pair as the authoritative mid-price and spread, anchoring all bin prices to the attacker's chosen value. LPs who deposit into the pool will have their positions priced at the manipulated rate; arbitrageurs (or the attacker themselves) can then sweep LP principal by trading against the mispriced bins. Because the pool is registered in the official factory (`isPool()` returns `true`), users and integrators have no on-chain signal that the price feed is unaudited.

The corrupted value is the `(bidPriceX64, askPriceX64)` pair that flows into every swap, directly determining how much token0/token1 each swap consumes and produces.

---

### Likelihood Explanation

Pool creation via `MetricOmmPoolFactory.createPool()` is **permissionless** — any address can call it. The only prerequisite is deploying a contract that returns the correct `token0()`/`token1()` addresses and a plausible `(bid, ask)` pair during normal observation, then switching to a manipulated price at attack time. No privileged role, no admin key, and no special token behavior is required.

---

### Recommendation

Add a recognized-factory registry to `MetricOmmPoolFactory` and enforce it inside `_validatePriceProvider`. The simplest form mirrors the design intent already documented in `AnchoredProviderFactory`:

```solidity
// In MetricOmmPoolFactory storage:
address[] public recognizedProviderFactories;

// In _validatePriceProvider, after the token check:
bool recognized = false;
for (uint256 i = 0; i < recognizedProviderFactories.length; i++) {
    if (IProviderFactory(recognizedProviderFactories[i]).isProvider(priceProvider)) {
        recognized = true;
        break;
    }
}
if (!recognized) revert UnrecognizedPriceProvider();
```

Alternatively, follow the TRST-M-2 fix pattern: instead of accepting a provider address, accept the factory address and feed parameters, then derive the provider address deterministically from the official factory.

---

### Proof of Concept

1. Attacker deploys `MaliciousPriceProvider` implementing `IPriceProvider`:
   - `token0()` returns the legitimate `WETH` address
   - `token1()` returns the legitimate `USDC` address
   - `getBidAndAskPrice()` returns `(1, 2)` initially (passes pool creation checks), then switches to `(1, type(uint128).max - 1)` after LPs have deposited

2. Attacker calls `MetricOmmPoolFactory.createPool(params)` with `params.priceProvider = address(MaliciousPriceProvider)`. `_validatePriceProvider` passes because `token0()`/`token1()` match.

3. Pool is registered: `factory.isPool(pool) == true`. LPs, seeing an official-factory pool for WETH/USDC, add liquidity.

4. Attacker triggers `MaliciousPriceProvider` to return an extreme ask price. All `zeroForOne` swaps now execute at a price near zero for token1 output. The attacker swaps a tiny amount of token0 and receives nearly all token1 in the pool's bins, draining LP principal. [9](#0-8) [1](#0-0) [5](#0-4)
