### Title
Pusher Delegation Signature Replay Allows Creator to Permanently Override Revocation Within Deadline Window - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `allowPushers` function in `CompressedOracleV1` uses a `deadline` as the sole replay-protection mechanism for pusher delegation signatures. Because no nonce is included in the signed payload and no "used-signature" bitmap is maintained, a creator can replay the same valid signature an unlimited number of times before the deadline expires — re-establishing a delegation that the pusher has already revoked.

---

### Finding Description

`allowPushers` verifies a pusher's EIP-191 consent signature over the tuple `(block.chainid, address(this), deadline, pusher, msg.sender)`: [1](#0-0) 

The code comment explicitly acknowledges the replay concern — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — and treats the `deadline` as the fix: [2](#0-1) 

However, the `deadline` only prevents use of the signature *after* it expires. It does **not** prevent the creator from calling `allowPushers` a second (or third, or Nth) time with the **identical** signature bytes before the deadline. There is no nonce, no consumed-signature set, and no monotonic counter in the signed payload or in contract state.

`revokePusher` zeroes the mapping: [3](#0-2) 

But `allowPushers` unconditionally overwrites it again with the same signature: [4](#0-3) 

`_ensureDeadline` only checks `block.timestamp <= deadline`: [5](#0-4) 

So the creator can loop: `allowPushers` → pusher calls `revokePusher` → creator calls `allowPushers` again with the same bytes → repeat until the deadline expires.

---

### Impact Explanation

While the delegation is forcibly re-established, every `fallback` push from the pusher is routed into the **creator's** namespace instead of the pusher's own: [6](#0-5) 

Consequences:

1. **Pusher namespace hijack**: The pusher cannot write to their own feeds — all pushes land in the creator's namespace — until the deadline expires. Any pool or provider reading the pusher's own `feedId` (keyed by the pusher's address) receives stale/zero data, which downstream providers reject as stale, breaking pool pricing.
2. **Forced price injection**: If the creator's pool is bound to a `feedId` in the creator's namespace, the creator can force the pusher's price stream into that pool's oracle slot, bypassing the pusher's intent to stop feeding that pool.
3. **Revocation is ineffective**: The security invariant stated in the NatSpec — that a pusher can always revoke consent — is broken for the entire deadline window.

---

### Likelihood Explanation

- Any creator who obtained a pusher's signature (the normal onboarding flow) retains the raw signature bytes.
- Replaying requires a single cheap `allowPushers` call with no additional privileges.
- The deadline is set by the creator at call time and can be arbitrarily far in the future (the contract imposes no maximum deadline).
- The pusher has no on-chain mechanism to invalidate the signature before the deadline expires.

Likelihood: **High** — the attack requires only the signature already exchanged during normal delegation setup.

---

### Recommendation

Add a per-pusher nonce to the signed payload and increment it on every successful `allowPushers` call (or maintain a `mapping(address => uint256) public pusherNonce`). The signed message should become:

```solidity
keccak256(abi.encode(
    block.chainid,
    address(this),
    deadline,
    pusherNonce[pusher],   // <-- added
    pusher,
    msg.sender
))
pusherNonce[pusher]++;
```

Alternatively, record consumed signature hashes in a `mapping(bytes32 => bool) usedSignatures` set and revert on reuse. Either approach ensures each signed consent can be consumed exactly once, matching the security model described in the NatSpec.

---

### Proof of Concept

```
1. Pusher signs: keccak256(abi.encode(chainid, oracle, deadline=T+1h, pusher, creator))
   → sig = 0xABCD...

2. Creator calls allowPushers(T+1h, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)

4. Creator calls allowPushers(T+1h, [pusher], [sig])  ← SAME sig, no revert
   → namespaceRemapping[pusher] = creator  ✗ (revocation undone)

5. Steps 3–4 repeat until block.timestamp > T+1h.
   During this window every push from the pusher feeds the creator's namespace,
   and the pusher's own feedIds receive no updates (stale/zero price).
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
