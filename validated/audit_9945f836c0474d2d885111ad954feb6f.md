### Title
Pusher delegation consent signature lacks a nonce, allowing a creator to replay a revoked `allowPushers` signature and silently redirect a pusher's oracle slot writes into the creator's namespace ŌĆö (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` signs `(chainid, address(this), deadline, pusher, creator)` with no nonce or one-time-use flag. After a pusher calls `revokePusher()` to clear `namespaceRemapping[pusher]`, the creator can immediately replay the original, still-valid signature to re-establish the delegation. The pusher's subsequent `fallback` pushes ŌĆö which the pusher believes land in their own namespace ŌĆö are silently redirected into the creator's namespace, feeding the creator's oracle slots with data the pusher never intended for them. Pools registered to the creator's feeds then consume those misrouted prices.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only replay protection is the `deadline` field, enforced by `_ensureDeadline`:

```solidity
require(block.timestamp <= deadline, DeadlineExceeded());
``` [2](#0-1) 

There is no nonce, no per-pusher revocation counter, and no consumed-signature registry. The code's own comment acknowledges the risk:

> "the deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The comment treats the deadline as the fix, but the deadline only prevents replay *after* it expires. Within the deadline window, the same signature is accepted an unlimited number of times. `revokePusher` clears `namespaceRemapping[pusher]` to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [4](#0-3) 

But `allowPushers` unconditionally overwrites it back to `msg.sender` (the creator) on replay:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [5](#0-4) 

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [6](#0-5) 

So after the creator replays the signature, every subsequent `fallback` call from the pusher writes into the creator's namespace, not the pusher's own. The pusher has no on-chain signal that the remapping was re-established.

---

### Impact Explanation

The creator's oracle slots receive price data the pusher intended for their own namespace. If the pusher's slot/position assignments differ from the creator's (e.g., the pusher's slot 0 / position 0 is ETH/USD while the creator's same coordinates are BTC/USD), the creator's BTC/USD feed is overwritten with ETH/USD prices. Any pool registered to the creator's feed via `OracleBase.register` then executes swaps against a misrouted bid/ask quote. This is a direct bad-price execution path: the pool's `swap` ŌåÆ `getBidAndAskPrice` ŌåÆ `oracle.price(feedId, pool)` chain reads the corrupted slot value and applies it to live trades.

Simultaneously, the pusher's own namespace receives no updates (it stays at the last value before revocation), so any pool or integrator reading the pusher's own feeds sees stale prices.

---

### Likelihood Explanation

The trigger is semi-privileged: the creator must hold a still-valid (non-expired) pusher consent signature. Consent signatures are routinely shared off-chain as part of the delegation workflow, and deadlines are typically set days or weeks in the future to avoid operational friction. A creator who loses a pusher (e.g., the pusher switches to a competing protocol) has a clear economic incentive to replay the old signature and keep their feeds live. The pusher has no on-chain mechanism to invalidate the signature before the deadline expires and no automatic notification that the remapping was re-established.

---

### Recommendation

Add a per-pusher revocation nonce to the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]
    ))
);
pusherNonce[pusher]++;   // invalidates all prior signatures for this pusher
```

Alternatively, record consumed signatures in a `mapping(bytes32 => bool) usedSignatures` set and revert on reuse. Either approach ensures that `revokePusher` permanently invalidates all previously issued consent signatures regardless of their deadline.

### Proof of Concept

```
1. Pusher P signs consent for creator A:
       digest = keccak256(chainid, oracle, deadline=T+30days, P, A)
       sig = sign(P_key, digest)

2. Creator A calls allowPushers(T+30days, [P], [sig])
       ŌåÆ namespaceRemapping[P] = A

3. P calls revokePusher()
       ŌåÆ namespaceRemapping[P] = address(0)
       P now believes its fallback pushes go to its own namespace.

4. Creator A calls allowPushers(T+30days, [P], [sig])  ŌåÉ SAME sig, still valid
       ŌåÆ namespaceRemapping[P] = A  (silently re-established)

5. P pushes slot word for its own ETH/USD feed (slot 0, pos 0):
       fallback: creator = namespaceRemapping[P] = A  ŌåÉ not address(0)
       key = A_namespace | slotId(0)
       ŌåÆ A's slot 0 is overwritten with P's ETH/USD price

6. Pool registered to A's feedIdOf(A, 0, 0) (A's BTC/USD feed) calls swap
       ŌåÆ oracle returns P's ETH/USD price as BTC/USD bid/ask
       ŌåÆ bad-price execution against live liquidity
```
