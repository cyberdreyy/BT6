### Title
Creator Can Replay Pusher Consent Signature to Re-Establish Revoked Delegation Within Deadline Window, Enabling Continued Bad-Price Injection - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`CompressedOracleV1.allowPushers` does not invalidate a pusher's consent signature after `revokePusher()` is called. Within the deadline window the creator can replay the original on-chain signature to immediately re-establish the delegation, making the pusher's self-revocation a no-op and allowing continued arbitrary writes into the creator's namespace — which flows directly into pool bid/ask pricing.

### Finding Description
`allowPushers` establishes pusher delegation by verifying an EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)`: [1](#0-0) 

The code comment explicitly acknowledges the replay risk: *"an undated signature could re-establish a delegation AFTER the pusher revoked it."* The deadline is the stated protection. However, the deadline only prevents re-establishment **after** it expires — not **within** the window. There is no nonce, no used-signature bitmap, and no on-chain record that a pusher ever revoked.

When a pusher calls `revokePusher()`, `namespaceRemapping[pusher]` is cleared to `address(0)`: [2](#0-1) 

The creator can immediately call `allowPushers` again with the **identical** `(deadline, pusher, signature)` tuple. `_ensureDeadline` passes (deadline not yet expired), `ECDSA.recover` returns the same pusher address, and `namespaceRemapping[pusher]` is written back to `creator`. The revocation is silently undone.

The signature is public — it was submitted as calldata in the original `allowPushers` transaction and is permanently visible on-chain.

The fallback push path resolves the namespace at call time: [3](#0-2) 

So any push from the pusher's address after the creator re-establishes the delegation lands in the creator's namespace, not the pusher's own. Those prices are then returned by `price()` and consumed by `ProtectedPriceProvider._getBidAndAskPrice()` → pool swap. [4](#0-3) 

### Impact Explanation
If a pusher's key is compromised and the pusher calls `revokePusher()` to stop the attacker from injecting bad prices, a creator colluding with the attacker can immediately replay the original consent signature to re-establish the delegation. The attacker continues pushing arbitrary prices into the creator's namespace. These prices flow through `CompressedOracleV1.price()` → `ProtectedPriceProvider._getBidAndAskPrice()` → pool swap, causing bad-price execution and direct loss of trader principal. The `ProtectedPriceProvider` staleness and price-guard checks do not help if the attacker pushes a fresh, in-range but manipulated price.

### Likelihood Explanation
Medium. The creator must be malicious or colluding with the attacker who holds the compromised pusher key. The original consent signature is permanently public on-chain (calldata of the first `allowPushers` call), so no additional off-chain coordination is needed to replay it. The window of exposure equals the remaining time until the deadline — which can be up to the full duration the pusher originally agreed to (e.g., days or weeks).

### Recommendation
Track consumed signatures or add a per-pusher revocation nonce. Include the nonce in the signed hash and increment it on every `revokePusher()` / `removePushers()` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher / removePushers:
pusherNonce[pusher]++;
```

This invalidates all prior signatures for that pusher the moment they revoke, regardless of the deadline.

### Proof of Concept
```
// 1. Pusher signs consent for creator with deadline = T + 1 day
sig = sign(keccak256(abi.encode(chainid, oracle, T+1day, pusher, creator)))

// 2. Creator establishes delegation
oracle.allowPushers(T+1day, [pusher], [sig])
// → namespaceRemapping[pusher] = creator

// 3. Pusher's key is compromised; pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
// → namespaceRemapping[pusher] = address(0)  ← appears to succeed

// 4. Creator (colluding with attacker) replays the SAME signature
vm.prank(creator);
oracle.allowPushers(T+1day, [pusher], [sig]);
// _ensureDeadline: T+1day > block.timestamp ✓
// ECDSA.recover(hash, sig) == pusher ✓
// → namespaceRemapping[pusher] = creator  ← revocation silently undone

// 5. Attacker pushes manipulated price via compromised pusher key
// → lands in creator namespace, consumed by pool swap

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
