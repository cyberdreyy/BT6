### Title
Creator Can Replay Pusher Consent Signature to Re-Establish Delegation After Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` accepts a deadline-bounded EIP-191 signature as the sole replay guard. The code comment explicitly acknowledges that the deadline is required to prevent re-establishment of delegation after a pusher revokes. However, the deadline only prevents use *after* it expires — there is no nonce or used-signature tracking. Within the deadline window, the creator can replay the exact same signature an unlimited number of times, re-establishing `namespaceRemapping[pusher] = creator` immediately after every `revokePusher()` call. This is the direct analog of the UStb M-02 bug: a state change (revocation) does not fully prevent the action (re-delegation) because a check is missing (no consumed-signature record).

### Finding Description

`allowPushers` in `CompressedOracle.sol`:

```solidity
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← overwrites unconditionally
    emit PusherAuthorized(pusher, msg.sender);
}
``` [1](#0-0) 

The code comment at line 189–191 states: *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* The deadline is presented as the fix. But `_ensureDeadline` only checks `block.timestamp <= deadline`: [2](#0-1) 

There is no mapping of consumed signatures, no per-pusher nonce, and no check that `namespaceRemapping[pusher]` is currently zero before writing. The creator holds the original signature bytes and can call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple at any point before the deadline expires.

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

But the creator can immediately replay the old signature to restore `namespaceRemapping[pusher] = creator`, creating a race condition that lasts for the entire deadline window (which can be set to days).

### Impact Explanation

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

After the creator re-establishes delegation, every subsequent fallback push from the pusher lands in the **creator's namespace** — not the pusher's own namespace as the pusher intended. If the pusher has revoked because they are now pushing data for a different asset or a different slot layout, that mismatched data overwrites the creator's feed slots. Any `AnchoredPriceProvider` or `PriceProvider` wired to the creator's `feedId` will read the corrupted price and deliver it to a pool swap, satisfying the bad-price execution impact gate. [5](#0-4) 

### Likelihood Explanation

The creator holds the original signature bytes off-chain and can replay them permissionlessly at any time before the deadline. No privileged role, no special setup, and no on-chain state change is required beyond the creator calling `allowPushers` again. The pusher can detect the re-delegation by reading `namespaceRemapping` and call `revokePusher` again, but the creator can immediately replay again — the race continues until the deadline expires. Deadlines are typically set to hours or days, making the window practically exploitable.

### Recommendation

Track consumed signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedSignatures` set. Alternatively, include a monotonically increasing nonce in the signed payload and store the last accepted nonce per pusher, so that a revocation implicitly invalidates all prior signatures:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
uint256 nonce = pusherNonce[pusher]++;
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, nonce, pusher, msg.sender))
);
```

Alternatively, bump `pusherNonce[pusher]` inside `revokePusher` so that any previously signed consent is invalidated on revocation.

### Proof of Concept

```
1. Pusher signs consent:
   sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+1day, pusher, creator)))

2. Creator calls allowPushers(T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes delegation is gone)

4. Creator calls allowPushers(T+1day, [pusher], [sig])  ← SAME sig, SAME deadline
   → _ensureDeadline passes (T+1day > block.timestamp)
   → ECDSA.recover returns pusher  ✓
   → namespaceRemapping[pusher] = creator  ← delegation silently restored

5. Pusher (now pushing for their own namespace, e.g. ETH/USD at slot 0 pos 0)
   calls fallback with ETH/USD price data.
   → creator = namespaceRemapping[pusher] = creator (not address(0))
   → data lands in creator's slot 0 pos 0 (the creator's BTC/USD feed)

6. AnchoredPriceProvider reads creator's BTC/USD feedId
   → gets ETH/USD price (~$3,000) instead of BTC/USD (~$100,000)
   → pool swap executes at a 33× wrong price
``` [6](#0-5) [7](#0-6)

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
