### Title
Pusher consent signature replay allows creator to re-establish a revoked delegation within the deadline window, routing the pusher's price data into the wrong namespace and leaving the pusher's own feeds stale — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature that commits to `(chainId, oracle, deadline, pusher, creator)` but includes **no nonce and no one-time-use guard**. After a pusher calls `revokePusher()`, the creator still holds the original signature and can replay it — with the same deadline, which is still valid — to silently re-establish the delegation. Every subsequent push from the pusher then lands in the creator's namespace instead of the pusher's own, leaving the pusher's own feeds with a stale timestamp. Any pool or price provider reading from the pusher's namespace receives a stale or zero-price quote.

---

### Finding Description

In `allowPushers` the signed digest is:

```solidity
// CompressedOracle.sol line 204-207
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
``` [1](#0-0) 

The only freshness gate is `_ensureDeadline(deadline)`:

```solidity
// OracleBase.sol line 124-126
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

`revokePusher` clears the mapping:

```solidity
// CompressedOracle.sol line 238-243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

Because the signature carries no nonce and no "already-used" registry exists, the creator can immediately call `allowPushers` again with the **identical** `(deadline, pusher, sig)` tuple. The deadline check passes (it has not expired), the ECDSA recovery succeeds (the signature is still valid), and `namespaceRemapping[pusher]` is overwritten back to `creator`. The revocation is silently undone.

The code comment at lines 189–191 acknowledges the replay risk but incorrectly concludes that the deadline alone prevents it:

> "an undated signature could re-establish a delegation AFTER the pusher revoked it" [4](#0-3) 

The deadline prevents replay only **after** it expires, not during its validity window (which the test suite sets to `block.timestamp + 1 days`, giving a 24-hour replay window after every revocation).

Once the delegation is re-established, the fallback push path resolves the namespace from the remapping:

```solidity
// CompressedOracle.sol line 315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

All subsequent pushes from the pusher write into the creator's namespace. The pusher's own namespace (`feedIdOf(pusher, slotIndex, positionIndex)`) receives no new data and its stored timestamp freezes at the last pre-revocation value.

---

### Impact Explanation

Any pool or `PriceProvider` configured to read from `feedIdOf(pusher, slot, pos)` will receive a stale price whose `refTime` is the timestamp of the last push before revocation. Depending on the consumer's `maxTimeDrift` setting:

- **If drift tolerance is tight**: every swap call that reads the feed reverts, making the pool's swap path permanently unusable — broken core pool functionality.
- **If drift tolerance is loose or absent**: the stale bid/ask quote passes through to `SwapMath`, executing trades at a price that no longer reflects the market — direct bad-price execution and potential loss of trader principal.

This matches the allowed impact gates: *bad-price execution (stale bid/ask quote reaches a pool swap)* and *broken core pool functionality causing unusable swap flows*.

---

### Likelihood Explanation

**Medium.** The creator already holds the original signature (they submitted it in the first `allowPushers` call and it is visible on-chain). The only precondition is that the deadline has not yet expired. Deadlines are typically set days or weeks in the future, giving the creator a large replay window after every revocation attempt. No special privilege or off-chain capability is required beyond retaining the calldata from the original transaction.

---

### Recommendation

Track consumed delegation signatures with a `usedSignatures` set, or include a per-pusher monotonic nonce in the signed message and increment it on every successful delegation or revocation:

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
