### Title
`PriceGuard` min/max bounds are stored but never enforced in `OracleBase._readPrice()`, allowing a near-zero oracle price to reach pool swaps — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase` exposes a `setPriceGuard(feedId, minPrice, maxPrice)` function and stores the resulting `PriceGuard` struct, but the internal `_readPrice()` function — the single path used by both `price(feedId, pool)` and `integratorPrice()` — never reads or enforces those bounds. A fresh, verified oracle push carrying a near-zero price passes through to the pool unchanged, exactly mirroring the M-07 pattern where no minimum floor existed on the auction's `newRatio`.

---

### Finding Description

`OracleBase.setPriceGuard` writes a `PriceGuard{min, max}` record:

```solidity
// OracleBase.sol L88-97
function setPriceGuard(bytes32 feedId, uint128 minPrice, uint128 maxPrice)
    external checkRole(feedId)
{
    require(minPrice < maxPrice);
    priceGuard[feedId] = PriceGuard({min: minPrice, max: maxPrice});
    emit PriceGuardUpdated(feedId, minPrice, maxPrice);
}
```

The wiki explicitly states: *"If the oracle returns a price outside `[min, max]`, the system treats the feed as invalid/stalled."*

However, `_readPrice()` — the only internal function that feeds both the pool path and the integrator path — simply returns raw storage:

```solidity
// OracleBase.sol L187-194
function _readPrice(bytes32 feedId)
    internal view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = _oracleDataRaw(feedId);
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
```

`priceGuard[feedId]` is never read here. The `feedExists` modifier only checks `timestampMs != 0`; it does not validate price. A Pyth or Chainlink push that carries a fresh timestamp but a near-zero price (e.g., `price = 1`) satisfies every existing guard and is returned verbatim to the `PriceProvider`.

The `PriceProvider` pipeline then computes:

```
delta = (midPrice * oracleSpread * confidenceParam) / CONFIDENCE_BASE
bid   = midPrice - delta   ≈ 0
ask   = midPrice + delta   ≈ 0
```

Both Q64-scaled values collapse toward zero, and the pool executes swaps at those prices.

---

### Impact Explanation

A pool swap priced at near-zero bid/ask allows a trader to receive the pool's token-B reserves while paying essentially nothing in token-A. LP principal is directly drained. The severity matches or exceeds M-07 because the oracle price is the sole pricing authority for every swap — there is no internal reserve ratio to act as a secondary floor.

---

### Likelihood Explanation

The external report's cited scenarios (Arbitrum downtime, Ethereum client fork, network congestion) are real precedents. During such events, Pyth or Chainlink data streams can deliver stale or anomalous prices. A fresh push with a near-zero price satisfies the monotonicity timestamp check and the `feedExists` guard, so it is stored and immediately served. No privileged actor is required; the trigger is a valid oracle update arriving during a network anomaly.

---

### Recommendation

Enforce the `PriceGuard` inside `_readPrice()`:

```solidity
function _readPrice(bytes32 feedId)
    internal view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    OracleData memory data = _oracleDataRaw(feedId);
    PriceGuard memory guard = priceGuard[feedId];
    if (guard.min != 0 || guard.max != 0) {
        require(
            data.price >= guard.min && data.price <= guard.max,
            PriceOutOfBounds(feedId, data.price)
        );
    }
    return (uint256(data.price), uint256(data.spread0), data.spread1, data.timestampMs.toSeconds());
}
```

Additionally, require `data.price > 0` unconditionally before returning, so an unpushed or zeroed feed never reaches a pool even without a configured guard.

---

### Proof of Concept

1. Admin calls `setPriceGuard(feedId, 1e8, 1e12)` — a guard is stored for a BTC/USD feed.
2. During a network anomaly, the Pyth relayer pushes a payload with `price = 1` (1 satoshi) and a fresh `timestampMs`.
3. `PythOracle` verifies the provider signature and stores the data via `oracleData[feedId] = OracleData{price: 1, ...}`.
4. A pool swap calls `PriceProvider.getBidAndAskPrice()` → `OracleBase.price(feedId, pool)` → `_readPrice(feedId)`.
5. `_readPrice` returns `mid = 1` without consulting `priceGuard[feedId]`.
6. `PriceProvider` computes `bid ≈ 0`, `ask ≈ 0` in Q64.
7. The pool executes the swap, transferring its full token-B reserve to the attacker for a negligible token-A input. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
