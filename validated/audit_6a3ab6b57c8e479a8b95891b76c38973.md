### Title
Replayed `allowPushers` Consent Signature Re-establishes Delegation After Pusher Revocation — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()`, the creator can replay the original signature (while the deadline is still valid) to silently re-establish delegation, redirecting the pusher's subsequent fallback pushes back into the creator's namespace without fresh consent.

### Finding Description

`allowPushers` constructs a hash over `(block.chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's signature against it:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));

namespaceRemapping[pusher] = msg.sender;
```

There is no used-signature bitmap, no per-pusher nonce, and no on-chain record that this exact `(deadline, pusher, creator)` tuple was already consumed. The code comment acknowledges the deadline is required to prevent re-establishing delegation after revocation, but the deadline only prevents *expired* signatures — a signature with a future deadline can be replayed an unlimited number of times within its validity window.

The attack sequence:

1. Pusher signs consent for creator with `deadline = block.timestamp + 7 days`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator immediately calls `allowPushers(deadline, [pusher], [sig])` again with the **identical** signature and deadline (still in the future).
5. Signature verification passes; `namespaceRemapping[pusher] = creator` is written again.
6. All subsequent fallback pushes from the pusher silently land in the creator's namespace, not the pusher's own.

The pusher has no on-chain mechanism to permanently invalidate the old signature short of waiting for the deadline to expire.

### Impact Explanation

After step 6, the pusher believes their pushes update their own namespace feeds (the registrationless default after revocation). Instead, every push overwrites the creator's namespace slots. If the creator's namespace feeds are consumed by a live pool via `AnchoredPriceProvider`, the pool receives prices the pusher did not intend to provide for those feeds — a bad-price execution path. Simultaneously, the pusher's own feeds remain at timestamp 0 (stale), so any pool or provider that was supposed to read the pusher's own namespace gets a zero/stale quote.

This satisfies the "bad-price execution: stale, inverted, unbounded, or unclamped bid/ask quote reaches a pool swap" impact gate.

### Likelihood Explanation

The creator is the party who calls `allowPushers` and therefore already possesses the pusher's signature. Re-replaying it requires no additional capability — just re-submitting the same calldata. The window is the full deadline duration (typically days). Any creator who wants to retain a pusher's data feed against the pusher's will can do so trivially.

### Recommendation

Mark each `(pusher, creator, deadline)` tuple as consumed on first use with a `mapping(bytes32 => bool) usedConsents` check, or include a per-pusher nonce in the signed message and increment it on each successful `allowPushers` call. Either approach ensures that `revokePusher()` permanently terminates the authority of any previously issued consent signature.

```solidity
// Option A: consumed-signature bitmap
bytes32 consentKey = keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender));
require(!usedConsents[consentKey], "consent already used");
usedConsents[consentKey] = true;
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

    /// @notice Allows a pusher to self-revoke their delegation. After revocation the
    ///         wallet pushes into its OWN namespace again (the registrationless default).
    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
