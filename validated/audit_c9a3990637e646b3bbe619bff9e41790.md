### Title
`allowPushers` consent signature is replayable within the deadline window, allowing a creator to silently re-establish delegation after `revokePusher()` and route bad prices into their namespace — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers()` verifies an EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()`, the creator can replay the original signature — while the deadline is still valid — to silently re-establish `namespaceRemapping[pusher] = creator`. The pusher's revocation is therefore not final within the deadline window. If the pusher revoked because their key was compromised, the creator (unaware) can re-establish delegation, allowing the compromised key to push bad prices into the creator's namespace and from there into any pool that reads those feeds.

---

### Finding Description

`allowPushers` hashes the consent as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no `usedSignatures` bitmap, and no per-pusher revocation counter. The only replay guard is the deadline check: [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But the original consent signature is still cryptographically valid until the deadline expires. The creator can immediately call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, passing the `_ensureDeadline` check and the ECDSA recovery check, and writing `namespaceRemapping[pusher] = creator` again. The code's own comment acknowledges this exact risk:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [4](#0-3) 

The comment treats the deadline as the mitigation, but the deadline only prevents replay **after** it expires — not within the window. A 30-day deadline leaves a 30-day window during which the creator can replay the signature an unlimited number of times, each time overwriting the pusher's revocation.

The `fallback` push path resolves the namespace at call time: [5](#0-4) 

So every push the pusher makes after believing they have revoked still lands in the creator's namespace, not their own.

---

### Impact Explanation

**Bad-price execution path**: A pusher revokes because their signing key is compromised. The creator, unaware of the compromise, replays the original consent signature to re-establish delegation. The attacker (holding the compromised key) now pushes arbitrary prices into the creator's namespace. Any pool whose `AnchoredPriceProvider` or `ProtectedPriceProvider` reads a feed from that creator's namespace receives the attacker-controlled mid/spread values and executes swaps at bad prices.

**DoS path**: A pusher revokes to push into their own namespace. The creator replays the signature. The pusher's own namespace (`feedIdOf(pusher, slot, pos)`) is never updated; any pool reading those feeds gets `price = 0 / timestamp = 0`, which every consumer rejects as stale (`FeedStalled`), making the pool unquotable.

---

### Likelihood Explanation

The creator holds the pusher's original signature from the on-chain `allowPushers` transaction (publicly visible in calldata). Replaying it costs one transaction. The creator may do this inadvertently (not knowing the pusher revoked for a security reason) or deliberately. Deadlines in practice are set days to weeks in the future, giving a wide replay window.

---

### Recommendation

Mark each consent signature as consumed on first use:

```solidity
mapping(bytes32 => bool) public usedConsentSignatures;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
        );
        require(!usedConsentSignatures[hash], "SignatureAlreadyUsed");
        require(pusher == ECDSA.recover(hash, signatures[i]));

        usedConsentSignatures[hash] = true;   // ← consume the signature
        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

Alternatively, include a per-pusher nonce in the signed digest so that each consent is unique and cannot be replayed even with a fresh deadline.

### Proof of Concept

```
1. Pusher P signs: keccak256(abi.encode(chainid, oracle, deadline=now+30d, P, C))
   → sig_P

2. Creator C calls allowPushers(deadline, [P], [sig_P])
   → namespaceRemapping[P] = C  ✓

3. P's key is compromised; P calls revokePusher()
   → namespaceRemapping[P] = address(0)  ✓

4. C (unaware) calls allowPushers(deadline, [P], [sig_P])  ← same sig, same deadline
   → _ensureDeadline passes (deadline not yet expired)
   → ECDSA.recover returns P  ✓
   → namespaceRemapping[P] = C  (revocation silently undone)

5. Attacker (holding P's compromised key) calls oracle.fallback with
   crafted slot word: price = 0, spread = BPS_BASE sentinel
   → lands in C's namespace (feedIdOf(C, slot, pos))

6. Pool reading feedIdOf(C, slot, pos) via AnchoredPriceProvider
   receives attacker-controlled mid/spread → bad-price execution
``` [6](#0-5) [7](#0-6)
