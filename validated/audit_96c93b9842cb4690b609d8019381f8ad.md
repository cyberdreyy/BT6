### Title
`allowPushers` Delegation Signature Lacks Nonce — Creator Can Replay Consent After Pusher Revokes, Silently Re-establishing Namespace Control — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` binds a pusher's EIP-191 signature to `(chainid, address(this), deadline, pusher, creator)`. There is no nonce and no "consumed signature" record. A creator who stores the original signature can call `allowPushers` again with the identical arguments at any point before the deadline, re-establishing delegation even after the pusher has explicitly called `revokePusher()`. The code comment on line 189 acknowledges the exact threat ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but the deadline alone does not close it — it only closes replay *after* the deadline expires, not replay *within* the deadline window.

---

### Finding Description

`allowPushers` in `CompressedOracle.sol` verifies the pusher's EIP-191 signature over:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

`_ensureDeadline` only checks `block.timestamp <= deadline`: [2](#0-1) 

There is no nonce, no per-pusher revocation flag, and no mapping of consumed signatures. After `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`: [3](#0-2) 

…the creator can immediately call `allowPushers` again with the original `(deadline, pusher, signature)` tuple. Because the deadline has not expired, `_ensureDeadline` passes, the hash recomputes identically, ECDSA recovery succeeds, and `namespaceRemapping[pusher]` is written back to the creator's address: [4](#0-3) 

The pusher's revocation is silently undone.

---

### Impact Explanation

After re-established delegation, every subsequent fallback push from the pusher's address resolves `namespaceRemapping[msg.sender]` to the creator and writes into the **creator's namespace**: [5](#0-4) 

The creator's oracle feeds — consumed by `price()` and ultimately by pool swaps — are updated with data the pusher no longer intends to provide for that namespace. If the pusher has revoked because they are now pushing data for a different asset, a different price scale, or a different purpose, the creator's oracle receives semantically wrong slot data. Any pool whose `PriceProvider` reads those feeds will execute swaps against a corrupted bid/ask, satisfying the **bad-price execution** impact gate (stale, inverted, or wrong-asset quote reaching a pool swap).

---

### Likelihood Explanation

- The creator must have retained the original signature bytes — trivially true for any on-chain or off-chain indexer watching the `allowPushers` calldata.
- The deadline must not have expired. Pushers commonly sign with multi-day or multi-week deadlines to avoid operational friction; the replay window is therefore large.
- The pusher's automated push infrastructure typically does not re-check `namespaceRemapping` before each push, so the redirect is invisible to the pusher.
- No privileged role is required; the creator is a semi-trusted actor with a valid prior signature.

---

### Recommendation

Introduce a per-pusher nonce that is incremented on every successful `allowPushers` call and on every `revokePusher` / `removePushers` call. Include the nonce in the signed hash:

```solidity
mapping(address => uint256) public pusherNonce;

// in allowPushers:
uint256 nonce = pusherNonce[pusher];
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;          // invalidate the consumed signature
namespaceRemapping[pusher] = msg.sender;

// in revokePusher / removePushers:
pusherNonce[pusher]++;          // invalidate any outstanding signed consent
namespaceRemapping[pusher] = address(0);
```

Incrementing the nonce on revocation ensures that any previously issued signature — regardless of its deadline — becomes permanently invalid the moment the pusher revokes.

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 30-day deadline.
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — deadline still valid.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));  // succeeds
assertEq(oracle.namespaceRemapping(pusher), creator);    // delegation re-established

// 5. Pusher's next push (intended for own namespace) lands in creator's namespace.
uint56 tsMs = uint56(block.timestamp * 1000 + 1);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, _packRaw(BAD_PRICE, 5, 5), tsMs));
assertTrue(ok);
// Creator's feed now holds BAD_PRICE; pools reading it execute against a corrupted quote.
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, U64x32.decode(BAD_PRICE));
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
