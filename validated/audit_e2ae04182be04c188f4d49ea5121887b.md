### Title
Oracle Spread Fetched but Discarded When `confidenceParam == 0` (Default), Causing Unclamped Tight Quotes — (`smart-contracts-poc/contracts/PriceProvider.sol`)

### Summary

`PriceProvider._getBidAndAskPrice()` reads the oracle's `spread` value (representing price uncertainty in bps) and validates it against the stall marker, but then multiplies it by `confidenceParam` before using it in price computation. Because `confidenceParam` is a storage variable that initializes to `0` and must be explicitly set by the factory, the oracle spread is silently discarded in the default state. The resulting bid/ask is computed with zero spread — purely from `marginStep` — regardless of how wide the oracle's actual uncertainty is.

This is the direct analog of the CurveStableSwapNG "dynamic fee calculated but not used" bug: a value is fetched and partially validated, but the computed result is zeroed out before it reaches the price output.

---

### Finding Description

In `PriceProvider._getBidAndAskPrice()`:

```solidity
(uint256 mid, uint256 spread, , uint256 refTime) =
    IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);
...
if (mid == 0 || spread >= ORACLE_BPS) {          // spread used only as stall marker
    return (0, type(uint128).max);
}
...
uint256 adjustedSpread = spread * confidenceParam; // confidenceParam == 0 by default
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
``` [1](#0-0) 

`_getBidAskFrom` with `confidence == 0`:

```solidity
uint256 delta = midPrice * confidence / CONFIDENCE_BASE; // == 0
bid = delta >= midPrice ? 0 : midPrice - delta;          // == mid
ask = midPrice + delta;                                   // == mid
``` [2](#0-1) 

So `bid == ask == mid`. The only spread in the final quote comes from `marginStep` via `_applyBidAdjustments` / `_applyAskAdjustments`. The oracle's actual uncertainty (`spread`) is completely absent from the price output.

Contrast with `AnchoredPriceProvider._computeBidAsk`, which always incorporates `spreadBps` directly into the reference band:

```solidity
uint256 half = spreadBps * ONE_BPS_E18 + minMargin;
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
``` [3](#0-2) 

`PriceProvider` has no equivalent safety net — there is no band clamp that enforces a minimum spread equal to oracle uncertainty.

---

### Impact Explanation

When `confidenceParam == 0` (the default for every newly deployed `PriceProvider`), the pool receives a bid/ask spread determined solely by `marginStep`, which can be arbitrarily small. If the oracle's true uncertainty (`spread`) is large — e.g., during high volatility or a Pyth/Chainlink confidence widening event — the pool still quotes a tight spread. Sophisticated traders can:

1. Observe that the oracle's reported spread is wide (true price is uncertain over a large range).
2. Trade against the pool at the artificially tight bid/ask.
3. The pool's LPs bear the adverse selection loss because the quote did not account for oracle uncertainty.

This constitutes **bad-price execution** (unclamped quote reaches a pool swap) and **direct loss of LP principal** above Sherlock thresholds when oracle spread is non-trivial.

---

### Likelihood Explanation

`confidenceParam` is a storage variable with no constructor argument and no default non-zero value. Every `PriceProvider` deployment starts with `confidenceParam == 0`. The factory must call `setConfidenceParam` with a non-zero value to activate spread incorporation, and there is a 1-minute cooldown between updates. Any pool that goes live before this call — or any pool whose factory owner never calls it — operates permanently in the zero-spread state. This is the default, not an edge case. [4](#0-3) 

---

### Recommendation

Remove the dependency on `confidenceParam` for incorporating the oracle's own spread. The oracle-reported `spread` should be applied unconditionally as the minimum half-width of the bid/ask band, mirroring `AnchoredPriceProvider`'s reference band logic. `confidenceParam` (if retained) should only be allowed to **widen** the spread beyond the oracle's floor, never to zero it out:

```solidity
// Minimum spread = oracle's own spread; confidenceParam can only widen it.
uint256 effectiveSpread = spread + spread * confidenceParam / CONFIDENCE_BASE;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, effectiveSpread);
```

Alternatively, enforce a band clamp analogous to `AnchoredPriceProvider`'s `refBid`/`refAsk` so that no quote can be tighter than `mid ± oracle_spread`.

---

### Proof of Concept

1. Deploy `PriceProvider` with `marginStep = 10` (tiny), `MAX_TIME_DELTA = 1 hour`. `confidenceParam` defaults to `0`.
2. Oracle reports `mid = 1e8` (price = $1.00), `spread = 500` (5% uncertainty, 500 bps).
3. Call `getBidAndAskPrice()`:
   - `adjustedSpread = 500 * 0 = 0`
   - `_getBidAskFrom(1e8, 0)` → `bid = 1e8`, `ask = 1e8`
   - `_applyBidAdjustments(1e8)` → `bidOut ≈ 1e8 * Q64 * (BPS_BASE_U - 10) / STEP_DENOM` (negligibly below mid)
   - `_applyAskAdjustments(1e8)` → `askOut ≈ 1e8 * Q64 * (BPS_BASE_U + 10) / STEP_DENOM` (negligibly above mid)
   - Effective spread: ~0.000001% (from marginStep=10), not 5% (oracle spread).
4. A trader sells into the pool at the ask (near mid), knowing the oracle's true uncertainty is ±5%. The LP absorbs the adverse selection loss. [5](#0-4)
