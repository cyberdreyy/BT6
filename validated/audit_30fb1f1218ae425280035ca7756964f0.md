### Title
Missing Minimum-Age Lower Bound in `revertIfAfterBlockTimeWithDrift` Allows Replay of Stale Pyth Lazer Payloads into Uninitialized Feeds â€” (`smart-contracts-poc/contracts/oracles/utils/LazerConsumer.sol`)

### Summary
`revertIfAfterBlockTimeWithDrift` only rejects timestamps in the future. It imposes no lower bound on how old a timestamp may be. An attacker who possesses a legitimately signed but old Pyth Lazer payload can replay it into a feed whose `timestampMs` is still zero, storing a price anchored to a stale market state that pools will subsequently read.

### Finding Description
`TimeMs.revertIfAfterBlockTimeWithDrift` enforces only `t0.toSeconds() <= block.timestamp + drift`. There is no symmetric check `t0.toSeconds() >= block.timestamp - MAX_TIME_DRIFT`. Pyth Lazer payloads are broadcast messages; old ones carry valid signatures indefinitely. The permissionless `PythOracle.fallback()` accepts any caller. For a feed whose `oracleData[feedId].timestampMs == 0`, the monotonicity guard (`isAfter`) is trivially satisfied by any non-zero timestamp, including one from 24 hours ago. The stale `OracleData` (price + spread + timestampMs) is written to storage and becomes the authoritative price for any registered pool.

### Impact Explanation
A pool registered for the affected feed calls `price(feedId, pool)` during a swap. `_readPrice` returns the stale `mid`, `spread0`, and `refTime`. If the pool does not independently enforce a maximum `refTime` age, it executes the swap at a price anchored to a market state that is hours old â€” constituting bad-price execution and potential loss of funds for one side of the trade.

### Likelihood Explanation
- Old Pyth Lazer payloads are publicly broadcast and trivially obtainable.
- The attack window is the period between feed creation and the first legitimate push. For a newly deployed oracle or a newly added feed, this window may be non-trivial.
- The fallback is permissionless; no special role is required.
- Likelihood is **Medium**: requires timing and access to old payloads, but no privileged access.

### Recommendation
Add a symmetric lower-bound check in `revertIfAfterBlockTimeWithDrift` (or a separate function called alongside it):

```solidity
function revertIfStale(TimeMs t0, uint256 maxAgeSeconds) view {
    require(t0.toSeconds() + maxAgeSeconds >= block.timestamp, StaleTimestamp());
}
```

Call it in `_verifyAndStore` after the future-timestamp check, using `MAX_TIME_DRIFT` (or a dedicated `MAX_AGE` parameter) as the maximum acceptable age.

### Proof of Concept
1. Deploy `PythOracle` with `MAX_TIME_DRIFT = 300`.
2. Mock `pythLazer.verifyUpdate` to return a valid payload whose `FeedUpdateTimestamp` is `(block.timestamp - 86400) * 1000` ms (24 h ago).
3. Call `PythOracle.fallback()` with this payload.
4. Assert `oracleData[feedId].timestampMs` equals the 24-hour-old value (stored successfully).
5. Register a pool for `feedId`, enter a swap, call `price(feedId, pool)`, assert `refTime` is 24 h stale and `mid != 0`.

The `revertIfAfterBlockTimeWithDrift` check passes because `(block.timestamp - 86400) <= block.timestamp + 300`. The `isAfter` check passes because the stored timestamp is 0. The stale price is returned to the pool.
