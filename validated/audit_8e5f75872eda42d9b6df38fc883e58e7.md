### Title
`allowPushers` Consent Signature Replay Nullifies Pusher Self-Revocation, Locking Feed Writes Into Creator Namespace — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracle.allowPushers` authorizes pusher delegation via an EIP-191 signature whose domain contains `(chainid, address(this), deadline, pusher, creator)` but **no nonce and no used-signature bitmap**. After a pusher calls `revokePusher()`, the creator can immediately replay the original consent signature — while the deadline is still live — to silently re-establish the delegation. The pusher's self-revocation is nullified in the same block.

### Finding Description

The signed hash in `allowPushers` is:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;   // overwrites the zero set by revokePusher
``` [1](#0-0) 

The code's own comment acknowledges the replay risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The deadline prevents replay **after** it expires, but not **before**. `revokePusher()` only zeroes `namespaceRemapping[pusher]`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

Because there is no per-pusher revocation counter, no invalidation flag, and no used-signature set, the creator can call `allowPushers` again with the **identical** `(deadline, pusher, signature)` tuple and restore `namespaceRemapping[pusher] = creator`. The corrupted state value is `namespaceRemapping[pusher]`, which should be `address(0)` after revocation but is written back to `creator`.

### Impact Explanation

Every subsequent fallback push from the pusher's address resolves its namespace through `namespaceRemapping`:

```solidity
// fallback push namespace resolution
address effectiveCreator = namespaceRemapping[msg.sender];
if (effectiveCreator == address(0)) effectiveCreator = msg.sender;
``` [4](#0-3) 

With the delegation forcibly re-established, the pusher's price words continue to overwrite the creator's feed slots — the exact slots that `AnchoredPriceProvider._readLeg` and `PriceProvider._getBidAndAskPrice` consume to produce bid/ask quotes for live pool swaps. A pusher who revoked because they detected price manipulation or key compromise cannot stop their data from reaching the pool's oracle read path until the deadline expires (up to 7 days, the `MAX_REF_STALENESS` ceiling). [5](#0-4) 

This is an **admin-boundary break**: the pusher's self-revocation right is a documented security invariant (`revokePusher` exists precisely to let a pusher exit), and the creator — a semi-trusted party — can bypass it without any privileged role.

### Likelihood Explanation

- **Trigger**: any creator who stored the pusher's original consent bytes (trivially available from the on-chain calldata of the first `allowPushers` call) can replay them in a single transaction immediately after `revokePusher` is mined.
- **No special access required**: the creator is the `msg.sender` in `allowPushers`; no admin key or factory role is needed.
- **Deadline window**: consent signatures are typically issued with multi-day or multi-week deadlines to avoid operational friction, giving the creator a large replay window.

### Recommendation

Track consumed signatures with a per-pusher revocation nonce or a `mapping(bytes32 => bool) usedConsents` bitmap. Increment or set the flag inside `revokePusher` / `removePushers` so that any previously issued signature for that `(pusher, creator)` pair is permanently invalidated:

```solidity
mapping(address => uint256) public pusherNonce; // incremented on every revocation

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline,
                         pusherNonce[pusher], pusher, msg.sender))
);

// In revokePusher / removePushers:
pusherNonce[pusher]++;
namespaceRemapping[pusher] = address(0);
```

### Proof of Concept

```
T=0  Pusher signs consent: hash(chainid, oracle, deadline=T+7days, pusher, creator)
T=0  Creator calls allowPushers(deadline, [pusher], [sig])
       → namespaceRemapping[pusher] = creator  ✓

T=1  Pusher discovers misuse; calls revokePusher()
       → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)

T=1  Creator (same block or next) calls allowPushers(deadline, [pusher], [sig])
       with the IDENTICAL sig from T=0 — deadline still valid, no nonce check
       → namespaceRemapping[pusher] = creator  ← RESTORED without pusher consent

T=2…  Every push from pusher's address still lands in creator's feed slots,
       which AnchoredPriceProvider reads to produce bid/ask for pool swaps.
       Pusher's only escape: stop pushing entirely (making the feed go stale).
``` [6](#0-5) [7](#0-6)

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
