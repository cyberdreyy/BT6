### Title
Unbounded `deadline` in `allowPushers` lets a creator permanently override a pusher's self-revocation, enabling stale-price DoS on dependent pools — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts an arbitrary `deadline` with no maximum upper bound. The protocol's own NatSpec acknowledges that "an undated signature could re-establish a delegation AFTER the pusher revoked it," yet `_ensureDeadline` only checks `block.timestamp <= deadline` — no cap is enforced. A creator who obtains a pusher's EIP-191 consent signed with `deadline = type(uint256).max` can replay that single signature indefinitely to re-establish delegation after every `revokePusher()` call, making the pusher's self-revocation permanently ineffective and, if the pusher stops pushing to escape the hijacked namespace, starving any dependent pool of price updates.

---

### Finding Description

`_ensureDeadline` in the compressed oracle's `OracleBase`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
```

Only a lower bound is checked. No maximum is enforced.

`allowPushers` in `CompressedOracleV1`:

```solidity
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;
    emit PusherAuthorized(pusher, msg.sender);
}
```

The `deadline` is embedded in the signed digest. A pusher who signs with `deadline = type(uint256).max` (or any far-future value) produces a signature that is valid forever. The creator stores this signature off-chain and can replay it at any time.

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

But immediately after the pusher calls `revokePusher()`, the creator calls `allowPushers` again with the same signature — the deadline has not expired, the signature is still valid, and `namespaceRemapping[pusher]` is set back to `msg.sender`. The pusher's revocation is undone in the same block.

The protocol's own comment on line 189–191 explicitly identifies this risk:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

Yet no maximum deadline is enforced, making `type(uint256).max` a valid, effectively undated deadline.

---

### Impact Explanation

The `CompressedOracleV1` is the price source for pools via `PriceProvider` / `AnchoredPriceProvider`. A pusher who is the sole price updater for a creator's namespace feeds the pool's bid/ask through:

```
pool.swap → provider.getBidAndAskPrice → oracle.price(feedId, pool) → getOracleData(feedId)
```

If the pusher cannot revoke their delegation, they face two choices:
1. **Keep pushing** — their price updates continue to land in the creator's namespace, which the creator controls (the creator can also push conflicting data into the same slot).
2. **Stop pushing** — the feed's `timestampMs` stops advancing; the provider's staleness check (`MAX_TIME_DELTA`) triggers, returning `(0, type(uint128).max)`, causing `getBidAndAskPrice` to revert with `FeedStalled`.

A `FeedStalled` revert propagates through every `pool.swap` call, making the pool completely unusable for swaps. LP positions are locked in an illiquid pool until a new price source is configured — a process that requires factory-level intervention.

---

### Likelihood Explanation

- The pusher must have signed a consent with a far-future deadline. This is a realistic mistake: off-chain tooling that generates "permanent" delegation keys (e.g., for automated market-making bots) would naturally use `type(uint256).max` as a convenience.
- The creator must be adversarial. In the registrationless model, any address is a creator; there is no vetting.
- No privileged role is required. The entire attack is executable by two unprivileged EOAs.
- The protocol's own comment acknowledges the exact replay scenario, confirming it is a known-but-unmitigated risk.

---

### Recommendation

Enforce a maximum deadline window in `_ensureDeadline` (or directly in `allowPushers`):

```solidity
uint256 public constant MAX_DEADLINE_WINDOW = 30 days;

function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
    require(deadline <= block.timestamp + MAX_DEADLINE_WINDOW, DeadlineTooFar());
}
```

This ensures that any signed consent expires within a bounded window, so a pusher who revokes today is guaranteed that the old signature cannot be replayed after `MAX_DEADLINE_WINDOW` seconds.

### Proof of Concept

```
1. Pusher signs: keccak256(abi.encode(chainid, oracle, type(uint256).max, pusher, creator))
   → produces sig_forever

2. Creator calls: allowPushers(type(uint256).max, [pusher], [sig_forever])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher calls: revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (revocation succeeds)

4. Creator calls: allowPushers(type(uint256).max, [pusher], [sig_forever])
   → _ensureDeadline passes (type(uint256).max >= block.timestamp)
   → ECDSA.recover returns pusher  ✓
   → namespaceRemapping[pusher] = creator  ✓ (revocation undone)

5. Steps 3–4 repeat indefinitely; pusher cannot permanently escape.

6. Pusher stops pushing to avoid writing into creator's namespace.

7. pool.swap → provider.getBidAndAskPrice → oracle.price(feedId, pool)
   → timestampMs stale → _isStale returns true → returns (0, type(uint128).max)
   → getBidAndAskPrice reverts FeedStalled
   → all swaps on the pool revert; pool is unusable.
