### Title
`revokePusher()` Self-Revocation Is Silently Undone by Replaying the Original `allowPushers` Consent Signature Within Its Deadline Window — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` verifies a pusher's EIP-191 consent signature but tracks no nonce and marks no signature as used. The only replay guard is the deadline. A creator who holds a valid, unexpired consent signature can call `allowPushers` again with the identical signature immediately after the pusher calls `revokePusher()`, atomically re-establishing the delegation. The pusher's self-revocation is therefore ineffective for the entire lifetime of the original deadline.

---

### Finding Description

`allowPushers` in `CompressedOracleV1` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The domain covers chain id, contract address, deadline, pusher, and creator. It does **not** include a nonce, a revocation counter, or any per-use marker. No `usedSignatures` mapping or equivalent is written after a successful call.

`_ensureDeadline` only checks `block.timestamp <= deadline`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) [3](#0-2) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [4](#0-3) 

Because the signature is not consumed on first use, the creator can immediately call `allowPushers` again with the same `(deadline, pusher, sig)` tuple, writing `namespaceRemapping[pusher] = creator` again. The revocation is undone in the same block.

The code comment itself acknowledges the replay concern but misidentifies the deadline as the complete fix:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [5](#0-4) 

The deadline prevents replay *after* it expires, but does nothing to prevent replay *within* the deadline window — which is exactly when `revokePusher()` is called.

---

### Impact Explanation

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [6](#0-5) 

Every slot word the pusher sends after the replay lands in the creator's namespace, not the pusher's own. If the creator's namespace feeds a pool via `AnchoredPriceProvider`, the pool's `getBidAndAskPrice` call consumes those slot values as the live oracle price. A pusher whose key is compromised and who calls `revokePusher()` to stop the leak cannot stop it: the creator (or an attacker who observed the original `allowPushers` calldata on-chain) replays the signature and the compromised key's pushes continue to reach the pool. This is a bad-price execution path: the pool executes swaps against prices produced by a key the legitimate pusher explicitly tried to revoke.

---

### Likelihood Explanation

The original `allowPushers` calldata — including the full signature — is permanently visible on-chain. Any party who can read chain history (including the creator themselves) can replay it. No special privilege is required beyond being the creator (`msg.sender` in the original call). The window is the full deadline duration, which the code's own test suite sets to `block.timestamp + 1 days`. [7](#0-6) 

---

### Recommendation

Mark each consent signature as used after the first successful `allowPushers` call. The minimal fix is a `mapping(bytes32 => bool) private _usedConsentSigs` keyed on the digest, set to `true` before writing `namespaceRemapping`, and checked at the top of the loop:

```solidity
bytes32 hash = ...; // existing digest
require(!_usedConsentSigs[hash], SignatureAlreadyUsed());
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedConsentSigs[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

This ensures each pusher consent is single-use, making `revokePusher()` immediately and permanently effective.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

// Foundry test demonstrating that revokePusher() is undone by replaying
// the original allowPushers signature within its deadline.

function testReplayAllowPushersAfterRevoke() public {
    uint256 deadline = block.timestamp + 1 days;

    // 1. Pusher signs consent
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher self-revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

    // 4. Creator replays the SAME signature — no revert, delegation restored
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // <-- replay succeeds
    assertEq(oracle.namespaceRemapping(pusher), creator); // delegation is back

    // 5. Pusher's next push lands in creator's namespace, not pusher's own
    uint56 tsMs = uint56(block.timestamp * 1000);
    uint48 raw = _packRaw(9_000_000, 4, 2); // attacker-controlled price
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
    assertTrue(ok);

    // Creator's namespace (and any pool reading it) now holds the bad price
    IOffchainOracle.OracleData memory data =
        oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
    assertEq(data.price, U64x32.decode(uint32(raw >> 16)));
}
```
