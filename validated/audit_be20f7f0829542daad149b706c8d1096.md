### Title
Creator Can Replay Unexpired Pusher Consent Signature to Silently Re-establish Revoked Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` binds a pusher's consent to a `deadline` but contains no used-signature or post-revocation guard. After a pusher calls `revokePusher()`, the creator can immediately replay the original (still-unexpired) EIP-191 consent signature to re-establish the delegation. The pusher's self-revocation is therefore ineffective for the entire remaining lifetime of the deadline, and any subsequent pushes the pusher makes for their own purposes are silently routed into the creator's namespace, potentially feeding bad prices into pools that consume that oracle.

---

### Finding Description

`allowPushers` enforces two guards: (1) `_ensureDeadline(deadline)` — the transaction must arrive before the deadline — and (2) ECDSA recovery of the pusher's consent signature over `(chainid, oracleAddress, deadline, pusher, creator)`. [1](#0-0) 

Neither guard tracks whether the signature has already been consumed, nor whether the pusher has since called `revokePusher()`. `revokePusher()` only zeroes `namespaceRemapping[msg.sender]`: [2](#0-1) 

Because the consent signature is not invalidated on revocation, the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` tuple at any point before the deadline expires, restoring `namespaceRemapping[pusher] = creator`. The code's own comment acknowledges the replay risk but only addresses the post-deadline case: [3](#0-2) 

The comment states the deadline prevents re-establishing delegation "after the pusher revoked it," but this is only true once the deadline has expired. For the entire window `[revocation_time, deadline]` the creator can replay the signature at will.

---

### Impact Explanation

Once re-delegated against their will, every `fallback()` push the pusher makes — even pushes intended for their own namespace — is routed into the creator's namespace: [4](#0-3) 

The creator's oracle slots receive price data the pusher is now generating for a different purpose (different asset pair, different scale, or deliberately stale). Any `PriceProvider` or `AnchoredPriceProvider` that reads those slots will surface the corrupted mid/spread to the pool's `getBidAndAsk()` call, causing bad-price execution: traders receive quotes derived from wrong or stale data, and the pool's swap conservation invariant is broken.

The slot-structure documentation confirms that a never-pushed slot reads as `timestamp = 0` and is rejected as stale, but a re-delegated pusher's active pushes carry a fresh timestamp and pass the monotonicity check, so the bad data is accepted: [5](#0-4) 

---

### Likelihood Explanation

- Any creator who holds an unexpired pusher consent signature (a normal operational artifact) can trigger this with a single on-chain call.
- Pushers routinely sign consents with long deadlines (days to months) to avoid frequent re-authorization; the replay window is therefore large.
- No privileged role or special setup is required beyond the creator having the original signature, which they necessarily possess.

---

### Recommendation

Track consumed signatures or record the revocation epoch. The simplest fix is a per-`(pusher, creator)` nonce or a `revokedAt` timestamp stored on `revokePusher()`, and require that the consent signature was produced after the last revocation:

```solidity
mapping(address => uint256) public pusherRevokedAt; // pusher → block.timestamp of last revoke

function revokePusher() external {
    ...
    pusherRevokedAt[msg.sender] = block.timestamp;
    ...
}

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    // Require the deadline was set AFTER the pusher's last revocation
    require(deadline > pusherRevokedAt[pusher], "signature predates revocation");
    ...
}
```

Alternatively, include a per-use nonce in the signed message and increment it on each successful `allowPushers` call, making each consent single-use.

---

### Proof of Concept

```
1. block.timestamp = T
2. Pusher signs consent: sig = sign(chainid, oracle, deadline=T+30days, pusher, creator)
3. Creator calls allowPushers(T+30days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓
4. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are free)
5. Creator calls allowPushers(T+30days, [pusher], [sig])  ← SAME signature, deadline still valid
   → _ensureDeadline passes (T < T+30days)
   → ECDSA.recover returns pusher  (signature is still valid)
   → namespaceRemapping[pusher] = creator  ← revocation silently undone
6. Pusher, now operating their own service, calls fallback() with new price data
   → push lands in creator's namespace, not pusher's own
   → creator's oracle slots receive wrong prices
   → AnchoredPriceProvider / pool getBidAndAsk() returns corrupted bid/ask
   → swap executes at bad price; pool conservation invariant broken
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
