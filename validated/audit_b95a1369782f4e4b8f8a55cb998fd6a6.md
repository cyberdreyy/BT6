### Title
`allowPushers` consent signature is replayable within its deadline window, permanently overriding a pusher's self-revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

After a pusher calls `revokePusher()` to withdraw consent, the creator can replay the original EIP-191 consent signature (still valid before its `deadline`) to forcibly re-establish `namespaceRemapping[pusher] = creator`. The pusher's explicit revocation is silently overridden, and their subsequent price pushes continue to land in the creator's namespace rather than their own, feeding potentially stale or compromised prices into any pool that reads from the creator's feeds.

---

### Finding Description

`allowPushers` validates a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The signed message commits to `(block.chainid, address(this), deadline, pusher, msg.sender)` but contains **no nonce and no consumed-flag**: [2](#0-1) 

`revokePusher()` clears the mapping to `address(0)`: [3](#0-2) 

After revocation, the original signature is still cryptographically valid — `_ensureDeadline` only checks `block.timestamp <= deadline`: [4](#0-3) 

The creator calls `allowPushers` a second time with the identical `(deadline, pushers, signatures)` arguments. All checks pass and `namespaceRemapping[pusher]` is written back to `creator`, overriding the revocation.

The code comment at lines 189–191 explicitly acknowledges this concern — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — and claims the deadline is the fix: [5](#0-4) 

But the deadline only **limits the replay window**; it does not prevent replay **within** the window. The invariant the comment promises — that revocation is final — is broken for the entire duration `[revocation_tx, deadline]`.

---

### Impact Explanation

Once the delegation is re-established, the pusher's `fallback()` price pushes are routed back into the creator's namespace: [6](#0-5) 

Any pool whose `PriceProvider` reads from the creator's `feedId` will consume those prices. If the pusher revoked because their key was compromised, the creator can force the compromised pusher to keep feeding prices into the creator's namespace until the deadline expires, causing **bad-price execution** in every pool anchored to those feeds. The pusher's own namespace simultaneously goes stale (their pushes no longer land there), breaking any other consumer of the pusher's own feeds.

---

### Likelihood Explanation

The trigger requires the creator to act against the pusher's explicit revocation. Creators are semi-trusted (they own their namespace) but are not fully trusted — the protocol's own comment treats post-revocation replay as a threat to prevent. The deadline window can be hours or days (the test suite uses `block.timestamp + 1 days`): [7](#0-6) 

Within that window the replay is a single permissionless transaction by the creator with no additional cost.

---

### Recommendation

Maintain a per-`(pusher, creator)` nonce that is included in the signed message and incremented on every successful `allowPushers` call. Any prior signature becomes invalid the moment the nonce advances — whether by a new delegation or by a revocation that bumps the nonce.

```solidity
mapping(address => mapping(address => uint256)) public pusherNonces;

// inside allowPushers, replace the hash with:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonces[pusher][msg.sender]   // <-- nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonces[pusher][msg.sender]++;        // consume the signature
namespaceRemapping[pusher] = msg.sender;
```

`revokePusher` and `removePushers` should also increment the nonce so that any outstanding signed consent is invalidated at the moment of revocation, not merely at deadline expiry.

### Proof of Concept

```solidity
// Setup: creator establishes delegation with a 1-day deadline
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// Pusher self-revokes (e.g. key compromise detected)
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// Creator replays the IDENTICAL signature — still before deadline
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
// BUG: revocation is overridden
assertEq(oracle.namespaceRemapping(pusher), creator);

// Pusher's next price push lands in creator's namespace, not their own
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(9_999_999, 4, 2); // attacker-controlled price
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);
// Creator's feed (read by pools) now contains the compromised price
assertEq(
    oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
    U64x32.decode(uint32(raw >> 16))
);
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
