### Title
Pusher Self-Revocation Can Be Silently Undone by Creator Replaying the Original Consent Signature — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`, but `allowPushers` contains no guard against re-establishing a delegation using the same signed consent after revocation. A creator who holds a pusher's signed consent with a non-expired deadline can call `allowPushers` a second time with the identical signature, silently re-mapping the pusher back into the creator's namespace. The pusher's self-revocation is therefore not final, and any automated pusher bot that continues operating after calling `revokePusher()` will unknowingly write price data into the creator's namespace rather than its own.

---

### Finding Description

`allowPushers` verifies the pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The signed message commits to `(chainid, oracle, deadline, pusher, creator)`. It does **not** commit to a nonce or a "used" flag, and `allowPushers` does not check the current value of `namespaceRemapping[pusher]` before overwriting it: [2](#0-1) 

`revokePusher` clears the mapping: [3](#0-2) 

Because the signature is not invalidated on revocation, the creator can immediately call `allowPushers` again with the same `deadline` and `signature` bytes. The ECDSA check passes (the signature is still mathematically valid), and `namespaceRemapping[pusher]` is restored to the creator's address. The pusher's revocation is silently undone.

The `fallback` push path resolves the namespace at call time: [4](#0-3) 

So any push the bot makes after believing it has revoked lands in the creator's namespace, not its own.

---

### Impact Explanation

An automated pusher bot that calls `revokePusher()` and then continues pushing (into what it believes is its own namespace) will instead write into the creator's namespace if the creator replays the consent. The creator's oracle feeds — which are consumed by `AnchoredPriceProvider` and ultimately by pool swaps — receive price data that the pusher no longer endorses. If the pusher's bot is in a degraded or transitional state at the time of revocation (e.g., pushing stale, incorrect, or zero prices while shutting down), those values land in the creator's live feed and propagate to pool bid/ask quotes. This satisfies the "bad-price execution" impact gate: an unclamped or stale quote reaches a pool swap because the oracle namespace attribution is wrong.

---

### Likelihood Explanation

The conditions are:
1. A pusher signs a consent with a deadline far in the future (common for long-running integrations).
2. The pusher calls `revokePusher()` to stop providing data.
3. The creator replays the original signature before the deadline expires.
4. The pusher's bot continues operating (pushing to what it believes is its own namespace).

Steps 1–3 require no special privilege — `allowPushers` is a public function and the creator already holds the signature. Step 4 is the normal behavior of any automated pusher that does not monitor `namespaceRemapping` after revocation. The protocol's own comment acknowledges the deadline is the only replay barrier: [5](#0-4) 

---

### Recommendation

Invalidate the consent on revocation by storing a per-pusher revocation nonce or a "revoked-at" timestamp, and include that value in the signed message. Alternatively, treat `revokePusher()` as burning the consent by recording the revoked `(pusher, creator, deadline)` triple in a mapping and rejecting any `allowPushers` call that matches a revoked triple:

```solidity
mapping(bytes32 => bool) public revokedConsents;

function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    ...
    // burn the consent so the creator cannot replay it
    bytes32 consentKey = keccak256(abi.encode(msg.sender, creator));
    revokedConsents[consentKey] = true;
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

And in `allowPushers`, reject if the consent has been revoked:

```solidity
bytes32 consentKey = keccak256(abi.encode(pusher, msg.sender));
require(!revokedConsents[consentKey], "consent revoked");
```

### Proof of Concept

```solidity
// 1. Pusher signs consent for Creator with deadline = now + 30 days
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // cleared

// 4. Creator replays the SAME signature — no revert, delegation restored
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // silently re-established

// 5. Pusher bot pushes (thinking it goes to its own namespace)
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(1, 0xff, 0xff); // sentinel / bad price
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);

// Bad price lands in CREATOR's namespace, not pusher's own
IOffchainOracle.OracleData memory creatorData =
    oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
// creatorData.price is now the bad value pushed by the "revoked" pusher
IOffchainOracle.OracleData memory pusherData =
    oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0));
assertEq(pusherData.price, 0); // pusher's own namespace is empty
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
