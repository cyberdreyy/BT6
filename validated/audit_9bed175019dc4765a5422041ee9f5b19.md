### Title
Signature Replay in `allowPushers` Lets Creator Re-Establish a Revoked Pusher Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` uses a deadline-bounded EIP-191 signature but **no nonce**. After a pusher self-revokes via `revokePusher()`, the creator can immediately replay the original, still-valid signature to re-establish the delegation. The pusher's revocation is nullified for the entire remaining lifetime of the deadline.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The digest commits to `chainid`, oracle address, `deadline`, `pusher`, and `msg.sender` (the creator). There is **no nonce or consumed-flag**. Once the pusher signs and the creator calls `allowPushers`, the signature remains valid for any future call to `allowPushers` with the same arguments until `block.timestamp > deadline`.

`revokePusher` simply zeroes the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

Because `allowPushers` has no guard against re-use of a previously consumed signature, the creator can call it again with the identical `(deadline, [pusher], [signature])` tuple immediately after the pusher revokes, restoring `namespaceRemapping[pusher] = creator` without any new consent from the pusher.

The code's own comment acknowledges the replay concern but only addresses the post-deadline case:

> *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it. The deadline is likewise required."* [3](#0-2) 

The deadline prevents replay **after** expiry, but not **before** it — the window between signing and deadline expiry is entirely unprotected.

---

### Impact Explanation

The `fallback` push path routes all of a pusher's slot words into `namespaceRemapping[msg.sender]`'s namespace:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

While delegated, **every push** the pusher makes lands in the creator's namespace, not the pusher's own. A pusher who also maintains their own feeds (used by their own pools) cannot push to their own namespace at all while the delegation is active. A malicious creator who replays the signature after revocation forces the pusher to choose between:

1. Stopping all pushes (starving their own pools of price updates, causing stale-price reverts), or  
2. Continuing to push (feeding the creator's namespace indefinitely against their will).

If the pusher's feeds are consumed by a `PriceProvider` / pool, stale prices cause swap reverts or bad-price execution — a broken core pool functionality impact.

---

### Likelihood Explanation

- The creator already holds the pusher's signed bytes from the initial `allowPushers` call; no brute force or off-chain work is needed.
- The creator simply calls `allowPushers` again with the same calldata after `revokePusher` is mined.
- Any deadline longer than a few blocks (typical practice: hours to days) gives the creator an ample replay window.
- The pusher has no on-chain mechanism to invalidate the signature short of waiting for the deadline.

Likelihood: **Medium** (requires a malicious or adversarial creator, but the exploit is trivially cheap once the signature is in hand).

---

### Recommendation

Introduce a per-pusher-per-creator consumed flag or an incrementing nonce that is burned on first use:

```solidity
mapping(address pusher => mapping(address creator => uint256)) public delegationNonce;

// inside allowPushers, after recovering pusher:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        delegationNonce[pusher][msg.sender]   // <-- add nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
delegationNonce[pusher][msg.sender]++;        // <-- consume
```

Alternatively, record a `bytes32` signature hash in a `usedSignatures` set and revert on re-use. Either approach ensures that a revoked delegation cannot be silently reinstated with a stale signature.

---

### Proof of Concept

```solidity
// 1. Pusher signs delegation for creator with deadline = block.timestamp + 7 days
bytes memory sig = _signDelegation(PUSHER_KEY, deadline, pusherAddr, creatorAddr);

// 2. Creator establishes delegation
vm.prank(creatorAddr);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusherAddr), creatorAddr);

// 3. Pusher revokes
vm.prank(pusherAddr);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusherAddr), address(0));

// 4. Creator replays the SAME signature — no new consent from pusher
vm.prank(creatorAddr);
oracle.allowPushers(deadline, pushers, sigs);  // succeeds
assertEq(oracle.namespaceRemapping(pusherAddr), creatorAddr); // delegation restored

// 5. Pusher's subsequent pushes still land in creator's namespace
vm.prank(pusherAddr);
(bool ok,) = address(oracle).call(slotWord);
assertTrue(ok);
// creator's feed is updated; pusher's own feed is NOT
```
