### Title
`allowPushers` Consent Signature Replay After `revokePusher` Permanently Defeats Pusher Self-Revocation Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` does not consume or invalidate the pusher's EIP-191 consent signature after first use. Within the deadline window the creator can replay the identical signature to silently re-establish delegation immediately after the pusher has called `revokePusher()`. The pusher's self-revocation is therefore permanently ineffective for the entire lifetime of the signed deadline, and any namespace the pusher intended to serve independently stays empty/stale, feeding bad prices to pools that read from it.

---

### Finding Description

`allowPushers` verifies a pusher-signed EIP-191 message and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no consumed-signature set, and no per-signature invalidation. The same `(deadline, pusher, creator)` tuple produces an identical hash every time, so the signature is replayable on every call to `allowPushers` as long as `block.timestamp <= deadline`.

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

But the creator can immediately call `allowPushers` again with the original signature to restore `namespaceRemapping[pusher] = creator`. The code comment acknowledges the deadline is the only replay guard, but the deadline only prevents replay *after* expiry — within the window the signature is unconditionally reusable:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [3](#0-2) 

The comment claims the deadline solves the problem, but it does not: within the deadline window the creator can replay the signature an unlimited number of times.

**Attack sequence:**

1. Pusher P signs consent for creator C with `deadline = now + 365 days`.
2. C calls `allowPushers` → `namespaceRemapping[P] = C`.
3. P calls `revokePusher()` → `namespaceRemapping[P] = 0`. P begins pushing to its own namespace to serve independent pools.
4. C immediately calls `allowPushers` with the same signature → `namespaceRemapping[P] = C` again.
5. P's fallback pushes are silently redirected to C's namespace; P's own namespace (`feedIdOf(P, slotIndex, positionIndex)`) receives no data.
6. Any pool or `AnchoredPriceProvider` reading from P's own namespace receives `price = 0 / timestamp = 0`, which every consumer treats as stale — bad-price execution or a halted pool.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push P makes after "revoking" still lands in C's namespace as long as C keeps replaying the signature.

---

### Impact Explanation

Any pool configured to read from P's own namespace (`feedIdOf(P, …)`) receives a permanently stale feed (timestamp 0, price 0). The `AnchoredPriceProvider` and every downstream price provider reject timestamp-0 data as stale, halting swaps or forcing the pool to use a zero/sentinel price. This is a direct bad-price execution path: the oracle feed that pools depend on is silently starved of updates while the creator diverts the pusher's data to a different namespace.

---

### Likelihood Explanation

Medium. The creator must hold the original consent signature (they received it when first calling `allowPushers`) and must act within the deadline window. Deadlines are typically set to days or weeks to avoid operational friction, giving the creator a large replay window. The pusher has no on-chain recourse other than waiting for the deadline to expire.

---

### Recommendation

Track consumed signatures with a `mapping(bytes32 => bool) usedSignatures` and mark each `(chainid, oracle, deadline, pusher, creator)` hash as used on first acceptance. Alternatively, include a per-pusher nonce in the signed payload so each consent can only be accepted once:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline,
                         pusher, msg.sender, pusherNonce[pusher]++))
);
```

This ensures that after `revokePusher()` the old signature is permanently invalid, and re-delegation requires a fresh signature over the new nonce.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with a 1-day deadline
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1); pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);        sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher self-revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — no new consent needed
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);            // succeeds
assertEq(oracle.namespaceRemapping(pusher), creator);    // delegation restored

// 5. Pusher's subsequent fallback pushes land in creator's namespace, not pusher's own
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(1_000_000, 5, 3);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);

// Pusher's own namespace is empty — stale feed for any pool reading from it
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0)).price, 0);
// Creator's namespace received the data instead
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price,
         U64x32.decode(uint32(raw >> 16)));
```
