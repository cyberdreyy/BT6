### Title
Pusher Signature Replay in `allowPushers` Lets Creator Re-Establish a Revoked Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` signs consent with `(chainid, oracle, deadline, pusher, creator)` but includes no nonce and tracks no consumed signatures. A creator who holds a valid (not-yet-expired) signature can call `allowPushers` repeatedly — including after the pusher has explicitly revoked via `revokePusher()` — to silently re-establish the delegation. The pusher has no on-chain way to permanently cancel consent until the deadline timestamp elapses.

### Finding Description

`allowPushers` verifies an EIP-191 signature whose domain is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

After a successful check the function unconditionally overwrites `namespaceRemapping[pusher]`:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [2](#0-1) 

`revokePusher` clears the mapping to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because no nonce is included in the signed payload and no "used-signature" set is maintained, the creator can immediately replay the original signature (provided `block.timestamp <= deadline`) to restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone. The code comment acknowledges the deadline is the only replay guard:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [4](#0-3) 

But the deadline is a future timestamp chosen by the creator, not the pusher. A creator who requested a signature with a far-future deadline (e.g., 30 days) retains replay capability for the entire window regardless of the pusher's intent.

### Impact Explanation

After the creator replays the signature:

1. Every subsequent `fallback()` push from the pusher resolves `namespaceRemapping[msg.sender]` to the creator's address and writes into the **creator's namespace**, not the pusher's own.
2. The pusher's own namespace receives no updates — any pool reading `feedIdOf(pusher, slotIndex, pos)` sees stale or zero data.
3. The creator's feeds receive data the pusher no longer intends to provide — potentially stale, wrong-asset, or adversarially crafted prices that flow through `getOracleData` → `price` → `AnchoredPriceProvider` / `ProtectedPriceProvider` → pool swap. [5](#0-4) 

The `price()` read path in `CompressedOracleV1` is permissionless (no pool registration gate), so any pool consuming the creator's feed immediately receives the replayed-delegation data. [6](#0-5) 

### Likelihood Explanation

- The trigger is **unprivileged**: any creator who holds a not-yet-expired pusher signature can replay it in a single transaction.
- The pusher has **no on-chain defense** between revocation and deadline expiry; calling `revokePusher` again after each creator replay is a losing race.
- Deadlines are set by the creator at delegation time; a pusher who signed with a 30-day deadline is exposed for up to 30 days after revoking.
- The `allowPushers` function is explicitly listed as a public, permissionless entry point in the audit scope. [7](#0-6) 

### Recommendation

Add a consumed-signature set. The simplest fix is to hash the signature bytes and mark it used on first consumption:

```solidity
mapping(bytes32 => bool) public usedSignatures;

// inside allowPushers loop:
bytes32 sigHash = keccak256(signatures[i]);
require(!usedSignatures[sigHash], SignatureAlreadyUsed());
usedSignatures[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful delegation, so every new consent requires a fresh signature.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with a 30-day deadline.
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = pusher.sign(
    keccak256(abi.encode(block.chainid, oracle, deadline, pusher, creator))
);

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// 3. Pusher revokes.
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// 4. Creator replays the SAME signature — no revert, delegation restored.
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator again ✗

// 5. Pusher's next fallback push lands in creator's namespace, not pusher's own.
vm.prank(pusher);
(bool ok,) = address(oracle).call(wordAt(slotId, pos, raw, tsMs));
assertTrue(ok);
assertEq(oracle.getOracleData(oracle.feedIdOf(creator, slotId, pos)).price, decoded);
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher,  slotId, pos)).price, 0); // pusher's own ns empty
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
