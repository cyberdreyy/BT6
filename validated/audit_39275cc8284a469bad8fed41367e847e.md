### Title
Permissionless `register` Clears ADMIN Blacklist, Allowing Blacklisted Pools to Bypass Oracle Read Protection — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary

The `register` function in the providers `OracleBase` is permissionless and unconditionally clears the ADMIN-set blacklist for any pool as a side effect of registration. Anyone can pay the default 1-wei registration fee to un-blacklist a pool the ADMIN has explicitly blocked, making the blacklist protection permanently bypassable.

### Finding Description

`OracleBase.register` (providers) is the permissionless paid-registration path that whitelists a pool for a given `feedId`:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);          // default: 1 wei
    require(pool != address(0));
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;                       // ← unconditional side-effect
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    ...
}
``` [1](#0-0) 

The `price` read path enforces two blacklist checks: one on `msg.sender` (the provider, via `notBlacklisted`) and one on the pool itself:

```solidity
require(!blacklisted[pool], Blacklisted(pool));
``` [2](#0-1) 

The ADMIN sets the pool-level blacklist via `setBlacklist`, which is restricted to `ADMIN_ROLE`:

```solidity
function setBlacklist(address account, bool value) external onlyRole(ADMIN_ROLE) {
``` [3](#0-2) 

Because `register` is permissionless and clears `blacklisted[pool]` as a side effect, any pool operator (or third party) can immediately undo an ADMIN blacklist by calling `register` with 1 wei and any approved factory that recognizes the pool. The ADMIN can re-blacklist, but the attacker can clear it again in the same block — the protection degenerates into a gas race the ADMIN cannot win.

The analog to TribeRedeemer is exact:

| TribeRedeemer | Metric OMM |
|---|---|
| Zero-balance check prevents early redemption | Blacklist check prevents unsafe pool from reading prices |
| Attacker sends tokens → balance non-zero → guard passes | Attacker calls `register` → `blacklisted[pool] = false` → guard passes |
| Users redeem at bad rates | Blacklisted pool reads live oracle prices and executes swaps |

### Impact Explanation

If ADMIN blacklists a pool because it is misbehaving (e.g., reading prices outside a legitimate swap, draining LP value via oracle manipulation, or exploiting a pool-level bug), the pool operator can immediately clear the blacklist by paying 1 wei. The pool then passes both the `notBlacklisted` provider check and the `require(!blacklisted[pool])` pool check inside `price`, receives a live oracle quote, and can execute swaps — directly exposing LP principal and protocol fees to the same exploit the blacklist was meant to stop.

### Likelihood Explanation

- The registration fee defaults to 1 wei, making the bypass essentially free.
- The only prerequisite is that an approved factory recognizes the pool — which is true for any legitimately deployed pool.
- The pool operator is the most motivated attacker and has all the information needed.
- ADMIN cannot win a re-blacklist race because `register` is a single transaction.

### Recommendation

Separate blacklist clearing from the registration flow. Remove the `if (blacklisted[pool]) { blacklisted[pool] = false; }` block from `register`. Only `setBlacklist` (ADMIN-gated) should ever clear a blacklist entry. If "paying rehabilitates" is an intended design goal, gate it on an explicit ADMIN approval step rather than a permissionless side-effect.

### Proof of Concept

1. ADMIN calls `setBlacklist(pool, true)` — pool is blocked from reading prices.
2. Attacker (or pool operator) calls `register(feedId, pool, approvedFactory)` with `msg.value = 1 wei`.
3. Inside `register`: `blacklisted[pool]` is `true`, so the branch executes `blacklisted[pool] = false`.
4. Pool calls its provider's `getBidAndAskPrice()` → provider calls `oracle.price(feedId, pool)`.
5. `notBlacklisted` passes (provider is not blacklisted), `require(!blacklisted[pool])` passes (just cleared), `registeredPool[feedId][pool]` is `true` (just set).
6. Oracle returns a live price; the pool executes swaps as if it were never blacklisted.
7. ADMIN re-blacklists; attacker calls `register` again for 1 wei — indefinitely repeatable.
