### Title
Slot-Level Shared Timestamp Allows Stale Per-Position Prices to Bypass Staleness Checks — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracleV1` packs four oracle positions into one 256-bit storage slot and stores a **single shared `uint56` timestamp** for all four positions. When a pusher updates any position in a slot, the entire slot — including the shared timestamp — is overwritten. Downstream consumers (e.g., `AnchoredPriceProvider`) receive this slot-level timestamp as `refTime` and use it for their staleness check. A pusher can therefore refresh the timestamp by updating one active position while supplying old (stale) price data for the other three positions, causing those stale prices to pass the staleness guard and reach pool swaps.

### Finding Description

The slot layout is:

```
bits 255…208 : oracle[0] (48 bits: 32-bit U64x32 price | s0 | s1)
bits 207…160 : oracle[1]
bits 159…112 : oracle[2]
bits 111… 64 : oracle[3]
bits  63…  8 : timestamp (uint56, unix milliseconds) ← SHARED across all 4 positions
bits   7…  0 : slotId (cleared before storing)
``` [1](#0-0) 

`_loadSlotLayout` reads the single timestamp from bits 63:8 and assigns it to `_layout.timestampMs`, which is then returned identically for whichever of the four positions is queried: [2](#0-1) 

`getOracleData` sets `data.timestampMs = _layout.timestampMs` for every position, and `_price` surfaces it as `refTime`: [3](#0-2) 

The documentation explicitly states that **the entire slot is overwritten on every update**, meaning a pusher who wants to update position 0 must also supply values for positions 1–3: [4](#0-3) 

The monotonicity guard only checks that the new timestamp is strictly greater than the stored one — it does not enforce that each position's price was actually refreshed: [5](#0-4) 

**Concrete attack path (non-malicious variant):**

1. A creator registers a slot with four positions: e.g., ETH/USD (pos 0), BTC/USD (pos 1), SOL/USD (pos 2), AVAX/USD (pos 3).
2. A pusher is authorized via `allowPushers` and regularly updates all four positions.
3. The SOL/USD data source becomes unavailable. The pusher continues updating positions 0, 1, and 3 with live prices.
4. On each push the pusher must supply all four lanes. They supply the last-known (now stale) price for position 2 alongside fresh prices for the other three.
5. The slot timestamp is set to `block.timestamp * 1000` (fresh). The monotonicity check passes.
6. A consumer calls `price(feedId_for_pos2, pool)`. It receives the stale SOL/USD price **and** the fresh slot timestamp as `refTime`.
7. The downstream staleness guard (`maxTimeDrift` check in `AnchoredPriceProvider`) compares `refTime` to `block.timestamp` — the check passes because the timestamp is fresh.
8. The stale SOL/USD price is used to compute the pool's bid/ask, causing bad-price execution.

**Malicious variant:** A semi-trusted pusher deliberately keeps a manipulated price in position 2 while continuously refreshing the slot timestamp via position 0 updates, permanently bypassing the staleness guard for position 2.

### Impact Explanation

Stale or manipulated prices bypass the `maxTimeDrift` staleness guard and reach pool swap math. This can cause:
- Swap conservation failure: a trader receives more output tokens than the oracle-anchored curve permits.
- Pool insolvency: the pool pays out more than it receives, eroding LP assets.

This is a direct loss of user principal / LP assets, satisfying the Critical/High impact gate.

### Likelihood Explanation

The non-malicious variant requires only that a pusher's data source for one position becomes unavailable while the others remain live — a realistic operational scenario. The malicious variant requires only a semi-trusted pusher (authorized by the creator), not an unprivileged actor. No special setup or privileged role beyond pusher authorization is needed.

### Recommendation

Replace the single slot-level timestamp with **per-position timestamps**. Each 48-bit lane has 16 bits currently unused (32-bit price + 8-bit s0 + 8-bit s1 = 48 bits, leaving 16 bits free per lane if the slot is restructured). Alternatively, store four independent 14-bit or 16-bit relative timestamps (delta from a base) alongside each lane, and have `getOracleData` return the per-position timestamp rather than the shared slot timestamp. Consumers must then validate freshness against the position-specific `refTime`.

uint56 tsMs_new = uint56(block.timestamp * 1000);           // now

    function _loadSlotLayout(bytes32 slotIndex) internal view returns (SlotLayout memory _layout) {
        uint256 slotValue;
        assembly ("memory-safe") {
            slotValue := sload(slotIndex)
        }

        _layout.timestampMs = toTimeMs(slotValue >> 8 & X56);

        _layout.oracle0 = _decodeCompressedOracleData(uint48((slotValue >> 208) & X48));
        _layout.oracle1 = _decodeCompressedOracleData(uint48((slotValue >> 160) & X48));
        _layout.oracle2 = _decodeCompressedOracleData(uint48((slotValue >> 112) & X48));
        _layout.oracle3 = _decodeCompressedOracleData(uint48((slotValue >> 64) & X48));
    }

## Slot Layout (Big-Endian)

```text
bits 255 … 208 : oracle[0] (48 bits)
bits 207 … 160 : oracle[1] (48 bits)
bits 159 … 112 : oracle[2] (48 bits)
bits 111 …  64 : oracle[3] (48 bits)
bits  63 …   8 : timestamp (uint56, unix milliseconds)
bits   7 …   0 : reserved (always 0 in storage)
```

Each oracle lane encodes:

- `p` (32 bits) — price using `U64x32` pseudo-float (`U64x32.decode`).
- `s0` (8 bits) — spread index in `Codebook256`.
- `s1` (8 bits) — spread index in `Codebook256`.

## Default State

A never-pushed slot reads as all zeros: `timestamp = 0`, so every consumer rejects it
as stale — no seeding is needed. A lane with `s0 = 0xFF, s1 = 0xFF` is the explicit
"stalled" marker (readers surface spread0/spread1 as `10_000`).
```

Updates overwrite the entire storage slot value.

- `fallback()` accepts one or more 32-byte slot words (no prefix; calldata length must
  be a non-zero multiple of 32).
- `updateBySignature(feedCreator, newSlotValue, signature)` accepts a slot word signed
  by the feed creator over `keccak256(abi.encode(chainid, oracleAddress, feedCreator,
  newSlotValue))`, and is callable by anyone.

There is **no deadline** on either push path: each word carries its own timestamp and
the per-slot monotonicity check neutralizes replay (a replayed word is "not newer" and
is skipped).
