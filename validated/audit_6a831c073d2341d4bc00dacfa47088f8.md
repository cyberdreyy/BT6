### Title
`allowPushers` Consent Signature Is Replayable Within the Deadline Window, Nullifying `revokePusher()` - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` accepts a pusher's EIP-191 consent signature and maps `namespaceRemapping[pusher] = msg.sender`. There is no nonce and no used-signature tracking. The same `(deadline, pusher, creator)` signature can be submitted by the creator an unlimited number of times as long as `block.timestamp <= deadline`. A creator can therefore replay the original `allowPushers` calldata immediately after a pusher calls `revokePusher()`, silently re-establishing the delegation the pusher just cancelled.

---

### Finding Description

The code's own NatSpec acknowledges the replay concern:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The deadline is intended to be the replay guard. However, the deadline only prevents replay **after** it expires. Within the deadline window — which callers are expected to set days or weeks in the future — the creator holds a fully valid, reusable credential.

`allowPushers` signature construction:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce field, no `usedSignatures` bitmap, and no state change that invalidates the signature after first use. The `_ensureDeadline` check in `OracleBase` only enforces `block.timestamp <= deadline`: [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But nothing prevents the creator from immediately calling `allowPushers` again with the identical `(deadline, pushers, signatures)` arguments extracted from the original on-chain transaction, restoring `namespaceRemapping[pusher] = creator`.

The existing test suite validates that a **third-party attacker** cannot replay a signature into a different namespace (the hash includes `msg.sender`), but it does not test the creator replaying after revocation: [4](#0-3) 

---

### Impact Explanation

The `fallback` push path resolves the effective namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

If a pusher's signing key is compromised, the pusher calls `revokePusher()` to stop the attacker's pushes from landing in the creator's namespace. The creator (or an automated keeper watching for `PusherRevoked` events) replays the original `allowPushers` calldata. The attacker's pushes resume landing in the creator's namespace. Those slots are the source of truth for `getOracleData` / `price`, which `AnchoredPriceProvider` and pool swaps consume. Manipulated mid-prices and spread indexes reach live pool swaps, causing bad-price execution: traders receive more than the oracle curve permits or the pool fails to receive owed input.

---

### Likelihood Explanation

- The original `allowPushers` transaction is permanently visible on-chain; the creator needs no off-chain secret to replay it.
- Automated keeper bots that maintain delegations (a common operational pattern) would replay on any `PusherRevoked` event without human review.
- The pusher has no on-chain mechanism to prevent the replay short of waiting for the deadline to expire.
- Likelihood is **medium**: requires either a malicious or an automated creator, plus a compromised pusher key.

---

### Recommendation

Mark each consent signature as consumed on first use. The minimal fix is a `usedSignatures` mapping keyed on the signature hash (or on `keccak256(abi.encode(deadline, pusher, creator))`):

```solidity
mapping(bytes32 => bool) private _usedConsents;

// inside allowPushers loop:
bytes32 consentKey = keccak256(abi.encode(deadline, pusher, msg.sender));
require(!_usedConsents[consentKey], ConsentAlreadyUsed());
_usedConsents[consentKey] = true;
```

Alternatively, include a per-pusher nonce in the signed message and increment it on each successful `allowPushers`, so a revoked pusher can refuse to sign a new nonce.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with a 30-day deadline.
uint256 deadline = block.timestamp + 30 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher self-revokes.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the IDENTICAL calldata — succeeds, no revert.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));

// 5. Delegation is restored; pusher's revocation is nullified.
assertEq(oracle.namespaceRemapping(pusher), creator); // ← passes
```

A compromised pusher key now continues to write into the creator's namespace via the `fallback` push path, and those prices propagate to pool swaps through `getOracleData` → `AnchoredPriceProvider` → `MetricOmmPool.swap`.
