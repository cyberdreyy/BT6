### Title
`allowPushers` Signature Replay Re-Establishes Revoked Pusher Delegation, Enabling Continued Bad-Price Injection into Pools — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` does not consume or invalidate the pusher's EIP-191 consent signature after first use. After a pusher calls `revokePusher()`, the creator can immediately replay the same still-valid signature to re-establish delegation within the deadline window. This is the direct analog of the ElectionCommission bug: a state transition (`revokePusher` → `namespaceRemapping = 0`) is not checked before the inverse transition (`allowPushers` → `namespaceRemapping = creator`) is re-applied, allowing the "revoked" state to be silently overwritten back to "delegated."

---

### Finding Description

`CompressedOracleV1.allowPushers` signs consent over `(chainid, oracle_address, deadline, pusher, creator)` and enforces only that the deadline has not expired:

```solidity
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← unconditional overwrite
    ...
}
``` [1](#0-0) 

The code's own NatSpec explicitly acknowledges the risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The comment claims the deadline solves this. It does not. The deadline only prevents using an **expired** signature. Within the deadline window the same bytes are accepted an unlimited number of times. There is no nonce, no used-signature bitmap, and no check that `namespaceRemapping[pusher] == address(0)` before overwriting.

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

But `allowPushers` immediately overwrites it back to `creator` with the same signature, restoring full write authority over the creator's namespace.

The `fallback` push path resolves the namespace from `namespaceRemapping[msg.sender]` at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the pusher makes after the creator re-establishes delegation lands in the creator's namespace, not the pusher's own.

---

### Impact Explanation

`CompressedOracleV1.price` is permissionless and ignores the `pool` argument — any pool backed by a creator's feed reads directly from that namespace's storage:

```solidity
function price(bytes32 feedId, address /* pool */)
    external view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    return _price(feedId);
}
``` [5](#0-4) 

If the creator re-establishes delegation after the pusher revoked, any data the pusher subsequently pushes — whether intended for their own namespace or injected by an attacker who holds the pusher's key — overwrites the creator's live feed slots. Pools consuming those feeds execute swaps against the corrupted mid/spread values, constituting bad-price execution.

---

### Likelihood Explanation

The scenario requires:
1. A pusher signs a consent with a future deadline (normal operation).
2. The pusher calls `revokePusher()` — e.g., because their key was compromised or they want to stop.
3. The creator calls `allowPushers` again with the same signature bytes (still valid until the deadline).
4. The pusher (or an attacker holding the pusher's key) continues pushing data that lands in the creator's namespace.

Step 3 is the only non-automatic step. A creator who does not know why the pusher revoked (e.g., they assume it was accidental) will naturally replay the same signature to restore service. The deadline window can be days long, giving ample time for this sequence.

---

### Recommendation

Track a per-pusher nonce or a used-signature set so each consent signature is accepted exactly once. Alternatively, check that `namespaceRemapping[pusher] == address(0)` before writing in `allowPushers` — if the pusher is already delegated, require a fresh signature; if the pusher previously revoked, require explicit re-consent with a new signature (new deadline or nonce). Either approach closes the replay window that the deadline alone cannot close.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline T+1 day.
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes (e.g., key compromise detected).
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — still within deadline.
//    allowPushers does NOT check current state; unconditionally overwrites.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored

// 5. Attacker (holding pusher key) pushes bad price into creator's namespace.
uint56 tsMs = uint56(block.timestamp * 1000 + 1);
uint48 badRaw = _packRaw(BAD_PRICE, 5, 5);
vm.prank(pusher); // attacker controls this key
(bool ok,) = address(oracle).call(_wordAt(slotId, pos, badRaw, tsMs));
assertTrue(ok);

// 6. Pool reads the corrupted price from creator's namespace.
(uint256 mid,,,) = oracle.price(oracle.feedIdOf(creator, slotId, pos), address(pool));
assertEq(mid, U64x32.decode(uint32(badRaw >> 16))); // bad price reaches pool
```
