### Title
`createPriceProvider` / `createAnchoredProvider` Use Plain `CREATE`, Enabling Reorg-Based Provider Hijack That Feeds Bad Prices to Pools — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`, `AnchoredProviderFactory.sol`)

---

### Summary

All three price-provider factories deploy their provider contracts with the plain `new` keyword (EVM `CREATE` opcode). The resulting address depends only on the factory's nonce. On any chain susceptible to block reorgs (Arbitrum, Polygon, Ethereum mainnet), an attacker can race to occupy the same nonce-derived address with a malicious provider, causing a pool that was legitimately configured to use that address to instead consume attacker-controlled prices.

---

### Finding Description

`PriceProviderFactory.createPriceProvider` deploys via:

```solidity
PriceProvider p = new PriceProvider(
    address(this), _oracle, _feedId, _marginStep, _maxTimeDelta, _baseToken, _quoteToken
);
``` [1](#0-0) 

`PriceProviderFactoryL2.createPriceProvider` is identical in structure: [2](#0-1) 

`AnchoredProviderFactory.createAnchoredProvider` does the same: [3](#0-2) 

None of these use `CREATE2` with a salt that includes `msg.sender`. The deployed address is therefore fully determined by `(factory_address, factory_nonce)`.

The pool factory's only provider validation is a token-pair check:

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1)
        revert PriceProviderTokenMismatch();
}
``` [4](#0-3) 

There is no `isProvider()` origin check, no factory-of-origin check, and no `CREATE2` salt binding. The pool deployer itself uses `CREATE2` with a user-supplied salt: [5](#0-4) 

— but the provider factories that feed prices into those pools do not.

---

### Impact Explanation

**Attack path (PriceProviderFactory — no envelope validation):**

1. Alice calls `createPriceProvider(legitimateOracle, legitimateFeedId, safeMarginStep, safeMaxTimeDelta, token0, token1)`. The factory is at nonce N; the provider lands at address X. Alice records X.
2. Alice submits `createPool(... priceProvider: X ...)`. Both transactions are in the mempool.
3. A reorg occurs. Bob observes the pending transactions and calls `createPriceProvider(maliciousOracle, maliciousFeedId, extremeMarginStep, 7 days, token0, token1)` at factory nonce N. Bob's provider lands at address X (same nonce → same address). `providerOwner[X] = Bob`.
4. Alice's `createPriceProvider` re-executes at nonce N+1, producing address Y. Alice's `createPool` still references address X (hardcoded in the submitted transaction).
5. Alice's pool is now bound to Bob's provider at X. Bob's malicious oracle returns arbitrary prices; `MAX_TIME_DELTA = 7 days` accepts maximally stale data; `marginStep` near `±BPS_BASE` produces extreme bid/ask spreads.

The pool's `getBidAndAskPrice()` call reaches Bob's provider: [6](#0-5) 

Bob can then execute swaps at prices far from market, draining LP principal. Because `PriceProviderFactory` has **no envelope validation**, Bob is free to choose any oracle address and any parameter values within the constructor's loose bounds (`marginStep ∈ (−1e18, 1e18)`, `maxTimeDelta ∈ (0, 7 days]`).

`AnchoredProviderFactory` is also vulnerable but the envelope constrains `minMargin`, `maxRefStaleness`, and `maxSpreadBps` to admin-approved ranges — the damage is bounded but not eliminated (attacker still controls which allow-listed oracle is used and can pick the loosest envelope parameters).

---

### Likelihood Explanation

- Polygon experienced a 157-block reorg in February 2023. Arbitrum and Optimism reorg on fraud proofs. Ethereum mainnet has had beacon-chain reorgs.
- The protocol explicitly targets L2 deployment (the `PriceProviderFactoryL2` variant exists for this purpose).
- The attack requires only that the attacker observe Alice's pending `createPriceProvider` transaction and submit a competing one — no privileged role, no special token, no malicious initial setup beyond choosing constructor parameters that pass the constructor's own checks.
- The two-step flow (create provider → create pool) is the natural integration pattern, making the window between the two transactions the attack surface.

---

### Recommendation

Deploy all three provider types via `CREATE2` with a salt that commits to `msg.sender` (and optionally a user-supplied nonce):

```solidity
bytes32 salt = keccak256(abi.encode(msg.sender, userNonce));
PriceProvider p = new PriceProvider{salt: salt}(...);
```

This makes the deployed address depend on the caller's identity, so an attacker cannot occupy the same address by replaying a different transaction at the same factory nonce.

**Malicious provider parameters that pass constructor checks in `PriceProvider`:**
- `_oracle`: any address implementing `IOffchainOracle` / `IPricedOracle` returning attacker-chosen prices
