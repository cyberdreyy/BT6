### Title
Consent-signature replay in `allowPushers` lets a creator permanently override a pusher's `revokePusher()` call, misdirecting oracle feeds and staling pool prices — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = creator`. The signed message is bound to `(chainid, oracle, deadline, pusher, creator)` but carries **no nonce and no one-time-use marker**. The only replay gate is `block.timestamp <= deadline`. A creator who holds the pusher's consent signature can replay it an unlimited number of times before the deadline expires, re-establishing a delegation the pusher explicitly cancelled via `revokePusher()`.

---

### Finding Description

`allowPushers` verifies the pusher's consent and unconditionally overwrites `namespaceRemapping`: [1](#0-0) 

The only replay protection is the deadline check in `_ensureDeadline`: [2](#0-1) 

This check prevents replay **after** the deadline expires, but not **before**. The code comment itself acknowledges the intent:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The comment is correct about the threat but wrong about the fix: the deadline only closes the window after expiry. Before expiry, the same `(deadline, pusher, creator)` tuple is valid on every call, so the creator can replay the identical signature immediately after `revokePusher()` clears the mapping: [3](#0-2) 

`revokePusher()` sets `namespaceRemapping[pusher] = address(0)`. The creator then calls `allowPushers(sameDeadline, [pusher], [sameSig])` and the mapping is restored to `creator`. This cycle can repeat indefinitely until the deadline timestamp passes.

The `fallback()` push path reads `namespaceRemapping[msg.sender]` at call time: [4](#0-3) 

So every push the pusher makes after a "revocation" that the creator has replayed still lands in the creator's namespace, not the pusher's own.

---

### Impact Explanation

**Direct oracle-feed misdirection:** If the pusher wants to push data for a second creator (Creator B) after revoking from Creator A, Creator A can keep replaying the old signature. Every push the pusher sends is written into Creator A's namespace. Creator B's feeds receive no updates and their timestamps remain at the last pre-revocation value.

**Stale price reaching pools:** `AnchoredPriceProvider._readLeg` checks staleness against `MAX_REF_STALENESS`: [5](#0-4) 

Once Creator B's feed timestamp exceeds `MAX_REF_STALENESS`, `_readLeg` returns `ok = false`, `_getBidAndAskPrice` returns `(0, type(uint128).max)`, and `getBidAndAskPrice` reverts with `FeedStalled`: [6](#0-5) 

Every swap on Creator B's pool halts. This is broken core pool functionality — swaps and liquidity flows become unusable for the duration of the original consent deadline (up to 7 days per `MAX_REF_STALENESS` bound, or longer if the deadline itself is far in the future).

---

### Likelihood Explanation

- The creator already holds the pusher's consent signature (they submitted it in the original `allowPushers` call).
- No new capability is required — replaying is a single public transaction with existing calldata.
- Pushers may sign long-lived consents (e.g., 30-day or 1-year deadlines) to avoid frequent re-authorisation; the longer the deadline, the longer the attack window.
- The pusher has no on-chain way to invalidate the signature before the deadline expires.

Likelihood: **Medium** (requires a malicious creator, but the mechanism is trivially executable with no additional privileges).

---

### Recommendation

Track a per-pusher revocation nonce and include it in the signed digest. When `revokePusher()` is called, increment the nonce. Any previously signed consent (which committed to the old nonce) becomes cryptographically invalid:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))

// In revokePusher:
pusherNonce[msg.sender]++;
namespaceRemapping[msg.sender] = address(0);
```

Alternatively, store a `revokedAt[pusher]` timestamp and reject any consent whose deadline was already valid at revocation time.

---

### Proof of Concept

```
T=0   Pusher signs: sig = sign(keccak256(abi.encode(chainid, oracle, deadline=T+30days, pusher, creatorA)))
T=1   creatorA calls allowPushers(T+30days, [pusher], [sig])
         → namespaceRemapping[pusher] = creatorA  ✓

T=2   Pusher calls revokePusher()
         → namespaceRemapping[pusher] = address(0)  ✓

T=3   creatorA calls allowPushers(T+30days, [pusher], [sig])  ← SAME sig, SAME deadline
         → namespaceRemapping[pusher] = creatorA  ← revocation undone

T=4   Pusher pushes data (intending to serve creatorB's pool)
         → fallback reads namespaceRemapping[pusher] == creatorA
         → data lands in creatorA's namespace, not creatorB's

T=5+  creatorB's feed timestamp goes stale → AnchoredPriceProvider._readLeg returns ok=false
         → getBidAndAskPrice reverts FeedStalled
         → all swaps on creatorB's pool revert
```

Steps T=2 and T=3 can be repeated by `creatorA` in every block until `block.timestamp > T+30days`.

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
```
