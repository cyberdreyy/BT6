### Title
Pusher revocation is bypassable via signature replay — creator can re-establish delegation without new consent (`CompressedOracle.sol`)

---

### Summary

`revokePusher()` clears `namespaceRemapping[pusher]` on-chain but does not invalidate the pusher's original EIP-191 signed consent. Because `allowPushers()` has no on-chain nonce or used-signature bitmap, a creator who holds the original signature can immediately replay it (while the deadline is still live) to re-establish the delegation the pusher just revoked — without any new consent from the pusher.

---

### Finding Description

`allowPushers()` accepts a pusher's EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)` and writes `namespaceRemapping[pusher] = creator`. [1](#0-0) 

The code comment explicitly acknowledges the replay risk:

> "the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The deadline is offered as the mitigation. But the deadline only prevents replay **after it expires**. If the deadline is still in the future when the pusher calls `revokePusher()`, the creator can immediately call `allowPushers()` again with the **same signature** to restore the mapping. [2](#0-1) 

`revokePusher()` only zeroes the mapping. There is no nonce increment, no used-signature set, and no other on-chain state that would cause the already-verified signature to be rejected on a second call to `allowPushers()`. [3](#0-2) 

Confirmed: no nonce or signature-invalidation storage exists anywhere in the production contract surface. [4](#0-3) 

---

### Impact Explanation

After the creator replays the signature, the pusher's `fallback()` push path reads `namespaceRemapping[msg.sender]` and routes every subsequent price update into the **creator's namespace** instead of the pusher's own. [5](#0-4) 

Consequences:

1. **Pusher's own namespace stays at price 0 / timestamp 0.** Any pool or `PriceProvider` reading `feedIdOf(pusher, slot, pos)` receives a zero-price, zero-timestamp value. Every consumer already treats timestamp 0 as stale, so the pool's oracle read fails or returns a bad price — matching the "stale, unbounded, or unclamped bid/ask reaches a pool swap" impact gate.
2. **The creator's namespace receives the pusher's live updates without the pusher's ongoing consent**, allowing the creator to keep a feed live that the pusher intended to abandon, or to attribute the pusher's price data to a pool the pusher did not intend to serve.
3. The creator can repeat the replay every time the pusher revokes, making revocation completely ineffective until the deadline expires — which can be months or years after the original consent was signed.

---

### Likelihood Explanation

- The pusher must have previously signed a consent with a future deadline (the normal operational flow).
- The creator must be adversarial and retain the original signature bytes (trivially available from the original `allowPushers` transaction calldata on-chain).
- No privileged role is required beyond being the creator who originally called `allowPushers`.
- The pusher has no on-chain mechanism to prevent this; their only recourse is to wait for the deadline to expire.

Likelihood: **Medium** (requires an adversarial creator, but the attack is fully permissionless once the creator decides to act).

---

### Recommendation

Add a per-pusher revocation nonce that is incremented by `revokePusher()` and embedded in the signed digest verified by `allowPushers()`:

```solidity
// storage
mapping(address => uint256) public pusherNonce;

// in allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);

// in revokePusher:
namespaceRemapping[msg.sender] = address(0);
pusherNonce[msg.sender]++;     // <-- invalidate all prior signatures
emit PusherRevoked(msg.sender, creator);
```

This mirrors the fix recommended for the Hats Protocol bug: the state that enables re-establishment must be destroyed at revocation time.

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 1-year deadline
uint256 deadline = block.timestamp + 365 days;
bytes memory sig = pusher.sign(
    keccak256(abi.encode(block.chainid, oracle, deadline, pusher, creator))
);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes — expects to push into own namespace from now on
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator immediately replays the SAME signature — no new consent needed
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds
assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation restored

// 5. Pusher's next push still lands in creator's namespace, not pusher's own
vm.prank(pusher);
(bool ok,) = address(oracle).call(slotWord);
assertTrue(ok);
// feedIdOf(creator, slot, pos).price == pusher's price  ← creator namespace
// feedIdOf(pusher,   slot, pos).price == 0              ← pusher namespace stale
```
