### Title
Pusher Revocation Bypass via Signature Replay — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` contains no used-signature tracking (no nonce, no per-pusher deadline watermark, no signature bitmap). `_ensureDeadline` only checks `block.timestamp <= deadline`. A creator who holds a pusher's consent signature with a far-future deadline can replay it an unlimited number of times, re-establishing `namespaceRemapping[pusher] = creator` immediately after every `revokePusher()` call, making revocation permanently ineffective.

---

### Finding Description

The `allowPushers` function verifies an EIP-191 signature over `(block.chainid, address(this), deadline, pusher, msg.sender)` and then unconditionally writes `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

The only replay guard is `_ensureDeadline`, which is a pure timestamp comparison with no state mutation: [2](#0-1) 

`revokePusher` clears the mapping to `address(0)`: [3](#0-2) 

Because no consumed-signature state is recorded anywhere, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, signature)` tuple. The signature still verifies (same hash inputs, deadline still in the future), and `namespaceRemapping[pusher]` is restored to `creator`. This cycle can repeat indefinitely.

The code comment on `allowPushers` explicitly acknowledges the deadline is the intended replay barrier:

> *"the deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The deadline alone is not sufficient — it only prevents using an *expired* signature, not replaying a still-valid one.

---

### Impact Explanation

- The pusher's `revokePusher()` call is rendered permanently ineffective as long as the creator holds a non-expired signature.
- Any subsequent fallback push by the pusher continues to land in the creator's namespace rather than the pusher's own namespace, because `namespaceRemapping[pusher]` is immediately restored.
- If the pusher attempts to migrate to a new creator by having the new creator call `allowPushers`, the old creator can overwrite that mapping back in the same block, creating an unresolvable race condition.
- The pusher's only escape is to stop pushing entirely, which may affect all their feeds and constitutes broken core oracle functionality.
- If the pusher's data is consumed by a live pool via an `AnchoredPriceProvider`, the creator retains unauthorized write authority over the oracle namespace that feeds pool pricing, which is a broken admin-boundary guarantee.

---

### Likelihood Explanation

- Any creator who stored the original `allowPushers` calldata (trivially available from chain history) can execute the replay with zero additional privileges.
- A pusher signing with `deadline = type(uint256).max` (a common pattern to avoid expiry management) makes the vulnerability permanent.
- The attack requires only a single on-chain transaction by the creator after each revocation.

---

### Recommendation

Record each consumed signature or track a per-pusher-creator nonce. The simplest fix is a `mapping(bytes32 => bool) private _usedConsents` keyed on the signature hash (or the pre-image hash), set to `true` on first use and checked at the top of `allowPushers`. Alternatively, track a monotonically increasing nonce per `(pusher, creator)` pair and include it in the signed message, so each consent can only be used once.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with far-future deadline
uint256 deadline = type(uint256).max;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — succeeds, no revert
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs); // ← replay

// 5. Revocation is bypassed
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored
```

The final assertion passes, confirming `revokePusher()` is ineffective against a creator holding a non-expired signature.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }

    function revokePusher() external {
        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
        namespaceRemapping[msg.sender] = address(0);
        emit PusherRevoked(msg.sender, creator);
    }
