### Title
`revokePusher()` is ineffective during the deadline window — creator can re-establish delegation using the same signature after pusher self-revokes - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

A pusher who calls `revokePusher()` to stop their pushes from landing in a creator's namespace can have that revocation immediately undone by the creator, who can replay the same valid (non-expired) EIP-191 signature through `allowPushers`. The code comment explicitly claims the deadline prevents this, but it does not: the deadline only blocks an *expired* signature, not a still-valid one reused after revocation.

### Finding Description

`allowPushers` requires the pusher's EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)` and checks `block.timestamp <= deadline`. The comment at line 186–191 states:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

This claim is incorrect. The deadline only prevents using a signature whose deadline has already passed. During the window `[now, deadline)`, the creator holds a fully valid, reusable signature. When the pusher calls `revokePusher()`, it sets `namespaceRemapping[pusher] = address(0)`. The creator can immediately call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments — the deadline check passes, the ECDSA recovery succeeds, and `namespaceRemapping[pusher] = creator` is written back. There is no nonce, no used-signature registry, and no on-chain record that the pusher revoked.

### Impact Explanation

After the creator re-establishes delegation, every subsequent fallback push from the pusher's address is routed into the creator's namespace instead of the pusher's own namespace. Concretely:

1. The pusher's own feeds (identified by `feedIdOf(pusher, slotIndex, positionIndex)`) receive no new data — their timestamps freeze.
2. Any `AnchoredPriceProvider` or pool that reads the pusher's own feeds will see a stale `refTime`, causing `_isStale` to return `true` and `getBidAndAskPrice` to revert with `FeedStalled`, halting swaps on those pools.
3. The creator's namespace continues to receive the pusher's data, keeping the creator's feeds live even though the pusher intended to sever the relationship.

The pusher's only escape is to stop pushing entirely and wait for the deadline to expire — which may be days or weeks away if a long-lived consent was signed.

### Likelihood Explanation

- A pusher signs consent with a deadline far in the future (common for operational convenience — e.g., a 7-day or 30-day window).
- The creator is malicious or the relationship sours after signing.
- The pusher calls `revokePusher()` expecting to redirect their pushes to their own namespace.
- The creator front-runs or immediately follows with `allowPushers(deadline, [pusher], [sig])` using the already-known signature (it was submitted on-chain when delegation was first established, so it is public).
- The entire attack requires only one additional transaction from the creator and zero new signatures.

### Recommendation

Track consumed delegation signatures with a per-pusher nonce or a `usedSignatures` mapping. Alternatively, include the current `namespaceRemapping[pusher]` value (or a monotonic revocation counter) in the signed payload so that a revocation increments the counter and invalidates all prior signatures. The `allowContractPushers` path is not affected because it uses a live `isPusher(creator)` call rather than a stored signature. [4](#0-3) 

### Proof of Concept

```
// Setup
uint256 deadline = block.timestamp + 7 days;
bytes memory sig = pusher.sign(keccak256(abi.encode(chainid, oracle, deadline, pusher, creator)));

// Step 1: creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator ✓

// Step 2: pusher revokes
vm.prank(pusher);
oracle.revokePusher();
// namespaceRemapping[pusher] == address(0) ✓

// Step 3: creator immediately re-establishes using the SAME sig — no new signature needed
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
// namespaceRemapping[pusher] == creator again ✓
// revokePusher() had zero lasting effect

// Step 4: pusher's next fallback push lands in creator's namespace, not pusher's own
vm.prank(pusher);
oracle.call(wordAt(slotId, pos, raw, tsMs));
// getOracleData(feedIdOf(creator, slotId, pos)).price == decoded price  ← creator's feed updated
// getOracleData(feedIdOf(pusher,   slotId, pos)).price == 0             ← pusher's own feed stays empty
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
