### Title
Signature Replay in `allowPushers()` Allows Forced Re-Delegation After Pusher Revokes — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers()` verifies an EIP-191 pusher-consent signature that commits to `(chainid, oracle, deadline, pusher, creator)` but includes **no nonce**. Any caller who observed the original delegation transaction can replay the same signature — before the deadline — to silently re-establish a delegation the pusher already revoked, or to hijack a pusher that has since been re-delegated to a different creator. The hijacked pusher's price updates land in the wrong namespace, making the legitimate creator's feeds permanently stale until the deadline expires.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The five fields bound in the digest are fixed for the lifetime of the deadline: `chainid` and `address(this)` are constants, `deadline` is chosen by the creator, `pusher` and `msg.sender` are the two parties. Nothing in the digest changes between calls. Consequently, the same `(deadline, pusher, sig)` triple is valid for every call to `allowPushers` until `block.timestamp > deadline`.

The code's own NatSpec acknowledges the deadline is needed to prevent re-delegation *after* revocation:

> "the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

But the deadline only prevents replay *after* it expires. It does **not** prevent replay *within* the deadline window, which is the exploitable gap.

`revokePusher` and `removePushers` both zero out `namespaceRemapping[pusher]`:

```solidity
function revokePusher() external {
    ...
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

After revocation, any party holding the original signature can call `allowPushers` again with the identical arguments and overwrite `namespaceRemapping[pusher]` back to the original creator — no new consent from the pusher is required.

The `fallback` push path reads `namespaceRemapping[msg.sender]` to decide which namespace receives the price update:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So a replayed delegation silently redirects all subsequent pushes from the pusher into the attacker-controlled namespace, while the legitimate namespace (or the pusher's own namespace) receives no updates and goes stale.

---

### Impact Explanation

A pool consuming a `CompressedOracleV1` feed via `PriceProvider` calls `getOracleData(feedId)`, which reads the slot for the feed's creator. If the pusher responsible for that creator's feeds has been silently re-delegated to a different creator's namespace, the legitimate creator's slot is never updated. `getOracleData` returns `timestampMs = 0` (the never-pushed sentinel), which every downstream consumer — including `AnchoredPriceProvider` and the pool's `getSellAndBuyPrices` — must reject as stale. The pool cannot execute swaps against that feed, or worse, if the staleness check is not tight enough, it executes against a frozen price. This is a **bad-price / stale-price execution** impact that directly affects LP assets and swap settlement.

---

### Likelihood Explanation

The replay requires only that the attacker observed the original `allowPushers` transaction (trivially available on-chain) and that the deadline has not yet expired. Deadlines are set by the creator and are typically far in the future to avoid operational friction. The attacker need not control any privileged key — only the original `(deadline, pusher[], sigs[])` calldata, which is public. The pusher's revocation is the natural trigger: a pusher who revokes to switch to a new creator is the exact victim. This is a **medium-to-high likelihood** event in any live deployment where pushers rotate between creators.

---

### Recommendation

Add a per-pusher nonce to the signed digest and increment it on every successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        uint256 nonce = pusherNonce[pusher]++;   // <-- consume nonce

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce))
        );
        require(pusher == ECDSA.recover(hash, signatures[i]));

        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

The nonce must also be incremented on `revokePusher` / `removePushers` so that any outstanding signed consent for the old delegation is invalidated immediately upon revocation.

### Proof of Concept

```
Setup:
  - CompressedOracleV1 deployed, deadline D = block.timestamp + 365 days
  - Pusher P signs: keccak256(abi.encode(chainid, oracle, D, P, creatorA))
  - creatorA calls allowPushers(D, [P], [sig])
    → namespaceRemapping[P] = creatorA  ✓

Step 1 – Pusher revokes:
  - P calls revokePusher()
    → namespaceRemapping[P] = address(0)
  - P signs new consent for creatorB and creatorB calls allowPushers
    → namespaceRemapping[P] = creatorB  ✓

Step 2 – creatorA replays old sig (deadline D still in future):
  - creatorA calls allowPushers(D, [P], [original_sig])
    → ECDSA.recover succeeds (same digest, same sig)
    → namespaceRemapping[P] = creatorA  ← hijacked

Step 3 – Consequence:
  - P pushes price updates → land in creatorA's namespace
  - creatorB's feeds: timestampMs = 0 (stale sentinel)
  - Pool consuming creatorB's feedId calls getOracleData → price=0, ts=0
  - Pool swap reverts or executes against frozen/zero price
``` [5](#0-4) [6](#0-5)
