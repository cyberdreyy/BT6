### Title
`updateBySignature` Lacks a Transaction Deadline, Allowing a Malicious Relayer to Delay Price Updates Until Stale — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`CompressedOracle.updateBySignature` carries no transaction deadline. Its inline comment asserts "no deadline is needed" because the 56-bit oracle timestamp embedded in `newSlotValue` neutralises replay via the monotonicity check. That reasoning conflates two distinct concepts: the oracle observation timestamp (when the price was sampled) and a transaction deadline (the latest block time at which the transaction may be included). A malicious relayer or block producer can hold a signed message indefinitely and submit it after the market price has moved, writing a stale price into the oracle that the downstream `PriceProvider` / `AnchoredPriceProvider` will accept as fresh.

---

### Finding Description

`updateBySignature` accepts a creator-signed `newSlotValue` word that encodes a 56-bit `timestampMs` and four compressed price observations. The only time-related guards are:

1. **Future-timestamp guard** (`revertIfAfterBlockTimeWithDrift`): rejects the word only if its embedded timestamp exceeds `block.timestamp + MAX_TIME_DRIFT`. As real time advances, `block.timestamp` grows, so this check becomes *easier* to pass — the transaction never expires.
2. **Monotonicity check**: the word is stored only if its timestamp is strictly newer than the currently stored one. [1](#0-0) 

Neither guard constitutes a transaction deadline. The signed payload `keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))` binds the chain, the oracle address, the creator, and the slot word — but not a wall-clock expiry for the transaction itself. [2](#0-1) 

Contrast this with `allowPushers`, which explicitly adds a `deadline` argument and includes it in the signed digest precisely because "the signed consent carries no timestamp of its own": [3](#0-2) 

The same omission applies to the `fallback()` direct-push path, though that path is less critical because the pusher controls the calldata timestamp and can immediately re-push with a newer timestamp to override a held transaction. [4](#0-3) 

The `revertIfAfterBlockTimeWithDrift` helper only checks the upper bound (not-too-future); it imposes no lower bound (not-too-past): [5](#0-4) 

---

### Impact Explanation

Once the stale slot word is stored, `getOracleData` returns the embedded `timestampMs` converted to seconds via `toSeconds()`: [6](#0-5) 

`PriceProvider._getBidAndAskPrice` accepts the price as fresh if `nowTs - refTime <= MAX_TIME_DELTA`. `MAX_TIME_DELTA` is bounded only to `[1 second, 7 days]`: [7](#0-6) [8](#0-7) 

`AnchoredPriceProvider._readLeg` applies the same staleness window (`MAX_REF_STALENESS`, also up to 7 days): [9](#0-8) 

If the relayer delays inclusion by any duration shorter than `MAX_TIME_DELTA`, the stale price passes the staleness gate and reaches the pool's swap. The pool then quotes bid/ask derived from the old price, allowing a trader to buy at a price that no longer reflects the market — a direct swap conservation failure and potential LP loss.

---

### Likelihood Explanation

The attack requires a relayer (or block producer) who receives a signed `updateBySignature` payload and withholds it. This is a realistic threat model:

- Creators commonly use relayer infrastructure to avoid holding ETH for gas.
- A compromised or malicious relayer can trivially delay submission.
- The window is bounded only by `MAX_TIME_DELTA` (up to 7 days), which is a very wide holding period.
- The creator has no on-chain signal that the transaction is being withheld; they may believe the update landed.
- The monotonicity check only protects if the creator independently signs and submits a *newer* slot word before the relayer acts — which requires the creator to detect the withholding.

---

### Recommendation

Add a `deadline` parameter to `updateBySignature` and include it in the signed digest, mirroring the pattern already used in `allowPushers`:

```solidity
function updateBySignature(
    address feedCreator,
    uint256 newSlotValue,
    uint256 deadline,          // ← add
    bytes calldata signature
) external override returns (bool) {
    _ensureDeadline(deadline); // ← add

    // ... existing timestamp / monotonicity checks ...

    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(
            block.chainid,
            address(this),
            feedCreator,
            newSlotValue,
            deadline           // ← bind deadline into the signed digest
        ))
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    // ...
}
```

`_ensureDeadline` already exists in `OracleBase`: [10](#0-9) 

Creators should set the deadline to a short window (e.g., 60–300 seconds) matching their expected block-inclusion latency.

### Proof of Concept

```
t=0   Creator signs newSlotValue encoding price P1 at timestampMs T1.
      Sends to relayer for gas-less submission.

t=0   Relayer receives the signed payload but withholds it.

t=Δ   Real market price moves from P1 to P2 (e.g., +5%).
      No newer updateBySignature has been submitted (creator believes T1 landed).

t=Δ   Relayer submits the T1 payload.
      revertIfAfterBlockTimeWithDrift: T1/1000 ≤ block.timestamp + MAX_TIME_DRIFT ✓
        (block.timestamp has grown; check is easier, not harder)
      Monotonicity: T1 > stored timestamp (no newer push) ✓
      Signature: valid ✓
      → oracleData slot stores price P1 with timestamp T1.

t=Δ   Pool swap calls PriceProvider.getBidAndAskPrice().
      refTime = T1/1000, nowTs = block.timestamp.
      nowTs - refTime = Δ ≤ MAX_TIME_DELTA ✓  (staleness gate passes)
      Pool quotes bid/ask derived from stale P1.
      Trader buys at P1 < P2 → LP receives less than market value.
```

    function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
        external
        override
        returns (bool)
    {
        require(feedCreator != address(0), InvalidNamespace());

        uint256 namespace;
        assembly ("memory-safe") {
            namespace := shl(96, feedCreator) // [creator:20][zeros:12]
        }

        uint8 slotId = uint8(newSlotValue); // LSB
        TimeMs timestampMs = toTimeMs(newSlotValue >> 8 & X56);
        timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
        bytes32 key = bytes32(namespace | uint256(slotId));
        uint256 old = uint256(_loadStorage(key));
        TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

        bool newer = timestampMs.isAfter(oldTimestampMs);
        if (!newer) {
            return false;
        }

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
        );
        require(feedCreator == ECDSA.recover(hash, signature));

        _writeStorage(key, bytes32(newSlotValue & ~uint256(0xff)));

        return true;
    }

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
