### Title
`lastDecayTs` Not Reset on Pool Unpause Causes Paused-Period Decay to Silently Disable Stop-Loss Protection — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` tracks per-bin high watermarks with a `lastDecayTs` timestamp. Because `lastDecayTs` is only updated inside `_checkAndUpdateWatermarks`, which is only reachable via `afterSwap`, and because swaps revert while a pool is paused, the timestamp is never advanced during a pause. When the pool is unpaused, the first swap computes `dt = block.timestamp − lastDecayTs` over the entire paused interval, applying the full accumulated decay in one shot. If the pause is long enough relative to `decayPerSecondE8`, the watermarks collapse to zero, and the stop-loss check becomes permanently inert for that swap — allowing a swap that should have been blocked to execute.

---

### Finding Description

`_checkAndUpdateWatermarks` computes elapsed time as:

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
``` [1](#0-0) 

It then passes `dt` to `_decayed`:

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;          // ← floors to zero
    return hwm - (hwm * factor) / E8;
}
``` [2](#0-1) 

`lastDecayTs` is only written in two places: inside `_checkAndUpdateWatermarks` (line 284) and inside `executeOracleStopLossHighWatermarks` (line 174). Neither is callable while the pool is paused. [3](#0-2) 

The pool's `swap()` reverts at any `pauseLevel > 0`: [4](#0-3) 

So during a pause, `lastDecayTs` is frozen at the timestamp of the last pre-pause swap. When the pool is unpaused and the first swap arrives, `dt` spans the entire paused interval. With the example decay rate of 58 E8/s (~5%/day), `factor = 58 × dt` reaches `1e8` after ≈ 19.95 days, at which point `_decayed` returns 0. With higher rates (up to the maximum of `1e8`/s), even a 1-second pause suffices.

When both watermarks are 0, `_applyWatermark(metric, 0, floorMultiplier)` returns `(metric, false)` — no breach — because `metric >= 0` is always true:

```solidity
function _applyWatermark(uint256 metric, uint256 hwm, uint256 floorMultiplier)
    private pure returns (uint256 newHwm, bool breached)
{
    if (metric >= hwm) return (metric, false);   // hwm == 0 → always takes this branch
    ...
}
``` [5](#0-4) 

The watermarks then ratchet up to the current (post-pause) metric, silently resetting the stop-loss baseline to whatever value the pool holds after the pause — including any value that was lost during the pause period.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain guard preventing swaps that would extract value from LPs beyond the configured drawdown threshold. When the watermarks are zeroed by pause-period decay, this guard is completely bypassed for the first post-unpause swap. An attacker who observes that the pool has been paused long enough to zero the watermarks can immediately execute a swap that drains LP value up to the drawdown limit without triggering `OracleStopLossTriggered`. This constitutes a direct loss of LP principal — the exact impact class the extension is designed to prevent.

---

### Likelihood Explanation

- Pool admins are semi-trusted and can pause pools for legitimate operational reasons (oracle outage, security incident, upgrade).
- The pause duration is unbounded; there is no maximum pause time enforced by the protocol.
- The decay rate is configurable up to `1e8`/s; even moderate rates (58/s ≈ 5%/day) zero the watermarks after ~20 days.
- The exploit requires no special privilege: any unprivileged swapper can submit the first post-unpause swap.
- The attacker can monitor on-chain pause events and `currentHighWatermarks` to know exactly when the watermarks have decayed to zero. [6](#0-5) 

---

### Recommendation

When a pool is unpaused, `lastDecayTs` should be advanced to `block.timestamp` for all active bins, so the paused interval is excluded from decay accounting — exactly analogous to updating `lastRewardBlock` on emission resumption.

Since the extension has no direct hook into the unpause path, the cleanest fix is to add a permissionless `syncDecayTimestamp(address pool_, int8 binIdx)` function that, when the pool is not paused, writes `hwmS.lastDecayTs = uint32(block.timestamp)` without altering the watermark values. Alternatively, `_checkAndUpdateWatermarks` can cap `dt` to a protocol-defined maximum (e.g., the pool's configured timelock) so that a single long gap cannot zero the watermarks in one step.

### Proof of Concept

**Setup:**
- Pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 58` (~5%/day).
- Bin 0 holds `t0 = 1000, t1 = 1000` at price `Q64`. Watermarks are set to the corresponding metrics after the first swap at `T0`.

**Step 1 — Pause at T0:**
Pool admin calls `factory.pausePool(pool)`. `lastDecayTs` for bin 0 remains at `T0`.

**Step 2 — Wait 21 days:**
`dt = 21 days = 1,814,400 s`. `factor = 58 × 1,814,400 = 105,235,200 > 1e8`. `_decayed` returns 0 for both watermarks.

**Step 3 — Unpause at T0 + 21 days:**
Pool admin calls `factory.unpausePool(pool)`. `lastDecayTs` is still `T0`.

**Step 4 — Attacker drains value:**
Attacker calls `swap(zeroForOne=true, ...)`. `_checkAndUpdateWatermarks` runs:
- `dt = (T0 + 21 days) − T0 = 1,814,400 s`
- `_decayed(hwm0, 58, 1,814,400)` → `factor ≥ 1e8` → returns `0`
- `_applyWatermark(metricT0_current, 0, 950_000)` → `metricT0_current >= 0` → `(metricT0_current, false)`
- `breach0 = false` → no revert

The swap executes even if the pool's value per share has dropped by more than 5% during the pause, bypassing the stop-loss entirely. The watermarks ratchet up to the current (degraded) metric, silently resetting the baseline. [7](#0-6)

    address pool_,
    int8 binIdx,
    uint256 metricT0,

    bool zeroForOne
