### Title
`allowPushers` EIP-191 Signature Is Replayable Within the Deadline Window, Making `revokePusher()` Ineffective — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

The `allowPushers` function accepts a pusher's EIP-191 signature that commits to `(chainid, oracle, deadline, pusher, creator)` but includes **no nonce and no one-time-use invalidation**. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original `allowPushers` call with the **identical signature**, re-establishing the delegation. `revokePusher()` is therefore ineffective for the entire lifetime of the deadline window — a direct analog of the `finishWithdrawal()` bug class where executed state is never cleared.

---

### Finding Description

`allowPushers` verifies the pusher's consent via ECDSA and writes `namespaceRemapping[pusher] = msg.sender`: [1](#0-0) 

The signed digest is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
```

It commits to `deadline`, `pusher`, and `creator` — but **not to any nonce or per-call state**. The same `(deadline, signature)` tuple is valid for every call until `block.timestamp > deadline`.

`revokePusher()` clears the mapping: [2](#0-1) 

But nothing marks the original signature as consumed. The creator can immediately call `allowPushers` again with the same arguments, writing `namespaceRemapping[pusher] = creator` back. The code's own comment reveals the developer's intent:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline limits the **window** but does not prevent **replay within that window**. The developer's stated goal — preventing re-establishment after revocation — is not achieved.

The `fallback()` push path resolves the effective namespace from `namespaceRemapping[msg.sender]`: [4](#0-3) 

So every push the pusher makes after their attempted revocation continues to land in the **creator's** namespace, not the pusher's own.

---

### Impact Explanation

A creator who holds a pusher's signed consent can:

1. Re-establish delegation atomically in the same block as the pusher's `revokePusher()` call, or in any subsequent block before the deadline.
2. Force a contract-pusher (e.g., an automated keeper) to keep writing price data into the creator's namespace indefinitely — up to the deadline — even after the pusher has explicitly revoked consent.
3. Keep the creator's oracle feeds fresh with the pusher's data stream while the pusher's **own** namespace feeds remain stale (all pushes are redirected), breaking the pusher's ability to serve their own consumers.

If the creator's feeds are consumed by a pool via `PriceProvider`, this allows the creator to maintain oracle liveness using a data source that has withdrawn consent, which is an oracle role-check bypass reachable by a semi-trusted party (the creator).

**Severity: Medium** — direct loss of oracle integrity for the pusher's own namespace; pool consumers of the creator's feeds receive prices from a source that has revoked authorization.

---

### Likelihood Explanation

- The creator must have obtained a pusher's signature with a non-trivial deadline (common in production deployments where long-lived keys are used).
- The creator must be motivated to maintain delegation after revocation (e.g., to keep their feeds live after a pusher dispute).
- No special privileges beyond holding the original signed message are required; the replay is a single public function call.

**Likelihood: Medium.**

---

### Recommendation

Add a per-pusher nonce to the signed digest and track it on-chain, or record used signature hashes:

```solidity
// Option A: nonce per (creator, pusher) pair
mapping(address => mapping(address => uint256)) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[msg.sender][pusher]++   // <-- invalidates after first use
    ))
);
```

Alternatively, after `revokePusher()` is called, increment a per-pusher nonce so any previously signed message is immediately invalidated regardless of its deadline.

---

### Proof of Concept

```
T=0:  Pusher signs: H = keccak256(abi.encode(chainid, oracle, deadline=T+365days, pusher, creator))
T=1:  Creator calls allowPushers(T+365days, [pusher], [sig])
        → namespaceRemapping[pusher] = creator  ✓

T=2:  Pusher calls revokePusher()
        → namespaceRemapping[pusher] = address(0)  (pusher believes they are free)

T=3:  Creator calls allowPushers(T+365days, [pusher], [sig])  // SAME sig, SAME deadline
        → namespaceRemapping[pusher] = creator  ✗ (revocation undone)

T=4..T+365days: Creator repeats step T=3 after every revokePusher() call.
        Pusher's pushes continue landing in creator's namespace.
        Pusher's own namespace feeds remain at timestamp 0 (stale / rejected by consumers).
```

The pusher's `revokePusher()` is fully neutralized for the entire deadline window with zero additional cost to the creator.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
