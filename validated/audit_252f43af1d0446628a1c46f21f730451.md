### Title
`revokePusher()` is ineffective before deadline expiry — creator can re-establish delegation using the same signature - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

`CompressedOracleV1.revokePusher()` is supposed to let a pusher permanently exit a creator's namespace. However, because `allowPushers()` performs no used-signature tracking and no check that the pusher has not already revoked, the creator can immediately replay the original EIP-191 consent (with the same `deadline`, `pusher`, and `msg.sender` tuple) to re-establish the delegation. The revocation is therefore ineffective for the entire window between the revocation call and the deadline timestamp — an exact structural analog to the `readOnlyMode` false-security pattern in the seed report.

---

### Finding Description

`allowPushers` signs consent as:

```
hash = EIP191( keccak256(abi.encode(chainid, address(this), deadline, pusher, creator)) )
``` [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)` — there is no nonce, no used-signature set, and no check that `namespaceRemapping[pusher]` is currently `address(0)`. [2](#0-1) 

`revokePusher()` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

Because the signature is still valid (deadline has not expired), the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments and overwrite `namespaceRemapping[pusher]` back to `creator`. The code's own NatSpec comment acknowledges the risk but mischaracterises the deadline as the fix:

> *"an undated signature could re-establish a delegation AFTER the pusher revoked it"* [4](#0-3) 

The deadline only prevents replay **after** it expires; before expiry the same signature is unconditionally accepted, so the pusher's revocation is fully reversible by the creator.

---

### Impact Explanation

The `fallback()` push path resolves the target namespace as:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

After the creator re-establishes the delegation, every push the pusher makes — even pushes the pusher believes are going into their **own** namespace — is silently written into the creator's namespace under `feedIdOf(creator, slotIndex, positionIndex)`.

If the pusher has revoked because they are now publishing prices for a **different asset** (e.g., they switched from ETH/USD to BTC/USD), those wrong prices land in the creator's namespace. Any pool whose `PriceProvider` reads from `feedIdOf(creator, ...)` will consume the wrong asset's price during `getBidAndAskPrice()`: [6](#0-5) 

This satisfies the **bad-price execution** impact gate: a stale or wrong-asset bid/ask quote reaches a live pool swap, causing traders to receive more than the oracle curve permits or LPs to suffer mispriced settlement.

---

### Likelihood Explanation

- The pusher must have signed a consent with a future deadline (normal operational practice for long-lived pushers).
- The pusher must have revoked and begun pushing a different feed.
- The creator must replay the old signature — a single on-chain transaction requiring no special privilege beyond possessing the original calldata.

No off-chain coordination, no flash loan, and no admin key is required. The window is as long as the original deadline, which in production deployments is typically days to weeks.

---

### Recommendation

Add a per-pusher revocation nonce stored in the oracle:

```solidity
mapping(address pusher => uint256) public pusherNonce;
```

Include the nonce in the signed message:

```solidity
keccak256(abi.encode(chainid, address(this), deadline, pusher, creator, pusherNonce[pusher]))
```

Increment `pusherNonce[pusher]` inside `revokePusher()`. Any previously issued signature becomes invalid immediately upon revocation, regardless of its deadline.

Alternatively, add a guard at the top of `allowPushers` that reverts if `namespaceRemapping[pusher] != address(0)`, forcing the pusher to explicitly re-sign a new consent after each revocation.

---

### Proof of Concept

```solidity
// 1. Setup
CompressedOracleV1 oracle = new CompressedOracleV1(admin, 0);
uint256 deadline = block.timestamp + 365 days;   // long-lived consent

// 2. Pusher signs consent for creator
bytes memory sig = sign(PUSHER_KEY,
    keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator)));

// 3. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);   // ✓ delegated

// 4. Pusher revokes — expects to push into own namespace from now on
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // ✓ cleared

// 5. Creator replays the SAME signature — no new consent from pusher
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));  // ← succeeds, no revert
assertEq(oracle.namespaceRemapping(pusher), creator);    // ✗ delegation restored

// 6. Pusher now pushes BTC/USD prices believing they go to own namespace;
//    they silently land in creator's ETH/USD pool feed → bad-price execution.
vm.prank(pusher);
(bool ok,) = address(oracle).call(btcUsdSlotWord);
assertTrue(ok);
// creator's pool reads feedIdOf(creator, slot, pos) → gets BTC/USD price → wrong swap
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
