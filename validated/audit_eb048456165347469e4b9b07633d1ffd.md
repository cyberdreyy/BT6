### Title
`allowPushers` Consent Signature Replay Within Deadline Window Re-Establishes Revoked Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers()` accepts a pusher's EIP-191 consent signature that is bound to `(chainid, address(this), deadline, pusher, creator)` but carries **no nonce and no check of the current `namespaceRemapping` state**. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original consent signature — as long as the deadline has not yet expired — to silently re-establish the mapping. The pusher's subsequent fallback pushes then land in the creator's namespace instead of their own, feeding the creator's oracle feeds with prices the pusher never intended for that namespace.

---

### Finding Description

The code's own NatSpec comment acknowledges the replay concern:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."

The deadline is intended to close this window, but it only prevents replay **after** the deadline expires. Within the deadline window the same signature is unconditionally valid:

```solidity
// CompressedOracle.sol L192-211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);                          // only checks deadline > block.timestamp
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;            // overwrites without checking current state
``` [1](#0-0) 

`revokePusher()` clears the mapping to `address(0)`:

```solidity
// L238-243
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [2](#0-1) 

Because `allowPushers` does not check whether `namespaceRemapping[pusher]` is already `address(0)` (i.e., the pusher has revoked), the creator can call it again with the identical signature to restore the mapping. The fallback push path then resolves the namespace from the remapping:

```solidity
// L315-316
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [3](#0-2) 

So any push the pusher makes after revoking — believing they are writing into their own namespace — silently lands in the creator's namespace instead.

---

### Impact Explanation

The creator's oracle feeds are consumed by `AnchoredPriceProvider` / pool `swap()` calls. If the creator re-establishes delegation after the pusher revokes, the pusher's subsequent price pushes (intended for their own namespace) populate the creator's slots. A pool reading `price(feedId, pool)` for the creator's feed receives the hijacked price. This constitutes **bad-price execution**: a bid/ask quote the pool was not supposed to receive reaches a live swap, causing traders to receive incorrect amounts and the pool to settle at a wrong price — a direct loss of user principal.

---

### Likelihood Explanation

- The attack requires only that the creator retain the original consent signature (trivially available from calldata history) and that the deadline has not yet expired.
- Deadlines are typically set days or weeks in the future (the test suite uses `block.timestamp + 1 days`).
- The pusher has no on-chain mechanism to invalidate the signature before the deadline; their only recourse is to stop pushing entirely and wait for the deadline to expire.
- The pusher may be unaware the delegation was re-established (no event is emitted that differs from the original `PusherAuthorized`), so they may continue pushing prices they believe are going to their own namespace.

---

### Recommendation

Add a per-pusher nonce to the signed message and increment it on every successful `allowPushers` call (or on every `revokePusher` call). This makes any previously issued consent signature immediately invalid after revocation:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;          // invalidates this signature for future replays
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, check that `namespaceRemapping[pusher] == address(0)` before writing, so a revoked pusher cannot be silently re-delegated without a fresh signature.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
