### Title
`allowPushers` consent signature is replayable within the deadline window, making `revokePusher()` ineffective — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` accepts a pusher's EIP-191 consent signature and writes `namespaceRemapping[pusher] = creator`. The signature is never invalidated after first use. A creator who holds a valid (not-yet-expired) consent signature can replay it an unlimited number of times. Because `revokePusher()` only zeroes the mapping entry, the creator can immediately re-establish delegation by replaying the original signature, making the pusher's revocation silently ineffective for the entire lifetime of the deadline.

---

### Finding Description

`allowPushers` computes the consent hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no per-pusher sequence counter, and no `usedSignatures` bitmap. The grep across all `.sol` files confirms zero instances of `usedSignatures`, `signatureUsed`, or any nonce tracking in the oracle contracts.

The code's own NatSpec comment acknowledges the exact attack class it is trying to prevent:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [2](#0-1) 

The deadline prevents replay **after** it expires, but does nothing to prevent replay **within** the deadline window. `revokePusher()` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

The creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple, restoring `namespaceRemapping[pusher] = creator`. The pusher has no on-chain way to detect this; they believe they have revoked.

The `fallback()` push path resolves the effective namespace at push time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

After the creator replays the signature, every subsequent push from the pusher silently lands in the creator's namespace again, not the pusher's own namespace as the pusher intended.

The creator's namespace is the source for `feedIdOf(creator, slotIndex, positionIndex)`, which is the `baseFeedId` (or `quoteFeedId`) bound immutably into an `AnchoredPriceProvider`: [5](#0-4) 

That provider's `getBidAndAskPrice()` → `_readLeg()` → `IPricedOracle.price(feedId, pool)` path reads directly from the hijacked namespace and delivers the resulting bid/ask to the pool swap: [6](#0-5) 

---

### Impact Explanation

A pusher who discovers they are pushing incorrect prices (e.g., a broken upstream feed, a misconfigured codebook index, or a compromised signing key) calls `revokePusher()` as a safety measure. The creator replays the original consent signature before the deadline expires. The pusher, believing they have revoked, continues to push — but every push still lands in the creator's namespace. The pool's `AnchoredPriceProvider` continues to consume those prices. The staleness guard (`MAX_REF_STALENESS`) and price guard (`priceGuard`) are the only remaining defences, but neither detects a price that is wrong-but-fresh and within the guard range. Bad bid/ask quotes reach the pool swap, causing swap conservation failure or incorrect LP settlement.

---

### Likelihood Explanation

- Deadlines are set by the creator and can be arbitrarily long (e.g., 30 days or more).
- The replay requires only a single additional `allowPushers` call — no special privilege, no gas-intensive operation.
- The pusher has no event or view to detect that their revocation was undone; `PusherRevoked` fires on revoke, but `PusherAuthorized` fires again on replay with no distinction from the original authorization.
- Any creator who has stored the original `(deadline, sig)` tuple off-chain (the normal operational pattern) can execute this at zero cost.

---

### Recommendation

Mark each consent signature as consumed on first use. Add a `mapping(bytes32 => bool) public usedConsentSignatures` and set it to `true` after the first successful `allowPushers` call for that hash:

```solidity
mapping(bytes32 => bool) public usedConsentSignatures;

// inside allowPushers, after ECDSA.recover succeeds:
bytes32 sigHash = keccak256(signatures[i]);
require(!usedConsentSignatures[sigHash], "consent already used");
usedConsentSignatures[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful delegation, so a revoked pusher must issue a fresh signature with the new nonce before delegation can be re-established.

---

### Proof of Concept

```
T=0:  Pusher signs consent: hash = keccak256(chainid, oracle, deadline=T+30d, pusher, creator)
T=1:  Creator calls allowPushers(T+30d, [pusher], [sig])
        → namespaceRemapping[pusher] = creator  ✓
T=2:  Pusher discovers bad prices; calls revokePusher()
        → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are safe)
T=3:  Creator calls allowPushers(T+30d, [pusher], [sig])  ← SAME sig, still valid
        → namespaceRemapping[pusher] = creator  ← silently restored
T=4+: Pusher continues pushing (into what they think is their own namespace)
        fallback: creator = namespaceRemapping[pusher] = creator (not address(0))
        → push lands in creator's namespace
        → AnchoredPriceProvider reads creator's namespace
        → bad bid/ask delivered to pool swap
```

The `_ensureDeadline` check in `OracleBase` only enforces `block.timestamp <= deadline`: [7](#0-6) 

It does not prevent the same signature from being submitted a second time within the window. No other guard in `allowPushers` or `OracleBase` closes this gap.
