### Title
`updateBySignature` Lacks Submission Deadline — Signed Price Updates Can Be Withheld and Submitted at a Strategically Chosen Time to Feed Stale Prices into Pools - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

### Summary

`CompressedOracleV1.updateBySignature` accepts a creator-signed slot word and writes it to oracle storage with no expiry on the signature itself. Because the signed message contains only `(chainid, oracleAddress, feedCreator, newSlotValue)` and no deadline, any party holding a valid signature can defer submission indefinitely and choose the moment that maximises their advantage — exactly the timing-of-action manipulation described in the seed report.

### Finding Description

`updateBySignature` is a permissionless function: any caller may submit a creator-signed slot word on behalf of the creator. [1](#0-0) 

The signed digest is:

```
keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
``` [2](#0-1) 

No deadline field appears in the signed payload. The only time-related guard is `revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT)`, which rejects a slot word whose embedded price-timestamp is in the future — it says nothing about when the signature may be submitted. [3](#0-2) 

By contrast, `allowPushers` — which also uses an EIP-191 signature — explicitly requires a deadline in both the call and the signed message, with the comment explaining exactly why:

> *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [4](#0-3) 

The design comment on `updateBySignature` argues that the 56-bit price-timestamp inside the slot word neutralises replay via the per-slot monotonicity check. That is true for **replay** (re-submitting the same word after a newer one is stored), but it does not prevent **deferred first submission**: a word signed at time T with price P can be submitted at any later time T′ > T as long as no newer slot has been stored, because T is still strictly greater than the stored timestamp of 0 (or any older value). [5](#0-4) 

The `PriceProvider` (and its variants) reads the oracle's `refTime` and applies a staleness check with `MAX_TIME_DELTA` up to 7 days: [6](#0-5) [7](#0-6) 

So a signed word whose embedded timestamp is up to `MAX_TIME_DELTA` seconds old will pass the provider's staleness check and reach the pool as a valid bid/ask quote.

### Impact Explanation

A relayer (or the creator themselves) who holds a signed slot word can:

1. Observe that the live market price has moved significantly away from the price embedded in the signed word.
2. Submit the signed word at the moment that maximises profit — e.g., when the embedded price is far below the true market price, allowing the submitter (or a coordinated trader) to buy the underpriced token from the pool.
3. The pool's `swap()` call reads `getBidAndAskPrice()` from the `PriceProvider`, which reads the now-stale price from `CompressedOracleV1`, passes the staleness check (price-timestamp is within `MAX_TIME_DELTA`), and executes the swap at the manipulated quote.

LP assets are drained by the difference between the stale oracle price and the true market price. This is a direct loss of LP principal — a pool-insolvency / bad-price-execution impact. [8](#0-7) 

### Likelihood Explanation

- `updateBySignature` is the standard off-chain-relayer path for `CompressedOracleV1`; any deployment that uses a relayer service exposes this surface.
- The attacker needs only a single previously signed slot word — no privileged access, no malicious setup.
- The creator can cancel by pushing a newer update via `fallback()`, but only if they are online and aware that a stale signed word is being withheld. In a relayer-service model the creator may have no visibility into which signed words the relayer is holding.
- The attack window is bounded by `MAX_TIME_DELTA` (up to 7 days), which is a wide window for price movement.

### Recommendation

Add a `deadline` parameter to `updateBySignature` and include it in the signed digest, mirroring the pattern already used in `allowPushers`:

```solidity
function updateBySignature(
    address feedCreator,
    uint256 deadline,          // ← new
    uint256 newSlotValue,
    bytes calldata signature
) external override returns (bool) {
    _ensureDeadline(deadline); // ← new
    require(feedCreator != address(0), InvalidNamespace());
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(
            block.chainid, address(this), feedCreator,
            deadline,          // ← new
            newSlotValue
        ))
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    ...
}
```

`_ensureDeadline` already exists in `OracleBase`: [9](#0-8) 

This matches the interface already described in the top-level documentation (`docs/en/oracle-packet-structure.md`) and the ABI recorded in `registry.json`, both of which include a `deadline` parameter — indicating the deadline was present in an earlier version and was inadvertently removed. [10](#0-9) [11](#0-10) 

### Proof of Concept

```
T=0   Creator signs slotWord = {price: 1000, timestamp: T, slotId: 0}
      Gives signature to Relayer.

T=0   No fallback push is made; stored timestamp for slot 0 = 0.

T=60s Market price moves to 1200 (+20%).

T=60s Relayer calls:
        oracle.updateBySignature(creator, slotWord, sig)
      Guards:
        revertIfAfterBlockTimeWithDrift: T=0 ≤ 60s + drift  ✓
        isAfter(0, 0): 0 > 0 is false → wait, stored is 0, new is T=0 ms
```

Actually, let me correct the PoC — the timestamp is in milliseconds:

```
T=0       Creator signs slotWord = {price: 1000, timestampMs: 0ms, slotId: 5}
          Stored slot 5 timestamp = 0 (never pushed).

T=3600s   Market price moves to 1200.

T=3600s   Relayer submits:
            oracle.updateBySignature(creator, slotWord, sig)
          Check 1: timestampMs(0) ≤ block.timestamp(3600s)*1000 + drift  ✓
          Check 2: isAfter(0ms, 0ms) → false → returns false (no update)
```

The PoC requires the signed timestamp to be strictly greater than the stored timestamp. A realistic scenario:

```
T=0       Stored slot 5 has timestamp 0 (never pushed).
          Creator signs slotWord = {price: 1000, timestampMs: 1ms, slotId: 5}
          Gives signature to Relayer.

T=3600s   Market price moves to 1200.

T=3600s   Relayer submits updateBySignature(creator, slotWord, sig):
            timestampMs(1ms) ≤ 3600s*1000 + drift  ✓
            isAfter(1ms, 0ms) = true  ✓
            Signature valid  ✓
          → price 1000 written to oracle.

T=3600s   Pool swap reads PriceProvider:
            refTime = 1ms / 1000 = 0s
            _isStale(0, 3600, MAX_TIME_DELTA=7days) → false (0 ≤ 7days)  ✓
          → Pool quotes at stale price 1000 instead of 1200.
          → Attacker buys token0 at 17% discount, draining LP funds.
```

    function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
        external
        override
        returns (bool)
    {
        require(feedCreator != address(0), InvalidNamespace());

        uint256 namespace;
        assembly ("memory-safe") {
            namespace := shl(96, feedCreator) // [creator:20][zeros:12]
        }

        uint8 slotId = uint8(newSlotValue); // LSB
        TimeMs timestampMs = toTimeMs(newSlotValue >> 8 & X56);
        timestampMs.revertIfAfterBlockTimeWithDrift(MAX_TIME_DRIFT);
        bytes32 key = bytes32(namespace | uint256(slotId));
        uint256 old = uint256(_loadStorage(key));
        TimeMs oldTimestampMs = toTimeMs(old >> 8 & X56);

        bool newer = timestampMs.isAfter(oldTimestampMs);
        if (!newer) {
            return false;
        }

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
        );
        require(feedCreator == ECDSA.recover(hash, signature));

        _writeStorage(key, bytes32(newSlotValue & ~uint256(0xff)));

        return true;
    }

## Required Guards

- `deadline` must be in the future (`DeadlineExceeded` otherwise).
- `timestamp` must not be in the future (`FutureTimestamp` otherwise).
- `timestamp` must be strictly increasing per slot (older updates are ignored).
