### Title
`revokePusher()` Provides No Finality Within Deadline Window — Creator Can Replay Original Consent Signature to Re-Delegate a Self-Revoked Pusher — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers()` accepts any valid EIP-191 signature whose deadline has not yet expired. There is no nonce, no used-signature tracking, and no check that the pusher has previously self-revoked. A creator who holds a pusher's original consent signature can call `allowPushers()` repeatedly — including after the pusher has called `revokePusher()` — to silently restore the delegation without any new consent from the pusher.

### Finding Description
`revokePusher()` sets `namespaceRemapping[msg.sender] = address(0)` but does not invalidate the pusher's original signed consent. [1](#0-0) 

`allowPushers()` only checks that the deadline has not passed and that the ECDSA-recovered signer equals the pusher. Both conditions remain true after revocation if the deadline is still in the future. [2](#0-1) 

`_ensureDeadline()` performs only a timestamp comparison — no revocation state is consulted. [3](#0-2) 

The code comment at lines 186–191 explicitly states the deadline is the mechanism that prevents re-delegation after revocation, but this reasoning is only valid once the deadline has expired — not within the window. [4](#0-3) 

### Impact Explanation
The pusher's `revokePusher()` call is ineffective for the entire remaining lifetime of the original signature. Any data the pusher (or an attacker who has compromised the pusher's key) pushes via the `fallback()` path will land in the creator's namespace rather than the pusher's own namespace, because `namespaceRemapping[pusher]` has been restored to `creator`. [5](#0-4) 

The most direct fund-loss path: pusher's key is compromised → pusher calls `revokePusher()` to protect the creator's feeds → creator (unaware) replays the old signature to restore the delegation → attacker (holding the compromised key) pushes manipulated prices into the creator's namespace → those prices reach pool swaps via the `AnchoredPriceProvider`, causing bad-price execution and direct loss of user principal.

### Likelihood Explanation
Likelihood is medium. It requires the creator to actively call `allowPushers()` a second time with the same signature. This can happen inadvertently (creator's automation re-registers pushers from a stored list of signatures) or deliberately (malicious creator). Signatures with long deadlines (e.g., 1 year) are a natural operational choice, making the window wide.

### Recommendation
Track used or revoked consent signatures. The simplest fix is a `mapping(bytes32 => bool) public revokedConsents` keyed on the consent hash. `revokePusher()` should compute and mark the hash of any active consent as revoked, and `allowPushers()` should reject any hash present in that mapping. Alternatively, introduce a per-pusher nonce that is incremented on revocation and included in the signed payload, so any pre-revocation signature becomes invalid immediately.
