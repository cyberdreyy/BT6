### Title
Pusher Consent Signature Replay Nullifies `revokePusher()` Within Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` in `CompressedOracleV1` verifies a pusher's EIP-191 consent signature but stores no record that the signature has been consumed. Because the only replay barrier is the deadline timestamp, a creator can call `allowPushers` a second (or third, …) time with the **identical** `(deadline, pushers[], signatures[])` arguments to silently re-establish a delegation that the pusher already cancelled via `revokePusher()`. The pusher's self-revocation is therefore ineffective for the entire lifetime of the original deadline, and a compromised pusher key can continue injecting prices into the creator's oracle namespace — and from there into any pool that reads those feeds.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

After verifying the signature it unconditionally writes:

```solidity
namespaceRemapping[pusher] = msg.sender;
``` [2](#0-1) 

There is no "used-signature" set, no per-pusher nonce, and no check that `namespaceRemapping[pusher]` is currently `address(0)`. The only guard is `_ensureDeadline`, which merely requires `block.timestamp <= deadline`. [3](#0-2) 

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [4](#0-3) 

But because the original consent bytes are still valid (deadline not yet expired, signature unchanged), the creator can immediately call `allowPushers` again with the same arguments, restoring `namespaceRemapping[pusher] = creator`. This loop can repeat until the deadline expires.

The code comment acknowledges the deadline is the sole protection against post-revocation re-establishment, but it only bounds the attack window — it does not prevent replay within that window: [5](#0-4) 

---

### Impact Explanation

Once the delegation is re-established, the `fallback()` push path routes any calldata sent by the pusher's address into the **creator's** namespace:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [6](#0-5) 

Prices written there are immediately readable by any pool via the permissionless `price(feedId, pool)` view: [7](#0-6) 

If the pusher's private key is compromised, the pusher calls `revokePusher()` as an emergency stop. The creator replays the original `allowPushers` calldata to override the revocation. The attacker holding the compromised key then pushes manipulated prices (arbitrary `U64x32`-encoded values, arbitrary spread indexes) into the creator's slot. Pools executing swaps against those feeds receive bad bid/ask quotes, causing traders to receive more output than the oracle permits or the pool to receive less input than owed — a direct swap conservation failure and potential pool insolvency.

---

### Likelihood Explanation

- The creator already possesses the original `allowPushers` calldata (they submitted it on-chain; it is trivially recoverable from transaction history).
- No special privilege is required beyond being the creator (`msg.sender` in the original call).
- Deadlines are set by the creator and can be arbitrarily long (days or weeks), giving a wide replay window.
- The pusher's revocation emits a `PusherRevoked` event, so an attacker monitoring the chain can front-run or immediately follow with the replay.

---

### Recommendation

Track consumed consent signatures with a per-pusher nonce or a `mapping(bytes32 => bool) usedConsents` set, and reject any signature whose digest has already been processed. The simplest fix mirrors the H-03 mitigation: add a `uint256 pusherNonce` to the signed payload and increment it on each successful `allowPushers` call, so a revoked-and-replayed signature carries a stale nonce and is rejected.

```diff
+ mapping(address => uint256) public pusherNonce;

  function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
      _ensureDeadline(deadline);
      uint256 l = pushers.length;
      require(l == signatures.length);
      for (uint256 i; i < l; i++) {
          address pusher = pushers[i];
          if (pusher == msg.sender) revert NoSelfRemapping();

+         uint256 nonce = pusherNonce[pusher];
          bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
-             keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
+             keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce))
          );
          require(pusher == ECDSA.recover(hash, signatures[i]));

+         pusherNonce[pusher] = nonce + 1;
          namespaceRemapping[pusher] = msg.sender;
          emit PusherAuthorized(pusher, msg.sender);
      }
  }
```

### Proof of Concept

```
1. Pusher signs: hash(chainid, oracle, deadline=T+7days, pusher, creator)
2. Creator calls allowPushers(T+7days, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher's key is compromised; pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓ (pusher believes they are safe)

4. Creator (or on-chain observer) calls allowPushers(T+7days, [pusher], [sig])
   with the IDENTICAL arguments — deadline still valid, signature still valid
   → namespaceRemapping[pusher] = creator  (revocation silently undone)

5. Attacker (holding compromised pusher key) calls oracle.fallback() with
   a crafted 32-byte slot word encoding an arbitrary price and spread indexes.
   → The word lands in creator's namespace (slot key = creator << 96 | slotId)

6. A pool reading oracle.price(feedIdOf(creator, slotId, pos), pool)
   receives the attacker-controlled price and executes a swap at that price.
   → Swap conservation failure / pool insolvency.

Steps 4–6 can repeat until T+7days expires.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
