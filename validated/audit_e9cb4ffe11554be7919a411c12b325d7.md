### Title
`revokePusher()` Is Ineffective Due to Signature Replay in `allowPushers()` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers()` in `CompressedOracleV1` uses a deadline-bounded EIP-191 signature to authorize pusher delegation. Because there is no nonce or one-time-use mechanism, the same signature is valid for every call to `allowPushers()` until the deadline expires. A creator who holds a valid (non-expired) consent signature can re-establish delegation immediately after the pusher calls `revokePusher()`, rendering the revocation permanently ineffective within the deadline window. The pusher's price data continues to flow into the creator's namespace and reach any pool consuming those feeds.

### Finding Description

`allowPushers()` signs consent over:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The code comment explicitly acknowledges the risk: *"the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* The deadline is presented as the mitigation. However, within the deadline window the same bytes can be submitted to `allowPushers()` an unlimited number of times — there is no nonce, no consumed-signature registry, and no check that `namespaceRemapping[pusher]` is currently zero before writing. [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But because `allowPushers()` only checks `_ensureDeadline(deadline)` and the ECDSA recovery — both of which still pass for the original signature — the creator can call `allowPushers()` again with the identical `(deadline, [pusher], [sig])` tuple and restore `namespaceRemapping[pusher] = creator` in the same block as the revocation.

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So any push the pusher makes after believing they have revoked — intending to write to their own namespace — is silently redirected to the creator's namespace if the creator has replayed the delegation in the same or a prior block.

### Impact Explanation

The pusher's `revokePusher()` call is a no-op as long as the creator holds a non-expired consent signature. The pusher cannot stop their price data from flowing into the creator's namespace until the deadline expires (which may be months or years away). If the pusher continues pushing after revocation — for example, because they have changed their price methodology or are now pricing a different asset — those prices land in the creator's namespace and are consumed by any pool whose `PriceProvider` / `AnchoredPriceProvider` reads that feed:

```
pool.swap → provider.getBidAndAskPrice → oracle.price(feedId, pool)
``` [5](#0-4) 

A pusher who revokes and then begins pushing prices for a different asset or at a different scale (thinking they are writing to their own namespace) will instead overwrite the creator's live feed. The pool's staleness and price-guard checks do not protect against a fresh, in-range price that is simply wrong for the configured asset pair. This satisfies the bad-price execution impact gate.

### Likelihood Explanation

- Any creator who received a pusher consent signature with a future deadline can perform this attack.
- The attack requires only a single additional `allowPushers()` call after each `revokePusher()` — no special privilege, no MEV, no front-running beyond watching the mempool for the revocation transaction.
- Consent signatures are typically shared off-chain and may carry deadlines of days to years.
- The pusher has no on-chain way to detect that their revocation was undone before their next push lands.

### Recommendation

Add a per-pusher revocation nonce that is included in the signed message and incremented on every `revokePusher()` or `removePushers()` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers():
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- bind to current nonce
    ))
);

// In revokePusher() / removePushers():
pusherNonce[pusher]++;        // invalidates all prior signatures
namespaceRemapping[pusher] = address(0);
```

This ensures that any signature issued before a revocation is cryptographically invalidated by the nonce increment, making `revokePusher()` a true one-way gate.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }

    fallback() override external {
        uint256 end;
        uint256 namespace;

        address creator = namespaceRemapping[msg.sender];
        if (creator == address(0)) creator = msg.sender;

        assembly ("memory-safe") {
            end := calldatasize()
            namespace := shl(96, creator) // [creator:20][zeros:12]
        }

        // 4 * 6 + 7 + 1 = 32 bytes per slot
        if (end == 0 || end % 32 != 0) revert BadCalldataLength();

        for (uint256 ptr = 0; ptr < end; ptr += 32) {
            uint256 word;
            assembly ("memory-safe") {
                word := calldataload(ptr)
