### Title
Signature Replay in `allowPushers` Defeats `revokePusher` Finality Within the Deadline Window — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary

`allowPushers` has no nonce and no used-signature tracking. A creator who holds a valid EIP-191 signature from a pusher can call `allowPushers` repeatedly with the same `(deadline, pusher, signature)` tuple for as long as `block.timestamp <= deadline`. Because `revokePusher` only zeroes `namespaceRemapping[pusher]` without invalidating the off-chain signature, the creator can immediately re-establish the delegation after every revocation attempt, making `revokePusher` a no-op within the deadline window.

### Finding Description

`allowPushers` verifies a signature over `(block.chainid, address(this), deadline, pusher, msg.sender)` and writes `namespaceRemapping[pusher] = msg.sender`. [1](#0-0) 

The only replay guard is `_ensureDeadline(deadline)`, which passes for any call made before the deadline. [2](#0-1) 

`revokePusher` zeroes the mapping entry but cannot touch the off-chain signature. [3](#0-2) 

The code comment explicitly acknowledges the concern ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but the deadline only prevents replay *after* expiry, not *within* the window. [4](#0-3) 

The `fallback` push path resolves the target namespace from `namespaceRemapping[msg.sender]`, so a silently re-delegated pusher's data lands in the creator's namespace, not the pusher's own. [5](#0-4) 

### Impact Explanation

A pusher who revokes and resumes pushing to their own namespace (e.g., to serve their own pools) will have their pushes silently redirected to the creator's namespace for the entire remaining deadline window. This produces two simultaneous bad outcomes:

1. **Pusher's own pools receive stale oracle data** — no updates land in the pusher's namespace, causing `timestampMs` to age past `maxTimeDrift`, which causes provider reads to revert or return a zero/stale price, breaking swap and liquidity flows for those pools.
2. **Creator's pools receive the pusher's data** — data formatted for the pusher's own feed layout is written into the creator's slots, potentially producing inverted or unbounded bid/ask quotes consumed by the creator's pools.

Both outcomes satisfy the allowed impact gate: stale/bad-price execution reaching a pool swap, and broken core pool functionality causing loss of funds or unusable withdraw/swap flows.

### Likelihood Explanation

The attack requires only that the creator retain the pusher's original signature (which they received during the initial `allowPushers` call) and that the deadline has not yet expired. No privileged role, no malicious pool setup, and no non-standard token behavior is needed. The creator is a semi-trusted party who already participated in the delegation flow. The window can be up to the full deadline duration (e.g., 1 day as in the proof idea).

### Recommendation

Track used signatures with a `mapping(bytes32 => bool) usedSignatures` and mark each signature hash as consumed on first use. Alternatively, include a monotonically increasing per-pusher nonce in the signed payload and store the last-used nonce on-chain. Either approach makes a replayed signature fail even within the deadline window.

// Foundry test sketch
function test_replayAfterRevoke() public {
    uint256 deadline = block.timestamp + 1 days;
    // pusher signs: keccak256(abi.encode(chainid, oracle, deadline, pusher, creator))
    bytes memory sig = _sign(pusherKey, abi.encode(block.chainid, address(oracle), deadline, pusher, creator));

    // Step 1: creator delegates pusher
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 2: pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 3: creator replays the SAME signature
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds — no replay guard
    assertEq(oracle.namespaceRemapping(pusher), creator);   // pusher is re-delegated

    // Step 4: pusher pushes (thinking it goes to own namespace)
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(slotWord);
    // data lands in creator's namespace, not pusher's
    assertEq(oracle.getOracleData(creatorFeedId).price, pushedPrice);
    assertEq(oracle.getOracleData(pusherFeedId).price, 0); // pusher's own feed is stale
}
