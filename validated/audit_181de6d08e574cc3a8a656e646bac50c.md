### Title
Pusher delegation cannot be truly revoked within the deadline window due to signature replay — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

### Summary
`allowPushers` in `CompressedOracleV1` uses an EIP-191 signature with a deadline to authorize pusher delegation. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature (with the same `deadline`, which is still valid) to silently re-establish the delegation. The pusher's revocation is therefore ineffective for the entire remaining deadline window, and the pusher's own oracle feeds remain permanently stale during that period, rendering any pools that depend on those feeds unable to execute swaps.

### Finding Description

`allowPushers` signs over:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no one-time-use flag, and no on-chain record that the signature was consumed. The only replay barrier is the deadline check in `_ensureDeadline`. After `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`: [2](#0-1) 

the creator can call `allowPushers` again with the **identical** `deadline` and `signature` arguments. The hash is identical, `ECDSA.recover` returns the same pusher address, and `namespaceRemapping[pusher]` is set back to the creator. The code comment in `allowPushers` explicitly acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but incorrectly treats the deadline as a sufficient mitigation — the deadline only bounds the window, it does not prevent replay within it. [3](#0-2) 

The `fallback()` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So while the delegation is re-active, every push from the pusher lands in the creator's namespace. The pusher's own namespace receives no updates; its timestamp stays at 0, which every consumer rejects as stale. [5](#0-4) 

### Impact Explanation

Any pool whose `PriceProvider` / `ProtectedPriceProvider` reads from a feed in the pusher's own namespace will receive `refTime = 0`, triggering the staleness guard and reverting with `FeedStalled` on every swap attempt: [6](#0-5) 

Swaps, liquidity additions, and withdrawals that require a live price quote are all blocked for the duration of the deadline window. This is broken core pool functionality — unusable swap/liquidity flows — satisfying the allowed impact gate.

### Likelihood Explanation

The pusher must have previously signed a consent with a future deadline (a normal operational practice for long-lived delegations). The creator must be willing to replay the signature after revocation. No privileged role is required; `allowPushers` is a public function callable by any address. The pusher has no on-chain mechanism to invalidate the signature before the deadline expires.

### Recommendation

Add a per-pusher nonce to the signed message and store it on-chain. Increment the nonce on every successful `allowPushers` call and on every `revokePusher` / `removePushers` call. Verify the nonce in `allowPushers` so that any previously issued signature is immediately invalidated upon revocation:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
keccak256(abi.encode(block.chainid, address(this), deadline,
                     pusherNonce[pusher], pusher, msg.sender))
// After successful delegation:
pusherNonce[pusher]++;

// In revokePusher / removePushers:
pusherNonce[pusher]++;
```

### Proof of Concept

```solidity
// 1. Pusher signs consent with a 1-year deadline.
uint256 deadline = block.timestamp + 365 days;
bytes memory sig = pusher.sign(
    keccak256(abi.encode(chainid, oracle, deadline, pusher, creator))
);

// 2. Creator establishes delegation.
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator);

// 3. Pusher revokes — intending to push into their own namespace.
vm.prank(pusher);
oracle.revokePusher();
assert(oracle.namespaceRemapping(pusher) == address(0));

// 4. Creator replays the SAME signature — delegation silently restored.
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator); // ← revocation undone

// 5. Pusher's subsequent pushes land in creator's namespace, not their own.
//    oracle.feedIdOf(pusher, slotId, pos) → timestampMs == 0 → FeedStalled
//    on every swap that reads from the pusher's own feeds.
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
