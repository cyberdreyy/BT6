### Title
`revokePusher()` is ineffective during the deadline window — creator can silently re-establish delegation using the original signature - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

---

### Summary

A pusher who calls `revokePusher()` to stop their pushes from landing in a creator's namespace can have their revocation immediately undone by the creator re-calling `allowPushers` with the same signature and deadline. Because `allowPushers` has no nonce or revocation check, the signed consent remains valid until the deadline expires, allowing the creator to re-establish the delegation at will and silently redirect the pusher's price data back into the creator's oracle namespace.

---

### Finding Description

The `allowPushers` function requires a pusher's EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)`. The code comment explicitly acknowledges the risk:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [1](#0-0) 

However, the deadline only prevents re-establishment **after it expires**. During the deadline window, the creator can call `allowPushers` again with the identical `(deadline, [pusher], [sig])` arguments, immediately overwriting `namespaceRemapping[pusher] = address(0)` back to `namespaceRemapping[pusher] = creator`.

The `revokePusher()` function sets `namespaceRemapping[msg.sender] = address(0)`: [2](#0-1) 

But there is no nonce, revocation flag, or one-time-use check on the signature. The creator retains the original signature and can replay it any number of times before the deadline. The `allowPushers` function only checks:
1. `_ensureDeadline(deadline)` — deadline not yet expired
2. `pusher == ECDSA.recover(hash, signatures[i])` — signature is valid

Both conditions remain satisfied after the pusher revokes, so the creator can call `allowPushers` again immediately. [3](#0-2) 

The `fallback()` push path resolves the namespace at call time from `namespaceRemapping[msg.sender]`: [4](#0-3) 

So after re-establishment, every subsequent push by the pusher silently lands in the creator's namespace again, not the pusher's own.

---

### Impact Explanation

After the pusher revokes and begins pushing to their own namespace (e.g., to serve their own pools via `feedIdOf(pusher, slotIndex, positionIndex)`), the creator's re-establishment silently redirects those pushes back to the creator's namespace. The pusher's own pools receive no price updates, causing their oracle feeds to go stale.

`AnchoredPriceProvider._readLeg` returns `ok = false` when the staleness check fires: [5](#0-4) 

This halts swaps in the pusher's pools (`FeedStalled`). If the pusher's pools have a lenient staleness window, they execute swaps against an outdated price — a direct bad-price execution impact on traders.

---

### Likelihood Explanation

Medium. The creator holds the original signature and can replay it at any time before the deadline. A creator who wants to retain a pusher's data feed (e.g., to keep their pools liquid after the pusher attempts to leave) has a direct incentive to do so. The pusher must actively monitor `namespaceRemapping[pusher]` after every revocation to detect re-establishment, which is not enforced by the protocol.

---

### Recommendation

Introduce a per-pusher revocation nonce included in the signed message. Maintain `mapping(address => uint256) public pusherNonce` and increment it on every `revokePusher()` or `removePushers()` call. Include the nonce in the `allowPushers` signature hash:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, pusherNonce[pusher]))
```

This invalidates all previously issued signatures the moment the pusher revokes, making `revokePusher()` truly permanent.

---

### Proof of Concept

```
1. Creator calls allowPushers(deadline = T+1day, [pusher], [sig])
   → namespaceRemapping[pusher] = creator

2. Pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)
   Pusher begins pushing to their own namespace for their own pools.

3. Creator immediately calls allowPushers(deadline = T+1day, [pusher], [sig])
   (same deadline, same signature — both checks still pass)
   → namespaceRemapping[pusher] = creator   ← revocation overwritten

4. Pusher's fallback() pushes now land in creator's namespace again.
   feedIdOf(pusher, slotIndex, positionIndex) receives no updates.

5. Pusher's pools → AnchoredPriceProvider._readLeg → _isStale() = true
   → getBidAndAskPrice() reverts FeedStalled, or executes at stale price.
```

This is the direct analog to the external bug: the pusher's `revokePusher()` call (the "adjustment") is overwritten by the creator re-calling `allowPushers` with the original signature (the "original value"), exactly as `_executeClaim` overwrote the owner's `adjust` by re-initializing `records[beneficiary].total` with the original `purchasedAmount`.

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
